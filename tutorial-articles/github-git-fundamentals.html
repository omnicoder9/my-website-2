Section 1

Git tracks history. GitHub shares it.

Git is the version control system running on your machine. It tracks file changes, lets you create commits, and makes branching cheap. GitHub is the hosting and collaboration layer around Git: remote repositories, pull requests, review, issues, and team visibility.

Repository

A project folder with Git history attached.

Commit

A saved checkpoint with a message explaining what changed.

Branch

An isolated line of work so you can change code without destabilizing main.

Pull Request

A GitHub review and merge proposal for a branch.

The core beginner mistake is treating Git and GitHub as the same thing. They are not. You can use Git without GitHub, but most modern teams still use GitHub as the shared place where the remote repository lives.

Section 2

Do the one-time setup first

Before touching a project, make sure Git knows who you are and that you can authenticate with GitHub. You only do most of this once per machine.

Terminal 
git --version
git config --global user.name "Your Name"
git config --global user.email "you@example.com"
git config --global init.defaultBranch main

On GitHub, create an account if you do not already have one. For authentication, choose either HTTPS with a credential manager or SSH keys. HTTPS is usually the easier starting point. SSH is common once you use GitHub daily.

Important: GitHub no longer accepts your account password for Git operations over HTTPS. Use the browser/device login flow, Git Credential Manager, or SSH keys.

Git identity

Your name and email are stored in each commit unless you override them per repository.

Default branch

Setting main up front prevents older defaults like master.

Authentication

HTTPS is fine. SSH is great. The important part is using a supported GitHub method.

Section 3

The quickest GitHub-first workflow

If you are new, the least confusing path is to create the repository on GitHub first and then clone it locally. That avoids remote setup details on day one.

Step 1: Create the repository on GitHub

  • Click New repository.
  • Choose a name such as git-practice.
  • Pick Public or Private.
  • Optionally add a README and a .gitignore.

Step 2: Clone it locally

Clone and inspect 
git clone https://github.com/your-username/git-practice.git
cd git-practice
git status

git clone copies the project and its entire history to your machine. git status is the command you should trust most when you are unsure what Git sees.

Step 3: Make a small change

Edit README.md or add a new file. Then stage and commit the change.

Stage and commit 
git status
git add README.md
git commit -m "Update README with setup notes"

The staging area is where you decide what belongs in the next commit. Think of git add as selecting the exact change set you want recorded.

Step 4: Push the commit to GitHub

Push to remote 
git push origin main

origin is the default name Git gives the remote repository you cloned from. main is the branch you are pushing to on GitHub.

Mental model: Your work is not safely shared until it exists both as a local commit and on the remote GitHub repository.

Section 4

Starting from an existing local folder

Sometimes you already have files locally and want to connect them to a brand-new GitHub repository. In that case, initialize Git yourself and then add the GitHub remote.

Existing project setup 
cd your-project
git init
git add .
git commit -m "Initial commit"
git branch -M main
git remote add origin https://github.com/your-username/your-project.git
git push -u origin main

The first push uses -u so Git remembers that your local main branch tracks origin/main. After that, plain git push and git pull usually work.

Section 5

Use branches for real work, then merge with a pull request

Teams usually do not commit directly to main. They create a feature branch, push it to GitHub, and open a pull request for review. This is the normal GitHub workflow.

Branch workflow 
git switch -c feature/readme-refresh
# edit files
git add README.md
git commit -m "Clarify local setup steps"
git push -u origin feature/readme-refresh

After pushing, GitHub will usually show a button to create a pull request. In the pull request, describe what changed, why it changed, and any review context your team needs.

1. Branch

Create a short-lived branch for a focused task.

2. Commit

Keep commits small enough that a reviewer can understand them.

3. Push

Publish the branch to GitHub with git push -u origin <branch>.

4. Open PR

Use GitHub to discuss, review, and merge back into main.

Once the pull request is merged, switch back to main, pull the latest changes, and delete the branch if it is done.

After merge 
git switch main
git pull --ff-only
git branch -d feature/readme-refresh

Section 6

Stay in sync with the remote repository

The most common operational mistake is starting new work on stale code. Before making substantial changes, update your local copy of the branch you depend on.

git fetch

Downloads remote history but does not merge it into your current branch.

git pull

Fetches and then integrates remote changes into your current branch.

git pull --ff-only

Safer when learning because it refuses surprise merge commits.

Safe sync routine 
git switch main
git pull --ff-only
git switch feature/my-work
git status

If your team uses pull requests heavily, this simple rhythm covers most of what you need: update main, branch from current main, work, commit, push, open PR, merge, repeat.

Section 7

When a merge conflict happens

A conflict means Git found overlapping edits it cannot safely resolve by itself. This is normal. It is not a sign that the repository is broken.

Conflict markers 
<<<<<<< HEAD
Your version of the code
=======
Incoming version of the code
>>>>>>> origin/main

Resolve the conflict with a disciplined sequence:

  1. Open the conflicted file and decide what the final code should be.
  2. Remove the conflict markers.
  3. Run tests or at least re-read the changed section carefully.
  4. Stage the resolved file with git add.
  5. Finish the merge or continue the rebase, depending on the workflow you used.
Beginner rule: If you do not understand a conflict, stop and inspect the surrounding code before clicking “accept current” or “accept incoming.” Conflict resolution is a code change, not a button press.

Section 8

Good habits that prevent avoidable Git pain

Commit small units

Small commits are easier to review, revert, and reason about.

Write useful messages

Prefer messages that explain the intent, not just “misc changes.”

Pull before you branch

Starting from stale main creates conflicts you did not need.

Never commit secrets

API keys, tokens, and private credentials should stay out of Git entirely.

Use .gitignore

Ignore build output, local environment files, and editor noise.

Trust git status

When uncertain, check status before doing anything else.

Section 9

Command reference for the fundamentals

Command What it does
git status Shows changed files, staged files, branch state, and tracking information.
git add <file> Stages a file for the next commit.
git commit -m "message" Creates a commit from the staged changes.
git push Uploads local commits to the tracked remote branch.
git pull --ff-only Updates your branch from the remote without creating an automatic merge commit.
git switch -c name Creates and switches to a new branch.
git branch -d name Deletes a local branch that has already been merged.
git clone <url> Copies a remote GitHub repository and its history to your machine.
git remote -v Shows the remotes connected to your local repository.

If you can reliably clone, status, add, commit, push, pull, branch, and open a pull request, you already understand the Git and GitHub workflow that most teams use every day.
