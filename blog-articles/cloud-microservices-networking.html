Technical Review · Cloud Infrastructure

Cloud & Microservices Networking:
Architecture, Patterns, and Frontiers

From virtual private clouds and subnet design to service meshes and east–west traffic management—a comprehensive technical survey of how modern distributed systems communicate.

·  by Nicholas Knapp  ·  2026

Abstract

The shift from monolithic applications to distributed microservices architectures has fundamentally transformed how networks must be designed, secured, and operated. This article surveys the core concepts of cloud and microservices networking, tracing the historical evolution from Amazon's first Virtual Private Cloud (VPC) in 2009 through to modern service meshes, eBPF-accelerated data planes, and zero-trust security paradigms. We examine the key infrastructure primitives—VPC design, public and private subnets, API gateways, and service meshes—alongside the canonical traffic patterns they govern: north–south (client-to-service) and east–west (service-to-service) flows. We also identify persistent engineering challenges and the innovations currently reshaping the landscape.

01 / IntroductionThe Networking Revolution in Cloud-Native Systems

Traditional enterprise networking was designed around a clear perimeter: a fortified boundary separated trusted internal systems from the hostile public internet. Monolithic applications communicated with their databases through well-understood, static connections within a corporate data centre. Security teams knew exactly where the network boundary lay, and a single firewall often sufficed to guard it.

The rise of cloud computing and microservices architecture has dismantled this model entirely. A single user request to a modern application might traverse dozens of discrete services—an API gateway, an authentication service, a product catalogue, a payments engine—each potentially running in different containers, on different physical hosts, possibly in different geographic regions, all communicating over a shared network fabric. [1]

This transformation has generated an entirely new discipline of cloud networking: one concerned with isolating multi-tenant workloads, encrypting service-to-service traffic, enforcing fine-grained policy without human operators touching every packet, and observing the behaviour of systems too complex to reason about manually. This article surveys that discipline from first principles to its present frontier.

⚡

Scope of this article: We focus on the networking layer of cloud and microservices architectures—specifically VPC design, subnet topology, API gateways, service meshes, and the traffic patterns (east–west and north–south) that define how these components interact. Security, observability, and emerging technologies such as eBPF are treated as they relate to networking.

02 / Historical ContextFrom Flat Networks to Virtual Isolation

To understand the sophistication of modern cloud networking, it helps to trace where it came from.

2006
Amazon launches EC2 on its original "EC2-Classic" flat network—a single shared network fabric where customer instances ran alongside one another, separated only by security groups applied at the instance level. [2]
2009
Amazon introduces the Virtual Private Cloud (VPC) on 25 August 2009, initially as a limited beta. The VPC allowed customers to provision logically isolated EC2 instances within a private IP subnet, with an optional VPN connection back to their on-premises data centre. [2,3]
2011
AWS significantly extends VPC capabilities: enhanced Security Groups with egress filtering, direct internet connectivity, custom routing tables, and Network ACLs for subnet-level traffic control. [2]
2013
VPC becomes the default networking model for new AWS accounts. EC2-Classic is formally deprecated and eventually retired in August 2022. [3]
2015
VPC Flow Logs announced, enabling capture of IP traffic metadata to CloudWatch Logs and S3—a foundational observability tool. [4]
2017
Istio announced by Google, IBM, and Lyft (Envoy team), heralding the mainstream arrival of the service mesh. Linkerd, derived from Twitter's internal RPC framework, had already been open-sourced in 2016. [5,6]
2019
IBM Cloud launches IBM Cloud VPC (June 2019); major cloud providers have by now converged on VPC as the universal isolation primitive. [7]
2022–2025
eBPF emerges as a kernel-level alternative to sidecar proxies. Istio's "Ambient Mesh" mode (sidecar-less) gains traction; Istio graduates into the CNCF. The Kubernetes Gateway API matures to address ingress controller limitations. [8,9,10]

03 / Core ConceptVirtual Private Cloud (VPC) Design

A Virtual Private Cloud (VPC) is an on-demand, configurable pool of shared cloud resources allocated within a public cloud environment, providing logical isolation between different tenants. The isolation is typically achieved through a private IP subnet allocation combined with virtual communication constructs such as VLAN tags or VXLAN overlays—mechanisms that ensure one customer's VPC traffic never mixes with another's, even when traversing shared physical hardware. [7]

From an operator's perspective, a VPC closely resembles a traditional data centre network. Administrators control IP address ranges via CIDR notation, define subnets, configure route tables, and attach gateways—all without touching physical hardware. [3] On AWS, for example, a VPC requires a primary IPv4 CIDR block ranging from /16 to /28, with secondary blocks available for address space expansion up to a default limit of five per VPC. [4]

"VPCs combine the scalability and convenience of public cloud computing with the data isolation of private cloud computing—imagine a public cloud as a crowded restaurant, and a VPC as a reserved table that only the party who made the reservation can access."

Cloudflare Learning Centre [1]

Key VPC Primitives

CIDR Block
The IP address range assigned to the VPC (e.g. 10.0.0.0/16). Carves out the space from which all subnets are allocated.
Route Table
A set of rules determining where VPC traffic is directed. Each subnet is associated with exactly one route table.
Internet Gateway
Attached to a VPC to provide resources in public subnets with bidirectional internet access.
NAT Gateway
Enables resources in private subnets to initiate outbound internet connections without being reachable from the internet.
Security Group
Stateful firewall applied at the instance/interface level, controlling inbound and outbound traffic rules.
Network ACL
Stateless firewall applied at the subnet boundary, offering an additional layer of access control.
VPC Peering
Enables private IP routing between two VPCs, even across AWS accounts or regions.
VPC Endpoint
Private connectivity to AWS services (e.g. S3) without traversing the public internet or requiring a gateway.

Subnet Design: Public vs. Private

Within a VPC, the network is subdivided into subnets—ranges of IP addresses that must reside entirely within a single Availability Zone. Subnet type is not a fixed attribute but rather emerges from how routing is configured. [11]

A public subnet has a route in its table that points to an internet gateway, enabling instances within it to communicate directly with the public internet and receive public IP addresses. Web servers, load balancers, and API gateway endpoints typically reside here. [11]

A private subnet has no direct route to an internet gateway. Instances can only access the internet through a NAT gateway located in a public subnet, and are not directly reachable from outside the VPC. Databases, application servers, message queues, and internal microservices typically inhabit private subnets. AWS explicitly recommends placing most resources in private subnets and using bastion hosts or NAT devices for any required internet access. [11]

Canonical Two-Tier VPC Architecture (Public + Private Subnets)

Multi-Cloud and Hybrid VPC Patterns

Modern organisations rarely confine themselves to a single cloud provider. In multi-cloud deployments, VPCs in different clouds (AWS, Azure, GCP) can be interconnected using private links or site-to-site VPNs, distributing workloads across providers while maintaining security and control and reducing the risk of vendor lock-in. [12] Hybrid cloud patterns extend this further, connecting on-premises data centres to cloud VPCs via VPN or dedicated interconnects (AWS Direct Connect, Azure ExpressRoute, GCP Cloud Interconnect), enabling gradual migration strategies where legacy systems remain on-premises while modern services adopt cloud infrastructure. [12]

04 / Core ConceptAPI Gateways

An API gateway is an API management tool that sits between clients and back-end services, acting as the single entry point into a distributed system and enabling multiple APIs or services to behave cohesively for the consumer. [6] When a client makes a request, the gateway can decompose it into multiple internal requests, route each to the appropriate service, aggregate the responses, and return a unified result to the caller—all transparently.

API gateways are primarily associated with managing north–south traffic: requests that originate outside the system boundary (the public internet, mobile clients, third-party APIs) and flow inward toward services. [13] Historically, this was their defining role—protecting and mediating access to back-end systems.

Core Capabilities

Modern API gateway implementations process up to 35,000 requests per second at peak load while maintaining consistent latency. [14] Their capability set has expanded considerably beyond simple request routing to include:

Authentication and authorisation — JWT validation, OAuth 2.0 flows, API key management, and integration with identity providers. Rate limiting and throttling — preventing abuse and protecting back-end services from traffic spikes, including defence against credential-stuffing attacks and DDoS. TLS termination — offloading HTTPS handling from individual services, reducing their complexity. Request transformation — translating between protocols (REST, gRPC, WebSocket), adapting payloads for legacy back-ends, and version management. Observability — centralised logging, distributed tracing integration, and metrics collection for all inbound traffic.

Notable Implementations

The ecosystem of API gateway software spans both open-source and managed cloud services. Kong (based on nginx/OpenResty), Apigee (Google), AWS API Gateway, Azure API Management, and Envoy-based projects such as Ambassador and Gloo are among the most widely deployed. The Kubernetes Gateway API, which reached its first stable release in 2023, provides a standardised, role-oriented abstraction over ingress controllers and is positioned to supersede the original Ingress resource. [15]

Example — Kubernetes Gateway API (YAML)
apiVersion: gateway.networking.k8s.io/v1

  name: payments-route

05 / Core ConceptService Meshes

A service mesh is a dedicated infrastructure layer that manages all service-to-service (east–west) traffic within a distributed system. It provides both functional operations—such as dynamic routing, load balancing, and traffic splitting—and non-functional support, including security policy enforcement, reliability patterns (retries, timeouts, circuit-breaking), and telemetry collection. [5]

The defining characteristic of a service mesh is that its capabilities are delivered at the platform level, without requiring application code changes. Services remain unaware of the mesh; they communicate as if they were speaking directly to one another, while the mesh layer intercepts and enriches that communication transparently.

Architecture: Control Plane and Data Plane

Service meshes are structured as two distinct planes. The data plane consists of lightweight proxy processes (sidecars) deployed alongside each service instance. They intercept all inbound and outbound network traffic and enforce the policies distributed by the control plane. Envoy Proxy—an open-source, cloud-native proxy developed originally at Lyft—is the dominant data plane component, used by Istio, Gloo, and others. [5]

The control plane acts as the brain: it maintains a live registry of all service instances and endpoints, distributes routing rules and security policies to the sidecar proxies via xDS APIs (Envoy's configuration protocol), acts as a certificate authority for mutual TLS (mTLS) identity, and aggregates telemetry. In Istio, the control plane is consolidated into a single binary called Istiod. [16]

"Istio was originally created by Google and IBM in partnership with the Envoy team from Lyft—uniting infrastructure teams from three distinct engineering cultures around a shared vision for service communication."

InfoQ, Service Mesh Ultimate Guide [5]

Key Capabilities in Practice

Mutual TLS (mTLS) is perhaps the most operationally significant capability: it authenticates both parties in every service-to-service connection and encrypts the channel, without any application-level code changes. The service mesh control plane acts as a certificate authority, automatically generating and rotating certificates. [17]

Traffic management enables sophisticated deployment patterns. Istio's VirtualService and DestinationRule resources allow operators to define percentage-based traffic splits (enabling canary releases and A/B tests), route based on HTTP headers, inject faults for chaos testing, and apply per-route retry and timeout policies. [18]

Observability is provided as a default: because all traffic passes through Envoy sidecars, the mesh automatically generates distributed traces, service-level metrics (latency, error rate, throughput), and access logs without instrumentation code in individual services.

06 / Traffic PatternsNorth–South and East–West Traffic

Cloud networking literature distinguishes traffic by the direction it flows relative to the system boundary—a metaphor borrowed from how network diagrams have traditionally been drawn.

North–South Traffic: Client-to-Service

North–south traffic crosses the cluster or data centre boundary. Inbound (northbound) traffic originates externally—from browsers, mobile applications, or third-party services—and flows into the system, typically entering at an API gateway or load balancer. Outbound (southbound) traffic originates within the cluster and leaves toward external dependencies, such as a payment processor or third-party data API. [13]

North–south traffic management focuses on edge concerns: TLS termination, authentication, rate limiting, WAF rules, DDoS mitigation, and protocol adaptation. The API gateway is the canonical tool for managing northbound flows, while egress controllers and DNS allowlists govern southbound egress. [13]

East–West Traffic: Service-to-Service

East–west traffic flows laterally within the cluster boundary—from one microservice to another. The term's origin is visual: traditional data centre diagrams were drawn with inter-service traffic moving horizontally across the page (east to west). [5] In a modern microservices deployment, east–west traffic typically dwarfs north–south traffic in volume; a single user-facing request may fan out into dozens of internal service calls.

East–west traffic management is the primary concern of service meshes. Key challenges include service discovery (knowing where other service instances currently reside), load balancing across dynamic instance pools, implementing circuit-breaker patterns to prevent cascading failure, and enforcing mTLS to prevent lateral movement in the event of a compromised service. [5]

North–South vs East–West Traffic Flow

The API Gateway vs. Service Mesh Distinction

A common simplification holds that API gateways handle north–south traffic while service meshes handle east–west. While this directional framing is useful as a mental model, it is not architecturally strict. [19] API gateways can manage east–west traffic through internal gateways governing inter-service communication, policy application, and versioning. Conversely, service meshes such as Istio handle north–south traffic through mesh-aware ingress gateways (Istio's Gateway resource) that bring external traffic flows under the same control plane as internal traffic. [19] The more meaningful distinction is one of purpose: API gateways converge traffic at a central point for boundary management, whereas service meshes distribute policy enforcement peer-to-peer across all service proxies. [19]

07 / Engineering ChallengesPersistent Problems in Cloud Networking

Despite significant advances, cloud and microservices networking carries substantial operational complexity. The challenges below represent active areas of engineering and research.

ChallengeSeverityDescription
Operational Complexity of Service Meshes High Running a production service mesh is hard. Sidecar proxies add resource overhead per pod; misconfiguration is subtle and consequential. For many teams, the operational burden outweighs the benefits. [8]
Observability at Scale High East–west traffic in large deployments generates enormous volumes of traces, logs, and metrics. Correlating these across hundreds of services to diagnose a latency regression is a significant unsolved operational problem.
Multi-Cluster & Multi-Cloud Networking High Cross-cluster service discovery and policy enforcement lack standardised formats and governance models. Under-standardisation risks creating fragmentation as brittle as the legacy systems cloud was meant to replace. [20]
Network Security Surface Area Medium Each microservice is a potential attack surface. API abuse (credential stuffing, DDoS) and lateral movement via compromised services represent concrete threats—illustrated by the 2024 Roku credential-stuffing attacks affecting 591,000 accounts. [21]
IP Address Exhaustion Medium Large Kubernetes clusters in IPv4-only environments can exhaust subnet allocations rapidly, particularly when each pod consumes its own IP. IPv6 dual-stack support in VPCs partly addresses this but introduces its own routing complexity. [11]
Latency Overhead of Sidecar Proxies Medium Each service-to-service call traverses two Envoy sidecars (source and destination), adding per-hop latency. For latency-sensitive applications or edge devices with constrained compute, this overhead is material. [22]
Configuration Drift and Misconfigurations Ongoing Cloud migration, continuous deployment, and the evolution of APIs increase the risk of security misconfigurations—open security groups, overly permissive ACLs, misconfigured mTLS policies. [21]

08 / Emerging InnovationsThe Next Frontier

eBPF: Kernel-Level Networking

Extended Berkeley Packet Filter (eBPF) allows safely running sandboxed programs in the Linux kernel without changing kernel source code. Applied to networking, eBPF enables high-performance packet processing, observability, and policy enforcement at kernel level—bypassing the user-space sidecar proxy entirely. Projects such as Cilium use eBPF to implement network policies, load balancing, and even a service mesh data plane, with significantly lower latency and memory overhead than sidecar architectures. [22,15] Standards bodies including CNCF increasingly recognise eBPF as foundational infrastructure for cloud-native networking. [20]

Ambient Mesh (Sidecar-less Service Mesh)

Istio's Ambient Mesh mode, announced in 2022 and under active development, restructures the service mesh data plane to eliminate per-pod sidecars. Instead, it uses per-node proxies ("ztunnel") for mTLS and basic telemetry, with optional Layer-7 waypoint proxies deployed only for services that require advanced routing. Early benchmarks suggest Ambient Mesh can reduce service mesh infrastructure costs by up to 92% compared to traditional sidecar deployments. [24] This represents a significant step toward making service meshes operationally invisible.

Zero Trust Networking

The zero-trust security model—"never trust, always verify"—is reshaping cloud network architecture. Rather than relying on VPC perimeter controls to grant implicit trust to internal traffic, zero-trust treats every request as potentially hostile and requires continuous authentication and authorisation at each hop. Research indicates that organisations adopting zero-trust architectures have achieved up to a 50% reduction in breach impact. [23] Practically, this manifests as mTLS everywhere (enforced by service meshes), SPIFFE-based workload identity, and fine-grained authorisation policies applied at the service level rather than the network level.

SPIFFE and Workload Identity

SPIFFE (Secure Production Identity Framework For Everyone) provides a standard for cryptographic workload identity in cloud-native environments. Rather than identifying services by their network location (IP address and port), SPIFFE issues short-lived X.509 certificates (SVIDs) tied to the workload's identity. This enables strong, portable authentication that survives IP address changes—essential in dynamic container environments where pod IPs change constantly. SPIFFE is increasingly adopted alongside eBPF and the Kubernetes Gateway API as an open-standard antidote to proprietary vendor lock-in. [20]

AI-Native Gateways

The proliferation of large language model APIs has introduced a new class of north–south traffic: inference requests destined for GPU clusters running LLM workloads. Vendors are beginning to ship AI-native gateway products that understand LLM-specific protocols (model context protocol, agent-to-agent communication), provide inference routing, and apply policies specific to AI workloads. This represents the latest evolution of the API gateway concept—extending established north–south management patterns to entirely new protocol semantics. [24]

09 / Conclusion

Cloud and microservices networking has evolved from a simple model of shared flat networks and perimeter firewalls into a rich, layered discipline involving logically isolated VPCs, sophisticated subnet topology, API gateways managing boundary traffic, and service meshes enforcing policy across thousands of service-to-service connections. The canonical patterns—north–south traffic managed at the edge, east–west traffic managed by the mesh—provide a useful architectural vocabulary, though the real-world boundary between these tools is increasingly blurred as both categories expand their capabilities.

The persistent challenges—operational complexity, multi-cloud fragmentation, latency overhead, and security surface area—are driving a new generation of innovations: eBPF-accelerated data planes, sidecar-less mesh architectures, zero-trust identity models, and AI-native gateways. The trajectory is toward networking infrastructure that becomes progressively more invisible to application developers while growing more capable and more secure—infrastructure that, as one observer put it, "just works." [8]

For practitioners, the practical implication is clear: invest in understanding VPC design and subnet topology as foundational; adopt API gateways thoughtfully as the control point for external traffic; evaluate service meshes against your organisation's operational maturity; and watch the eBPF and ambient mesh space closely as these technologies approach production readiness at scale.
