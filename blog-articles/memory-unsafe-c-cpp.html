Unsafe

A field guide to the four bugs that have defined four decades of insecure software: buffer overflows, use-after-free, null-pointer dereferences, and data races in C and C++.

A technical longread · with bidirectional Vancouver citations

In 1988, a graduate student's worm crippled an estimated ten percent of the computers then connected to the internet by, among other tricks, feeding an oversized string into a 512-byte stack buffer that a Unix daemon read with the notoriously unsafe gets() function.1 Nearly forty years later, the descendants of that same bug class still account for the majority of severe security vulnerabilities in the world's most important software.

The reason is structural. C and, by inheritance, C++ were designed to give the programmer direct, unmediated control over memory. There is no runtime that checks whether an array index is in bounds, no garbage collector that decides when an object dies, and no enforced rule that a pointer still points at something valid. That control is precisely what makes the languages fast and suitable for operating systems, embedded firmware, browsers, and cryptographic libraries. It is also what makes them dangerous: the compiler will happily emit code that writes past the end of a buffer, reads freed memory, or dereferences a null pointer, and the standard declares the result not an error but undefined behavior — a blank cheque that the implementation may cash however it likes.

This article walks through the four canonical memory-management defects of C and C++ — what they are, where they came from, how they are exploited, and how they are defended against — with runnable source code and the historical incidents that made each one infamous.

01 / FoundationsThe memory model that makes it possible

A running C or C++ program divides its address space into regions. Code and read-only constants sit in their own segments. Global and static variables live in the data and BSS segments. Two regions matter most for the bugs that follow: the stack, which grows and shrinks automatically as functions are called and returned, holding local variables and bookkeeping such as saved return addresses; and the heap, a pool of memory the programmer allocates and frees explicitly with malloc/free in C or new/delete in C++.

The defining property of these languages is that the programmer, not the runtime, is responsible for the rules. An array does not know its own length. A pointer does not know whether the object it addresses is still alive. Freeing memory does not change the value of the pointers that referenced it. None of these operations is checked at runtime, because checking would cost cycles that the languages were specifically designed not to spend.

When a program violates a rule the standard cares about — indexing out of bounds, reading an uninitialized value, using a pointer after the object's lifetime has ended — the behavior is undefined. The compiler is entitled to assume such situations never happen, and it optimizes accordingly. In practice this means a memory bug rarely produces a tidy, immediate error. It may appear to work, corrupt unrelated data, crash much later, or hand control of the process to an attacker. This gap between "the program seems to run" and "the program is correct" is the soil in which every defect below grows.

Four bugs, one root cause: in C and C++, nothing at runtime is obligated to stop you from touching memory you have no right to touch.

The security community has given these defects stable identifiers in the Common Weakness Enumeration (CWE) maintained by MITRE. Throughout this article they serve as precise names for what would otherwise be loosely described "memory bugs."

02 / CWE-787 & CWE-125Buffer overflows

A buffer overflow occurs when a program writes (or reads) past the bounds of a fixed-size region of memory. Because C arrays carry no length and most string functions stop only at a terminating byte or a caller-supplied count, supplying more data than the buffer can hold simply spills into whatever lies adjacent — other variables, saved registers, or, most dangerously, the function's saved return address.

The bug that started the field

The canonical example is the Morris worm of November 2, 1988. One of its propagation vectors targeted the BSD fingerd daemon, which read a network request into a fixed stack buffer using gets() — a function that performs no bounds checking whatsoever.12 By sending a carefully constructed string longer than the buffer, the worm overwrote the saved return address on the stack so that, when the function returned, execution jumped into attacker-supplied code rather than back to the caller.2 The incident infected roughly six thousand machines, prompted the creation of the first CERT coordination center, and produced the first conviction under the U.S. Computer Fraud and Abuse Act.2

The technique was largely folklore until 1996, when Elias Levy — writing as "Aleph One" — published Smashing the Stack for Fun and Profit in Phrack issue 49. It was the first high-quality, step-by-step public explanation of how stack buffer overflows work and how to exploit them, and it shaped a generation of both attackers and defenders.3

Anatomy of a stack smash

Consider a function that copies an external string into a local array. The vulnerable version trusts the input; the fixed version bounds it.

vulnerable.cCWE-787
#include <string.h>

void handle(const char *input) {
    char name[64];            // fixed 64-byte stack buffer
    strcpy(name, input);       // no length check: copies until '\0'
    // if input is longer than 63 bytes, strcpy writes past `name`,
    // clobbering saved registers and the return address.
    use(name);
}                              // <- return jumps to whatever now sits in the slot
hardened.cbounded
#include <string.h>

void handle(const char *input) {
    char name[64];
    // copy at most 63 bytes, always leave room for the terminator
    snprintf(name, sizeof name, "%s", input);
    name[sizeof name - 1] = '\0';  // defensive: guarantee NUL
    use(name);
}

The fix is not "use a bigger buffer" — it is "never write more than the buffer holds." Functions like gets, strcpy, and sprintf have no notion of a destination size; their bounded counterparts (fgets, strncpy/strlcpy, snprintf) do.

On the stack, the local buffer typically sits below the saved frame pointer and return address. Writing past the buffer marches upward into those slots. Overwriting the return address with the address of injected code (classic stack smashing) or with the address of an existing function or instruction sequence (return-oriented programming, the modern technique used once stacks became non-executable) redirects control flow.3

Heap overflows and over-reads

Overflows are not confined to the stack. Writing past a heap allocation can corrupt allocator metadata or adjacent objects, and reading past a buffer — an over-read, CWE-125 — leaks whatever happens to be in adjacent memory. The most consequential over-read in history is Heartbleed (CVE-2014-0160).

Case study · Heartbleed (CVE-2014-0160)

OpenSSL's implementation of the TLS heartbeat extension trusted a length field supplied by the remote peer. A client could claim its small payload was up to 64 KB long; OpenSSL allocated a response buffer and memcpy'd that many bytes from the request — copying far past the actual payload and returning whatever lived in adjacent process memory, including private keys, passwords, and session cookies.1415

The flaw was a single missing bounds check before a memcpy using non-sanitized user input as the length.15 Introduced in 2012 and disclosed in April 2014, it affected OpenSSL 1.0.1 through 1.0.1f and was fixed in 1.0.1g by validating the declared length against the real record length.14

Heartbleed illustrates two points. First, a memory bug need not write anything to be catastrophic — reading too much was enough to undermine confidentiality across a large fraction of the web's servers. Second, the defect was textbook: trusting an attacker-controlled length without checking it against the buffer's real size.

Primary mitigations

Defenders layered several mechanisms in response to stack smashing: stack canaries, a random value placed before the return address and checked on return; non-executable memory (NX/DEP), which marks the stack and heap as data-only so injected shellcode cannot run; and Address Space Layout Randomization (ASLR), which randomizes the base addresses of the stack, heap, and libraries so an attacker cannot reliably predict where to jump.3 Compiler-side, GCC and Clang's _FORTIFY_SOURCE and -fstack-protector add bounds-aware versions of common functions and canary checks. None of these eliminates the bug; they raise the cost of exploiting it.

03 / CWE-416Use-after-free & double-free

Use-after-free (UAF) is the act of dereferencing a pointer to memory that has already been freed. In C and C++, calling free or delete returns the memory to the allocator but does not alter the pointer — it becomes a dangling pointer still holding the old address. Reading or writing through it touches memory whose contents and ownership are now undefined.

uaf.cppCWE-416
struct Session { int id; char token[32]; };

void logout(Session *s) {
    delete s;                  // memory returned to the allocator...
}                              // ...but the caller's pointer still holds the address

void flow() {
    Session *s = new Session{7, "secret"};
    logout(s);
    // s is now DANGLING. The next line is undefined behavior:
    printf("%d\n", s->id);   // may print garbage, crash, or read attacker data
}
raii.cppownership-managed
#include <memory>

struct Session { int id; char token[32]; };

void flow() {
    auto s = std::make_unique<Session>(Session{7, "secret"});
    printf("%d\n", s->id);
    // unique_ptr frees the Session automatically at end of scope.
    // There is no raw pointer left to dangle, and no manual delete.
}

RAII — "Resource Acquisition Is Initialization" — ties the object's lifetime to a scope. std::unique_ptr and std::shared_ptr express ownership in the type system so the compiler frees memory deterministically and no dangling raw pointer remains.

Why UAF is dangerous, not just buggy

The reason UAF is a security problem and not merely a crash is the allocator's behavior. After free, the allocator may hand the same block to the next allocation request. An attacker who can trigger an allocation of the right size between the free and the dangling use can place their own data where the freed object used to be — a technique called heap grooming or "feng shui." If the dangling pointer is then used as, say, a C++ object with a vtable, the attacker controls the function pointers it invokes.

A close relative is the double-free: calling free twice on the same pointer. This corrupts the allocator's internal free lists and, in many classic allocators, can be manipulated into an arbitrary write. Microsoft's analysis of its own vulnerability data found use-after-free among the recurring root causes of memory-safety CVEs, alongside heap out-of-bounds access, type confusion, and uninitialized use.10

A freed pointer is a loaded weapon left on the table. C and C++ never clear it for you.

Primary mitigations

The modern C++ answer is to stop managing lifetimes by hand: prefer stack objects and RAII, express ownership with unique_ptr (single owner) and shared_ptr (reference-counted shared owner), and use weak_ptr to hold non-owning references that can be checked for validity. Setting raw pointers to nullptr immediately after freeing converts a silent UAF into a more visible null dereference. At the allocator level, hardened allocators such as Android's Scudo are specifically designed to resist heap-based use-after-free, double-free, and overflow exploitation.20

04 / CWE-476Null-pointer dereferences

A null-pointer dereference is an attempt to read or write through a pointer whose value is null — a sentinel meaning "points at nothing." The concept itself has a famous origin and a famous apology.

The billion-dollar mistake

Sir Tony Hoare introduced the null reference in 1965 while designing the type system for ALGOL W, reasoning that it was trivial to implement. In a 2009 talk he renounced it: he called it his "billion-dollar mistake," arguing that the single null reference had since caused innumerable errors, vulnerabilities, and system crashes.4

In C and C++ the consequences are sharper than in managed languages. Dereferencing null is itself undefined behavior. On a typical desktop operating system the virtual page at address zero is unmapped, so the access faults and the process is killed — a denial of service, but a relatively clean one. On embedded systems without memory protection, or where address zero is mapped, the same operation may silently read or write real memory. Worse, because the standard says dereferencing null cannot happen, optimizing compilers sometimes delete the very null checks programmers add after a dereference, on the logic that the pointer must have been non-null to reach that point — turning a defensive check into a removed one.

null.cppCWE-476
Config *load(const char *path);   // returns nullptr on failure

int port_of(const char *path) {
    Config *c = load(path);      // may fail and return nullptr
    return c->port;             // dereference without checking -> UB / crash
}
checked.cppvalidated
#include <optional>

std::optional<int> port_of(const char *path) {
    Config *c = load(path);
    if (c == nullptr)           // handle the absent case explicitly
        return std::nullopt;
    return c->port;
}
// Better still: design APIs to return references or std::optional,
// so "no value" is a type the caller is forced to handle.

The structural fix is to make absence visible in the type. A std::optional<T> or a reference (T&, which cannot legally be null) forces the caller to confront the missing-value case at compile time rather than discover it at runtime.

Defensively, the rules are simple: check any pointer that can be null before using it; prefer references over pointers when the value is mandatory; and model optionality with std::optional rather than a bare nullable pointer. Static analyzers and the compiler's own -Wnull-dereference warning catch many of these paths before they ship.

05 / CWE-362Data races & the memory model

The first three defects are about space — touching the wrong bytes. A data race is about time. Before 2011, the C and C++ standards did not even acknowledge threads; multithreading was a platform extension layered on top of a single-threaded language. C++11 (and C11) changed that by defining a formal multithreaded memory model, and with it, a precise definition of the data race.

Under that model, two evaluations conflict if they access the same memory location and at least one is a write. A program has a data race if two conflicting accesses occur on different threads, at least one is non-atomic, and neither happens-before the other. If a data race occurs, the behavior of the entire program is undefined.16

A data race is not "a bug that sometimes prints the wrong number." It is undefined behavior for the whole program.

This is stronger than intuition suggests. Programmers often imagine a racy counter++ simply loses some increments. In reality, because the behavior is undefined, the compiler may assume the race never happens and optimize on that basis, producing outcomes that no interleaving of the source statements could explain.

race.cppCWE-362
#include <thread>

int counter = 0;                 // plain int, shared, non-atomic

void work() {
    for (int i = 0; i < 100000; ++i)
        ++counter;                // read-modify-write, unsynchronized
}

int main() {
    std::thread t1{work}, t2{work};   // two threads, same `counter`
    t1.join(); t2.join();
    // DATA RACE -> undefined behavior. The total is not merely
    // "less than 200000"; the program has no defined meaning.
}
atomic.cppsynchronized
#include <thread>
#include <atomic>

std::atomic<int> counter{0};      // atomic: no data race by definition

void work() {
    for (int i = 0; i < 100000; ++i)
        counter.fetch_add(1, std::memory_order_relaxed);
}

int main() {
    std::thread t1{work}, t2{work};
    t1.join(); t2.join();
    // Defined behavior; final value is exactly 200000.
    // (Use a std::mutex instead when guarding multiple variables.)
}

cppreference states the rule directly: three threads incrementing a plain shared int is undefined behavior; the same threads incrementing a std::atomic<int> are well-defined.16

Happens-before and memory order

The way to make conflicting accesses safe is to establish a happens-before relationship between them. Acquiring and releasing a std::mutex does this: releasing a mutex synchronizes-with a later acquisition of the same mutex by another thread, ordering everything before the release ahead of everything after the acquire.16 Atomic operations do it explicitly through std::memory_order. The default, memory_order_seq_cst, gives the strongest guarantee — a single global order all threads agree on — at some performance cost; weaker orders such as acquire/release and relaxed trade ordering guarantees for speed and must be used with care.17

A related but distinct hazard is the race condition of the time-of-check-to-time-of-use (TOCTOU) variety, where correctness depends on timing even without a low-level data race — for instance, checking that a file exists and then opening it, when another process can swap the file in between. CWE-362 covers improper synchronization broadly; the data race is its sharpest, language-level form.

06 / ToolingDetection & defense

Because the compiler will not reject memory-unsafe code, the ecosystem has built an arsenal of tools that find these bugs at compile time, at run time, or by brute force.

Sanitizers

AddressSanitizer (ASan), introduced by a Google team in 2012, instruments a program at compile time with "shadow memory" and redzones so that out-of-bounds accesses, use-after-free, and double-frees are caught at the moment they happen, with a precise stack trace, at roughly a 2× slowdown.21 Its sibling ThreadSanitizer (TSan) detects data races at run time by tracking the happens-before relationship between accesses, flagging exactly the conflicting unsynchronized pairs the C++ memory model forbids. MemorySanitizer targets uninitialized reads, and UndefinedBehaviorSanitizer catches a grab-bag of other UB. These are enabled with a single compiler flag (e.g. -fsanitize=address) and are now standard in CI pipelines.

Dynamic analysis & fuzzing

Valgrind's Memcheck tool finds leaks and invalid accesses without recompilation, at a heavier performance cost. Fuzzing — feeding a program vast quantities of malformed input to provoke crashes — pairs naturally with sanitizers: the fuzzer produces the haystack of inputs, and ASan turns each silent corruption into a loud, diagnosable failure. Google's OSS-Fuzz has found tens of thousands of bugs in open-source C/C++ this way.

Static analysis & safer language constructs

Compiler warnings (-Wall -Wextra), the Clang Static Analyzer, and commercial tools reason about code paths without running them. Within C++ itself, the discipline that prevents whole bug classes is consistent: prefer standard containers (std::vector, std::string) over raw arrays and manual buffers; prefer smart pointers and RAII over manual new/delete; prefer std::atomic and std::mutex over hand-rolled synchronization; and prefer references and std::optional over nullable raw pointers. The C++ Core Guidelines codify these into rules a tool can enforce.

The uncomfortable lesson

Microsoft's data is sobering: despite decades of code review, training, static analysis, and fuzzing, roughly 70% of the vulnerabilities it assigned CVEs each year remained memory-safety issues.11 The conclusion drawn by Microsoft, Google, and later by the U.S. NSA and CISA was that tooling alone, layered on a memory-unsafe language, hits a ceiling — because it depends on developers never making a mistake.12

07 / OutlookThe road to memory safety

The most significant shift of the last decade is the argument that these bugs should be prevented by construction rather than caught after the fact. A memory-safe language — one whose type system and runtime guarantee that out-of-bounds access, use-after-free, and data races cannot occur in safe code — eliminates the class rather than chasing instances. Rust is the prominent systems-level example: its ownership-and-borrowing model enforces, at compile time, that a value has a single owner, that references cannot outlive what they point to, and that shared mutable access across threads is rejected outright.

The data from large codebases is striking. Google reported that the number of memory-safety vulnerabilities in Android fell from 223 in 2019 to fewer than 50 in 2024, and their share of total vulnerabilities dropped from 76% to about 24% over the same period — well below the roughly 70% industry norm — as new code was increasingly written in Rust rather than C and C++.1819 By 2025 that share fell below 20% for the first time, and Google described a roughly 1000× lower memory-safety vulnerability density in its Rust code than in its C and C++.20 In 2023, U.S. and allied cybersecurity agencies formally urged software manufacturers to publish "memory-safe roadmaps."11

None of this makes C and C++ disappear. They run the kernels, the firmware, the cryptographic libraries, and the trillion lines of existing software that the rest of the stack depends on. The realistic posture is layered: harden new C/C++ with the constructs and sanitizers above, fuzz relentlessly, isolate untrusted parsing, and write the riskiest new components — the ones that touch attacker-controlled input — in a memory-safe language where it is feasible. The four bugs in this article are not exotic. They are the predictable consequence of a design that trusts the programmer completely. Understanding exactly how that trust is betrayed is the first step toward not betraying it.
