Almost every graph problem you meet early in your training is the same problem wearing a different costume. A map of land and sea, a list of course prerequisites, a dictionary of words that differ by one letter — none of them looks like a graph, yet each one is. The whole skill at this level is learning to see the graph hiding inside the prompt, because once you can, the answer is nearly always one of two traversals with a small twist bolted on.

The seven problems below cover the moves you will reuse for years: flood fill and connected components, measuring a region, breadth-first search for shortest paths and elapsed time, multi-source and reverse traversal, deep-copying a structure as you walk it, and ordering items under dependencies.5 For each one we name the pattern it teaches, build a small worked example, then give complete, idiomatic solutions in all six languages. Use the switcher at the top of the page to read every snippet in the language you think in.

Depth-first asks “how far can I get from here?” Breadth-first asks “what can I reach in one more step?” Almost everything else is bookkeeping.

Foundations· read this first

01The toolkit

A grid is a graph in which every cell is a node and edges connect it to its orthogonal neighbours — up, down, left, right. You rarely build that graph explicitly; you walk it directly on the 2-D array, with one piece of bookkeeping: a way to remember where you have already been. The recurring trick for stepping to neighbours is a small table of (row, column) deltas.

One rule prevents most beginner bugs: mark a node seen the moment you first reach it — as you enqueue it, or on entry to a DFS call — never when you later pull it off the queue. Mark on dequeue and the same node gets enqueued many times before it is processed, which is both slow and a source of wrong answers. On a grid, the simplest marker is to overwrite the cell itself (“sink” the land, mark the orange rotten). When you are not allowed to mutate the input, carry a separate visited matrix or set.

Which traversal?

Reach for DFS when the question is about existence, counting, or the size of a region: it is the least code and recursion does the bookkeeping for you. Reach for BFS when the question is the shortest path, the minimum number of steps, or the earliest time something happens — because BFS visits nodes in order of distance, the first time it reaches a node is along a shortest route.1 Two refinements show up constantly: multi-source BFS, where many origins start in the queue at once, and topological order, for items with dependencies where you need a valid ordering or just a cycle check.

Rule of thumb

Counting or measuring regions → DFS flood fill. Shortest path, fewest steps, or elapsed time on an unweighted graph → BFS. Many things spreading at once → seed every source into the BFS queue together. Dependencies and ordering → topological sort.

Pattern · Flood fill / connected components

02Number of Islands

LeetCode 200MediumDFS or BFSTime O(R·C)Space O(R·C)

You are given a grid whose cells are land (1) or water (0). An island is a maximal group of land cells joined horizontally or vertically. Count the islands.

The idea

Each island is a connected component. Scan the grid in reading order; the first time you land on a 1 you have not seen, you have discovered a new island, so add one to the count and then flood the whole component — visiting every cell reachable through land — so none of those cells starts a second count. Overwriting each visited cell with water is the cleanest “seen” marker and costs no extra memory. The scan touches every cell, but the flood guarantees each component is entered exactly once.

The solution below uses recursive DFS. If a single island could span a grid large enough to overflow the call stack, the identical flood fill runs iteratively with an explicit stack or a queue — switching DFS for BFS here changes the traversal order but not the answer.

Variations & follow-ups

  • Max Area of Island (next section): return the largest component instead of the count.
  • Number of Distinct Islands (LC 694): hash each island’s shape — the sequence of moves taken from its origin — and count the unique shapes.
  • Surrounded Regions (LC 130) and Number of Closed Islands (LC 1254): flood inward from the border first to spare anything touching an edge, then work on what remains.
  • Making a Large Island (LC 827): give each island an id and a size, then try every water cell as a one-cell bridge between neighbouring islands.
  • Union-Find alternative: union adjacent land cells and count distinct roots. This shines when land is added incrementally — Number of Islands II (LC 305).

Pattern · Region measurement (a DFS that returns a value)

03Max Area of Island

LeetCode 695MediumDFSTime O(R·C)Space O(R·C)

Same land/water grid. Instead of counting islands, return the number of cells in the largest one — or 0 if there is no land at all.

The idea

The structural change from the previous problem is tiny: the flood fill now returns a number. A land cell contributes 1 for itself plus whatever its four recursive calls report; an out-of-bounds, water, or already-seen cell contributes 0. Run the fill from every unseen land cell and keep the maximum. This “return a value from DFS” shape is the workhorse behind region sizes, perimeters, sums, and depths — learn it once and a whole family of problems opens up.

Variations & follow-ups

  • Island Perimeter (LC 463): count the cell edges that touch water or the grid boundary — often a simple per-cell formula, no DFS required.
  • Area thresholds and sums: count islands with area ≥ k, or total all areas — same fill, different aggregation.
  • Diagonal adjacency: extend the direction table to eight neighbours when islands connect through corners.
  • Id-labelling: store each region’s size keyed by an id so later queries (bridges, merges) are O(1) lookups.

Pattern · Multi-source BFS over time

04Rotting Oranges

LeetCode 994MediumMulti-source BFSTime O(R·C)Space O(R·C)

A grid of empty cells (0), fresh oranges (1), and rotten oranges (2). Each minute, every fresh orange orthogonally adjacent to a rotten one also rots. Return the number of minutes until no fresh orange remains, or −1 if some orange can never rot.

The idea

Elapsed time is simply BFS depth. Seed the queue with every orange that is already rotten — they all act at minute 0 — then expand one full layer at a time, counting each layer as one minute. Track the number of fresh oranges as you go; if any are still fresh once the queue empties, they were unreachable and the answer is −1. The mechanism that turns BFS into a clock is processing the queue layer by layer: capture the current queue size, drain exactly that many cells, then tick the minute.

Two edges that bite

Return 0 when there were no fresh oranges to start with — nothing has to happen, so the time is zero, not the number of layers. And increment the minute counter per layer, never per cell, or a single busy minute inflates into many.

Variations & follow-ups

  • Walls and Gates (LC 286) and 01 Matrix (LC 542): seed every gate (or every zero) at once and BFS outward, filling each cell with its distance to the nearest source — the same multi-source engine.
  • Shortest Bridge (LC 934): flood-fill one island, then run multi-source BFS from its entire border until it touches the second island.
  • As Far from Land as Possible (LC 1162): seed all land cells and BFS over water; the last distance reached is the answer.

Pattern · Reverse / multi-source traversal & intersection

05Pacific Atlantic Water Flow

LeetCode 417MediumDFS or BFSTime O(R·C)Space O(R·C)

A matrix of heights. Water flows from a cell to an orthogonal neighbour whose height is less than or equal to it. The Pacific borders the top and left edges; the Atlantic borders the bottom and right. Return every cell from which water can reach both oceans.

The idea

Taken literally — from each cell, can I flow downhill to each ocean? — you re-walk the grid from thousands of starting points. Flip the question: standing at the shoreline, which cells could the ocean reach if water ran uphill? Flood inward from every Pacific border cell, then again from every Atlantic border cell, this time stepping only to neighbours that are equal or higher. A cell drains to both oceans precisely when both floods have marked it — so the answer is the intersection of the two reachable sets. Reversing the edge rule and starting from the goal is a move worth keeping: whenever “from every source to one target” is expensive but “from one target outward” is cheap, turn the search around.

Mind the inequality

The comparison direction is the classic slip. Forward flow allows lower-or-equal, so the reverse flood must move to higher-or-equal. Keep two separate visited matrices — one per ocean — and intersect them at the end.

Variations & follow-ups

  • Trapping Rain Water II (LC 407): a priority-queue BFS inward from the border, always processing the lowest surrounding wall first.
  • The same reverse-flood reasoning underlies 01 Matrix and Walls and Gates from the previous section — compute distances from the targets, not to them.

Pattern · Traversal that builds a parallel structure

06Clone Graph

LeetCode 133MediumDFS or BFSTime O(V+E)Space O(V)

You are handed a reference to one node of a connected, undirected graph. Return a deep copy: a brand-new graph with the same shape and the same values, sharing no nodes with the original.

The idea

A single hash map from each original node to its clone does two jobs at once — it is your visited set and your wiring table. The first time you reach a node, create its clone and register the pair immediately, before recursing. That ordering is the whole trick: when a neighbour’s edges loop back to a node you are still copying, you return the clone already in the map instead of building a second one and recursing forever. Once the clone exists, copy each neighbour and attach it. DFS and BFS both work; the map is what makes it correct on cyclic graphs.

On the Rust version

Because the problem guarantees node values are unique (1–100), the Rust solution keys the map by value, which sidesteps the awkwardness of hashing Rc<RefCell<Node>> identities. In a graph with repeated values you would instead key by the allocation’s address (e.g. Rc::as_ptr).

Variations & follow-ups

  • Copy List with Random Pointer (LC 138): the same old→new map on a linked list whose nodes carry one arbitrary extra pointer.
  • N-ary trees and labelled / weighted graphs: copy the payload alongside the structure; the traversal is unchanged.
  • The cardinal pitfall: recursing before registering the clone — any cycle then sends the copy into infinite recursion.

Pattern · Topological sort / cycle detection (directed graph)

07Course Schedule

LeetCode 207MediumTopological sortTime O(V+E)Space O(V+E)

There are numCourses courses labelled 0 … n−1 and a list of prerequisite pairs [a, b] meaning course b must be taken before course a. Decide whether it is possible to finish every course.

The idea

Model each course as a node and each prerequisite as a directed edge from the prerequisite to the course that needs it. A complete schedule exists exactly when this graph is a DAG — when it contains no cycle, because a cycle is a set of courses that each wait on one another forever. Two equivalent checks. Kahn’s algorithm (shown): repeatedly take any course with no remaining prerequisites (in-degree 0), “finish” it, and decrement each dependent’s count; if you manage to finish all n, the graph was acyclic, and if you stall before then, a cycle blocked you.2 The DFS alternative colours nodes white / grey / black and declares a cycle the instant it follows an edge back to a grey (still-in-progress) node.3

Get the direction right

The edge points from the prerequisite to the dependent course, and in-degree 0 means “ready to take now.” Reverse the edges by accident and you compute the wrong order — or miss the cycle entirely.

Variations & follow-ups

  • Course Schedule II (LC 210): the same Kahn’s loop, but append each finished course to a list to return an actual valid order.
  • Parallel Courses (LC 1136): each Kahn layer is one semester — the number of layers is the minimum time to graduate.
  • Alien Dictionary (LC 269): derive letter-ordering edges by comparing adjacent words, then topologically sort the alphabet.
  • Course Schedule IV (LC 1462): reachability queries — precompute the transitive closure.
  • Minimum Height Trees (LC 310): peel leaves layer by layer, a topological cousin on an undirected tree.

Pattern · Shortest path on an implicit graph (BFS)

08Word Ladder

LeetCode 127HardBFSTime O(N·L²)Space O(N·L)

Given a beginWord, an endWord, and a dictionary of equal-length words, change one letter at a time so that every intermediate word is in the dictionary, turning the begin word into the end word. Return the length of the shortest such chain (the number of words, counting both ends), or 0 if no chain exists.

The idea

Words are nodes; an edge joins two words that differ in exactly one letter. The shortest chain is then a shortest path on an unweighted graph — which is BFS. You never materialise the edge list: to find a word’s neighbours, try all 26 letters at each position and keep the candidates that appear in the dictionary. Remove each word from the dictionary as you enqueue it so it is visited once. The BFS layer counter, started at 1 for beginWord, is the answer.1

Making it faster

Two standard upgrades, in rough order of payoff:

  • Wildcard buckets. Precompute patterns such as h*t → {hot, hit, hat}. A word’s neighbours are then the union of its L pattern buckets, found in O(L) instead of O(26·L) — a real win when the alphabet or word length grows.
  • Bidirectional BFS. Search forward from beginWord and backward from endWord simultaneously, always expanding whichever frontier is smaller; the two searches meet in the middle, cutting the breadth explored roughly from bd to 2·bd/2.6
Count words, not steps

The result is the number of words in the chain — i.e. the number of BFS layers — not the number of letter changes. A chain of N words has length N, not N−1. Off-by-one here is the most common wrong answer.

Variations & follow-ups

  • Word Ladder II (LC 126): return every shortest transformation — BFS to build the layered graph, then DFS / backtracking to reconstruct all paths.
  • Minimum Genetic Mutation (LC 433): the identical pattern over the 4-letter alphabet ACGT.
  • Open the Lock (LC 752): BFS over four-wheel lock states, skipping a set of forbidden “deadends.”

09Pattern recognition

Step back and the seven problems line up as a small set of reusable moves. The hard part in a contest is rarely the code — it is recognising, from an unfamiliar prompt, which graph is hiding inside and which traversal it wants. Habits · the things that actually break

11Pitfalls & habits

Across all seven problems the same handful of mistakes account for most wrong answers and time-limit failures. Build these into reflexes:

  • Mark on first sight. Set a node seen the moment you enqueue it (BFS) or enter its call (DFS), never when you later dequeue it — otherwise the same node is added many times before it is processed.
  • Mutate or remember. Sinking a grid cell is free bookkeeping; when the input must stay intact, keep an explicit visited matrix or set instead.
  • Freeze the layer. For “minimum steps” or “elapsed time,” snapshot the queue size and drain exactly that many before ticking the counter.
  • Register clones before recursing. On any cyclic structure this single ordering is all that stands between you and infinite recursion.
  • Reverse the rule when you reverse the search. Pacific–Atlantic flips “flow to lower-or-equal” into “climb to higher-or-equal”; the same care applies to any search run from the goal.
  • Mind edge direction in DAGs. Prerequisite → course, and in-degree 0 means ready. Flip it and you solve a different problem.
  • Count the right thing. Word Ladder counts words (layers), not transitions; Rotting Oranges counts minutes (layers), not cells.
  • Watch recursion depth. A worst-case grid is one long snake; deep recursive DFS can overflow the stack, so prefer an explicit stack or queue on very large inputs.
Learn the toolkit, not the answers. Seven solutions you can rebuild from the pattern beat seventy you can only recite.
