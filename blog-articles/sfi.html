Untrusted Code, Trusted Process

The Sandbox in the Same Address Space

Software Fault Isolation turns a hostile binary into a well-behaved tenant — not by walling it off with hardware, but by rewriting its arithmetic until escape becomes impossible. A field guide to address masking, control-flow integrity, Native Client, WebAssembly, and the bypasses that keep the discipline honest.

An Engineering Deep-Dive Topic / SFI Level / Advanced 27 References

The cheapest way to contain a program you do not trust is to give it its own address space. Spin up a process, let the memory-management unit and the kernel's privilege rings do the work, and a fault in the guest cannot reach across the page tables into the host. This is the textbook isolation boundary, and for coarse-grained, loosely-coupled components it remains the right answer. The problem is that the boundary is expensive precisely where you most want it to be cheap: between two pieces of code that need to talk to each other constantly. Every cross-domain call becomes a system call or an IPC round-trip, with a trap into the kernel, a context switch, TLB pressure, and cache pollution on each side of the transition. When Wahbe, Lucco, Anderson, and Graham introduced the technique that would come to be called Software Fault Isolation, their opening observation was exactly this: placing each module in a separate address space gives fault isolation, but for tightly-coupled modules the context-switch overhead is prohibitive.1

The motivating scenario in 1993 was extensibility. Systems researchers wanted to load third-party modules — a database access method, a packet filter, a rendering plug-in — directly into the address space of a host application or operating system, where they could call host services and be called back at the speed of a direct branch. The danger is obvious: a buggy or malicious extension sharing the host's address space can read any secret, scribble over any data structure, and divert control anywhere it likes. The question SFI answered was whether you could keep the in-process performance while recovering the isolation guarantee that the process boundary normally provides — and do it in software, on stock hardware, without trusting the module's source language, its compiler, or even its good intentions.

That question never went away; it only changed clothes. The same structural tension — untrusted code that must run in-process, fast, with a narrow and policed interface to everything around it — reappears in the browser running a downloaded native module, in the edge-compute platform multiplexing thousands of customer functions inside one server process, in the application that links an untrusted media or font parser written in memory-unsafe C, and in the serverless runtime that wants microsecond cold starts. Each of these is a setting where a fresh process per tenant is too slow, too heavy, or too coarse, and where the only economical place to draw the trust boundary is inside a single virtual address space. SFI is the family of techniques that makes such a boundary enforceable.

What an in-process boundary has to defend

To appreciate the design space it helps to be precise about the adversary. The untrusted module is delivered as machine code (or as bytecode that the host will compile to machine code). We assume it may be actively malicious and may have been crafted with full knowledge of the host and the isolation mechanism. Once it executes, it shares the host's hardware-defined address space, register file, and — absent countermeasures — the host's full authority. A successful escape is any of: reading host memory outside the region the module was granted; writing host memory outside its writable region; or transferring control to any instruction the policy did not sanction, including into the middle of a host instruction or into the module's own code at an unintended offset. The last category matters because on variable-length instruction sets the byte stream is not self-delimiting: a jump to the wrong offset can decode an entirely different, attacker-chosen instruction sequence out of bytes the compiler intended as something else.

The classical reference-monitor formulation gives us the language to reason about this. A reference monitor mediates every access a subject makes to an object, and to be trustworthy it must be tamper-proof, always invoked, and small enough to verify — the properties James Anderson laid out in 1972.2 Schneider later formalized exactly which security policies are enforceable by a monitor that runs alongside the target and can halt it: the safety properties, those expressible as a prohibition on some bad finite prefix of execution, recognizable by a security automaton.3 SFI is best understood as a way to compile such a monitor into the untrusted code itself, so that the checks are inlined ahead of every dangerous operation and there is no separate, bypassable guard to trap into. That insight — the monitor as inlined instrumentation rather than an external sentry — is what later work named the inline reference monitor, and it is the connective tissue running through everything that follows.

Why not just use the MMU?

Page-table protection is real isolation, and SFI does not replace it — production systems layer the two. The MMU's limitation is granularity and transition cost, not strength. It protects at page boundaries with a handful of permission bits, switches domains only via the kernel, and cannot express “this thread may call that function but may not read this adjacent object.” SFI buys sub-page, intra-process, language-agnostic mediation at the cost of trusting a rewriter/verifier and paying a per-operation instruction tax. The interesting engineering is in making that tax small while keeping the verifier small enough to trust.

SFI does not ask the untrusted code to behave. It rewrites the code so that misbehavior is not expressible in the instruction stream the CPU will accept.

§ 02 — Fundamentals

Software-Based Fault Isolation, from first principles

The original construction is elegant enough to state in a paragraph. Partition the host address space and reserve a contiguous, power-of-two-aligned region for the untrusted module — Wahbe et al. called it a fault domain.1 Choose the region so that every legal address inside it shares a common high-order bit pattern, the segment identifier. The module's code and its data each get such a region. Now transform the module's machine code so that every memory store, every memory load you wish to confine, and every computed (indirect) control transfer is preceded by instructions that force the target address into the correct region before it is used. Because the confinement is established immediately before each unsafe instruction, the module can compute any address it likes — the instrumentation guarantees the address that actually reaches the bus or the program counter lies inside the sandbox. The transformation is applied by a trusted tool; a separate, very small verifier then checks the finished binary to confirm the invariants hold, so that the much larger rewriter need not itself be trusted.

Sandboxing versus checking

There are two ways to enforce the address invariant, and the distinction recurs in every later system. The first is address checking: before the unsafe access, compare the target against the region bounds and branch to a fault handler if it is out of range. This catches the violation and can report precisely which instruction misbehaved, but it adds a compare-and-conditional-branch — and branches are exactly what you do not want sprinkled through a hot loop. The second is address sandboxing (also called address masking or forcing): instead of checking whether the address is legal, unconditionally coerce it to be legal, typically by clearing the high bits and setting the segment identifier with a couple of bitwise operations. Sandboxing is branchless and therefore friendlier to the pipeline, but it sacrifices the ability to pinpoint the offending access — an illegal pointer is silently redirected somewhere harmless inside the sandbox rather than reported. Wahbe et al. chose sandboxing for performance and accepted the loss of diagnosis; most production SFI since has done the same.1

Address sandboxing — the canonical idiom (pseudo-x86); goal: confine the address in %rax to the data region whose
; segment id is in the high bits encoded by DATA_MASK / SEG_ID.
; rewriter emits this immediately before the real store.

    and   %rax, $DATA_MASK    ; clear the high "segment" bits → offset within region
    or    %rax, $SEG_ID       ; stamp in the sandbox's segment identifier
    mov   (%rax), %rbx        ; now provably in-region; the store cannot escape

A naive reading suggests the masking sequence is two extra instructions per memory operation, which would be ruinous. The 1993 paper's central contribution was showing it need not be. Three optimizations carried the day. First, a dedicated register holds the segment identifier so the or is a register operation rather than an immediate load, and the rewriter reserves a small number of registers (five, in the original RISC implementation) as dedicated — guaranteed by the verifier never to be used as the base of an unsandboxed access. Second, the mask-then-stamp can frequently be folded: if the segment shift is a constant, a single masking step can both clear out-of-region bits and rely on the region's base being a known constant. Third, and most importantly, the cost is paid only at the point of an unsafe operation, and the compiler's existing optimizations (hoisting an invariant address computation out of a loop, for instance) hoist the sandboxing with it. The measured result was an average runtime overhead in the low single digits to low double digits depending on the workload — small enough that the in-process model was clearly worth it for tightly-coupled extensions.1

The crucial subtlety on CISC: where does an instruction begin?

On a fixed-width RISC instruction set, control flow is comparatively tame: every instruction starts on a four-byte boundary, so a verifier can decode the program unambiguously and an indirect jump can be masked to land on a valid instruction boundary by clearing its low bits. On a variable-length CISC architecture such as x86, the same byte stream decodes to different instructions depending on where you start reading. An attacker who can direct a jump into the middle of a legitimate instruction can synthesize instructions — including the privileged or unsafe ones the rewriter carefully avoided emitting — out of the immediate operands and bytes of innocent code. This is the same primitive that return-oriented programming exploits to assemble malicious computation from borrowed instruction fragments, or gadgets.4 Confining data accesses is therefore not enough on x86; you must also confine control flow to a set of instruction boundaries you actually intended.

McCamant and Morrisett solved this for x86 in their PittSFIeld system with a disarmingly simple idea: artificially impose alignment that the hardware does not. Divide the code region into fixed-size chunks — sixteen bytes in their implementation — and have the rewriter insert no-op padding so that no instruction ever straddles a chunk boundary, every jump target is moved to the start of a chunk, and every call is placed at the end of a chunk so the return address it pushes is itself chunk-aligned. The control-flow sandboxing mask then clears the low four bits of any computed jump target, which forces it to a chunk start — and because each chunk is, by construction, an atomic unit you cannot enter except at its first instruction, you can never land mid-instruction.5 Their measured overheads were modest, and just as significant, they verified the safety of the scheme with a machine-checked proof in ACL2, establishing a methodological standard — a small trusted verifier whose correctness is mechanically proved — that later systems would inherit.5

The architecture of a complete SFI system

Strip away implementation detail and every SFI system is the same three-part machine. A transformer (in the compiler back end or as a binary rewriter) emits code that already satisfies the sandboxing invariants. A verifier — the only large piece that must be trusted, and deliberately kept tiny — independently re-checks the delivered binary against the invariants without trusting how it was produced. And a runtime / trusted runtime mediates the narrow set of operations the sandbox is allowed to perform on the outside world: entering and leaving the sandbox, and invoking host services through a controlled gate. The separation of transformer from verifier is the linchpin of the trust argument. You can use an aggressive, bug-prone optimizing compiler to generate the module, because nothing the compiler does is believed; the small verifier has the final word, and if the binary does not provably stay inside the lines, it is rejected before it ever runs.

§ 03 — Memory

Address masking and guard zones

The memory side of SFI reduces to one obligation: no load or store performed by the untrusted module may touch memory outside the regions it was granted. There are two broad strategies for discharging it, and the cleverest implementations combine them.

Bit-masking and the one-instruction store

If the sandbox region is 2^k bytes, aligned to a 2^k boundary, then an address is in-region exactly when its high bits equal the region's base. Masking enforces this in one or two ALU operations, as shown earlier. The refinement that makes it nearly free is choosing the region geometry to collapse the mask into a single instruction. If the architecture offers segmentation, you can do better still: load the segment register once with the sandbox base and limit, and let the hardware add the base and bounds-check every access for free. This is precisely why the first version of Native Client on 32-bit x86 was so fast — it leaned on the legacy x86 segmentation hardware to confine data accesses with essentially zero per-access software cost, reserving the software masking only for control flow.11 The catch is that segmentation in this form does not exist on x86-64 (the segment limits were largely dropped) or on ARM, which forced the field back onto pure software masking and motivated the next idea.

Guard zones: trading address space for instructions

A guard zone (or guard page) is a band of unmapped, inaccessible virtual address space placed immediately around the sandbox region. Its purpose is to let you weaken the masking obligation and recover speed. Consider an access of the form base + index where index is a small, bounded displacement — the common case for structure-field and array accesses with a compile-time-bounded offset. If you have masked base to lie within the region, then base + index can stray at most index bytes past the region's end. If the guard zone is at least as large as the maximum possible displacement, any such stray access lands in unmapped memory and faults rather than escaping — so you do not need to re-mask the full effective address, only the base. On 64-bit machines, where virtual address space is effectively free, you can be lavish: reserve a multi-gigabyte region with enormous guard bands on either side, mask a pointer to a fixed number of low bits, and let any 32-bit displacement be absorbed harmlessly by the guard. This is the structural trick behind both 64-bit Native Client and the WebAssembly linear-memory model discussed below.

Read / write asymmetry

Many systems sandbox writes aggressively and treat reads more permissively, because an out-of-bounds write that corrupts host state is catastrophic, while an out-of-bounds read is “merely” an information leak. Confining only writes (and control flow) is sometimes enough for an integrity-only threat model and roughly halves the instrumentation. Whether that is acceptable depends entirely on whether confidentiality of host memory is in scope — and, as the bypasses section shows, transient-execution attacks have made “merely a read” far less comfortable than it once sounded.

Data sandboxing as a verified policy

The masking discipline is only sound if the verifier can be convinced that every dangerous memory operation is preceded by a correct mask, with no path that reaches the operation while skipping the mask. This is harder than it looks because the mask and the access can be separated by other instructions and by control flow. Zeng, Tan, and Morrisett showed how to make the verification tractable and trustworthy by combining a control-flow-integrity substrate with a static data-flow analysis: CFI first pins down the set of possible execution paths, and a lightweight range analysis over that constrained graph then proves that the masked value reaching each access has not been clobbered in between, allowing redundant masks to be elided while preserving a checkable safety argument.12 The lesson is structural and worth stating plainly: memory sandboxing depends on control-flow integrity. If the adversary can jump to the access while bypassing its guard, no amount of masking helps. That dependency is why CFI is not a separate topic bolted on beside SFI but a load-bearing primitive underneath it.

§ 04 — Control Flow

Control-Flow Integrity as an SFI primitive

Control-Flow Integrity, introduced by Abadi, Budiu, Erlingsson, and Ligatti at CCS 2005, states a property and an enforcement mechanism in one breath. The property: program execution must follow a path through a control-flow graph (CFG) determined ahead of time, so that every indirect branch — every computed call, every indirect jump, every function return — transfers control only to a target permitted by the graph.7 The mechanism: assign each valid target a small label (an identifying bit pattern embedded at the destination), and instrument each indirect transfer to check, immediately before it jumps, that the destination carries the expected label, halting otherwise.7 Direct branches need no check because their targets are fixed in the code and protected by the same integrity argument that protects the code region itself. The authors gave the property a formal treatment and a companion paper establishing its guarantees against a powerful adversary who can write arbitrary data memory at any time — exactly the adversary SFI must survive.8

Label-based CFI check on an indirect call (schematic); at the call site: ensure the target is a sanctioned entry point
    mov   (%rax), %r11        ; fetch the bytes at the call target
    cmp   %r11, $LABEL_F      ; does it carry function F's CFI label?
    jne   cfi_violation       ; mismatch → halt; never branch into the void
    call  %rax                ; only now is the indirect transfer permitted

Why SFI needs CFI, and why CFI alone is not SFI

The relationship runs in one direction with a twist. SFI requires a form of CFI: the entire data-sandboxing argument collapses if control can reach a memory operation while skipping its mask, or land mid-instruction to decode an unmasked store. So every SFI system embeds at least a tailored control-flow discipline — Native Client's bundle alignment and PittSFIeld's chunk masking are CFI mechanisms specialized to the sandbox's needs. The twist is that general CFI, as a defensive retrofit for trusted-but-buggy programs, aims at a different goal than SFI's. CFI's classic use case is hardening a program you wrote and compiled yourself against an attacker who has already achieved a memory-corruption foothold and wants to hijack control. SFI's use case is confining a program you did not write and do not trust at all. The threat models overlap but are not identical, and that difference shows up in how precise the CFG needs to be.

Forward edges, backward edges, and the precision problem

Indirect transfers split into forward edges (indirect calls and jumps, dispatched through function pointers and vtables) and backward edges (returns, which pop a target off the stack). The two demand different defenses. Forward-edge protection has been productized: the compilers shipped a coarse but cheap form in which each indirect call is checked against a class of permitted targets. Tice and colleagues described the forward-edge CFI implementations added to GCC and LLVM, including virtual-table verification and indirect-function-call checks, with overheads low enough for production use.9 Backward edges are harder: the precise defense is a shadow stack, a protected, separate copy of return addresses against which each return is validated, which is exactly the “protected shadow call stack” the original CFI paper proposed building on top of the CFI substrate.8

The precision problem is the crux of CFI's security and the seam most bypasses pry at. The static CFG is necessarily an over-approximation: pointer analysis cannot resolve every indirect call to a single target, so many call sites are permitted to reach an entire equivalence class of targets sharing one label. The coarser the labelling — the larger those classes — the more freedom an attacker retains while still satisfying the check. A single shared label for “any function that is ever called indirectly” is cheap but admits enormous latitude. We return to what that latitude buys an attacker in the bypasses section; for now the point is that CFI's strength is a function of CFG precision, and precision costs analysis effort and runtime checks. XFI, from Erlingsson and colleagues, is the system that most directly fused this CFI machinery with full SFI memory confinement and a host-call interface, packaging the whole inline-reference-monitor stack — control-flow integrity, a protected stack, and data-access guards — into a single binary-level mechanism for sandboxing system extensions.10

CFI is the floor under memory sandboxing. Take away the guarantee that execution follows a known graph, and every other invariant in the system becomes a suggestion.

§ 05 — Native Client

Google Native Client: SFI shipped to a billion browsers

Native Client (NaCl) is the system that took SFI from a systems-research technique to production infrastructure. Yee and colleagues described it in 2009 as a sandbox for portable, untrusted x86 native code, built to give browser applications the computational performance of native code without compromising safety, by combining software fault isolation with a secure runtime that funnels all system interaction and side effects through interfaces NaCl controls.11 The goal was to let a web page ship a compiled C or C++ module — a physics engine, a codec, a game — and run it at near-native speed inside the browser's process while guaranteeing it could not touch anything outside its sandbox or reach the operating system except through a vetted gate.

Two sandboxes, nested

NaCl's architecture is a sandbox within a sandbox. The inner sandbox is classic SFI: the untrusted module is compiled and rewritten so that its memory accesses and control flow are confined, and a static validator checks the binary before execution to confirm the confinement. The 32-bit x86 implementation took the elegant shortcut of using the legacy x86 segmentation hardware to bound data accesses, so that data confinement cost essentially nothing per access; control flow was confined in software by requiring that the code be laid out in 32-byte bundles, that no instruction cross a bundle boundary, and that every indirect branch be masked to a bundle start — the same chunk-alignment idea PittSFIeld pioneered, with NaCl supplying a careful, auditable validator.11 The outer sandbox is a second, independent line of defense at the system-call boundary: even if the inner sandbox were defeated, the trusted service runtime constrains what the module can ask the operating system to do, so a single failure does not hand over the host.11

Entering and leaving the sandbox: trampolines and springboards

Because the sandboxed code is forbidden from executing the segment-manipulation and far-call instructions that a domain transition requires, NaCl provides trusted control-transfer stubs that are allowed to contain those instructions. A trampoline carries control out of the sandbox into the trusted runtime to request a service; a springboard carries control back in. These stubs are patched into known, aligned addresses during module loading and are the only sanctioned doorways through the wall, so the set of ways control can cross the boundary is finite and audited rather than open-ended.11 This is the inline-reference-monitor pattern made concrete: the “mediation of every cross-domain access” is compiled into a fixed gateway the untrusted code cannot bypass.

A verification milestone

The NaCl x86 validator is small but security-critical, and getting its instruction decoder exactly right on x86 is notoriously error-prone. Morrisett, Tan, Tassarotti, Tristan, and Gan built RockSalt, a checker that is smaller and faster than NaCl's original validator and carries a much smaller trusted computing base, because the bulk of its analysis is generated automatically from a declarative description tied to a formal model of a subset of the x86 instruction set — and machine-checked in Coq.13 RockSalt is a landmark: a complete, executable, formally proved-correct SFI checker for a real instruction set.

Portability: from x86-64 and ARM to PNaCl

The 32-bit segmentation trick did not survive the move to x86-64 and ARM, neither of which offers the same cheap hardware bounding. Sehr and colleagues re-engineered SFI for these architectures using pure software masking with guard regions, and reported the result as the best-known SFI implementations for those targets at the time, with average performance overhead under 5% on ARM and around 7% on x86-64 — low enough that the in-process sandbox remained eminently practical on contemporary hardware.6 Other user-level x86 sandboxes of the era, such as Ford and Cox's Vx32, reached similar conclusions about the viability of confining native code without kernel support.27 They also observed that the instrumentation tends to overlap well with instruction-level parallelism and has particularly small impact on data-memory-bound workloads, properties that flatter the technique on modern out-of-order cores.6

Two further problems had to be solved for the web. First, native code is not portable across instruction sets, which is anathema to the web's “write once, run anywhere” ideal; Google's answer was Portable Native Client (PNaCl), which shipped a target-independent LLVM-bitcode-based format that the client translated to NaCl for the local architecture. Second, just-in-time compilers and self-modifying code generate executable instructions at runtime, which the static validator never saw; Ansel and colleagues extended the model with mechanisms to sandbox dynamically generated code and JIT output without abandoning the validation guarantee.14 NaCl and PNaCl ultimately shipped in Chrome and ran real applications at scale. They were, however, browser-specific and architecturally heavyweight, and when a cross-vendor standard emerged that captured the same safety properties in a cleaner package, Google announced it would deprecate PNaCl in favor of WebAssembly.15 NaCl's intellectual legacy is enormous: it proved that compiled, untrusted native code could be run safely in-process at scale, and nearly every idea in it — nested sandboxes, a small audited validator, masked control flow, a fixed host-call gateway — reappears, refined, in what came next.

§ 06 — WebAssembly

The WebAssembly sandbox model

WebAssembly (Wasm) is what you get when the SFI agenda is pursued not by rewriting and validating arbitrary machine code, but by designing a new low-level bytecode whose very structure makes the dangerous operations inexpressible. Haas and colleagues described it in 2017 as a portable low-level bytecode, collaboratively designed by engineers from the major browser vendors, offering compact representation, fast and simple validation and compilation, and safe execution with low to no overhead.16 Where NaCl confines code after the fact, Wasm achieves confinement by construction — and crucially, it is the first mainstream language designed with a formal semantics from the outset, so its safety claims can be stated and reasoned about precisely.16 A revised account later appeared in Communications of the ACM,17 and the W3C standardized the core specification, giving the model a single normative definition across implementations.18

How the structure does the work

Several deliberate design choices, taken together, are what let a Wasm engine run untrusted code safely with little of SFI's explicit per-access instrumentation:

  • Separation of code and data, by representation. Wasm code is not bytes in the same linear memory the program reads and writes; it lives in a distinct space the program cannot address as data, cannot read, and cannot overwrite. This makes code injection and self-modification structurally impossible at the bytecode level — the single most consequential difference from native machine code.19
  • Linear memory as a single bounded sandbox. A module's heap is one contiguous, zero-based linear memory — a flat byte array. Every memory access is an offset into this array, and the engine guarantees the access stays within bounds, either by an explicit bounds check or, on 64-bit hosts, by the guard-zone trick: reserve a large virtual region with guard bands so that a 32-bit offset cannot reach host memory and an out-of-bounds access faults harmlessly. This is SFI memory sandboxing, expressed as a property of the memory model rather than as masking instructions the programmer wrote.
  • Structured control flow. Wasm has no arbitrary goto. Control flow is built from structured blocks, loops, and if constructs with statically determined, properly nested branch targets; branches can only target enclosing constructs. There is no way to jump to an arbitrary byte offset, so the “land mid-instruction” problem simply does not arise.
  • Typed, table-mediated indirect calls. Indirect calls do not take a raw address. They take an index into a function table, and the call is checked at runtime against the callee's declared type signature. This is CFI promoted into the type system: the set of indirect targets is the table, and a type mismatch traps. An attacker who corrupts a function-pointer-equivalent can at worst select a different, type-compatible table entry — not jump to arbitrary code.19
  • A protected, non-addressable call stack and typed locals. Return addresses and the operand stack are managed by the engine and are not part of the linear memory the program can corrupt. Local variables and the value stack are typed and validated. Classic return-address smashing has nothing to smash.
  • Linear-time validation. Before a module runs, the engine type-checks the bytecode in a single pass: every instruction's operand types are verified, every branch target is checked to be in scope, every function's signature is confirmed. Validation is the inline-reference-monitor's “verifier” role, made cheap enough to do on every module load.16

The capability boundary follows the same philosophy. A Wasm module has no ambient authority — it cannot make system calls, open files, or touch the network on its own. It can only call functions the host explicitly imports into it. On the web that host is the JavaScript engine and the browser APIs; outside the browser, the WebAssembly System Interface (WASI) defines a capability-style set of host functions. Either way, the “host-call gateway” that NaCl built out of trampolines is, in Wasm, just the import table: the finite, declared set of functions through which the sandbox can affect the outside world.

NaCl asked, “how do we confine this machine code?” WebAssembly asked, “what would a machine code look like that needs no confining?”

Beyond the browser

Because the safety properties are intrinsic to the bytecode rather than to a browser, Wasm became the default substrate for in-process isolation far from its origin. Edge and serverless platforms run many mutually distrusting customer functions inside a single server process, isolated as Wasm instances, to get near-zero cold-start times that a process-per-tenant model cannot match. And in a striking inversion of the original SFI motivation — untrusted plug-ins inside a trusted host — Mozilla used Wasm to sandbox trusted-but-buggy third-party C libraries inside Firefox: a font shaper or media decoder is compiled to Wasm and then back to native code, so that a memory-safety bug in the library is contained to its linear memory and cannot corrupt the browser. The RLBox framework by Narayan and colleagues made this retrofit practical by using the type system to force the host code to validate every value crossing the sandbox boundary, catching the integration mistakes that otherwise undermine such isolation.20

§ 07 — Reference Monitors

Inline reference monitors and policy enforcement

Step back from the individual systems and SFI is a special case of a more general idea: the inline reference monitor (IRM). A reference monitor, in Anderson's original sense, is a mechanism that mediates every access and can be trusted because it is always invoked, tamper-proof, and verifiable.2 The classical realization is external — a kernel that traps every system call. An inline reference monitor instead merges the monitor into the target program itself, rewriting the code so that the policy checks execute inline, ahead of each regulated action, with no separate guard to bypass and no trap to pay. SFI is the IRM whose policy is precisely “stay inside your memory regions and follow your control-flow graph.”

What policies can an IRM enforce?

The theory here is Schneider's. The policies enforceable by a monitor that observes execution and can halt the program are exactly the safety properties — those that forbid some bad thing from ever happening, characterizable as a security automaton that reads the execution as a sequence of events and rejects on the first prohibited step.3 Liveness properties (“something good eventually happens”) and information-flow properties that quantify over multiple possible executions fall outside this class and cannot, in general, be enforced by inlined runtime checks alone. This is not an academic footnote: it tells you in advance which guarantees SFI can deliver (no out-of-bounds access ever occurs; no disallowed control transfer ever occurs — both safety properties) and which it cannot (e.g., “this module never leaks a secret through a timing side channel,” a non-safety property no inlined check can catch).

Erlingsson and Schneider's SASI work demonstrated the IRM approach concretely by merging a security automaton, specified as a policy, directly into the instruction stream of the target, and reflecting on what that buys and costs.3 The crucial move that makes inlining sound is the same one that makes SFI sound: the rewriter is not trusted, the verifier is. After the policy automaton is woven in, a checker confirms that the result genuinely mediates every regulated event — that there is no path to a dangerous operation that evades the inlined check — and only then is the program permitted to run.

The IRM design contract

An inline reference monitor is trustworthy only if three obligations are met, and every SFI system is an exercise in meeting them. Complete mediation: no path reaches a regulated operation without passing its inlined check — this is what control-flow integrity guarantees, and why CFI is foundational. Tamper-proofness: the untrusted code cannot rewrite the checks or the policy state at runtime — which requires the code region to be immutable and the monitor's data (shadow stacks, dedicated registers) to be out of the sandbox's reach. Verifiability: a small, trusted checker can confirm the first two hold. Break any one and the monitor is theater.

Generalizing the policy

Once you see SFI as an IRM, richer policies become natural. XFI is the clearest example: it layers a configurable IRM atop the SFI substrate, enforcing not just memory and control-flow confinement but additional integrity constraints — a protected stack, restrictions on which host functions may be called and with what arguments, and fine-grained access rules on specific memory regions — all compiled inline and independently verified.10 The general pattern is: express the policy as a safety automaton over the events you can observe at the instruction level (memory accesses, control transfers, host calls), inline the automaton's checks, and verify complete mediation. SFI's memory-and-control-flow policy is the indispensable base layer because it is what guarantees the higher-level checks cannot themselves be bypassed.

§ 08 — Limits

Limitations and known bypasses

A discipline this old has a well-mapped failure surface. The bypasses fall into a few families, and understanding them is the difference between deploying SFI as real isolation and deploying it as a comforting story.

1 · CFI is only as strong as its graph

Coarse-grained control-flow integrity — the kind cheap enough to deploy widely — was systematically shown to be insufficient. Because static analysis cannot resolve indirect targets precisely, practical CFI lumps many call sites and many targets into large equivalence classes that share a label, and an attacker who confines their gadget chain to transfers that satisfy those coarse labels can still compute arbitrary results. Göktaş and colleagues demonstrated this directly, building working exploits that respect a coarse CFI policy yet defeat its intent.23 Davi and colleagues independently stitched together gadget chains that pass coarse-grained checks, arguing such protections are ineffective against a determined attacker.24 Most pointedly, Carlini and colleagues showed with their “control-flow bending” work that even fully precise static CFI — the strongest the model allows without a shadow stack — can be subverted: an attacker can make the program follow a path that is entirely valid in the CFG yet still performs the attacker's chosen malicious computation, because a static graph cannot capture which valid path is the intended one at a given moment.25 The remediation is precision plus a shadow stack for backward edges — but precision and shadow stacks cost performance and engineering, and the gap between “CFI” as a checkbox and CFI as a real defense is exactly this cost.

2 · Confinement is not memory safety within the sandbox

This is the most widely misunderstood limitation, and Wasm makes it vivid. SFI guarantees the untrusted module cannot escape its sandbox; it says nothing about whether the module corrupts itself inside the sandbox. Lehmann, Kinder, and Pradel showed that many classic memory-corruption vulnerabilities, which modern native toolchains have largely neutralized with stack canaries, ASLR, guard pages, and hardened allocators, are fully exploitable inside a WebAssembly module precisely because the protective mitigations of the native world are absent from the linear-memory model.19 Their attack primitives let an adversary write arbitrary locations within linear memory, overwrite data the source language considered constant, and corrupt the heap via a stack overflow — and because Wasm's linear memory has no internal page permissions, a buffer overflow in the module can clobber the module's own “read-only” data or function-table indices, redirecting a type-compatible indirect call.19 The sandbox holds — the host is safe — but the computation inside the sandbox can be fully hijacked, which matters enormously when the module itself processes secrets or makes security decisions. The structural irony they named is exact: everything old is new again.

3 · Transient execution: the microarchitecture below the model

SFI's guarantees are about the architectural execution the instruction set defines. Modern CPUs also execute transiently — speculatively, down mispredicted paths, with the wrong-path effects later squashed at the architectural level but leaving measurable traces in caches and other microarchitectural state. Spectre-class attacks weaponize exactly this gap, inducing the processor to speculatively perform accesses that the architectural semantics forbid and then exfiltrating the result through a timing side channel.21 For SFI this is devastating in principle: a sandboxed module can train the branch predictor so that the CPU speculatively executes past a bounds check or a CFI label check that, architecturally, would have trapped — reading host memory outside the sandbox during the speculation window and leaking it through the cache. The masking and checking instructions SFI relies on are bypassed not by defeating them but by racing ahead of them. Worse, transient execution turns the “merely a read” concession of read-permissive sandboxes into a full confidentiality breach. Mitigations exist but are unpleasant: serializing fences that stall speculation (with large overheads), masking that is designed to be speculation-safe, and hardware features (memory-protection keys, control-flow-enforcement instructions) repurposed to bound speculation. Narayan and colleagues' Swivel hardened WebAssembly specifically against Spectre, presenting both a software-only scheme for existing CPUs and a hardware-assisted scheme using newer processor extensions, and quantifying the uncomfortable spread between cheap-but-probabilistic mitigation and expensive-but-complete elimination.22

4 · The trusted computing base is small but not zero

Every SFI system rests on a verifier, a runtime, and an instruction decoder, and a bug in any of them is a full escape. The history is littered with validator bugs — subtle errors in modeling the x86 instruction set, in handling instruction prefixes, in the corner cases of self-modifying or JIT-generated code — which is precisely the motivation for the formal-verification lineage from PittSFIeld's ACL2 proof through RockSalt's Coq-checked decoder.13 The same concern drove work to push the trust further back: rather than verify the binary after an untrusted compiler produces it, prove the compiler's sandboxing pass correct once and for all, so that no separate per-binary verifier is needed — the approach Kroll, Stewart, and Appel took with portable software fault isolation built on a verified compiler.26 Even with proofs, the trust does not vanish; it relocates to the proof's assumptions, the proof checker, and the formal model's fidelity to the real silicon — which, as Spectre demonstrated, can diverge from the architecture in security-relevant ways.

5 · Side channels, resource exhaustion, and the policy gap

Finally, the safety-property boundary from Schneider's theory is a hard limit, not a soft one.3 SFI's inlined checks cannot enforce confidentiality against timing or cache side channels, cannot bound a sandboxed module's CPU or memory consumption without a separate scheduling/quota mechanism, and cannot stop a module from doing perfectly legal things maliciously through whatever host interface it was granted. A sandbox that exports a powerful host call confines memory and control flow flawlessly while still permitting abuse through that call — confinement of the mechanism is not confinement of the capability. This is why mature deployments pair SFI's intra-process confinement with the very process and OS isolation it was invented to economize on: SFI for speed and fine granularity at the inner boundary, hardware and kernel isolation as the outer backstop, exactly the layered posture NaCl modeled with its two nested sandboxes.11

The standing assessment

None of this makes SFI a failure; it makes it a tool with a precisely known envelope. For integrity confinement of untrusted code against an adversary who has not reached down to the microarchitecture, a well-implemented, precisely-CFI'd, verifier-backed SFI system — WebAssembly being the current high-water mark — is fast, sound within its model, and deployed at planetary scale. Its limits are real and they cluster at the edges of that model: in the imprecision of static control-flow graphs, in the absence of intra-sandbox memory safety, in the chasm between architectural and transient execution, and in the residual trust placed in a small but non-empty trusted computing base. The thirty-year arc from Wahbe's fault domains to Wasm's linear memory is, read one way, the story of shrinking that trusted computing base and tightening that control-flow graph — and read another, the story of attackers patiently finding the layer of the machine that the current model forgot to describe.
