Nmap occupies a peculiar position in the red team toolkit: it is simultaneously the most comprehensively documented network scanner in existence and the tool most thoroughly understood by the blue teams tasked with detecting its use. That tension — full-spectrum capability versus high signature density — defines every Nmap invocation in an adversarial engagement.

The question is never whether to use Nmap. It is which invocation, from which infrastructure, at what cadence, blended into what traffic profile, with the detection surface mapped in advance. A default nmap -sV -O target run from attacker-controlled infrastructure against a mature SOC is an announcement, not a reconnaissance mission. Understanding why — and how to use Nmap at the other extreme — is the subject of this article.

The treatment assumes working familiarity with TCP/IP, privilege-level requirements for raw socket access, and the general red team engagement model. The focus is operational precision: choosing the right probe, at the right timing, with the right evasion stack, informed by knowledge of what detection engineering looks for on the other side.

§ 01

Origins and Lineage

Gordon Lyon, operating under the pseudonym Fyodor, published the first public release of Nmap in September 1997 as a technical article in Phrack Magazine, issue 51.[1] The article, titled "The Art of Port Scanning," introduced not merely a tool but a systematic taxonomy of scanning techniques and their corresponding TCP/IP behaviours. The timing was significant: the mid-1990s had produced a crop of rudimentary port scanners — strobe, portscan, and early iterations of SATAN — but none integrated host discovery, multiple scan types, OS fingerprinting, and structured output into a single instrument with operator-controllable timing.[1],[2]

Lyon's initial implementation focused on three capabilities that distinguished it from contemporaries: half-open SYN scanning to reduce application-layer visibility, OS fingerprinting derived from TCP/IP stack behavioural differences across operating system implementations, and machine-readable output that could feed downstream tooling. These three pillars remain central to Nmap's identity more than two decades later.

The 2009 publication of Nmap Network Scanning, Lyon's official reference text, codified the tool's architecture, technique rationale, and operational theory into a canonical document that continues to serve as the primary source for both implementation detail and operational guidance.[2] The codebase is written primarily in C and C++, with the scripting engine layer implemented in Lua.[2] Development has been continuous, with the Nmap Scripting Engine introduced in the 4.x release series around late 2007 representing the tool's most significant architectural expansion post-launch.[4]

Nmap's cultural reach beyond technical circles is notable: it has appeared in films including The Matrix Reloaded and multiple other productions depicting network intrusion, has received LinuxWorld's Editor's Choice Award, and was ranked first in the "Top 100 Network Security Tools" survey for several consecutive years.[2] For red teams, this visibility cuts both ways: detection engineers are familiar with Nmap's signatures precisely because the tool is so widely known and studied.

"Detection engineers are familiar with Nmap's signatures precisely because the tool is so widely known and studied. Ubiquity is not the same as invisibility."

§ 02

Nmap Architecture

Nmap operates across four conceptually distinct phases on each invocation: host discovery, port scanning, service/version detection, and optional script execution.[2] These phases are largely sequential per host, though Nmap parallelises aggressively across hosts within a target range to minimise wall-clock time. Understanding the phase architecture matters operationally because each phase has an independent noise profile and OPSEC cost.

Host Discovery establishes which addresses in the target scope respond before committing scanning resources to dead hosts. By default on non-local networks, Nmap sends an ICMP echo request, a TCP SYN packet to port 443, a TCP ACK packet to port 80, and an ICMP timestamp request.[2] On local-segment targets, ARP is used instead. Each of these has a different detection profile and firewall-traversal probability.

Port Scanning generates crafted packets to each discovered host and interprets responses to categorise ports as open, closed, or filtered. The distinction relies on TCP handshake semantics, ICMP unreachable messages, or the absence of response within configurable timeout windows.[2],[15]

Service/Version Detection (-sV) sends application-layer probes to open ports and matches responses against a database of thousands of service and version signatures maintained in nmap-service-probes.[2]

Script Execution invokes user-specified or default NSE scripts against discovered hosts and ports, extending enumeration into application-layer territory.[10]

Architecture Note

The -A flag enables OS detection, version detection, script scanning, and traceroute simultaneously. It is a convenience flag, not an OPSEC flag. On any adversarial engagement, -A should be decomposed into its constituent components and applied selectively based on the intelligence requirement and the noise budget for each target.

§ 03

Host Discovery Without the Noise

The default host discovery mechanism is detectable by most modern IDS platforms. ICMP echo requests are blocked or monitored at virtually every enterprise perimeter; TCP ACK to port 80 generates RST from nearly all stateful firewalls; TCP SYN to 443 completes the first step of a tracked connection. For red teams, the choice of discovery probes is a first-order OPSEC decision, not a default-accepted setting.[5],[6]

-Pn disables host discovery entirely, treating all targets as live. The operational cost is scan time against dead addresses; the OPSEC benefit is eliminating the distinct multi-protocol discovery signature. Use when the target IP range is small and confirmed populated, or when perimeter filtering makes discovery unreliable anyway.

-PS<portlist> sends TCP SYN packets to specified ports. -PS80,443 blends with normal web traffic from the port selection perspective, though the timing and source still require management. -PA<portlist> uses TCP ACK, which can traverse certain stateless ACLs that block SYN but permit established-session traffic.

-PU<portlist> (UDP ping) can reach hosts that block all ICMP and TCP probes. Sending UDP to high closed ports elicits ICMP port-unreachable responses, confirming the host is live without a TCP handshake footprint.

ARP ping (-PR) is the most reliable discovery mechanism on local Ethernet segments and produces no routable traffic visible to network-layer sensors. On a compromised internal host performing internal reconnaissance, -PR is the preferred starting point for local-segment host enumeration.

OPSEC

For engagements requiring maximum stealth across a large range: sequence a slow ARP sweep (-PR -T1) on local subnets first, then TCP SYN pings to common ports (-PS80,443,8080 with --scan-delay 10s) against remote ranges. Never issue the default multi-protocol discovery blast against an environment with active network monitoring.

bash — Low-noise host discovery 
# Local segment — ARP only, T1 timing, randomised host order
nmap -PR -sn -T1 --randomize-hosts 192.168.10.0/24

# Remote range — TCP SYN on web ports only, no ICMP, 5-second probe delay
nmap -PS80,443,8080,8443 -sn -Pn --scan-delay 5s --randomize-hosts 10.0.0.0/16

# Single host confirmation — TCP ACK (bypasses some stateless ACLs)
nmap -PA443 -sn -Pn 10.0.1.45
§ 04

Port Scanning Taxonomy

MITRE ATT&CK categorises active port scanning under T1595 (Active Scanning) for pre-compromise reconnaissance and T1046 (Network Service Discovery) for post-compromise internal enumeration.[6],[7] The distinction matters for red team planning: the noise tolerance, available privileges, and evasion stack differ substantially between external and internal operations.

4.1 — SYN Scan (-sS)

The SYN scan sends a TCP SYN segment and evaluates the response. An open port responds with SYN-ACK; Nmap immediately transmits RST, aborting the handshake before the connection completes. A closed port responds with RST directly. Filtered ports produce no response, or ICMP unreachable messages.[2],[15]

The "half-open" nature of the SYN scan means that the connection never progresses to an established state, and no application-layer log entry is generated on the target service. However, stateful firewalls do track the SYN in their session table briefly before it ages out. The SYN scan is Nmap's default when executed with sufficient privileges (raw socket access, typically root or equivalent) and represents the best balance of reliability, speed, and application-layer invisibility for most targets.[2]

bash — SYN scan with common port targeting 
# Targeted SYN scan, top 1000 ports, T2 timing, no version detection
nmap -sS -T2 --top-ports 1000 -Pn 10.0.1.0/24

# Specific service ports, OPSEC-conscious timing
nmap -sS -p 22,80,443,445,3389,8080,8443 -T1 --scan-delay 2s -Pn 10.0.1.45

4.2 — TCP Connect Scan (-sT)

The connect scan delegates connection establishment to the operating system's TCP stack, completing the full three-way handshake. This is detectable at the application layer: the connection registers in server logs, appears in netstat on the target, and creates full session state in firewalls and SIEM correlation.[2]

It is the only scan type available to unprivileged users, as it requires no raw socket access. On post-compromise internal operations where the attacker is running as a low-privileged service account, -sT or native OS tooling (see §9) is the realistic option. The noise profile makes it inappropriate for any stealthy external reconnaissance.

4.3 — FIN, NULL, and Xmas Scans (-sF / -sN / -sX)

These scan types exploit a specific provision in RFC 793: a closed port receiving a segment with no SYN, ACK, or RST flag set should respond with RST, while an open port should silently discard it.[15] Three implementations exploit this:

FlagNameTCP Flags SetEvasion Target
-sFFIN ScanFINStateless ACLs permitting non-SYN
-sNNULL ScanNoneSame; minimal signature
-sXXmas ScanFIN, PSH, URGNamed for "lit-up" flag pattern

These scans can bypass stateless packet filters that inspect only SYN-bearing packets and permit established-session traffic. A firewall rule permitting tcp any any established without proper stateful inspection may allow FIN/NULL/Xmas probes through while blocking SYN.[2],[8]

The critical limitation: Windows targets do not comply with RFC 793 in this respect. Microsoft's TCP/IP stack responds with RST regardless of port state when receiving these anomalous packets, making it impossible to distinguish open from closed ports on Windows systems with these scan types.[2] Additionally, any modern stateful inspection device or IDS monitoring for non-SYN TCP without established session context will flag these scans immediately.

Detection Surface

FIN, NULL, and Xmas packets are generated by zero legitimate applications in normal operation. Any detection system monitoring for TCP segments with these flag combinations arriving outside an established session — including Snort and Suricata with default rulesets — will alert immediately regardless of timing.[9] Reserve these scan types for specific scenarios where stateless ACL bypass is the objective and IDS detection is assessed as absent or misconfigured.

4.4 — IDLE / IPID Scan (-sI) T1595

The IDLE scan is operationally the most sophisticated technique in Nmap's core feature set. It was discovered by Salvatore Sanfilippo (Antirez) and published to the Bugtraq security mailing list in December 1998.[3] The technique exploits the incremental IP identification (IPID) field present in IPv4 headers of certain "zombie" hosts — machines that are otherwise idle and therefore maintain a predictable, monotonically incrementing IPID counter.[3],[12]

The mechanism operates in three steps:[2]

  1. Probe the zombie host to read its current IPID value (e.g., IPID = 1000).
  2. Send a spoofed SYN packet to the target, with the source address set to the zombie's IP. If the target port is open, it replies with SYN-ACK to the zombie; the zombie (surprised by an unexpected SYN-ACK) sends RST and its IPID increments.
  3. Probe the zombie again. If IPID has incremented by 2 (e.g., 1002), the target port is open. If by 1, the port is closed or filtered.

The scanning host's IP address never appears in traffic to the target. The scan appears to originate entirely from the zombie. The operational value for attribution-sensitive operations is substantial: properly executed IDLE scanning attributes all reconnaissance traffic to a third-party IP address.[2]

OPSEC

Zombie selection is the critical constraint. The zombie must have a globally incrementing (not random, per-connection, or per-flow) IPID field, must be accessible by the attacker, and must be reachable by the target. Most modern operating systems (Linux kernels ≥ 3.x, Windows Vista and later) use randomised IPIDs, making them unsuitable as zombies. Suitable candidates are often older embedded devices, printers, or network appliances. Verify zombie suitability with nmap -O --osscan-limit -p <port> zombie_ip and inspect IPID sequence values in the output.

bash — IDLE scan syntax 
# -sI zombie_host:probe_port — probe_port is open on zombie (to elicit IPID increments)
nmap -sI 192.168.5.20:80 -p 22,80,443,3389 10.0.1.100

# Verify zombie IPID incrementing predictability first
nmap -O -v --packet-trace -p80 192.168.5.20 2>&1 | grep "IPID"

4.5 — UDP Scanning (-sU)

UDP scanning is inherently slower and more ambiguous than TCP scanning. With no handshake mechanism, Nmap relies on ICMP port-unreachable messages to confirm closed ports; open UDP ports typically produce no response, making them indistinguishable from filtered ports under packet loss conditions.[2]

The practical implication: UDP scans generate high false-positive rates (open|filtered) without service-layer probes. Combining -sU with -sV allows Nmap to send application-specific UDP payloads (DNS queries to port 53, SNMP GetRequest to port 161, SSDP to port 1900) and validate responses. UDP service detection is one area where targeted rather than exhaustive scanning has clear operational value: scan only the UDP services relevant to the engagement (DNS, SNMP, TFTP, NTP, LDAP) rather than issuing blanket -sU --top-ports 1000 runs.

OPSEC

Rate-limit UDP scans aggressively. --max-rate 5 (5 probes per second) prevents the high volume of ICMP unreachable responses from creating detectable bursts on the target's network egress. ICMP rate limiting on the target host can cause false open|filtered results at high probe rates — another reason for slow, targeted UDP enumeration.

§ 05

Service and OS Detection

5.1 — Service Version Detection (-sV)

Nmap's service detection engine sends a sequential series of application-layer probes to each open port, matching responses against a database of service and version signatures maintained in nmap-service-probes.[2] The database has grown to encompass thousands of fingerprints across hundreds of service types, enabling confident version identification of common services without operator-supplied hints.

The --version-intensity flag (0–9) controls the aggressiveness of probing: intensity 0 sends only the most statistically likely probes for a given port, while intensity 9 exhausts every probe in the database regardless of port assignment.[2] The performance and noise implications are significant: intensity 9 against a large port range generates substantial traffic volume and wall-clock time. For OPSEC-sensitive operations, --version-intensity 2 against a curated subset of interesting open ports is the appropriate posture.

Version detection against a confirmed Apache/Nginx web service can be meaningfully replaced with a manual banner grab or a single targeted HTTP request, avoiding the multi-probe signature of -sV entirely. Craft the minimal operation that retrieves the intelligence requirement; -sV at default intensity is rarely that operation.

bash — Targeted version detection 
# Version detection on confirmed open ports only, low intensity
nmap -sV --version-intensity 2 -p 22,80,443 10.0.1.45

# Manual banner grab (no Nmap binary involved)
nc -w2 10.0.1.45 22
curl -sI --max-time 3 http://10.0.1.45/ | grep -i "server:"

5.2 — OS Fingerprinting (-O)

Nmap's OS fingerprinting engine transmits a carefully designed battery of probes — TCP SYN, SYN+ECN, FIN/PSH/URG, ACK, and several others against both open and closed ports, plus ICMP and UDP probes — and analyses the responses against a database of OS fingerprint records.[2],[11] The discriminating features include initial window size, TTL, the DF (Don't Fragment) bit, TCP options ordering and values, IP ID generation pattern, and RST sequence number generation behaviour.[2]

Accuracy degrades significantly when targets are behind NAT, reverse proxies, load balancers, or network address translation middleboxes, as these devices often normalise TCP/IP header fields before forwarding.[11] The --osscan-limit flag restricts OS detection to hosts with at least one open and one closed port, avoiding inconclusive and time-consuming fingerprinting attempts against fully filtered targets.[2]

The OS fingerprinting probe battery is one of the most distinctive signatures Nmap produces from a packet-analysis perspective. The combination of unusual TCP flag combinations, specific option sequences, and rapid sequential probes against both open and closed ports creates a detectable pattern in passive network monitoring even at T1 timing.[13] OS fingerprinting should be reserved for specific intelligence requirements that justify its noise cost.

Detection Surface

Zeek (formerly Bro) and similar network security monitoring frameworks perform passive OS fingerprinting through TCP SYN analysis and can identify Nmap probe sequences from the distinctive TCP option patterns alone, independent of timing.[13] p0f achieves similar passive fingerprinting. Neither is dependent on rate-based thresholds — they identify packet-level anomalies. Timing evasion does not defeat signature-based OS fingerprinting detection.

§ 06

The Nmap Scripting Engine

6.1 — Architecture and Execution Model

NSE was integrated into Nmap in the 4.x release series, around late 2007, representing the tool's most substantial post-launch architectural addition.[4],[10] Scripts are authored in Lua 5.x and executed in parallel using a coroutine-based cooperative scheduler within the Nmap process.[10],[16] The engine exposes a layered API through standard libraries:

LibraryPurpose
nmapCore API: host/port objects, packet sending, interface access
stdnseUtilities: output formatting, sleep, debug logging, conditional execution
shortportPort matching helpers: shortport.http, shortport.ssl, etc.
httpFull HTTP client with cookie management, redirect following
bruteGeneric credential testing framework with lockout avoidance
vulnsStandardised vulnerability reporting structures
msrpc, smbWindows protocol enumeration

NSE scripts define four optional execution hooks: prerule (before host discovery), hostrule (per-host, before port scanning), portrule (per open port), and postrule (after all scanning completes). The portrule hook is the most common and is where targeted service enumeration scripts execute.[10]

6.2 — Script Categories

The 600+ bundled NSE scripts are organised into 14 categories that form a spectrum from passive information gathering to active exploitation.[10]

CategoryOPSEC CostRelevance

For adversary emulation, the vuln category is operationally relevant but requires careful selection. Category-wide invocation (--script vuln) runs every vulnerability check script, many of which probe paths and service behaviours that generate WAF alerts, SIEM hits, and EDR telemetry. Targeted single-script invocation (--script http-shellshock, --script smb-vuln-ms17-010) reduces the signature to a specific, understood interaction.

6.3 — Targeted NSE Development

Custom NSE scripts allow red teams to perform highly specific checks without generating the broad-spectrum traffic of generic vulnerability scanners. Developing a targeted script for a known-vulnerable endpoint in the target environment is meaningfully stealthier than running a commercial scanner against the same target: one precisely-crafted HTTP request versus hundreds of probe packets.[10],[17]

Lua — Minimal NSE script structure 
local http      = require "http"
local shortport = require "shortport"
local stdnse    = require "stdnse"
local vulns     = require "vulns"

description = [[
  Checks for an exposed administration endpoint on HTTP services.
  Returns a finding if the path is accessible and returns HTTP 200.
]]

author     = "Red Team Operator"
license    = "Same as Nmap--See https://nmap.org/book/man-legal.html"
categories = { "vuln", "safe" }

-- Only execute against standard HTTP ports
portrule = shortport.http

action = function(host, port)
  local target_path = "/admin/config.php"
  local response    = http.get(host, port, target_path)

  if not response then
    return nil
  end

  if response.status == 200 then
    local vuln_report = vulns.Report:new(SCRIPT_NAME, host, port)
    local vuln = {
      title  = "Exposed administration endpoint",
      state  = vulns.STATE.VULN,
      descr  = "Path " .. target_path .. " returned HTTP 200",
      refs   = { "https://owasp.org/www-project-top-ten/" }
    }
    return vuln_report:make_output(vuln)
  end

  return nil
end

Operational considerations for custom script development:[10]

  • Respect scan delays — use stdnse.sleep() to honour --scan-delay settings within multi-request scripts.
  • Structured output — use the vulns library for findings; -oX XML output will include structured data consumable by post-processing tooling.
  • Error handling — network timeouts and nil responses must be handled explicitly; uncaught exceptions abort the script and print debug output to stdout.
  • No subprocess spawning — NSE scripts execute within the Nmap process; spawning external processes via io.popen creates unexpected artefacts on the scanning host.
bash — Script invocation patterns 
# Single targeted script — surgical, minimal signature
nmap --script smb-vuln-ms17-010 -p 445 10.0.1.0/24

# Custom script from local path
nmap --script /opt/redteam/scripts/check-admin-endpoint.nse -p 80,443 10.0.1.45

# Multiple scripts, explicit list — avoids category-wide blast
nmap --script "http-title,http-headers,http-auth-finder" -p 80,443,8080 10.0.1.45

# Script argument passing
nmap --script http-brute --script-args brute.mode=user -p 80 10.0.1.45
§ 07

OPSEC and Detection Avoidance

7.1 — Timing Templates

Nmap's -T<0–5> timing templates are compound controls that adjust probe timeouts, retry limits, probe parallelism, and inter-probe delays simultaneously.[2] They are the most visible OPSEC lever but the least granular.

TemplateNameMin Probe DelayOperational Use
T0Paranoid5 min / probeDefeat all rate-based detection; serialised (no parallelism)
T1Sneaky15 s / probeIDS evasion; practical for targeted scans
T2Polite400 ms / probeReduce bandwidth; not an OPSEC template
T3NormalDefaultDefault; unsuitable for adversarial engagements
T4Aggressive~10 msInternal trusted networks with monitoring disabled
T5Insane~5 msLab/CTF environments only

T0 and T1 defeat most rate-based detection rules. A common Snort/Suricata detection threshold triggers on 20 or more SYN packets per second from a single source; T1's 15-second inter-probe delay reduces the rate to ~0.07 probes/second — well below any rate-based threshold.[9]

The template system is a convenience abstraction. For precise control, use individual timing flags: --scan-delay (minimum delay between probes to a single host), --min-rtt-timeout/--max-rtt-timeout (round-trip timeout bounds), and --max-rate (global rate ceiling in probes/second). These allow combining a generous per-host delay with a reasonable global rate across a large target range.

bash — Fine-grained timing control 
# 10-second minimum between probes per host, 2-probe/sec global ceiling
nmap -sS --scan-delay 10s --max-rate 2 -p 80,443,22,445 10.0.0.0/22

# T1 with further restriction on overall scan rate
nmap -sS -T1 --max-rate 1 --randomize-hosts -p 22,80,443 10.0.0.0/24

7.2 — Decoys (-D)

The -D flag inserts additional spoofed source IP addresses into the scan packet stream alongside the real source, creating a pool of apparent origins that complicates attribution by defenders reviewing connection logs or NetFlow data.[2]

bash — Decoy examples 
# Named decoys — ME marks the true source position in the list
nmap -sS -D 10.0.5.10,10.0.5.22,ME,10.0.5.44 -p 80,443 10.0.1.100

# Random decoys — 8 random IPs plus the true source
nmap -sS -D RND:8 -p 22,80,443 10.0.1.100

Operational constraints that limit decoy effectiveness:[2],[8]

  • Decoys must be live hosts. Spoofed addresses belonging to down hosts generate no SYN-ACK responses on open ports, creating a detectable asymmetric pattern: decoy IPs never receive SYN-ACK responses, the real source does.
  • The real source IP is still present in the packet stream. Defenders correlating which addresses receive SYN-ACK replies from the target can isolate the real scanner.
  • Asymmetric path analysis defeats decoys entirely. NetFlow or packet capture between the attacker and target will identify the true source regardless of decoy count.
OPSEC

For credible decoys, use live IP addresses from the same autonomous system or subnet as the target to avoid routing anomalies. RND:N decoys are easily identified as random if defenders observe the range — fabricated IPs from outside the relevant address space do not blend with expected traffic patterns.

7.3 — IP Fragmentation (-f / --mtu)

IP fragmentation splits probe packets into smaller fragments that must be reassembled before content inspection. -f sets a fragment size of 8 bytes; -ff sets 16 bytes; --mtu N sets a custom size (must be a multiple of 8).[2],[14] The intent is to defeat IDS signatures that match on complete packet contents.

The practical effectiveness of fragmentation as an evasion technique has declined significantly. Most modern IDS/IPS platforms, including Suricata and Snort with default configurations, perform fragment reassembly before applying detection rules.[9] Fragmentation is most relevant against:

  • Legacy network appliances (older firewalls and IDS sensors) that forward fragments without reassembly.
  • High-throughput links where reassembly is computationally expensive and detection devices may disable it under load.

Combined with decoys and timing delay, fragmentation adds modestly to the overall evasion profile, but should not be treated as a primary detection avoidance mechanism against any modern, maintained sensor stack.

7.4 — Source Port Manipulation (--source-port)

Many network access control lists permit traffic appearing to originate from source port 53 (DNS), 80 (HTTP), or 443 (HTTPS) under the assumption that these are replies from trusted services. --source-port N forces Nmap to use a specific source port for all probes.[2],[8]

bash — Source port manipulation 
# Use source port 53 to traverse ACLs that trust DNS reply traffic
nmap -sS --source-port 53 -p 1-1024 10.0.1.45

# HTTPS source port — appears as reply from an HTTPS server
nmap -sS --source-port 443 -p 22,3389,5985 10.0.1.45

This technique is effective against stateless packet filters and poorly implemented ACLs, but ineffective against stateful inspection, which validates that source port 53 traffic is part of an established DNS session. Additionally, a single source IP generating traffic from port 53 to hundreds of diverse destination ports is anomalous and detectable via NetFlow analysis — no DNS client legitimately queries dozens of non-DNS endpoints in rapid succession.[8]

7.5 — Packet Shaping and Randomisation

Several Nmap flags influence packet-level characteristics that feed into signature matching:

--data-length N appends N bytes of random payload data to probe packets, altering their size away from the fixed-size signatures produced by default Nmap probes.[2] Many IDS signatures match on precise packet lengths; this flag disrupts that matching.

--ttl N sets a custom TTL value in probe packets, which can be used to mimic specific OS behaviour or to avoid TTL-based path analysis.

--spoof-mac <MAC|0|vendor> spoofs the Ethernet MAC address of probe packets. Effective only on local-segment scans where Ethernet frames are not re-encapsulated at Layer 3 boundaries.

--randomize-hosts randomises the order in which target hosts are scanned, disrupting the sequential scanning pattern that is one of the primary heuristics used in flow-based scan detection.[8] This flag should be considered mandatory for any OPSEC-sensitive scan involving a range of more than a handful of hosts.

bash — Combined evasion stack 
# Layered evasion: T1 timing + source port 443 + data padding +
# randomised host order + two named decoys from same /24
nmap -sS \
  -T1 \
  --source-port 443 \
  --data-length 48 \
  --randomize-hosts \
  -D 10.0.0.14,10.0.0.27,ME \
  -p 22,80,443,445,3389,5985,8080,8443 \
  -Pn \
  -oX /tmp/scan-$(date +%Y%m%d).xml \
  10.0.0.0/24
§ 08

What Blue Teams See — The Detection Engineering Perspective

Adversary emulation requires internalising the detection engineer's model. Understanding precisely what telemetry each Nmap feature produces allows red teams to calibrate noise budgets, predict alert thresholds, and design engagement scenarios that replicate specific threat actor behaviours rather than generic "scanner noise."[5],[17]

"Realistic adversary emulation replicates targeted, slow, protocol-legitimate traffic — not the broadside of a default Nmap invocation."

Rate-Based Detection

Suricata, Snort, and equivalent IDS platforms can trigger on probe rates exceeding configurable thresholds within observation windows.[9] Example: a detection rule triggering on 20 SYN packets per second from a single source catches default Nmap at T3 immediately. T1 timing (15 seconds between probes) reduces the observable rate to ~0.07 probes/second — defeating all rate-based thresholds but requiring operator patience.

Detection Surface — Rate-Based

Rate-based rules are the most common Nmap detection mechanism and the easiest to defeat with timing controls. However, defeating rate-based detection is necessary but not sufficient: signature-based, anomaly-based, and behavioural detection mechanisms operate independently and are not defeated by timing alone.

TCP/IP Fingerprint Anomalies

Nmap's OS detection probes generate TCP segments that no legitimate application produces: RST-ACK packets to closed ports with specific IP ID generation patterns, ACK probes with unusual window sizes, FIN probes without established sessions. Passive fingerprinting tools including Zeek and p0f analyse these features at wire rate without signatures — they identify the behavioural pattern regardless of timing.[13] Timing evasion does not defeat passive TCP/IP fingerprint anomaly detection.

Sequential Scanning Patterns

Default Nmap scans hosts and ports in sequential order unless --randomize-hosts is specified. The pattern of incrementing destination addresses at regular intervals is detectable via NetFlow and IPFIX correlation at the collector level, independent of IDS signature matching.[5],[8] Any SOC with flow-based analysis capability can identify sequential scanning regardless of timing or source port manipulation. --randomize-hosts disrupts the sequential address pattern; randomising the port list (--port-ratio or -r flag absence) disrupts the sequential port pattern.

Adversary Emulation Calibration

A realistic APT reconnaissance profile against a mature environment involves: one to five target hosts per engagement day, T1 timing with additional scan delay, SYN scan only (no OS fingerprinting), version detection at intensity 2 or below against confirmed interesting ports, no category-wide NSE execution, and output suppressed to prevent on-disk artefacts on scanning infrastructure. This profile generates minimal NetFlow anomalies, no rate-based alerts, and marginal passive fingerprint signatures — at the cost of slow intelligence development.[18]

§ 09

Living Off the Land — When Nmap Is Wrong for the Job

Nmap binary execution on a compromised host generates filesystem artefacts, spawns a recognisable process visible in system telemetry, and may trigger EDR behavioural detections based on raw socket access, high-volume TCP connection attempts, or process hollowing heuristics.[7] The calculus for post-compromise internal enumeration is direct: Nmap is a tool for controlled red team infrastructure, not for host-level post-exploitation in mature defended environments.

Several native operating system mechanisms support minimal-footprint port enumeration that blends with legitimate administrative activity.[5] None are invisible — process-level logging (auditd, PowerShell Script Block logging, Sysmon) and network-layer analysis still apply — but their process signatures differ substantially from Nmap execution.

bash — /dev/tcp port check (Linux, no binary) 
# Single port — no tooling required, uses bash built-in
timeout 1 bash -c "echo > /dev/tcp/10.0.1.45/443" 2>/dev/null \
  && echo "OPEN" || echo "CLOSED/FILTERED"

# Port sweep across a range (generates sequential pattern — rate-limit)
for port in 22 80 443 445 3389 5985; do
  timeout 0.5 bash -c "echo > /dev/tcp/10.0.1.45/$port" 2>/dev/null \
    && echo "Port $port: OPEN"
  sleep 3
done
powershell — Native Windows port check 
# Test-NetConnection — generates PowerShell telemetry, not Nmap telemetry
$target = "10.0.1.45"
@(22, 80, 443, 445, 3389, 5985) | ForEach-Object {
  $result = Test-NetConnection -ComputerName $target -Port $_ `
              -InformationLevel Quiet -WarningAction SilentlyContinue
  if ($result) { Write-Output "Port $($_): OPEN" }
  Start-Sleep -Seconds 3
}
python — Socket-level enumeration 
import socket, time

target = "10.0.1.45"
ports  = [22, 80, 443, 445, 3389, 5985]

for port in ports:
    s = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
    s.settimeout(0.8)
    result = s.connect_ex((target, port))
    if result == 0:
        print(f"Port {port}: OPEN")
    s.close()
    time.sleep(3)       # rate-limit between probes
OPSEC — LotL Enumeration

Native OS tools reduce the binary execution footprint but do not eliminate network-layer visibility. /dev/tcp, Test-NetConnection, and Python socket calls all produce TCP SYN packets on the wire. The OPSEC advantage is in the host-level telemetry: no nmap.exe or nmap process, no raw socket capability invocation at the process level — just what looks like a script making individual TCP connections. Pair with per-probe delays of 3–10 seconds and a targeted port list to minimise the detectable scanning pattern.
