// prologue

The language that ended up everywhere

No language of comparable reach was designed in less time or under stranger constraints. JavaScript was sketched as a prototype in roughly ten days in May 1995 by Brendan Eich at Netscape, meant as a lightweight glue for web pages while a “serious” language handled the real work.[1][2] Thirty years later it is the only programming language that runs natively in every web browser on earth, and by the most direct measure of usage it is the most widely used language in professional software: 66 % of respondents to the 2025 Stack Overflow Developer Survey reported writing it in the past year — a position it has held every year since 2015.[3][4]

That gap between humble origins and total ubiquity is the story of JavaScript. It is a language that was never supposed to be important, standardized under a name it was only licensed to borrow, nearly torn apart by its own committee, and then rescued by a browser war, a fast engine, and a runtime that let it escape the browser altogether. This article traces that arc — the history and the versions, the design philosophy of its creators, the machinery that makes it fast, its standing against other languages, and the libraries, frameworks, runtimes, and integrations that turned a scripting toy into the substrate of the modern web.

// history

Ten days in 1995

In 1994 Netscape's Navigator was the dominant browser, and the web was almost entirely static — HTML documents with no way to react to a user. Netscape wanted a scripting language that designers could drop into a page to validate a form or animate an element without a round trip to the server.[1] Eich was hired in April 1995 and, under intense schedule pressure, produced a working prototype in about ten days.[1][2]

The language went through three names in a single year. It was built as Mocha, shipped in a Netscape Navigator 2.0 beta as LiveScript in September 1995, and finally renamed JavaScript that December.[5][1] The rename was a marketing manoeuvre: Netscape and Sun Microsystems closed a deal in which Netscape licensed the “Java” name, positioning the little scripting language as a companion to Sun's much-hyped Java, which would supposedly handle the heavyweight components.[2] The two languages are unrelated, and the borrowed name has confused newcomers ever since. When Microsoft shipped its own compatible implementation in Internet Explorer, it could not use the trademarked name and called it JScript instead.[5]

The name still causes friction today. Oracle inherited the JavaScript trademark when it acquired Sun, and in 2024 the Deno team petitioned to have it cancelled as an abandoned, generic term — a dispute that was still winding through the process as of 2025.[6]

// versions & change

ECMAScript: the standard behind the name

Because “JavaScript” was a trademark, the language was standardized under a different label. Netscape submitted it to Ecma International, which published the first edition of ECMA-262 in June 1997; the standardized language is called ECMAScript, and in practice the two names are used interchangeably.[7] The standard is maintained by a committee known as TC39, and the relationship is simple to state: ECMAScript is the specification, and JavaScript is its overwhelmingly dominant implementation, alongside engines like V8 and SpiderMonkey.[7]

The early editions arrived quickly — the first edition in 1997, a mostly editorial second edition in 1998, and a substantial third edition in December 1999 that added regular expressions, try/catch exception handling, and better string handling.[7] Then progress stalled for a decade, and the reason is one of the more instructive episodes in language governance.

The fourth edition that never shipped

A proposed fourth edition (ES4) aimed to remake the language: classes, modules, namespaces, and an optional static type system.[8] The committee split over whether such an ambitious overhaul was wise, and at a TC39 meeting in Oslo in July 2008 the effort was abandoned outright.[8] In its place the group adopted a “Harmony” strategy of modest, consensus-driven change. The immediate result was ECMAScript 5 (December 2009), a careful, incremental release whose headline feature was strict mode, an opt-in subset with tighter error checking.[7][8] The lesson — that a language used by millions cannot be rebooted, only evolved — would shape every decision that followed.

ES6 and the annual cadence

The turning point was ECMAScript 2015, the sixth edition, still widely known as ES6. Developed for years under the “Harmony” banner, it delivered the modern language most developers now take for granted: let and const, arrow functions, classes, template literals, destructuring, modules, promises, and much more.[7] With that release TC39 also switched to a yearly release train — features ship the moment they are ready, and each June's stable set is stamped with the year, giving ES2016, ES2017, and so on.[7] The most recent edition, ECMAScript 2025, was approved by Ecma's General Assembly on 25 June 2025 as the 16th edition of the standard.[9][10]

Modern proposals move through a five-stage pipeline (Stage 0 through Stage 4); reaching Stage 4 means a feature is finished and slated for the next edition. ES2025 added iterator helper methods, native Set operations, JSON modules via import attributes, a RegExp.escape() method, and Promise.try(), among others.[9][10]

es2025.js
// ES2025 — three of the 16th edition's additions
import config from "./config.json" with { type: "json" }; // JSON modules

const primes = new Set([2, 3, 5, 7]);
const odds   = new Set([1, 3, 5, 7, 9]);
primes.intersection(odds);            // Set { 3, 5, 7 } — native set math

const squares = [1, 2, 3, 4, 5, 6]
  .values()
  .map(n => n * n)
  .take(3)
  .toArray();                         // [1, 4, 9] — lazy iterator helpers

The full arc, from a two-week prototype to a specification revised every year, looks like this:

  • 1995
    Mocha → LiveScript → JavaScript

    Brendan Eich builds a prototype at Netscape in ~10 days; it ships in Navigator 2.0 and is renamed after a Sun licensing deal.[1][2]

  • 1997–1999
    ECMA-262, editions 1–3

    Standardized as ECMAScript; the third edition adds regular expressions and exception handling.[7]

  • 2008
    ES4 abandoned

    TC39 scraps the ambitious fourth edition in Oslo and adopts the incremental “Harmony” path.[8]

  • 2008–2009
    V8, then Node.js — and ES5

    Google ships the fast V8 engine; Ryan Dahl uses it to build Node.js; ECMAScript 5 introduces strict mode.[11][7]

  • 2012
    TypeScript

    Microsoft releases a typed superset that compiles to plain JavaScript.[12]

  • 2015
    ES6 / ECMAScript 2015

    The landmark modern edition; TC39 switches to yearly releases.[7]

  • 2018–2022
    Deno, then Bun

    New runtimes challenge Node with security-by-default, native TypeScript, and raw speed.[13][14]

  • 2025–2026
    ES2025 — and the AI era

    The 16th edition ships; TypeScript becomes the most-used language on GitHub; Temporal lands for ES2026.[9][15][16]

// design philosophy

A hybrid, born under pressure

Eich has said he was hired with the expectation of putting Scheme — a minimalist dialect of Lisp — into the browser. The deal with Sun changed the brief: management wanted something that looked like Java to ride its popularity.[2] The result is a language of deliberate contradictions. On the surface it wears Java's C-style syntax, with curly braces and familiar keywords. Underneath, it is something else entirely: first-class functions and closures borrowed from Scheme, and a prototype-based object model taken from a research language called Self.[2] It looks like a dynamic Java and behaves like a scripting Lisp in Java's clothing.

The language you could argue was designed for hundred-line scripts is now used, with regularity, to write applications a hundred thousand lines long.

— paraphrasing Anders Hejlsberg on why he built TypeScript, 2012 [17]

The most consequential design commitment is not a feature but a discipline: backward compatibility. The abandonment of ES4 hardened an unwritten rule that TC39 still follows — do not break the web. Code written in 1997 is expected to keep running in a 2026 browser, which is why the language accretes rather than replaces, and why its odder early decisions have never been removed.[8] This is the source of both JavaScript's remarkable durability and its reputation for quirks: the tolerance that makes a page load rather than fail is the same tolerance that turns "5" - 1 into the number 4.

That tension — forgiving, dynamic, and quick to write, versus predictable, typed, and safe to scale — is the axis along which the entire ecosystem has evolved. Nearly every major development of the last decade, from TypeScript to the design of new runtimes, is an attempt to keep JavaScript's expressiveness while taming its looseness.

// technical merits

Under the hood

Three ideas do most of the work in JavaScript, and understanding them explains both its power and its idiosyncrasies: functions are values, objects inherit from other objects, and concurrency happens without threads.

Functions are first-class

A function in JavaScript is an ordinary value. It can be stored in a variable, passed as an argument, returned from another function, and it captures the variables in scope where it was defined — a closure. This one property, inherited from Scheme, is what makes the language's callback-driven, event-oriented style possible, and it underpins nearly every library API in the ecosystem.[2]

closure.js
// The returned function "closes over" count and
// remembers it across calls — a closure, borrowed from Scheme.
function counter(start = 0) {
  let count = start;
  return () => ++count;               // arrow function keeps the scope
}

const next = counter(10);
next();  // 11
next();  // 12  — state lives in the closure, not in a global

Objects inherit from objects

JavaScript has no classes in the traditional sense — the class keyword added in ES6 is syntactic sugar over a prototype mechanism taken from the language Self.[2] Every object holds a hidden link to another object, its prototype; when you read a property the engine walks this chain until it finds the property or reaches null. Inheritance is therefore just objects delegating to other objects, which is unusually flexible and, for newcomers from class-based languages, unusually surprising.

Concurrency without threads: the event loop

JavaScript runs on a single thread with a single call stack — it does exactly one thing at a time.[18] It nonetheless handles thousands of simultaneous network requests, timers, and user events, and it does so through the event loop, the single most characteristic idea in the language. The engine itself only runs synchronous code; every asynchronous capability comes from the surrounding host — the browser's Web APIs or Node's C++ layer — which perform slow work off to the side and hand back a callback when it is done.[19]

Those callbacks wait in queues. Callbacks from promises land in a high-priority microtask queue that is drained completely on every turn; callbacks from timers, I/O, and UI events wait in the macrotask queue, from which the loop takes one per turn.[19] Whenever the call stack is empty, the event loop pulls the next piece of work and runs it. This is why JavaScript is described as non-blocking: instead of parking a thread while it waits, it registers interest and moves on.

Programming this model directly with nested callbacks became notorious as “callback hell.” ES6 promises flattened it, and the async/await syntax of ES2017 made asynchronous code read like ordinary sequential code while preserving the non-blocking behaviour underneath.[7]

load-user.js
// async/await reads top-to-bottom yet never blocks the thread.
async function loadUser(id) {
  const res = await fetch(`/api/users/${id}`);   // yields to the loop
  if (!res.ok) throw new Error(`HTTP ${res.status}`);
  const user = await res.json();
  return user.name;
}

loadUser(42)
  .then(name => console.log(name))
  .catch(err => console.error(err));

The engines that make it fast

For its first decade JavaScript was slow, and that changed abruptly in 2008 when Google shipped V8, the engine at the heart of Chrome.[20] Modern engines are sophisticated virtual machines written in C++. V8 parses source to bytecode run by an interpreter called Ignition, then watches which functions run often (“hot” code) and progressively recompiles them through faster tiers — Sparkplug, Maglev, and the optimizing TurboFan compiler — producing machine code that can approach native speed. A concurrent garbage collector reclaims memory in the background.[21]

V8 is not alone. Mozilla's SpiderMonkey — a direct descendant of the original engine Eich wrote — powers Firefox with its own multi-tier compilers; Apple's JavaScriptCore drives Safari and can embed JavaScript inside native apps; and Meta's Hermes optimizes for fast startup on mobile.[20][21] The competition between these engines during the “browser wars” is a large part of why JavaScript became fast enough to build serious software in.

// beyond the browser

Runtimes: JavaScript escapes the page

The second event that changed JavaScript's trajectory was Node.js. In 2009 Ryan Dahl took Google's V8 engine and wrapped it in a runtime for the server, betting that JavaScript's event-driven, non-blocking model was ideal for scalable network programs.[11][22] The bet paid off. Node let developers write both client and server in one language, and it brought with it npm, a package registry that has grown into the largest software registry in existence, with well over two million packages.[23][24] Node remains the default: the State of JS 2025 survey put its share of backend runtime use around 90 %, and it underpins production systems at Netflix, Uber, PayPal, and LinkedIn.[24][11] Recent Node releases have added first-class ES modules, a built-in test runner, a permissions API, and native TypeScript execution; the current Node 24 “Krypton” line carries long-term support into 2028.[13][24]

Node's very success invited successors. In 2018 Dahl announced Deno — an anagram of “Node” — as an explicit correction of what he regretted about his first project.[13] Built on V8 and written in Rust, Deno runs in a permission sandbox with no file, network, or environment access unless you grant it, ships with TypeScript support built in, and includes a formatter, linter, and test runner out of the box.[23][14] In 2022 Jarred Sumner released Bun, which chases raw speed instead: it is built on Apple's JavaScriptCore engine and written in the systems language Zig, and it bundles a runtime, package manager, bundler, and test runner into one binary whose installs and startups are several times faster than Node's.[14][22] The three now define the runtime landscape — Node for maturity and ecosystem, Deno for security and standards, Bun for speed — while State of JS 2025 still shows Node far in front, with Bun and Deno as fast-growing challengers.[24]

// where it stands

How popular, really?

“Most popular language” has no single answer, because the major rankings measure different things — and JavaScript's position shifts depending on which ruler you pick.

by developer usageJavaScript — #1

The 2025 Stack Overflow survey, which asks what people actually work in, puts JavaScript first at 66 %, ahead of HTML/CSS, SQL, and Python. It has topped this list every year since 2015.[3][25][4]

by search interestPython — #1, JS #6

The TIOBE index, which tracks search-engine mentions, ranked Python first in July 2026 with JavaScript sixth, behind C, C++, Java, and C#.[26][27]

by code on GitHubTypeScript — #1

In August 2025 TypeScript — typed JavaScript — overtook Python and plain JavaScript to become the most-used language on GitHub by monthly contributors, GitHub's biggest language shift in over a decade.[15][28]

the family viewJS + TS dominate the web

Counting TypeScript as the JavaScript dialect it is, the JavaScript family leads by every reasonable measure of general software activity, even as Python owns AI and data science.[15]

The GitHub milestone is worth dwelling on. TypeScript closed August 2025 with about 2.64 million monthly contributors, having added over a million in a single year — a 66 % jump — and GitHub tied the surge directly to AI-assisted coding, since typed code lets tools and compilers catch machine-generated errors earlier, and most modern frameworks now scaffold new projects in TypeScript by default.[28][15] TypeScript is not a rival to JavaScript but an extension of it; its rise is JavaScript's rise, wearing types. Meanwhile the two most-admired languages in the 2025 survey were Rust and its peers, a reminder that admiration and usage are not the same thing.[3]

// notable products

What is built with it

JavaScript's footprint reaches well past web pages. The clearest evidence is Electron, a framework that packages a web app — HTML, CSS, and JavaScript running on Node and Chromium — into a cross-platform desktop application.[29] The list of software built this way is striking: Microsoft's Visual Studio Code, the most popular code editor in the world and itself written largely in TypeScript; the chat clients Slack, Discord, and Signal; productivity tools Notion, Obsidian, Trello, and Loom; developer tools GitHub Desktop, Postman, and Docker Desktop; and, fittingly for the current moment, the desktop apps for both OpenAI's ChatGPT and Anthropic's Claude.[29][30]

Serious Electron apps typically adopt a hybrid architecture: the interface is JavaScript, while performance- or security-critical cores are written in another language — VS Code uses a specialized text data structure and V8 startup snapshots, Discord runs voice processing in C++, and Signal keeps its cryptography in Rust.[31] JavaScript, in other words, is the connective tissue and the interface layer even where it is not doing the heavy lifting. On the server, Node powers the backends of Netflix, PayPal, Uber, and LinkedIn; in the browser it is, of course, universal.[11] It is difficult to spend a day on a computer without running software whose primary language is JavaScript.

// notable libraries

The libraries

JavaScript's ecosystem is defined as much by its libraries as by the language. The most historically important is jQuery, released by John Resig at a BarCamp event in early 2006 to paper over the maddening inconsistencies between browsers of the era.[32][33] With a concise, CSS-like syntax for selecting and manipulating page elements, handling events, and making Ajax requests, it became — and, remarkably, remains — the most deployed JavaScript library by a wide margin, still present on roughly three-quarters of the top ten million websites.[33][34] Many of the conveniences it pioneered were later absorbed into the browser itself.

Beyond jQuery, a handful of libraries recur across the industry:

data visualizationD3.js

Released in 2011 by Mike Bostock, Jeff Heer, and Vadim Ogievetsky, D3 (“Data-Driven Documents”) binds data to the DOM and SVG to build custom, interactive charts, from simple bar charts to complex scientific plots.[35][36]

3D graphicsThree.js

A widely used wrapper over WebGL that makes real-time 3D rendering in the browser approachable, powering everything from product configurators to browser games.[35]

utilitiesLodash & friends

Lodash and its predecessor Underscore provide consistent helpers for arrays, objects, and functions; libraries like Axios (HTTP) and Socket.IO (real-time messaging) fill similarly universal roles.[35][36]

charts & motionChart.js, GSAP

Chart.js offers ready-made charts on the canvas, while animation libraries such as GSAP drive the high-end motion seen on marketing and portfolio sites.[35]

// notable frameworks

The frameworks

If libraries are tools, frameworks are opinions about how to structure an entire application. The dominant one is React, released by Facebook (now Meta) in 2013, which popularized building interfaces from composable components and reconciling changes through a virtual representation of the DOM.[37] It is, strictly, a library rather than a full framework, but it anchors the largest ecosystem in front-end development and consistently leads usage — the 2025 Stack Overflow survey placed it first among web frameworks, used by roughly 68 % of respondents.[38]

Around it sit a set of well-defined alternatives, each expressing a different philosophy:

AngularGoogleA complete, TypeScript-first framework for large enterprise apps; powerful but with a steep learning curve. [37][38]

Layered on top of these are meta-frameworks that add server-side rendering, routing, and build tooling: Next.js for React, Nuxt for Vue, SvelteKit for Svelte, plus Remix and the islands-oriented Astro.[40][39] On the server, JavaScript frameworks are equally established — the minimalist Express remains the reference point, with the TypeScript-first NestJS and the performance-focused Fastify as common alternatives.[11]

// integrations

How it connects to everything else

Part of JavaScript's reach comes from how well it interoperates with other technologies. Four integrations matter most.

TypeScript: types without leaving JavaScript

TypeScript, first released by Microsoft on 1 October 2012 and led by Anders Hejlsberg — the architect of C# — is a strict syntactic superset of JavaScript: every valid JavaScript program is valid TypeScript.[12][41] It adds optional static types that are checked at build time and then erased, compiling down to ordinary JavaScript that runs in any browser or runtime.[12] Hejlsberg's motivation was blunt — teams were routinely building hundred-thousand-line applications in a language whose loose typing made large codebases hard to maintain.[17] More than a decade on, that argument has been decisively won: TypeScript is the way most large JavaScript projects are now written, and, as noted above, it became the single most-used language on GitHub in 2025.[15]

WebAssembly: a partner for heavy lifting

WebAssembly (Wasm), a low-level binary format that runs at near-native speed, is a compilation target for languages like C, C++, and Rust and is explicitly designed to run alongside JavaScript rather than replace it.[42][43] Through the WebAssembly JavaScript API, a JavaScript application loads a Wasm module, calls its exported functions, and shares memory with it, so the two exchange work: JavaScript keeps the interface, the DOM, and the glue, while Wasm handles compute-heavy tasks such as image filtering, video processing, or 3D physics.[42] It is the standard way to bring desktop-class performance into a web app without giving up JavaScript's flexibility.

The package ecosystem and the toolchain

The npm registry and its lockfile-driven workflow tie the whole ecosystem together, and the build tooling around it — bundlers like webpack, Vite, esbuild, and Rollup, and package managers like npm, Yarn, and pnpm — is what turns modern source into something a browser can load.[23][40] A notable recent trend is that much of this performance-critical tooling is itself being rewritten in Rust for speed, even as it serves JavaScript developers.[13]

Desktop, mobile, and the edge

Beyond Electron on the desktop, JavaScript reaches mobile through React Native and similar frameworks that render native interfaces from JavaScript code, and it increasingly runs at the network edge, where lightweight runtimes execute JavaScript close to users for low-latency serverless functions.[20][44] The same language spans browser, server, desktop, phone, and edge — a span no other mainstream language matches.

// the other side

The famous rough edges

No account of JavaScript is complete without its criticisms, most of which trace back to the same root: the refusal to break old code. Its type coercion is the stuff of conference talks — adding an array to an object, or subtracting a string from a number, produces results that are technically defined and practically baffling. These behaviours cannot be fixed without breaking existing pages, so they remain, and the ecosystem routes around them with linters, strict mode, and TypeScript.[8]

The other recurring critique is the dependency model. The ease of publishing to npm and pulling in packages has produced deep, sprawling dependency trees, which are convenient until a single small package breaks or is compromised — a structural risk that the security-first design of newer runtimes like Deno was created to address.[13][23] The through-line of the last decade is that the community has chosen to manage JavaScript's looseness with tooling rather than to replace the language — a pragmatic bet that the reach is worth the rough edges.

// the road ahead

What comes next

JavaScript now evolves through TC39's disciplined, yearly process, and the near-term additions are more about smoothing sharp corners than adding grand new paradigms — exactly the philosophy that emerged from the ES4 collapse. The headline arrival is Temporal, which reached Stage 4 in March 2026 and is bound for ECMAScript 2026: a modern, immutable replacement for the language's error-prone Date object, with proper time-zone and calendar support, retiring the need for heavy third-party date libraries.[16] Also landing are decorators and explicit resource management via using declarations, part of a broader push to make JavaScript a more “batteries-included” language whose standard library covers what developers once reached for libraries to do.[45][46]

Further out, a Signals proposal aims to standardize the fine-grained reactive state model that frameworks like Solid, Svelte, and Angular already implement independently, so that reactivity becomes a shared language primitive rather than a per-framework invention.[46] A long-running pattern matching proposal would bring expressive, structural branching of the kind found in Rust and Scala.[47] Not everything survives the process — the once-anticipated Records and Tuples proposal for immutable value types was withdrawn in 2025 after the committee could not reach consensus, a healthy sign that the gate still works.[48]

The largest force now shaping the language is not a syntax proposal at all but AI-assisted development, which has rewarded the reliability of typed code and helped push the JavaScript family further ahead in the count of code actually being written.[28] Thirty years after a ten-day prototype, a language that was never meant to be important is being written, checked, and generated at a scale its creator could not have imagined — and it is still, by the most direct measure, the language the world reaches for first.[3][15]
