.text:00401000 Orientation
What a thick client is // no WAF here

0x00The program is the perimeter

A thick client — also called a fat client, rich client, or simply a desktop application — is software installed on and executed by the user's own machine, interacting with local resources, the operating system, and frequently a remote backend. Unlike a thin client, it retains substantial functionality whether or not it is connected to a network.1 Banking platforms, trading terminals, healthcare systems, industrial engineering suites, and enterprise line-of-business tools are overwhelmingly thick clients, and they routinely handle an organisation's most sensitive data and most critical operations.2

That distinction is not academic. When you assess a web application, the attacker sits on the far side of a network boundary you can instrument, throttle, and shield with a web application firewall. A thick client inverts every assumption. The code runs on the attacker's hardware, which means an adversary can attach a debugger to the running process, dump its memory to harvest secrets, modify the binary on disk before execution, and reverse engineer the entire codebase offline — and no perimeter device will ever see it happen.2 Applying a web-application playbook to a desktop binary misses most of the attack surface, because the surface is fundamentally different: local file storage, the registry, inter-process communication channels, native code defects, insecure update mechanisms, weak local cryptography, and the binary itself are all in play.2

Two tiers or three

Architecture dictates attack vectors, so the first question in any engagement is how the application is layered.2 In a two-tier design, the client communicates directly with a database server; the data tier is reachable from the endpoint, which is inherently less secure because the end user can talk straight to the database.3 In a three-tier design, the client talks to an application server, which in turn brokers access to the database — most processing moves server-side, and the user is prevented from communicating directly with the data store, materially reducing risk.4 A two-tier client that ships a database connection string inside its own binary is the canonical thick-client finding, and the rest of this guide will return to it repeatedly.

Proxy-aware versus proxy-unaware

The second orienting question is whether you can see the traffic. A proxy-aware thick client has built-in support for an HTTP proxy — clients such as collaboration suites, mail clients, and file-sync tools fall here, and interception with an intercepting proxy is straightforward.4 A proxy-unaware client offers no proxy configuration of its own, and frequently speaks a non-HTTP or custom binary protocol, forcing the tester to reach for transparent redirection, traffic-capture tooling, or runtime instrumentation instead. Section 0x04 is devoted to the techniques that pry these clients open.

Mental model

Treat a thick-client engagement as four overlapping investigations that feed one another: what the binary contains (reverse engineering), what it says on the wire (protocol analysis), what it lets a local user become (privilege escalation), and whether its own defences hold (anti-tamper and memory-corruption review). Findings in one almost always unlock the next.

A consensus methodology already exists

You are not improvising. The community has converged on a small set of reference documents that purpose-built thick-client testing around. The OWASP Thick Client Application Security Verification Standard (TASVS) establishes an open standard for designing, building, and testing thick-client applications, explicitly filling the gap between OWASP's web-focused ASVS and its Mobile ASVS — the latter can be applied to desktop software but is not an ideal fit.5 Complementing it, the OWASP Desktop App Security Top 10 is a consensus awareness document enumerating the most critical risks to desktop applications across Windows and *nix, derived from publicly known exploits and CVEs.6 Practitioner checklists aligned to both — covering reconnaissance through exploitation and reporting — package the workflow into eighty-plus concrete test cases.78 The remainder of this article is structured along that methodology, deepened with the reverse-engineering and exploitation craft that distinguishes a real assessment from an automated scan.

Mapping what it touches // enumerate every entry point

0x01The surface a desktop binary exposes

Before a single byte is decompiled, the surface is mapped. A thick client touches far more of the host than its window suggests, and each contact point is an entry point: network connections, local file storage, registry keys, inter-process communication channels, and hardware interactions all need to be documented.2

The local execution model is precisely what makes this both rewarding and complex. There is no WAF between the attacker and the application logic — debuggers attach directly to running processes, memory is dumped and searched, and binaries are reverse engineered entirely offline.7 A useful reconnaissance pass walks the application's behaviour with a tool such as Sysinternals Process Monitor and notes everything it does: which files it reads and writes, which registry keys it queries, which named pipes or sockets it opens, and what it loads at startup.8 The reward for this patience is concrete. A classic walkthrough of a deliberately vulnerable client shows Process Monitor revealing that the application stores and retrieves user credentials directly in the registry — an immediate data-storage finding that surfaces only because the tester observed the program rather than merely running it.9

The ten failure modes worth memorising

The OWASP Desktop App Security Top 10 gives the engagement its risk vocabulary. Map every finding to it; reviewers unfamiliar with thick-client testing may not immediately grasp why a disabled UI button or a world-writable install directory is critical, and the mapping supplies the missing context.7

Where the secrets actually hide

Across real assessments the same locations yield repeatedly. Configuration files frequently hold cleartext credentials or, worse, "encrypted" values protected by trivially recoverable schemes; database connection strings live in config or are reconstructed in memory; cryptographic and licensing keys sit in the binary itself; and insecure local storage leaks across the board.2 The remediation direction is settled: hardcoding credentials, API keys, or encryption keys into application binaries exposes a major attack surface that attackers extract through reverse engineering or memory dumping, so secrets belong in OS-provided secure storage — DPAPI on Windows, Keychain on macOS — rather than in the application's own files.10 A purpose-built vulnerable target, the Damn Vulnerable Thick Client Application, was deliberately seeded with the exact catalogue a tester should expect to find: insecure local data storage, insecure logging, weak cryptography, lack of code obfuscation, exposed decryption logic, SQL injection, CSV injection, sensitive data in memory, DLL hijacking, cleartext data in transit, and client-side protection bypasses achieved through reverse engineering.11 Each of those is a section heading in the work that follows.

Field note — the disabled button

A control greyed out in the UI is not a security boundary. Thick-client GUIs routinely hide privileged functionality behind disabled or invisible form objects, and re-enabling them — or invoking the underlying handler directly — is a standard early test.8 If the only thing standing between a user and an administrative function is button.Enabled = false, that function is already reachable.

.text:00402310 Reconnaissance
Fingerprinting the binary // know the runtime first

0x02Information gathering & binary triage

Everything downstream depends on correctly identifying what you are holding. The very first task is to determine the runtime: is this native machine code, a .NET assembly, a Java application, an Electron bundle, or something packed? The answer decides whether you reach for a decompiler that recovers near-source, or a disassembler that gives you assembly.

The triage toolkit is small and stable. A PE-inspection utility — CFF Explorer, PE-bear, or Detect It Easy (DIE) — reveals file type, headers, imports, the compiler or packer signature, and the assembly type; Strings dumps embedded ASCII and Unicode literals; and the Sysinternals Suite (Process Monitor, Process Explorer) reveals runtime behaviour.8 Walking a real target, an analyst opens the executable in CFF Explorer, reads off that the assembly type is a .NET assembly, and on that basis switches to a managed-code decompiler — the single decision that determines the entire static-analysis path.9

The information-gathering checklist

  1. Architecture. Two-tier or three-tier? The presence of a direct database connection changes the threat model entirely.8
  2. Technology stack. Languages and frameworks in use — native C/C++, .NET, Java, Delphi, or a web stack wrapped in Electron.8
  3. Network communication. Identify every endpoint the client contacts; a quick capture in Wireshark on the relevant interface reveals the remote hosts and whether traffic is plaintext.9
  4. Process and filesystem activity. Run Process Monitor with a filter on the target and catalogue the files, registry keys, and pipes it touches — this is where credential-in-registry and config-file findings first appear.9
  5. Entry points and trust boundaries. Every input, every IPC channel, every place the app reads data it did not generate.8
  6. Authentication model. Local credentials, domain authentication, smart cards, biometrics; online and offline behaviour.8

Two configuration artefacts deserve special scrutiny because they are where the cheapest wins live. The application's own folder frequently contains a .config file or equivalent holding connection parameters, and running the program once while Process Monitor watches reveals exactly which keys and files it reaches for.9 When a secret is not on disk in cleartext, it is often recoverable from process memory: a tool such as Process Hacker can be filtered to extract the live memory of the target, and in documented assessments that memory yields a cleartext database connection string that a tester then uses directly against the backend with an ordinary SQL client.3

Triage decision tree

Managed (.NET / Java) → decompiles to near-original source; reach for a managed decompiler and read it like code. Native (C/C++/Delphi) → disassemble and decompile to pseudo-C; expect to annotate. Packed / obfuscated → identify the protector first (Section 0x06), because nothing readable appears until the outer layer is peeled.

Where the secrets actually live

Part of triage is knowing where a thick client is likely to have hidden the things worth finding, because the storage locations are predictable across the platform. Configuration files sitting beside the executable — XML, INI, JSON, or a bundled local database — are the first stop, and they routinely hold connection strings, endpoints, and credentials in plaintext or trivially-reversible encoding.4 Beyond the file system, the Windows registry under the user and machine hives is a common store for application settings and secrets, and the methodology treats registry inspection as a standard reconnaissance step rather than an afterthought.3 The more interesting cases involve platform secret stores: secrets protected with the Windows Data Protection API (DPAPI) — and on macOS, items held in the Keychain — are encrypted, but because the protection is tied to the user's own context, an attacker operating as that user can frequently decrypt them, which is exactly why a "DPAPI-protected" secret is not the end of the analysis but the start of it.10 Finally, the process's own memory is a store in its own right: whatever the client decrypts to use, it must hold in cleartext at some point, so a memory capture taken while the application is unlocked often yields the very secret the on-disk encryption was meant to protect.3 An assessor enumerates all four tiers — files, registry, platform key stores, and live memory — before concluding that a secret is safely handled.

.text:00403C80 Reverse engineering
Static & dynamic analysis // asm → meaning

0x03Reverse engineering the client

Reverse engineering is the deconstruction of a program to understand its inner workings, algorithms, and data structures.1 Because a thick client uses compiled binaries that hide logic and secrets, the assessor applies decompilation to analyse code paths, inspect DLLs, and uncover hardcoded credentials or weak encryption — and the same work reveals how the client talks to its backend.2

Two axes: static vs dynamic, disassembly vs decompilation

The discipline splits along two axes. Static analysis examines the program without running it; dynamic analysis observes it live, attaching a debugger to watch registers, memory, and control flow as the program executes. Independently, a disassembler converts raw machine code into assembly language, while a decompiler lifts that assembly into high-level, C-like pseudocode — and the latter is what makes modern reverse engineering tractable, because it spares the analyst from reading raw assembly line by line.1 Real work braids all four together: you decompile statically to form a hypothesis, then confirm it dynamically in a debugger.

Managed code: the gift of near-source recovery

When the binary is a .NET assembly or Java application, the assessor is fortunate. The compilers for these platforms emit intermediate representations — .NET CIL, Java bytecode — that decompile back to something very close to the original source. A .NET decompiler such as dnSpy or .NET Reflector, or JD-GUI for Java, often renders the program in fully readable form.2 A documented assessment of a managed thick client opens the executable in dnSpy and finds the decompiled output unobfuscated, exposing FTP credentials hardcoded directly in a button-click handler and the decryption logic for a "protected" database password sitting in a bundled DLL — every secret legible without a single instruction of assembly.11

Native code: Ghidra and IDA Pro

When the binary is native C, C++, or Delphi, the two dominant platforms are Ghidra and IDA Pro, and a serious practitioner is fluent in both. Ghidra is the software reverse-engineering framework released publicly by the United States National Security Agency at the RSA Conference in 2019 under the Apache 2.0 licence; its capabilities include disassembly, assembly, decompilation, graphing, and scripting, across a wide range of processor instruction sets and executable formats.12 It is written in Java with a Swing GUI, its decompiler component is implemented in C++, and analysts automate it with scripts written in Java or Python via Jython.13 Its decisive feature for our purposes is a decompiler that turns unfamiliar assembly into annotatable pseudo-C; with version 11.3 in 2025 Ghidra added an integrated debugger that connects to GDB, LLDB, and WinDbg, putting static and dynamic analysis in a single view and cutting context-switching.14

IDA Pro, from Hex-Rays, is the long-standing commercial standard, an interactive multi-processor disassembler and static-analysis platform whose Hex-Rays decompiler transforms machine code into readable C-like pseudocode generated on the fly in seconds, freeing the analyst from learning assembly for routine work.15 IDA supports more than sixty processor architectures by default and offers specialised decompilers across a dozen hardware platforms, with variable names and types editable on the fly and changes propagating between the disassembly and pseudocode views.16 Founded in 2005 by Ilfak Guilfanov, Hex-Rays equips IDA with technologies that matter directly to thick-client work: FLIRT (Fast Library Identification and Recognition Technology) recognises statically linked library functions so you do not waste hours reversing strcpy, and the bundled gooMBA plug-in simplifies obfuscated Mixed Boolean-Arithmetic expressions by bridging to an SMT solver.17

What you are hunting for in the listing

Hardcoded credentials, API keys, encryption keys, and connection strings; authentication logic that can be inverted; licensing and key-derivation routines; hidden or backdoor functionality; and the cryptographic primitives the client trusts. Static analysis surfaces the constants and the logic; dynamic analysis confirms which path actually executes.2

The deeper question this section raises — and the next one answers — is what happens when the client speaks to its backend. Decompilation tells you the protocol exists and where it is constructed; protocol analysis tells you what crosses the wire.

0x04Protocol analysis & traffic interception

Thick clients are not isolated; they communicate. The most important single task in many engagements is intercepting the requests the application fires, and how you do it depends entirely on whether the client cooperates.3

The cooperative case: proxy-aware clients

A proxy-aware client exposes proxy settings of its own, which makes interception with an intercepting proxy such as Burp Suite or OWASP ZAP straightforward — point the client at the proxy and read, replay, and tamper with HTTP/HTTPS at will.4 Standard API testing applies here: authentication bypass, broken authorisation, injection, and excessive data exposure are all in scope, and the recurring findings are hardcoded API keys, weak encryption, cleartext credentials in transit, and replayable requests.3

The stubborn case: proxy-unaware and custom protocols

Many thick clients offer no proxy configuration and may not speak HTTP at all, using a custom or binary protocol over raw TCP.3 Here the tester escalates through a ladder of techniques: transparent redirection at the host level (hosts-file or routing tricks that send the client's traffic to your listener), full packet capture with Wireshark on the relevant adapter to read whatever the client actually emits, and — when transport encryption stands in the way — runtime instrumentation to defeat it from inside the process.9 Wireshark frequently settles the most damaging question outright: a capture of a client's session reveals, for example, that its file-transfer channel is plaintext, exposing credentials and data because the protocol is unencrypted.9

Certificate validation and the TLS soft spot

Transport security is only as strong as the client's willingness to verify it. A core protocol test asks whether the application accepts self-signed certificates and whether a man-in-the-middle position can therefore be established — many thick clients skip proper TLS validation, and where sensitive data is transmitted in clear text, where session tokens are poorly randomised, or where requests can be replayed, the protocol layer is exposed.3

Frida: instrumenting the running process

When a client pins its certificate or buries its protocol behind native crypto, dynamic instrumentation is the lever. Frida is a dynamic instrumentation toolkit for developers, reverse engineers, and security researchers that injects your own scripts into black-box processes: you can hook any function, spy on crypto APIs, or trace private application code with no source code, editing and reloading scripts without recompiling or restarting the target, and it runs on Windows, macOS, Linux, iOS, and Android.18 Created by Ole André V. Ravnås and supported by NowSecure, its core is written in C and injects a JavaScript engine into the target process, where scripts run with full access to memory, hook functions, and call native functions over a bidirectional channel.19 In practice this means a tester hooks the function that validates a certificate and forces it to return success, or hooks the routine that encrypts an outbound buffer and reads the plaintext before it is sealed — restoring visibility into a channel the developer thought was opaque.

Interception ladder — escalate in order

1. Configure the client's own proxy → Burp/ZAP. 2. No proxy support → transparent redirection + Wireshark capture. 3. Custom/binary protocol → decode from the capture, cross-referenced with the decompiled construction code from Section 0x03. 4. Pinned TLS or native crypto → Frida hooks on the validation or crypto routine.

The local attack surface beyond the wire

Not all of a thick client's communication leaves the host, and the channels that stay local are exactly the ones a network capture never sees. Desktop applications routinely split themselves into a privileged background service and an unprivileged user-facing front end, and the two halves talk over local inter-process communication — named pipes, loopback-bound TCP or UDP sockets, shared memory, or COM/DCOM and RPC endpoints. Each of these is attack surface in its own right, and the security misconfiguration category in the desktop risk taxonomy calls out misconfigured named pipes and services specifically as a source of serious weakness.6 The dangerous pattern is a trust assumption: a high-privileged helper that exposes a named pipe or local socket and then acts on whatever the low-privileged GUI sends it, without authenticating the caller or validating the request. An attacker who can reach that endpoint — and any local user often can — may be able to invoke privileged operations, inject commands, or escalate, turning an internal convenience into a local-privilege-escalation primitive.3 A complete assessment therefore enumerates the client's local listeners and IPC objects alongside its network traffic, because the most damaging thick-client findings frequently live on channels that never touch the network at all.1

.text:00406A40 Privilege escalation
SYSTEM from a user shell // T1574 — hijack exec flow

0x05Local privilege escalation

Installed software runs somewhere in the privilege hierarchy, and when it runs higher than the user who can influence it, the gap is a ladder. Local privilege escalation in thick clients almost always reduces to one MITRE ATT&CK technique family: Hijack Execution Flow (T1574) — abusing how the operating system locates the programs and libraries a process runs.20

DLL hijacking and side-loading

Dynamic-link libraries are shared code loaded by multiple programs, and they are abused through search-order hijacking, side-loading, and phantom DLL hijacking.21 The mechanics that matter to a thick-client assessor are simple: if a privileged application loads a DLL by name from a directory the current user can write to, and the search order reaches that directory before the legitimate one, a planted DLL executes inside the privileged process — and if that application runs at a higher privilege level, the attacker-controlled library runs there too, which is precisely how the technique becomes privilege escalation rather than mere persistence.21 The deliberately vulnerable DVTA client demonstrates the full chain: an attacker with a low-privileged foothold finds DVTA running with administrative rights, identifies a missing DLL the application searches for using Process Monitor, drops a malicious DLL of that name into the application's directory, and gains an elevated shell when the application next loads it.22

Unquoted service paths

When a Windows service is registered with an executable path that contains spaces and is not wrapped in quotation marks, the path becomes ambiguous, and an adversary can place an executable in a higher-level directory so that Windows launches it instead of the intended binary.23 The canonical example: a service whose path is C:\Program Files\Sub Dir\service.exe can be hijacked by planting C:\Program.exe, and because services frequently run as SYSTEM, exploiting an unquoted service path yields SYSTEM privileges.24

Weak service permissions — file and registry

Two related weaknesses round out the family. Where the permissions on a service's binary are weak, an adversary simply replaces the executable that runs at service start with their own.25 Where the permissions on the service's registry keys are weak, the adversary repoints the service at their own program — and on restart, the attacker's code executes in the account the service runs under, whether that is a local or domain account, SYSTEM, LocalService, or NetworkService.26 The same registry tree offers further leverage: a failure-command key can be set to run in an elevated context whenever the service fails, and a performance key can be made to point at a malicious DLL.26

Why install location is a security control

A thick client installed into a world-writable directory and run with elevated rights collapses several of these techniques into one trivial finding. If any user can write to the folder a SYSTEM-level service or an admin-run application loads code from, that user can become SYSTEM. Auditing the application's install path, service configuration, and DLL search behaviour with Process Monitor and an ACL checker is therefore not optional — it is the core of the local-privilege-escalation pass.22

The tooling here is light and well known: Process Monitor to observe what the application loads and from where, an access-control checker to find writable paths and weak service ACLs, and privilege-enumeration scripts to surface unquoted paths and misconfigured services automatically. The judgement is in recognising that a greyed-out button, a missing DLL, and a writable install folder are not separate curiosities but the rungs of a single escalation.

.text:00408D20 Licensing & anti-tamper
Client-side trust review // trust the server, not the host

0x06License logic & software-protection review

This is where thick-client testing confronts a structural truth: any check a program makes about itself, on a machine the user controls, is a check the user can rewrite. Assessing license and activation logic is not about defeating a vendor's revenue model; it is about measuring how much trust a product places in the endpoint, and advising where that trust belongs instead.

Scope & legality first

Everything in this section presumes explicit written authorisation to test the software in question. Anti-tamper and licensing review is performed against your own product, a client's product under contract, or a target inside an agreed scope — never as a route to piracy. The legal frame is discussed at the close of this section and again under standards in Section 0x09.

The anti-pattern: client-side enforcement

The hero of this article is a license check that returns true when a key equals a constant, evaluated entirely on the client. It is not a strawman. The DVTA project includes, by design, "client-side protection bypasses using reverse engineering" precisely because they are among the most common real-world thick-client weaknesses its author encountered.11 The failure mode generalises: when authentication, entitlement, or feature-gating decisions are made locally — an offline validation routine, a key checked against a locally derived value, a feature flag toggled by a registry entry — an analyst who can decompile the binary can locate the deciding branch and understand exactly how it works.3 A decompiler renders the logic legible; from there the conceptual distance to a bypass is short, which is the entire point a security assessment must make to the developer.

The remediation is architectural, not cosmetic. Decisions that must not be subverted belong on a server the user does not control, with the client treated as untrusted input. The assessor's deliverable is rarely "your obfuscation is weak"; it is "this entitlement decision is enforced in a place the user owns, and must move server-side." That reframing — from client-side trust to server-side authority — is the most valuable sentence in many thick-client reports.

What protection mechanisms actually buy

Vendors do defend their binaries, and recognising the defences is part of triage: is the binary packed, obfuscated, equipped with anti-debugging checks, or anti-tampering measures? Understanding the protections is the prerequisite to assessing them.3 The techniques form a layered repertoire. Code obfuscation renames symbols, reorders blocks, encrypts strings, and inserts junk to frustrate both static and dynamic analysis.27 Anti-tampering uses checksums and digital signatures to detect modification, and may detect virtualised or sandboxed execution as a proxy for analysis.27 Anti-debugging detects and obstructs debuggers — Windows exposes signals such as the NtGlobalFlag field that protected code inspects — to make analysis slower and costlier.28 And packers and protectors add layers of encryption and anti-analysis on top of the program, with heavyweight commercial protectors such as VMProtect and Themida going further into code virtualisation.29

The crucial framing for both attacker and defender is that none of this is absolute. The goal of packing and obfuscation is not to make reverse engineering impossible but to make it so time-consuming and expensive that it is not worth the effort, forcing the analyst to peel back layers of protection before understanding can even begin.29 Protection raises cost; it does not change the fundamental fact that code which executes on the user's machine can, with enough effort, be understood and altered. A security assessment quantifies that cost honestly — high enough to deter casual abuse, never high enough to substitute for server-side enforcement of anything that truly matters.

The legal boundary every assessor must know

In the United States, the anti-circumvention provisions of the Digital Millennium Copyright Act, codified at Section 1201 of the Copyright Act, prohibit circumventing technological measures that protect access to copyrighted works.30 Critically for security work, Section 1201 contains a set of permanent exemptions — for reverse engineering, encryption research, and good-faith security testing among them — though each applies only within the specific circumstances the statute defines, and a triennial rulemaking can add temporary exemptions.31 The boundary that trips researchers is overlap: good-faith security research that qualifies under Section 1201 may nonetheless incur liability under other laws, including the Computer Fraud and Abuse Act, so qualifying for the DMCA exemption does not, by itself, exempt a researcher from the CFAA or other applicable law.32 The practical rule follows directly: authorisation, scope, and a documented rules-of-engagement are not bureaucratic overhead — they are what keeps lawful security testing lawful.

.text:0040B500 Binary exploitation
Memory corruption // EIP is a suggestion

0x07Binary exploitation

Native thick clients are written in memory-unsafe languages, and the deepest class of finding is the one where malformed input does not just confuse the program but seizes its control flow. Memory-corruption review is the most technically demanding part of the discipline, and it is the part automated scanners cannot reach.

The bug classes

The family is familiar to anyone who has studied low-level security. The classic stack-based buffer overflow overwrites a function's saved return address by writing past a fixed-size buffer, redirecting execution when the function returns.33 Beyond it lie Structured Exception Handler (SEH) overwrites, heap corruption in its many forms — use-after-free, double-free, type confusion, uninitialised memory — format-string defects, and integer overflows that lead to undersized allocations. A serious exploit-development curriculum treats exactly this spread: in-depth vulnerability analysis and exploitation in Windows user-mode applications, custom exploit development for stack, heap, and integer overflows as well as format-string and use-after-free vulnerabilities.34

The mitigations between a bug and an exploit

Modern operating systems and compilers stack several defensive layers, and understanding them is what separates "the program crashed" from "the program is exploitable." Stack canaries place a known value before the saved return address; if a buffer overflow overwrites it, the program detects the change and aborts before returning.35 Data Execution Prevention (DEP / NX) marks data regions such as the stack and heap non-executable, so injected shellcode cannot simply run from the buffer it landed in.36 Address Space Layout Randomization (ASLR) randomises the base addresses of the stack, heap, libraries, and — with position-independent executables — the binary itself, so attacks that depend on hardcoded addresses fail.35 These are most effective in combination: even if one layer is bypassed, such as a canary defeated through an information leak, the others can still prevent full exploitation.35

How exploitation adapts: ROP and information leaks

The defining technique of modern exploitation is the response to DEP. Because injected code cannot execute, attackers turn to Return-Oriented Programming (ROP), a code-reuse method that chains together short instruction sequences — "gadgets," each ending in a ret — already present in the program's own address space, bypassing the no-execute bit by reusing executable code rather than injecting new code.35 A ROP chain is used precisely to bypass Data Execution Prevention, and where the program lacks the needed instructions in its own image, a return-into-libc variant reuses code from linked libraries instead.36 ASLR raises the difficulty further by randomising where those gadgets live — but a single byte of leaked address information can be enough to defeat it, which is why memory-disclosure primitives are so prized: the modern exploit first leaks an address to unmask the layout, then builds its ROP chain against the now-known target.35

This is no longer a curiosity of the past. Although buffer overflows were discovered decades ago, they persist in legacy codebases, embedded systems, and performance-critical software where developers disable safety checks, and the bypass techniques have kept pace with the defences.33 A formal study of a vulnerable native binary illustrates the reasoning a tester applies: observing that NX is enabled but ASLR is not, the analyst concludes the overflow is exploitable only via a ROP chain to achieve code execution, and reasons through how an address leak would be required where library randomisation is in force.36 That chain of reasoning — bug, then mitigation inventory, then the specific bypass each mitigation demands — is the core competency.

Mitigation ↔ bypass, at a glance

Stack canary → leak it, or corrupt around it via a non-contiguous write. DEP / NX → ROP / return-to-libc, reusing existing executable code. ASLR / PIE → an information leak to unmask the base address before building the chain. Control-Flow Guard → find an indirect-call path it does not cover. No single mitigation is sufficient alone, and the assessment's value is in identifying which layers a given client actually ships — and which it forgot.35

This work has a recognised home in the certification landscape, and the toolchain that supports it — the disassembler, the debugger, and the gadget finder — is the subject of the next section. The point worth carrying forward is that a thick-client memory-corruption finding is rarely a one-line crash report; it is a demonstration that input crosses a trust boundary and, through a chain of mitigation bypasses, reaches control of execution.

.text:0040D000 The toolchain
Disassemble · debug · instrument // three views of one binary

0x08The toolchain

A thick-client assessment is only as good as the analyst's ability to see what the binary is doing — statically, as a frozen listing, and dynamically, as a running process. Three categories of tool provide that sight: the disassembler-decompiler that turns machine code back into something readable, the debugger that stops the program mid-execution to inspect and alter its state, and the instrumentation framework that rewrites behaviour on the fly. The three named in nearly every methodology — Ghidra, IDA Pro, and x64dbg — map exactly onto the first two categories, with Frida covering the third.

The decompilers: Ghidra and IDA Pro

The two reference disassemblers sit at opposite ends of the cost spectrum and yet converge on the same workflow. Ghidra is the free, open-source software reverse-engineering suite released by the United States National Security Agency in 2019, built in Java with a multi-platform GUI and, crucially, a decompiler that lifts disassembly to readable C-like pseudocode across a wide range of processor architectures.12 Its 2019 public release was a turning point precisely because a capable decompiler — historically the most expensive component of the reverse-engineering toolkit — became available at no cost, and recent versions have folded in an integrated debugger that bridges the long-standing gap between Ghidra's static analysis and the dynamic inspection that previously required a separate tool.14 Its extensibility through Java and Python scripting, and a collaborative multi-user mode, made it a fixture in both training and professional practice almost immediately.13

IDA Pro, from Hex-Rays, is the long-standing commercial standard: an interactive disassembler and debugger supporting more than fifty processor families, paired with the Hex-Rays Decompiler that converts native machine code into a readable C-like representation and dramatically accelerates analysis of large or unfamiliar binaries.15 The decompiler is the feature most analysts pay for, transforming the slow work of reading assembly into the far faster work of reading reconstructed source.16 Around it Hex-Rays has built decades of refinements — FLIRT signatures that automatically recognise and label statically-linked library functions so the analyst is not re-deriving printf by hand, and more recent additions aimed at seeing through compiler optimisation and obfuscation — which together explain why, despite Ghidra's arrival, IDA Pro retains its place in many shops.17

The debugger: x64dbg

Where the decompiler reads the program at rest, the debugger watches it run. x64dbg is an open-source x64/x32 debugger for Windows, purpose-built for user-mode binary analysis and distributed under a permissive licence — the project that, more than any other, gave the Windows reversing community a free, modern, actively-developed alternative to the aging commercial debuggers.37 It ships as a unified package: x32dbg and x64dbg for 32- and 64-bit targets, launched through the x96dbg selector, with a familiar multi-pane interface showing disassembly, registers, the stack, and memory simultaneously.38 Its capability comes from the libraries it integrates — a Qt-based interface, the TitanEngine debugging core, the Zydis disassembler engine, Yara for pattern scanning, Scylla for import reconstruction during unpacking, and an analysis engine for control-flow recovery — and it exposes an assembler-like scripting language and a documented plugin API, so the community has built extensions for everything from automated unpacking to deobfuscation.37 For a thick-client tester, x64dbg is where a license check is observed actually executing, where a patched jump is tested live before it is written to disk, and where the value in a register at the moment of a comparison settles a question the static listing left ambiguous.

Static and dynamic are not rivals — they are two passes

The decompiler tells you what the program is supposed to do; the debugger tells you what it actually does with this input, on this machine, right now. Anti-debugging defeats the second; obfuscation defeats the first; packing defeats both until unpacked. A real engagement moves constantly between Ghidra or IDA Pro open on one monitor and x64dbg paused on a breakpoint on the other — a decompiled hypothesis confirmed or refuted by a live register value, then back again.

The instrumentation layer and the supporting cast

Between static and dynamic analysis sits dynamic instrumentation. Frida is a dynamic instrumentation toolkit that injects a JavaScript engine into a running native process, letting the tester hook arbitrary functions, read and rewrite arguments and return values, and trace execution without source code or a recompile.18 For thick clients it is the standard answer to certificate pinning and custom cryptography, hooking the validation or encryption routine in memory so traffic can be read or a check neutralised.19 Surrounding these headline tools is a supporting cast no assessment omits: dnSpy, .NET Reflector, and JD-GUI for managed-code decompilation;2 the Sysinternals utilities — Process Monitor for file, registry, and process telemetry and Process Explorer for live handle and DLL inspection — for runtime reconnaissance and for catching the missing-DLL and weak-permission conditions that drive privilege escalation;3 a triage layer of CFF Explorer, PE-bear, Detect It Easy, and Strings for header parsing and packer identification;10 WinDbg and Process Hacker for deep kernel-aware debugging and memory extraction;4 and the network pair Burp Suite, OWASP ZAP, and Wireshark for the protocol-analysis pass.1

.text:0040E800 Standards & compliance
What governs the work // methodology is defensible

0x09Standards, frameworks & compliance

Ad-hoc poking does not survive a client's legal review or a regulator's audit. A professional thick-client assessment is anchored to published standards on two axes at once: a subject-matter standard that says what to test in a desktop application, and a process standard that says how a penetration test is structured, evidenced, and reported. Knowing both is what turns a collection of findings into a defensible engagement.

The subject-matter standards: OWASP for thick clients

The reference catalogue specific to this work is OWASP's. The Thick Client Application Security Verification Standard (TASVS) provides a structured, requirement-by-requirement basis for verifying the security of desktop applications, the direct analogue for thick clients of the more familiar standards OWASP maintains for web and mobile.5 Alongside it, the OWASP Desktop App Security Top 10 enumerates the most critical risk categories — from injection and broken authentication through sensitive-data exposure, improper cryptography, weak authorization, and insufficient logging — giving an assessment a shared, citable vocabulary for classifying what it finds.6 Used together, TASVS supplies the depth of a verification checklist and the Top 10 supplies the breadth of a risk taxonomy, and a report that maps each finding to both is immediately legible to any security-literate reader.

The process standards: how a test is run

The methodology that frames the engagement comes from a small set of widely-recognised standards. The U.S. National Institute of Standards and Technology's SP 800-115, the Technical Guide to Information Security Testing and Assessment, structures a security test into four phases — planning, discovery, attack, and reporting — a sequence general enough to govern a thick-client engagement end to end.39 Complementing it, OWASP catalogues the established penetration-testing methodologies the industry recognises — among them the Penetration Testing Execution Standard (PTES), the Open Source Security Testing Methodology Manual (OSSTMM), and the NIST guidance itself — each offering a repeatable structure that makes an assessment auditable rather than improvised.40 The adversary-behaviour layer is supplied by MITRE ATT&CK, whose technique taxonomy — the same one that catalogues the Hijack Execution Flow family used in the privilege-escalation section — lets a tester describe what an attacker would do in a vocabulary defenders already use, tying a local finding to a named, documented technique.20

Where regulation makes it mandatory

For software handling payment-card data, penetration testing is not advisory but contractual. Under PCI DSS 4.0, the penetration-testing requirement was renumbered to Requirement 11.4 — relocated from the previous 11.3 — and its expectations became enforceable for assessments after the standard's transition deadline.41 The standard does not merely require that a test happen; it requires that the test follow a documented methodology grounded in an industry-accepted approach such as NIST SP 800-115, the OWASP guidance, or PTES, and that testing include genuine manual exploitation rather than an unverified automated scan — a bar that a thoroughly reverse-engineered, manually-exploited thick-client finding meets squarely.41 The same requirement is where tester qualification enters: organisations are expected to use suitably skilled testers, and recognised certifications such as OSCP, CREST, GPEN, and others are the customary evidence of that competence.41

Scoring and the legal envelope

However the testing is framed, findings are communicated in a common currency of severity. The Common Vulnerability Scoring System (CVSS), in its v3.1 and newer v4.0 revisions, provides the standard numeric basis for rating and prioritising each issue, so that a thick-client report's severities are comparable with those from any other assessment a client receives.7 And enclosing all of it is the legal envelope examined in Section 0x06: the DMCA's Section 1201 anti-circumvention rules with their security-research and reverse-engineering exemptions,30 the permanent and triennial exemption structure that defines exactly when circumvention for good-faith research is permitted,31 and the overlap with the Computer Fraud and Abuse Act that means a documented authorization and rules-of-engagement are the foundation on which every other standard rests.32

From findings to a report that drives remediation

Every framework converges on the same final act: the report. In the SP 800-115 structure, reporting is the fourth and closing phase, the stage where raw discovery becomes a documented, prioritised, actionable record rather than a pile of notes.39 For a thick-client engagement that means each finding carries a CVSS severity so the client can triage consistently against everything else on their plate, a reproduction path precise enough that a developer can trigger the issue unaided, the evidence that demonstrates impact, and a concrete remediation mapped to the relevant OWASP category.7 Where the work falls under a regulatory regime, the report must also satisfy that regime's evidentiary bar — a payment-card assessment, for instance, has to show that genuine manual exploitation was performed and that the methodology followed a recognised standard, not merely that a scanner was run.41 A report that ties each finding to a severity, a reproduction, a fix, and the standard or regulation that frames it is what converts a technically impressive engagement into organisational change — which is, in the end, the only outcome that matters.

The compliance stack, bottom to top

Authorization & scope (the legal floor — DMCA 1201 exemptions, CFAA boundaries, signed rules-of-engagement) → process methodology (NIST SP 800-115, PTES, OSSTMM) → subject-matter standard (OWASP TASVS, Desktop App Security Top 10) → adversary mapping (MITRE ATT&CK) → severity scoring (CVSS v3.1/v4.0) → regulatory driver where applicable (PCI DSS 4.0 Req 11.4). A defensible engagement can point to each layer by name.

.text:0040FC00 Building capability
Certs · training · labs · channels // the long apprenticeship

0x0ABuilding capability

No single course produces a thick-client tester. The competence is assembled from overlapping disciplines — reverse engineering, exploit development, Windows internals, and protocol analysis — each with its own canonical training, certification, and practice ground. What follows is the established path through them, from the certifications that signal the skill to the free channels where the craft is actually taught.

Certifications worth the name

The certification that maps most directly onto the deepest part of this work is OffSec's OSED — Offensive Security Exploit Developer — earned through the course EXP-301, Windows User Mode Exploit Development. Its exam is a punishing, proctored hands-on assessment: roughly forty-eight hours to solve three exploit-development challenges and produce a professional report, with no multiple-choice in sight.42 OSED is one of the three certifications that compose OffSec's OSCE³ expert designation, alongside the web-focused OSWE and the penetration-focused OSEP, and unlike many credentials it does not expire — a reflection of the fact that the underlying skill, not a snapshot of current tooling, is what it certifies.42 The curriculum behind it is the same one referenced earlier: in-depth analysis and custom exploitation of stack, heap, integer-overflow, format-string, and use-after-free vulnerabilities in Windows user-mode software.34

For the reverse-engineering and malware-analysis dimension — the skills that carry over directly to understanding a hostile or heavily-protected binary — the benchmark is GIAC's GREM (GIAC Reverse Engineering Malware). Its exam is hands-on in the most literal sense, incorporating a CyberLive component that requires candidates to perform real analysis tasks in a live environment rather than answer questions about them, and its pass rate reflects the difficulty of that format.43 Underpinning both, and assumed by most employers as the entry ticket to offensive work generally, is OffSec's OSCP — not thick-client-specific, but the credential whose presence (alongside CREST, GPEN, and CEH) is the customary proof of tester competence that compliance regimes like PCI DSS expect.41

Where the skills are taught

Behind the certifications sit the courses. OffSec's EXP-301 is the structured route into Windows exploit development, building from disassembly and debugging through the mitigation-bypass techniques — DEP, ASLR, and Control-Flow Guard evasion, SEH-based exploitation, and shellcode creation — that define modern memory-corruption work.34 For reverse engineering and malware analysis, SANS FOR610, Reverse-Engineering Malware, is the long-standing reference: a multi-day course built around hands-on labs using a curated toolset including Ghidra, that teaches the systematic examination of a hostile binary's behaviour and internals.44 And the deepest specialist training in exploit development comes from Corelan, run by Peter Van Eeckhoutte, whose Bootcamp, Advanced, and heap-exploitation classes are widely regarded as the most thorough available — the same lineage that produced the mona.py plugin and a large body of free, openly-published exploit-development tutorials that have trained a generation of researchers.45 For those building toward these on a smaller budget, more accessible practical-malware and ethical-hacking courses from providers such as TCM Security cover the foundational ground before the specialist material.44

Labs: where it is practised

Training without a target is theory. The purpose-built practice application for this exact discipline is the Damn Vulnerable Thick Client App (DVTA), a deliberately insecure two-tier .NET client created to let testers rehearse the full methodology — insecure storage, traffic interception, DLL hijacking, and more — against a safe, legal target.11 Worked end-to-end walkthroughs of DVTA, demonstrating exactly how each class of flaw is found and exploited, are freely available and form the standard first project for anyone entering the field.9 Beyond it, the broader practice grounds are the same ones the offensive community uses for every discipline: crackmes for pure reverse-engineering and license-logic puzzles, and platforms such as Hack The Box, TryHackMe, and pwn.college for graded binary-exploitation and reversing challenges that scale from first principles to professional difficulty.34

The free curriculum: channels and communities

Much of the best instruction in this field is published free on video. OALabs (Open Analysis) maintains an extensive library of practical reverse-engineering and malware-analysis tutorials, walking through real tools and real samples at a level that bridges beginner and professional.46 LiveOverflow offers one of the most approachable on-ramps to binary exploitation and reverse engineering, building intuition for memory corruption and CTF-style problem-solving from the ground up.47 And curated roundups of the wider scene point to the rest of the essential channels — stacksmashing, John Hammond, and other reverse-engineering educators — so that a newcomer can assemble a complete self-taught syllabus without spending a dollar before deciding to invest in formal training.48

A realistic progression

Start free: LiveOverflow and OALabs for intuition, crackmes and DVTA for hands-on reps. Build foundations: OSCP for offensive fundamentals, FOR610 or TCM material for reverse engineering. Specialise: EXP-301/OSED for Windows exploit development, GREM for malware analysis, Corelan for the deepest exploit work. Practise continuously: Hack The Box, TryHackMe, and pwn.college keep the skill from decaying. The discipline rewards patience — it is an apprenticeship measured in years, not a weekend certification.

Best practices, for both sides of the table

The assessment's value is realised only in what changes afterward, and the recommendations divide cleanly. For developers, the recurring lessons are architectural: never trust the client for a decision that matters, so move licensing, authorization, and entitlement enforcement to a server the user does not control; never embed secrets — credentials, keys, connection strings — in the binary or its configuration, because everything shipped to the user is readable; encrypt sensitive data at rest using platform key stores rather than home-grown schemes; validate all input crossing a trust boundary, including inter-process and local network channels; install into protected directories with correct ACLs, quote service paths, and sign code to close the privilege-escalation and DLL-hijacking avenues; and keep bundled third-party components current. For testers, the discipline is method: map the full attack surface before touching exploitation, work the standards in parallel so every finding is classified and scored consistently, treat static and dynamic analysis as two passes of one investigation, prove impact rather than assert it, and — above all — keep authorization and scope documented, because the line between security research and a computer-crime allegation is the paperwork signed before the work began.

.text:00410000 Return
The thesis, restated // ret

0x0BThe program is the perimeter

Every technique in this article reduces to a single inversion of the web tester's assumption. There is no server enforcing the rules at a distance; the logic, the secrets, and the defences are all sitting on a machine the adversary fully owns — and what executes on the user's machine can, with enough effort, be read and rewritten.

That is why the work moves between two views of the same binary. There is the opaque face — the hex, the opcodes, the packed and obfuscated layers a vendor hopes will hold — and the recovered meaning the analyst pulls back out of it with a decompiler, a debugger, and an instrumentation hook. The license check that returns true on a constant, the FTP password sitting in a button handler, the unquoted service path that yields SYSTEM, the missing DLL planted into a writable directory, the overflow that survives every mitigation but one — each is a place where the gap between those two faces became a finding.

The honest conclusion is the one that should reach the client most clearly: client-side protection raises the cost of an attack, and that is worth doing, but it never substitutes for enforcing anything that truly matters in a place the user does not control. The thick-client assessor's most valuable deliverable is rarely a clever exploit. It is the sentence that moves a trust decision off the user's machine and onto the server where it belongs — translated, like everything else in this discipline, from the opaque listing into recovered, actionable meaning.
