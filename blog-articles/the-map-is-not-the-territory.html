The Map Is Not the Territory

Why every engineered system can be hacked — and why “unhackable” is not a hard problem but a category error.

00 — Prologue

A System Is a Model

Every system we build is a model, and a model is always a partial, frozen abstraction of something far more fluid than itself. A login prompt, a firewall, a biometric scanner, a legal identity document — each takes some set of measurable properties and treats them as standing in for an entity. Passwords stand in for an authorized user; fingerprints for a body; packet headers for an intention. The model, however, is not the thing. It is a map, and as Alfred Korzybski put it in the founding text of general semantics, the map is not the territory.1

A map is useful precisely because it omits. It selects, quantizes, and freezes a few aspects of a world that is continuous, high-dimensional, and in motion. That selective blindness is not a defect to be engineered away; it is what makes the map small enough to use. But the same property that makes a model tractable makes it exploitable, because everything the model discarded still exists in the territory, fully available to anyone who cares to read it.

This is the whole of hacking, stated abstractly: it is the systematic exploitation of the difference between the map a system maintains and the territory the system claims to govern. A password is a string; the “authorized user” is a biological, psychological, and social phenomenon. Obtain the string and you have not become the user — you have hijacked the model’s proxy for the user, which is all the model ever consults. A fingerprint reader sees ridges, not a person; reproduce the ridges and you are, to the reader, the person.

The argument of this essay is that the “unhackable system” is not a distant engineering milestone we have yet to reach. It is incoherent in roughly the way a map identical to its territory is incoherent — such a map would simply be the territory again, and would secure nothing. Three results, none of them from security engineering, converge on this conclusion: the lossiness of abstraction, the non-existence of a fixed identity, and the impossibility of certain justification. Taken together they show that the attack surface is not an artifact of sloppy code but a permanent feature of representation itself. What follows traces each thread, then returns to what security can actually be once invulnerability is off the table.

01 — Abstraction

The Abstraction Gap

To represent something is to choose which of its effectively infinite properties to record and which to throw away. The discarded remainder does not vanish. It stays in the territory, and the territory is where the adversary lives. The recurring shape of a vulnerability is therefore simple: an interaction the model has no vocabulary to describe, occurring in the part of reality the model declined to represent.

A canonical early software demonstration is the buffer overflow. A program models its input as “a value of a certain type that fits in a certain space.” The machine beneath the program models something coarser and more honest: bytes, addresses, and the electrical state of memory cells. When the supplied input exceeds the space the program imagined, the surplus bytes do not politely disappear — they are written into memory the program assumed it owned, including the very addresses that determine what the processor will execute next.2 The gap between the bounded type a programmer pictures and the unbounded stream a CPU will faithfully copy is a place an attacker can stand.

handler.c/* The function models its input as "a name that fits in 64 bytes."
   The machine models bytes. Reality is under no obligation to agree. */
void greet_user(char *input) {
    char name[64];
    strcpy(name, input);   /* no bound check: bytes past 64 fall outside the model */
    print_greeting(name);
}

The deeper the layer, the more it has elected not to see. Consider memory itself. Software models RAM as an array of independent, stable cells; device physics models DRAM as a dense field of leaky capacitors that influence their neighbors. Repeatedly hammering one row of cells can flip bits in the physically adjacent rows — data corrupted in a location the program never touched, through an electrical interaction the logical model simply has no rule to forbid. When the effect was first characterized in detail, more than eighty percent of the commodity modules tested, drawn from all three major vendors, proved vulnerable.3 No line of the program was wrong. The program’s model of memory was wrong, in the specific sense of being a map that omitted the physics.

Timing tells the same story one level up. A processor’s architectural model promises that instructions executed speculatively and then discarded leave no observable trace. The microarchitecture underneath does not honor that promise: speculation leaves measurable residue in shared caches, and an attacker who can time memory accesses can read it. The Spectre class of attacks induces a victim to perform, speculatively, operations that would never occur during correct execution, then exfiltrates the secret through a channel the instruction-set specification refuses to acknowledge exists.4 Once more the breach is not in the logic but in the difference between the clean abstraction and the messy substrate it was abstracted from.

Every reduction creates an attack surface, because the surface is exactly what the reduction threw away.The general form of the abstraction gap
02 — Identity

Identity Without Essence

Authentication asks the oldest question security knows — who or what is this? — and in asking it presumes that there is a stable “who” to be found and pinned down. Philosophy has been undermining that presumption for two and a half thousand years.

Heraclitus held that you cannot step into the same river twice: the self, like the river, is a process rather than an object, the same in name only across its perpetual replacement.5 The Buddhist doctrine of anattā, non-self, denies that any enduring essence underlies the changing aggregates of experience; what we call a person is a bundle of momentary, causally linked states with no owner behind them.6 Hume, looking inward for the self, reported finding only a bundle of perceptions in ceaseless flux and no impression of a subject doing the perceiving.7 Derek Parfit pressed the point into rigorous form: personal identity over time consists in nothing beyond physical and psychological continuity — there is no further fact, no deep additional ingredient, that makes a later person be an earlier one.8

A security system cannot operate on a process that has no essence. It must reify. Your identity becomes a database row, a private key, a behavioral baseline — a snapshot of attributes treated as if it were persistent and intrinsic. The system then checks new evidence against that snapshot. But a snapshot is a description, and descriptions can be copied.

auth.rs// To the system, "you" is the truth-value of a single comparison.
fn authenticate(supplied: Secret, on_file: Hash) -> Verdict {
    if hash(supplied) == on_file {
        Verdict::You      // the model now believes the referent is present
    } else {
        Verdict::NotYou
    }
}

The fingerprint sensor makes the reduction physical: it models a finger as a pattern of ridges, and a pattern of ridges can be cast in gelatin. When a cryptographer methodically tested this, artificial “gummy” fingers moulded from a few dollars of household gelatin were accepted by all eleven commercial fingerprint devices in the study at least some of the time, with reported acceptance probabilities ranging from more than 67% to 100% — across optical and capacitive systems, including some that may have employed so-called live-and-well detection.9 Nothing was “broken.” The sensor did exactly what it was built to do: it confirmed the presence of the ridges it models as standing in for a person.

This is the general form of every authentication failure that is not a software bug. To authenticate is to check for the signs of an identity, never the identity itself, which is not a fact in the world that can be exhaustively inspected. The stolen credential, the cloned hardware token, the replayed biometric, the deepfaked voice on the phone — each produces the signs without being the referent, and the model, which only ever consulted the signs, is satisfied exactly. Being you is not something a system can verify; it can only verify a narrative you maintain, and narratives can be performed by others.

03 — Epistemology

The Regress of Trust

Grant, for argument’s sake, both a thing to verify and a snapshot to check it against. Verification still rests on prior knowledge — a shared secret, a trusted authority, a measured characteristic — and that knowledge rests on further assumptions, which rest on assumptions of their own. Security inherits, in full, epistemology’s unfinished business with certainty.

Edmund Gettier showed in three pages that even justified true belief can fail to be knowledge: one can hold a belief that is true, and be rationally entitled to it, and still not know, because the justification is connected to the truth only by luck.10 Every verdict of “this request is authenticated” is precisely such a belief — justified by a credential, quite possibly true, and possibly true only by the accident that the adversary has not yet forged the credential in question. The system’s confidence and the system’s correctness are different things, and nothing internal to the system can tell them apart.

Push the demand for justification all the way down and you arrive at an argument the ancient skeptics already knew as Agrippa’s trilemma, revived in the twentieth century by Hans Albert as the Münchhausen trilemma.11 Any attempt at ultimate justification must end in one of three unacceptable places: an infinite regress of reasons, a circle in which the conclusion justifies its own premise, or a dogmatic stopping point asserted without further ground.12 Security’s “root of trust” is the dogmatic horn, given a name and often anchored in hardware: a key we simply decide to trust, because the regress has to be cut somewhere and the hardware is a common place to cut it.

A root of trust is a regress we have agreed to stop questioning.On the Münchhausen trilemma in security

Ken Thompson made the regress unforgettably concrete in his Turing Award lecture. A compiler can be taught to insert a backdoor into the system’s login program — and, crucially, to insert that very teaching into any compiler it subsequently compiles. The malicious source can then be deleted entirely. You may audit every line of source code in your possession and still inherit the betrayal, because you did not write the compiler that compiled your compiler, nor the one that compiled that. His conclusion was that you cannot fully trust code you did not create yourself13 — and no one creates all of it. Modern supply-chain compromise is not an aberration from this principle; it is the principle arriving on schedule. Trust is always delegated, and every delegation is an assumption an adversary may have reached first.

04 — Ontology

The Closed World and the Open World

An engineered system operates by imposing a closed, formalized world: an explicitly modeled set of states, a catalogue of rules, and a defined space of permitted transitions. It can respond only to inputs that its ontology is prepared to treat as inputs. This is not a weakness of particular systems — it is what it means to be a system at all. Raymond Reiter gave the idea its canonical statement as the closed-world assumption: whatever is not known to be true is taken to be false.14 That assumption is what makes inference tractable and what makes the world, from inside the system, appear finite and complete.

Reality is not closed. The space of possible interactions is not enumerable in advance, and an attacker wins not by making a wrong move within the game but by making a move the game has no rule for. Even the most rigorous adversary models pay for their rigor by drawing the boundary explicitly — and thereby inheriting it. The Dolev–Yao model, the foundation of formal protocol analysis, grants the attacker total control of the network while stipulating that the cryptographic primitives themselves cannot be broken.15 The proofs built on it are entirely real, but they secure the model’s world. An attack that violates the model’s assumptions — a leaked key, a timing side channel, a user persuaded to approve the transaction — is not a counterexample to the theorem. It is simply outside the theorem.

Nor can formal verification be stretched to cover everything, even in principle. Turing’s halting problem and Rice’s theorem together establish that, for any non-trivial semantic property of program behavior, no algorithm can decide whether arbitrary programs have that property.1617 We can prove specific properties of specific, constrained systems, and that work is invaluable. What we cannot build is a general oracle that certifies an arbitrary system safe. The set of behaviors a model declares “known” is bounded by the model; the universe of behaviors a real system can exhibit is not; and the asymmetry between them is mathematical rather than budgetary. No amount of engineering effort closes it.

A recent illustration is also one of the most literal. Large language models embedded in applications erase the very boundary the closed world depends on — the line between data to be processed and instructions to be obeyed. Researchers demonstrated that text placed in content the model will later read — a web page, a shared document, an incoming email — can hijack the surrounding system, because the model has no reliable ontological category separating “the world I am reading about” from “the commands I must follow.”18 Indirect prompt injection is the closed-world assumption failing in natural language: an input the designers’ ontology never contained, interpreted by the system as one it did.

05 — Learned Models

The Statistical Shadow

Machine-learned models make the thesis vivid because they never even pretended to inspect reality — only a statistical projection of it. Szegedy and colleagues discovered that an imperceptible, carefully chosen perturbation could flip a well-trained network’s classification, and, more disquietingly, that these “adversarial examples” transfer: an example crafted against one network often fools another trained on different data. The blind spots are systematic, not random — properties of the represented space rather than accidents of a single training run.19 Goodfellow and colleagues explained the phenomenon as a consequence of the model’s near-linearity in very high dimensions and produced adversarial examples in closed form.20

x′ = x + ε · sign( ∇x J(θ, x, y) )   such that   ‖x′ − x‖∞ ≤ ε,   yet   f(x′) ≠ f(x)

A perturbation too small for a person to notice, aligned with the model’s own gradient, is enough to move an input across a decision boundary — without moving it, perceptibly, anywhere in the territory.

Eykholt and colleagues then carried the attack out of the dataset and onto the road. A few unremarkable black-and-white stickers placed on a real stop sign produced targeted misclassification in one hundred percent of lab images and in 84.8% of video frames captured from a moving vehicle.21 To any human in the territory the sign still plainly reads STOP. The classifier, reading the statistical shadow it mistook for the sign, sees a speed-limit sign instead.

The moral generalizes well beyond machine learning. The classifier’s “stop sign” was always a region in a feature space, never the painted metal on the post. Every system — learned or hand-written, digital or physical — only ever inspects a representation that is under-specified relative to the actual situation. Which is to say there is always a way to make the model see something that is not there, and always a way to make it miss something that is.

06 — Synthesis

Why the Attack Surface Is Permanent

Assemble the three threads. From abstraction: every model is a reduction, and the reduction’s residue is an attack surface that cannot be removed without removing the abstraction itself. From identity: there is no fixed essence available to authenticate, only signs, and signs can be reproduced. From epistemology: there is no certain justification, only a regress halted by an assumption, and assumptions can be mistaken or subverted.

The conclusions stack. If no verification can be absolute, then no authentication can be conclusive. If identity has no intrinsic essence, then no credential can be more than a forgeable description. If reality is always richer than the model that represents it, then there will always be a way to make the model misperceive. These are not three separate engineering shortfalls awaiting three separate fixes. They are three faces of one fact: a representation is, by definition, less than the thing it represents.

This is why the correct statement is not “we have not yet built the unhackable system.” A model that discarded nothing would not be a model; it would be a duplicate of the world, and securing it would be no easier than securing the world itself. The unhackable system has the same shape as a map identical to its territory — and a map identical to its territory is not a map. The impossibility is structural and therefore permanent. New systems do not shrink the gap between model and world toward zero. They relocate it. Close the overflow and the side channel opens; harden the credential and the human operator becomes the target; formalize the protocol and the assumptions the formalization required become the new frontier.

The unhackable system is not far away. It is the same shape as a map identical to its territory — and that is not a map.The category error
07 — Practice

Security as Gap Management

If invulnerability is incoherent, the discipline needs a different aim, and in practice it already has one. Security is not the pursuit of a perfect model; it is the continuous management of the distance between the model and the modeled — narrowing that distance where it is dangerous, bounding the damage where it cannot be closed, and adapting each time an adversary reveals where it has reopened.

The classic design principles, read in this light, are quietly a confession that the gap is permanent. Saltzer and Schroeder’s catalogue reads less like a recipe for safety than like a discipline for living with a model you know to be incomplete: economy of mechanism, because a smaller model leaves less residue to exploit; fail-safe defaults, because when the model is uncertain it should deny rather than permit; complete mediation, because every past verdict is a stale snapshot that must be re-checked; least privilege and separation of privilege, because any single misreading should be survivable; and the principle of open design, because the secrecy of a model is no substitute for its soundness.22 Each is a way of taking the gap seriously rather than pretending it away.

The contemporary posture states the epistemic humility outright. “Assume breach” and zero-trust architectures abandon the fantasy of a perimeter that, once crossed, can be trusted forever. They treat every request as unproven, because identity is only ever a snapshot and trust is only ever a delegated assumption; they re-verify continuously and minimize what any single verification is allowed to grant. This is not pessimism. It is belief calibrated to what can actually be known.

Social engineering remains the purest demonstration of the whole argument, and the reason the human layer so often decides the outcome. The attacker need not break a single cryptographic primitive when the “knowledge” a person acts on — a familiar-sounding voice, a plausible address, an urgent and authoritative tone — is itself a fallible sign of an intent that cannot be directly inspected.23 A comprehensive survey of the field is, at bottom, a catalogue of such gaps and of the partial, provisional, perpetually revised measures that keep each one small enough to live with.24 That catalogue is never finished, and its incompleteness is not a sign of failure. It is the discipline behaving exactly as a discipline aimed at a moving, unclosable gap should behave.

The map is not, and never can be, the territory. A system inspects the map; an adversary walks the territory; and the difference between them is where hacks keep finding room to happen.

Security, properly understood, is the engineering practice that takes this seriously — not by closing the gap, which is impossible, but by watching it, narrowing it where it is dangerous, bounding what falls through it, and revising the map each time the world reminds us, again, of everything it left out.
