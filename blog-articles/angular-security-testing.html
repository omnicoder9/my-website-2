§ 01 — Orientation

Why an Angular app still needs testing

Angular is, by design, one of the safer ways to render untrusted data in a browser. Its template engine treats every interpolated value as untrusted and sanitizes or escapes it before it reaches the DOM.1 But that built-in safety covers only one layer of the stack — and it can be deliberately switched off.

The official guidance is explicit that the framework's protections stop at the rendering boundary: Angular's security guide covers cross-site scripting and a small set of HTTP-level helpers, but states plainly that it does not address application-level concerns such as authentication and authorization.1 Those concerns — together with session handling, API design, and file handling — are exactly where the most damaging modern vulnerabilities live. Broken Access Control sits at the top of the OWASP Top 10:2021, having moved up from fifth place; the contributing data showed access-control weaknesses in a larger share of applications than any other category.23

A second reason for testing is that Angular's own escape hatches reintroduce risk. The DomSanitizer.bypassSecurityTrust* family of methods exists so that developers can render trusted HTML, URLs, or resource URLs — but if the input flowing into them is attacker-controlled, the application is back to a classic XSS vulnerability.45 Direct DOM access via ElementRef.nativeElement, server-side template construction, and missing Content-Security-Policy headers all open the same kinds of gaps.15

The framework treats all values as untrusted by default — but the moment you mark something trusted, the responsibility for that decision moves entirely onto you. — Paraphrasing Angular's security model1

This manual organizes Angular security testing into three complementary layers. Static analysis reads the source code without running it, catching dangerous patterns at author time. Dynamic testing attacks the running application as a black box. Penetration testing applies human judgement to the areas automated tools handle poorly — business logic, authorization boundaries, and chained exploits. Each layer finds bugs the others miss, which is why mature programs run all three.

§ 02 — Foundations

What Angular protects, and what it leaves to you

Before testing, it helps to know the framework's baseline so you can tell a real finding from expected behaviour. Angular's protections cluster into three areas.

Contextual sanitization and security contexts

When a value is inserted into the DOM through a binding or interpolation, Angular sanitizes and escapes it. Sanitization is context-dependent: a value that is harmless in CSS can be dangerous in a URL. The framework defines four security contexts — HTML, Style, URL, and Resource URL — and sanitizes untrusted values for HTML and URLs. Resource URLs cannot be sanitized because they load and execute code, so they require explicit trust.1

component template — interpolation vs. innerHTML
<!-- Interpolated content is always escaped: angle brackets render as text -->
<p>{{ htmlSnippet }}</p>

<!-- Bound to innerHTML: Angular sanitizes, stripping <script> but keeping <b> -->
<p [innerHTML]="htmlSnippet"></p>

// htmlSnippet = 'Template <script>alert("0wned")</script> <b>Syntax</b>'
// Angular recognizes the value as unsafe and removes the script element.

This example is drawn from the framework's own documentation: binding attacker-controllable data into innerHTML would normally cause XSS, and Angular's automatic sanitization is what neutralizes it.1

HTTP-level helpers: XSRF and XSSI

Angular's HttpClient implements the client half of a cookie-to-header CSRF defense. An interceptor reads a token from a cookie named XSRF-TOKEN by default and echoes it in an X-XSRF-TOKEN header. Because only code on your own origin can read that cookie, the backend can be confident the request came from your application. The header is sent on mutating requests such as POST — but not on GET or HEAD — and the server must be configured to set the cookie and verify the header, or the protection is inert.1

For cross-site script inclusion (XSSI, the "JSON vulnerability"), HttpClient automatically strips the well-known non-executable prefix )]}',\n from responses, provided the server prepends it.1

Defense in depth: CSP, Trusted Types, and AOT

Two web-platform features operate below Angular at the DOM level, where they cannot be bypassed by lower-level APIs: a Content Security Policy and Trusted Types enforcement. Angular strongly encourages both. The framework also relies on the Ahead-of-Time (AOT) compiler in production, which eliminates a whole class of template-injection vulnerabilities by refusing to compile templates from concatenated user input at runtime.1

Audit target Any call to bypassSecurityTrustHtml, bypassSecurityTrustScript, bypassSecurityTrustStyle, bypassSecurityTrustUrl, or bypassSecurityTrustResourceUrl is flagged in Angular's docs as security-sensitive and must be reviewed by hand. These are the first grep targets in any Angular security review.1
§ 03 — Layer one

Static analysis: catching bugs before they run

Static Application Security Testing (SAST) inspects source code without executing it, looking for vulnerabilities, misconfigurations, and insecure patterns before the application ever starts.9 For an Angular codebase, SAST is most valuable for spotting the framework's escape hatches (the bypassSecurityTrust* calls), insecure storage of tokens, dangerous DOM access, and weak cryptographic primitives — all of which are visible in the code itself.

Two open-source tools dominate the Angular/TypeScript SAST space and complement each other well: ESLint with security plugins, which lives inside the developer's existing lint step, and Semgrep, which offers far richer pattern and dataflow matching across many languages. The sections below cover both.

§ 04 — SAST tool A

ESLint security rules

Angular projects already lint with ESLint (via angular-eslint), which makes ESLint the lowest-friction place to add security checks — they run on every save and every commit with no new tooling. The community-maintained eslint-plugin-security adds 14 rules that flag potential security hotspots in JavaScript and Node.js code, including code injection, unsafe regular expressions, file-system vulnerabilities, and timing-attack patterns.67

Representative rules

RuleWhat it detectsWhy it matters in Angular
detect-eval-with-expressionDynamic eval() on a non-literal argumentDynamic code execution is a direct injection vector
detect-unsafe-regexRegexes prone to catastrophic backtrackingReDoS can hang the event loop and the UI thread6
detect-possible-timing-attacksSequential comparison of secrets with === / ==Token/HMAC comparison should be constant-time6
detect-pseudoRandomBytesUse of non-cryptographic randomnessPredictable IDs/tokens; prefer the Web Crypto API6

A companion plugin, eslint-plugin-security-node, adds rules such as detect-insecure-randomness, which flags Math.random() used to generate tokens, IDs, or UUIDs. Math.random() is cryptographically insecure and can produce predictable values; the recommended replacement is window.crypto.getRandomValues(), which yields cryptographically strong random numbers.8

eslint.config.js — flat config (ESLint ≥ 8.23 / Angular)
const eslint = require("@eslint/js");
const tseslint = require("typescript-eslint");
const angular = require("angular-eslint");
const security = require("eslint-plugin-security");

// eslint-plugin-security ships a flat-config "recommended" preset.
module.exports = tseslint.config(
  {
    files: ["**/*.ts"],
    extends: [
      eslint.configs.recommended,
      ...tseslint.configs.recommended,
      ...angular.configs.tsRecommended,
      security.configs.recommended,   // turns on the 14 security rules
    ],
    processor: angular.processInlineTemplates,
    rules: {
      // project-specific hardening
      "security/detect-possible-timing-attacks": "error",
    },
  },
  {
    files: ["**/*.html"],
    extends: [...angular.configs.templateRecommended],
  },
);

The plugin is added under devDependencies and integrates with the existing ESLint run; for the flat-config format introduced in modern ESLint, the recommended preset is referenced directly as shown.67

Known limitation ESLint security plugins operate on single-file syntax patterns, so they generate false positives and miss vulnerabilities that span files or require dataflow tracking. Independent benchmarking of 17 ESLint security plugins against real-world vulnerable patterns found wide detection gaps, with most plugins missing a substantial share of cases. Treat ESLint as a fast first pass, not a complete SAST solution.6
§ 05 — SAST tool B

Semgrep

Semgrep — a portmanteau of "semantic" and "grep" — is an open-source static-analysis engine that parses source code into Abstract Syntax Trees rather than searching it as text, so it understands code structure and intent.1011 It is language-agnostic, supports TypeScript, and lets you write rules in YAML whose patterns look like the code you want to find. Originally built at Facebook in 2009, it now backs a registry of over 2,500 community rules, including rule sets that map to the OWASP Top 10.1112

Rules combine pattern operators: pattern for a single expression, patterns for a logical AND, pattern-either for a logical OR, and pattern-regex for a PCRE-compatible regex.11 This is enough expressiveness to encode Angular-specific anti-patterns that ESLint cannot describe. Below is a custom rule that flags the framework's sanitization bypass methods.

rules/angular-bypass-security.yaml — custom Semgrep rule
rules:
  - id: angular-dom-sanitizer-bypass
    languages: [typescript]
    severity: WARNING
    message: >
      DomSanitizer.bypassSecurityTrust* disables Angular's built-in
      sanitization. Trace the argument's data flow and confirm it is
      not attacker-controlled before approving this finding.
    metadata:
      cwe: "CWE-79: Improper Neutralization of Input (XSS)"
      owasp: "A03:2021 - Injection"
    pattern-either:
      - pattern: $S.bypassSecurityTrustHtml(...)
      - pattern: $S.bypassSecurityTrustScript(...)
      - pattern: $S.bypassSecurityTrustStyle(...)
      - pattern: $S.bypassSecurityTrustUrl(...)
      - pattern: $S.bypassSecurityTrustResourceUrl(...)
shell — running Semgrep locally and in CI
# run the auto config (curated registry rules for the detected languages)
semgrep scan --config auto

# run a curated security ruleset plus your custom Angular rules
semgrep scan --config "p/typescript" --config ./rules/ src/

# emit machine-readable output for the pipeline
semgrep scan --config auto --json > semgrep-report.json

Semgrep's deterministic SAST is well suited to classic flaws such as XSS and SQL injection, while its newer AI-assisted detection targets context-heavy logic flaws like insecure direct object references (IDOR) and broken authorization — the kinds of issues rule-based engines historically struggle with.910 The open-source CLI is free; cross-file and cross-function dataflow analysis, which materially reduces false positives, is part of the commercial platform.10

Static analysis tells you what could go wrong in the code you wrote. Dynamic testing tells you what actually goes wrong in the application you shipped.
§ 06 — Layer two

Dynamic testing: attacking the running app

Dynamic Application Security Testing (DAST) runs active tests against the live application, exercising it the way an attacker would and surfacing vulnerabilities that only appear at runtime — in authentication flows, APIs, and JavaScript-heavy single-page applications.1314 Because Angular renders its UI client-side, a DAST tool for Angular must do two things well: drive the application like a browser so it discovers AJAX-loaded routes, and proxy the HTTP/HTTPS traffic so every API request can be inspected and replayed.

The two staples here are OWASP ZAP, an automated and semi-automated scanner, and Burp Suite Community Edition, the manual-testing proxy of choice. They overlap but are used differently: ZAP shines at breadth and automation; Burp at precise, hands-on manipulation of individual requests.

§ 07 — DAST tool A

OWASP ZAP

The Zed Attack Proxy (ZAP) is an open-source DAST tool founded in 2010 by Simon Bennetts as an OWASP project; it has since become one of the most widely used web-application security scanners in the world. The project left OWASP in 2023 for the Linux Foundation and, in 2024, was adopted by Checkmarx and rebranded "ZAP by Checkmarx," remaining open-source and free.13 ZAP sits as a manipulator-in-the-middle proxy between browser and application, capturing HTTP/HTTPS traffic and observing how the application responds to potentially malicious requests.14

Core capabilities for an Angular SPA

CapabilityWhat it does
Traditional spiderCrawls reachable URLs to map the attack surface13
AJAX spiderDrives a headless browser to reach content invisible to traditional crawlers — essential for JavaScript-heavy SPAs15
Passive scanInspects traffic without altering it; flags missing security headers and insecure cookies — considered safe to run continuously14
Active scanFires attack payloads at discovered endpoints to surface SQL injection, XSS, and more14
Automation framework & REST APIScriptable scans for CI/CD integration13
SPA gotcha Active scans hit authenticated APIs without the bearer token unless ZAP is taught the auth scheme — so API findings come back empty. For an Angular app, configure authentication (and a logged-in session/token) before scanning, then manually walk the app through the proxy so ZAP discovers the routes only visible to authenticated users.14
shell — automated baseline scan via the ZAP Docker image
# Passive "baseline" scan against a running Angular build
docker run -t ghcr.io/zaproxy/zaproxy:stable \
  zap-baseline.py -t https://app.example.com -r zap-report.html

# Full active scan (use only against systems you are authorized to test)
docker run -t ghcr.io/zaproxy/zaproxy:stable \
  zap-full-scan.py -t https://app.example.com -r zap-full.html

ZAP runs on Windows, macOS, and Linux, works in test or production environments, and commonly detects SQL injection, broken authentication, sensitive data exposure, and cross-site scripting.16 It holds OWASP Flagship Project status and is listed in the OWASP Testing Guide as a reference DAST tool.15

§ 08 — DAST tool B

Burp Suite Community Edition

Burp Suite, built by PortSwigger, has been an industry-standard web-application testing platform for over a decade. The Community (free) Edition is engineered around a core intercepting proxy over HTTP and WebSocket protocols, and ships the essential tools for manual testing.1719 Burp listens by default on 127.0.0.1:8080; you point the browser at that proxy and install Burp's CA certificate to avoid TLS warnings.18

Proxy

intercept · history · scope

  • Pause, edit, and forward every request
  • Or pass traffic through and review the HTTP history afterwards
  • Right-click any request to send it to the other tools18

Repeater

manual replay

  • Resend a single modified request as many times as needed
  • Ideal for careful, targeted verification of one endpoint19

Sequencer

entropy analysis

  • Tests the randomness of session tokens for predictability19
  • Directly relevant to Angular session-management testing

Decoder & Comparer

encoding · diffing

  • Decoder handles Base64, URL, and Hex transforms19
  • Comparer shows exactly what changed between two responses20
Edition boundaries Community Edition includes Proxy, Repeater, Decoder, Sequencer, Comparer, the BApp Store, and a throttled (demo) Intruder. The automated web vulnerability Scanner is Professional-only — at roughly US$449/year — and Professional also removes Intruder's rate throttling. For speed-sensitive fuzzing in CE, the free Turbo Intruder BApp restores throughput.161720

The practical division of labour: use ZAP to crawl and scan an Angular app at breadth and to gate CI; use Burp Community to take the interesting requests ZAP (or the browser) surfaced and pull them apart by hand — changing one parameter, replaying with a different user's session, or fuzzing an upload endpoint. Manual modification and replay of requests is how you test input validation, session handling, and other controls that automated scanners report imprecisely.18

§ 09 — Layer three

Penetration testing areas

Automated scanners are necessary but not sufficient: the highest-severity, highest-likelihood modern vulnerabilities — broken access control and authentication failures — require human reasoning about who should be allowed to do what.23 What follows is a structured checklist across the seven areas that matter most for an Angular SPA backed by an API, with the specific tests, the tool to reach for, and the underlying principle.

1 · Authentication

Identification and Authentication Failures (A07:2021, formerly "Broken Authentication") covers weaknesses that let attackers in: credential stuffing against endpoints with no rate limiting, weak password rules, and session-fixation flaws (CWE-384).21 In an Angular app the front end never authenticates anyone — it only holds and presents a credential — so every authentication test is really a test of the API plus how the SPA stores the token.

  • Token storage. Inspect where the access token lives. OWASP's guidance is explicit: do not store session identifiers in local storage, because the data is always accessible to JavaScript; an HttpOnly cookie keeps a token unreadable by injected script.2526 A JWT in localStorage can be exfiltrated by any XSS; with an HttpOnly cookie the attacker can still ride the session but cannot steal the token for later use.25 The widely recommended pattern is a short-lived access token held in memory plus a refresh token in a Secure; HttpOnly cookie.27
  • Brute-force / credential stuffing. Replay the login request in Burp Repeater with varied credentials; confirm the API enforces rate limiting and account lockout.19
  • Token validation. Tamper with the JWT (alter the alg header, strip the signature, extend exp) and confirm the server rejects it — the token must be validated server-side, never trusted because the client sent it.

2 · Authorization

Broken Access Control is the #1 OWASP risk. Typical failures include violating least-privilege/deny-by-default, bypassing checks by tampering with the URL or API request, viewing or editing another user's account by supplying its identifier (insecure direct object references), and missing controls on POST/PUT/DELETE.3

Angular-specific trap Route guards (CanActivate) and *ngIf="isAdmin" are UX, not security. They run in code the user controls and can be removed in the browser. Every authorization decision must be re-enforced on the server. Test by calling the protected API directly — bypassing the Angular UI entirely — using Burp Repeater with a low-privilege user's token.
  • IDOR. Change an id in an API path or body to another user's resource and confirm a 403, not the data.3
  • Forced browsing & verb tampering. Hit admin endpoints directly; swap GET for DELETE; confirm method-level access control.3
  • Privilege escalation. Replay a high-privilege request with a low-privilege session in Burp; the server must reject it.

3 · Session management

Session integrity is, per OWASP, critical to defending against authentication attacks alongside identity confirmation.21 Key tests:

  • Token entropy. Feed a large sample of session tokens into Burp Sequencer to confirm they are unpredictable and resistant to guessing.19
  • Invalidation on logout. OWASP advises that stateful session identifiers be invalidated server-side on logout, while stateless JWTs should be short-lived to shrink the attacker's window; longer-lived JWTs should follow OAuth revocation standards.3 Capture a token, log out, then replay the token — it must be dead.
  • Cookie flags. Verify session cookies carry Secure, HttpOnly, and an appropriate SameSite value; ZAP's passive scan flags missing flags automatically.14

4 · API security

Because an Angular SPA is a thin client over an API, the API is the real attack surface. Beyond authorization (above), test for Cryptographic Failures (A02:2021): confirm all traffic uses TLS with forward-secrecy ciphers, that no sensitive data is sent over legacy plaintext protocols, and that passwords are stored with strong adaptive salted hashes such as Argon2, scrypt, bcrypt, or PBKDF2.22 Also probe for excessive data exposure (the API returning more fields than the UI shows), missing rate limiting, and Server-Side Request Forgery — note that the 2025 OWASP revision folds SSRF into Broken Access Control, reflecting that it is fundamentally an over-permissioned trust problem.28

Angular SSR note If you use Angular's server-side rendering, the framework now validates Host and X-Forwarded-* headers against an allowlist to prevent header-based SSRF; X-Forwarded-* headers are treated as untrusted and stripped unless explicitly allowed via trustProxyHeaders. Audit your allowedHosts configuration — a wildcard * reopens host-header injection.1

5 · File uploads

Unrestricted file upload can lead to remote code execution — the highest-severity outcome of an upload flaw.24 The decisive point for an Angular tester: client-side validation is a convenience feature, not a security control, and is trivially bypassed with browser dev tools, curl, or Burp. Every rule must be enforced server-side, and you test by bypassing the Angular checks entirely.24

6 · Cross-site scripting (XSS)

XSS lets attackers inject code that steals login data or impersonates the user, and remains one of the most common web attacks.1 Angular's default escaping blocks much of it, so testing focuses on where developers stepped outside the framework:

  • Sanitization bypasses. Trace every bypassSecurityTrust* call (find them with the Semgrep rule above) and test whether user input can reach it.45
  • Direct DOM access. Unless Trusted Types are enforced, browser DOM APIs reached through ElementRef or third-party libraries are not auto-sanitized; inject payloads where such code handles user data.1
  • Reflected / stored / DOM-based. Use ZAP's active scan to inject payloads into every discovered input and report which execute, then confirm by hand in Burp.514
  • CSP presence. Confirm a Content-Security-Policy header is set as defense-in-depth; ZAP's passive scan flags its absence.114

7 · Cross-site request forgery (CSRF)

In a CSRF attack a malicious page silently sends a state-changing request to an application where the victim is logged in, relying on the browser to attach the auth cookie automatically.1 Angular's HttpClient provides the cookie-to-header defense described in §02, but it protects only the client half — the backend must set the XSRF-TOKEN cookie and verify the X-XSRF-TOKEN header, or the protection is ineffective.1

  • Header echo. Confirm mutating requests carry the X-XSRF-TOKEN header and the server rejects requests where it is missing or mismatched.1
  • Custom names. If the backend uses different names, confirm the Angular client is configured with withXsrfConfiguration({ cookieName, headerName }).1
  • SameSite backstop. Verify cookies set SameSite=Lax or Strict as an additional layer, since the token scheme protects only requests issued through HttpClient.26
§ 10 — Operationalizing

Wiring it into the pipeline

Tools find nothing if they run only once. The practical assembly for an Angular project:

  1. Pre-commit / IDE. ESLint with eslint-plugin-security runs on save and blocks the obvious hotspots before they are committed.6
  2. Pull request. Semgrep scans the diff with curated rules plus your custom Angular rules, posting findings as PR comments; it integrates cleanly into CI and can publish to ecosystems such as GitLab's SAST.1011
  3. Nightly / staging. ZAP's baseline scan (passive) runs against every deploy; a scheduled full active scan runs against an authorized staging environment via the automation framework or Docker image.1316
  4. Periodic manual pentest. A human, armed with Burp Community, works the seven areas above — especially authorization and business logic, which automation reports poorly.18
Rule of engagement Active scanning and manual fuzzing send real attack payloads. Run them only against systems you own or are explicitly authorized to assess, and keep destructive tests off production.18
§ 11 — Closing

Closing notes

Angular gives you a strong default posture: untrusted-by-default rendering, contextual sanitization, a CSRF helper, XSSI stripping, and — increasingly — host-header validation for server-rendered apps.1 None of that covers authentication, authorization, sessions, the API, or file handling, and all of it can be deliberately bypassed. A credible Angular security program therefore layers static analysis to catch dangerous code as it is written, dynamic scanning to attack the running build, and human penetration testing for the judgement-heavy access-control questions that automated tools cannot answer.

The good news is that the entire toolchain described here — ESLint, Semgrep's CLI, OWASP ZAP, and Burp Suite Community Edition — is free and open enough to start using today. Keep the framework current so you inherit its security patches, audit every bypassSecurityTrust* call, and re-enforce on the server every decision the client appears to make.1
