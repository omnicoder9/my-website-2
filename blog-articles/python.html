>>> import origins · section 01

A hobby project, over Christmas

In December 1989 a Dutch programmer named Guido van Rossum went looking for a project to occupy the quiet week around Christmas.1 What he started as a diversion would, within a generation, become one of the most widely used programming languages on Earth.

Van Rossum was not starting from nothing. For years at the Centrum Wiskunde & Informatica (CWI) in the Netherlands he had worked on ABC, a language designed to teach programming to beginners without sacrificing expressive power. ABC was clean and readable, but it had hard limits: it could not easily be extended with new modules, and it lacked proper exception handling.1 Van Rossum absorbed what he admired and catalogued everything he would do differently.

Python was conceived in the late 1980s explicitly as a successor to ABC, and implementation began that December.2 From ABC it inherited two decisions that still define the language: the use of indentation to group statements, and a small set of powerful, very-high-level data types such as the list and the dictionary.1 Where ABC was a closed world, though, Python was built to be extended — highly modular, in the spirit of Niklaus Wirth’s Modula languages — so that functionality could live in libraries rather than the core.1

The first public version, Python 0.9.0, appeared in 1991; van Rossum posted the code to the Usenet group alt.sources in February of that year.23 Even at this earliest stage the language already carried classes with inheritance, exception handling, functions, the core list, dict and str types, and a module system borrowed from Modula-3.3 The name, for the record, honours the British comedy troupe Monty Python’s Flying Circus — not the snake, however hard the logo’s two serpents work to suggest otherwise.21

the founding conviction

Underneath every later design choice sits one belief van Rossum held from the start: that a programmer’s time is worth more than a computer’s, and that code is read far more often than it is written.45 Almost everything distinctive about Python follows from taking that seriously.

>>> import evolution · section 02

Versions, and one deliberate break

Python’s history is mostly one of patient accretion. Ideas arrived, were absorbed, and rarely left — with a single, defining exception.

Python 1.0 shipped in January 1994, adding the functional-programming tools lambda, map, filter and reduce — contributed, van Rossum later recalled, by a Lisp enthusiast who missed them.3 Python 2.0 followed in October 2000 with list comprehensions, a cycle-detecting garbage collector, and Unicode support.6

Then came the break. Python 3.0 — nicknamed “Python 3000” or “Py3K” — was released on 3 December 2008 and was deliberately not backward-compatible with the 2.x line.3 Its guiding principle was to reduce feature duplication by removing the old ways of doing things, in direct service of the Zen of Python’s insistence that there should be one obvious way to do something.3 The migration would take the community more than a decade; the 2.x line was finally retired in 2020. Along the way, from version 3.5 onward, Python gained optional static typing — keywords and syntax for type hints that tools could check without changing how the code runs.2

Governance evolved too. For nearly three decades van Rossum served as Python’s Benevolent Dictator for Life, holding final say over the language’s design. On 12 July 2018 he stepped down, worn out after a bruising debate over PEP 572, the proposal that added the assignment-expression (“walrus”) operator.25 In January 2019 the core developers elected a five-member Steering Council to lead the project in his place;2 the non-profit Python Software Foundation had already anchored the project’s stewardship since 2001.5 Van Rossum, meanwhile, joined Microsoft in November 2020 to work on making Python faster.4

Today the project runs on a predictable annual cadence with a five-year support window per release.7 Python 3.14.0 arrived on 7 October 2025; by mid-2026 the current maintenance release was 3.14.6, Python 3.15 was in beta ahead of an October 2026 release, and 3.16 had opened as the main development branch.87

Version 3.14 is no mere point release. Its headline change is that free-threaded Python is now officially supported (PEP 779), alongside deferred evaluation of annotations (PEP 649), template string literals or “t-strings” (PEP 750), multiple interpreters in the standard library (PEP 734), a Zstandard compression module (PEP 784), and a zero-overhead external-debugger interface (PEP 768).9 The official Windows and macOS binaries also ship an experimental just-in-time compiler, and official Android builds are now available.10 Free-threading follows the three-phase plan laid out in PEP 703: the single-threaded performance penalty of running without the Global Interpreter Lock has fallen to roughly 5–10%, down from around 40% in the experimental 3.13 build.1112

>>> import merits · section 03

What the language is made of

Ask why Python won and you get the same answer in a dozen forms: it optimises relentlessly for the human reading the code.

The most-cited virtue is readability treated as an engineering property, not a nicety. Python uses plain English keywords in place of symbols, reserves punctuation for cases where it is already conventional in written English or algebra, and makes indentation itself part of the grammar — so the visual shape of a program and its logical structure cannot drift apart.13 The effect is code that reads close to executable pseudocode.

word_count.py
# Count the five most common words in a file — the whole program.
from collections import Counter

with open("speech.txt") as f:
    words = f.read().lower().split()

for word, n in Counter(words).most_common(5):
    print(f"{word:>12}  {n}")

Beyond legibility, Python is dynamically typed and interpreted, which shortens the loop between writing code and running it, and it ships with a famously large standard library — the “batteries included” philosophy. When Netflix explained why its engineers reach for Python, it named exactly these traits: a rich built-in library, a clean and expressive syntax, and a vast ecosystem of third-party packages to draw on.14 Because the language was built to be extended through modules rather than by bloating its core, that ecosystem could grow without waiting on the language itself.2

Python’s other structural advantage is that it is superb glue. Its flexibility and convenience make it particularly useful for stitching together small utility programs written in other languages into a larger whole,1 which is why so much high-performance numerical and systems code hides behind a Python interface. That same design points at Python’s classic weakness: as an interpreted language whose Global Interpreter Lock long capped multi-core throughput, raw CPU-bound performance has never been its strength. The traditional remedy was to push hot loops down into C; the 2025 remedy, as the previous section noted, is the pairing of free-threading and a JIT.11

the rule at Google

The trade-off has a famous one-line summary from Python’s early adopters at Google: “Python where we can, C++ where we must.”14 Reach for the readable language by default; drop to the fast one only where the profiler insists.

>>> import this · section 04

The Zen, and other intangibles

Some of what makes Python Python cannot be put in a feature list. It lives in a set of aphorisms and in a temperament.

In 1999 the core developer Tim Peters tried to write down van Rossum’s unstated design principles and posted the result — nineteen aphorisms — to the Python mailing list.15 He left a slot open for a twentieth “for Guido to fill in”; it never has been. The list was released into the public domain, formalised as PEP 20, and hidden inside the interpreter as an Easter egg you summon by typing import this.15 (Peters, incidentally, is also the inventor of Timsort, the sorting algorithm Python still uses.)16

The aphorisms are less a rulebook than a sensibility: Explicit is better than implicit. Simple is better than complex. Readability counts. There should be one— and preferably only one —obvious way to do it.2 They rhyme with the concrete choices van Rossum made in the syntax — his stated rule that punctuation should be used conservatively, in line with its ordinary use in written English or high-school algebra, so that Python does not use $ to mark variables or ! to signal side effects.13 The deeper goals were admittedly subjective ones: elegance, simplicity, learnability, and a language that would make everyday tasks quick and programmers productive.1

Code that honours this spirit is called Pythonic, and the word carries real weight in the community. The principles are guidelines, not laws — Python has taken criticism for accumulating features and “bloat,” and the standing defence is precisely that the Zen is a guideline rather than a rule.2 That tension is not academic: it was a disagreement over whether a new operator fit the language’s character, not whether it worked, that ultimately ended van Rossum’s reign as Benevolent Dictator for Life.2

an in-joke, thirty-five years deep

One aphorism reads: Although that way may not be obvious at first unless you’re Dutch. It is a wink at van Rossum himself.16 The Zen of Python is a serious statement of values delivered, characteristically, as a joke.

>>> import popularity · section 05

Where Python stands

No single ranking tells the whole story — each measures a different thing — but almost all of them now point the same way.

On the search-based TIOBE index, the longest-running popularity gauge, Python held first place through 2026, with C in second.17 Python’s rating peaked at nearly 27% in July 2025 and, even after cooling to around 21.8% in early 2026, still led C by roughly ten percentage points — a moat no other language comes close to.1819 On PYPL, which tracks how often language tutorials are searched, Python ranks first in the United States, India, Germany, the United Kingdom and France alike, each time with more than a 40% share.19

The most telling movements are on GitHub, where the data reflect code actually written. In 2024 Python overtook JavaScript as the most-used language on the platform, ending a ten-year run and reflecting the surge in data science and machine learning.20 Then, in August 2025, TypeScript overtook both to claim the top spot by monthly contributors — a sign of how strongly AI-assisted coding rewards statically typed languages.2122

Yet the same reports make Python’s real position unmistakable: it is the language of AI. Python powers nearly half of all new AI repositories on GitHub — some 582,000 projects, growing more than 50% year over year — and added over 850,000 contributors in a single year.23 Repositories using Jupyter notebooks, Python’s data-science workbench, grew about 75% to roughly 2.4 million.24 Whatever the monthly leaderboard says, the generative-AI boom runs on Python.

>>> import production · section 06

In production, at scale

The real test of a language is what people are willing to ship in it. By that measure Python has little left to prove.

Instagram runs what its own engineers describe as the world’s largest deployment of the Django web framework, written entirely in Python, and it scaled to hundreds of millions of users on that Python-and-Django stack before Facebook acquired it.2514 Spotify engineers roughly 80% of its backend services in Python — a constellation of services wired together over ZeroMQ — and leans on the Python data-pipeline tool Luigi for the analytics behind its recommendations.25 Dropbox wrote both its server infrastructure and its desktop client in Python, which is part of why founder Drew Houston could stand up a working prototype in about two weeks.26

The list runs on: Reddit, Pinterest and Uber all lean on Python, Quora’s founders chose it for its expressiveness and development speed, and even YouTube began life with Python at its core.1425 At Google, Python has been woven through the stack since the earliest days — the company was an early and influential adopter, and much of its internal tooling and services rely on it.14 For a language conceived as a Christmas hobby, that is an unusually load-bearing role in the modern internet.

>>> import numpy as np · section 07

The scientific stack

Python did not become the language of data because of the language. It became so because of a handful of libraries built on top of it.

The foundation is NumPy. In 2005 and 2006 Travis Oliphant merged two older numerical projects — Numeric, dating to 1995, and Numarray — into a single, fast, n-dimensional array type, the ndarray.2728 Because NumPy’s core is written in C and Fortran with a thin Python layer over it, Python could suddenly work with large matrices at speeds close to those compiled languages.28 Its role as bedrock infrastructure was formally recognised when the paper Array programming with NumPy appeared in Nature in 2020.29

On top of the array came the table. pandas was built by Wes McKinney in 2008 while he worked at the quantitative hedge fund AQR Capital Management, frustrated by the absence of good tools for analysing financial data in Python.2728 Its DataFrame became the second core structure of the ecosystem, the thing nearly every later data tool speaks to. Around these two grew the rest: Matplotlib, John D. Hunter’s 2003 library for publication-quality plots,28 and SciPy, the broader collection of scientific algorithms whose 1.0 release was documented in Nature Methods in 2020.29

Much of this software is now sustained by NumFOCUS, the non-profit — co-founded by Oliphant — that supports open-source scientific computing.29 Together these libraries are the substrate on which modern machine learning was built.29

why Python beat R

For years R and Python were framed as rivals for data work. The ndarray and the DataFrame settled it: a shared, high-performance substrate that let general-purpose software, web backends and machine-learning models all speak the same numerical language — something a specialised statistics environment could not match.29

>>> import frameworks · section 08

Frameworks for the web

Two philosophies have defined Python on the web — include everything, or include almost nothing — and a third has quietly become the standard for AI.

Django, created by Adrian Holovaty and Simon Willison in 2005, is the “batteries-included” option: a full framework following the model-template-views pattern and marketed, memorably, as “the web framework for perfectionists with deadlines.”30 It ships an object-relational mapper, an admin interface and authentication out of the box — the machinery behind Instagram’s backend.30

Flask is its opposite. Armin Ronacher released it in April 2010 — it began, famously, as an April Fool’s joke that turned out to be popular enough to make serious.31 A deliberately minimal microframework built on Ronacher’s own Werkzeug and Jinja2 libraries (later homed under the Pallets project), Flask provides a solid core and trusts developers to add only the extensions they need.31 For years it has sat second only to Django in GitHub stars among Python web frameworks, and it topped the Python Developers Survey every year from 2018 through 2022.31

The newcomer is FastAPI, first released by the Colombian developer Sebastián Ramírez in December 2018.32 Built on the Starlette toolkit and the Pydantic validation library, it is designed around Python’s type hints, which it uses to validate requests and generate interactive documentation automatically.32 Pydantic’s version-2 core was rewritten in Rust for roughly a twentyfold speedup, and the combination of speed, async support and clean integration with the ML ecosystem has made FastAPI the de facto standard for serving machine-learning models.32

>>> import torch · section 09

The engines of machine learning

If Python is the language of AI, two frameworks are its engines — and they represent two philosophies of how to describe a computation.

TensorFlow came first. Google Brain’s Jeff Dean and Rajat Monga open-sourced it on 9 November 2015 under the Apache 2.0 licence, as the successor to Google’s internal DistBelief system from 2011.3334 Its model is to express a computation as a data-flow graph that the framework then executes; in practice, developers write Python that drives a high-performance C++ core.34 The 2.0 release in 2019 adopted the friendlier Keras API and eager execution to soften the framework’s early reputation for difficulty.33

PyTorch arrived from Meta’s AI Research lab in 2016 and won over researchers with the opposite approach: a dynamic computation graph.35 Instead of defining the whole graph up front, PyTorch builds it on the fly using ordinary Python control flow — loops, conditionals, recursion — and records operations on a tape so its autograd system can compute gradients in reverse.35 The result is model code that behaves like normal Python and can be debugged like it, and PyTorch is now the framework of choice across a large share of deep-learning research and production.35

Both frameworks are, in the end, the “glue” philosophy taken to its conclusion: a readable Python surface stretched over hand-optimised CUDA and C++ kernels. Which is exactly where the last section of this portrait begins.

>>> import ctypes · section 10

The glue: implementations and integrations

The thing that lets a slow, interpreted language power high-performance software is that it was designed, from the beginning, to hand work off to faster ones.

What almost everyone means by “Python” is CPython, the reference implementation and by far the most widely used. It has always been an interpreter rather than a JIT compiler: it turns your source into portable bytecode, executes that bytecode, manages memory through reference counting, and exposes its internals to other languages through a C-API declared in Python.h.3637

That C-API is why Python is such effective glue. The standard library’s ctypes lets you load a shared library at runtime — a .so or a .dll — and call its C functions as though they were ordinary Python, with no wrapper and no build step.37 The related CFFI does the same with a cleaner interface and is the recommended tool on the PyPy runtime.38

ffi_demo.py
# Borrow a C function from a shared library at runtime — no build step.
from ctypes import CDLL, c_double

libm = CDLL("libm.so.6")           # the system C math library
libm.cos.restype  = c_double
libm.cos.argtypes = [c_double]

print(libm.cos(c_double(0.0)))     # -> 1.0

For heavier lifting, Cython — a superset of Python that compiles to C, and one of the oldest such tools still standing — is used inside the numeric stack itself to make libraries like NumPy and SciPy fast, while Numba just-in-time-compiles numerical Python through the LLVM toolchain.39

And CPython is not the only engine available. PyPy is an alternative implementation whose tracing JIT compiler often runs pure-Python code substantially faster than CPython does; Jython runs Python on the Java Virtual Machine, and IronPython on Microsoft’s .NET.36 A quieter recent improvement ties the ecosystem together: CPython’s stable ABI now lets a single compiled extension run across multiple interpreter versions, easing the long-standing pain of rebuilding native packages for every release.38

It is a fitting place to end. Python’s enduring trick is not that it is fast — it usually is not — but that it made the readable language and the fast one cooperate so gracefully that, thirty-five years on, most of the software world decided it would rather write the readable one.
