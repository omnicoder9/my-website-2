Action Model Learning: Automating Domain Knowledge Acquisition in AI Planning

A technical examination of the algorithms, frameworks, and machine learning techniques developed to automatically infer PDDL domain models from observed plan executions

by Nicholas Knapp | May 9, 2026

Back to all articles

Automated planning systems require a formal, machine-readable description of the actions an agent can perform — commonly called a domain model. Encoding such models by hand is laborious, error-prone, and demands expert knowledge of both the target domain and the modelling language in question. Kambhampati identified this knowledge-acquisition bottleneck as a fundamental obstacle limiting the practical deployment of AI planning technology.[7] In response, a substantial body of research has developed over the past three decades devoted to action model learning (AML): the automatic induction of planning domain models from observations of agent behaviour. This article surveys the theoretical underpinnings, landmark algorithms, contemporary tools, and open challenges of this field.

Section 01Historical Context and Motivation

Automated planning emerged in the 1960s when research in operations research, theorem proving, and the study of human problem-solving began coalescing, with an impetus driven significantly by challenges in robotics.[2] The seminal contribution was STRIPS (STanford Research Institute Problem Solver), introduced by Fikes and Nilsson in 1971, which defined actions through three components: a set of preconditions (what must hold for the action to be applicable), a delete list (fluents that become false after execution), and an add list (fluents that become true).[5]

This representation was later formalised and extended into the Planning Domain Definition Language (PDDL), introduced in 1998 to provide a common interchange format for planning system benchmarks.[6] PDDL supports typed objects, quantified formulae, conditional effects, numeric fluents, durative actions, and temporal constraints across successive version releases, making it the de facto standard for domain specification in the international planning community.

Despite the maturity of planning algorithms — including the Fast Forward heuristic planner and the Fast Downward planning system — the construction of accurate domain models remains a persistent bottleneck. Building a PDDL model from scratch is considered difficult and time-consuming even for planning experts, and this is considerably harder when dealing with temporal or numeric domains.[10] This impasse motivated the development of techniques for learning models automatically.

Early work by Wang (1994) — the Observer algorithm — was among the first systematic attempts to extract action models from execution traces, operating under the assumption of full observability, noise-free data, and deterministic actions.[19] Subsequent decades saw progressively more sophisticated approaches capable of handling partial observability, noisy data, and relational (lifted) action representations.

Section 02Formal Foundations

2.1 The Classical Planning Model

A classical planning problem is a tuple P = ⟨D, I⟩, where D is a domain description and I is a problem instance specifying an initial state and a goal. The domain consists of a set of action schemas, each with typed parameters, preconditions expressed as conjunctions of literals, and effects split into positive add-effects and negative delete-effects. Ground instantiation of schema parameters over objects in the problem yields a state graph in which nodes represent world states and labelled edges represent action transitions.[8]

STRIPS Action Schema (Formal Definition). An action schema A = ⟨name, params, pre, eff⁺, eff⁻⟩ where params is a tuple of typed variables, pre is a conjunction of literals (the precondition), eff⁺ is the set of positive effects (facts made true), and eff⁻ is the set of negative effects (facts made false). For conditional effects, each effect has the form ⟨c, e⟩ such that if antecedent condition c holds in the current state when action a is applied, then result e is asserted in the successor state.[9]

2.2 The Action Model Learning Problem

The action model learning problem may be stated as follows. Given a set of observation traces — sequences of state-action interleavings — the task is to induce a domain model D such that the model is consistent with the observations and generalises to unseen problem instances. Formally, a trace τ = ⟨s₀, a₁, s₁, a₂, s₂, …, aₙ, sₙ⟩ records a sequence of states and the actions that caused each transition.

Several design axes differentiate AML approaches: the observability of states and actions (fully, partially, or not observed); whether data is noisy or deterministic; whether learning proceeds online (incrementally as actions are executed) or offline (from a pre-collected dataset); and whether the output model is propositional (ground) or lifted (using schematised variables that generalise across objects).[24]

Input: Σ = {τ₁, τ₂, …, τₖ} ← set of observation traces A = {a₁, a₂, …, aₘ} ← known action names/signatures Output: D = {Â₁, Â₂, …, Âₘ} ← learned action schemas where each Âᵢ = ⟨preᵢ, eff⁺ᵢ, eff⁻ᵢ⟩ Subject to: D is consistent with every trace in Σ

The input to AML algorithms differs from conventional machine learning in two important respects: the training data consists of variable-length plan traces rather than fixed-dimension feature vectors, and the output is a structured symbolic model (conjunctions of literals per action) rather than a scalar prediction.[6]

Section 03Core Algorithms

3.1 Observer — Set-Based Intersection (Wang, 1994)

The Observer algorithm, one of the earliest and most straightforward AML methods, operates under full observability and assumes noise-free, deterministic traces.[19] For each action type a, it maintains candidate sets for preconditions and effects by intersecting observations across all plan traces. The preconditions of a are initialised to all fluents true in the first observed pre-state; subsequent observations narrow this candidate set via set intersection. Effects are determined by comparing pre- and post-states.

Algorithm 1 — Observer (Wang, 1994) Set-Intersection Method
  1. Initialise PRE(a) ← S₀ for each action type a, using the first observed pre-state.
  2. For each subsequent observation of action a in state sᵢ:
    1. PRE(a) ← PRE(a) ∩ sᵢ
  3. For each transition ⟨sᵢ, a, sᵢ₊₁⟩:
    1. EFF⁺(a) ← EFF⁺(a) ∪ (sᵢ₊₁ \ sᵢ)
    2. EFF⁻(a) ← EFF⁻(a) ∪ (sᵢ \ sᵢ₊₁)
  4. Return ⟨PRE(a), EFF⁺(a), EFF⁻(a)⟩ for each action.

The Observer's principal limitation is its strict requirement for full state observability. In practice, many domains expose only partial information — sensor readings, incomplete logs, or aggregated telemetry. More sophisticated algorithms were required to address these constraints.

3.2 ARMS — Action-Relation Modelling System (Yang, Wu & Jiang, 2007)

ARMS (Action-Relation Modelling System) was a landmark contribution that addressed the practical difficulty of obtaining complete intermediate state information.[18] Unlike Observer, ARMS does not require knowledge of states between actions; it operates on plan traces where intermediate states may be entirely absent. The system extracts statistical information about frequently co-occurring action-fluent relationships from the input plans, encodes this information as a weighted MAX-SAT problem, and uses a solver to identify the most probable consistent action schemas.

The MAX-SAT formulation assigns weights to constraints based on the frequency of supporting evidence in the plan corpus. The solver maximises the total weight of satisfied constraints, effectively performing a form of probabilistic inference over the space of possible domain models. ARMS does not require typed parameters and operates at the ground level, which simplifies the constraint encoding but limits scalability to large object sets.

Python (MacQ / ARMS usage) Example: Generating traces and extracting a domain model with ARMS 
from macq import generate, extract
from macq.observation import IdentityObservation, AtomicPartialObservation

# Step 1: Generate traces from a planning.domains problem
# Problem 123 is a Blocksworld instance.
traces = generate.pddl.VanillaSampling(
    problem_id = 123,
    plan_len   = 20,
    num_traces = 100
).traces

# Step 2: Inspect a trace
trace  = traces[0]
action = trace[0].action
pre_states  = trace.get_pre_states(action)   # states before each action
post_states = trace.get_post_states(action)  # states after each action

# Step 3: Tokenize — simulate partial observability (60% fluents hidden)
observations = traces.tokenize(
    Token     = AtomicPartialObservation,
    percent_missing = 0.6
)

# Step 4: Apply the ARMS extraction algorithm
# ARMS uses weighted MAX-SAT to learn preconditions and effects.
model = extract.ARMS(
    observations,
    min_support   = 2,     # minimum co-occurrence frequency
    action_weight = 110,
    info_weight   = 100,
    threshold     = 0.6,
    output        = "pddl"
)

# Step 5: Retrieve learned action schemas
for action_name, schema in model.actions.items():
    print(f"Action: {action_name}")
    print(f"  Preconditions: {schema.precond}")
    print(f"  Add effects:   {schema.add}")
    print(f"  Del effects:   {schema.delete}")

3.3 SLAF — Simultaneous Learning and Filtering (Amir & Chang, 2008)

SLAF (Simultaneous Learning and Filtering) addresses partially observable deterministic domains in which only a subset of fluents is revealed at each time step.[25] The algorithm maintains a propositional formula — the transition-belief formula — which represents the set of all action models consistent with observations accumulated so far. As each new action-observation pair is received, SLAF updates this formula through a logical filtering operation. The final formula encodes all learned models simultaneously, and a SAT or CNF solver is used to extract a specific consistent model.

SLAF proceeds in an online fashion: it updates the belief formula incrementally rather than processing all traces in batch. The algorithm offers polynomial-time updates when the belief formula is maintained in k-CNF form, making it tractable for domains with hundreds of fluents.[25] Experimental evaluations demonstrated correctness on standard planning domains including Driverlog and Safe World.

PDDL Example: Blocksworld domain fragment — expected output of a learned model 
; Learned STRIPS domain — Blocksworld (representative output)
(define (domain blocksworld-learned)
  (:requirements :strips :typing)
  (:types block)
  (:predicates
    (on ?b1 - block ?b2 - block)
    (ontable ?b - block)
    (clear ?b - block)
    (holding ?b - block)
    (handempty))

  ; Learned action: pick up a block from the table
  (:action pickup
    :parameters   (?b - block)
    :precondition (and (clear ?b) (ontable ?b) (handempty))
    :effect       (and (holding ?b)
                       (not (ontable ?b))
                       (not (clear ?b))
                       (not (handempty))))

  ; Learned action: stack one block on another
  (:action stack
    :parameters   (?b1 - block ?b2 - block)
    :precondition (and (holding ?b1) (clear ?b2))
    :effect       (and (on ?b1 ?b2)
                       (clear ?b1)
                       (handempty)
                       (not (holding ?b1))
                       (not (clear ?b2))))
)

3.4 LOCM — Learning Object-Centred Models (Cresswell, McCluskey & West, 2013)

LOCM takes a fundamentally different approach from constraint-based methods. Rather than reasoning about state fluents directly, LOCM analyses plan sequences — sequences of action names and their object arguments — without access to any state information whatsoever.[12] It constructs finite state machines (FSMs) that capture how each action transitions the lifecycle state of each participating object. From these FSMs, PDDL predicates and their role in action preconditions and effects are inferred.

The algorithm operates in two phases: first constructing FSMs for each object sort from action sequences, then translating these automata into PDDL predicates. LOCM2 extended the original by handling cases where two objects mutually transition in an action. LOP (LOCM with Optimised Plans) further incorporated static predicates and used a set of optimal plans as an additional signal to disambiguate structurally similar domains.[15]

Pseudocode LOCM Phase 1 — FSM Construction from action sequences 
# Input: plan sequence π = [a₁(obj₁,..), a₂(obj₂,..), ...]
# For each object sort S do:

for sort S in sorts:
    # Extract subsequences of actions that involve objects of sort S
    action_seqs = extract_subsequences(π, S)

    # Build a transition hypothesis table
    # Rows = actions, Cols = (B_i, B_j) where B_i is state before, B_j after
    T = build_transition_table(action_seqs)

    # Construct deterministic FSM: states are object lifecycle states
    FSM[S] = construct_fsm(T)

# Phase 2: Translate FSM transitions to PDDL predicates
for sort S, state q in FSM[S].states:
    predicate = create_predicate(f"state_{S}_{q}", params=[S])

for (q_i, action, q_j) in FSM[S].transitions:
    # Precondition: object in state q_i
    # Effect: object moves from q_i to q_j
    action.pre  += predicate(q_i)
    action.eff⁺ += predicate(q_j)
    action.eff⁻ += predicate(q_i)

Figure 1. Simplified pseudocode for LOCM Phase 1 (FSM construction) and Phase 2 (PDDL predicate generation). Each object sort receives its own automaton capturing its lifecycle transitions across actions in the plan sequence.

3.5 FAMA — Fully-Observable and Minimal-Observability Action Model Acquisition (Aineto et al., 2019)

FAMA represents a state-of-the-art approach that reformulates the action model learning task itself as a classical planning problem.[26] Given input observations, FAMA compiles the learning task into a planning problem whose solutions correspond to action models consistent with the observations. This formulation is particularly powerful because it can operate when both actions and states are only partially observed — a realistic condition in sensor-based systems.

The FAMA compilation encodes the space of possible action models as planning states, and the goal is a model that satisfies all constraints imposed by the observations. FAMA additionally provides an evaluation mechanism for assessing the quality of a learned model both syntactically (with respect to the ground-truth model) and semantically (with respect to the observed behaviours).[26]

Section 04Advanced Topics

4.1 Learning with Noisy and Partial Data

Real-world observations are rarely clean. Sensor malfunction, incomplete logging, and stochastic action outcomes all introduce noise and uncertainty into plan traces. SLAF's CNF-FSLAF variant handles incomplete observability by maintaining a set of candidate models rather than committing to a single hypothesis.[16] The NOLAM algorithm was explicitly designed to learn action models from noisy trajectories — using a filtering mechanism to discount spurious transitions before extracting model constraints.[13]

The IRALe (Incremental Relational Action Learning) algorithm addresses this problem from a rule-learning perspective. It combines relational reinforcement learning with active learning, starting with an empty rule set and revising it incrementally as new observations arrive — generalising rules when appropriate and specialising them when contradicted by new examples.[1]

4.2 Online Learning

Most early AML algorithms were designed for the offline setting: a dataset of traces is provided, and the model is learned in batch. However, agents that explore unknown environments — a common scenario in robotics and autonomous systems — must learn their domain model incrementally, updating it with each new action execution.[4]

The 3SG (3-Set Generator) algorithm of Certicky (2014) provided a real-time online method that maintains three hypothesis sets for each action — conditions that are always true before execution, always added after, and always deleted after — and updates them at each step in constant time.[17] More recently, Lamanna and colleagues (2021) presented a fully online PDDL learning algorithm that interleaves model learning with plan execution, managing action failures and incomplete information in real time.[4]

4.3 Temporal and Numeric Domain Learning

Standard AML research addresses STRIPS-like propositional domains. Temporal domains, where actions have durations and can execute concurrently, present additional complexity. TempAMLSI extended existing techniques via a STRIPS translation layer that converts temporal domains into non-temporal ones, applies an AML algorithm, and then restores the temporal structure — demonstrating that accurate temporal PDDL domains can be learned without human proofreading.[10]

Numeric domain learning — where fluents take continuous values and action effects modify them by mathematical expressions — remains an active research frontier. Work on learning safe numeric planning action models employs constraint satisfaction over linear arithmetic to infer numeric preconditions and effect functions while guaranteeing that the learned model does not permit unsafe action application.[13]

4.4 Neural-Symbolic and Vision-Based Approaches

A growing line of work learns domain models directly from raw perceptual data — video frames or images — rather than symbolic state descriptions. The LatPlan system (Asai and Fukunaga, 2018) introduced a two-stage pipeline: a neural State AutoEncoder maps images to propositional states, and an Action Model Acquisition module then derives symbolic action schemas from the resulting state-transition graphs.[3]

The DSAMA (Dual-Stage Action Model Acquisition) system extended this approach by training Random Forest classifiers to learn the preconditions and effects over the propositionally encoded states, then compiling these classifiers into PDDL formulae. While the resulting models were competitive in accuracy with black-box neural approaches, the complexity of the generated PDDL exposed a translation bottleneck in the SAS+ component of the Fast Downward planner.[3]

Python (Conceptual) Neural-symbolic pipeline — learning action models from image transitions 
import numpy as np
from sklearn.ensemble import RandomForestClassifier
from sklearn.preprocessing import Binarizer

# ─── Stage 1: State AutoEncoder (simplified representation) ───────────────────
# In LatPlan/DSAMA, a trained VAE maps image pairs → binary propositional states.
def encode_states(image_pairs, state_ae):
    """Map raw image observations to binary propositional state vectors."""
    states_before = [state_ae.encode(img_pre)  for img_pre, _  in image_pairs]
    states_after  = [state_ae.encode(img_post) for _,  img_post in image_pairs]
    return np.array(states_before), np.array(states_after)

# ─── Stage 2: Learn precondition classifiers per action ───────────────────────
def learn_preconditions(pre_states, n_propositions):
    """For each proposition p, train a classifier: does p hold before action a?"""
    classifiers = {}
    for p_idx in range(n_propositions):
        labels = pre_states[:, p_idx]       # 1 if prop holds, 0 otherwise
        clf = RandomForestClassifier(n_estimators=50, random_state=42)
        clf.fit(pre_states, labels)
        classifiers[p_idx] = clf
    return classifiers

# ─── Stage 3: Derive add/delete effects ───────────────────────────────────────
def compute_effects(pre_states, post_states):
    """Effects = propositions that change value between pre and post states."""
    delta     = post_states - pre_states
    add_effs  = (delta ==  1)    # true in post, false in pre
    del_effs  = (delta == -1)    # false in post, true in pre
    return np.any(add_effs, axis=0), np.any(del_effs, axis=0)

# ─── Stage 4: Compile to PDDL ─────────────────────────────────────────────────
def compile_to_pddl(action_name, pre_clf, add_mask, del_mask, prop_names):
    """Generate a PDDL action schema string from learned components."""
    # Preconditions: propositions consistently required
    pre_props  = [prop_names[i] for i, clf in pre_clf.items()
                  if clf.predict_proba([[1]])[0][1] > 0.9]
    add_props  = [prop_names[i] for i, v in enumerate(add_mask) if v]
    del_props  = [prop_names[i] for i, v in enumerate(del_mask) if v]

    pre_str = " ".join([f"({p})" for p in pre_props])
    add_str = " ".join([f"({p})" for p in add_props])
    del_str = " ".join([f"(not ({p}))" for p in del_props])

    return (
        f"(:action {action_name}\n"
        f"  :precondition (and {pre_str})\n"
        f"  :effect (and {add_str} {del_str}))"
    )

Section 05Tools, Libraries, and Frameworks

5.1 MacQ — A Unified Framework for Model Acquisition

The MacQ project (Callanan et al., 2022), developed collaboratively by Queen's University and IBM Research AI, represents the most comprehensive unified library for action model acquisition currently available.[23] It surveys over 43 published AML techniques and provides Python implementations of a representative subset, designed around three composable pipeline stages.

The MacQ framework exposes three core components: (1) trace generation, which produces observation traces from existing PDDL planners or the planning.domains API; (2) observation tokenization, which simulates various observability conditions including partial and noisy observation; and (3) model extraction, which applies a chosen AML algorithm to produce learned domain models.[19] Implemented algorithms include Observer, ARMS, SLAF, LOCM, and AMDN (Action Model Discovery with Neural networks), among others.

Python (MacQ) MacQ — comparing multiple extraction techniques on the same trace set 
from macq import generate, extract
from macq.observation import IdentityObservation, AtomicPartialObservation

# Generate a shared set of traces for fair comparison
traces = generate.pddl.VanillaSampling(
    problem_id = 456, plan_len = 15, num_traces = 80
).traces

# Tokenize for full observability
full_obs = traces.tokenize(Token=IdentityObservation)

# Tokenize for partial observability (30% fluents hidden)
partial_obs = traces.tokenize(
    Token=AtomicPartialObservation, percent_missing=0.3
)

# --- Extraction 1: Observer (requires full observability) ---
observer_model = extract.Observer(full_obs)

# --- Extraction 2: ARMS (handles partial observability) ---
arms_model = extract.ARMS(
    partial_obs,
    min_support=2, action_weight=110,
    info_weight=100, threshold=0.6
)

# --- Extraction 3: LOCM (only needs action sequences, no states) ---
# LOCM requires a special tokenizer that strips state information entirely
action_only_obs = traces.tokenize(Token=AtomicPartialObservation, percent_missing=1.0)
locm_model = extract.LOCM(action_only_obs)

# --- Compare: output PDDL domain strings ---
for name, model in {"Observer": observer_model,
                      "ARMS":     arms_model,
                      "LOCM":     locm_model}.items():
    pddl_str = model.to_pddl()
    print(f"\n=== {name} ===\n", pddl_str[:500])

5.2 PDDL Ecosystem Tools

Several additional tools support the AML pipeline. The planning.domains platform (Muise, 2016) provides an online repository of PDDL domain and problem instances, accessible via API, serving as a shared benchmark infrastructure. Fast Downward (Helmert, 2006) is the most widely used deterministic planner for evaluating learned models by testing whether the induced domain can solve held-out problem instances. The VAL validator (Howey et al., 2004) verifies that a given plan is correct with respect to a domain model, making it useful for semantic evaluation of learned domains.

For neural-symbolic pipelines, LatPlan (Asai, 2019) provides a complete implementation of the image-to-STRIPS pipeline using Keras/TensorFlow, supporting puzzle domains such as the 8-puzzle, Lights Out, and Sokoban.[3]

Section 07Evaluation Methodology

Evaluating AML systems poses unique challenges. Two primary evaluation dimensions are employed in the literature. Syntactic evaluation measures the structural fidelity of the learned model against a ground-truth domain: metrics include precision and recall of correctly inferred preconditions and effects at the level of individual literals. Semantic evaluation assesses whether the learned model supports correct planning: a planner is applied to a set of held-out problem instances using the learned domain, and success is measured by plan correctness and coverage rate.

FAMA's compilation approach introduced a formal evaluation methodology that operationalises both dimensions within its planning-based framework, enabling automatic assessment without requiring access to the ground-truth model in the semantic case.[26] Standard benchmark domains used in AML evaluation include Blocksworld, Logistics, Driverlog, Zenotravel, Depots, and Satellite — all available through the ICAPS International Planning Competition benchmark archives.

Python Syntactic evaluation — precision and recall of learned action schemas 
def evaluate_learned_model(learned_model, ground_truth_model):
    """
    Compute per-action precision and recall of learned preconditions/effects.
    Both models are dicts: {action_name: ActionSchema}
    ActionSchema has .precond, .add, .delete as frozensets of literal strings.
    """
    results = {}

    for action_name in ground_truth_model:
        gt     = ground_truth_model[action_name]
        lrn    = learned_model.get(action_name)

        if lrn is None:
            results[action_name] = {"precision": 0.0, "recall": 0.0}
            continue

        # Pool all literals from precond + effects for comparison
        gt_literals  = gt.precond  | gt.add  | gt.delete
        lrn_literals = lrn.precond | lrn.add | lrn.delete

        tp = len(gt_literals & lrn_literals)
        fp = len(lrn_literals - gt_literals)
        fn = len(gt_literals - lrn_literals)

        precision = tp / (tp + fp) if (tp + fp) > 0 else 0.0
        recall    = tp / (tp + fn) if (tp + fn) > 0 else 0.0

        results[action_name] = {
            "precision": round(precision, 3),
            "recall":    round(recall,    3),
            "f1":        round(2*(precision*recall)/(precision+recall+1e-9), 3)
        }

    # Macro-average over all actions
    avg_f1 = sum(r["f1"] for r in results.values()) / len(results)
    print(f"Macro-avg F1: {avg_f1:.3f}")
    return results

Section 08Applications

Action model learning has found application in a diverse range of domains. In robotics, where hand-coding sensor-to-action models is impractical, AML enables robots to acquire planning capabilities from demonstration or self-exploration. SLAM-adjacent approaches draw directly on SLAF-style filtering to construct both map representations and action models simultaneously.[7]

In video game testing and verification, AML pipelines have been applied to automatically derive PDDL models from game execution logs, enabling subsequent formal verification or automated test generation. The workflow requires minimal PDDL modelling expertise: informal action descriptions are used to define predicates, game source code is extended for structured logging, and an AML tool generates the complete domain model.[5]

In business process management, AML techniques can recover formal workflow models from process execution logs, effectively providing a bridge between process mining and symbolic planning. In human-robot collaboration, observed human action sequences serve as training data for learning models of human task structure, which may then be used for robot anticipation and assistance.

Section 09Open Challenges and Future Directions

Despite substantial progress, action model learning remains an open problem in several important respects. The generalisation from ground observations to lifted schemas with typed parameters is non-trivial: most propositional methods require a separate predicate invention step to produce usable PDDL output, and this step is not well-studied. The SIFT algorithm (Segovia-Aguas et al., 2024) is a recent contribution addressing lifted learning — learning full STRIPS schemas with negation directly from action traces without intermediate grounding steps.[8]

Scalability is a persistent concern: as the number of fluents and objects increases, both the hypothesis space and the size of the SAT/planning encodings grow rapidly. Neural approaches offer scalability but sacrifice interpretability and formal guarantees. The integration of large language models (LLMs) as sources of domain knowledge — particularly for predicate invention and schema suggestion — is a nascent but rapidly growing research direction that may substantially reduce the number of observations required for accurate learning.

Safety and correctness guarantees represent another frontier. A learned model that is overly optimistic — one that permits actions that are actually inapplicable — may lead a planner to generate unsafe plans. Work on safe learning of PDDL domains with conditional effects (Chitnis et al., 2023) directly addresses this concern, proposing algorithms that provide formal guarantees that the learned model does not over-approximate the true transition function.[9]

The integration of AML with plan recognition — inferring both the goal and the domain model of an observed agent — is a compelling bidirectional challenge. As the planning community moves toward deployable autonomous systems, the ability to automatically maintain and update domain models in response to environmental change will be essential.
