Introduction

The Performance Imperative

Angular's reputation across the front-end ecosystem has undergone a quiet but dramatic rehabilitation. Once dismissed as the "heavyweight" option favoured only by enterprise teams entrenched in opinionated architecture, the framework has, through versions 17 to 19, delivered a series of performance primitives that rival—and in several measurable dimensions, surpass—any competing toolchain. [1]

Performance is not a single axis. A web application can be fast to first byte (TTFB) yet slow to become interactive; it can render blazingly quickly on a desktop while crawling on a mid-range Android handset; it can score green on Lighthouse while delivering a janky scroll experience at 30 fps. Optimising Angular therefore demands reasoning about multiple distinct problem spaces simultaneously: bundle size and code-splitting strategy; image delivery and layout stability; change-detection frequency and computational cost; and the architecture of server-to-client HTML transfer.

This article examines five of the most consequential mechanisms Angular exposes for performance engineering. Each section combines conceptual foundations with production code and, where available, empirical benchmarks from the official documentation and third-party analysis. The reader is assumed to be fluent in TypeScript and comfortable with Angular's component model, dependency injection, and reactive programming via RxJS or Signals.

Section 01

Deferrable Views

Introduced as a stable feature in Angular 17 (and becoming the most-praised release highlight in the 2024 developer survey [1]), deferrable views allow a component template to declare that certain subtrees—and their full dependency graphs, including directives, pipes, and CSS—should be loaded lazily, on demand, rather than as part of the initial bundle. This is fundamentally different from the older route-level lazy loading: it operates at the template level, giving developers surgical control over exactly which rendering work happens at which moment. [2]

"Unlike traditional lazy loading that works at the route level, deferrable views let you defer the loading of specific components, directives, or template sections until certain conditions are met."

— TutsCoder Angular Guide, 2025 [5]

The practical consequence is a measurable reduction in the initial JavaScript payload delivered to the browser. Deferring heavy components—those that bundle large third-party packages such as charting or rich-text-editing libraries—until they are needed can dramatically improve Largest Contentful Paint (LCP) and Time to First Byte (TTFB). [3] Deferring components that are below the fold also eliminates Cumulative Layout Shift (CLS) contributions from off-screen content rendering before the layout has settled. [4]

Anatomy of the @defer Block

A deferrable view is declared using the @defer template block. Alongside it, Angular provides three companion blocks that manage the states of the deferred content's lifecycle: @placeholder, @loading, and @error. [6] 

HTML (Angular Template)<!-- dashboard.component.html -->

@defer (on viewport) {
  <!-- Lazily loaded: heavy chart component and all its deps -->
  <app-analytics-chart [data]="analyticsData" />
  <app-data-grid       [rows]="gridRows"     />
}
@placeholder (minimum 200ms) {
  <!-- Lightweight placeholder shown before deferred content triggers -->
  <div class="chart-skeleton"></div>
}
@loading (minimum 300ms; after 150ms) {
  <!-- Shown during the network fetch -->
  <app-spinner size="md" />
}
@error {
  <p class="error-msg">Failed to load analytics dashboard.</p>
}

A critical implementation detail: always provide a @placeholder. Without it, there is no content in the DOM before the defer trigger fires, which means the browser cannot reserve the requisite layout space. This results in a CLS contribution as the component materialises. [4] The minimum parameter on @placeholder prevents a flash when deferred content loads faster than the human eye can resolve.

⚠ Avoid Cascading Loads

Nesting @defer blocks creates cascading asynchronous loads where each resolved block triggers the next. This produces a "waterfall" of network requests, compounding latency. Prefer shallow defer hierarchies and batch related heavy components within a single block. [4]

Trigger Catalogue

The on keyword accepts a set of built-in triggers that control exactly when the deferred block begins fetching and rendering its content. A single @defer block may combine a when condition (a boolean expression evaluated reactively) with an on trigger. [6]

Trigger Description Typical Use Case
on viewport Fires when the placeholder enters the browser viewport via IntersectionObserver. Below-the-fold widgets, infinite scroll items, off-screen panels.
on interaction Fires on first user interaction with the placeholder (click or keydown). Dialogs, dropdowns, context menus, rich-text editors.
on hover Fires when the pointer moves over the placeholder element. Tooltip-triggered heavy components, preview panes.
on idle Uses requestIdleCallback to load when the browser is not busy. Low-priority analytics panels, background data fetching.
on timer(n) Fires after a fixed millisecond delay. Staggered onboarding flows, delayed promotional content.
when <expr> Fires when a reactive boolean expression turns true. Feature-flag gating, permission-based feature panels. 
HTML (Angular Template)<!-- Using 'when' with a signal for permission-gated content -->
@defer (when isAdmin(); prefetch when isNearAdminPanel()) {
  <app-admin-console />
}
@placeholder {
  <div class="admin-placeholder">Admin tools require elevated access.</div>
}

<!-- Combining triggers: load on idle but also on explicit interaction -->
@defer (on idle; on interaction) {
  <app-recommendation-engine />
}
@placeholder {
  <button class="btn-secondary">Show Recommendations</button>
}
ℹ Duplicate Triggers

Angular prohibits duplicate triggers of the same type within a single @defer block. Combining two on viewport triggers, for example, will produce a compile-time error. Combine different trigger types to achieve additive conditions. [6]

Prefetch Strategies

Beyond controlling when a block renders, Angular's defer API separates the concern of when to prefetch (download the JS bundle) from when to render. This allows teams to implement sophisticated strategies such as downloading the heavy bundle during browser idle time while only rendering once the user actually scrolls into view: 

TypeScript + HTML// statistics.component.ts
@Component({
  selector: 'app-statistics',
  template: `
    @defer (on viewport; prefetch on idle) {
      <app-complex-chart [dataset]="chartData" />
    }
    @placeholder {
      <div class="chart-area-placeholder"></div>
    }
  `,
})
export class StatisticsComponent {
  readonly chartData = input<ChartDataset[]>();
}

In this pattern, the JavaScript for <app-complex-chart> is fetched as soon as the browser reports an idle period—meaning it is likely already in the cache before the user has scrolled. Rendering is still deferred to the viewport intersection event, preserving LCP hygiene. This is arguably the most powerful two-phrase cadence in Angular's performance toolkit. [7]

Figure 1. The @defer block lifecycle: from placeholder through loading to resolved content, with the separate prefetch channel

The @for + @defer Interaction

A subtle architectural decision arises when deferring components within loops. Consider the two placements: 

HTML (Angular Template)<!-- Option A: @defer wraps @for
     Creates ONE deferrable view managing N component instances.
     Lower overhead—preferred when components are homogeneous. -->
@defer (on viewport) {
  @for (item of items; track item.id) {
    <app-feed-card [data]="item" />
  }
}
@placeholder { <app-feed-skeleton /> }

<!-- Option B: @defer inside @for
     Creates N independent deferrable views, each with its own trigger.
     Use when each iteration has a distinct heavy dependency. -->
@for (item of items; track item.id) {
  @defer (on interaction) {
    <app-rich-preview [id]="item.id" />
  }
  @placeholder { <app-card-summary [item]="item" /> }
}

Angular must manage a separate embedded view context for every @defer block in the template. Placing @defer inside @for for a list of 500 items creates 500 deferrable view contexts with independent IntersectionObserver instances. The approach is only warranted when each row genuinely needs a distinct trigger or distinct error handling. [6]

Section 02

Image Optimisation with NgOptimizedImage

Images are, statistically, one of the largest contributors to poor Core Web Vitals scores across the web. The LCP element—the largest visible content in the initial viewport—is a hero image in the majority of pages. Misconfiguring a single hero image can single-handedly push a page into the "Needs Improvement" or "Poor" LCP band. [8]

Angular's answer is the NgOptimizedImage directive, developed in partnership with Google's Chrome Aurora team. First released as stable in Angular 15 (backported to 13.4.0 and 14.3.0), it packages the most impactful image loading best practices into a declarative API that is little more complex than writing a standard <img> element. [9]

Setup and Activation

NgOptimizedImage is exported from @angular/common. In a standalone component, it is added directly to the imports array. In an NgModule application, it is added to the module's imports. The directive is activated by replacing the src attribute with ngSrc. [10] 

TypeScript — Standalone Componentimport { Component } from '@angular/core';
import { NgOptimizedImage } from '@angular/common';

@Component({
  selector: 'app-product-card',
  standalone: true,
  imports: [NgOptimizedImage],
  template: `
    <!-- LCP hero image: marked priority -->
    <img
      ngSrc="https://cdn.example.com/hero.jpg"
      width="1200"
      height="630"
      priority
      alt="Product hero image"
    />

    <!-- Non-priority image: lazy-loaded by default -->
    <img
      ngSrc="https://cdn.example.com/thumbnail.jpg"
      width="400"
      height="300"
      alt="Product thumbnail"
    />
  `,
})
export class ProductCardComponent {}

LCP Prioritisation

Marking an image as priority triggers several automatic behaviours that together minimise LCP: [9]

  • Sets fetchpriority="high" on the <img> element, instructing the browser's preload scanner to fetch the resource immediately.
  • Overrides the default lazy-loading behaviour by setting loading="eager".
  • Generates a <link rel="preconnect"> hint to the image origin, reducing DNS and TCP handshake latency before the fetch begins.
  • In development mode, warns if the marked image is not actually the LCP element, preventing misclassification.

Non-priority images receive the opposite treatment by default: loading="lazy" is applied automatically, deferring the download until the image approaches the viewport. This eliminates the common mistake of eagerly loading dozens of below-the-fold thumbnails. [11]

⚠ Width & Height Are Mandatory

NgOptimizedImage enforces explicit width and height attributes. Their absence causes a runtime error in development. This is intentional: the browser requires the aspect ratio information to reserve layout space before the image loads, which directly reduces CLS. Percentage-based dimensions via CSS are not a substitute—they must be concrete pixel values. [12]

Image Loaders and CDN Integration

The directive's most powerful capability is its integration with image CDN loader functions. A loader transforms the value of ngSrc into a full CDN URL, enabling automatic srcset generation. The browser can then select the appropriately sized image for the current display density and viewport, eliminating the most common image-size mistake of serving desktop-resolution images to mobile clients. [8] 

TypeScript — app.config.ts (built-in Cloudinary loader)import { ApplicationConfig } from '@angular/core';
import { provideCloudinaryLoader } from '@angular/common';

export const appConfig: ApplicationConfig = {
  providers: [
    provideCloudinaryLoader('https://res.cloudinary.com/my-cloud'),
  ],
};
TypeScript — Custom Image Loaderimport { IMAGE_LOADER, ImageLoaderConfig } from '@angular/common';

// Provide in app.config.ts providers array
{
  provide: IMAGE_LOADER,
  useValue: (config: ImageLoaderConfig) => {
    const { src, width, loaderParams } = config;
    const quality = loaderParams?.['quality'] ?? 75;
    return `https://img.example.com/${src}?w=${width}&q=${quality}&fmt=webp`;
  },
}

Angular 17.2 extended NgOptimizedImage with support for base64-encoded Low-Quality Image Placeholders (LQIPs). While the actual image loads, a blurred, base64-encoded thumbnail is displayed as a CSS background-image, providing a perceptually complete experience without layout shift. [13] 

HTML — LQIP Placeholder (Angular 17.2+)<img
  ngSrc="products/hero.jpg"
  width="1200"
  height="630"
  placeholder="data:image/png;base64,iVBORw0KGgoAAAANSUhEUgAAAAQAAAACCAYAAAB/..."
  [placeholderConfig]="{ blur: true }"
  priority
  alt="Hero image"
/>
✔ Angular 17 Expanded Guidance

From Angular 17 onwards, image performance warnings are emitted for all Angular applications, not only those using NgOptimizedImage. If the framework detects that the LCP image is being lazy-loaded, or that a downloaded image file is significantly larger than its rendered size, it logs a development-mode warning. [8]

Section 03

Mitigating Zone Pollution

What is Zone.js?

Zone.js is a library that monkey-patches browser APIs—setTimeout, setInterval, Promise, fetch, DOM event listeners, and others—to intercept asynchronous tasks and report their completion to a central scheduler. Angular uses Zone.js as a signalling mechanism: when Zone.js reports that a task has finished, Angular schedules a change detection cycle to check whether any component state has changed and whether the DOM needs updating. [14]

This is the foundation of Angular's original "magic"—developers can mutate component properties in event handlers, HTTP callbacks, or timer callbacks, and Angular will automatically refresh the view without any explicit notification call. The framework achieves this because Zone.js notifies it every time any of these async contexts settle.

Zone Pollution: Causes and Cost

Zone pollution occurs when asynchronous operations that have no effect on application state are run inside Angular's zone. Because Zone.js has no way to know a priori whether a given callback will mutate component state, it treats every intercepted operation as a potential state change and triggers a change detection cycle. The most common sources are: [14]

  • Frequent timers: requestAnimationFrame loops, high-frequency setInterval polling (e.g., real-time dashboards refreshing at 60 fps).
  • Third-party libraries that set up internal async loops—charting libraries like Plotly, animation engines, analytics SDKs—typically without knowledge of Angular's zone. [15]
  • DOM event listeners on broad targets such as window or document (scroll, mousemove, resize) that fire at high frequency.
  • WebSocket message handlers receiving data continuously from a streaming server.

"Zone pollution occurs when scheduled tasks or microtasks don't make changes in the data model, making change detection unnecessary. This leads to performance degradation, wasted CPU resources, and sluggish UI."

— Medium: Zone Pollution in Angular, 2026 [15]

Mitigation with NgZone.runOutsideAngular()

The primary remedy is the NgZone service, injected via Angular's DI system. Its runOutsideAngular() method executes a callback in a context that Zone.js does not intercept. Angular will not schedule change detection when that callback's async operations complete. 

TypeScript — runOutsideAngularimport { Component, NgZone, OnInit, OnDestroy, inject } from '@angular/core';
import * as Plotly from 'plotly.js-dist-min';

@Component({
  selector: 'app-live-chart',
  template: '<div id="plotly-host"></div>',
})
export class LiveChartComponent implements OnInit, OnDestroy {
  private readonly ngZone = inject(NgZone);
  private pollingId: ReturnType<typeof setInterval>;

  ngOnInit(): void {
    // Initialise Plotly outside Angular: its internal event listeners
    // won't trigger change detection.
    this.ngZone.runOutsideAngular(() => {
      Plotly.newPlot('plotly-host', initialData, layout);

      // High-frequency polling: 60× per second but state doesn't change
      this.pollingId = setInterval(() => {
        const newData = this.fetchLivePoint();
        Plotly.extendTraces('plotly-host', { y: [[newData]] }, [0]);
      }, 16); // ~60 fps
    });
  }

  ngOnDestroy(): void {
    clearInterval(this.pollingId);
  }
}

When a state mutation does occur outside the zone and the view must be updated, two re-entry strategies are available: call NgZone.run() to schedule full change detection, or inject ChangeDetectorRef and call detectChanges() for a targeted, local update. [16] 

TypeScript — Selective zone re-entryimport { Component, NgZone, ChangeDetectorRef, inject, signal } from '@angular/core';

@Component({
  selector: 'app-sensor-feed',
  template: `<p>Latest reading: {{ latestReading() }} °C</p>`,
})
export class SensorFeedComponent {
  private readonly ngZone = inject(NgZone);
  private readonly cdr    = inject(ChangeDetectorRef);
  readonly latestReading = signal(0);

  startStream(socket: WebSocket): void {
    this.ngZone.runOutsideAngular(() => {
      socket.addEventListener('message', (event) => {
        const reading = JSON.parse(event.data).temperature;

        if (reading !== this.latestReading()) {
          // Re-enter zone only when the value actually changed
          this.ngZone.run(() => {
            this.latestReading.set(reading);
          });
        }
      });
    });
  }
}

Zone Coalescing

Angular 16+ exposes provideZoneChangeDetection with coalescing options that batch multiple rapid Zone.js signals into a single change detection pass. eventCoalescing collapses multiple events fired within the same microtask queue flush; runCoalescing reduces redundant scheduler calls in the same task. [17] 

TypeScript — app.config.tsimport { ApplicationConfig, provideZoneChangeDetection } from '@angular/core';

export const appConfig: ApplicationConfig = {
  providers: [
    provideZoneChangeDetection({
      eventCoalescing: true,   // Collapses rapid event bursts
      runCoalescing:   true,   // Dedups scheduler calls
    }),
  ],
};

The Zoneless Horizon

Angular's long-term trajectory is toward zoneless change detection, where Zone.js is removed entirely and the framework relies on Signals (and, for backwards compatibility, ChangeDetectorRef and async pipe) to know exactly which components need re-rendering. As of Angular 20 this mode is considered mature enough for production use. [17] 

TypeScript — Enabling Zoneless (Angular 18+)import { ApplicationConfig, provideExperimentalZonelessChangeDetection } from '@angular/core';

export const appConfig: ApplicationConfig = {
  providers: [
    provideExperimentalZonelessChangeDetection(),
  ],
};

// Also remove Zone.js from polyfills in angular.json:
// "polyfills": []  ← remove "zone.js"
Section 04

Slow Computations in Change Detection

Even after eliminating zone pollution, an application can remain sluggish if the work Angular performs during each change detection cycle is itself expensive. Every cycle, Angular synchronously evaluates all template expressions for every component whose change detection strategy permits it, and executes four lifecycle hooks: ngDoCheck, ngAfterContentChecked, ngAfterViewChecked, and ngOnChanges. [18]

ℹ Change Detection Cycle Work

In a Default-strategy component tree, Angular walks every component from root to leaf on every cycle. A single expensive template expression or a computationally heavy ngDoCheck body blocks the entire synchronous traversal, delaying all subsequent component updates. Angular runs all computations sequentially on the main thread. [18]

Profiling with Angular DevTools

Angular DevTools (a Chrome/Firefox extension from the Angular team) includes a change detection profiler. In the Profiler tab, recording a session and clicking on a cycle bar reveals a breakdown of how long Angular spent in each component's template evaluation and lifecycle hooks. This is the definitive starting point for diagnosing slow-computation bottlenecks. [18]

Common antipatterns revealed by the profiler include:

  • Method calls in templates: {{ formatData(item) }} calls the function on every cycle. If formatData sorts, filters, or maps a large array, the cost compounds with every change detection pass.
  • Heavy ngDoCheck bodies: ngDoCheck fires on every cycle—including cycles triggered by unrelated components. Even an O(n) array comparison in a large list becomes expensive at 60 cycles/second. [19]
  • Layout-triggering DOM reads in lifecycle hooks: Reading offsetHeight, getBoundingClientRect(), or scrollTop inside ngAfterViewChecked forces the browser to perform a synchronous reflow before the current paint can complete.

Pure Pipes as Computed Cache

A pure pipe in Angular is guaranteed by the framework to be idempotent: given the same inputs, it will always produce the same output. Angular exploits this contract by memoising the last result: if a pure pipe is called again with identical arguments, Angular returns the cached value rather than re-executing the transform function. [18] 

TypeScript — Pure Pipe (memoised by Angular)import { Pipe, PipeTransform } from '@angular/core';

@Pipe({ name: 'expensiveSort', pure: true }) // pure: true is the default
export class ExpensiveSortPipe implements PipeTransform {
  transform(items: Item[], sortKey: string): Item[] {
    // Executed only when `items` reference or `sortKey` changes
    return [...items].sort((a, b) => a[sortKey] < b[sortKey] ? -1 : 1);
  }
}

HTML — Template Usage<!-- Angular evaluates the sort only when `products` or `sortKey` changes.
     Without the pipe, sorting would happen on every change detection cycle. -->
@for (product of (products | expensiveSort: sortKey); track product.id) {
  <app-product-row [product]="product" />
}

Multi-Result Memoisation

Pure pipes cache only the most recent result. A multi-argument transform with many distinct input combinations benefits from a richer memoisation strategy—for example, a Map-backed cache keyed on the argument tuple: 

TypeScript — Manual Memoisationimport { Pipe, PipeTransform } from '@angular/core';

@Pipe({ name: 'localeDateFormat', pure: true })
export class LocaleDateFormatPipe implements PipeTransform {
  private readonly cache = new Map<string, string>();

  transform(timestamp: number, locale: string, format: string): string {
    const key = `${timestamp}::${locale}::${format}`;
    if (this.cache.has(key)) return this.cache.get(key)!;

    const result = new Intl.DateTimeFormat(locale).format(new Date(timestamp));
    this.cache.set(key, result);
    return result;
  }
}

The OnPush Change Detection Strategy

The most impactful structural optimisation available is adopting the OnPush change detection strategy on components. With OnPush, Angular skips the subtree rooted at a component unless one of four conditions is met: an @Input() reference changes, an event originates from the component, the async pipe emits, or a Signal within the template changes. [17] 

TypeScript — OnPush Strategyimport { Component, ChangeDetectionStrategy, input } from '@angular/core';

@Component({
  selector: 'app-product-card',
  changeDetection: ChangeDetectionStrategy.OnPush,
  template: `
    <h3>{{ product().name }}</h3>
    <p>{{ product().price | currency }}</p>
  `,
})
export class ProductCardComponent {
  // Signal-based input: change detection fires only when reference changes
  readonly product = input.required<Product>();
}
Figure 2. Default strategy checks the entire tree; OnPush skips subtrees marked clean, reducing evaluation cost proportionally to the number of unchanged components
Section 05

Hydration

Server-side rendering (SSR) solves the initial load problem by generating HTML on the server and sending a fully rendered page to the browser. The user sees content immediately—before a single byte of JavaScript has been parsed. However, the rendered HTML is static: it has no event listeners, no Angular component instances, no reactive state. The process of converting this static HTML into a live Angular application is called hydration. [20]

The Pre-Hydration Problem

Before Angular 16 introduced non-destructive hydration, the SSR/client lifecycle was destructive: Angular would receive the server-rendered HTML, immediately discard it, and re-render the entire application from scratch on the client. This caused a visible flicker—the entire DOM disappeared and reappeared—and negated most of the perceived performance benefit of SSR for metrics like Cumulative Layout Shift. [21]

"Historically when you used SSR with Angular, Angular would just delete all of the DOM and replace it with newly rendered elements, but in 2025 we have hydration. This means that instead of deleting the elements, we wire up all of the behaviors and functionality into the existing DOM."

— fluin.io: The State of SSR in Angular, 2025 [21]

Non-destructive hydration, stable from Angular 17, preserves the server-rendered DOM. Angular walks the existing DOM, reconciles it with its internal component tree, and attaches event listeners and reactive state to the already-present nodes. Zero nodes are destroyed and re-created during this process. The result is a seamless transition from static to interactive with no layout shift and no visual flicker. [20]

Enabling Hydration

If the application was created or migrated using the Angular CLI's SSR support (ng add @angular/ssr), hydration is enabled automatically. For custom server setups, provideClientHydration is imported from @angular/platform-browser and added to the application's bootstrap providers: 

TypeScript — app.config.tsimport { ApplicationConfig }          from '@angular/core';
import { provideClientHydration,
         withEventReplay }             from '@angular/platform-browser';
import { provideServerRendering }       from '@angular/platform-server';

export const appConfig: ApplicationConfig = {
  providers: [
    provideClientHydration(
      withEventReplay(), // Buffer events during hydration and replay
    ),
  ],
};

withEventReplay(), available from Angular 18, solves the "hydration gap": the window of time between the server HTML being displayed and Angular finishing its hydration pass. Without event replay, user interactions during this gap are silently dropped. With it, they are buffered and replayed once the component tree is live.

Opting Out: ngSkipHydration

Components that manipulate the DOM directly—using Renderer2, ElementRef.nativeElement, or third-party libraries that bypass Angular's view engine—cannot be hydrated safely. The server-rendered DOM and the client's expected DOM will diverge, causing Angular's reconciliation to fail. Such components must opt out of hydration using the ngSkipHydration host attribute: [21] 

TypeScript — Opting Out of Hydration@Component({
  selector: 'app-legacy-chart-wrapper',
  host: {
    'ngSkipHydration': 'true',
  },
  template: '<div id="d3-host"></div>',
})
export class LegacyChartWrapperComponent implements AfterViewInit {
  ngAfterViewInit(): void {
    // Direct DOM manipulation via D3 — incompatible with hydration
    const svg = d3.select('#d3-host').append('svg');
    // ...
  }
}

Incremental Hydration (Angular 19+)

Angular 19 introduced Incremental Hydration—a feature announced at Google I/O 2024 and formerly called Partial Hydration. It builds directly on the @defer API: deferred blocks rendered on the server are not hydrated eagerly; instead, they are hydrated on-demand as their corresponding defer trigger fires. [22]

The consequence is a dramatic reduction in the JavaScript that must be parsed and executed before the application is fully interactive. A page with ten heavy @defer blocks hydrates only the blocks that are in the viewport, leaving the rest as inert server-rendered HTML until the user scrolls or interacts. 

TypeScript — Enabling Incremental Hydrationimport { provideClientHydration,
         withIncrementalHydration } from '@angular/platform-browser';

export const appConfig: ApplicationConfig = {
  providers: [
    provideClientHydration(
      withIncrementalHydration(),
    ),
  ],
};
HTML — Template: Incrementally Hydrated Defer Block<!-- This block is rendered on the server but NOT hydrated immediately.
     Hydration occurs when the viewport intersection trigger fires. -->
@defer (hydrate on viewport) {
  <app-comment-thread [postId]="post.id" />
}
@placeholder {
  <div class="comments-placeholder">Loading comments…</div>
}
Figure 3. With Incremental Hydration, only the above-fold block is hydrated at startup. Below-fold blocks remain as inert server HTML until their defer trigger activates
Section 06

Angular Signals: The Reactive Substrate

Signals, introduced as stable in Angular 17, are Angular's native reactive primitive. A Signal is a lightweight wrapper around a value that tracks its consumers. When the signal's value changes, Angular knows exactly which template expressions and computed values need re-evaluation—without walking the entire component tree. [17]

Combined with the OnPush strategy and eventually with zoneless mode, Signals eliminate the two primary sources of unnecessary change detection work: over-broad zone notification and full-tree traversal. The performance contract becomes explicit and local—a signal write schedules only its direct dependants. 

TypeScript — Signal-Driven Component (no zone dependency)import { Component, signal, computed, effect,
         ChangeDetectionStrategy } from '@angular/core';

@Component({
  selector: 'app-cart',
  changeDetection: ChangeDetectionStrategy.OnPush,
  template: `
    <p>Items: {{ itemCount() }}</p>
    <p>Total: {{ total() | currency }}</p>
    <button (click)="addItem()">Add to Cart</button>
  `,
})
export class CartComponent {
  private readonly items = signal<CartItem[]>([]);

  // Derived signals: recomputed only when `items` changes
  readonly itemCount = computed(() => this.items().length);
  readonly total     = computed(() =>
    this.items().reduce((sum, i) => sum + i.price * i.qty, 0)
  );

  addItem(): void {
    this.items.update(current => [...current, newItem]);
    // Angular schedules only this component's template re-evaluation
  }
}

The synergy between Signals and Incremental Hydration is particularly compelling. Hydration knows precisely which components are driven by Signals and can defer attaching Angular's reactive graph to each component until that component's @defer block hydrates. The JavaScript for inactive components is never parsed.

✔ The Convergent Architecture

The emerging best practice converges on four concurrent choices: Signals for reactive state, OnPush change detection, @defer blocks for non-critical subtrees, and Incremental Hydration for SSR pages. Together, these minimise Time to Interactive, eliminate redundant re-renders, and reduce the JavaScript parse budget to the minimum necessary for the current viewport. [22]

Strategy Decision Matrix

Technique Primary Metric Improved When to Apply Risk / Trade-off
@defer views LCP, bundle size, TTI Non-critical or below-fold components with heavy dependencies Placeholder required; nested @defer creates waterfall loads
NgOptimizedImage LCP, CLS Any application displaying images Width/height mandatory; CDN loader required for srcset generation
runOutsideAngular CPU, frame rate Third-party libs, animations, high-frequency timers/events Must manually re-enter zone or use CDR for UI updates
Zone coalescing CPU, redundant cycles Event-dense UIs, live data feeds May delay UI updates by one event loop tick
OnPush + Signals CPU, frame rate All presentational components State must be immutable or communicated via Signals/inputs
Pure pipes / memoisation CPU per cycle Expensive template expressions run frequently Cache invalidation complexity for multi-argument functions
SSR + Hydration FCP, TTFB, SEO Content-heavy or SEO-critical pages Avoid direct DOM manipulation; server/browser environment divergence
Incremental Hydration TTI, JS parse budget SSR pages with @defer blocks Angular 19+ developer preview; requires careful defer boundary design
