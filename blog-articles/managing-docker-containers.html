Managing Docker ContainersDriving a process through its lifecycle from the command line

A container is an ordinary Linux process wearing a costume of isolation. Managing one means moving it through a small, well-defined set of states — and knowing exactly which command causes each transition. This guide maps those states, the engine that enforces them, and the discipline that keeps them safe in production.

Building an image is a one-time act of authorship; running the container that comes out of it is an ongoing act of operation, and the two demand different instincts. A well-crafted Dockerfile produces an artifact that is finished and immutable. The container launched from it is anything but: it starts, serves, is inspected, is paused, crashes, is restarted, leaks memory, is killed, and is finally swept away. That arc — not the image — is where uptime is won or lost, where a single mis-set flag becomes a 3 a.m. incident, and where the difference between a hobby project and a production system actually lives.

The good news is that the arc is small and knowable. Underneath the sprawling surface of the Docker command-line interface sits a compact finite-state machine with roughly seven states, and almost every management command you will ever type is a request to move a container from one of those states to another. created becomes running becomes exited becomes gone. Once you can see that machine clearly — the states, the legal transitions, and the verb attached to each edge — the CLI stops being a grab-bag of memorized incantations and becomes a control surface you can reason about.

This guide is organized around that machine. It begins with a short history, because the odd shape of today’s tooling (a client that talks to a daemon that talks to a second daemon that talks to a third program) is a fossil record of real engineering decisions. It then traces what actually happens inside the engine when you press Enter, so that later, when a container is wedged in restarting or refuses to stop, you know which layer to interrogate. From there it works through the lifecycle commands and the observation commands in the order you meet them in real operations, and closes with the practices and the tooling that turn “it runs on my laptop” into “it runs, safely, on a hundred hosts.” Every factual claim is cited; sources are collected at the end in Vancouver order.

◆How to read the command blocks

Terminal blocks show the prompt as $, comments in muted italic, and command output indented beneath. A green dot in the block header marks a lifecycle command (it changes state); an amber dot marks an observation command (it reads state without changing it). That single distinction — mutate versus read — is the most useful mental model you can bring to container operations.

HIST · ORIGINS

A five-minute lightning talk

On 15 March 2013, at PyCon in Santa Clara, a founder named Solomon Hykes was given a five-minute lightning-talk slot and used it to demonstrate a small internal tool his company had built. The company was dotCloud, a platform-as-a-service competing with Heroku; the tool was a way to package an application and its dependencies so that, in his framing, the perennial problem that “shipping code to the server is hard” might finally get easier.1 Lightning talks at PyCon happen on the main stage, which Hykes reportedly did not know; the demo was recorded, leaked to Hacker News, and went viral. Within the year dotCloud had renamed itself Docker and abandoned its original business to pursue the side project full-time.1

What made the tool land was not novelty of mechanism — Linux had shipped process-isolation primitives for years — but novelty of ergonomics. Docker was, as its creators later put it, built on the shoulders of giants: the isolation and resource-control features of the Linux kernel, copy-on-write filesystems, and a set of Git-like semantics for versioning and sharing images.2 Its contribution was to wrap those pieces in a single command-line tool and an image format so approachable that a developer with no background in kernel namespaces could isolate an application in one line. Dirk Merkel’s early and widely-cited write-up captured the appeal precisely: the promise of packaging applications and their dependencies into lightweight containers that move easily between distributions, start quickly, and stay isolated from one another.3 The wider arc of the field — from the low-level Linux Containers (LXC) project, through Docker’s popularization, to cluster orchestrators — was mapped contemporaneously by David Bernstein, who saw early that the interesting action would move from the single container to the management of many.4

From a monolith to a layered stack

The earliest Docker was a single monolithic daemon that did everything, and it drove containers by shelling out to LXC. Neither of those facts survived. The execution machinery was extracted into a dedicated, reusable library, and in 2015 the resulting low-level runtime, runc, was donated to a new, vendor-neutral standards body: the Open Container Initiative (OCI), formed in June 2015 by Docker, CoreOS, and other industry participants under the Linux Foundation.5 The OCI produced the specifications the whole ecosystem now shares — a Runtime Specification describing how to execute an unpacked “filesystem bundle,” an Image Specification describing the on-disk format, and a Distribution Specification for moving images around.5 Around the same time the libcontainer code was contributed and runc was born as the reference implementation, giving anyone building a runtime a well-defined target to hit.6

Why break a working monolith into a client, two daemons, and a runtime? The split solved concrete operational problems. Extracting the low-level runtime behind a specification meant the piece that touches the kernel could be audited, replaced, or hardened independently of everything above it.6 Separating the high-level lifecycle manager from the user-facing daemon meant the long-lived supervision of running containers no longer shared a fate with the front-end process; combined with the per-container shim, that is what lets the Docker daemon be upgraded — or even crash — without taking every running container down with it.7 The layering that can look like incidental complexity is largely the residue of making the system operable, upgradable, and hardenable at scale.

The high-level lifecycle logic became a separate project too. containerd — an industry-standard, high-level runtime that manages the complete container lifecycle on a host, from image pull and storage through supervision and networking — graduated to become a top-level project of the Cloud Native Computing Foundation and is used today by both Docker and Kubernetes.7 The practical upshot for anyone managing containers is that when you type docker, you are standing at the top of a deliberately layered tower. Understanding that tower is the difference between treating the engine as magic and engineering with it, so the next section walks down it, rung by rung.

ARCH · THE ENGINE

What a command actually does

Docker uses a client–server architecture. The thing you type into — the docker command — is a thin client that holds almost no logic of its own; it turns your instruction into an HTTP request and sends it to a long-running background daemon called dockerd, which does the heavy lifting of building, running, and distributing containers.8 The two communicate over a REST API carried on a UNIX socket or a network interface, which is why the client and daemon need not even run on the same machine.8 On Linux that socket is, by default, a UNIX domain socket at /var/run/docker.sock, accessible only to root or to members of the docker group — a detail that will matter a great deal when we discuss security.9

A useful consequence of this design is that the docker command enjoys no special status: it is simply one client of the Engine API, and anything that speaks that API is an equal peer. Docker Compose, the Portainer web UI, a CI pipeline, and the observability tools in the ecosystem section all drive the same daemon through the same interface you do.8 When you automate container management, then, you are not really scripting the CLI so much as talking to the daemon — which is why the mental model of “issue a request, cause a transition” carries over unchanged from a single typed command to a whole orchestration platform.

But dockerd does not itself create containers. When it receives a request to run one, it delegates to containerd, which manages the container’s lifecycle and, for each container, generates an OCI bundle and invokes the low-level runtime. That runtime is runc, the OCI reference implementation, whose single job is to talk to the Linux kernel: it creates the namespaces, configures the control groups, mounts the root filesystem, applies the security profiles, and executes your process — and then exits, leaving a small containerd-shim as the container’s parent.57 That shim is what keeps the container alive and its output collected even if dockerd or containerd is restarted, and it is why you can upgrade the Docker daemon without killing every running container on the host.7

It is worth being concrete about what runc actually receives, because doing so demystifies the entire descent. The OCI runtime specification defines a filesystem bundle: a directory holding the container’s unpacked root filesystem alongside a single config.json that declares everything about how the process is to run — which namespaces to create, which control-group limits to apply, which Linux capabilities to retain, which mounts and devices to set up, the user to run as, and the command to execute.5 containerd is what assembles that bundle, translating your chosen image and the flags you passed to run into that declarative document; runc then does nothing more than enact it faithfully and step aside. Seen from this angle the abstractions line up with startling neatness: every flag on your docker run line is ultimately a field in that config.json, the writable layer is the top of the bundle’s root filesystem, and the whole multi-daemon engine is one long pipeline for turning the words “docker run” into a bundle the kernel can be asked to instantiate. Nothing magical remains — only translation, layer by layer, from your intent down to a system call.

The two kernel primitives everything rests on

Strip away the daemons and a container reduces to a normal process that the kernel has been told to lie to and to fence in. The lying is done by namespaces, which give a process its own private view of some global resource; the Linux manual page enumerates the kinds — among them the mount, PID, network, IPC, UTS, user, and cgroup namespaces — each isolating a different facet so that, for instance, the process inside sees itself as PID 1 with its own process table, its own network interfaces, and its own hostname.10 The fencing is done by control groups (cgroups), a kernel mechanism that organizes processes into hierarchical groups and enforces limits and accounting on the resources they may consume — CPU time, memory, process counts, block I/O.11 Modern distributions and current Docker releases default to the unified cgroup v2 hierarchy, in which all controllers live under a single tree rather than the parallel hierarchies of v1.11

This is the physical meaning behind the abstractions we are about to manipulate. When you later cap a container’s memory with --memory=512m, you are writing a value into that container’s cgroup; when you pause it, you are freezing its cgroup; when you exec into it, you are launching a new process and joining it to the container’s existing namespaces. The commands are ergonomic names for kernel operations. Keeping that in mind turns most container mysteries into ordinary systems questions with ordinary answers.

Images, layers, and the writable layer

One more concept makes the lifecycle commands legible: the relationship between an image and the container that runs it. A Docker image is not a single opaque blob but a stack of read-only layers, each the filesystem delta of one build step, sharing the copy-on-write and Git-like versioning semantics the engine was built on.2 Because those layers are immutable and content-addressed, many containers and many images can share the same underlying layers on disk, which is a large part of why containers are so cheap to create and so fast to start — launching a container copies no files; it merely stacks one thin new layer on top of layers that already exist.

That thin new layer is the container’s writable layer, and it is the physical thing the lifecycle commands allocate and reclaim. docker create adds it; every file the running process writes that is not on a mounted volume lands in it; docker rm deletes it. This is why a stopped container still consumes disk (its writable layer persists), why docker diff can enumerate exactly what changed relative to the image, and why the operations section insists that durable state belong in volumes: anything written only to the writable layer dies with the container.8

◉Why the isolation is not a security boundary by itself

Because a container is a kernel-fenced process rather than a virtual machine, it shares the host kernel. That sharing is what makes containers start in milliseconds and run at near-native speed — measured comparisons find container overhead close to bare metal and generally lower than a hypervisor’s12 — but it also means the isolation is only as strong as the kernel surface exposed to the container. This is the seed of every hardening practice in the operations section.

LIFE · THE STATE MACHINE

The lifecycle and its commands

A container is always in exactly one state, and you can always read that state directly: docker inspect reports it as the .State.Status field, whose value is one of a small, fixed vocabulary — created, running, paused, restarting, removing, exited, or dead.13 Figure 1 drew the legal moves between them; the table below is the same information in reference form. Commit the states to memory and the commands arrange themselves naturally, because each command exists to cause one transition.

The container states (docker inspect .State.Status)

runningThe main process (PID 1 inside the container) is alive and scheduled.docker start · runCPU, memory, I/O
pausedEvery process is frozen in place via the cgroup freezer; memory is retained.docker pauseMemory (frozen)
restartingTransient: the daemon is bringing the container back up under a restart policy.--restart policyBriefly, on each attempt

Two of these deserve emphasis up front because they trip up newcomers. First, exited is not the same as removed: by default a container’s filesystem survives its own death, which is exactly what makes post-mortem debugging possible — you can inspect the final state and read the logs of something that crashed hours ago.13 Second, the top-level commands you already know (docker run, docker ps, docker logs…) are convenience aliases for their docker container equivalents (docker container run, docker container ls…); the longer forms are grouped under a single management command that is worth skimming once to see the full inventory.14

create, start, and run: bringing a container to life

The clearest way to understand docker run is to see it as a composite. docker create takes an image and produces a container in the created state — it allocates the writable layer and records the configuration, but starts nothing. docker start then takes that created (or previously stopped) container and executes its process, moving it to running. docker run simply does both at once, pulling the image first if it is not present locally.15 The equivalence is worth internalizing because it explains a great deal of behavior: anything you can configure at run time is really being configured at create time and is therefore fixed for the container’s life; to change most settings you must create a new container, not reconfigure a running one.

run ≡ create + start
# These two lines are equivalent to the single `docker run` beneath them:
$ docker create --name web -p 8080:80 nginx:1.27-alpine
3f2a1c9e...b7
$ docker start web
web

# One step. -d detaches (runs in the background); the daemon prints the full id.
$ docker run -d --name web -p 8080:80 nginx:1.27-alpine
4bed76d3ad428b889c56c1ecc2bf2ed95cb08256db22dc5ef5863e1d03252a19

Three families of flags account for most of what you will pass to run in practice. The first controls how you attach to the process: -d / --detach runs it in the background, while -it (the combination of --interactive and --tty) gives you a live terminal into it, which is how you get an interactive shell.15 The second controls the container’s connections to the outside world: --name assigns a stable identifier so you need not paste hashes, -p publishes a container port to the host, -v or --mount attaches storage, and -e sets environment variables.15 The third, which we return to at length later, governs lifetime and limits: --rm, --restart, --memory, --cpus, and their kin.

an interactive, self-cleaning container
# --rm removes the container (and its anonymous volumes) the moment it exits,
# so throwaway debugging shells never accumulate as clutter.
$ docker run --rm -it alpine:3.20 sh
/ # cat /etc/os-release | head -1
NAME="Alpine Linux"
/ # exit
# ...container is already gone. Verify with `docker ps -a` — nothing left behind.

The --rm flag is one of the highest-leverage habits in day-to-day use: it removes the container, and any anonymous volumes created with it, as soon as the process exits, which keeps a workstation from silting up with hundreds of dead one-off containers.15 Combining it with a restart policy is an error, for the obvious reason that a container cannot both be automatically restarted and automatically deleted.15

Foreground, detached, and attaching

By default docker run runs in the foreground: your terminal is attached to the container’s process, its output streams to your screen, and pressing Ctrl-C sends SIGINT to that process, which usually stops it. Adding -d detaches instead, returning your prompt and leaving the container running in the background.15 The two are bridged by docker attach, which reconnects your terminal to a running container’s main process — its standard input, output, and error — as though you had never left.14 The catch that surprises people is that attach connects you to PID 1 itself, so a reflexive Ctrl-C while attached will signal and likely stop the container; to leave without killing it, use the detach sequence Ctrl-P Ctrl-Q. This is exactly the distinction from exec: attach rejoins the existing main process, whereas exec starts a separate one, which is why exec is the safer way to poke at a container you care about.

attaching without accidentally stopping it
# Reconnect to a detached container's live streams:
$ docker attach web
# ...to leave WITHOUT stopping it, press:  Ctrl-P Ctrl-Q   (never Ctrl-C)

# Block until a container exits, then surface its exit code (handy in scripts):
$ docker wait batch-job
0

stop, kill, and restart: ending and cycling

Stopping a container is a negotiation, not an execution. docker stop sends the container’s main process a SIGTERM and waits — ten seconds by default — for it to shut down gracefully; only if the grace period elapses does the daemon follow up with an unignorable SIGKILL.14 That grace window is precisely where an application flushes buffers, finishes in-flight requests, and closes connections, which is why a well-behaved server installs a SIGTERM handler and why stateful workloads that need longer should be given a longer timeout. docker kill, by contrast, skips the negotiation: it delivers SIGKILL (or a signal of your choosing with -s) immediately.14 Reach for kill only when a container is genuinely wedged; reach for stop everywhere else.

graceful vs. forceful termination
# Ask nicely, but only wait 30s before SIGKILL (good for a slow, stateful app):
$ docker stop -t 30 db
db

# Send a specific signal without stopping — e.g. tell nginx to reopen its logs:
$ docker kill -s HUP web
web

# restart = stop (with the same grace period) followed by start:
$ docker restart -t 20 web
web

A subtlety hides in restart: because it is a stop followed by a start, it passes through the transient restarting state and honors the same grace period, so a container with a long-running graceful shutdown will take just as long to cycle as it does to stop. If a container is already exited, start and restart both simply bring it back; the difference only matters when it is currently running.

pause and unpause: freezing without stopping

Occasionally you want a container to stop consuming CPU without losing its in-memory state — to quiesce it during a host maintenance window, say, or to take a consistent snapshot. docker pause freezes every process in the container using the kernel’s cgroup freezer, which the processes cannot intercept or refuse, and docker unpause thaws them.14 A paused container still holds its memory; it simply is not scheduled. This is distinct from stopping: nothing is signaled, nothing shuts down, and no restart policy fires.

rm and pruning: reclaiming what is finished

A stopped container is inert but not free — it still occupies its writable layer on disk. docker rm deletes a container; by default it refuses to remove one that is still running, a guard rail you can override with -f (which kills the process first) and extend with -v to also drop the container’s anonymous volumes.14 Because dead containers accumulate quietly, Docker also provides bulk reclamation through the prune family, and the idiom below — listing stopped-container ids and feeding them to rm — is worth knowing even though docker container prune now does the same thing more safely.

removing and reclaiming
# Remove one stopped container:
$ docker rm web

# Force-remove a running one (SIGKILL, then delete) and its anonymous volumes:
$ docker rm -f -v stuck-worker

# Reclaim every stopped container at once:
$ docker container prune -f
Deleted Containers:
a1b2c3... 9f8e7d...
Total reclaimed space: 214.6MB

Restart policies: automatic recovery

Everything so far has been manual: you issue a verb, the container obeys. Restart policies delegate one class of transition — exit back to running — to the daemon itself, so a container that dies is brought back without a human in the loop. You attach a policy at creation with --restart, and there are exactly four, each with a precise contract.16

The four restart policies (--restart)
PolicyRestarts when…Survives a daemon restart / reboot?Use it for
noNever (this is the default).NoOne-off jobs, migrations, debug containers.
on-failure[:N]The process exits non-zero; optionally capped at N attempts.Only if it had failed and not exhausted NBatch workers where a clean exit means “done.”
alwaysThe process exits for any reason, including a clean exit.Yes — even if you had manually stopped itRarely the right choice; see below.
unless-stoppedAny exit, but not if you deliberately stopped it.Yes, unless you stopped it firstThe default choice for long-running services.

The distinction between always and unless-stopped is the one that bites: both bring a container back after a crash and after the host reboots, but always will resurrect a container you intentionally stopped the next time the daemon starts, whereas unless-stopped remembers your intent and leaves it down.16 For most production services, unless-stopped is the right balance of automatic recovery and manual control.

Crucially, a restart policy is a safety net, not a fix. To keep a broken container from thrashing the host, Docker inserts an exponential back-off between attempts: the delay starts at 100 milliseconds and doubles on each retry, up to a ceiling of one minute, and it resets only after the container has stayed up for at least ten seconds.16 The operational consequence is easy to miss: a container stuck in a crash loop will, after enough failures, appear in docker ps as restarting yet sit idle for up to a minute between attempts — it looks alive but serves nothing. A climbing restart count is therefore always a signal to investigate the root cause, never evidence that the policy is “working,” and you can read that count straight from the container’s metadata.13

is this policy actually healthy, or looping?
# A high RestartCount is a red flag, not a green light:
$ docker inspect --format '{{.Name}} restarts={{.RestartCount}} exit={{.State.ExitCode}}' api
/api restarts=37 exit=1

# Pause a policy without recreating the container, to break the loop and debug:
$ docker update --restart no api
$ docker logs --tail 50 api   # now read why it keeps dying

Watching a full lifecycle

It is worth seeing the whole machine turn over once. The session below runs a container and drives it through create→running, a pause and unpause, a graceful stop, and a final removal, while a second terminal follows docker events so you can watch the daemon narrate each transition as it happens.14 Every line of the event stream corresponds to exactly one edge in Figure 1.

one container, from birth to reclamation
# Terminal 2 — follow the daemon narrating this container's transitions:
$ docker events --filter container=demo --format '{{.Status}}'

# Terminal 1 — drive the container through its states:
$ docker run -d --name demo nginx:1.27-alpine   # create + start
9c1f2e...
$ docker pause demo
demo
$ docker unpause demo
demo
$ docker stop demo                             # SIGTERM, then reap
demo
$ docker rm demo
demo

# ...meanwhile Terminal 2 has printed, in order — one line per edge:
create
start
pause
unpause
kill      # the SIGTERM that `stop` delivers
die
stop
destroy  # the writable layer is gone

Read that event log next to the diagram and the abstraction becomes concrete: run produced a create and a start; stop expanded into a kill (the signal), a die (the process ending), and a stop (the state settling); and rm ended in a destroy. Nothing the daemon does to a container falls outside this vocabulary, which is why docker events is the definitive way to answer “what just happened to my container?”

OBS · SEEING INSIDE

Monitoring and debugging

If the lifecycle commands are how you change a container’s state, the observation commands are how you read it — and in an incident you will spend far more time reading than changing. The five that follow answer five different questions, and knowing which tool answers which question is most of the skill. What is running? ps. What did it say? logs. What is it doing right now, from the inside? exec. What is it consuming? stats. How, exactly, is it configured? inspect.

ps and ls: what is running

docker ps lists running containers, one per line, with their id, image, command, status, published ports, and name; adding -a widens the view to include stopped and created containers, which is indispensable when something has crashed and vanished from the default list.17 Two refinements turn it from a human-readable list into a scripting primitive: -q prints bare ids (ideal for piping into another command), and --filter plus --format let you select and shape the output with Go templates rather than parsing columns by eye.17

listing, filtering, shaping
$ docker ps
CONTAINER ID   IMAGE               STATUS          PORTS                  NAMES
4bed76d3ad42   nginx:1.27-alpine   Up 3 minutes    0.0.0.0:8080->80/tcp   web

# All containers, including the dead — note the Exited status and code:
$ docker ps -a --filter status=exited
9f8e7d6c5b4a   my-api:2.3          Exited (1) 6 min ago                          api

# Just the ids of every container, to feed another command:
$ docker ps -aq
4bed76d3ad42
9f8e7d6c5b4a

logs: what it said

A container’s logs are simply whatever its main process wrote to standard output and standard error. docker logs retrieves that stream in a batch, and its flags make it a real debugging instrument: -f / --follow tails new output as it is produced, --tail N shows only the last N lines, --since and --until bound the window in time, and --timestamps prefixes each line with a precise RFC 3339 timestamp so you can correlate events across services.18 The one caveat that surprises people is that docker logs reads from Docker’s own logging mechanism, so it works with the default json-file (and journald) drivers but not with drivers that ship logs elsewhere — if you have pointed a container at a remote log sink, the local command will tell you it cannot read them.18

following a live log with timestamps
# Last 20 lines, then follow, each line timestamped:
$ docker logs --tail 20 -f --timestamps web
2026-07-09T14:22:07.114Z 172.18.0.1 - GET /health 200
2026-07-09T14:22:08.501Z 172.18.0.1 - GET /api/users 200
# ^C to stop following. Bound a past window instead of following:
$ docker logs --since 2026-07-09T14:00:00 --until 2026-07-09T14:05:00 api

exec: what it is doing, from the inside

Sometimes reading output is not enough and you need to be in the container. docker exec starts a brand-new process and joins it to the running container’s namespaces, so the process you launch sees the container’s filesystem, network, and process table exactly as the main process does.19 Combined with -it, this is the canonical “get me a shell in there” move; on its own it is a way to run a single diagnostic command — check environment variables, list processes, test a health endpoint from inside the network namespace — without disturbing the main process. Two limits are worth stating plainly: exec only works on a container that is running (there is no process tree to join in a stopped one), and it can only run tools that actually exist inside the image, which is why minimal images sometimes lack even a shell.

a shell, and a one-shot diagnostic
# Interactive shell inside the running container:
$ docker exec -it web sh
/ # nginx -t
nginx: configuration file /etc/nginx/nginx.conf test is successful
/ # exit

# One command, no shell — confirm what the process actually sees:
$ docker exec web ps -o pid,comm
  PID COMMAND
    1 nginx
   31 nginx
$ docker exec -e DEBUG=1 web env | grep DEBUG
DEBUG=1

stats: what it is consuming

docker stats attaches to one or more containers and prints a live, continuously updating stream of resource usage: CPU percentage, memory usage against its limit, network and block I/O totals, and the number of processes.20 Those numbers come straight from the cgroup accounting the kernel is already keeping, which is why they are cheap and accurate.11 By default the display refreshes in place like top; passing --no-stream prints a single snapshot and exits, which is what you want in a script or a one-off check.20

a one-shot resource snapshot
$ docker stats --no-stream
CONTAINER   NAME   CPU %   MEM USAGE / LIMIT     MEM %   NET I/O         PIDS
4bed76d3    web    0.04%   6.1MiB / 512MiB       1.19%   1.2MB / 900kB   3
7a1c9e2b    api    17.8%   402MiB / 512MiB      78.5%   40MB / 55MB     44
# api is at 78% of its memory cap — a candidate to raise --memory or fix a leak.

inspect: how it is configured, exactly

When you need ground truth — not a summary but the complete configuration and state the daemon holds — docker inspect returns it as a low-level JSON document.13 That document is exhaustive: the resolved command and entrypoint, every mount and network, the environment, the restart policy, and the full State object with the status, the process id on the host, the exit code, and the start and finish timestamps. Dumping the whole blob is rarely useful; the power is in the --format flag, which evaluates a Go template against the structure so you can extract exactly the field you need for a script or a one-line answer.13

extracting single facts with --format
# The exact state, and the host PID of the container's process:
$ docker inspect --format '{{.State.Status}} pid={{.State.Pid}}' web
running pid=28841

# Why did it stop? Exit code and finish time, no JSON spelunking required:
$ docker inspect --format '{{.State.ExitCode}} @ {{.State.FinishedAt}}' api
1 @ 2026-07-09T14:16:41.6Z

# Resolve a container's IP on its network:
$ docker inspect --format '{{range .NetworkSettings.Networks}}{{.IPAddress}}{{end}}' web
172.18.0.2

The rest of the toolkit

Five commands cover the overwhelming majority of operational work, but a handful more are worth knowing when the usual five come up short, and all of them live under the same docker container group.14 docker top shows the processes running inside a container using the host’s process table, which is handy when a container runs more than one process. docker diff reports every file that has been added, changed, or deleted relative to the image — a fast way to see what a running container has written. docker cp copies files in and out, useful for extracting a core dump or a log a process wrote to a file rather than to stdout. docker port prints the published-port mappings for a container. And docker events streams the daemon’s real-time event feed — creates, starts, health-status changes, out-of-memory kills, dies — which is the single best way to watch what is actually happening to your containers over time rather than polling their state.14

Assembled into a routine, these five commands form a fast triage path. The table below maps a symptom you observe to the command that most quickly reveals its cause — a compressed version of the reasoning an experienced operator runs almost without thinking.

▲Debug the container, not just the process

A recurring trap is to treat a container purely as its main process and forget the fence around it. A service that “can’t reach the database” may be perfectly healthy but sealed in the wrong network namespace; a process being “randomly killed” is often the kernel’s OOM killer enforcing a --memory cap you forgot you set, visible instantly in docker inspect as an OOMKilled flag and exit code 137. When behavior is baffling, check the fence — the namespaces and cgroups from the architecture section — before you suspect the code.

OPS · PRODUCTION

Operating safely in production

The commands so far are value-neutral: they will just as happily run a hardened service as a root-privileged foothold for an attacker. What separates the two is a set of operational defaults, and the most authoritative catalogue of them is the U.S. National Institute of Standards and Technology’s Application Container Security Guide (SP 800-190), which analyzes the risks specific to the container model and prescribes countermeasures.21 The practices below distill that guidance, and the Docker documentation that implements it, into the flags and habits you actually apply when launching containers.

Least privilege: assume the container will be breached

Because a container shares the host kernel, the guiding assumption should be that a compromised process inside it will try to reach the host, and every default should narrow what that process can do. Four measures do most of the work. Run as a non-root user, either with USER in the image or -u at launch, so that even a full application compromise does not start as UID 0. Drop all Linux capabilities and add back only the few a service genuinely needs, rather than shipping the permissive default set. Mount the root filesystem read-only with --read-only, granting writable tmpfs mounts only where the app truly must write. And forbid privilege escalation so a process cannot gain more rights than it started with. NIST states the principle plainly — containers should run with the least privilege required, dropping unneeded capabilities and never running as root where it can be avoided — and Docker warns in the same breath that --privileged, which switches all of this off, does not produce a securely sandboxed process and should be treated as a last resort.2115

a least-privilege launch
$ docker run -d --name web \
    --user 10001:10001 \            # not root inside the container
    --read-only \                    # immutable root filesystem
    --tmpfs /tmp:rw,size=64m \        # the one place it may write
    --cap-drop ALL \                  # start from zero capabilities
    --cap-add NET_BIND_SERVICE \      # add back only what is needed
    --security-opt no-new-privileges \ # no setuid escalation
    nginx:1.27-alpine

Resource limits: contain the blast radius

An unbounded container is a denial-of-service waiting to happen: a memory leak, a fork bomb, or a runaway loop in one container can starve every other container on the host and the host itself. The cgroup machinery from the architecture section exists precisely to prevent this, and three flags expose it. --memory caps the container’s memory, after which the kernel’s OOM killer terminates the offending process rather than the whole host sliding into swap. --cpus limits how many CPU cores the container may consume. And --pids-limit caps the number of processes it may spawn, which neutralizes fork bombs.1511 NIST frames resource controls as a core mitigation, since without them a single container can exhaust host resources and degrade every neighbor.21

bounded resources
$ docker run -d --name api \
    --memory 512m --memory-swap 512m \  # hard RAM cap, no swap escape
    --cpus 1.5 \                        # at most 1.5 cores
    --pids-limit 200 \                  # fork-bomb ceiling
    my-api:2.3

One concern per container

A recurring source of operational pain is the container that tries to be a whole host — running an app, a database, a cron daemon, and an SSH server behind a homegrown init script. The model assumes the opposite: a container is a runnable instance of a single application, and keeping it to one primary concern is what makes every other practice work.8 With one process, the lifecycle is unambiguous (the container is up exactly when that process is up), SIGTERM reaches the thing that matters, the logs are that process’s output rather than an interleaved jumble, resource limits bound one workload rather than several competing ones, and scaling means running more copies of a small unit instead of duplicating a monolith. When several processes genuinely must cooperate, the container-native answer is several containers composed together — the subject of the ecosystem section — not several processes crammed into one.

Trust what you run

Least privilege limits the damage a compromised container can do; image provenance limits the odds that it is compromised to begin with. A container is only as trustworthy as the image beneath it, so NIST treats the image supply chain as a first-class control: prefer minimal base images to shrink the attack surface, obtain images only from trusted sources, scan them for known vulnerabilities before deployment, and pin them to an immutable digest rather than a mutable tag so that a reference like my-api:2.3 cannot be silently repointed to different bytes under you.21 Pinning by digest has a second payoff that is squarely a management concern: it makes deployments reproducible, because the same reference always resolves to the identical image — the container-operations equivalent of a locked dependency file.

Recovery and health: make “up” mean “working”

A long-running service should carry a restart policy so that a crash or a host reboot brings it back automatically; as established earlier, --restart unless-stopped is the sensible default for services.16 But a restart policy only knows whether the process is alive, not whether it is working — a server can be running while wedged, deadlocked, or returning errors. That gap is what a health check closes: a HEALTHCHECK baked into the image (or its equivalent in an orchestrator) runs a command on an interval and marks the container healthy or unhealthy, and that status shows up in docker ps and the event stream so both humans and schedulers can act on real readiness rather than mere liveness.14 Pairing the two — restart for liveness, health check for correctness — is what makes automated recovery trustworthy.

Graceful shutdown: respect the SIGTERM

Recall that docker stop sends SIGTERM and waits before escalating to SIGKILL. Whether that window is used well is a property of the image, and it is the source of a surprising number of production bugs. The application’s main process must actually receive and handle SIGTERM, which fails silently when the process is wrapped in a shell — a shell-form ENTRYPOINT or CMD runs the program as a child of /bin/sh, which does not forward signals, so the app never learns it is being asked to stop and is hard-killed ten seconds later.15 Using the exec form keeps your process as PID 1 where it receives signals directly. For slow, stateful shutdowns, extend the grace window with --stop-timeout, and when your PID 1 is not designed to reap orphaned children, add a minimal init such as the one Docker bundles behind --init, which forwards signals and reaps zombies on the container’s behalf.9

Logging, ephemerality, and the daemon socket

Three more habits round out safe operation. First, control your logs: with the default json-file driver, a chatty container will grow its log file until it fills the disk, so cap it with --log-opt max-size and --log-opt max-file to enable rotation, and ship logs to a central system for anything beyond a single host.18 Second, treat containers as ephemeral and immutable. A container is a runnable instance you can create, start, stop, and delete at will, and it is meant to be disposable: persistent state belongs in volumes or external stores, not in the container’s writable layer, and the correct response to a needed change is to build a new image and replace the container, not to exec in and mutate a running one.821 Reserving docker exec for read-only investigation keeps your running fleet identical to what your images describe. Third, guard the daemon socket. Access to /var/run/docker.sock is effectively root on the host — anyone who can reach it can launch a privileged container that mounts the host filesystem — which is why membership in the docker group is equivalent to root, why the socket must never be exposed over an unencrypted network, and why mounting it into a container is a decision to make with great care.9

▲The one-line summary of container security

Minimize what runs (one concern per container, least privilege, drop capabilities), bound what it can consume (memory, CPU, PIDs), verify that it works rather than merely runs (health checks, not just restart policies), and never forget that the daemon socket is the keys to the host. Every item here is an application of a single idea: the isolation is real but shallow, so you deepen it deliberately.

TOOL · THE ECOSYSTEM

The surrounding ecosystem

The raw CLI manages one container at a time, which is the right altitude for learning and for debugging but the wrong one for real systems, where applications are several cooperating containers and fleets span many hosts. A layered ecosystem picks up where docker run leaves off, and the useful move is to see each tool as automating one of the manual acts from the previous sections.

Docker Compose: many containers, one file

The moment an application is more than one container — a web service, a database, a cache — passing long run commands by hand becomes error-prone. Docker Compose replaces them with a single declarative YAML file that defines the services, their images, ports, volumes, environment, and dependencies; docker compose up then creates and starts the whole application, and docker compose down tears it down, as one unit.22 The same lifecycle and observation verbs you already know apply to the group, so the mental model carries over intact.

compose.yaml · a service with its datastore
services:
  web:
    image: my-api:2.3
    ports: ["8080:80"]
    restart: unless-stopped
    depends_on: [db]
    deploy:
      resources:
        limits: { memory: 512m, cpus: "1.5" }
  db:
    image: postgres:16-alpine
    volumes: ["pgdata:/var/lib/postgresql/data"]  # state lives in a volume
    restart: unless-stopped
volumes:
  pgdata:

Notice that everything hard-won from the operations section — a restart policy, resource limits, state kept in a named volume rather than the writable layer — is expressed declaratively here, which is exactly why teams graduate from ad-hoc commands to a checked-in Compose file: the file becomes the reviewable, reproducible record of how the application is meant to run.

Observability beyond the built-ins

The stats, logs, and events commands are enough to diagnose a handful of containers, but watching many of them over time calls for purpose-built tooling. Terminal dashboards such as ctop and lazydocker present a live, sortable overview of every container’s resource use and logs; Portainer offers the same as a web UI for teams. For metrics that feed a monitoring stack, Google’s cAdvisor discovers running containers automatically and exports their per-container CPU, memory, filesystem, and network usage, most commonly scraped by Prometheus and graphed in Grafana — turning the momentary snapshot of docker stats into a historical record you can alert on.23 The image-inspection tool dive rounds out the set by letting you explore an image layer by layer to find wasted space. None of these replaces the CLI; they scale it.

Alternative runtimes: swapping the bottom of the stack

Because the interface between the daemon and the low-level runtime is the standardized OCI runtime specification, the reference runc is replaceable, and you can select an alternative per container with --runtime.515 The choices reflect a spectrum of trade-offs between isolation and overhead. crun is a lighter, faster C implementation that is otherwise a drop-in for runc. For workloads where the shared-kernel caveat from the architecture section is unacceptable, gVisor interposes a user-space kernel that intercepts the container’s system calls, and Kata Containers goes further, wrapping each container in a lightweight virtual machine with its own kernel — buying much stronger isolation at some cost in speed and density. The point is that the same docker run you have used throughout can, with one flag, sit atop a very different enforcement mechanism.

Rootless mode and orchestration

Two directions extend the model outward. Downward, in privilege: rootless Docker runs the daemon and its containers as an unprivileged user by leaning on user namespaces, so that even a daemon compromise does not hand an attacker root on the host — the strongest available answer to the socket-is-root hazard from the operations section.9 Outward, in scale: once an application must span many hosts, single-host management gives way to an orchestrator that schedules containers across a cluster, restarts them on failure, and handles rollout and service discovery. Docker’s own Swarm mode is the gentlest step up; Kubernetes is the industry standard and, fittingly, drives its nodes through the very same containerd that sits inside Docker; Nomad is a lighter-weight scheduler. The trajectory from LXC to Docker to cluster orchestration was exactly the arc observed in the field’s early years, and understanding the single-container lifecycle is the prerequisite for all of it, because an orchestrator is in the end an automated operator issuing the same transitions you now know by hand.4 It is worth noting, finally, that the ecosystem is genuinely standards-based rather than single-vendor: because the OCI defines the image and runtime formats, tools such as Podman can run the same images with a compatible, daemonless CLI.5

END · SYNTHESIS

The shape of the whole thing

Step back and the sprawl collapses into three ideas. First, a container is a state machine: it is always in exactly one of a small set of states, and every management command — create, start, run, stop, kill, restart, pause, rm — exists to move it along one labeled edge, while ps, logs, exec, stats, and inspect read where it is without moving it. Hold Figure 1 in your head and the CLI is no longer a list to memorize but a map to navigate.

Second, that machine is enforced by a layered, standards-based engine. Your docker client speaks to dockerd, which delegates to containerd, which drives runc, which asks the Linux kernel for namespaces and cgroups. Each layer is swappable because the seams between them are open specifications, and knowing the layers tells you which one to interrogate when something goes wrong — and reminds you that the isolation, being kernel-shared, is real but shallow.

Third, safe operation is the deliberate deepening of that shallow isolation. Run as non-root, drop capabilities, mount read-only, and forbid escalation; bound memory, CPU, and process counts; recover with a restart policy but verify with a health check; respect the SIGTERM; keep state in volumes and rebuild rather than mutate; and guard the daemon socket as the root-equivalent that it is. Every one of those practices is the same lesson applied: the container will be treated as hostile, so give it as little as it needs and watch what it does. Master the small machine, respect the shallow fence, and the largest fleet is only the smallest unit, managed at scale.
