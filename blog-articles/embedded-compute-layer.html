Layer 2 — The Compute Substrate

The Embedded
Compute Layer

The processing intelligence sitting between raw sensor data and the networked world — microcontrollers, single-board computers, industrial controllers, and reconfigurable silicon.

Peer-Reviewed Technical Article Vancouver Citation Format ~7,200 words 2025 Edition

Every sensor that probes the physical world — measuring temperature, detecting motion, reading pressure, capturing photons — generates raw electrical signals. Those signals, on their own, are meaningless until something acts upon them: sampling them at precise intervals, converting analogue voltages into digital integers, running calibration algorithms, deciding whether a threshold has been crossed. That "something" is the embedded compute layer — the silicon brain attached to the sensor, responsible for transforming physical phenomena into structured, actionable information.[1]

Embedded computing is among the oldest disciplines in digital electronics. The field predates the internet, predates the personal computer, and in many respects predates the very vocabulary we use to describe it. [2] Today it encompasses an enormous design space — from microcontrollers drawing less than a microampere in deep sleep to NVIDIA GPU-accelerated modules running full neural network inference on a factory floor. Choosing the right node in this space is one of the most consequential architectural decisions an engineer faces.

This article surveys that design space systematically. It examines the four major hardware classes — microcontrollers (MCUs), single-board computers (SBCs), industrial controllers (PLCs, RTUs, DCS), and programmable logic devices (FPGAs and ASICs) — tracing their histories, technical architectures, canonical examples, and ideal deployment contexts. It concludes with a decision framework and a look at emerging innovations reshaping the boundary between these categories.

§ 01 — Historical Background

From Calculator Chips to Intelligent Edge Nodes

The story of embedded compute begins not with the internet of things but with the calculator wars of the early 1970s. Semiconductor manufacturers, racing to produce cheaper calculator chips, inadvertently invented the computer-on-a-chip — and with it, the embedded processor as we know it today.[3]

1971
Intel 4004 & TI TMS1802NC. Intel's 4004 — a 4-bit, 2,300-transistor CPU developed for Busicom calculators — became the first commercially available microprocessor sold as a general-purpose component on November 15, 1971.[4] That same year, Texas Instruments engineers Gary Boone and Michael Cochran completed the TMS1802NC, a single-chip device integrating CPU, ROM, RAM, and I/O — recognised by the Smithsonian Institution as the first true microcontroller.[5]
1974
TMS1000 — first general-purpose MCU. Texas Instruments released the TMS1000, the first high-volume, general-purpose commercial microcontroller. Sold for approximately $2 per unit in volume, it combined a 4-bit CPU with ROM, RAM, and I/O on a single PMOS die. By 1979, roughly 26 million units were shipped per year. Applications included the Speak & Spell educational toy and the Simon electronic game.[5,6]
1976–1980
Intel 8048 & 8051; MCS-48 family. Intel responded to TI's success with the 8048 (1976) and later the immensely influential 8051 (1980), which became the architectural template for an entire generation of 8-bit MCUs still manufactured today. The 8051's Harvard architecture with separate program and data memories established a pattern that ARM Cortex-M cores continue to honour.[7]
1990s
PIC, AVR, and 32-bit transitions. Microchip Technology's PIC and Atmel's AVR introduced RISC-based designs that dramatically simplified programming and reduced power consumption, making hobbyist embedded development viable. Simultaneously, the transition from 8-bit and 16-bit MCUs to 32-bit processors accelerated through the decade.[8]
2004–2010
ARM dominance & the Cortex-M era. ARM's Cortex-M profile, beginning with the Cortex-M3, established a new standard for 32-bit embedded computing. The architecture offered deterministic real-time behaviour, a Nested Vectored Interrupt Controller (NVIC) for sub-cycle interrupt latency, and the Thumb-2 instruction set for high code density — eventually capturing the majority of the 32-bit MCU market.[9]
2010s
IoT catalyst & security focus. The explosion of IoT deployments drove MCU manufacturers to integrate wireless connectivity (Wi-Fi, Bluetooth LE, Zigbee), hardware security modules, and eventually neural-network accelerators directly onto MCU silicon. The 2010s also saw increased emphasis on security features in embedded devices, driven by the vulnerability landscape of internet-connected hardware.[8]
2020–present
TinyML, RISC-V, and heterogeneous compute. The convergence of machine learning with embedded systems — branded as TinyML — enables on-device inference on MCUs with as little as a few hundred kilobytes of RAM. The open RISC-V ISA provides a royalty-free alternative to ARM. FPGAs increasingly integrate hard processor cores, blurring the line between reconfigurable logic and SoCs.[10,11]
§ 02 — Hardware Class I

Microcontrollers (MCUs)

A microcontroller is a single integrated circuit that consolidates a central processing unit, flash memory (program storage), RAM (working memory), and programmable I/O peripherals — timers, UART, SPI, I²C, ADCs — into one package. Unlike a microprocessor, which requires external components to function, the MCU is a self-contained computer designed to execute a fixed or narrowly-scoped task.[12] This integration is precisely what makes MCUs the dominant choice for battery-powered, cost-sensitive, latency-sensitive edge nodes: nothing is wasted on capabilities the application does not need.

Most MCUs employ a Harvard architecture, with physically separate buses for program memory and data memory, allowing simultaneous instruction fetch and data access — critical for deterministic real-time performance. This contrasts with the Von Neumann (Princeton) architecture used in general-purpose computers, where instructions and data share a single bus.

[6] 8-bit MCUs have been at the heart of embedded development since the 1980s and continue to play a major role in IoT development even as 32-bit architectures have become more prevalent. Choosing between 8-bit, 16-bit, and 32-bit architectures involves trade-offs of cost, power, code density, and computational throughput that are highly application-specific.[6]

ARM Cortex-M Series Market Leader

The ARM Cortex-M is a family of 32-bit RISC processor cores developed by Arm for microcontroller-based embedded systems, emphasising low power consumption, compact code size, and deterministic real-time operation to support IoT, industrial automation, automotive, and consumer electronics applications.[9] The architecture evolves from Armv6-M for entry-level cores to Armv8-M, which introduces enhanced security via TrustZone technology — hardware isolation between secure and non-secure execution states.[9]

The Cortex Microcontroller Software Interface Standard (CMSIS) delivers standardised APIs for accessing core peripherals like the NVIC and system tick timer, enabling consistent software reuse without vendor-specific code.[9] This ecosystem — combined with the open-source Mbed OS, Zephyr RTOS, and FreeRTOS — gives Cortex-M the broadest software ecosystem in embedded computing.

ESP32 Connectivity-First

Espressif Systems' ESP32 (2016) and its successors disrupted the MCU market by integrating dual-core Xtensa LX6 processors (later evolved to RISC-V in the ESP32-C series) with Wi-Fi 802.11b/g/n and Bluetooth 4.2/5.0 on a single low-cost SoC, typically available for under $5 in volume. This eliminated the need for a separate wireless module in the majority of IoT designs, making the ESP32 the dominant platform for consumer IoT prototyping and production alike.[1]

The ESP-IDF (IoT Development Framework) provides a comprehensive development environment, and Arduino-compatible wrappers lower the barrier to entry significantly. The ESP32-S3 added dedicated vector instructions for accelerating TinyML workloads, while the ESP32-P4 (2024) introduced a high-performance dual-core RISC-V with MIPI camera/display interfaces — explicitly targeting embedded AI vision applications.

STM32 Family Industrial Grade

STMicroelectronics' STM32 line is built on ARM Cortex-M cores (M0 through M7 and M33) combined with ST's proprietary low-power silicon IP, non-volatile memory technology, and hardware accelerators including CORDIC (trigonometric computation) and FMAC (filtering).[13] The STM32 range spans hundreds of variants across power consumption, flash size, peripheral mix, and package option — all unified under the STM32Cube development environment. The STM32L series targets ultra-low-power applications (sub-microamp sleep currents), while STM32H7 (Cortex-M7 at up to 550 MHz with L1 cache) targets high-performance industrial control.

AVR & PIC 8-bit Classics

Atmel's AVR architecture (now owned by Microchip Technology) became famous through the Arduino platform, which abstracted AVR programming into a C++-based framework accessible to non-engineers. The ATmega328P — the original Arduino Uno's heart — runs at 16 MHz, consumes milliwatts in active mode, and exemplifies the qualities that made 8-bit MCUs the workhorses of a generation of embedded systems. Microchip's PIC (Peripheral Interface Controller) family, dating from the mid-1970s, similarly dominates industrial and automotive embedded markets with its compact instruction set and deterministic timing model. Both architectures retain market relevance for simple, cost-constrained applications where the additional capability of a 32-bit MCU is unjustified.[8]

§ 03 — Hardware Class II

Single-Board Computers (SBCs)

When an application's computational demands exceed what a bare-metal MCU can offer — running a full Linux operating system, executing computer vision pipelines, handling complex network stacks, or orchestrating multiple concurrent services — engineers escalate to a single-board computer. An SBC integrates a high-performance application processor (often multi-core ARM Cortex-A), DRAM, storage interfaces, and rich I/O onto a single compact board, running a full OS with the associated scheduler, file system, and driver ecosystem.[1]

The trade-off is power, cost, boot latency, and the loss of hard real-time guarantees. A general-purpose Linux OS with a non-realtime scheduler cannot guarantee interrupt latency below a few hundred microseconds — making SBCs unsuitable for hard real-time control loops, but highly suitable for data-intensive edge processing, machine learning inference, protocol mediation, and complex decision logic.

Raspberry Pi Foundation

The world's most widely deployed SBC platform. The Pi 5 features the Broadcom BCM2712 quad-core Arm Cortex-A76 @ 2.4 GHz — delivering 2–3× the CPU performance of the Pi 4 — with 4/8/16 GB LPDDR4X RAM. Power consumption peaks at ~10 W under load. Its massive community, extensive Python/C++ library support, and optional Hailo-8L AI HAT (13 TOPS) make it the dominant prototyping and education platform.

The SBC market has bifurcated sharply. General-purpose boards (Raspberry Pi, Orange Pi, Rock Pi) serve prototyping and medium-complexity edge tasks; AI-focused boards (NVIDIA Jetson Orin series, BeagleBone AI-64, Google Coral Dev Board) serve applications that require GPU-accelerated inference or dedicated neural processing units. The Jetson Orin Nano's 67 TOPS makes it capable of running compact large language models (LLMs) locally, performing real-time multi-object detection, and powering autonomous robots — tasks that would be impossible on even the most powerful MCU.[14,15]

The global edge AI market was valued at approximately $20.78 billion in 2024 and is projected to reach $66.47 billion by 2030. This growth is a direct driver of investment in edge-capable SBCs and AI-accelerated embedded platforms, with predictions that 75% of enterprise data will be processed at the edge by 2025.[16]

§ 04 — Hardware Class III

Industrial Controllers

In factory automation, oil and gas infrastructure, power grids, and water treatment plants, the requirements diverge sharply from the consumer IoT design space. Reliability over decades, not months. Deterministic scan cycles measured in milliseconds. Certifications for hazardous environments (ATEX, IECEx). Communication over legacy fieldbus protocols (PROFIBUS, MODBUS, HART) and modern industrial Ethernet (PROFINET, EtherNet/IP). Here, industrial-grade controllers — PLCs, RTUs, and DCS systems — hold dominant position, and for very good engineering reasons.[17]

Programmable Logic Controllers (PLCs)

The programmable logic controller was invented in the late 1960s by Dick Morley and his colleagues at Bedford Associates to replace relay-logic panels in automotive manufacturing. The original Modicon 084 could be reprogrammed without rewiring — a transformative operational advantage. [18] Modern PLCs are specialised industrial computers with hardened housings, extended operating temperature ranges (–40 °C to +70 °C is typical), deterministic scan-cycle execution, and support for IEC 61131-3 standardised programming languages: Ladder Diagram (LD), Structured Text (ST), Function Block Diagram (FBD), Sequential Function Chart (SFC), and Instruction List (IL).[19]

PLCs have more sophisticated embedded control capabilities than RTUs and are used in applications where local, fast, standalone control is required — assembly lines, robotic cells, packaging machines, and conveyor systems.[20] Major vendors include Siemens (S7 series), Rockwell Automation (Allen-Bradley ControlLogix), Schneider Electric (Modicon M580), and Mitsubishi Electric (MELSEC).

Remote Terminal Units (RTUs)

A Remote Terminal Unit is a microprocessor-based device designed for remote monitoring and control of field instruments, typically in geographically dispersed infrastructure: pipeline monitoring stations, electricity substations, weather monitoring points, and water distribution networks.[21] RTUs are distinguished from PLCs by their extremely low power consumption (enabling solar and battery operation), tolerance for low-bandwidth, intermittent communication links (radio, GSM, satellite), and the ability to buffer data locally during communication outages.[22]

The SCADA (Supervisory Control and Data Acquisition) architecture is built around RTUs transmitting measurements and alarms to a central supervisory computer, which presents them to human operators via HMI screens and stores them in a historian database for post-event analysis.[21]

Distributed Control Systems (DCS)

A Distributed Control System takes the functionality of PLCs and expands it across a network of controllers that communicate with one another over a high-speed, deterministic proprietary fieldbus, each managing a segment of a complex continuous process.[22] DCS is the architecture of choice for process industries — oil refining, chemical manufacturing, pulp and paper, pharmaceuticals — where tightly coupled feedback loops across hundreds of control variables must be coordinated in real time.[22,23]

A distinguishing characteristic of modern DCS is the "smart field device": instruments with onboard intelligence sufficient to process their own sensor data, perform self-diagnostics, and communicate digitally over FOUNDATION Fieldbus or HART, without requiring the controller to intervene in measurement processing.[22] DCS vendors include Emerson (DeltaV), Honeywell (Experion PKS), ABB (System 800xA), and Yokogawa (CENTUM VP).

Industrial control systems are typically organised under the ISA-95 / Purdue Enterprise Reference Architecture. Level 0/1 contains field instruments and local control (PLCs, RTUs). Level 2 is supervisory control (HMI, SCADA servers). Levels 3–4 are plant information management and enterprise systems. DCS spans Levels 1–2; SCADA typically sits at Level 2–3. This layered model defines trust boundaries critical for industrial cybersecurity (IEC 62443).[18]

§ 05 — Hardware Class IV

FPGAs and ASICs

At the apex of embedded compute performance and determinism sit field-programmable gate arrays (FPGAs) and application-specific integrated circuits (ASICs). Where MCUs and SBCs execute software instructions sequentially on a fixed silicon architecture, FPGAs allow engineers to define the actual hardware structure — creating custom digital circuits that execute logic in parallel, with deterministic, sub-microsecond latency and no operating system overhead.[24]

Field-Programmable Gate Arrays (FPGAs)

An FPGA is an integrated circuit that can be configured after manufacture by the end user. Its fabric consists of an array of configurable logic blocks (CLBs), programmable interconnects, DSP slices, block RAM, and I/O blocks, all described in a hardware description language — VHDL or Verilog — and synthesised to a bitstream that defines the hardware's behaviour.[25]

FPGAs excel in three domains where software-executed processors fundamentally struggle:

Ultra-low, deterministic latency. Because FPGAs implement custom hardware pipelines with no global scheduler, they achieve highly predictable, fixed latency. FPGAs excel in low-latency environments due to their deterministic execution model, making them particularly valuable in industrial automation, autonomous systems, and safety-critical applications where timing predictability is paramount.[26] In discrete manufacturing, motion-control loops frequently operate at cycle times below 1–5 ms, and jitter beyond a few hundred microseconds can degrade synchronisation accuracy.[27]

High-throughput parallel processing. FPGAs can process multiple independent data streams simultaneously by instantiating parallel hardware pipelines — an impossibility for sequential CPU cores. This makes them ideal for high-speed signal processing, packet inspection, radar processing, and real-time image preprocessing for machine vision.[24]

Custom I/O and protocol implementation. FPGAs can implement any digital interface protocol — including custom or proprietary ones — directly in silicon, with bit-accurate timing. This makes them invaluable in high-energy physics experiments, telecommunications, and any system that must interface with non-standard hardware.[24]

Application-Specific Integrated Circuits (ASICs)

Where an FPGA offers reconfigurable hardware at the cost of power and area efficiency, an ASIC implements a fixed digital circuit on silicon optimised for exactly one function. ASICs achieve the best possible power-performance-area ratio for a given workload — but at enormous non-recurring engineering (NRE) cost, typically ranging from hundreds of thousands to tens of millions of dollars for tape-out and mask sets at advanced process nodes.[28]

The ASIC approach is justified when: (a) volumes are sufficiently high (millions of units) to amortise NRE costs; (b) the application's performance or power requirements cannot be met by any off-the-shelf component; or (c) IP protection requirements prevent use of a software-programmable device. The Google Tensor Processing Unit (TPU), Apple's Neural Engine, and Qualcomm's Hexagon DSP are all examples of custom ASICs deployed in high-volume products where the economics of silicon design are justified.[29]

FPGAs offer a favourable price-performance ratio compared to ASICs for medium-volume markets, thanks to their reconfigurable logic, embedded DSP cores, and off-the-shelf availability. ASICs become economically superior only when volume exceeds the crossover point — typically 100,000–1,000,000 units — at which per-unit cost advantage overcomes NRE investment. For IoT applications at scale (billions of units, e.g., consumer earbuds with on-device ML), ASIC remains the only viable approach.[28]

§ 06 — Engineering Decision Framework

Selecting the Right Compute Node

No single hardware class is universally optimal. The appropriate choice is determined by the intersection of computational requirements, real-time constraints, power budget, connectivity needs, environmental specifications, cost targets, and software ecosystem maturity. The following matrix provides a first-pass decision guide.

Parallel data streams ✗ Sequential execution ◑ Multi-thread/multi-core ◑ Multi-channel I/O ✓ True hardware parallelism

In practice, modern embedded systems frequently employ heterogeneous architectures — combining two or more hardware classes. A common pattern pairs an FPGA with a microcontroller: the FPGA provides deterministic, parallel AI acceleration for inference or preprocessing, while the MCU manages system logic, power sequencing, and communication.[27] Similarly, SBC-based edge nodes often offload time-critical sensor sampling to a co-processor MCU, preserving hard real-time behaviour while allowing the SBC to handle higher-level computation asynchronously.

§ 07 — Emerging Innovations

TinyML, RISC-V, and the AI-Native MCU

The embedded compute landscape is undergoing its most disruptive transition since the advent of the 32-bit MCU. Three forces are reshaping what is possible at the sensor edge: the maturation of TinyML, the rise of RISC-V as an open alternative to ARM, and the emergence of hardware neural-network accelerators integrated directly into MCU silicon.

TinyML: Machine Learning at the Edge

TinyML is a rapidly expanding interdisciplinary field at the convergence of machine learning, software, and hardware, focused on deploying deep neural network models on embedded (microcontroller-driven) systems.[30] The field directly addresses what was, until recently, considered a fundamental impossibility: running neural network inference in environments with less than 256 KB of RAM, no operating system, and a milliwatt power budget.

Several enabling techniques make this feasible. Post-training quantisation converts 32-bit floating-point weights to 8-bit integers, achieving 3–4× storage reduction with minimal accuracy loss — a study published in Scientific Reports demonstrated deployable flash footprints of 286–536 KB on a 1 MB flash / 256 KB SRAM budget, with on-device inference latency of 3.47–14.98 ms per frame.[31] Neural Architecture Search (NAS) automates the discovery of architectures that fit within hardware constraints. Knowledge distillation trains smaller "student" models to replicate the behaviour of larger "teacher" models.[32]

The software ecosystem has matured significantly. TensorFlow Lite for Microcontrollers (TFLite Micro) — Google's solution specifically for microcontrollers without OS support — can create models as small as 2 KB and is optimised for ARM Cortex-M processors.[33] Edge Impulse provides a web-based end-to-end TinyML development platform. The MLPerf Tiny benchmark suite (v1.2 published in 2024, v1.3 in 2025) enables standardised comparison of MCU inference performance across vendors.[34]

Applications span keyword spotting, gesture recognition, anomaly detection in industrial sensors, predictive maintenance (vibration and acoustic analysis), and always-on person detection — all running continuously on a coin cell battery. TinyML will pave the way for novel edge-level services and applications that survive on distributed edge inferring and independent decision-making rather than server computation.[30]

RISC-V: Open-Source Silicon

The RISC-V instruction set architecture (ISA), developed at UC Berkeley in 2010 and now governed by RISC-V International, offers a royalty-free, open-standard alternative to proprietary ISAs such as ARM. For the embedded world, RISC-V's significance is twofold: it enables custom silicon design at dramatically reduced cost (particularly relevant for ASIC tape-outs), and it allows manufacturers to extend the ISA with domain-specific instructions — for example, adding hardware acceleration for neural network operations or cryptographic primitives — without licensing restrictions.[35]

Espressif has already deployed RISC-V cores in the ESP32-C and ESP32-H families. SiFive, GigaDevice, and WCH produce RISC-V MCUs targeting the industrial and consumer embedded markets. In the TinyML domain, RISC-V's extensibility makes it an attractive platform for custom ML accelerators, and several academic and commercial projects have demonstrated RISC-V cores with integrated neural processing units that outperform equivalent ARM Cortex-M devices on ML workloads at similar power budgets.[32]

Integrated Neural Processing Units (NPUs)

The most significant near-term development in MCU architecture is the integration of dedicated neural processing units alongside traditional CPU cores. The ARM Cortex-M55 with Helium vector ISA represents the first step — adding DSP and ML instructions that enable efficient matrix operations for TinyML.[34] The Ethos-U55 and Ethos-U65 are ARM's dedicated Microcontroller NPUs, designed to pair with Cortex-M55 cores to accelerate inference workloads by 4–32× versus software-only execution on the same MCU.

Meanwhile, neuromorphic computing — hardware that mimics the sparse, event-driven computation of biological neurons — remains an active research area, with Intel's Loihi 2 demonstrating orders-of-magnitude power efficiency gains for specific workloads such as sensory processing and constraint optimisation. Whether neuromorphic chips will cross from research into embedded production is an open question, but their existence reflects the intensity of innovation pressure on the hardware compute substrate.[35]

§ 08 — Software Layer

Real-Time Operating Systems and Firmware Architectures

Hardware without software is inert. The embedded compute layer's effectiveness depends critically on its software execution model. Embedded software architecture is sometimes divided into an application layer, a middleware layer, and a firmware layer — and may or may not include a real-time operating system.[1]

At the lowest level, bare-metal firmware runs directly on the MCU with no OS: a single main() loop with interrupt service routines (ISRs) handles all tasks. This approach minimises overhead and provides maximum control over timing, but becomes difficult to maintain as complexity grows.

A Real-Time Operating System (RTOS) adds a scheduler that can pre-emptively context-switch between tasks with guaranteed latency. FreeRTOS is the most widely deployed embedded RTOS, running on ARM, RISC-V, AVR, and dozens of other architectures. Zephyr RTOS, backed by the Linux Foundation, has rapidly grown in adoption due to its strong security features (PSA Certified support), device driver model, and native support for Bluetooth LE, Wi-Fi, and Thread/Zigbee. For higher-end MCUs and SBC co-processors, Mbed OS provides a comprehensive connectivity-focused RTOS with integrated support for TLS, MQTT, and CoAP.[9]

For industrial PLCs and DCS controllers, software is standardised under IEC 61131-3, which defines five programming languages — a deliberate design choice to abstract away the underlying hardware and allow engineers from instrumentation backgrounds to program controllers without software engineering expertise.[17]

§ 09 — Security Considerations

Hardware Security in Embedded Systems

The proliferation of internet-connected embedded devices has created an enormous attack surface. The 2010s saw an increased focus on security features in embedded devices, driven by the rise of IoT and the vulnerability landscape exposed by large-scale botnet attacks (Mirai, 2016) targeting internet-facing MCUs with weak or default credentials.[8]

Modern MCUs respond with hardware security primitives. ARM's TrustZone for Microcontrollers (available on Cortex-M23, M33, M35P, M55) creates a hardware-enforced partition between a "Secure World" (trusted firmware, cryptographic keys, secure boot chain) and a "Normal World" (application code) — ensuring that a compromised application cannot access sensitive assets.[9] ARM's PSA Certified (Platform Security Architecture) framework, introduced in 2017, provides threat models and security analyses for common IoT product categories, allowing device makers to design, evaluate, and certify their hardware's security posture against documented threat scenarios.[14]

Hardware security modules (HSMs), secure elements (SEs), and on-die true random number generators (TRNGs) are now standard offerings in MCUs targeting industrial and medical applications. For industrial control systems, the IEC 62443 standard series provides a structured security framework for OT (operational technology) environments, emphasising the importance of network segmentation, defence-in-depth, and role-based access control across the Purdue model's layers.[18]

§ 10 — Synthesis

Conclusion

The embedded compute layer is not a single technology but a spectrum — from 8-bit MCUs drawing nanoamps in sleep mode to FPGA-based signal processing boards handling gigabytes of data per second with nanosecond latency. Each class in this spectrum reflects a different resolution of fundamental engineering trade-offs: cost versus capability, power versus performance, determinism versus programmability, flexibility versus optimisation.

The current decade is witnessing the most significant architectural disruption in embedded computing since the ARM Cortex-M displaced 8-bit MCUs in high-volume IoT applications. TinyML is pushing machine learning inference deep into the sensor node, closing the gap between edge and cloud. RISC-V is introducing open-source silicon design economics that were previously unavailable outside major semiconductor companies. And heterogeneous compute platforms — MCU + FPGA, SBC + NPU co-processor, SoC FPGA with hard ARM cores — are becoming the standard answer to applications whose requirements span multiple points on the performance-power-determinism spectrum.

Understanding the technical character of each hardware class — its memory architecture, interrupt model, power profile, software ecosystem, and certification landscape — is a prerequisite for sound embedded system design. The sensor, however sophisticated, is only as useful as the compute substrate that gives its data meaning.
