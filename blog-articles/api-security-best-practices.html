// 01 — Overview

The Threat Landscape

Application Programming Interfaces have become the connective tissue of the modern internet. Every mobile app, microservice, IoT device, and cloud integration communicates via APIs. This explosive growth has created an equally vast attack surface. APIs now represent the dominant vector through which sensitive data is exposed, exfiltrated, and abused.

The scale of the problem is difficult to overstate. Salt Security's Q1 2025 State of API Security report found that 99% of organizations encountered API security problems in the preceding year.[1] Traceable AI's 2025 report found that 57% of organizations experienced API-related data breaches in the prior two years.[2] Meanwhile, the IBM Cost of a Data Breach Report 2024 placed the global average cost of a breach at $4.88 million, falling to $4.44 million in 2025 as detection and response times improved.[3,4]

Several high-profile breaches illustrate the stakes. In January 2024, a buggy API exposed 650,000 sensitive messages including passwords. The Trello breach (January 2024) compromised data of over 15 million users via an exposed API that linked private email addresses to accounts. In May 2024, Dell suffered a breach of 49 million customer records through an exploited partner portal API.[5] A 2025 fintech API breach exposed 2.6 million financial records via misconfigured endpoints.[6]

"95% of API attacks originate from authenticated sessions — meaning attackers already hold valid credentials and exploit the logic of the API itself, not its perimeter."

Salt Security, 2025 State of API Security Report [1]

This is the defining characteristic of modern API attacks: they originate from the inside. An attacker with a valid token can probe business logic, harvest data through authorized-but-abused endpoints, and extract entire databases — all while appearing as a legitimate user. Traditional firewall-based security fails entirely against this threat model.

Only 21% of organizations report high ability to detect attacks at the API layer, and only 7.5% of organizations have implemented dedicated API testing and threat modeling programs.[2,5] The gap between threat severity and defensive investment is the central challenge of API security in 2025 and beyond.

// 02 — Vulnerability Taxonomy

OWASP API Security Top 10 (2023)

The Open Web Application Security Project (OWASP) released its first API-specific Top 10 list in 2019 and issued the first major update on June 5, 2023, reflecting the rapidly evolving threat landscape.[7] The 2023 edition consolidates two earlier categories (Excessive Data Exposure and Mass Assignment) and adds two new entries: Server Side Request Forgery (SSRF) and Unsafe Consumption of APIs — both reflective of cloud-native architectures and the growth of third-party integrations.[8]

  • API1
    Broken Object Level Authorization (BOLA)
    APIs fail to verify that a user is authorized to access a specific object. Present in ~40% of all API attacks and the #1 vulnerability since 2019. Exploited by manipulating object IDs (e.g., /api/users/1234 → /api/users/1235).
  • API2
    Broken Authentication
    Weak authentication mechanisms allow credential stuffing, brute-force, and token theft. The 2023 edition expands coverage to microservice authentication, missing JWT expiry validation, and password confirmation bypass.
  • API3
    Broken Object Property Level Authorization
    Consolidation of Excessive Data Exposure (2019:API3) and Mass Assignment (2019:API6). APIs return or accept full data objects without filtering sensitive fields, enabling data harvesting or mass property assignment attacks.
  • API4
    Unrestricted Resource Consumption
    Missing limits on request rates, payload sizes, execution timeouts, and third-party spending create denial-of-service vectors and enable brute-force authentication attacks at scale.
  • API5
    Broken Function Level Authorization (BFLA)
    Administrative or privileged API functions lack proper authorization checks, allowing non-privileged users to invoke them. Attackers discover hidden admin endpoints by probing predictable naming patterns.
  • API6
    Unrestricted Access to Sensitive Business Flows
    New in 2023. Business logic flows (e.g., purchasing, referral programs, reservations) are exposed to automated abuse — bots bulk-buying inventory, mass claiming referral credits, or scraping at scale.
  • API7
    Server Side Request Forgery (SSRF)
    New in 2023. When an API fetches a remote resource based on user-supplied URLs without validation, attackers redirect requests to internal services, cloud metadata endpoints (e.g., 169.254.169.254), or attacker-controlled hosts.
  • API8
    Security Misconfiguration
    A broad category covering missing security headers, permissive CORS, verbose error messages, default credentials, and insecure cloud storage configurations. Injection vulnerabilities are now treated as a subcategory of this risk in the 2023 list.
  • API9
    Improper Inventory Management
    Outdated, undocumented, or shadow API versions remain accessible and unmonitored. Attackers prefer forgotten endpoints precisely because they are unpatched and unguarded.
  • API10
    Unsafe Consumption of APIs
    New in 2023. Applications trust third-party API responses implicitly — bypassing authentication controls, failing to sanitize returned data, or following malicious redirects — creating transitive trust vulnerabilities.

The 2023 list explicitly removed Injection and Insufficient Logging from the top ten, not because they are resolved, but because OWASP considers them general application risks without unique API-specific characteristics — Injection is now subsumed under Security Misconfiguration.[9]

// 03 — Network Security

Transport Security

All API communication must be encrypted in transit. TLS (Transport Layer Security) is non-negotiable. A 2024 F5 report found that nearly a third of customer-facing APIs lack basic encryption — a baseline failure that enables trivial credential interception, token theft, and man-in-the-middle attacks.[10]

TLS Configuration Hardening

// Insecure
  • TLS 1.0 / TLS 1.1 (deprecated, broken)
  • SSL 2.0 / 3.0 (critically broken)
  • NULL cipher suites
  • RC4, DES, 3DES cipher suites
  • Self-signed certificates in production
  • HTTP allowed as fallback
  • HSTS not configured
// Secure
  • TLS 1.3 preferred, TLS 1.2 minimum
  • ECDHE key exchange with forward secrecy
  • AES-GCM or ChaCha20-Poly1305 ciphers
  • Valid CA-signed certificates (short TTL)
  • Strict HSTS with preload and subdomains
  • HTTP→HTTPS redirect (301 Permanent)
  • Certificate Transparency (CT) monitoring
nginx — TLS hardening configuration
# /etc/nginx/conf.d/api.conf — TLS hardening
server {
    listen 443 ssl http2;
    server_name api.example.com;

    ssl_certificate     /etc/ssl/api.example.com.crt;
    ssl_certificate_key /etc/ssl/api.example.com.key;

    # Protocol versions
    ssl_protocols TLSv1.2 TLSv1.3;
    ssl_prefer_server_ciphers on;

    # Cipher suites — forward secrecy, AEAD only
    ssl_ciphers 'ECDHE-ECDSA-AES256-GCM-SHA384:ECDHE-RSA-AES256-GCM-SHA384:ECDHE-ECDSA-CHACHA20-POLY1305';

    # HSTS: 1 year, include subdomains, preload
    add_header Strict-Transport-Security "max-age=31536000; includeSubDomains; preload" always;

    # Security headers
    add_header X-Content-Type-Options    "nosniff"      always;
    add_header X-Frame-Options            "DENY"         always;
    add_header Content-Security-Policy    "default-src 'none'" always;
    add_header Referrer-Policy            "no-referrer"  always;

    # CORS — explicit origins only, no wildcard
    add_header Access-Control-Allow-Origin  "https://app.example.com";
    add_header Access-Control-Allow-Methods "GET, POST, PUT, DELETE";
    add_header Access-Control-Allow-Headers "Authorization, Content-Type";
}

Mutual TLS (mTLS) for Service-to-Service APIs

For internal microservice APIs, mutual TLS (mTLS) provides bidirectional authentication: the server validates the client's certificate in addition to the client validating the server. This eliminates entire classes of unauthorized service invocation attacks and is the recommended approach for service meshes (Istio, Linkerd) and zero-trust architectures.

ℹ

Certificate Pinning: Mobile API clients should pin the server's certificate or public key to prevent interception via rogue CA certificates. However, pinning requires careful key rotation planning — a failed rotation causes a complete application outage.

// 04 — Authentication

Authentication & JSON Web Tokens

Authentication is the mechanism by which an API verifies the identity of a caller. Broken authentication (OWASP API2:2023) encompasses a wide range of failures, from trivially crackable secrets to subtle cryptographic vulnerabilities in token handling.

JSON Web Tokens (JWT): Architecture

JWT (RFC 7519) is the dominant API authentication token format. A JWT consists of three Base64URL-encoded segments separated by dots: header.payload.signature. The header specifies the signing algorithm; the payload carries claims (identity, permissions, expiry); the signature binds the two.

JWT structure — decoded
// Header
{ "alg": "RS256", "typ": "JWT" }

// Payload
{
  "sub":  "user_9a4f2b",
  "iss":  "https://auth.example.com",
  "aud":  "https://api.example.com",
  "iat":  1716000000,
  "exp":  1716003600,   // 1 hour TTL
  "jti":  "c7f3a8e1-4b29-4d1e-891c-a2d9f04bc3f7",
  "roles": ["user"]
}

// Signature (RS256) — server signs with PRIVATE key, verifies with PUBLIC key
RSASHA256(base64url(header) + "." + base64url(payload), privateKey)

Critical JWT Vulnerabilities

1. Algorithm Confusion Attack (alg:none / RS256→HS256)

The most devastating JWT vulnerability is algorithm confusion. JWT headers contain an attacker-controlled alg field. If the server trusts this field and uses the same public key for both asymmetric verification (RS256) and as an HMAC secret (HS256), an attacker can forge valid tokens.[11,12]

The attack works as follows: the attacker obtains the server's RSA public key (often exposed at /.well-known/jwks.json), modifies the JWT header to specify HS256, and re-signs the token using the public key as the HMAC secret. Since the server uses the public key to verify HS256 tokens, the forged token passes validation.[13] Algorithm confusion vulnerabilities arise from flawed library implementations that provide a single algorithm-agnostic verification method.[14] CVE-2024-54150 is a recent example of this class of vulnerability.[15]

✗

Vulnerable pattern: verify(token, publicKey) — trusts the alg field from the token itself. An attacker can set alg: "HS256" and sign with the public key to bypass verification entirely.

Node.js — vulnerable vs. secure JWT verification
const jwt = require('jsonwebtoken');

// ✗ VULNERABLE — trusts "alg" from token header
function verifyUnsafe(token, publicKey) {
  return jwt.verify(token, publicKey);  // alg confusion risk
}

// ✓ SECURE — enforce algorithm server-side, validate all critical claims
function verifySecure(token) {
  return jwt.verify(token, process.env.JWT_PUBLIC_KEY, {
    algorithms: ['RS256'],          // explicit whitelist — NEVER trust header alg
    issuer:     'https://auth.example.com',
    audience:   'https://api.example.com',
    clockTolerance: 0,               // no clock skew tolerance
  });
}

// ✓ Additional: use decode() ONLY for inspection, NEVER for auth decisions
// jwt.decode() does NOT verify signature — common developer error

2. Signature Stripping (alg:none)

Some libraries accept tokens with alg: "none" and no signature, effectively treating the unsigned payload as a valid token. Developers sometimes confuse the library's decode() method (which only Base64-decodes, no verification) with verify().[16]

3. jwk / jku Header Injection

JWT headers may include a jwk parameter (an embedded public key) or a jku parameter (a URL from which to fetch keys). Servers that trust these headers allow attackers to supply their own signing key, generating arbitrarily valid tokens. Always whitelist keys — never accept keys from the token itself.[16]

4. Weak Secret / Brute-Force (HS256)

Symmetric HS256 tokens signed with short or guessable secrets are vulnerable to offline brute-force using tools like hashcat with the --hash-type 16500 flag. HMAC secrets should be at minimum 256 bits of cryptographic randomness.

JWT Security Checklist

  • Enforce algorithms server-side; never trust the header alg field
  • Validate exp, iat, iss, aud, and jti on every request
  • Use short-lived access tokens (15 minutes to 1 hour maximum)
  • Use asymmetric keys (RS256, ES256) in preference to symmetric HS256
  • Whitelist public keys; reject jwk and jku header parameters
  • Maintain a token revocation list / jti blocklist for logout and compromise
  • Use ES256 (ECDSA P-256) for compact tokens with strong security
  • Store tokens in HttpOnly, SameSite=Strict cookies — not localStorage
  • Rotate signing keys regularly with a key versioning scheme (kid header)
  • Use well-maintained, audited JWT libraries — never hand-roll JWT parsing

API Keys

API keys are long-lived credentials used primarily for server-to-server or developer tool access. Key principles: generate with cryptographic randomness (minimum 256 bits), scope to minimum necessary permissions, enforce per-key rate limits, support rotation without downtime, and detect exposure immediately (e.g., GitHub secret scanning). A March 2024 breach exposed nearly 13 million API secrets through public GitHub repositories.[5]

⚠

Hardcoded credentials: The Rabbit R1 AI assistant (June 2024) had API keys hardcoded in its firmware, exposing all historical responses.[5] Never embed credentials in source code, containers, or mobile binaries. Use secrets management (HashiCorp Vault, AWS Secrets Manager, or equivalent).

// 05 — Delegated Authorization

OAuth 2.0 & PKCE

OAuth 2.0 (RFC 6749) is the industry standard for delegated authorization — enabling third-party applications to act on behalf of a user without exposing their credentials. In January 2025, the IETF published RFC 9700: Best Current Practice for OAuth 2.0 Security, consolidating years of practical security research and formally deprecating several insecure patterns.[17]

RFC 9700 Key Mandates

Practice RFC 9700 Status Severity
Authorization Code Flow + PKCE MUST use for all public clients Required
Implicit Grant DEPRECATED — replaced by Auth Code + PKCE Deprecated
Resource Owner Password Credentials (ROPC) DEPRECATED — violates separation of concerns Deprecated
Exact redirect URI matching MUST — no wildcards or substring matching Required
PKCE with S256 challenge method MUST for authorization servers Required
Refresh token rotation on use RECOMMENDED — detect replay attacks Recommended
Sender-constrained tokens (DPoP / mTLS) RECOMMENDED for high-value APIs Recommended

Authorization Code Flow with PKCE

PKCE (Proof Key for Code Exchange, RFC 7636, pronounced "pixie") prevents authorization code interception attacks by cryptographically binding each authorization request to the specific client instance that initiated it.[18]

// OAuth 2.0 Authorization Code Flow with PKCE
Client: generate code_verifier (random 43-128 chars)
→
code_challenge = BASE64URL(SHA256(code_verifier))
→
Auth request with code_challenge (S256)
→
Auth Server stores challenge; returns auth code
→
Token request: auth code + code_verifier
→
Auth Server: SHA256(verifier) must equal stored challenge
→
Access token issued
JavaScript — PKCE implementation
// Generate cryptographically random code_verifier
function generateCodeVerifier() {
  const array = new Uint8Array(32);
  crypto.getRandomValues(array);
  return btoa(String.fromCharCode(...array))
    .replace(/\+/g, '-').replace(/\//g, '_').replace(/=+$/, '');
}

// Derive code_challenge using S256 method (only secure method per RFC 9700)
async function generateCodeChallenge(verifier) {
  const encoder = new TextEncoder();
  const data    = encoder.encode(verifier);
  const digest  = await crypto.subtle.digest('SHA-256', data);
  return btoa(String.fromCharCode(...new Uint8Array(digest)))
    .replace(/\+/g, '-').replace(/\//g, '_').replace(/=+$/, '');
}

// Build authorization URL — exact redirect URI, narrow scopes
async function startAuthFlow() {
  const verifier   = generateCodeVerifier();
  const challenge  = await generateCodeChallenge(verifier);
  const state      = crypto.randomUUID();  // CSRF protection

  sessionStorage.setItem('pkce_verifier', verifier);
  sessionStorage.setItem('oauth_state', state);

  const params = new URLSearchParams({
    response_type:         'code',
    client_id:             'my-client-id',
    redirect_uri:          'https://app.example.com/callback',  // exact URI
    scope:                 'openid profile email',               // minimal scopes
    state,
    code_challenge:        challenge,
    code_challenge_method: 'S256',
  });

  window.location.href = `https://auth.example.com/authorize?${params}`;
}

Dangerous OAuth Patterns to Avoid

✗

Implicit Flow exposes access tokens directly in the URL fragment, leaking through browser history, referrer headers, and server logs. RFC 9700 deprecates it unconditionally — migrate all SPAs and native apps to Authorization Code + PKCE.[19]

✗

Open Redirectors: Authorization servers must not accept redirect URIs containing open redirectors. Use exact string matching — no wildcard domains, no path prefixes. Validate on every request, not just registration.[18]

⚠

Scope Minimization: Request the minimum scopes necessary for the operation. Broad scopes (e.g., *, admin) dramatically amplify the impact of token theft. Prefer fine-grained scopes: orders:read, profile:read.

// 06 — Authorization

Access Control

Authentication establishes identity; authorization determines what that identity may do. The distinction matters enormously — BOLA and BFLA, the two most common API vulnerabilities, are exclusively authorization failures, occurring in systems where authentication is functional.

Broken Object Level Authorization (BOLA)

BOLA occurs when an API endpoint receives an object identifier (user ID, order ID, document ID) from the client and retrieves that object without verifying the caller owns or has permission to access it. It has ranked #1 on the OWASP API Security Top 10 since 2019, comprising approximately 40% of all API attacks.[7]

Python / FastAPI — BOLA: vulnerable vs. secure
# ✗ VULNERABLE — blindly serves any order_id in the URL
@app.get("/api/orders/{order_id}")
async def get_order_vulnerable(order_id: int, db: Session = Depends(get_db)):
    order = db.query(Order).filter(Order.id == order_id).first()
    return order  # any authenticated user gets any order

# ✓ SECURE — enforce ownership at the query level
@app.get("/api/orders/{order_id}")
async def get_order_secure(
    order_id: int,
    current_user: User = Depends(get_current_user),
    db:           Session = Depends(get_db)
):
    order = db.query(Order).filter(
        Order.id == order_id,
        Order.user_id == current_user.id,  ← ownership enforced in query
    ).first()

    if not order:
        raise HTTPException(status_code=404, detail="Not found")  ← 404, not 403
    return order
ℹ

Return 404, not 403 for unauthorized objects. Returning 403 confirms the object exists, enabling enumeration. A 404 reveals nothing about the existence of resources the requester cannot access.

Role-Based and Attribute-Based Access Control

RBAC (Role-Based Access Control) assigns permissions to roles and roles to users. ABAC (Attribute-Based Access Control) extends this with contextual attributes — time of day, resource owner, data classification, network location. For complex domains, implement a policy engine (Open Policy Agent, Cedar, or Casbin) rather than embedding authorization logic in application code.

Open Policy Agent (OPA) — API authorization policy (Rego)
# policy/api_access.rego
package api.authz

import future.keywords.if

# Default deny
default allow := false

# Users may only access their own resources
allow if {
    input.method == "GET"
    input.path[0] == "api"
    input.path[1] == "orders"
    order := data.orders[input.path[2]]
    order.user_id == input.user.id          # ownership check
}

# Admins may access any resource
allow if {
    input.user.roles[_] == "admin"
}

# Write operations require explicit write permission
allow if {
    input.method in {"POST", "PUT", "PATCH", "DELETE"}
    "write" in input.user.permissions
    input.user.id == data.resources[input.resource_id].owner_id
}

Function-Level Authorization (BFLA)

Administrative endpoints (DELETE /api/users/{id}, POST /api/admin/role) must require explicit privilege checks beyond basic authentication. Common failures include trusting HTTP method semantics without re-checking role, exposing admin endpoints under predictable paths, and using UI-only hiding as the sole access control mechanism.

⚠

Security through obscurity is not access control. Hiding admin endpoints under non-obvious URLs provides no protection. Attackers systematically enumerate paths. Every sensitive function must perform server-side role/permission verification.

Principle of Least Privilege

Service accounts, API keys, and OAuth tokens should carry the minimum permissions required for their specific task. Implement fine-grained scopes, time-limited tokens, and purpose-specific API keys. Audit and rotate credentials regularly. Revoke immediately on suspicion of compromise.

// 07 — Input Security

Input Validation & Processing Security

Every value entering your API — path parameters, query strings, headers, request bodies, file uploads — is potentially adversarial. Robust input validation is the foundation of injection defense, business logic integrity, and mass assignment prevention.

Schema Validation

Define and enforce strict request schemas. Reject any request that does not conform. Use battle-tested validation libraries, not hand-written regex. OpenAPI schemas double as both documentation and validation contracts.

Python — Pydantic schema validation with strict type enforcement
from pydantic import BaseModel, Field, EmailStr, field_validator, ConfigDict
from typing import Annotated
import re

class CreateUserRequest(BaseModel):
    # Strict mode: rejects extra fields (prevents mass assignment)
    model_config = ConfigDict(extra='forbid', strict=True)

    username: Annotated[str, Field(min_length=3, max_length=32, pattern=r'^[a-zA-Z0-9_]+$')]
    email:    EmailStr
    age:      Annotated[int, Field(ge=13, le=120)]
    bio:      Annotated[str, Field(max_length=500)] | None = None

    @field_validator('bio')
    @classmethod
    def sanitize_bio(cls, v: str | None) -> str | None:
        if v is None: return None
        # Strip HTML tags; server-side only
        return re.sub(r'<[^>]+>', '', v).strip()

# Extra fields are rejected — prevents mass assignment of internal fields
# e.g.: {"username":"x","email":"x@x.com","age":25,"is_admin":true}  → 422

SQL Injection Prevention

SQL injection remains the second most common API vulnerability in 2025 at 20% of incidents.[6] The defense is absolute: use parameterized queries or prepared statements — never string concatenation. ORMs mitigate risk but don't eliminate it if raw query escape hatches are used carelessly.

SQL — parameterized queries (safe) vs string interpolation (fatal)
# ✗ FATAL — SQL injection: input: ' OR '1'='1
query = f"SELECT * FROM users WHERE email = '{user_input}'"
#         → SELECT * FROM users WHERE email = '' OR '1'='1'

# ✓ SAFE — parameterized query; driver escapes all input
cursor.execute(
    "SELECT * FROM users WHERE email = %s AND active = %s",
    (user_input, True)
)

Server-Side Request Forgery (SSRF) Prevention

SSRF (OWASP API7:2023) occurs when an API accepts a URL from a client and fetches it server-side without validation, allowing attackers to probe internal services, access cloud metadata endpoints (AWS: 169.254.169.254, GCP: metadata.google.internal), or scan internal networks.[7]

Python — SSRF prevention with URL allowlisting
import ipaddress, socket
from urllib.parse import urlparse

ALLOWED_SCHEMES = {'https'}
ALLOWED_HOSTS   = {'cdn.partner.com', 'images.trusted.io'}
PRIVATE_RANGES  = [
    ipaddress.ip_network('10.0.0.0/8'),
    ipaddress.ip_network('172.16.0.0/12'),
    ipaddress.ip_network('192.168.0.0/16'),
    ipaddress.ip_network('169.254.0.0/16'),   # link-local / metadata
    ipaddress.ip_network('127.0.0.0/8'),       # loopback
]

def validate_url(url: str) -> str:
    parsed = urlparse(url)

    if parsed.scheme not in ALLOWED_SCHEMES:
        raise ValueError(f"Scheme '{parsed.scheme}' not allowed")

    if parsed.hostname not in ALLOWED_HOSTS:
        raise ValueError(f"Host not in allowlist")

    # Resolve DNS and block private IP ranges (DNS rebinding protection)
    ip = ipaddress.ip_address(socket.gethostbyname(parsed.hostname))
    for private_range in PRIVATE_RANGES:
        if ip in private_range:
            raise ValueError(f"Private IP blocked: {ip}")

    return url

File Upload Security

  • Validate file type by magic bytes (content inspection), not extension or MIME type header
  • Set maximum file size limits and aggregate upload quotas per user
  • Store uploaded files outside the web root; never in a publicly accessible directory
  • Scan with antivirus / content inspection before processing
  • Rename files server-side; strip path traversal sequences (../) from filenames
  • Serve user-uploaded content from a separate, sandboxed domain
// 08 — Resource Protection

Rate Limiting & Resource Protection

Unrestricted Resource Consumption (OWASP API4:2023) covers any failure to bound the resources an API consumes on a caller's behalf. Without limits, a single client can exhaust CPU, memory, database connections, third-party API quotas, and network bandwidth — enabling denial of service, credential brute-forcing, and data scraping at scale.[7]

Rate Limiting Strategies

AlgorithmBehaviorUse Case
Fixed Window N requests per window; resets at boundary Simple scenarios; beware boundary bursts
Sliding Window N requests in any rolling time window Smoother limiting, higher memory cost
Token Bucket Burst allowed up to bucket capacity; refills at rate R APIs with legitimate burst needs (file uploads)
Leaky Bucket Requests queued at fixed output rate Strict downstream rate control
Python — Redis sliding window rate limiter
import time, redis
from fastapi import HTTPException, Request

r = redis.Redis(host='localhost', decode_responses=True)

def rate_limit(request: Request, limit: int = 100, window: int = 60):
    # Key: per user + endpoint; rate limits must be endpoint-specific
    user_id  = request.state.user.id
    endpoint = request.url.path
    key      = f"rl:{user_id}:{endpoint}"
    now      = time.time()
    window_start = now - window

    pipe = r.pipeline()
    pipe.zremrangebyscore(key, 0, window_start)   # remove old entries
    pipe.zadd(key, {str(now): now})                # add current request
    pipe.zcard(key)                                  # count in window
    pipe.expire(key, window)
    _, _, count, _ = pipe.execute()

    if count > limit:
        raise HTTPException(
            status_code=429,
            detail="Rate limit exceeded",
            headers={
                "Retry-After":       str(window),
                "X-RateLimit-Limit": str(limit),
                "X-RateLimit-Reset": str(int(window_start + window)),
            }
        )

Rate Limit Scope

  • Rate limit by user ID (authenticated), IP (unauthenticated), and API key
  • Apply stricter limits to authentication endpoints (login, password reset, OTP)
  • Implement request body size limits and execution timeout ceilings
  • Cap third-party API spending: SMS, email, payment operations per period
  • Return 429 Too Many Requests with Retry-After and rate limit headers
  • Implement progressive back-off and CAPTCHA after repeated failures
  • Set database query timeouts and result set size limits server-side

Business Logic Rate Limiting

Beyond raw request rates, APIs must protect sensitive business flows (OWASP API6:2023). A ticket-purchase endpoint may only receive 1 request per user per event. A referral endpoint may only award credit N times per user. A password-reset flow must detect and block rapid enumeration of phone numbers or emails. These constraints cannot be expressed as simple rate limits — they require understanding of the application's business rules.[7]

// 09 — Output Security

Output Encoding & Data Handling

Response Filtering and Data Minimization

APIs should return only the fields the requesting principal is authorized to see, never the full database record. Broken Object Property Level Authorization (OWASP API3:2023) occurs when the API returns fields the user cannot see (information disclosure) or accepts fields the user cannot set (mass assignment). These share the same root cause: absent field-level authorization.[7]

Python — response serialization with field-level authorization
from pydantic import BaseModel

# Public user view — safe to return to any authenticated caller
class UserPublicResponse(BaseModel):
    id:       int
    username: str
    bio:      str | None
    # ← does NOT include: password_hash, email, phone, is_admin, api_key

# Own-account view — returned only to the account owner
class UserSelfResponse(UserPublicResponse):
    email:    str
    phone:    str | None

# Admin view — returned only to admin role
class UserAdminResponse(UserSelfResponse):
    is_admin: bool
    login_ip: str
    created:  str

@app.get("/api/users/{user_id}")
async def get_user(user_id: int, caller: User = Depends(get_current_user)):
    user = db.get_user(user_id)
    if 'admin' in caller.roles:
        return UserAdminResponse.model_validate(user)
    if caller.id == user_id:
        return UserSelfResponse.model_validate(user)
    return UserPublicResponse.model_validate(user)

Error Handling and Information Disclosure

Verbose error messages, stack traces, database error strings, and internal path disclosures in API responses are a significant information leakage vector. In production: return generic error messages to clients, log full details server-side only, use correlation IDs to link client-facing errors to server logs.

Generic error response schema
// ✗ Information leaking — never return this to clients
{
  "error": "PsycopgError: relation 'users' does not exist at character 25",
  "file":  "/app/src/database/queries.py",
  "line":  142
}

// ✓ Safe — generic message, correlation ID for internal lookup
{
  "error":          "An internal error occurred",
  "correlation_id": "7f3a9c12-bc4e-4d21-8f99-2a1c05e3b80a"
}

HeaderRecommended ValuePurpose
Content-Type application/json; charset=utf-8 Prevent MIME sniffing attacks
X-Content-Type-Options nosniff Block MIME type sniffing
Strict-Transport-Security max-age=31536000; includeSubDomains; preload Force HTTPS for 1 year

CI/CD & DevSecOps Integration

Security in the CI/CD pipeline — often called "shifting left" — catches vulnerabilities before they reach production. The core principle is that security checks should be automated, repeatable, and part of the normal development workflow rather than a gating step performed by a separate security team.

Pipeline Security Gates

// API Security Gate: CI/CD Pipeline
Code Push
→
SAST (Semgrep, Bandit)
→
Dependency Scan (OWASP DC, Snyk)
→
Secret Detection (Gitleaks, truffleHog)
→
Container Scan (Trivy, Grype)
→
DAST (OWASP ZAP, Nuclei)
→
API Contract Test
→
Deploy
GitHub Actions — API security pipeline
# .github/workflows/api-security.yml
name: API Security Gates

on: [push, pull_request]

jobs:
  security:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with: { fetch-depth: 0 }

      # Static analysis — catches injection patterns, hardcoded secrets
      - name: SAST (Semgrep)
        uses: semgrep/semgrep-action@v1
        with:
          config: "p/owasp-top-ten p/secrets"

      # Secret scanning — blocks commits containing credentials
      - name: Secret Detection
        uses: gitleaks/gitleaks-action@v2
        env:
          GITLEAKS_LICENSE: ${{ secrets.GITLEAKS_LICENSE }}

      # Dependency vulnerabilities — fails on HIGH+ CVEs
      - name: Dependency Audit (Python)
        run: pip-audit --require-hashes -r requirements.txt

      # Container image scanning
      - name: Container Scan (Trivy)
        uses: aquasecurity/trivy-action@master
        with:
          image-ref: 'ghcr.io/myorg/api:${{ github.sha }}'
          severity:  'HIGH,CRITICAL'
          exit-code: '1'

      # API schema validation — ensures OpenAPI spec stays current
      - name: OpenAPI Lint
        run: npx @stoplight/spectral-cli lint openapi.yaml

      # DAST against staging environment
      - name: DAST (OWASP ZAP)
        uses: zaproxy/action-api-scan@v0.7.0
        with:
          target: 'https://api-staging.example.com'
          format: 'openapi'
          rules_file_name: '.zap/rules.tsv'

API Versioning and Inventory Management

Improper inventory management (OWASP API9:2023) is a primary enabler of attacks against legacy endpoints. Attackers specifically seek deprecated API versions because they are unmonitored and unpatched.[7] Best practices:

  • Maintain a complete, machine-readable API inventory (OpenAPI / AsyncAPI specs)
  • Version APIs explicitly (/v1/, /v2/) with documented end-of-life dates
  • Automatically discover APIs in production (traffic-based discovery tools)
  • Retire old versions fully — 301 redirect to new version, then decommission
  • Include API version headers in all responses for traceability
  • Treat internal and partner APIs with identical security rigor as public APIs

Infrastructure as Code Security

  • Scan IaC (Terraform, CloudFormation) with Checkov or tfsec before deployment
  • Enforce security groups deny all inbound except necessary ports
  • Validate that API Gateways have authentication and rate limiting enabled
  • Run Kubernetes admission controllers (OPA/Gatekeeper) to enforce security policies
  • Sign container images; verify signatures in deployment admission webhooks
// 11 — Detection & Response

Monitoring & Incident Response

Only 21% of organizations report a high ability to detect attacks at the API layer, and 38% of organizations discovered API breaches only after external reporting — meaning more than a third of breaches are disclosed by researchers, journalists, or the attackers themselves rather than the victim's own detection capability.[2,6] Effective monitoring is the gap between a security incident and a public breach.

What to Log

Log at the API gateway or middleware layer to capture all requests uniformly, without scattering logging logic across services. Each log entry should contain:

JSON structured API access log — schema
{
  "timestamp":      "2025-05-17T14:32:01.447Z",
  "correlation_id": "8e3f7b29-c014-4a9d-b6d1-03ee4a1c5f80",
  "method":         "GET",
  "path":           "/api/v2/orders/4891",
  "status":         200,
  "latency_ms":     43,
  "user_id":        "usr_7a4f2b",
  "client_ip":      "203.0.113.42",
  "user_agent":     "MyApp/3.1 iOS/17",
  "api_version":    "v2",
  "token_jti":      "c7f3a8e1-4b29-4d1e-891c",  // for revocation audit
  "request_size":   0,
  "response_size":  1247,

  // NEVER log: passwords, tokens, payment card numbers, SSNs, PII in bodies
  // Redact Authorization header value; log only presence/absence
  "auth_present": true
}

Threat Detection Signals

PatternLikely AttackResponse
Sequential object IDs across different user accounts BOLA enumeration Rate limit; alert; block IP
Repeated 401/403 responses from a single source Credential stuffing / brute force Progressive lockout; CAPTCHA; MFA prompt
Spike in 429 responses; high-frequency identical requests DoS / scraping Block; WAF rule; notify NOC
JWT validation failures: expired, wrong issuer, bad sig Token replay / forgery Revoke token family; alert
Requests to internal hostnames or metadata IPs SSRF Block; high-severity alert immediately
Unusual off-hours access to admin endpoints Insider threat / compromised account Session revoke; escalate to security team
Access to deprecated API versions Exploitation of unpatched endpoints Block; decommission; audit history

Incident Response Playbook

  • Detect: SIEM alerts, anomaly detection, external reports — triage within 15 minutes
  • Contain: Revoke affected tokens/keys, block source IPs, isolate services if necessary
  • Investigate: Correlate logs by correlation_id, determine blast radius, identify compromised data
  • Eradicate: Patch the root vulnerability; rotate all potentially exposed credentials
  • Recover: Restore from verified clean state; re-enable services progressively
  • Notify: Regulatory notification (GDPR: 72 hours), affected users, public disclosure per policy
  • Review: Post-incident retrospective; update playbooks and detection rules

Security Testing Cadence

// Continuous (Automated)
  • SAST on every commit
  • Dependency vulnerability scanning
  • Secret detection in git hooks
  • Contract testing (API schema drift)
  • DAST on staging deploys
// Periodic (Manual)
  • Annual third-party penetration test
  • Quarterly internal red team exercise
  • Bug bounty program (continuous)
  • Annual threat model review
  • After major architecture changes
