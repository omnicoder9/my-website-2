Section 01 — Context

The cloud-native inflection point

Modern Spring applications are cloud-native by default, not by exception. The question facing an architect in 2026 is no longer whether a Spring service will run in a container under an orchestrator, but how to make that path so frictionless that a developer can carry an idea from a feature branch to a horizontally-scaled, observable production deployment without leaving their editor. That shift is now backed by hard adoption data.

The Cloud Native Computing Foundation's most recent annual survey found that 82% of container users run Kubernetes in production, up from 66% two years earlier, and that 98% of surveyed organizations have adopted cloud-native techniques in some form.1 The prior year's report had already recorded production Kubernetes usage climbing to 80% — an annual growth rate above twenty percent — with 93% of organizations using, piloting, or actively evaluating it.2 The economic substrate beneath that adoption is expanding just as fast: the application container market was valued at roughly $5.85 billion in 2024 and is projected to reach $31.5 billion by 2030, a compound annual growth rate of about 33.5%.3

For the Spring ecosystem specifically, this is not a bolt-on. Spring Cloud exists precisely to industrialize the recurring patterns of distributed systems — configuration management, service discovery, circuit breakers, intelligent routing, a control bus, short-lived services, and contract testing — so that teams can stand those patterns up quickly and run them identically on a laptop, in a bare-metal datacenter, or on a managed platform.4 The release train cadence keeps it aligned with Spring Boot: the 2025.1.x line (codenamed Oakwood) tracks Spring Boot 4.0, moving Spring Cloud Kubernetes to a 5.0.x line to align with the rest of the train and continuing the long deprecation sunset of the Netflix-era components.5

Why "twelve-factor" still anchors the discipline

Cloud-native Spring is, at heart, a disciplined application of twelve-factor principles. Configuration lives in the environment, not in the artifact. Backing services — databases, brokers, caches — are attached resources reached by URL and credentials injected at runtime. Processes are stateless and disposable, designed to start fast and shut down gracefully so an orchestrator can move them at will. Logs are event streams written to stdout, not files an operator tails. Each of these maps directly onto a concrete Spring or platform mechanism, and the rest of this guide is, in effect, a tour of that mapping.

The unit of deployment is no longer the application. It is the contract between an immutable image, a declarative manifest, and a pipeline that can reproduce both on demand.// the cloud-native thesis

What "good" looks like, measured

The DevOps Research and Assessment (DORA) program gives us the yardstick. Its 2024 Accelerate State of DevOps report — drawn from tens of thousands of practitioners — classifies delivery organizations into elite, high, medium, and low tiers, with roughly 19% reaching the elite cluster.6 The gap between the top and bottom is not marginal: elite performers were found to deploy on the order of 182 times more frequently, with change lead times around 127 times faster, change-failure rates roughly eight times lower, and recovery from a failed deployment thousands of times quicker.7 Those multipliers are the prize that a well-built Spring delivery pipeline is reaching for.

Reality check The same 2024 research surfaced a cautionary signal worth holding in mind throughout this guide: a 25% increase in AI adoption correlated with a small decrease in delivery throughput and a larger decrease in stability.8 Automation and tooling do not buy performance on their own; small batch sizes, fast feedback, and trustworthy pipelines do. Everything below is in service of that.
Section 02 — Architecture

Anatomy of a Spring microservices estate

To make the deployment story concrete, we will carry one reference system through the entire article: an e-commerce backend called Aurora. It is deliberately minimal but architecturally complete — everything you need to exercise the full cloud-native toolchain, nothing you would have to apologize for in a design review.

The services

Aurora decomposes into an edge-gateway (the single ingress point), a catalog-service (products and inventory), an order-service (order placement and orchestration), a payment-service, plus the platform services that hold the estate together: a config-server and a discovery-server. The order service synchronously validates stock against the catalog and asynchronously emits an event the payment service consumes — giving us both request/response and event-driven communication to deploy and observe.

Configuration: externalized and versioned

Spring Cloud Config Server centralizes configuration for the whole estate in a versioned backing store — typically a Git repository — eliminating the duplication and drift that otherwise accumulates across environments.9 Each service is a config client: it reads its profile-specific properties at startup and can refresh them at runtime via the actuator. In Kubernetes you will often complement or replace this with Spring Cloud Kubernetes, which projects ConfigMap and Secret objects directly into the Spring Environment, but the Config Server remains the portable, platform-agnostic choice.

catalog-service/src/main/resources/application.yml
spring:
  application:
    name: catalog-service
  config:
    import: "optional:configserver:http://config-server:8888"
  datasource:
    url: ${CATALOG_DB_URL:jdbc:postgresql://localhost:5432/catalog}
    username: ${CATALOG_DB_USER:aurora}
    password: ${CATALOG_DB_PASSWORD}   # injected from a Secret
management:
  endpoints:
    web:
      exposure:
        include: health,info,metrics,prometheus
  endpoint:
    health:
      probes:
        enabled: true   # exposes liveness/readiness groups

Service discovery and routing

Discovery lets services register themselves and locate each other dynamically rather than depending on hard-coded hosts. Spring Cloud Netflix Eureka remains the canonical self-hosted registry, with clients registering on startup and refreshing the registry on a configurable interval.9 In front of it sits Spring Cloud Gateway, the modern, reactive API gateway built on Spring WebFlux and Project Reactor; because it runs on the Netty event-loop runtime rather than a servlet container, it handles high-fan-out routing and filtering with a small thread footprint.10 With discovery integration enabled, the gateway can route to a registered service by name using the lb://service-name scheme, deferring instance selection to the client-side load balancer.

edge-gateway/src/main/resources/application.yml
spring:
  cloud:
    gateway:
      discovery:
        locator:
          enabled: true
          lower-case-service-id: true
      routes:
        - id: catalog
          uri: lb://catalog-service
          predicates:
            - Path=/api/catalog/**
          filters:
            - CircuitBreaker=name=catalogCB,fallbackUri=forward:/fallback/catalog
        - id: orders
          uri: lb://order-service
          predicates:
            - Path=/api/orders/**
Migration note Treat the Netflix lineage as a museum of working ideas, not a target. Ribbon has been superseded by Spring Cloud LoadBalancer, and Hystrix by Spring Cloud Circuit Breaker over Resilience4j.11 Eureka is still actively used, but on Kubernetes the platform's own DNS-based service discovery often makes a dedicated registry redundant. Choose discovery to match the floor you deploy on.

Event-driven communication

Synchronous calls are the wrong tool for the order-to-payment hop: we do not want order placement to fail because the payment service is briefly slow, and we do want the order recorded even if payment is processed seconds later. This is where event-driven messaging earns its place, and Spring Cloud Stream supplies the abstraction — a binder-based programming model for message-driven microservices that decouples your code from the specific broker (Kafka, RabbitMQ, or a cloud-managed equivalent) behind it.9 A producer is a Supplier or an explicit send; a consumer is a Consumer<T> bean; the binder wires both to topics declared in configuration. The deployment consequence is important: an event-driven service can be down for maintenance and lose no work, because the broker buffers events until it returns — which is precisely the disposability twelve-factor asks for.

order-service: publishing an OrderPlaced event
// functional style: the bean name binds to a configured destination
@Bean
public Supplier<Flux<OrderPlaced>> orderEvents(OrderSink sink) {
    return sink::drain;   // emitted to topic 'orders.placed'
}
# application.yml — broker is a deployment detail, not a code detail
spring:
  cloud:
    stream:
      bindings:
        orderEvents-out-0: { destination: orders.placed }
      kafka:
        binder: { brokers: ${KAFKA_BROKERS:localhost:9092} }

Because the broker URL is injected from the environment, the same image consumes a local Kafka in development, a managed broker in production, and a different broker entirely on another cloud — without a recompile. Combined with the circuit breaker on the synchronous catalog call, Aurora now mixes both interaction styles deliberately: synchronous where it needs an immediate answer, asynchronous where it needs durability and decoupling.

Section 03 — Reliability

Resilience and observability before you deploy

A microservice that has not been made resilient and observable is not ready for an orchestrator, because the orchestrator will move it, restart it, and scale it on signals it cannot read without help. Two concerns deserve attention before any container is built: fault tolerance and telemetry.

Circuit breakers, retries, and bulkheads

The circuit breaker is the load-bearing resilience pattern in a distributed system: it detects when a downstream dependency is failing and short-circuits further calls, returning a fallback rather than letting failures cascade. Spring Cloud provides a vendor-neutral abstraction over several implementations, with Resilience4j as the recommended backend now that Hystrix is deprecated.11 Resilience4j layers four cooperating concerns — circuit breaking, retry, rate limiting, and bulkheading — configured declaratively per instance.

order-service: resilience4j configuration
resilience4j:
  circuitbreaker:
    instances:
      catalogCB:
        sliding-window-size: 20
        failure-rate-threshold: 50            # % failures to trip open
        wait-duration-in-open-state: 10s
        permitted-number-of-calls-in-half-open-state: 5
  retry:
    instances:
      catalogCB:
        max-attempts: 3
        wait-duration: 200ms
@CircuitBreaker(name = "catalogCB", fallbackMethod = "reservedFromCache")
@Retry(name = "catalogCB")
public StockStatus checkStock(String sku) {
    return catalogClient.stockFor(sku);   // OpenFeign HTTP call
}

private StockStatus reservedFromCache(String sku, Throwable t) {
    return stockCache.lastKnown(sku).orElse(StockStatus.UNKNOWN);
}

Observability: the Micrometer era

Spring Framework 6 and Spring Boot 3 introduced Spring Observability, an initiative built on Micrometer and Micrometer Tracing that replaced the older Spring Cloud Sleuth.12 The practical payoff is that a service emits dimensional metrics and propagates trace context across boundaries with almost no bespoke code: you add a tracing bridge dependency — micrometer-tracing-bridge-otel for OpenTelemetry or -brave for OpenZipkin Brave — plus an exporter, and Spring Boot auto-configures the necessary beans.13 Because instrumentation lives in the libraries (web clients, Kafka, JDBC), trace IDs flow through the request path automatically, and crucially, the approach works in natively-compiled images where Java agents do not.

The Spring team has continued to deepen first-class OpenTelemetry support, treating OTLP over Micrometer Tracing as the recommended, vendor-neutral wire format for metrics, traces, and logs.14 In a Kubernetes deployment, the canonical topology is to run an OpenTelemetry Collector in-cluster and have each service export to it over OTLP (gRPC on 4317 or HTTP on 4318), letting the Collector fan out to backends such as Prometheus, Tempo, Jaeger, or a commercial APM.13

common observability dependencies + properties
<!-- pom.xml -->
<dependency>
  <groupId>io.micrometer</groupId>
  <artifactId>micrometer-tracing-bridge-otel</artifactId>
</dependency>
<dependency>
  <groupId>io.opentelemetry</groupId>
  <artifactId>opentelemetry-exporter-otlp</artifactId>
</dependency>
# application.yml
management:
  tracing:
    sampling:
      probability: 1.0          # sample everything in dev; lower in prod
  otlp:
    tracing:
      endpoint: http://otel-collector:4318/v1/traces

Section 04 — Packaging

Containerizing Spring without footguns

A container image is the immutable artifact at the center of cloud-native delivery. The naive approach — FROM eclipse-temurin, COPY app.jar, java -jar — works, but it wastes build-cache efficiency, bloats images, and bakes in security debt. Spring offers three materially better paths.

Path 1: Layered jars and multi-stage Dockerfiles

Spring Boot's layered jar format splits the fat jar into layers ordered by how often they change — dependencies, then snapshot dependencies, then loaders, then your application classes and resources. A multi-stage Dockerfile that extracts these layers means that when only your code changes, only the small final layer is rebuilt and re-pushed; every other layer is reused from cache.15 The wins compound across hundreds of CI runs.

Dockerfile — layered, multi-stage
# --- stage 1: extract layers ---
FROM eclipse-temurin:21-jdk AS builder
WORKDIR /app
COPY target/order-service.jar app.jar
RUN java -Djarmode=tools -jar app.jar extract --layers --launcher

# --- stage 2: minimal runtime ---
FROM eclipse-temurin:21-jre
WORKDIR /app
COPY --from=builder /app/app/dependencies/ ./
COPY --from=builder /app/app/spring-boot-loader/ ./
COPY --from=builder /app/app/snapshot-dependencies/ ./
COPY --from=builder /app/app/application/ ./
USER 1001
ENTRYPOINT ["java", "-jar", "app.jar"]

Path 2: Cloud Native Buildpacks (no Dockerfile)

Since Spring Boot 2.3, the framework has integrated Cloud Native Buildpacks, so a single Maven goal — mvn spring-boot:build-image — or Gradle task — gradle bootBuildImage — produces an OCI-conformant image from source, with no Dockerfile to maintain.16 The Paketo buildpacks that back this integration apply sane, security-patched base images and automatically exploit the layered-jar structure, so only the small application layer changes between builds.15 For most teams this is the lowest-toil default; you get reproducible, well-structured images without owning base-image hygiene.

# build directly to a tagged image, no Dockerfile
./mvnw spring-boot:build-image \
  -Dspring-boot.build-image.imageName=registry.example.com/aurora/order-service:1.4.0

Path 3: GraalVM native images

For the lowest startup latency and smallest memory footprint, you can compile a Spring Boot application ahead of time into a GraalVM native executable. You can do this either with the GraalVM native-image tool directly or, more conveniently, through buildpacks — the Paketo native-image buildpack downloads GraalVM and produces a container holding a statically-compiled binary with no JVM inside, which is why the resulting images are dramatically smaller.17 The performance delta is the headline: a service that took several seconds to start and held hundreds of megabytes of heap at idle on the JVM can start in roughly 150 milliseconds and run comfortably in tens of megabytes once compiled natively.18

~150ms
Native image cold start, versus multiple seconds on the JVM.18
~64MB

5–20min

Tradeoff Native images compile for a target OS and architecture, so you must build inside a Linux container for Linux deployment, and AOT compilation closes the world: reflection, dynamic proxies, and resources that Spring cannot infer require explicit hints.18 Reserve native images for workloads where cold-start and density genuinely matter — serverless, scale-to-zero, and very dense estates — rather than adopting them reflexively.

Whichever path you choose, the non-negotiables are the same: run as a non-root user, expose the actuator health probes the orchestrator will poll, externalize all configuration, and keep secrets out of the image entirely. The image should be inert and identical across every environment; only its injected configuration distinguishes staging from production.

Section 05 — Orchestration

Orchestration with Kubernetes

Kubernetes is the substrate. It holds roughly 92% of the container-orchestration market and counts millions of developers in its community, and it has matured from an emerging technology into the assumed default for running containers at scale.19 Architecturally relevant to anyone operating it: the project follows an N-2 support policy — the three most recent minor versions receive fixes — on a roughly 15-week release cadence, giving any given release about fourteen months of support.20 That cadence is brisk, and a cloud-native Spring estate must budget for cluster upgrades as routine, not exceptional, work.

The objects that matter for Spring

Deploying a Spring service to Kubernetes is a matter of declaring a handful of objects: a Deployment (the desired replica set of pods), a Service (a stable in-cluster name and virtual IP), a ConfigMap and Secret (externalized configuration and credentials), an Ingress or Gateway (external exposure), and a HorizontalPodAutoscaler (scaling on metrics). The single most important integration point between Spring and the platform is the health-probe contract.

Probes: the contract that makes a service movable

Spring Boot Actuator exposes liveness and readiness probe groups out of the box. The orchestrator polls readiness to decide whether to send traffic to a pod and liveness to decide whether to restart it. Wiring these correctly is what lets Kubernetes roll, drain, and reschedule your service without dropping requests.

k8s/order-service-deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: order-service
spec:
  replicas: 3
  selector:
    matchLabels: { app: order-service }
  template:
    metadata:
      labels: { app: order-service }
    spec:
      containers:
        - name: order-service
          image: registry.example.com/aurora/order-service:1.4.0
          ports: [{ containerPort: 8080 }]
          envFrom:
            - configMapRef: { name: order-config }
            - secretRef: { name: order-secrets }
          readinessProbe:
            httpGet: { path: /actuator/health/readiness, port: 8080 }
            initialDelaySeconds: 10
          livenessProbe:
            httpGet: { path: /actuator/health/liveness, port: 8080 }
            initialDelaySeconds: 20
          resources:
            requests: { cpu: "250m", memory: "512Mi" }
            limits:   { memory: "768Mi" }
k8s/order-service-svc-hpa.yaml
apiVersion: v1
kind: Service
metadata: { name: order-service }
spec:
  selector: { app: order-service }
  ports: [{ port: 80, targetPort: 8080 }]
---
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata: { name: order-service }
spec:
  scaleTargetRef: { apiVersion: apps/v1, kind: Deployment, name: order-service }
  minReplicas: 3
  maxReplicas: 12
  metrics:
    - type: Resource
      resource: { name: cpu, target: { type: Utilization, averageUtilization: 65 } }

Spring Cloud Kubernetes

When you commit to Kubernetes as the floor, Spring Cloud Kubernetes lets the platform's own primitives do work that Spring Cloud Netflix used to do in-process: ConfigMap and Secret objects become property sources, and Kubernetes DNS-based service discovery and load balancing can replace Eureka and Ribbon entirely. In the 2025.x release train this module aligned to a 5.0.x line with the rest of Spring Cloud.5 The architectural decision is whether your services should be Kubernetes-aware or Kubernetes-agnostic; the former is leaner on a single platform, the latter more portable across platforms and local development.

Zero-downtime rollouts depend on graceful shutdown

A rolling update only avoids dropped requests if each terminating pod finishes in-flight work before exiting. Two settings make this reliable. First, enable Spring Boot's graceful shutdown so the embedded server stops accepting new requests but completes active ones within a grace period. Second, align Kubernetes' terminationGracePeriodSeconds and a preStop hook with that window, so the kubelet does not SIGKILL the container mid-request. The readiness probe does the rest: as soon as the pod receives its termination signal it should fail readiness, which removes it from the Service's endpoints and stops new traffic arriving while it drains.

graceful shutdown — Spring + Kubernetes, in concert
# application.yml
server:
  shutdown: graceful
spring:
  lifecycle:
    timeout-per-shutdown-phase: 25s
# deployment spec excerpt
terminationGracePeriodSeconds: 30
lifecycle:
  preStop:
    exec: { command: ["sh", "-c", "sleep 5"] }   # let endpoints update first

A platform that moves under you

Kubernetes evolves quickly, and recent releases have steadily reduced the friction of running stateful, resilient Spring workloads. Native sidecar containers — long the de-facto pattern for proxies and agents — graduated to stable, giving sidecars proper lifecycle ordering so they start before and terminate after the main container.21 In-place pod resource resizing reached general availability, allowing CPU and memory adjustments without restarting pods — a meaningful win for stateful Spring services that previously had to be recreated to be vertically scaled.22 Tracking these changes is part of operating a cloud-native estate responsibly.

A pod is cattle, not a pet — but only if your Spring service starts fast, drains gracefully, and tells the kubelet the truth about its own health.// the probe contract, restated
Section 06 — Enterprise Platform

The OpenShift layer

Red Hat OpenShift is an enterprise Kubernetes distribution: it takes upstream Kubernetes and adds a curated, opinionated set of tooling, security policy, and developer workflow on top, aiming to deliver a complete, batteries-included application platform rather than a bare orchestrator.23 For Spring teams in regulated or large enterprises, the distinction is operationally significant.

What OpenShift adds over vanilla Kubernetes

Several additions matter directly to how you deploy Spring services. Routes predate and complement Kubernetes Ingress, providing integrated HAProxy-based external routing with automatic TLS, sticky sessions, and native support for blue-green and canary release patterns.24 Security Context Constraints (SCCs) extend Kubernetes RBAC with fine-grained, container-level restrictions — by default OpenShift refuses to run containers as root, which is exactly the discipline your Spring images should already follow. Source-to-Image (S2I) can build a runnable image straight from your Git repository without a Dockerfile, and the Operator Lifecycle Manager (OLM) plus a curated OperatorHub automate the lifecycle of databases, brokers, and other backing services your estate depends on.24 The platform also bundles a web console, integrated CI/CD Pipelines (built on Tekton), GitOps based on Argo CD, and an internal registry — a turnkey toolchain rather than one you assemble.24

Deployment flavors and the support proposition

OpenShift ships in several forms: a self-managed Container Platform for on-prem, air-gapped, and hybrid use; fully Red Hat-managed offerings; and co-managed cloud services — ROSA (Red Hat OpenShift Service on AWS) integrated into AWS billing and IAM, and ARO (Azure Red Hat OpenShift) with a joint Microsoft support model.24 The enterprise value proposition is concrete: independent analysis named Red Hat a Leader in multicloud container platforms, and its managed services carry SLAs of 99.95% for the public-cloud editions — a level of engineered availability that goes beyond what native cloud Kubernetes services typically commit to.25

When to choose it Reach for OpenShift when you need certified support, strict security defaults, an integrated developer platform, and consistency across on-prem and multiple clouds — common in financial services and other regulated sectors. If you want maximum flexibility, minimal opinion, and the leanest cost, vanilla managed Kubernetes is the better fit. A Spring service deploys to either with the same image; what differs is the platform contract around it.
Section 07 — Delivery

CI/CD: Jenkins versus GitLab, pragmatically

The pipeline is where deployment frequency is won or lost. For Spring estates the two dominant self-hostable choices remain Jenkins and GitLab CI/CD, and they embody genuinely different philosophies.

Jenkins: maximal flexibility, maximal ownership

Jenkins is an open-source automation server, Java-based and runnable anywhere a JRE exists, configured through pipelines written in Groovy and scaled via a controller/agent (historically master/agent) architecture that distributes builds across machines.26 Its defining asset is an ecosystem of well over 1,800 plugins that let it integrate with essentially any tool or environment.27 That flexibility is also its cost: setup is manual, plugins must be maintained, and enforcing consistency across many pipelines demands real platform-engineering discipline.26

Jenkinsfile — declarative pipeline (excerpt)
pipeline {
  agent { kubernetes { yamlFile 'build-pod.yaml' } }
  stages {
    stage('Test')       { steps { sh './mvnw verify' } }
    stage('Image')      { steps { sh './mvnw spring-boot:build-image' } }
    stage('Scan')       { steps { sh 'trivy image $IMAGE' } }
    stage('Deploy')     {
      steps { sh 'kubectl set image deploy/order-service order-service=$IMAGE' }
    }
  }
}

GitLab CI/CD: integrated by design

GitLab CI/CD is built into the GitLab platform, defined by a single .gitlab-ci.yml file, and ships with native runners, an integrated container registry, and embedded security scanning — eliminating the external-tool and plugin sprawl that an equivalent Jenkins setup accumulates.26 In microservices environments, that consolidation pays off: a microservices estate may demand dozens or hundreds of independent pipelines, each building containers, scanning, and deploying to Kubernetes, and the integrated model reduces the per-pipeline overhead. One reported migration from a fragmented Jenkins setup to a unified GitLab CI architecture cut average commit-to-deploy time by 45% — with no application code change, only a change in pipeline architecture.28

.gitlab-ci.yml — Spring service pipeline
stages: [test, build, scan, deploy]

test:
  stage: test
  image: eclipse-temurin:21-jdk
  script: ["./mvnw verify"]

build-image:
  stage: build
  script: ["./mvnw spring-boot:build-image -Dimage=$CI_REGISTRY_IMAGE:$CI_COMMIT_SHORT_SHA"]

deploy:
  stage: deploy
  environment: production
  script:
    - "kubectl set image deploy/order-service order-service=$CI_REGISTRY_IMAGE:$CI_COMMIT_SHORT_SHA"
  rules: [{ if: '$CI_COMMIT_BRANCH == "main"' }]

GitOps: the deployment endgame

Whichever CI tool builds the image, the most robust deployment model decouples delivery from the pipeline using GitOps. A controller such as Argo CD — the engine beneath OpenShift GitOps — continuously reconciles the cluster's live state against declarative manifests in Git.24 The CI pipeline's job shrinks to building, scanning, and updating an image tag in a Git repository; the GitOps controller observes the change and rolls it out. The benefits are auditability (every change is a commit), trivial rollback (revert the commit), and drift correction (the cluster is pulled back to the declared state automatically). This is the architecture that lets elite-tier deployment frequency stay safe at scale.

Section 08 — Landing Zone

AWS, Azure, and Google Cloud

Managed Kubernetes from the three hyperscalers — Amazon EKS, Azure AKS, and Google GKE — is now the standard way to run containerized Spring workloads in production. All three absorb the undifferentiated heavy lifting of operating a Kubernetes control plane: high availability, version updates, and deep integration with each provider's native services.29 The differences that matter to a Spring architect are cost structure, availability commitments, version cadence, and ecosystem fit.

Control plane cost and availability

The first divergence is the control-plane fee. Amazon EKS charges roughly $0.10 per cluster-hour — about $72 per month per cluster — for control-plane management, a cost that compounds with cluster count.30 Azure AKS is notable as the only one of the three offering a free control plane for its standard tier, and GKE does not charge separately for the control plane on standard clusters either.30 On availability, EKS publishes a 99.95% SLA on its standard control plane and 99.99% on a higher-cost provisioned tier.29 For all three, worker-node compute, storage, and egress dominate the real bill far more than the headline control-plane line.31

Picking a landing zone for Spring

The decision rarely turns on Kubernetes itself, which is broadly equivalent across the three, and far more on the surrounding services your Spring estate consumes — managed PostgreSQL, the message broker, the secrets manager, the identity provider, and the observability backend. Azure AKS holds the broadest US federal compliance footprint, including FedRAMP High and multiple DoD impact levels through Azure Government, making it the conservative choice for regulated workloads; GKE consistently runs the most upstream Kubernetes and is typically first to support new features, given Google created Kubernetes and donated it to the CNCF in 2014.29 The portable discipline that protects you regardless of choice is the one this guide has insisted on throughout: an inert image, externalized configuration, declarative manifests, and a GitOps reconciliation loop. With those in place, the cloud underneath becomes a substitutable detail.

Section 09 — The Walkthrough

Deploying the complete application, end to end

Now we assemble everything into a single, reproducible path that takes Aurora from source to a running estate. The sequence below is platform-portable; the only cloud-specific step is provisioning the cluster.

Step 1 — Provision the cluster

Create a managed cluster on your chosen provider. The commands differ only in surface; the resulting Kubernetes API is the same.

# AWS EKS
eksctl create cluster --name aurora --region us-east-1 --nodes 3

# Azure AKS
az aks create -g aurora-rg -n aurora --node-count 3 --generate-ssh-keys

# Google GKE
gcloud container clusters create aurora --num-nodes 3 --region us-east1

Step 2 — Build and push every service image

Use buildpacks so there is no Dockerfile to maintain, and tag images with the commit SHA for traceability. In CI this is a matrix over the service modules.

for svc in config-server discovery-server edge-gateway \
            catalog-service order-service payment-service; do
  (cd $svc && ./mvnw spring-boot:build-image \
     -Dspring-boot.build-image.imageName=registry.example.com/aurora/$svc:$GIT_SHA)
  docker push registry.example.com/aurora/$svc:$GIT_SHA
done

Step 3 — Create namespaces, config, and secrets

Configuration and credentials live in the cluster, never in the image. Secrets should ultimately be sourced from a real secrets manager (AWS Secrets Manager, Azure Key Vault, GCP Secret Manager, or HashiCorp Vault) via an external-secrets operator; the literal form below is for clarity only.

kubectl create namespace aurora

kubectl create configmap order-config -n aurora \
  --from-literal=SPRING_PROFILES_ACTIVE=prod \
  --from-literal=CATALOG_DB_URL=jdbc:postgresql://aurora-pg:5432/catalog

kubectl create secret generic order-secrets -n aurora \
  --from-literal=CATALOG_DB_PASSWORD='***'   # prefer external-secrets in prod

Step 4 — Deploy backing services first

Stand up PostgreSQL and Kafka (via Helm charts or, on OpenShift, via Operators) before the application services, so that readiness probes on the apps succeed on first roll. Per-service databases preserve the data-isolation boundary that makes the services independently deployable.

Step 5 — Deploy platform services, then application services

Apply the config server and discovery server first, wait for them to report ready, then apply the application services and finally the gateway. Order matters because the application services resolve configuration and register on startup.

kubectl apply -n aurora -f k8s/config-server.yaml
kubectl apply -n aurora -f k8s/discovery-server.yaml
kubectl rollout status -n aurora deploy/config-server

kubectl apply -n aurora -f k8s/catalog-service.yaml
kubectl apply -n aurora -f k8s/order-service.yaml
kubectl apply -n aurora -f k8s/payment-service.yaml
kubectl apply -n aurora -f k8s/edge-gateway.yaml

Step 6 — Expose the gateway

On Kubernetes, an Ingress (or the newer Gateway API) routes external traffic to the gateway Service. On OpenShift you would instead create a Route, gaining automatic TLS and built-in support for blue-green and canary strategies without extra controllers.24

apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: aurora-edge
  namespace: aurora
spec:
  rules:
    - host: api.aurora.example.com
      http:
        paths:
          - path: /
            pathType: Prefix
            backend:
              service: { name: edge-gateway, port: { number: 80 } }

Step 7 — Verify health and trace a request

Confirm every service reports ready, then exercise a real path and follow it across services in your tracing backend — the payoff of the Micrometer/OpenTelemetry wiring from Section 03.

kubectl get pods -n aurora
curl https://api.aurora.example.com/actuator/health
curl -X POST https://api.aurora.example.com/api/orders \
  -H 'Content-Type: application/json' \
  -d '{"sku":"AUR-9","qty":2}'
# one traceId now spans gateway -> order -> catalog -> kafka -> payment

Section 10 — Day Two

Operating the estate in production

Deploying the first time is the easy part. The cloud-native discipline is what you do on day two and every day after.

Progressive delivery

A naive rolling update replaces pods in place; progressive delivery makes releases observable and reversible. Blue-green runs the new version alongside the old and flips traffic atomically once health checks pass. Canary shifts a small percentage of traffic to the new version, watches its RED metrics, and either proceeds or rolls back automatically. OpenShift Routes support both patterns natively;24 on vanilla Kubernetes, a progressive-delivery controller such as Argo Rollouts or a service mesh provides the traffic-shaping. Because elite delivery is defined by small, frequent, low-failure changes,7 canary releases of single services are precisely the batch size the DORA data rewards.

Scaling and cost

Horizontal pod autoscaling handles load within a cluster; cluster autoscaling adds and removes nodes beneath it. The newly-GA in-place pod resizing lets you tune CPU and memory of stateful Spring services without a restart, reducing the disruption that vertical scaling used to require.22 On cost, remember that worker-node compute and egress — not the control plane — dominate the bill;31 right-sizing requests, using spot/preemptible nodes for tolerant workloads, and scaling native-image services to zero are the levers that move the number.

Security as a continuous property

Treat the image scan in your pipeline as a gate, not a report. Run as non-root — a default OpenShift will enforce via SCCs.24 Pull secrets from a manager at runtime rather than baking them in. Apply network policies so the order service can reach only the catalog and payment services it actually calls. None of this is Spring-specific, but all of it is the price of running a Spring estate that auditors and incident responders can trust.

Cloud-native is not a destination you deploy to. It is a feedback loop — build, observe, learn, redeploy — that you keep tight enough to outrun your own failures.// day two, every day
Section 11 — Synthesis

Closing the loop

The thread running through every section is a single discipline: collapse the distance between a code change and a safe production deployment, and make the whole journey reproducible. Spring Cloud gives you the distributed-systems patterns — configuration, discovery, resilience, observability — in a portable, well-maintained form aligned to each Spring Boot release.4 Buildpacks and layered jars turn your code into inert, cache-efficient images, and GraalVM native compilation buys order-of-magnitude startup and memory wins where you need them.18 Kubernetes — and OpenShift when you need the enterprise contract around it — runs those images declaratively and self-heals them. Jenkins or GitLab CI/CD, fronting a GitOps controller, automates the path; AWS, Azure, or GCP provides the landing zone.

The payoff is measured, not asserted: the elite tier of the DORA research deploys vastly more often, recovers vastly faster, and fails vastly less, and roughly one organization in five reaches it.67 A Spring estate built along the lines drawn here — small services, immutable images, declarative manifests, automated progressive delivery, and a feedback loop tight enough to trust — is how a team gets there. The cloud-native era did not make Spring obsolete; it gave Spring its most natural home.
