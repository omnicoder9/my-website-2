Abstract

This article examines the security posture, known vulnerabilities, and penetration testing methodology applicable to end-of-life Microsoft Windows operating systems—specifically Windows 95, Windows 98, and Windows XP—that remain embedded in critical infrastructure worldwide. Drawing on public security advisories, government audit findings, and documented attack campaigns, we survey the architectural weaknesses inherent to these platforms, map the exploitation toolchain used by contemporary threat actors, and provide actionable guidance for security assessors working in regulated or air-gapped environments. Special attention is given to ATM networks, medical imaging and infusion systems, and the United States Federal Aviation Administration's legacy automation infrastructure. Defensive countermeasures and compensating controls are reviewed alongside the relevant regulatory and compliance frameworks.

1. Historical Context

To understand why legacy Windows systems remain so deeply entrenched in critical infrastructure, one must appreciate the eras in which they were designed. Windows 95 and XP were not simply older versions of a living product; they were products of entirely different assumptions about connectivity, threat landscape, and operational longevity.

1.1 Windows 9x Architecture and Its Security Model

Released on 24 August 1995, Windows 95 introduced the first mainstream graphical desktop environment for IBM-compatible personal computers. Its immediate commercial success was staggering: Microsoft shipped one million copies in the first four days and seven million in the first five weeks.[1] The architecture, however, was a hybrid that grafted a 32-bit graphical shell atop a 16-bit DOS kernel—a compromise driven by the need for backward compatibility with the installed base of MS-DOS software rather than by any coherent security philosophy.

Windows 9x (encompassing Windows 95, 98, and 98 Second Edition) operated with a flat memory model in which all applications ran in Ring 0—the highest privilege level on x86 architecture. There was no meaningful separation between user-mode and kernel-mode code. A misbehaving or malicious application could write directly to arbitrary memory, halt the system, or corrupt disk structures without any intervention from the operating system. There were no access control lists on the file system if FAT16 or FAT32 was used (NTFS was not available for the 9x series), no concept of least privilege, and no built-in firewall.[2]

Networking support was added via the Win32 WinSock API and NetBIOS over TCP/IP (NBT), but authentication was treated as optional. Default file sharing under Windows 98 required no password. The Remote Registry service, if enabled, was accessible from any network host. These design choices were entirely rational for a 1990s LAN environment where connectivity meant a local Ethernet segment of trusted workstations, not global internet exposure.

Architecture Note

Windows 9x uses a virtual machine manager (VMM32.VXD) to manage hardware access. Because all VxD drivers run in Ring 0, any driver vulnerability grants kernel-level access immediately—there is no privilege escalation step required by an attacker who can reach the driver interface.

1.2 The Windows XP Era: Security as an Afterthought

Windows XP, released on 25 October 2001, represented a genuine architectural leap: it was the first consumer-oriented Windows to be built on the NT kernel, bringing with it a proper hardware abstraction layer (HAL), protected memory, NTFS with access control lists, and a true separation between user and kernel mode.[3] In many respects, XP was the most important Windows release in the product's history, consolidating the consumer and enterprise lines into a single codebase.

Its security record, however, was catastrophic. The default installation of Windows XP SP0 (and SP1) shipped with the following services enabled and exposed on the network: the Server service (for SMB file sharing), the Messenger service, Remote Procedure Call (RPC), Universal Plug and Play (UPnP), and NetBIOS name service. The local Administrator account had a blank password by default. The built-in firewall (Internet Connection Firewall) was present but disabled by default until Service Pack 2 in 2004.[4]

This configuration, deployed on hundreds of millions of machines connected to a rapidly expanding internet, produced one of the most dangerous attack surfaces in computing history. The worms and exploits that followed—Blaster, Sasser, Nimda, Conficker—were not exotic zero-days but straightforward exploitations of enabled services with no authentication.

"A computer running Windows XP with no firewall connected directly to the internet could be compromised within minutes of first boot—before Windows Update could even download a single patch."

— Observed security research consensus, circa 2003–2004, documented in multiple CERT/CC advisories

1.3 End of Life and Its Security Meaning

Microsoft's product lifecycle policy defines "end of life" (EOL) as the date on which the company ceases to release security patches for a product. Windows 95 reached EOL on 31 December 2001. Windows 98 and 98 SE followed on 11 July 2006. Windows XP's EOL was 8 April 2014, after a support period of 12 years—the longest in Microsoft's history, extended twice in response to slow enterprise migration rates.[5]

EOL does not mean the software becomes suddenly more vulnerable on a specific date. What it means is that newly discovered vulnerabilities—and vulnerabilities in these platforms are discovered continuously by security researchers and threat actors—will never receive an official vendor patch. The attack surface is thus permanently growing. Every CVE published that touches a shared code component, every new exploitation technique targeting the Windows kernel, and every supply-chain tool that assumes patching is possible becomes permanently unaddressed on an EOL system.

2. The Persistence Problem: Why These Systems Never Left

The continued operation of Windows XP and, in some cases, Windows 9x in critical infrastructure is not a product of negligence alone. It reflects genuine economic, regulatory, and operational constraints that make replacement far more difficult than patching a developer workstation.

2.1 ATMs and Financial Infrastructure

Automated teller machines (ATMs) are perhaps the most widely cited example of Windows XP persistence. At the time of XP's EOL in April 2014, an estimated 95% of the world's approximately 3.2 million ATMs were running Windows XP.[6] The financial industry's reliance on XP was a product of the ATM hardware replacement cycle: a typical ATM has a physical lifespan of 10–15 years, and the software stack is certified against the hardware platform at the time of deployment. Replacing the OS requires re-certification of the entire device, not merely a software update.

Major ATM manufacturers including NCR, Diebold, and Wincor Nixdorf (now Diebold Nixdorf) had all committed to XP as the standard platform when Windows 7 was the natural successor—but the regulatory certification overhead made upgrades prohibitively slow. The UK's Link ATM network, which operates over 70,000 machines, reported that approximately 75% of its network was still on XP as late as early 2016, two years after EOL.[7]

Several major banks, including JPMorgan Chase and Bank of America in the United States and RBS and Lloyds in the UK, paid Microsoft for custom support agreements—essentially private patch contracts at significant per-device cost—to cover the gap period while migration proceeded.[8] This bespoke support was not available to smaller banks or credit unions, leaving many machines on the open EOL platform.

Financial Infrastructure Reality

Even today, ATMs in developing economies, small credit unions, and rural markets continue to operate Windows XP. Jackpotting attacks—malware that physically dispenses cash from ATM mechanisms—have specifically targeted XP-era vulnerabilities to achieve code execution on the device controller. The US Secret Service issued warnings in 2018 and 2020 regarding active jackpotting campaigns against US ATMs.[9]

2.2 Medical Devices and Healthcare Systems

The healthcare sector's legacy Windows problem is arguably more consequential than the ATM problem because the consequences of compromise are measured not in monetary loss but in patient safety. Medical devices are regulated as Class II or Class III devices under the US Food and Drug Administration (FDA) framework, which means the software embedded in them is part of the regulatory submission. When the FDA clears a device, it clears it with a specific software configuration. Changing the operating system—even via a security patch—may constitute a substantial device modification requiring a new 510(k) or PMA submission.

As of the FDA's 2016 guidance on postmarket cybersecurity management, the agency acknowledged that many medical devices run software that was never designed to be updated, cannot support modern antivirus tools due to processor or memory constraints, and cannot be isolated from hospital networks without disrupting clinical workflows.[10]

The categories of affected equipment are broad. GE Healthcare, Philips, Siemens Healthineers, and similar vendors have publicly acknowledged XP-based systems in their portfolios, including MRI control workstations, CT scanner consoles, PACS (Picture Archiving and Communication System) servers, infusion pump programming stations, and radiation therapy control systems. A 2019 report by CyberMDX found that 56% of connected medical devices in US hospitals ran an end-of-life operating system.[11]

Patient Safety Implication

In 2020, a ransomware attack on Düsseldorf University Hospital in Germany disrupted IT systems sufficiently that an ambulance was rerouted 30km to an alternate facility. A patient died during the delay. While prosecutors ultimately did not attribute criminal liability for the death to the attackers, the incident established that healthcare system compromise has direct patient mortality implications.[12]

2.3 Air Traffic Control and the FAA

The United States Federal Aviation Administration (FAA) operates some of the most complex and safety-critical computing infrastructure on earth—and some of the oldest. The Government Accountability Office (GAO) has repeatedly flagged the FAA's reliance on legacy systems as a significant risk. A 2015 GAO report identified that the FAA operates more than 1,000 information systems, many of which are interconnected and share infrastructure with systems running unsupported commercial off-the-shelf (COTS) operating systems.[13]

The FAA's Standard Terminal Automation Replacement System (STARS), deployed at Terminal Radar Approach Control (TRACON) facilities, was designed in the 1990s and uses a layered architecture with COTS components that have independently aged. The HOST computer system at en route air traffic control centres (ARTCCs) has been in operation since the 1980s and uses IBM 3083 mainframe descendants; however, the workstation and peripheral infrastructure feeding into these systems often runs on Windows XP or Windows Embedded Standard 2009 (which is itself XP-derived).[14]

The FAA's replacement programme, the En Route Automation Modernization (ERAM) project—itself a troubled multi-billion-dollar programme—was intended to replace legacy HOST computers but has faced persistent schedule and budget overruns. A 2023 GAO follow-up found that multiple legacy system replacement programmes at the FAA remained behind schedule and that cybersecurity risk management processes were inconsistently applied across the NAS (National Airspace System).[15]

"The FAA faces a significant challenge in protecting its air traffic control systems from cyber threats due to the presence of legacy systems with known vulnerabilities, inadequate security controls, and insufficient monitoring."

— GAO Report GAO-15-370, April 2015[13]

2.4 Industrial Control Systems and SCADA

Beyond aviation, Windows XP remains embedded in industrial control system (ICS) and SCADA environments across power generation, water treatment, oil and gas pipelines, and manufacturing. Engineering workstations running Wonderware InTouch, GE iFIX, Rockwell FactoryTalk, and similar HMI (human-machine interface) software were certified and deployed on XP and, in some cases, were never tested against newer platforms. Vendors sometimes explicitly prohibit OS upgrades in their support terms, citing risk of HMI instability.

CISA (the Cybersecurity and Infrastructure Security Agency, formerly ICS-CERT) has issued advisories since 2010 noting that a significant fraction of reported ICS vulnerabilities affect software components that run on Windows XP or XP Embedded. The agency's own surveys of critical infrastructure operators found that many could not enumerate how many XP-based systems they operated—let alone provide a remediation timeline.[16]

3. The Attack Surface: Known Vulnerabilities

The vulnerability landscape of Windows XP and Windows 9x is not merely historical record. The relevant CVEs remain exploitable on unpatched production systems today, and many are implemented in mature, maintained exploitation frameworks. What follows is a technical survey of the most significant vulnerability classes.

3.1 SMB Vulnerabilities: The Defining Attack Surface

The Server Message Block (SMB) protocol is the most consequential attack surface in legacy Windows. SMBv1—the version implemented in Windows 9x and XP—is a protocol with no encryption, no message signing by default, and numerous implementation flaws that have been exploited across three decades of attacks.

MS17-010 / CVE-2017-0144

EternalBlue — SMBv1 Remote Code Execution

CVSS 9.3 — Critical

EternalBlue exploits a heap overflow in the Windows SMBv1 srv.sys driver during the handling of a specially crafted Trans2 request. On Windows XP, the lack of ASLR and DEP in meaningful configurations makes reliable exploitation trivial. The exploit was developed by the NSA as part of its TAO (Tailored Access Operations) toolkit and was leaked by the Shadow Brokers group in April 2017. It subsequently formed the propagation mechanism of WannaCry and NotPetya. The vulnerability affects Windows XP through Windows 8—but XP and Vista receive no patch unless a custom support arrangement is in place. Microsoft issued an emergency out-of-band patch for XP (KB4012212) in May 2017 in response to the WannaCry outbreak—a rare exception to EOL policy.[17]

Affected: Windows XP, Vista, 7, 8, Server 2003/2008 Vector: Network / No auth Metasploit module: exploit/windows/smb/ms17_010_eternalblue

MS08-067 / CVE-2008-4250

NetAPI Stack Overflow — Remote Code Execution

CVSS 10.0 — Critical

A stack buffer overflow in the NetpwPathCanonicalize() function of netapi32.dll, reachable via the Server service's RPC interface on TCP port 445. The flaw exists in the canonicalization of file paths passed to the Windows Server service and allows unauthenticated remote code execution as SYSTEM. Conficker exploited this vulnerability in late 2008, ultimately infecting an estimated 9–15 million machines worldwide. On Windows XP SP2 and SP3 without ASLR, exploitation is nearly 100% reliable from a local network. This remains one of the most commonly used modules in legacy Windows assessments.[18]

Affected: Windows 2000, XP, Server 2003 Vector: Network / No auth Metasploit module: exploit/windows/smb/ms08_067_netapi

MS03-026 / CVE-2003-0352

DCOM RPC Buffer Overflow — Blaster Worm

CVSS 10.0 — Critical

A buffer overflow in the DCOM interface of the Windows RPC service, reachable on TCP port 135. The flaw is in the RPCSS.DLL component's handling of malformed DCOM activation requests. The Blaster worm (also known as Lovsan or MSBlast) exploited this vulnerability in August 2003, causing an estimated $320 million in damage and infecting over 500,000 machines. A related vulnerability in the same component (MS03-039) was exploited by the Nachi/Welchia worm. On Windows XP without SP2's firewall, the RPC service is exposed on all network interfaces by default.[19]

Affected: Windows NT 4.0, 2000, XP, Server 2003 Vector: Network / No auth Metasploit module: exploit/windows/dcerpc/ms03_026_dcom

3.2 RPC/DCOM and Named Pipe Vulnerabilities

The Windows RPC subsystem has historically been a rich attack surface. On Windows XP, multiple RPC services bind to dynamically assigned high ports in addition to port 135, and the RPC Endpoint Mapper (portmapper) running on TCP 135 responds to queries that reveal which interfaces are bound to which ports. This makes RPC-based services enumerable and targetable even through partial network segmentation.

Named pipes over SMB provide another RPC transport that is often overlooked. A significant number of Windows XP services expose their RPC interfaces via named pipes, meaning they are reachable through the IPC$ administrative share. If null sessions are permitted (the default on some XP configurations), these pipes are accessible without credentials. The rpcdump utility and its Metasploit equivalent can enumerate all exposed RPC interfaces in seconds.

4. Notable Threat Events

The theoretical vulnerabilities described above have been exercised at scale by real threat actors, producing some of the most consequential cybersecurity incidents in history. Understanding these campaigns is essential context for any security assessor working in legacy environments.

2003
Blaster Worm (MS03-026)

Exploited the DCOM RPC buffer overflow to propagate without user interaction across Windows 2000 and XP machines. At its peak, it infected an estimated 8–16 million machines. The worm's payload triggered a denial-of-service attack against windowsupdate.com—the very mechanism by which victims would obtain a patch. Machines that were patched against MS03-026 were then targeted by the Nachi worm (which exploited the same family of vulnerabilities) in a perverse attempt to "help" clean up Blaster infections. The incident caused power grid disruptions in the United States when SCADA systems running NT/2000 were affected.[21]

2004
Sasser Worm (MS04-011 — LSASS)

Exploited a buffer overflow in the Local Security Authority Subsystem Service (LSASS) to achieve SYSTEM-level code execution without user interaction. Sasser caused disruptions at Delta Air Lines (which cancelled 40 flights), the British coastguard, Goldman Sachs offices, and at least one hospital radiology department in Sweden where XP workstations controlling medical imaging equipment were compromised. The worm's author, a German student aged 18, was arrested, tried, and given a suspended sentence.[22]

2008–2011
Conficker (MS08-067 + WPAD + Autorun)

Conficker combined three propagation vectors: the MS08-067 NetAPI exploit, brute-force of weak administrative credentials over SMB, and AutoRun from removable media. It was sophisticated enough to disable Windows Security Center and Windows Update, block access to antivirus vendor websites, and establish a peer-to-peer command-and-control infrastructure. At peak infection, Conficker had compromised more than 15 million machines, including computers in the UK Ministry of Defence, French navy aircraft carriers (grounding Rafale fighters because flight plans were stored on infected machines), and multiple hospital networks. Conficker's advanced P2P C2 mechanism became a template for subsequent nation-state malware.[23]

2017 — May
WannaCry (EternalBlue + DoublePulsar)

WannaCry used EternalBlue to spread across networks without user interaction, delivering a ransomware payload that encrypted victim files and demanded payment in Bitcoin. The attack infected more than 200,000 machines in 150 countries within 48 hours. The UK National Health Service was among the worst-affected organisations: 80 of 236 NHS trusts in England were affected, with 19,000 appointments cancelled and ambulances diverted. The root cause was XP-based medical devices and legacy Windows 7 machines that had not received MS17-010 patches.[24] North Korea's Lazarus Group was attributed with high confidence by the NSA, NCSC, and Five Eyes partners.[25]

2017 — June
NotPetya (EternalBlue + Mimikatz + WMIC)

NotPetya was initially disguised as ransomware but was in fact a destructive wiper targeting Ukrainian infrastructure. Unlike WannaCry, NotPetya combined EternalBlue with credential harvesting via a modified Mimikatz to move laterally using legitimate Windows management tools (WMIC and PSEXEC), making it effective even against patched machines if any unpatched machine in the domain had been compromised. The malware overwrote the master boot record. Global collateral damage was massive: Maersk (shipping), Merck (pharmaceuticals), FedEx/TNT, Mondelez, and Reckitt Benckiser all suffered hundreds of millions of dollars in losses. Total economic damage has been estimated at $10 billion.[26]

2018–2020
ATM Jackpotting — "Ploutus" and "Tyupkin" Malware

A series of "jackpotting" attacks targeted ATMs running Windows XP in the United States, Europe, and Latin America. Attackers would gain physical or remote access to the ATM's PC component (typically via a technician service port), load ATM-specific malware (Ploutus, Tyupkin, or GreenDispenser), and command the cash dispenser to empty its cassettes. The US Secret Service issued alerts in January 2018 warning that attacks were moving from Latin America to the US market. The attack specifically required the target machine to be running Windows XP, as the malware used XP-specific APIs and process injection techniques that were incompatible with Windows 7 or later.[9]

5. Penetration Testing Legacy Windows Systems

Assessing the security of legacy Windows environments is a common requirement for financial institutions, healthcare organisations, utilities, and government contractors—and one that demands specific methodology adjustments compared to testing modern systems. Legacy systems are often more fragile, more consequential if disrupted, and more likely to produce unexpected results from standard exploitation attempts.

5.1 Legal, Ethical, and Operational Scope

Before any technical work begins, the assessor must establish written authorisation that explicitly covers the target systems, the permitted testing window, the acceptable actions (e.g., can exploit code actually be run, or only to the point of vulnerability identification?), and the notification chain if a system is destabilised. Legacy systems running in OT or medical device environments are frequently classified as safety-critical; unauthorised disruption carries legal liability under multiple frameworks including CFAA (US), Computer Misuse Act (UK), and sector-specific regulations.

Critical Warning

Do not run denial-of-service checks, aggressive scanning profiles, or live exploitation against systems described as safety-critical without explicit written approval from both the system owner and, where applicable, the responsible clinical or operational engineer. The fragility of legacy Windows systems means that a single malformed packet can trigger a BSOD—and a crashed ATM or medical device control station has real-world consequences. Always maintain a kill switch communication with the operational team.

5.2 Reconnaissance

Reconnaissance against legacy Windows environments requires distinguishing between open-source intelligence (OSINT) for internet-facing assets and passive internal enumeration for air-gapped or segmented OT networks.

For internet-facing systems, Shodan's proprietary fingerprinting database remains an invaluable first step. Shodan indexes banner responses, OS fingerprints, and SSL certificate metadata from billions of internet-connected devices. Queries targeting Windows XP machines can be constructed using the Shodan search API:

# Shodan CLI — find internet-exposed Windows XP systems
shodan search "os:Windows XP" --fields ip_str,port,org,hostnames

# More targeted: XP machines with SMB open
shodan search "os:Windows XP port:445"

# XP machines exposing RDP (common in legacy remote access)
shodan search "os:Windows XP port:3389"

For passive internal discovery in OT networks where active scanning is restricted, network traffic analysis using tools like Zeek (formerly Bro) or Wireshark can identify Windows XP hosts via their distinctive SMB negotiation sequences: XP defaults to SMBv1 dialect negotiation and advertises the "Windows XP" OS string in the Session Setup AndX Response packet.

# Wireshark filter for SMBv1 session setups containing "Windows XP"
smb.os_level == 0x0105 and smb.native_os contains "Windows XP"

# Zeek script to log SMB OS strings
event smb1_session_setup_andx_response(c: connection, hdr: SMB1::Header,
    request: SMB1::SessionSetupAndXRequest, response: SMB1::SessionSetupAndXResponse)
{
    if /Windows XP/ in response$native_os {
        print fmt("XP host detected: %s", c$id$resp_h);
    }
}

5.3 Scanning and Enumeration

Nmap remains the standard tool for host discovery and service enumeration. For legacy Windows environments, specific scan profiles are warranted to balance thoroughness against the risk of crashing fragile targets.

OS Fingerprinting: Nmap's OS detection relies on TCP/IP stack behaviour that is highly distinctive for Windows XP. XP's TCP initial window size (65535 on some configurations), IPID behaviour, and TCP options set (absent SACK and TS in early service packs) produce reliable fingerprints:

# Safe Nmap profile for legacy Windows (slower, less likely to crash targets)
nmap -sS -sV -O --osscan-guess -T2 --max-retries 1 \
     --script smb-os-discovery,smb-security-mode \
     --open -p 21,22,23,25,53,80,110,135,139,443,445,3389,8080 \
     192.168.1.0/24

# SMB-specific enumeration (very useful for XP targets)
nmap -p 445 --script smb-vuln-ms17-010,smb-vuln-ms08-067 \
     --script-args unsafe=1 192.168.1.100

Stability Warning

BlueKeep exploitation involves kernel pool grooming techniques that are sensitive to the exact memory state of the target machine at the time of the attack. False negatives and target crashes are common, especially on Windows XP where pool layout differs from Windows 7. Do not run BlueKeep exploitation against clinical devices or ATMs in production without explicit approval and a confirmed recovery procedure. The MS08-067 or MS17-010 modules are significantly more stable for XP targets.

WDigest on XP

Windows XP stores plaintext credentials in LSASS memory via the WDigest authentication provider by default—a behaviour that was not disabled until Windows 8.1 / Server 2012 R2. On XP targets, Mimikatz's sekurlsa::wdigest will almost always return cleartext passwords for any interactively logged-on user, making lateral movement trivial.

5.5 Post-Exploitation and Lateral Movement

Post-exploitation on XP focuses on persistence, credential harvesting, and documenting the blast radius—the set of systems reachable from the compromised host using harvested credentials and network access.

# Pivoting through an XP machine into a segmented OT network
meterpreter > run post/multi/manage/autoroute SUBNET=10.0.50.0 NETMASK=255.255.255.0

# Scan the OT subnet from behind the pivot
msf6 > use auxiliary/scanner/portscan/tcp
msf6 auxiliary(tcp) > set RHOSTS 10.0.50.0/24
msf6 auxiliary(tcp) > set PORTS 102,502,20000,44818   # Modbus, S7, DNP3, EtherNet/IP

# Document persistence options on XP (for report, NOT deployment in prod)
meterpreter > run post/windows/manage/persistence_exe STARTUP=SCHEDULER
meterpreter > run post/windows/gather/smart_hashdump   # SAM + domain hashes
meterpreter > run post/windows/gather/credentials/credential_collector

# Screenshot for evidence (report documentation)
meterpreter > screenshot
meterpreter > run post/multi/gather/screen_spy

For assessors working in regulated OT environments, the post-exploitation phase should be limited to documentation—screenshot, hash dump for crack-rate evidence, and route mapping—without deploying persistence mechanisms. The goal is to demonstrate blast radius, not to establish a genuine foothold that could be later exploited by a real adversary or introduce system instability.

6. Defensive Strategies for Legacy Windows Environments

The security of legacy Windows systems cannot be achieved through patching alone—by definition, patches are unavailable. Defence must be achieved through compensating controls: architectural choices and configurations that reduce the exploitable attack surface without requiring the operating system to be modified.

6.1 Network Segmentation and Micro-Segmentation

The most effective single control for legacy Windows is network isolation. A Windows XP machine that cannot receive inbound SMB traffic cannot be exploited via EternalBlue or MS08-067, regardless of patch status. NIST Special Publication 800-82 (Guide to Industrial Control System Security) provides a tiered segmentation model suitable for environments where legacy systems coexist with modern IT networks: legacy systems should be placed in a dedicated DMZ-equivalent zone, with firewall rules that permit only the specific protocols and port pairs required for their operational function.[27]

At minimum, the following ports should be blocked inbound to any Windows XP or 9x host from all non-authorised networks: TCP 135–139, 445 (SMB/NetBIOS), 3389 (RDP), and all UDP 137–138 (NetBIOS). Outbound connections from legacy hosts should be explicitly enumerated and all others denied.

6.2 Virtual Patching via Intrusion Prevention Systems

An IPS (Intrusion Prevention System) deployed at the network ingress to a legacy Windows segment can enforce virtual patches for known signatures—detecting and dropping exploit traffic before it reaches the vulnerable host. Major IPS vendors including Trend Micro, Palo Alto Networks, and Fortinet maintain signature databases for EternalBlue, BlueKeep, and MS08-067 that are effective against the published exploitation traffic patterns.

Virtual patching has significant limitations: it operates on network traffic signatures and can be evaded by modified payloads, tunnelled traffic, or insider threats that originate within the same network segment. It is a compensating control, not a substitute for remediation. CISA explicitly categorises virtual patching as a temporary measure.[28]

6.3 Application Whitelisting

On Windows XP, application control can be implemented via Software Restriction Policies (SRP) under Group Policy. While SRP lacks the sophistication of modern AppLocker or Windows Defender Application Control, it can prevent the execution of arbitrary executables—including Meterpreter payloads—when configured correctly. An SRP that permits only signed, path-restricted executables from known-good directories significantly raises the bar for post-exploitation payload execution.

For ATM environments, major ATM vendors provide their own application control layers (NCR's POS Protector, Wincor's TPM-based application control) that operate at the device-specific software layer and are compatible with XP's architecture.

6.4 Disabling High-Risk Services

Many of the vulnerabilities discussed in this article affect services that are enabled by default but not operationally required for most legacy deployments. The following services should be disabled on Windows XP hosts that do not require them: Server service (if file sharing is not required), Messenger service, Remote Registry, UPnP Host, SSRP (SQL Server Resolution Protocol if no SQL Server is installed), and Telnet. Disabling the Server service entirely removes the SMBv1 attack surface from the network perspective.

# Disable SMBv1 (where Group Policy is available)
# Registry method for XP:
reg add "HKLM\SYSTEM\CurrentControlSet\Services\LanmanServer\Parameters" ^
    /v SMB1 /t REG_DWORD /d 0 /f

# Disable unnecessary services via sc command
sc config messenger start= disabled
sc config RemoteRegistry start= disabled
sc config SSDPSRV start= disabled   # UPnP discovery
sc config upnphost start= disabled  # UPnP hosting
sc stop messenger
sc stop RemoteRegistry

6.5 Host-Based Monitoring and Logging

Legacy Windows systems often have limited or no centralised logging. Windows XP's Event Log can be configured to forward events to a syslog server using a lightweight agent (Snare for Windows, NXLog) that does not require modern OS features. The minimum log set for a legacy Windows host should include: process creation events (Security Event 592 on XP), logon/logoff events (Security Events 528, 538, 552), and network share access events. These are the minimum necessary to detect post-exploitation activity.

6.6 Migration Planning

All compensating controls are temporary. The only long-term solution to the legacy Windows problem is managed migration to a supported platform. For ATMs, this means a structured refresh programme coordinated with the device manufacturer's certification timeline. For medical devices, it requires engagement with the FDA's current guidance on pre-market and post-market cybersecurity submissions, including the 2023 Omnibus cybersecurity requirements for new device submissions.[29]

The FAA's NextGen programme and its specific modernisation tracks for STARS and ERAM represent the equivalent infrastructure replacement path for air traffic control. Congress appropriated $206 million specifically for FAA cybersecurity improvements in the FY2023 omnibus appropriations bill, partly in response to GAO findings about legacy system risk.[30]

7. Regulatory and Compliance Landscape

Multiple regulatory frameworks address legacy Windows systems explicitly, though enforcement mechanisms vary significantly by sector.

7.1 Healthcare: FDA and HIPAA

The FDA's 2016 guidance on postmarket cybersecurity management for medical devices establishes that manufacturers are responsible for identifying and communicating cybersecurity risks in their products throughout the product lifecycle, including risks arising from EOL operating systems.[10] The 2023 Consolidated Appropriations Act (Omnibus) added Section 524B to the FD&C Act, requiring that medical devices submitted for FDA clearance after March 2023 include a Software Bill of Materials (SBOM) and demonstrate a process for regular security updates.[29]

HIPAA's Security Rule (45 CFR §164.312) requires covered entities to implement technical safeguards that protect electronic protected health information (ePHI). While HIPAA does not prescribe specific controls, HHS has issued guidance that operating systems beyond their EOL date without compensating controls constitute a failure to implement reasonable and appropriate technical safeguards. HHS Office for Civil Rights has cited legacy operating system use in enforcement actions.[31]

7.2 Financial Services: PCI-DSS and FFIEC

PCI DSS Requirement 6.3.3 (v4.0) requires that all system components are protected from known vulnerabilities by installing applicable security patches. The PCI Security Standards Council's guidance explicitly states that running an operating system past its vendor end-of-life date creates a non-compliant state unless compensating controls are documented and accepted by the acquirer.[32] ATM operators are subject to PCI DSS as payment devices; the persistence of XP in ATM networks has been a recurring finding in PCI assessments.

7.3 Critical Infrastructure: CISA and NERC CIP

CISA's Known Exploited Vulnerabilities (KEV) catalogue—a binding operational directive for federal agencies—includes both MS17-010 (EternalBlue) and CVE-2019-0708 (BlueKeep), requiring federal agencies to remediate these vulnerabilities on a mandatory timeline. For private sector critical infrastructure, the KEV is advisory, but CISA has issued emergency directives to private sector OT operators when exploitation has been confirmed in their sectors.[33]

NERC CIP-007-6 (Systems Security Management) requires electric utility operators to manage ports and services, apply patches within defined timelines, and document cases where patching is not feasible. For unpatched legacy Windows systems, NERC CIP requires a compensating mitigation plan with documented risk acceptance—a framework that other critical infrastructure sectors lack equivalent mandates for.

8. Conclusion

The continued operation of Windows XP and Windows 9x in critical infrastructure is not a historical accident that will resolve itself. It is a structural property of industries where regulatory certification, long asset lifecycles, and the complexity of safety-critical software interactions make replacement far slower than the evolution of the threat landscape. An ATM, a ventilator control system, or a TRACON workstation that was designed and certified in 2003 will not simply become obsolete because Microsoft stopped releasing patches in 2014.

From an offensive security perspective, legacy Windows represents a target class with an extraordinarily high ratio of reliable exploits to mitigating controls. The availability of mature, maintained exploit modules in Metasploit for MS08-067, MS17-010, and BlueKeep—all of which require no authentication and produce SYSTEM-level access—means that network access to an uncompensated legacy Windows host is functionally equivalent to administrative access. The Conficker, WannaCry, and NotPetya campaigns demonstrated at global scale what is achievable with this access when credential harvesting and lateral movement are layered on top of initial exploitation.

The defensive posture for organisations that operate these systems must acknowledge this asymmetry. Compensating controls—network isolation, virtual patching, application whitelisting, and aggressive logging—can substantially reduce the risk, but they require active maintenance and skilled operational security oversight. The systems themselves are permanently in a state of accumulating technical debt, and each passing year adds new vulnerabilities in shared code components for which no patch will ever exist.

The long-term answer is managed migration, and the evidence suggests this is happening—slowly, unevenly, and against the backdrop of an adversary community that is actively and successfully exploiting the gap. Until migration is complete, legacy Windows security is not a problem to be solved but a risk to be managed with diligence, honesty, and the understanding that the margin for error is considerably smaller than on any supported platform.
