Section 01

Django's Architecture Philosophy

Django emerged in 2005 from a Kansas newspaper's content-management needs and deliberately chose pragmatism over purity.[5] Its batteries-included ethos — ORM, admin, auth, forms, templating bundled together — accelerates initial development but creates architectural gravity: features accumulate in models, views sprawl into utilities, and business logic disperses everywhere.[6]

The framework's official documentation describes an MTV (Model–Template–View) structure, a variant of MVC where Django's View acts as a controller and Template handles presentation.[7] This is a perfectly adequate structure for CRUD applications, but at scale — teams of 10+, hundreds of models, complex business rules — it becomes insufficient. The architecture described in this guide layers well-established software engineering patterns atop Django's foundation.

Django gives you a head start. Architecture gives you somewhere to go.

— Adapted from Two Scoops of Django[8]

The guiding principle throughout is the Dependency Inversion Principle: high-level business rules should not depend on low-level details like ORM specifics, HTTP request objects, or third-party SDKs.[9] Each pattern in this guide enforces this boundary in its own way.

Section 02

Service Layer Patterns

The most impactful single change you can make to a Django codebase is extracting business logic into a dedicated service layer. This section traces the path from the "fat model" anti-pattern through disciplined service objects and the repository pattern.

The Fat Model Problem

The Django community long advocated "fat models, thin views" — placing business logic in model methods rather than views. While better than fat views, fat models couple domain logic to persistence, making unit testing expensive (database hits required) and reuse impossible across serializers, management commands, and API endpoints.[10]

Python
models.py — Anti-pattern: fat model
# ❌ Anti-pattern: business logic bleeding into the model
class Order(models.Model):
    user = models.ForeignKey(User, on_delete=models.CASCADE)
    status = models.CharField(max_length=20, default='pending')
    total = models.DecimalField(max_digits=10, decimal_places=2)
    created_at = models.DateTimeField(auto_now_add=True)

    def confirm(self):
        # Validation, payment, email, inventory — all tangled here
        if self.status != 'pending':
            raise ValueError("Order already processed")
        charge_credit_card(self.user, self.total)   # side-effect!
        send_confirmation_email(self.user)           # side-effect!
        update_inventory(self)                       # side-effect!
        self.status = 'confirmed'
        self.save()  # Always hits DB — untestable without it

Service Objects

A service object (sometimes called a service function or use-case handler) is a plain Python class or function that orchestrates domain logic without inheriting from any Django base class.[11] Services receive their dependencies as arguments — enabling dependency injection and straightforward mocking.

Python
services/order_service.py
from dataclasses import dataclass
from decimal import Decimal
from typing import Protocol
from myapp.models import Order, User

# ── Ports (interfaces) the service depends on ──────────────────
class PaymentGateway(Protocol):
    def charge(self, user: User, amount: Decimal) -> str: ...

class Notifier(Protocol):
    def send_order_confirmation(self, user: User, order_id: int) -> None: ...

class InventoryService(Protocol):
    def reserve(self, order: Order) -> None: ...

# ── Input / Output DTOs ─────────────────────────────────────────
@dataclass
class ConfirmOrderInput:
    order_id: int
    requesting_user_id: int

@dataclass
class ConfirmOrderResult:
    order_id: int
    transaction_id: str
    status: str

# ── Service ─────────────────────────────────────────────────────
class OrderConfirmationService:
    def __init__(
        self,
        payment: PaymentGateway,
        notifier: Notifier,
        inventory: InventoryService,
    ):
        self._payment = payment
        self._notifier = notifier
        self._inventory = inventory

    def confirm(self, cmd: ConfirmOrderInput) -> ConfirmOrderResult:
        order = Order.objects.select_for_update().get(
            pk=cmd.order_id, user_id=cmd.requesting_user_id
        )
        if order.status != 'pending':
            raise ValueError(f"Order {order.pk} already in state {order.status}")

        txn_id = self._payment.charge(order.user, order.total)
        self._inventory.reserve(order)

        order.status = 'confirmed'
        order.save(update_fields=['status'])

        self._notifier.send_order_confirmation(order.user, order.pk)

        return ConfirmOrderResult(
            order_id=order.pk,
            transaction_id=txn_id,
            status='confirmed',
        )
✓ Testing Advantage
Every dependency is injected. In tests, swap PaymentGateway with a mock; no database required for validating the branching logic. The entire confirm flow can be exercised in milliseconds.

Repository Pattern

The repository pattern abstracts data access behind an interface, decoupling domain logic from ORM specifics.[12] In Django this is often considered redundant — the ORM already provides an abstraction. But there are scenarios where it adds genuine value: swapping data sources (PostgreSQL to Elasticsearch), testing without a database, or enforcing aggregate boundaries in DDD contexts.

Python
repositories/order_repository.py
from abc import ABC, abstractmethod
from typing import Optional
from myapp.models import Order

class AbstractOrderRepository(ABC):
    @abstractmethod
    def get_by_id(self, order_id: int) -> Optional[Order]: ...

    @abstractmethod
    def get_pending_for_user(self, user_id: int) -> list[Order]: ...

    @abstractmethod
    def save(self, order: Order) -> Order: ...


class DjangoOrderRepository(AbstractOrderRepository):
    def get_by_id(self, order_id: int) -> Optional[Order]:
        try:
            return Order.objects.select_related('user').get(pk=order_id)
        except Order.DoesNotExist:
            return None

    def get_pending_for_user(self, user_id: int) -> list[Order]:
        return list(
            Order.objects.filter(user_id=user_id, status='pending')
                         .order_by('-created_at')
        )

    def save(self, order: Order) -> Order:
        order.save()
        return order


# In-memory implementation for tests ─────────────────────────────
class InMemoryOrderRepository(AbstractOrderRepository):
    def __init__(self):
        self._store: dict[int, Order] = {}
        self._counter = 1

    def get_by_id(self, order_id: int) -> Optional[Order]:
        return self._store.get(order_id)

    def get_pending_for_user(self, user_id: int) -> list[Order]:
        return [o for o in self._store.values()
                if o.user_id == user_id and o.status == 'pending']

    def save(self, order: Order) -> Order:
        if not order.pk:
            order.pk = self._counter; self._counter += 1
        self._store[order.pk] = order
        return order

Clean Architecture in Django

Robert C. Martin's Clean Architecture proposes concentric rings: innermost is enterprise business rules (Entities), then application business rules (Use Cases), then interface adapters (Controllers/Presenters), and outermost is frameworks and drivers.[9] The fundamental rule: source code dependencies must point inward only. 

  ┌──────────────────────────────────────────────────────┐
  │            FRAMEWORKS & DRIVERS (outermost)          │
  │   Django, DRF, PostgreSQL, Redis, Celery, S3, etc.   │
  │  ┌─────────────────────────────────────────────────┐ │
  │  │          INTERFACE ADAPTERS                     │ │
  │  │   Views, Serializers, Management Commands,      │ │
  │  │   Admin, Celery Tasks, GraphQL resolvers         │ │
  │  │  ┌────────────────────────────────────────────┐ │ │

  │  │  │   Use Cases / Interactors / Services       │ │ │
  │  │  │  ┌──────────────────────────────────────┐  │ │ │
  │  │  │  │    ENTERPRISE BUSINESS RULES         │  │ │ │
  │  │  │  │   Domain Entities, Value Objects,    │  │ │ │
  │  │  │  │   Domain Events, Business Rules      │  │ │ │

Use Case Objects

Use cases represent a single application capability. They depend on abstract ports (interfaces) not on concrete implementations, keeping the inner layers isolated from framework churn.[13]

Python
application/use_cases/register_user.py
from dataclasses import dataclass
from typing import Protocol
import re

# ── Domain Entity (innermost layer — pure Python) ───────────────
@dataclass
class UserRegistrationRequest:
    email: str
    password: str
    display_name: str

    def __post_init__(self):
        if not re.match(r'.+@.+\..+', self.email):
            raise ValueError("Invalid email address")
        if len(self.password) < 10:
            raise ValueError("Password must be at least 10 characters")

# ── Ports ───────────────────────────────────────────────────────
class UserRepository(Protocol):
    def exists_by_email(self, email: str) -> bool: ...
    def create(self, req: UserRegistrationRequest) -> int: ...  # returns user_id

class WelcomeMailer(Protocol):
    def send(self, email: str, name: str) -> None: ...

# ── Use Case ────────────────────────────────────────────────────
class RegisterUserUseCase:
    def __init__(self, users: UserRepository, mailer: WelcomeMailer):
        self._users = users
        self._mailer = mailer

    def execute(self, request: UserRegistrationRequest) -> int:
        if self._users.exists_by_email(request.email):
            raise ValueError(f"Email {request.email!r} is already registered")
        user_id = self._users.create(request)
        self._mailer.send(request.email, request.display_name)
        return user_id

# ── Interface Adapter: DRF View ─────────────────────────────────
from rest_framework.views import APIView
from rest_framework.response import Response
from rest_framework import status

class RegisterView(APIView):
    def __init__(self, use_case: RegisterUserUseCase, **kwargs):
        super().__init__(**kwargs)
        self._use_case = use_case   # injected from DI container

    def post(self, request):
        try:
            req = UserRegistrationRequest(
                email=request.data['email'],
                password=request.data['password'],
                display_name=request.data['display_name'],
            )
            user_id = self._use_case.execute(req)
            return Response({'user_id': user_id}, status=status.HTTP_201_CREATED)
        except (KeyError, ValueError) as e:
            return Response({'error': str(e)}, status=status.HTTP_400_BAD_REQUEST)
💡 Dependency Injection in Django
Django has no built-in DI container. Popular options include python-dependency-injector, lagom, or simple factory functions in apps.py. For most teams, a factory function per use-case in di.py is the pragmatic choice before reaching for a full container.[14]
Section 04

CQRS — Command Query Responsibility Segregation

CQRS, popularised by Greg Young and Udi Dahan, separates operations that change state (Commands) from operations that return state (Queries).[15] In a Django context this is a spectrum: from a lightweight convention (separate modules for reads and writes) to full physical separation with distinct read and write databases.

A command changes state and returns nothing. A query returns state and changes nothing.

— Bertrand Meyer's Command-Query Separation principle[16]

The Command Side

Commands are intent-carrying data objects: "ConfirmOrder with ID 42 for user 7." They go through the full service/use-case stack: validation, business rules, database writes, events emitted. Because they're write-heavy and concern-isolated, they're easily distributed to background workers.

Python
cqrs/commands.py + command_bus.py
from dataclasses import dataclass
from typing import Any, Callable, Type

# ── Command definitions ─────────────────────────────────────────
@dataclass(frozen=True)
class ConfirmOrderCommand:
    order_id: int
    user_id: int

@dataclass(frozen=True)
class CancelOrderCommand:
    order_id: int
    user_id: int
    reason: str

# ── Simple in-process command bus ──────────────────────────────
class CommandBus:
    def __init__(self):
        self._handlers: dict[Type, Callable] = {}

    def register(self, command_type: Type, handler: Callable) -> None:
        self._handlers[command_type] = handler

    def dispatch(self, command: Any) -> Any:
        handler = self._handlers.get(type(command))
        if handler is None:
            raise LookupError(f"No handler for {type(command).__name__}")
        return handler(command)

# ── Wiring (in apps.py or di.py) ────────────────────────────────
def build_command_bus() -> CommandBus:
    bus = CommandBus()
    bus.register(
        ConfirmOrderCommand,
        OrderConfirmationService(
            payment=StripeGateway(),
            notifier=SendGridNotifier(),
            inventory=DjangoInventoryService(),
        ).confirm_from_command,
    )
    bus.register(CancelOrderCommand, CancelOrderService().cancel)
    return bus

The Query Layer

Queries bypass the service/domain layers entirely and speak directly to the read store — a database, cache, or search index — returning view-model DTOs shaped for the consumer. This avoids the N+1 ORM overhead that plagues generic serializers.[17]

Python
cqrs/queries/order_queries.py
from dataclasses import dataclass
from datetime import datetime
from django.db import connection

# ── Read-model DTOs ─────────────────────────────────────────────
@dataclass
class OrderSummaryView:
    order_id: int
    user_display_name: str
    total: str
    status: str
    item_count: int
    created_at: datetime

# ── Query handler — raw SQL for maximum read performance ────────
class OrderQueryService:

    def list_user_orders(
        self, user_id: int, page: int = 1, page_size: int = 20
    ) -> list[OrderSummaryView]:
        offset = (page - 1) * page_size
        sql = """
            SELECT
                o.id           AS order_id,
                u.display_name AS user_display_name,
                o.total::text  AS total,
                o.status,
                COUNT(oi.id)   AS item_count,
                o.created_at
            FROM orders_order o
            JOIN auth_user u    ON u.id = o.user_id
            LEFT JOIN orders_orderitem oi ON oi.order_id = o.id
            WHERE o.user_id = %s
            GROUP BY o.id, u.display_name
            ORDER BY o.created_at DESC
            LIMIT %s OFFSET %s
        """
        with connection.cursor() as cursor:
            cursor.execute(sql, [user_id, page_size, offset])
            rows = cursor.fetchall()
        return [
            OrderSummaryView(
                order_id=r[0], user_display_name=r[1], total=r[2],
                status=r[3], item_count=r[4], created_at=r[5]
            ) for r in rows
        ]

Read-Model Projections

In advanced CQRS, write and read models use separate storage. Writes go to a normalised relational database; a projection process (triggered by events) materialises denormalised read models into Redis, Elasticsearch, or a dedicated read database.[18] This enables sub-millisecond reads with no joins.

Write Side
  • Normalised PostgreSQL tables
  • Strong consistency, transactions
  • Rich domain validation
  • Through service layer
  • Emits domain events
Read Side
  • Denormalised Redis hashes / ES docs
  • Eventual consistency
  • No domain logic
  • Direct query to store
  • Rebuilt from event log
Section 05

Domain-Driven Design with Django

Eric Evans' Domain-Driven Design provides a vocabulary and set of patterns for modelling complex business domains with fidelity.[19] In Django, DDD is not about abandoning the ORM — it's about organising code around domain concepts rather than technical ones.

Entities and Value Objects

An Entity has a unique identity that persists over time (an Order is still that order even after its status changes). A Value Object has no identity; it is defined entirely by its attributes (a Money amount, a postal address).[19]

Python
domain/value_objects.py + domain/entities.py
from dataclasses import dataclass
from decimal import Decimal
from enum import Enum

# ── Value Objects ────────────────────────────────────────────────
@dataclass(frozen=True)     # frozen → immutable → safe to share
class Money:
    amount: Decimal
    currency: str             # ISO-4217: "USD", "EUR", etc.

    def __post_init__(self):
        if self.amount < 0:
            raise ValueError("Money cannot be negative")
        if len(self.currency) != 3:
            raise ValueError("Currency must be a 3-letter ISO code")

    def __add__(self, other: 'Money') -> 'Money':
        if self.currency != other.currency:
            raise ValueError("Cannot add different currencies")
        return Money(self.amount + other.amount, self.currency)

@dataclass(frozen=True)
class Address:
    street: str
    city: str
    postcode: str
    country_code: str         # ISO-3166-1 alpha-2

# ── Domain Entity ────────────────────────────────────────────────
class OrderStatus(Enum):
    PENDING   = 'pending'
    CONFIRMED = 'confirmed'
    SHIPPED   = 'shipped'
    CANCELLED = 'cancelled'

class OrderLine:
    def __init__(self, product_id: int, quantity: int, unit_price: Money):
        if quantity <= 0:
            raise ValueError("Quantity must be positive")
        self.product_id = product_id
        self.quantity = quantity
        self.unit_price = unit_price

    @property
    def subtotal(self) -> Money:
        return Money(self.unit_price.amount * self.quantity, self.unit_price.currency)

class Order:                  # Domain entity — pure Python, no Django
    def __init__(self, order_id: int, user_id: int):
        self.order_id = order_id
        self.user_id = user_id
        self.status = OrderStatus.PENDING
        self._lines: list[OrderLine] = []
        self._events: list = []

    def add_line(self, line: OrderLine) -> None:
        if self.status != OrderStatus.PENDING:
            raise ValueError("Cannot modify a non-pending order")
        self._lines.append(line)

    @property
    def total(self) -> Money:
        if not self._lines:
            raise ValueError("Order has no lines")
        result = self._lines[0].subtotal
        for line in self._lines[1:]:
            result = result + line.subtotal
        return result

    def confirm(self) -> 'OrderConfirmed':
        if self.status != OrderStatus.PENDING:
            raise ValueError(f"Cannot confirm order in state {self.status}")
        self.status = OrderStatus.CONFIRMED
        event = OrderConfirmed(order_id=self.order_id, total=self.total)
        self._events.append(event)
        return event

    def pull_events(self) -> list:
        events, self._events = self._events, []
        return events

Aggregates and Aggregate Roots

An aggregate is a cluster of domain objects that change together, accessed through a single Aggregate Root.[19] External code may only hold a reference to the root — never to interior objects directly. In Django, this maps to a model that "owns" related models and ensures invariants across them.

⚠ Django ORM Caveat
Django's ORM makes it easy to directly access related objects across aggregate boundaries (order.lines.all()). Enforce aggregate boundaries through code review, linting rules (pylint-django), or architectural tests that fail if inner models are accessed from outside their aggregate module.[20]

Bounded Contexts

A bounded context is an explicit boundary within which a domain model applies. The word Order may mean different things in the Payments context (a charge record) vs the Fulfilment context (a shipment task) vs the Analytics context (a revenue event). Each context has its own models, ubiquitous language, and service layer.[21]

Section 06

Microservices vs. The Modular Monolith

The microservices hype of the 2010s convinced many teams to distribute their applications prematurely. Martin Fowler's MonolithFirst pattern argues that starting with a well-structured monolith is almost always the right choice, extracting services when you understand the domain well enough to draw stable boundaries.[22]

Dimension Modular Monolith Microservices
Deployment Single artefact, simple CI/CD Per-service pipeline, registry, K8s complexity
Development speed Fast — in-process calls, shared ORM Slow — network calls, versioning, contracts
Data consistency ACID transactions across modules Eventual consistency, sagas required
Independent scaling Scale whole app; less efficient at extremes Scale hot services independently
Team autonomy Shared codebase needs governance Teams own services end-to-end
Observability Single log stream, easier tracing Distributed tracing essential (Jaeger, Tempo)
Right for Teams <50, unclear domain boundaries Large orgs, distinct SLAs per service

The Modular Monolith in Django

A modular monolith enforces bounded-context boundaries through module rules rather than network boundaries. Cross-context communication happens through well-defined interfaces (service classes or domain events), not direct model imports.[23]

Python
orders/application/interfaces.py — Anti-corruption layer
# The Orders context declares what it NEEDS from Catalogue,
# expressed in Orders' own language — not Catalogue's models.
from typing import Protocol
from dataclasses import dataclass

@dataclass(frozen=True)
class ProductDetails:   # Orders' view of a Product
    product_id: int
    name: str
    unit_price_cents: int
    currency: str
    in_stock: bool

class ProductLookup(Protocol):
    def get_product(self, product_id: int) -> ProductDetails: ...

# ── Catalogue adapter implements the Orders interface ────────────
# (lives in catalogue app, depends on catalogue models)
from catalogue.models import Product as CatalogueProduct

class CatalogueProductLookup:            # Implements ProductLookup
    def get_product(self, product_id: int) -> ProductDetails:
        p = CatalogueProduct.objects.get(pk=product_id)
        return ProductDetails(
            product_id=p.pk,
            name=p.name,
            unit_price_cents=int(p.price * 100),
            currency=p.currency,
            in_stock=p.stock_count > 0,
        )

Signals for Extraction

When a bounded context grows large enough to warrant extraction into a microservice, the transition is smoother if it was already communicating through events. You replace the in-process event handler with a message queue consumer — the producer remains unchanged.[24]

Section 07

Event-Driven Architecture

Event-driven systems communicate through facts that have occurred — OrderConfirmed, UserRegistered, PaymentFailed — rather than direct calls. Producers are decoupled from consumers: they emit an event and don't care who (if anyone) listens. This enables extensibility, asynchrony, and eventual consistency.[25]

Django Signals: The In-Process Event Bus

Django's built-in signals (post_save, pre_delete, custom signals) provide an in-process publish-subscribe mechanism. They're synchronous and coupled to the same transaction — useful for cross-cutting concerns (audit logs, cache invalidation), but not suitable for durable cross-service communication.[26]

Python
orders/signals.py — Domain events via Django signals
from django.dispatch import Signal, receiver
from django.db.models.signals import post_save
from orders.models import Order

# Custom domain event signals
order_confirmed  = Signal()   # kwargs: order_id, user_id, total
order_cancelled  = Signal()   # kwargs: order_id, reason

# Emit domain events from service layer (not from model.save())
class OrderConfirmationService:
    def confirm(self, order: Order) -> None:
        # ... business logic ...
        order.status = 'confirmed'
        order.save()
        order_confirmed.send(
            sender=Order,
            order_id=order.pk,
            user_id=order.user_id,
            total=order.total,
        )

# Handlers — decoupled listeners
@receiver(order_confirmed)
def handle_order_confirmed_audit(sender, order_id, user_id, total, **kw):
    from audit.models import AuditLog
    AuditLog.objects.create(event='order_confirmed', ref_id=order_id)

@receiver(order_confirmed)
def handle_order_confirmed_cache(sender, order_id, **kw):
    from django.core.cache import cache
    cache.delete(f'order:{order_id}')
    cache.delete(f'user_orders:{kw.get("user_id")}')

Durable Messaging: Celery + Redis/RabbitMQ

For reliable, distributed event processing, Django teams overwhelmingly adopt Celery as a task queue with Redis or RabbitMQ as the broker.[4] The key pattern is transactional outbox: write the event to the database in the same transaction as the state change, then a background process publishes it to the broker. This prevents the dual-write problem (state saved but event lost if the process crashes).[27]

Python
outbox/models.py + outbox/publisher.py — Transactional Outbox
# ── Outbox model ────────────────────────────────────────────────
import uuid, json
from django.db import models

class OutboxEvent(models.Model):
    class Status(models.TextChoices):
        PENDING   = 'pending',   'Pending'
        PUBLISHED = 'published', 'Published'
        FAILED    = 'failed',    'Failed'

    id         = models.UUIDField(primary_key=True, default=uuid.uuid4)
    event_type = models.CharField(max_length=100)
    payload    = models.JSONField()
    status     = models.CharField(max_length=20, choices=Status.choices,
                                  default=Status.PENDING, db_index=True)
    created_at = models.DateTimeField(auto_now_add=True)
    published_at = models.DateTimeField(null=True)

    class Meta:
        ordering = ['created_at']
        indexes  = [models.Index(fields=['status', 'created_at'])]

# ── Service writes to outbox transactionally ────────────────────
from django.db import transaction

class OrderConfirmationService:
    @transaction.atomic
    def confirm(self, order_id: int) -> None:
        order = Order.objects.select_for_update().get(pk=order_id)
        order.status = 'confirmed'
        order.save(update_fields=['status'])

        OutboxEvent.objects.create(        # same transaction!
            event_type='orders.OrderConfirmed',
            payload={'order_id': order_id, 'user_id': order.user_id,
                     'total': str(order.total)},
        )
        # If anything raises here, BOTH saves roll back — no dual-write problem

# ── Periodic relay task ─────────────────────────────────────────
from celery import shared_task
from django.utils import timezone
from kombu import Connection

@shared_task
def relay_outbox_events():
    with transaction.atomic():
        events = (
            OutboxEvent.objects
            .select_for_update(skip_locked=True)
            .filter(status=OutboxEvent.Status.PENDING)[:100]
        )
        for event in events:
            publish_to_broker(event.event_type, event.payload)
            event.status = OutboxEvent.Status.PUBLISHED
            event.published_at = timezone.now()
            event.save(update_fields=['status', 'published_at'])
✓ skip_locked = True
select_for_update(skip_locked=True) is critical in the relay task. It allows multiple Celery workers to process the outbox concurrently without blocking each other, preventing a thundering-herd on the outbox table at scale.[28]

Event Sourcing

Event sourcing takes the outbox concept to its logical conclusion: the system's state is derived entirely from a log of domain events, not from a current-state table. The event log becomes the source of truth; current state is a projection from that log.[29] This is a significant commitment — not appropriate for every bounded context — but provides powerful capabilities: full audit trail, time-travel debugging, and the ability to replay events to build new projections.

Python
eventsourcing/aggregate.py — Minimal event-sourced aggregate
from dataclasses import dataclass, field
from datetime import datetime
from typing import Any
import uuid

@dataclass
class DomainEvent:
    event_type: str
    aggregate_id: str
    sequence: int
    occurred_at: datetime
    data: dict[str, Any]

class EventSourcedOrder:
    def __init__(self, aggregate_id: str):
        self.aggregate_id = aggregate_id
        self.sequence = 0
        self.status = 'new'
        self.lines: list = []
        self._uncommitted: list[DomainEvent] = []

    # ── Commands raise events; events apply state ──────────────────
    def add_item(self, product_id: int, qty: int, price_cents: int) -> None:
        self._raise(DomainEvent(
            event_type='ItemAdded', aggregate_id=self.aggregate_id,
            sequence=self.sequence + 1, occurred_at=datetime.utcnow(),
            data={'product_id': product_id, 'qty': qty, 'price': price_cents},
        ))

    def confirm(self) -> None:
        if self.status != 'pending':
            raise ValueError("Order not in pending state")
        self._raise(DomainEvent(
            event_type='OrderConfirmed', aggregate_id=self.aggregate_id,
            sequence=self.sequence + 1, occurred_at=datetime.utcnow(),
            data={},
        ))

    def _raise(self, event: DomainEvent) -> None:
        self._apply(event)
        self._uncommitted.append(event)

    def _apply(self, event: DomainEvent) -> None:
        self.sequence = event.sequence
        handler = getattr(self, f'_on_{event.event_type}', None)
        if handler:
            handler(event.data)

    def _on_ItemAdded(self, data: dict) -> None:
        self.status = 'pending'
        self.lines.append(data)

    def _on_OrderConfirmed(self, data: dict) -> None:
        self.status = 'confirmed'

    # ── Rehydrate from event stream ─────────────────────────────────
    @classmethod
    def from_events(cls, aggregate_id: str, events: list[DomainEvent]) -> 'EventSourcedOrder':
        obj = cls(aggregate_id)
        for event in events:
            obj._apply(event)
        return obj
Section 08

Scaling Strategies for Django

Architecture patterns reduce complexity; operational scaling handles load. The two are complementary. Well-structured code scales more predictably because bottlenecks are isolated and visible.[30]

Database Optimisation

The Django ORM is lazy and composable, but this ease makes N+1 queries trivially common. Use select_related() for ForeignKey traversals and prefetch_related() for ManyToMany or reverse FK access. Use Django's QuerySet.explain() (Django 3.0+) to introspect query plans directly from Python.[31]

Python
Optimisation patterns
# ── N+1 problem and solution ────────────────────────────────────
# BAD: issues 1 + N queries
orders = Order.objects.filter(status='pending')
for o in orders:
    print(o.user.email)      # DB hit per iteration!

# GOOD: 2 queries total
orders = Order.objects.select_related('user').filter(status='pending')

# ── Bulk operations ─────────────────────────────────────────────
# BAD: N INSERTs
for item in items:
    OrderItem.objects.create(**item)

# GOOD: 1 INSERT
OrderItem.objects.bulk_create([OrderItem(**i) for i in items], batch_size=500)

# ── Read-only queries: use values() to avoid model instantiation ─
totals = (
    Order.objects
         .filter(status='confirmed')
         .values('user_id')
         .annotate(spend=Sum('total'))
         .order_by('-spend')
)

# ── Database routing for read replicas ─────────────────────────
class ReadReplicaRouter:
    READ_PREFIXES = ('ORDER BY', 'SELECT')

    def db_for_read(self, model, **hints):
        return 'replica'

    def db_for_write(self, model, **hints):
        return 'default'

    def allow_migrate(self, db, app_label, model_name=None, **hints):
        return db == 'default'

Async Django (5.x)

Django 4.1 introduced async ORM support; Django 5.x completes the picture with async-capable middleware, views, signals, and cache backends.[3] For I/O-bound workloads — external API calls, file uploads, streaming responses — async views under ASGI can dramatically increase concurrency without additional processes.

Python
views.py — async Django view with async ORM
from django.http import JsonResponse
import asyncio, httpx

async def dashboard_view(request):
    # Fetch DB and external API concurrently
    orders_qs, weather = await asyncio.gather(
        Order.objects.filter(user=request.user).acount(),
        fetch_external_weather_async(request.user.city),
    )
    return JsonResponse({'order_count': orders_qs, 'weather': weather})

async def fetch_external_weather_async(city: str) -> dict:
    async with httpx.AsyncClient() as client:
        r = await client.get(f'https://api.weather.example.com/?city={city}')
        return r.json()

# Async ORM API (Django 4.1+)
async def get_pending_orders(user_id: int) -> list:
    return [o async for o in
            Order.objects.filter(user_id=user_id, status='pending')
                         .select_related('user')]

Caching Architecture

Django supports multiple cache backends: in-memory (development), Memcached, and Redis. The recommended production approach is a two-level cache: per-process in-memory cache (using django-cacheops or a simple lru_cache) backed by a shared Redis cluster.[32]

Horizontal Scaling Checklist

  • Stateless application servers — all session/state in Redis or database; any instance can serve any request.[33]
  • Database connection pooling — use PgBouncer in transaction mode between Django and PostgreSQL to handle hundreds of application connections efficiently.[34]
  • Celery worker autoscaling — use --autoscale=10,3 to scale workers between 3 and 10 processes based on queue depth.
  • Read replicas — route read-heavy query sets through a replica using Django's database router (shown above).
  • Content Delivery Networks — serve media and static files from CDN (e.g. CloudFront + S3 via django-storages).
  • Observability — instrument with OpenTelemetry; ship traces to Jaeger or Tempo; use django-prometheus for metrics.[35]
