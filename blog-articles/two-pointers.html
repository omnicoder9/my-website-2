The Technique

The two-pointer technique is an algorithmic pattern that uses two index variables — called pointers — to traverse a data structure simultaneously. Rather than considering every possible pair of indices in O(n²) nested loops, the pointers share a single pass through the data, producing O(n) solutions to problems that would otherwise require quadratic time.

The core insight is that the relative positions of the pointers encode information about what you've already ruled out. When both ends of a sorted array are examined, any movement of either pointer permanently eliminates a family of candidates that provably cannot yield a solution — so no work is ever repeated.

The fundamental contract: every time you move a pointer, you must be able to prove that the elements you're skipping cannot possibly be part of a valid answer. If that proof holds, you get O(n). If it doesn't, you need a different approach.

When to Reach for Two Pointers

Look for these signals in a problem statement:

The input is sorted (or can be sorted without violating constraints). The problem asks for a pair or triplet summing to a target. The problem involves removing, partitioning, or compacting an array in-place. The problem involves a substring or subarray with a size or sum constraint. The structure is a palindrome or requires symmetric comparison.

Complexity Baseline

The canonical convergent-ends pattern gives O(n) time with O(1) auxiliary space — both optimal for most two-pointer problems. The 3Sum pattern adds a sort pre-step (O(n log n)) plus a nested two-pointer loop (O(n²) total). Sliding window variants are O(n) amortized because each element enters and exits the window exactly once.

Four Archetypes

Every two-pointer problem maps to one of four structural patterns:

① Convergent Ends
L→ [ 1 3 5 7 9 ] ←R move whichever pointer brings you closer to target

Pointers start at opposite ends of a sorted array and meet in the middle. Classic for pair-sum, water container, palindrome checks.

② Slow & Fast
[ 1 1 2 3 3 4 5 ] S→ F→ → slow writes; fast reads

Both pointers move in the same direction. The slow pointer marks the write position; the fast pointer scans ahead. Used for in-place compaction.

③ Sliding Window
[ 2 3 1 2 4 3 ] target=7 L→ R→ expand right, shrink left when condition is met

A variable-width window bounded by two pointers, both moving right only. Each element enters and exits at most once → O(n) amortized.

④ Three Pointers
[ 2 0 2 1 1 0 ] lo mid hi Dutch National Flag; also reverse-merge

Three index variables partition the array into three regions. Used for 3-way partitioning (Sort Colors) and backward merge of two sorted arrays.

    for left < right {
        // skip non-alphanumeric from the left
        for left < right && !unicode.IsLetter(rune(s[left])) &&

Medium Convergent Ends LC 167
Time

Pattern
Convergent

Given a 1-indexed sorted array of integers and a target, return the 1-based indices of the two numbers that add up to the target. Exactly one solution is guaranteed. You may not use the same element twice.

Because the array is sorted, comparing the current sum to the target tells you exactly which pointer to move. Sum too small? The left pointer's value is too small — move it right. Sum too large? The right value is too large — move it left. Each comparison eliminates an entire row or column of the implicit n×n matrix.

func twoSum(numbers []int, target int) []int {
    left, right := 0, len(numbers)-1
    for left < right {
        sum := numbers[left] + numbers[right]
        switch {
        case sum == target:
            return []int{left + 1, right + 1} // 1-indexed
        case sum < target:
            left++  // need a larger value
        default:
            right-- // need a smaller value
        }
    }
    return []int{-1, -1} // guaranteed to find solution
}

import "sort"

func threeSum(nums []int) [][]int {
    sort.Ints(nums)
    result := [][]int{}
    n := len(nums)

    for i := 0; i < n-2; i++ {
        // skip duplicate outer values
        if i > 0 && nums[i] == nums[i-1] {
            continue
        }
        // early exit: smallest possible triple is positive
        if nums[i] > 0 {
            break
        }
        left, right := i+1, n-1
        for left < right {
            sum := nums[i] + nums[left] + nums[right]
            if sum == 0 {
                result = append(result, []int{nums[i], nums[left], nums[right]})
                // skip duplicates on both sides
                for left < right && nums[left] == nums[left+1] {
                    left++
                }
                for left < right && nums[right] == nums[right-1] {
                    right--
                }
                left++
                right--
            } else if sum < 0 {
                left++
            } else {
                right--
            }
        }
    }
    return result
}

Container With Most Water

Medium Convergent Ends LC 11
Time
O(n)
Space
O(1)
Pattern
Greedy Convergent

Given an integer array height where each element represents a vertical line of that height, find two lines that together with the x-axis form a container that holds the most water. Return the maximum amount of water.

Area is determined by the shorter of the two walls times the distance between them: area = min(height[L], height[R]) × (R - L).

The greedy key: moving the taller pointer can never improve the area because the bottleneck is always the shorter wall. Only moving the shorter pointer can potentially reveal a taller wall that yields more water, so that's always the correct move.

func maxArea(height []int) int {
    left, right := 0, len(height)-1
    maxWater := 0
    for left < right {
        h := height[left]
        if height[right] < h {

        }
    }
    return maxWater

The slow/fast pointer pattern extends naturally to linked lists. A slow pointer advances one node per step; a fast pointer advances two. This creates two fundamental primitives:

Cycle detection (Floyd's algorithm, LC 141/142): if a cycle exists, the fast pointer will eventually lap the slow pointer and they will meet inside the cycle. If no cycle exists, fast reaches null first.

The counting trick is elegant: when the window [left, right] has product < k, the number of valid subarrays ending at right is exactly right - left + 1 (every subarray ending at right whose left boundary is between left and right inclusive).

        // every subarray ending at 'right' with left boundary in [left..right]
        count += right - left + 1;
    }
    return count;
}
function numSubarrayProductLessThanK(nums: number[], k: number): number {
    if (k <= 1) return 0; // all values positive, no product can be < 1

    for (let right = 0; right < nums.length; right++) {
        product *= nums[right];
        // shrink window until product is valid again
        while (product >= k) product /= nums[left++];
        // every subarray ending at 'right' with left boundary in [left..right]

    }
    return count;

Pattern
k-Sum Generalisation

Find all unique quadruplets in an integer array that sum to a given target. The solution must not contain duplicate quadruplets. This is the natural extension of 3Sum: fix two outer indices, run Two Sum II on the inner subarray.

The pattern generalises to any k-Sum: sort once, fix k-2 outer loops each skipping duplicates, then run Two Sum II on the remaining subarray. Time complexity is O(n^(k-1)) for k ≥ 2.

Two key pruning optimisations: if the smallest possible quadruplet with the current outer indices already exceeds the target, break (everything larger will too). If the largest possible quadruplet falls short, continue to the next outer index. These prunes don't change worst-case complexity but dramatically speed up average-case performance on sparse inputs.

func fourSum(nums []int, target int) [][]int {
    sort.Ints(nums)
    n := len(nums)
    result := [][]int{}

    for i := 0; i < n-3; i++ {
        if i > 0 && nums[i] == nums[i-1] { continue } // skip dup i
        for j := i + 1; j < n-2; j++ {
            if j > i+1 && nums[j] == nums[j-1] { continue } // skip dup j
            // pruning: smallest/largest quadruplet with current i,j
            if nums[i]+nums[j]+nums[j+1]+nums[j+2] > target { break }
            if nums[i]+nums[j]+nums[n-2]+nums[n-1] < target { continue }

            left, right := j+1, n-1
            for left < right {
                sum := nums[i] + nums[j] + nums[left] + nums[right]
                if sum == target {
                    result = append(result, []int{nums[i], nums[j], nums[left], nums[right]})
                    for left < right && nums[left]  == nums[left+1]  { left++  }
                    for left < right && nums[right] == nums[right-1] { right-- }
                    left++; right--
                } else if sum < target {
                    left++
                } else {
                    right--
                }
            }
        }
    }
    return result
}

Common Pitfalls

Off-by-one on the merge loop. In Merge Sorted Array, the condition is p1 >= 0 && p2 >= 0; when one exhausts, only the other needs copying — and if it's nums1, the elements are already in place.

Forgetting to skip duplicates in 3Sum. After recording a triplet, advance both pointers past consecutive equal values before the final left++; right-- — otherwise you'll record the same triplet repeatedly.

Moving mid after swapping a 2 in Sort Colors. When nums[mid] == 2, you swap it with hi and decrement hi — but you must not increment mid because the element you just brought in from hi has not been examined yet.

Assuming the sliding window always works. The sliding window pattern requires a monotone condition: adding an element always makes the window "more valid" (or "more invalid"), and removing always reverses it. If elements can be negative, this monotonicity breaks and you may need a different structure.

Interview Playbook

Recognising a two-pointer problem under interview pressure is 80% of the work. The rest is mechanical. Here is a battle-tested mental checklist.

When NOT to Use Two Pointers

Two pointers are not a universal hammer. Reach for a hash map when the array is unsorted and sorting would violate index-preservation constraints (classic Two Sum). Reach for a monotonic stack when you need the nearest greater or lesser element for each position (Largest Rectangle in Histogram). Reach for binary search when the search space is huge and a single midpoint test halves it decisively. Reach for prefix sums when a range query must run in O(1) after linear pre-processing.

The key disqualifier for two pointers is lack of monotonicity: if moving a pointer cannot reliably bring you closer to a solution — because the condition you're optimising for is non-monotone — the convergence argument breaks and you need a different data structure.

Step 1 — Signal Scan (30 seconds)

Read the problem statement and tag every signal word: sorted, pair, triplet, palindrome, in-place, contiguous subarray, minimum/maximum length, without extra space, cycle. Two or more of these in combination is a strong indicator. A problem that asks to remove or partition elements without allocating extra memory is almost always the slow/fast pattern.

Step 2 — Brute Force Aloud

State the O(n²) or O(n³) brute force immediately — it shows you understand the problem and gives the interviewer something to confirm before you optimise. Then say: "I notice that if I sort the array, the relationship between elements becomes monotone — moving a pointer in one direction always changes the sum/product in a predictable direction. That lets me replace the inner loop with a converging scan."

Step 3 — Prove the Move

Before writing a single line of code, articulate why moving each pointer is safe. For convergent ends: "When the sum is too large, the right pointer's value must be discarded because even pairing it with the smallest remaining left value exceeds the target — so every candidate involving this right value has been eliminated in one step." This proof is the heart of the technique and what interviewers are actually testing.

Step 4 — Handle Duplicates Explicitly

Duplicate-skipping is the most common source of bugs in 3Sum and 4Sum. Write the skip loops before the final increment/decrement, not after. Test your solution on [-2,-2,0,0,2,2] with target 0 — this catches both the outer-index duplicate skip and the inner-pointer duplicate skip.

The k-Sum Recursive Template

Two Sum II is the base case. Every higher k-Sum reduces to it by fixing one element and recursing. The template below works for any k ≥ 2 in any language that supports recursion: 

import "sort"

// kSum finds all unique k-tuples in nums[start:] that sum to target.
func kSum(nums []int, target, k, start int) [][]int {
    result := [][]int{}
    n := len(nums)

    // base case: Two Sum II using convergent ends
    if k == 2 {
        left, right := start, n-1
        for left < right {
            sum := nums[left] + nums[right]
            if sum == target {
                result = append(result, []int{nums[left], nums[right]})
                for left < right && nums[left]  == nums[left+1]  { left++  }
                for left < right && nums[right] == nums[right-1] { right-- }
                left++; right--
            } else if sum < target {
                left++
            } else {
                right--
            }
        }
        return result
    }

    // recursive case: fix one element, reduce to (k-1)-Sum
    for i := start; i < n-k+1; i++ {
        if i > start && nums[i] == nums[i-1] { continue } // skip duplicates
        // pruning
        if nums[i]*k > target || nums[n-1]*k < target { break }
        for _, sub := range kSum(nums, target-nums[i], k-1, i+1) {
            result = append(result, append([]int{nums[i]}, sub...))
        }
    }
    return result
}

fn k_sum(nums: &[i32], target: i64, k: usize, start: usize) -> Vec> {
    let mut result: Vec> = Vec::new();
    let n = nums.len();

    // base case: Two Sum II using convergent ends
    if k == 2 {
        let (mut left, mut right) = (start, n.wrapping_sub(1));
        while left < right && right < n {
            let sum = nums[left] as i64 + nums[right] as i64;
            match sum.cmp(&target) {
                std::cmp::Ordering::Equal => {
                    result.push(vec![nums[left], nums[right]]);
                    while left < right && nums[left]  == nums[left  + 1] { left  += 1; }
                    while left < right && nums[right] == nums[right - 1] { right -= 1; }
                    left += 1; if right > 0 { right -= 1; }
                }
                std::cmp::Ordering::Less    => left  += 1,
                std::cmp::Ordering::Greater => { if right == 0 { break; } right -= 1; }
            }
        }
        return result;
    }

    // recursive case: fix one element, reduce to (k-1)-Sum
    for i in start..n.saturating_sub(k - 1) {
        if i > start && nums[i] == nums[i - 1] { continue; } // skip duplicates

            sub.insert(0, nums[i]);
            result.push(sub);
        }
    }
    result
