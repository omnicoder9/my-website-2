Overview

Introduction: The Cloud Attack Surface

Cloud computing has fundamentally restructured the attack surface that security teams must defend. Where on-premises infrastructure concentrated risk within physical perimeters, cloud environments distribute it across APIs, identity planes, ephemeral compute, globally distributed storage, and vendor-managed abstractions — each introducing categories of vulnerability that have no precise equivalent in traditional data centres.[1]

The Cloud Security Alliance’s Pandemic Eleven report identifies the most critical cloud-native threats, led overwhelmingly by misconfiguration, inadequate identity controls, and insecure interfaces.[1] Gartner analyst Neil MacDonald’s now-famous assertion — that through 2025, 99% of all cloud security failures would be the customer’s own fault — frames this starkly: the cloud provider is rarely the weakest link.[2]

This article dissects the principal categories of cloud-specific vulnerability — misconfiguration, identity and access risks, shared-responsibility failures, insecure cloud services, and container and serverless threats — with concrete technical examples, real-world incidents, and practical defensive guidance.

Section 01

Misconfiguration

Misconfiguration is the single largest source of cloud security incidents. Cloud provider consoles expose hundreds of configuration toggles; a single wrong setting can expose sensitive data at global scale within seconds. The self-service, rapid-provisioning model that makes cloud valuable also makes accidental misconfiguration extremely easy and widespread.[3]

Publicly Exposed Storage

Object storage — Amazon S3, Azure Blob Storage, Google Cloud Storage — is the most frequently misconfigured cloud resource. Buckets default to private, but a single policy statement or ACL override can render terabytes of data publicly readable or writable.[4] Wiz Research’s 2023 state-of-cloud report found that organisations routinely expose sensitive categories including PII, authentication tokens, and intellectual property through object storage misconfigurations.[4]

⚠ Real-World Incident

Capital One Breach — 2019 (106 million records)

A misconfigured Web Application Firewall permitted a Server-Side Request Forgery (SSRF) attack. The attacker used the SSRF to query the AWS Instance Metadata Service (IMDS), retrieve IAM role credentials attached to the WAF’s EC2 instance, and then use those credentials to list and download over 100 million records from S3 buckets. The root cause was an overly permissive IAM role combined with an unpatched SSRF surface — not a breach of AWS’s own systems.[3]

A common offending S3 bucket policy pattern grants public access to all principals:

Insecure S3 bucket policy — grants unauthenticated read to any principal 
{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    // WARNING: Principal * = any unauthenticated internet user
    "Principal": "*",
    "Action": "s3:GetObject",
    "Resource": "arn:aws:s3:::my-data-bucket/*"
  }]
}

The corrected pattern restricts access to a specific service account principal and requires TLS:

Hardened S3 bucket policy — principal-restricted with TLS enforcement 
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::123456789012:role/DataServiceRole"
      },
      "Action": ["s3:GetObject", "s3:ListBucket"],
      "Resource": [
        "arn:aws:s3:::my-data-bucket",
        "arn:aws:s3:::my-data-bucket/*"
      ]
    },
    {
      // Deny all HTTP (non-TLS) requests
      "Effect": "Deny",
      "Principal": "*",
      "Action": "s3:*",
      "Resource": ["arn:aws:s3:::my-data-bucket/*"],
      "Condition": { "Bool": { "aws:SecureTransport": "false" } }
    }
  ]
}

Microsoft Power Apps portals were the subject of a 2021 exposure event where default settings on more than 1,000 organisations’ apps made tables publicly accessible, exposing approximately 38 million records including vaccination statuses and employment data — a direct consequence of a permissive default configuration that users failed to override.[4]

Open Security Groups

Security groups and cloud firewall rules serve as the virtual network perimeter around cloud resources. Unit 42’s 2022 Cloud Threat Report found that 63% of analysed cloud environments contained security groups with SSH (port 22) or RDP (port 3389) exposed to the entire internet (0.0.0.0/0), and 80% of organisations had at least one publicly readable storage bucket.[5]

Improper IAM Policies

Identity and Access Management policies in cloud environments govern not only human users but also service accounts, EC2 instance profiles, Lambda execution roles, and cross-account trust relationships. Overly permissive IAM policies — particularly the use of wildcard actions (*) or wildcard resources — are among the most dangerous and pervasive misconfigurations.[8]

Over-permissive IAM policy — full S3 and EC2 access via wildcards 
{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    // WARNING: Action wildcard = all S3 and EC2 operations
    "Action": ["s3:*", "ec2:*"],
    // WARNING: Resource wildcard = all resources in account
    "Resource": "*"
  }]
}
Least-privilege IAM policy — scoped to specific bucket and read-only actions 
{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Action": [
      "s3:GetObject",
      "s3:GetObjectVersion",
      "s3:ListBucket"
    ],
    "Resource": [
      "arn:aws:s3:::production-assets",
      "arn:aws:s3:::production-assets/*"
    ],
    // Condition restricts to specific source VPC endpoint
    "Condition": {
      "StringEquals": {
        "aws:SourceVpce": "vpce-0abc1234def56789a"
      }
    }
  }]
}

Rhino Security Labs documented 21 distinct IAM privilege-escalation paths in AWS — methods by which an attacker holding limited permissions can chain policy manipulations, role assumptions, or Lambda deployments to achieve effective administrator-level access without ever being explicitly granted it.[17]

Section 02

Identity and Access Risks

In cloud environments, identity is the new perimeter. Traditional network boundaries are largely absent; what prevents an adversary from reading a database or invoking a privileged API is the correctness of identity controls — IAM roles, service account credentials, OAuth tokens, and MFA bindings. Failures in any of these create pathways that are often easier to exploit than any software vulnerability.[9]

Excessive Privileges

The principle of least privilege is widely understood but rarely achieved in practice. Lacework’s 2022 Cloud Threat Report found that 71% of cloud security incidents involved excessive IAM permissions, and that on average 90% of granted permissions were never used in a 90-day period.[9] This “permissions gap” represents latent blast radius: if credentials are compromised, the attacker inherits all granted rights, not just those the legitimate user exercises.

On average, 90% of granted cloud permissions are never exercised within a 90-day window — each unused permission is a latent blast-radius multiplier waiting for a credential compromise.

Common privilege excess patterns include:

Anti-PatternDescriptionRisk
Wildcard actions on *Grant all actions on all resourcesFull account takeover on credential theft
AdministratorAccess on service rolesLambda / EC2 roles with root-equivalent permissionsSSRF → IMDS → admin compromise
Cross-account trust with * principalAny account may assume the roleAccount-hopping by external attacker
Unrotated service account keysLong-lived access keys exposed in code reposPersistent silent access after leakage
PassRole without restrictionAbility to assign arbitrary roles to Lambda/EC2Privilege escalation to admin via new resource

Tools such as AWS IAM Access Analyzer, Google Cloud’s Policy Analyzer, and open-source projects like PMapper can enumerate these paths automatically, but remediating them requires sustained commitment to reviewing and narrowing permissions — a process that must be built into CI/CD pipelines and periodic audits.[8]

Stolen Cloud Credentials

Cloud credentials — AWS access key pairs, Azure service principal secrets, GCP service account JSON keys — are high-value targets because they provide direct programmatic access to cloud APIs. Verizon’s 2023 Data Breach Investigations Report identified stolen credentials as the single most common attack vector across all breach categories, accounting for the majority of web application attacks.[10]

⚠ Real-World Incident

Tesla Kubernetes Cryptojacking — 2018

RedLock researchers discovered that Tesla’s Kubernetes administrative console was accessible without authentication. Attackers accessed it, retrieved AWS credentials stored in Kubernetes secrets, and used those credentials to provision EC2 instances running a Monero mining operation. The payload was configured to use non-standard mining pool endpoints and throttled CPU usage to avoid detection. Tesla had no idea the breach had occurred until RedLock disclosed it.[21]

AWS IMDSv2, which requires a session-oriented token request before metadata retrieval, significantly mitigates SSRF-based credential theft. Enforcing IMDSv2 at the account level — so that no instance can respond to IMDSv1 requests — is a high-priority hardening action.[14]

AWS CLI — enforce IMDSv2 on new and existing EC2 instances 
# Require IMDSv2 at instance launch
aws ec2 run-instances \
  --image-id ami-0abcdef1234567890 \
  --instance-type t3.medium \
  --metadata-options HttpTokens=required,HttpPutResponseHopLimit=1

# Patch existing running instances to require IMDSv2
aws ec2 modify-instance-metadata-options \
  --instance-id i-1234567890abcdef0 \
  --http-tokens required \
  --http-put-response-hop-limit 1

Weak MFA Implementation

Multi-factor authentication is a critical compensating control against credential theft, but not all MFA implementations provide equal protection. CISA’s 2022 guidance on phishing-resistant MFA distinguishes between phishable MFA methods (TOTP codes, SMS OTPs, push notifications) and phishing-resistant methods (FIDO2/WebAuthn hardware tokens, PIV/CAC smart cards).[24]

⚠ Real-World Incident

Uber MFA Fatigue Attack — 2022

An attacker acquired an Uber contractor’s VPN credentials, then bombarded the contractor with push MFA notifications until the user, fatigued by the alerts, approved one. The attacker then found hardcoded Thycotic PAM credentials in a network share, which granted access to Uber’s AWS, GCP, and internal systems. The attack succeeded not because cryptography was broken but because the MFA implementation relied on user discipline under deliberately created pressure.[24]

Section 03

Shared Responsibility Risks

Every major cloud provider publishes a shared responsibility model that delineates which security controls they manage and which remain the customer’s obligation. In practice, this boundary is frequently misunderstood, particularly by teams migrating from on-premises environments, leading to entire categories of exposure being left unaddressed because each party assumed the other was responsible.[16]

LayerAWS ResponsibilityCustomer Responsibility
Physical data centres✓ Full—
Hypervisor / virtualisation✓ Full—
Managed service (e.g. RDS)✓ OS, patching, HADB config, encryption, access control
Networking (VPC, routing)Underlying fabric✓ Security groups, NACLs, VPC design
EC2 guest OS—✓ Patching, hardening, endpoint security
IAM, policies, credentials—✓ Full
Data encryptionProvides tools (KMS, SSE)✓ Must enable and configure
Application code—✓ Full

A common failure occurs with managed database services. Customers assume that because AWS manages RDS patching and availability, security is handled — but RDS security groups, parameter groups, audit logging, encryption-at-rest, and access controls remain fully within customer scope. Similarly, enabling AWS CloudTrail or Azure Monitor is not automatic; teams that do not configure audit logging have no forensic trail after a breach.[20]

The shared responsibility model does not divide security evenly — it divides it precisely. Every undefined gap lands in the customer’s column by default.

ℹ
SaaS vs IaaS DistinctionFor SaaS products (Salesforce, Microsoft 365, Workday), the provider’s share expands significantly — they own the full stack below the application data. However, data governance, access management, DLP configuration, and integration security remain entirely customer responsibilities even in full SaaS deployments.

The Cloud Security Alliance’s Cloud Controls Matrix (CCM v4.0) provides a comprehensive mapping of controls to cloud service types (IaaS, PaaS, SaaS) and maps them against ISO 27001, NIST 800-53, PCI DSS, and other frameworks, providing a structured basis for responsibility assignment.[20]

Section 04

Cloud Service Risks

Beyond misconfiguration and identity, cloud platforms introduce structural risks that arise from their architecture: APIs that provide the only control plane, multi-tenant infrastructure that shares physical resources, vendor relationships that create lock-in, and data placement obligations that conflict with regulatory requirements. These risks require design-level thinking, not just configuration hardening.[7]

Insecure APIs

Cloud provider APIs, customer-deployed application APIs, and inter-service APIs all represent attack surface. OWASP’s 2023 API Security Top 10 identifies Broken Object Level Authorization (BOLA) as the most critical API vulnerability class — where API endpoints fail to verify that the requesting identity is authorised to access the specific object being requested.[7]

BOLA vulnerability — API returns any user’s data if ID is known 
# Attacker calls API with manipulated user ID
GET /api/v1/users/8823/invoices HTTP/1.1
Authorization: Bearer <attacker-token-for-user-id-9001>

# Insecure server: validates token exists, but does NOT
# verify that user 9001 owns resource belonging to user 8823
HTTP/1.1 200 OK
{
  "invoices": [
    { "id": "INV-8823-001", "amount": 4200.00, "ssn_last4": "7742" }
  ]
}

Cloud-native API Gateways (AWS API Gateway, Azure API Management, Google Cloud Endpoints) provide built-in capabilities for authentication enforcement, rate limiting, and request validation. Failing to enable these controls — for example, deploying a Lambda function URL without an authoriser — creates unauthenticated endpoints at cloud scale.[8]

Multi-Tenancy Concerns

Public cloud infrastructure is shared by default. Multiple customers’ workloads run on the same physical hosts, share the same hypervisor, and in some cases share the same CPU. While hypervisor isolation is robust in practice, speculative execution vulnerabilities such as Spectre (CVE-2017-5753) and Meltdown (CVE-2017-5754) demonstrated that hardware-level side-channel attacks can defeat software isolation boundaries.[25]

🚨
Cross-Tenant Side-Channel RiskSpectre-class attacks exploit CPU branch prediction and out-of-order execution to leak data across process boundaries. In a cloud co-tenancy context, this theoretically permits one tenant to read another tenant’s memory. All major providers mitigated this via microcode updates and hypervisor patches, but the discovery fundamentally altered assumptions about isolation guarantees.

Beyond hardware side channels, multi-tenancy introduces several operational risks:

RiskMechanismMitigation
Noisy NeighbourCo-tenant resource exhaustion affects shared infrastructureReserved capacity; dedicated hosts
Shared IP reputationMalicious co-tenant blacklists shared egress IPDedicated egress IPs; NAT gateway
Data remanenceDeallocated storage retains prior tenant data in unwiped blocksEncrypted volumes; provider zero-fill attestations
Cross-tenant API abuseProvider control-plane APIs misroute requests in multi-tenant SaaSTenant isolation validation in SaaS architecture reviews

For highly sensitive workloads, providers offer dedicated tenancy options — AWS Dedicated Hosts, Azure Dedicated Hosts, and GCP Sole-Tenant Nodes — which provide physical isolation at the cost of reduced elasticity and higher pricing.[8]

Vendor Lock-In Issues

Vendor lock-in is a security risk as well as a commercial one. Organisations that have deeply integrated proprietary cloud services — AWS Lambda, Azure Cognitive Services, Google BigQuery — face significant barriers to migrating workloads if a provider suffers a prolonged outage, introduces an untenable pricing change, or fails to meet emerging regulatory requirements.[1]

The security dimension of lock-in manifests in several ways. First, proprietary encryption key management services, while convenient, mean that key custody is partially delegated to the provider. Second, logging and SIEM integrations built on provider-native formats (CloudWatch, Azure Monitor) may not export cleanly to third-party systems, reducing portability of forensic evidence. Third, cloud-native IAM models differ substantially between AWS, Azure, and GCP, making multi-cloud consistent policy enforcement technically complex.

✔
Lock-In Mitigation StrategiesPrefer open standards where security-equivalent: OpenTelemetry over proprietary metrics, OIDC federation over native IAM users, LUKS/AES encryption over provider-only CMKs where exportability matters. Conduct annual provider migration feasibility assessments as part of business continuity planning.

Data Residency Concerns

Cloud providers operate data centres globally, and data may replicate across regions by default for availability reasons. This creates compliance risk where specific regulations require data to remain within defined geographic boundaries — GDPR in the EU, China’s Data Security Law, India’s PDPB, and sector-specific rules such as FedRAMP in the U.S. government context.[23]

The Court of Justice of the European Union’s 2020 Schrems II ruling (Case C-311/18) invalidated the EU-U.S. Privacy Shield, dramatically complicating transatlantic data transfers. It established that even contractual mechanisms (Standard Contractual Clauses) may be insufficient if the destination country’s surveillance laws enable access to personal data in ways incompatible with EU fundamental rights.[23]

Schrems II was not a technical ruling — it was a constitutional one. No amount of encryption or configuration can satisfy it if the data is legally accessible to a foreign intelligence apparatus at the provider’s legal obligation.

Data residency controls available in major cloud platforms include region-locked S3 buckets, Azure Policy initiatives that deny resource creation outside approved regions, and GCP Organisation Policy constraints on resource location. These must be configured explicitly — none are enforced by default.[20]

Section 05

Container and Serverless Risks

Containers and serverless functions have become the dominant deployment targets for cloud-native applications. They introduce their own threat models that differ markedly from virtual machine security: image supply chain integrity, kernel namespace sharing, ephemeral execution contexts, and event-driven attack surfaces that existing security tooling was not designed to address.[6]

Vulnerable Container Images

Container images aggregate operating system packages, language runtimes, third-party libraries, and application code into a single layered artifact. Each layer is a potential source of known vulnerabilities. Snyk’s 2023 Docker security research found that the majority of popular Docker Hub base images contained high or critical severity CVEs, many of which had been patched in later versions but were embedded in outdated base images in active use.[15]

Dockerfile — insecure practices vs. hardened equivalent 
# ===== INSECURE Dockerfile =====
FROM ubuntu:latest          # Mutable tag; build not reproducible
RUN apt-get install -y curl wget python3 vim git  # Excessive tooling
COPY . /app                 # Copies secrets, .env files, .git history
RUN pip install -r requirements.txt  # No version pinning
USER root                   # Container runs as root
CMD ["python3", "app.py"]

# ===== HARDENED Dockerfile =====
FROM python:3.12.3-slim-bookworm  # Pinned digest; minimal base

WORKDIR /app

# Install only production dependencies as non-root
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt \
    && rm -rf /root/.cache

# Copy only application source; use .dockerignore
COPY --chown=appuser:appuser src/ ./src/

# Drop to unprivileged user
RUN groupadd -r appuser && useradd -r -g appuser appuser
USER appuser

# Drop all Linux capabilities; read-only filesystem
# (enforced at runtime via securityContext)
CMD ["python3", "-m", "src.main"]

Docker Hub’s 2019 security breach exposed data from approximately 190,000 accounts, including access tokens for GitHub and Bitbucket repositories linked to automated builds.[22] The incident illustrated the supply chain risk posed by public container registries: images pulled from Docker Hub may be compromised at the registry layer, not just within their own build process.

Image scanning must be integrated into CI/CD pipelines using tools such as Trivy, Grype, or AWS ECR image scanning. Admission controllers in Kubernetes (OPA/Gatekeeper, Kyverno) can enforce that only scanned and policy-compliant images may be deployed to production clusters.[6]

Container Escape

Container escape vulnerabilities allow a process running inside a container to break the namespace isolation boundary and execute code on the host kernel or adjacent containers. NIST SP 800-190 identifies this as one of the most severe container-specific risks.[6] CVE-2019-5736 demonstrated a practical runc escape: a malicious container image could overwrite the host’s runc binary, executing arbitrary commands as root on the host the next time any container was started.[18]

Container escapes commonly exploit one or more of the following misconfigurations:

The 2021 OMIGOD vulnerability (CVE-2021-38647) illustrated cloud-specific container risk: Azure’s Open Management Infrastructure (OMI) agent was silently installed alongside several popular Azure services on Linux VMs. An unauthenticated HTTP request to its UNIX socket allowed remote code execution as root. Wiz Research estimated that approximately 65% of Azure Linux VM customers were affected at time of disclosure.[19]

Insecure Serverless Functions

Serverless platforms — AWS Lambda, Azure Functions, Google Cloud Run — abstract infrastructure management but retain application-layer vulnerability surfaces. OWASP’s Serverless Top 10 identifies injection via event data, insecure function configurations, and overpermissive execution roles as the leading serverless risks.[12]

Unlike traditional applications, serverless functions receive input from diverse event sources — API Gateway request bodies, S3 event notifications, SQS messages, SNS topics, DynamoDB streams — any of which may carry attacker-controlled data. Injection vulnerabilities arise when this event data is passed unsanitised to database queries, shell commands, or template engines.

Lambda SQL injection via S3 event — attacker controls bucket/key name 
# Attacker uploads file with malicious key name:
# "reports/'; DROP TABLE users; --"

# Vulnerable Lambda handler (Python)
import boto3, pymysql

def handler(event, context):
    key = event['Records'][0]['s3']['object']['key']
    conn = pymysql.connect(host=DB_HOST, ...)
    cursor = conn.cursor()
    # UNSAFE: direct string interpolation into SQL
    cursor.execute(f"INSERT INTO processed_files (path) VALUES ('{key}')")

# Secure equivalent: parameterised query
    cursor.execute(
        "INSERT INTO processed_files (path) VALUES (%s)",
        (key,)          # Database driver handles escaping
    )

Serverless functions are frequently granted IAM execution roles with far broader permissions than their logic requires. A Lambda that only needs to read from one DynamoDB table may be given DynamoDB full access, or even broader administrative policies, by engineers following convenience over least-privilege. This dramatically amplifies the impact of any code execution vulnerability within the function.[12]

⚠
Serverless Dependency RiskServerless functions consume third-party dependencies bundled in deployment packages or layers. The 2021 npm ua-parser-js supply chain attack demonstrated that a single compromised package can affect thousands of deployed functions simultaneously, as no patching or restart is required — the malicious code executes on the next invocation of any consuming function.

Additional serverless-specific risks include function event-data tampering (when upstream event sources like SQS are accessible to less-trusted principals), cold-start timing side channels that reveal internal logic, and insufficient function timeout and concurrency controls that enable denial-of-service amplification attacks via unbounded invocation.[12]

Section 06

Defensive Posture

Effective cloud security requires a layered defensive posture that addresses each vulnerability category with controls appropriate to that layer. NIST SP 800-207’s Zero Trust Architecture provides the conceptual framework: assume breach, verify explicitly, grant least privilege, and continuously monitor all resource access regardless of network location.[13]

IBM’s 2023 Cost of a Data Breach report found that organisations with extensive deployment of security AI and automation reduced breach costs by an average of $1.76 million compared to those without, and detected breaches an average of 108 days faster.[11] Cloud-native SIEM integration, automated compliance scanning, and infrastructure-as-code security linting (tfsec, checkov, KICS) operationalise this advantage.

Cloud security is a continuous practice, not a one-time audit. Threat actors actively scan for newly created misconfigured resources — automated scanners claim open S3 buckets and exposed Kubernetes dashboards within minutes of creation. Continuous Cloud Security Posture Management (CSPM) integrated with CI/CD pipelines, combined with developer security training focused on cloud-native anti-patterns, represents the state of the art in organisational cloud defence.[13]
