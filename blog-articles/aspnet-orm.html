1. Thesis: ORM as an architectural boundary

Object-relational mapping in ASP.NET Core is not merely a convenience layer that turns rows into objects. It is a boundary where application semantics, database semantics, transaction semantics, observability, performance, and team workflow meet. Microsoft documents Entity Framework Core as an ORM that lets .NET developers work with a database using .NET objects and removes much of the typical data-access code, but ASP.NET Core teams often choose alternatives when they want tighter SQL control, different performance tradeoffs, legacy compatibility, or richer domain-model patterns than their current EF Core usage provides.1

This article now treats Entity Framework Core as the baseline mainstream ORM and compares it with three alternatives that occupy distinct positions on the abstraction spectrum:

Entity Framework Core

The mainstream Microsoft ORM for modern .NET. Microsoft describes EF Core as a lightweight, extensible, open-source, cross-platform version of Entity Framework that lets .NET developers work with databases using .NET objects and avoid much of the data-access code normally written by hand.1 EF Core supports LINQ queries, change tracking, updates, schema migrations, and a provider model for SQL Server, SQLite, PostgreSQL, MySQL, Azure Cosmos DB, and other stores.17

Dapper

A SQL-first micro-ORM. It extends ADO.NET connections with concise methods for executing SQL and materializing results. Dapper’s official documentation describes it as a NuGet library that enhances DbConnection via extension methods and supports synchronous, asynchronous, buffered, and non-buffered queries.2

RepoDB

A hybrid ORM. RepoDB’s project documentation describes it as an open-source .NET ORM that bridges micro-ORMs and full ORMs, allowing basic and advanced operations from a connection and connection string.3

NHibernate

A mature full ORM for .NET. NHibernate’s site describes it as an actively developed, fully featured object-relational mapper used in thousands of projects.4 Its current reference documentation is version 5.6.5

The key design question is not “which ORM is best?” It is “which persistence model gives this bounded context the best failure modes?” A reporting endpoint that joins fifteen tables and returns a DTO has different needs from a write-heavy aggregate root with invariants, cascades, optimistic locking, and a transactional outbox.

Practical rule: treat data access as an explicit architectural policy. Decide per bounded context, not per developer preference. A single ASP.NET Core system can legitimately use EF Core for mainstream CRUD and migrations, Dapper for read models, RepoDB for administrative CRUD and bulk ingestion, and NHibernate for complex domain aggregates.

2. Historical context: from ADO.NET boilerplate to layered persistence choices

The .NET data-access stack began with explicit connection, command, reader, and transaction objects. ADO.NET remains the substrate underneath many ORMs: connection pooling, parameterized commands, transactions, and provider-specific behavior still define how ASP.NET Core applications communicate with relational databases. Microsoft’s ADO.NET documentation states that SQL Server connection pooling reduces the number of times new connections must be opened by keeping physical connections alive per configuration, returning an available pooled connection when an application calls Open.6

ORMs emerged to reduce repetitive data-reader mapping and to align persistence code with application models. Full ORMs such as Entity Framework Core and NHibernate aim to represent persistent entities and relationships, track changes, and generate SQL. Micro-ORMs such as Dapper keep SQL visible and focus on parameter binding and object materialization. Hybrid ORMs such as RepoDB try to reduce repetitive CRUD and bulk-operation code while staying closer to explicit SQL and connection-oriented programming than a full unit-of-work ORM.

In modern ASP.NET Core, the dependency-injection container becomes the composition root for data access. ASP.NET Core supports dependency injection as a first-class design pattern for inversion of control, and Microsoft’s documentation emphasizes constructor injection and request-scoped services for database work.7 That matters because lifetime errors are persistence errors: a singleton service holding a request-scoped session, a long-lived connection, or an unbounded transaction can corrupt system behavior under load.

3. Data-access mental model in ASP.NET Core

Every ORM integration should be reasoned about through five layers:

  1. Provider layer. ADO.NET provider objects such as DbConnection, DbCommand, DbTransaction, and provider-specific implementations such as SqlConnection, NpgsqlConnection, or MySqlConnection.
  2. Mapping layer. The rules that map rows, columns, joins, identifiers, and relationships into .NET types.
  3. Unit-of-work layer. The boundary in which changes are tracked or explicitly composed before commit.
  4. ASP.NET Core lifetime layer. The DI scopes that decide how long connections, sessions, factories, and repositories live.
  5. Operational layer. Logging, metrics, retries, migrations, connection pooling, health checks, tracing, and query review.

Figure 1. EF Core, Dapper, RepoDB, and NHibernate differ above the provider layer. They still depend on provider behavior, pooling, transactions, and SQL execution semantics.

That diagram also explains why “ORM performance” debates can be misleading. If a request spends 120 ms in a database join, a 2 ms mapper difference is noise. If an endpoint maps millions of rows, buffering, allocation, and column-to-property resolution dominate. If the application leaks connections, connection-pool starvation will dwarf all mapper concerns.

Dapper and RepoDB are closest to ADO.NET. They usually do not want long-lived connections. EF Core and NHibernate are different: their context/session objects are persistence contexts with identity and unit-of-work behavior, so lifetime is a design decision. For EF Core, the default ASP.NET Core pattern is a scoped DbContext; for NHibernate, the session is itself the persistence context and identity map, so its lifetime is a design decision. A typical web application uses one NHibernate session per request or command, not one per entity method call and not one singleton session for the entire application.

5. Sample domain and database schema

To keep examples concrete, assume a small order-management service with customers, orders, and line items. The schema below uses SQL Server syntax, but the same concepts apply to PostgreSQL, MySQL, and SQLite with provider-specific adjustments.

CREATE TABLE dbo.Customers (
    CustomerId       uniqueidentifier NOT NULL PRIMARY KEY,
    Email            nvarchar(320)    NOT NULL UNIQUE,
    DisplayName      nvarchar(200)    NOT NULL,
    CreatedUtc       datetime2        NOT NULL DEFAULT sysutcdatetime()
);

CREATE TABLE dbo.Orders (
    OrderId          uniqueidentifier NOT NULL PRIMARY KEY,
    CustomerId       uniqueidentifier NOT NULL,
    Status           nvarchar(32)     NOT NULL,
    CreatedUtc       datetime2        NOT NULL DEFAULT sysutcdatetime(),
    PaidUtc          datetime2        NULL,
    RowVersion       rowversion       NOT NULL,
    CONSTRAINT FK_Orders_Customers FOREIGN KEY (CustomerId)
        REFERENCES dbo.Customers(CustomerId)
);

CREATE TABLE dbo.OrderLines (
    OrderLineId      uniqueidentifier NOT NULL PRIMARY KEY,
    OrderId          uniqueidentifier NOT NULL,
    Sku              nvarchar(80)     NOT NULL,
    Quantity         int              NOT NULL,
    UnitPriceCents   int              NOT NULL,
    CONSTRAINT FK_OrderLines_Orders FOREIGN KEY (OrderId)
        REFERENCES dbo.Orders(OrderId)
);

CREATE INDEX IX_Orders_CustomerId_CreatedUtc
    ON dbo.Orders(CustomerId, CreatedUtc DESC);

The corresponding .NET types can be written as records for read models and classes for mutable domain entities. With Dapper and RepoDB, immutable DTOs often work well for reads; with NHibernate, persistent entities are usually modeled to satisfy proxying, constructor, identity, and change-tracking requirements.

public sealed record OrderSummaryDto(
    Guid OrderId,
    Guid CustomerId,
    string CustomerEmail,
    string Status,
    DateTime CreatedUtc,
    int LineCount,
    long TotalCents);

public sealed class Order
{
    public virtual Guid OrderId { get; protected set; }
    public virtual Guid CustomerId { get; protected set; }
    public virtual string Status { get; protected set; } = "Draft";
    public virtual DateTime CreatedUtc { get; protected set; }
    public virtual DateTime? PaidUtc { get; protected set; }
    public virtual byte[] RowVersion { get; protected set; } = Array.Empty<byte>();
    public virtual IList<OrderLine> Lines { get; protected set; } = new List<OrderLine>();

    protected Order() { } // Useful for ORMs that materialize entities.

    public Order(Guid customerId, DateTime nowUtc)
    {
        OrderId = Guid.NewGuid();
        CustomerId = customerId;
        CreatedUtc = nowUtc;
    }

    public virtual void AddLine(string sku, int quantity, int unitPriceCents)
    {
        if (quantity <= 0) throw new ArgumentOutOfRangeException(nameof(quantity));
        if (unitPriceCents < 0) throw new ArgumentOutOfRangeException(nameof(unitPriceCents));

        Lines.Add(new OrderLine(OrderId, sku, quantity, unitPriceCents));
    }

    public virtual void MarkPaid(DateTime nowUtc)
    {
        if (Status == "Paid") return;
        if (Lines.Count == 0) throw new InvalidOperationException("Cannot pay an empty order.");
        Status = "Paid";
        PaidUtc = nowUtc;
    }
}

public sealed class OrderLine
{
    public virtual Guid OrderLineId { get; protected set; }
    public virtual Guid OrderId { get; protected set; }
    public virtual string Sku { get; protected set; } = "";
    public virtual int Quantity { get; protected set; }
    public virtual int UnitPriceCents { get; protected set; }

    protected OrderLine() { }

    public OrderLine(Guid orderId, string sku, int quantity, int unitPriceCents)
    {
        OrderLineId = Guid.NewGuid();
        OrderId = orderId;
        Sku = sku;
        Quantity = quantity;
        UnitPriceCents = unitPriceCents;
    }
}

6. ASP.NET Core integration patterns

ASP.NET Core’s DI conventions support grouped service registration through extension methods, and Microsoft documents Add{GROUP NAME} extension methods as the framework convention for registering related services.7 Use that convention for persistence infrastructure. It keeps Program.cs readable and gives you a single place to enforce lifetimes.

6.1 Connection factory abstraction

Dapper and RepoDB should usually receive a factory, not a singleton open connection. In SQL Server, connection pooling is automatic for the provider unless disabled, and disposing a pooled connection returns the underlying physical connection to the pool instead of necessarily closing a TCP connection.6

using System.Data.Common;
using Microsoft.Data.SqlClient;

public interface IDbConnectionFactory
{
    ValueTask<DbConnection> OpenConnectionAsync(CancellationToken cancellationToken = default);
}

public sealed class SqlConnectionFactory(IConfiguration configuration) : IDbConnectionFactory
{
    private readonly string _connectionString =
        configuration.GetConnectionString("Orders")
        ?? throw new InvalidOperationException("Missing connection string: Orders");

    public async ValueTask<DbConnection> OpenConnectionAsync(CancellationToken cancellationToken = default)
    {
        var connection = new SqlConnection(_connectionString);
        try
        {
            await connection.OpenAsync(cancellationToken).ConfigureAwait(false);
            return connection;
        }
        catch
        {
            await connection.DisposeAsync().ConfigureAwait(false);
            throw;
        }
    }
}

6.2 Service registration

public static class PersistenceServiceCollectionExtensions
{
    public static IServiceCollection AddOrderPersistence(this IServiceCollection services)
    {
        services.AddSingleton<IDbConnectionFactory, SqlConnectionFactory>();

        // Dapper-style repositories.
        services.AddScoped<IOrderReadRepository, DapperOrderReadRepository>();

        // RepoDB-style repositories.
        services.AddScoped<IOrderBulkImportRepository, RepoDbOrderBulkImportRepository>();

        // NHibernate registrations are shown later; session factory should be singleton,
        // while sessions are usually scoped per request/command.

        return services;
    }
}

Then the application root stays simple:

var builder = WebApplication.CreateBuilder(args);

builder.Services.AddControllers();
builder.Services.AddOrderPersistence();

var app = builder.Build();
app.MapControllers();
app.Run();

6.3 Minimal API endpoint shape

app.MapGet("/customers/{customerId:guid}/orders", async (
    Guid customerId,
    IOrderReadRepository repository,
    CancellationToken cancellationToken) =>
{
    var orders = await repository.GetRecentOrdersAsync(customerId, cancellationToken);
    return Results.Ok(orders);
});

The endpoint has no ORM dependency. That is deliberate. Controllers and endpoints should depend on application-facing ports such as repositories, query services, or command handlers. The ORM is an implementation detail unless the application layer intentionally models a session or unit of work.

7. Entity Framework Core deep dive

Entity Framework Core is the default starting point for many ASP.NET Core applications because it combines a high-level programming model with Microsoft documentation, provider integrations, migrations, LINQ, relationship mapping, and change tracking. Microsoft describes EF Core as a modern object-database mapper for .NET that supports LINQ queries, change tracking, updates, and schema migrations, with database support provided through provider plug-ins.17

7.1 Installing EF Core packages

A SQL Server-backed ASP.NET Core service commonly installs the runtime provider plus the design-time package used by migration tooling. Replace SQL Server with Npgsql.EntityFrameworkCore.PostgreSQL, Pomelo.EntityFrameworkCore.MySql, Microsoft.EntityFrameworkCore.Sqlite, or another provider when the database changes.

dotnet add package Microsoft.EntityFrameworkCore.SqlServer
 dotnet add package Microsoft.EntityFrameworkCore.Design
 dotnet tool install --global dotnet-ef

EF Core design-time tooling is used for commands such as adding migrations, updating databases, and scaffolding models; Microsoft documents dotnet ef as the .NET Core command-line tooling for EF Core design-time development.18

7.2 Modeling entities and DbContext

EF Core usually starts with entity classes and a DbContext. The DbContext represents a session with the database and combines unit-of-work and repository-like behavior for querying and saving entity instances. Keep it short-lived and scoped to a request, command, or message handler rather than storing it in singletons.

using Microsoft.EntityFrameworkCore;

public sealed class OrderingDbContext(DbContextOptions<OrderingDbContext> options)
    : DbContext(options)
{
    public DbSet<Customer> Customers => Set<Customer>();
    public DbSet<Order> Orders => Set<Order>();
    public DbSet<OrderLine> OrderLines => Set<OrderLine>();

    protected override void OnModelCreating(ModelBuilder modelBuilder)
    {
        modelBuilder.Entity<Customer>(b =>
        {
            b.ToTable("Customers", "dbo");
            b.HasKey(x => x.CustomerId);
            b.Property(x => x.Email).HasMaxLength(320).IsRequired();
            b.Property(x => x.DisplayName).HasMaxLength(200).IsRequired();
            b.HasIndex(x => x.Email).IsUnique();
        });

        modelBuilder.Entity<Order>(b =>
        {
            b.ToTable("Orders", "dbo");
            b.HasKey(x => x.OrderId);
            b.Property(x => x.Status).HasMaxLength(32).IsRequired();
            b.Property(x => x.RowVersion).IsRowVersion();
            b.HasOne(x => x.Customer)
             .WithMany(x => x.Orders)
             .HasForeignKey(x => x.CustomerId);
        });

        modelBuilder.Entity<OrderLine>(b =>
        {
            b.ToTable("OrderLines", "dbo");
            b.HasKey(x => x.OrderLineId);
            b.Property(x => x.Sku).HasMaxLength(64).IsRequired();
            b.Property(x => x.UnitPrice).HasPrecision(18, 2);
            b.HasOne(x => x.Order)
             .WithMany(x => x.Lines)
             .HasForeignKey(x => x.OrderId);
        });
    }
}

public sealed class Customer
{
    public Guid CustomerId { get; set; }
    public string Email { get; set; } = "";
    public string DisplayName { get; set; } = "";
    public DateTime CreatedUtc { get; set; }
    public List<Order> Orders { get; set; } = [];
}

public sealed class Order
{
    public Guid OrderId { get; set; }
    public Guid CustomerId { get; set; }
    public Customer Customer { get; set; } = null!;
    public string Status { get; set; } = "Draft";
    public DateTime CreatedUtc { get; set; }
    public DateTime? PaidUtc { get; set; }
    public byte[] RowVersion { get; set; } = [];
    public List<OrderLine> Lines { get; set; } = [];
}

public sealed class OrderLine
{
    public Guid OrderLineId { get; set; }
    public Guid OrderId { get; set; }
    public Order Order { get; set; } = null!;
    public string Sku { get; set; } = "";
    public int Quantity { get; set; }
    public decimal UnitPrice { get; set; }
}

7.3 ASP.NET Core registration

The common registration pattern is AddDbContext, which registers the context with scoped lifetime. For background jobs, queue consumers, and code that needs to create contexts outside a request scope, AddDbContextFactory can be used to create short-lived contexts explicitly.

builder.Services.AddDbContext<OrderingDbContext>(options =>
{
    options.UseSqlServer(builder.Configuration.GetConnectionString("Ordering"));
    options.EnableDetailedErrors(builder.Environment.IsDevelopment());
});

builder.Services.AddScoped<EfCoreOrderService>();

7.4 Code-first migrations

Code-first migrations are EF Core’s built-in schema versioning mechanism. A migration captures model differences as C# operations and can generate provider-specific SQL. Microsoft’s migrations documentation describes migrations as a way to incrementally apply schema changes to keep the database in sync with the EF Core model while preserving existing data.19

# Create the first migration from the current model.
dotnet ef migrations add InitialOrderingSchema \
  --project src/Ordering.Infrastructure \
  --startup-project src/Ordering.Api \
  --context OrderingDbContext

# Apply pending migrations to the configured database.
dotnet ef database update \
  --project src/Ordering.Infrastructure \
  --startup-project src/Ordering.Api \
  --context OrderingDbContext

# Generate SQL for review/deployment pipelines.
dotnet ef migrations script \
  --idempotent \
  --output artifacts/sql/ordering-migrations.sql

In serious production environments, do not let every application instance mutate schema automatically on startup. Prefer reviewed migration scripts, controlled deployment steps, and CI checks that verify migrations can apply to an empty database and to a recent production-like schema. EF Core migrations are excellent for authoring and versioning changes; deployment policy still belongs to the platform team.

7.5 Change Tracker API

EF Core’s change tracker is the reason ordinary property assignments can become SQL UPDATE statements. Microsoft documents that tracking queries are the default for entity results, and changes to tracked entity instances are persisted by SaveChanges.20 The change tracker maintains entity state, original values, current values, relationship fix-up, and identity resolution within the context.

public sealed class EfCoreOrderService(OrderingDbContext db)
{
    public async Task MarkPaidAsync(Guid orderId, CancellationToken cancellationToken)
    {
        var order = await db.Orders.SingleAsync(x => x.OrderId == orderId, cancellationToken);

        order.Status = "Paid";
        order.PaidUtc = DateTime.UtcNow;

        // Inspect tracked changes before saving.
        var entry = db.Entry(order);
        bool statusChanged = entry.Property(x => x.Status).IsModified;
        EntityState state = entry.State;

        await db.SaveChangesAsync(cancellationToken);
    }
}

The ChangeTracker API is also useful for audit fields, domain-event collection, soft deletes, and diagnostics. Interceptors are often cleaner for cross-cutting policies, but direct inspection is valuable when debugging why EF Core generated an unexpected insert, update, or no-op.

foreach (var entry in db.ChangeTracker.Entries())
{
    Console.WriteLine($"{entry.Metadata.ClrType.Name}: {entry.State}");

    foreach (var property in entry.Properties.Where(p => p.IsModified))
    {
        Console.WriteLine($"  {property.Metadata.Name}: {property.OriginalValue} -> {property.CurrentValue}");
    }
}

For read-only endpoints, use AsNoTracking() to avoid tracking overhead, or AsNoTrackingWithIdentityResolution() when no-tracking queries still need a single instance per key in the result graph. Microsoft documents no-tracking queries as useful for read-only scenarios because they are generally quicker to execute and avoid setting up change-tracking information.20

7.6 Eager loading

Eager loading loads related data as part of the original query with Include and ThenInclude. EF Core’s documentation describes eager loading as a way to specify related data to include in query results and notes that EF Core automatically fixes up navigation properties for entities already loaded in the context.21

var order = await db.Orders
    .Include(o => o.Customer)
    .Include(o => o.Lines)
    .SingleAsync(o => o.OrderId == orderId, cancellationToken);

Eager loading is the safest default for API request handlers because the query shape is visible near the use case. For large collection graphs, consider projections or split queries; Microsoft documents that JOIN-based eager loading can create performance issues such as cartesian explosion and data duplication, and EF Core provides split queries as an alternative.22

var orderDto = await db.Orders
    .Where(o => o.OrderId == orderId)
    .Select(o => new OrderDetailsDto(
        o.OrderId,
        o.Customer.Email,
        o.Lines.Select(l => new OrderLineDto(l.Sku, l.Quantity, l.UnitPrice)).ToList()))
    .AsNoTracking()
    .SingleAsync(cancellationToken);

7.7 Explicit loading

Explicit loading loads a navigation after the root entity is already available. It is useful when a workflow conditionally needs a relationship and you want the additional database call to be obvious in code. EF Core supports explicit loading through Entry(...).Reference(...).LoadAsync() and Entry(...).Collection(...).LoadAsync().23

var order = await db.Orders.SingleAsync(o => o.OrderId == orderId, cancellationToken);

if (includeLines)
{
    await db.Entry(order)
        .Collection(o => o.Lines)
        .LoadAsync(cancellationToken);
}

await db.Entry(order)
    .Reference(o => o.Customer)
    .LoadAsync(cancellationToken);

7.8 Lazy loading

Lazy loading loads related data when a navigation property is accessed. EF Core supports lazy loading with proxies when the Microsoft.EntityFrameworkCore.Proxies package is installed and UseLazyLoadingProxies is enabled; navigation properties must generally be virtual on inheritable entity classes.24

dotnet add package Microsoft.EntityFrameworkCore.Proxies
builder.Services.AddDbContext<OrderingDbContext>(options =>
    options.UseLazyLoadingProxies()
           .UseSqlServer(builder.Configuration.GetConnectionString("Ordering")));

public class Order
{
    public Guid OrderId { get; set; }
    public virtual Customer Customer { get; set; } = null!;
    public virtual ICollection<OrderLine> Lines { get; set; } = new List<OrderLine>();
}

Lazy loading can be convenient in server-rendered applications or small administrative screens, but it is risky in JSON APIs because serialization can accidentally traverse navigation properties and trigger N+1 query storms. Prefer eager loading or projection for API responses unless the team has strict query-count tests and logging.

7.9 EF Core strengths

  • Integrated schema evolution. Code-first migrations keep model and schema changes in one workflow.
  • Productive CRUD. LINQ, DbSet, relationships, tracking, and SaveChanges reduce routine repository code.
  • Strong ASP.NET Core fit. The Microsoft documentation, provider ecosystem, logging integration, and DI patterns are broadly understood by .NET teams.
  • Flexible query shape. EF Core supports tracked entity queries, no-tracking read models, projections, eager loading, explicit loading, lazy loading, raw SQL, transactions, interceptors, and concurrency tokens.

7.10 EF Core pitfalls

  • Assuming LINQ means efficient SQL. Always inspect generated SQL for critical paths.
  • Overusing tracked entities for reads. Use projections and AsNoTracking() for read-heavy endpoints.
  • Hiding database ownership behind repositories. A generic repository over EF Core often removes useful query capabilities while adding little safety.
  • Running migrations casually in production. Generate and review scripts; separate authoring from deployment.
  • Lazy-loading surprises. Unplanned navigation access can turn one request into hundreds of queries.
Use EF Core when the team wants a mainstream, Microsoft-supported ORM with code-first migrations, LINQ-first development, change tracking, relationship mapping, and strong ASP.NET Core integration. Use Dapper beside it when a hot query or report needs hand-authored SQL, and keep schema ownership explicit when mixing EF Core migrations with other tools.

9. Dapper deep dive

Dapper is the smallest abstraction in this article. Its value proposition is explicit: keep SQL in the application while eliminating repetitive parameter creation, command execution, and data-reader mapping. The official GitHub README says Dapper adds extension methods to DbConnection and exposes key APIs such as Execute for insert/update/delete operations and query APIs for reading rows.2 NuGet describes Dapper as a simple micro-ORM for developers who like SQL but dislike ADO.NET boilerplate.8

9.1 Installing Dapper

dotnet add package Dapper
dotnet add package Microsoft.Data.SqlClient

As of the NuGet package page consulted for this article, Dapper is published as version 2.1.79.8 Always pin and review package versions through your own dependency-management process.

9.2 Query object with explicit SQL

using Dapper;

public interface IOrderReadRepository
{
    Task<IReadOnlyList<OrderSummaryDto>> GetRecentOrdersAsync(
        Guid customerId,
        CancellationToken cancellationToken);
}

public sealed class DapperOrderReadRepository(IDbConnectionFactory connectionFactory)
    : IOrderReadRepository
{
    public async Task<IReadOnlyList<OrderSummaryDto>> GetRecentOrdersAsync(
        Guid customerId,
        CancellationToken cancellationToken)
    {
        const string sql = """
        SELECT TOP (25)
            o.OrderId,
            o.CustomerId,
            c.Email AS CustomerEmail,
            o.Status,
            o.CreatedUtc,
            COUNT(ol.OrderLineId) AS LineCount,
            COALESCE(SUM(CONVERT(bigint, ol.Quantity) * ol.UnitPriceCents), 0) AS TotalCents
        FROM dbo.Orders AS o
        INNER JOIN dbo.Customers AS c ON c.CustomerId = o.CustomerId
        LEFT JOIN dbo.OrderLines AS ol ON ol.OrderId = o.OrderId
        WHERE o.CustomerId = @CustomerId
        GROUP BY o.OrderId, o.CustomerId, c.Email, o.Status, o.CreatedUtc
        ORDER BY o.CreatedUtc DESC;
        """;

        await using var connection = await connectionFactory
            .OpenConnectionAsync(cancellationToken)
            .ConfigureAwait(false);

        var command = new CommandDefinition(
            sql,
            new { CustomerId = customerId },
            cancellationToken: cancellationToken);

        var rows = await connection.QueryAsync<OrderSummaryDto>(command)
            .ConfigureAwait(false);

        return rows.AsList();
    }
}

The query is intentionally not hidden. A senior reviewer can reason about indexes, grouping, cardinality, and data shape from a single artifact. That is Dapper’s main strength.

9.3 Commands and transactions

For writes, Dapper forces you to be explicit about what changes happen and in what order. That can be good for correctness because no hidden change tracker is making assumptions; it can also be error-prone because every update statement is hand-authored.

public sealed class DapperOrderCommandRepository(IDbConnectionFactory connectionFactory)
{
    public async Task MarkPaidAsync(Guid orderId, DateTime paidUtc, CancellationToken ct)
    {
        await using var connection = await connectionFactory.OpenConnectionAsync(ct);
        await using var transaction = await connection.BeginTransactionAsync(ct);

        const string sql = """
        UPDATE dbo.Orders
        SET Status = @Status,
            PaidUtc = @PaidUtc
        WHERE OrderId = @OrderId
          AND Status <> @Status;
        """;

        var affected = await connection.ExecuteAsync(new CommandDefinition(
            sql,
            new { OrderId = orderId, Status = "Paid", PaidUtc = paidUtc },
            transaction: transaction,
            cancellationToken: ct));

        if (affected == 0)
        {
            // Could mean already paid or missing. Decide the business semantics deliberately.
        }

        await transaction.CommitAsync(ct);
    }
}

9.4 Multi-mapping

Dapper can map joined rows into multiple objects, but the more object graph behavior you need, the more manual coordination you write. A common pattern is to avoid pretending Dapper is a full ORM and instead return purpose-built read models.

public sealed record OrderLineDto(Guid OrderLineId, string Sku, int Quantity, int UnitPriceCents);
public sealed record OrderDetailsDto(Guid OrderId, string Status, List<OrderLineDto> Lines);

public async Task<OrderDetailsDto?> GetDetailsAsync(Guid orderId, CancellationToken ct)
{
    const string sql = """
    SELECT o.OrderId, o.Status,
           ol.OrderLineId, ol.Sku, ol.Quantity, ol.UnitPriceCents
    FROM dbo.Orders AS o
    LEFT JOIN dbo.OrderLines AS ol ON ol.OrderId = o.OrderId
    WHERE o.OrderId = @OrderId;
    """;

    var lookup = new Dictionary<Guid, OrderDetailsDto>();
    await using var connection = await _connectionFactory.OpenConnectionAsync(ct);

    await connection.QueryAsync<OrderDetailsDto, OrderLineDto, OrderDetailsDto>(
        sql,
        (order, line) =>
        {
            if (!lookup.TryGetValue(order.OrderId, out var existing))
            {
                existing = order with { Lines = new List<OrderLineDto>() };
                lookup.Add(existing.OrderId, existing);
            }

            if (line is not null)
                existing.Lines.Add(line);

            return existing;
        },
        new { OrderId = orderId },
        splitOn: "OrderLineId");

    return lookup.Values.SingleOrDefault();
}

9.5 Dapper strengths

  • SQL transparency. Query plans and database behavior are easier to review.
  • Low abstraction cost. Dapper stays close to ADO.NET and is easy to mix with provider-specific features.
  • DTO-first reads. Query endpoints can project exactly the response shape they need.
  • Incremental adoption. A team can add Dapper to a legacy ADO.NET or EF Core codebase without a full persistence rewrite.

9.6 Dapper pitfalls

  • Stringly typed SQL. Column renames and schema drift fail at runtime unless guarded by tests and database tooling.
  • Manual graph persistence. Cascades, orphan handling, optimistic concurrency, and aggregate updates are yours to implement.
  • SQL scattering. Without conventions, SQL fragments spread through controllers, handlers, and repositories.
  • Provider-specific syntax. Portability is limited by the SQL you write.
Use Dapper when the database schema is treated as a first-class API, SQL review is part of engineering culture, and the application mostly needs query projections or explicit command handlers rather than automatic graph persistence.

10. RepoDB deep dive

RepoDB positions itself between micro-ORMs and full ORMs. Its documentation says it bridges the gaps of micro-ORMs and full ORMs and can be used through a connection string and feature set once initialized.3 The GitHub repository uses the same “hybrid ORM” language.9 In practice, this means RepoDB can reduce the boilerplate for routine CRUD operations while still letting you drop down into SQL-oriented thinking.

10.1 Installing RepoDB

dotnet add package RepoDb
dotnet add package RepoDb.SqlServer
dotnet add package Microsoft.Data.SqlClient

The RepoDb and RepoDb.SqlServer NuGet pages consulted for this article list version 1.13.1 and target .NET 6.0 and .NET Standard 2.0, with newer prerelease versions shown on NuGet.10,11

10.2 Bootstrapping RepoDB

RepoDB requires provider-specific bootstrap. In ASP.NET Core, call bootstrap once during application startup.

using RepoDb;
using RepoDb.SqlServer;

public static class RepoDbBootstrapper
{
    private static int _bootstrapped;

    public static void Bootstrap()
    {
        if (Interlocked.Exchange(ref _bootstrapped, 1) == 1)
            return;

        GlobalConfiguration.Setup().UseSqlServer();
    }
}

// Program.cs
RepoDbBootstrapper.Bootstrap();

A more defensive implementation wraps this in a persistence registration method and tests application startup so a missing bootstrap fails before production traffic arrives.

10.3 Entity mapping

RepoDB can infer mappings by convention, but production systems should make mappings explicit when table names, schemas, or columns diverge from CLR names.

using RepoDb.Attributes;

[Map("dbo.Customers")]
public sealed class CustomerRow
{
    [Primary]
    public Guid CustomerId { get; set; }

    public string Email { get; set; } = "";
    public string DisplayName { get; set; } = "";
    public DateTime CreatedUtc { get; set; }
}

[Map("dbo.Orders")]
public sealed class OrderRow
{
    [Primary]
    public Guid OrderId { get; set; }
    public Guid CustomerId { get; set; }
    public string Status { get; set; } = "";
    public DateTime CreatedUtc { get; set; }
    public DateTime? PaidUtc { get; set; }
}

10.4 CRUD repository

using RepoDb;

public sealed class RepoDbCustomerRepository(IDbConnectionFactory connectionFactory)
{
    public async Task<CustomerRow?> FindByEmailAsync(string email, CancellationToken ct)
    {
        await using var connection = await connectionFactory.OpenConnectionAsync(ct);

        var rows = await connection.QueryAsync<CustomerRow>(
            c => c.Email == email,
            cancellationToken: ct);

        return rows.SingleOrDefault();
    }

    public async Task InsertAsync(CustomerRow customer, CancellationToken ct)
    {
        await using var connection = await connectionFactory.OpenConnectionAsync(ct);
        await connection.InsertAsync(customer, cancellationToken: ct);
    }

    public async Task UpdateDisplayNameAsync(Guid customerId, string displayName, CancellationToken ct)
    {
        await using var connection = await connectionFactory.OpenConnectionAsync(ct);

        await connection.UpdateAsync<CustomerRow>(
            entity: new CustomerRow { CustomerId = customerId, DisplayName = displayName },
            fields: Field.From(nameof(CustomerRow.DisplayName)),
            cancellationToken: ct);
    }
}

The exact overloads available can vary by RepoDB version and provider package, so production code should be compiled against pinned versions and wrapped behind your own repository or gateway interfaces. The architectural point is that RepoDB can express CRUD without hand-writing every statement, but it remains closer to a connection-oriented model than NHibernate.

10.5 Bulk operations

RepoDB is often attractive when applications ingest or synchronize tabular data. For example, a nightly import job may receive 50,000 customer updates and should avoid one statement per row.

public sealed class RepoDbOrderBulkImportRepository(IDbConnectionFactory connectionFactory)
{
    public async Task ImportCustomersAsync(
        IReadOnlyCollection<CustomerRow> customers,
        CancellationToken ct)
    {
        if (customers.Count == 0) return;

        await using var connection = await connectionFactory.OpenConnectionAsync(ct);
        await using var transaction = await connection.BeginTransactionAsync(ct);

        // Use the appropriate RepoDB bulk method for your provider/version.
        // For SQL Server, RepoDb.SqlServer provides provider-specific bulk capabilities.
        await connection.BulkMergeAsync(customers, transaction: transaction, cancellationToken: ct);

        await transaction.CommitAsync(ct);
    }
}

Bulk operations should be benchmarked with realistic data. They interact with indexes, constraints, triggers, lock escalation, transaction-log throughput, tempdb usage, and provider-specific batch behavior. The ORM call is only the top of the stack.

10.6 RepoDB strengths

  • Less repetitive CRUD. Routine insert, update, delete, and query operations are concise.
  • Hybrid escape hatch. You can mix generated operations with explicit SQL-style design.
  • Bulk-friendly posture. It is a natural candidate for import, sync, and administrative data workflows.
  • Moderate learning curve. Less conceptual machinery than a full ORM.

10.7 RepoDB pitfalls

  • Hybrid ambiguity. Teams need rules for when to use generated CRUD, query expressions, raw SQL, or stored procedures.
  • Provider package coupling. SQL Server, PostgreSQL, MySQL, and SQLite behavior must be validated separately.
  • Less mainstream than Dapper or EF Core. Hiring, examples, and ecosystem integrations may require more internal documentation.
  • Domain graphs still need design. RepoDB does not remove the need for aggregate boundaries and consistency rules.
Use RepoDB when you want a structured data-access layer with concise CRUD and bulk operations, but you do not want the full session/change-tracking model of NHibernate or EF Core.

11. NHibernate deep dive

NHibernate is the full ORM in this comparison. The community site describes it as a mature, open-source object-relational mapper for .NET that is actively developed, fully featured, and used in many projects.4 The project documentation page lists the current reference as NHibernate Reference v5.6.5 Hibernate’s own site describes NHibernate as a port of Hibernate ORM to the .NET Framework and independent of the Hibernate project family hosted there.12

NHibernate introduces a different set of concepts: ISessionFactory, ISession, mappings, identifiers, proxies, lazy loading, dirty checking, cascading, flushing, HQL, QueryOver, LINQ providers, first-level cache, optional second-level cache, and transaction management. This power is useful when the domain model is central and persistence should follow object graph semantics. It is a liability when developers treat it like Dapper with extra magic.

11.1 Installing NHibernate

dotnet add package NHibernate
dotnet add package FluentNHibernate
dotnet add package Microsoft.Data.SqlClient

The NuGet page consulted for NHibernate lists version 5.6.1.13 FluentNHibernate is a common mapping alternative; its NuGet page lists version 3.4.1 and targets .NET Core 2.0, .NET Standard 2.0, and .NET Framework 4.6.1 or higher.14

11.2 Session factory registration

Build the ISessionFactory once. It is expensive and thread-safe for application-wide reuse. Open ISession per unit of work, commonly per request or command handler.

using FluentNHibernate.Cfg;
using FluentNHibernate.Cfg.Db;
using NHibernate;

public static class NHibernateRegistration
{
    public static IServiceCollection AddNHibernatePersistence(
        this IServiceCollection services,
        IConfiguration configuration)
    {
        var connectionString = configuration.GetConnectionString("Orders")
            ?? throw new InvalidOperationException("Missing connection string: Orders");

        services.AddSingleton<ISessionFactory>(_ =>
            Fluently.Configure()
                .Database(MsSqlConfiguration.MsSql2012.ConnectionString(connectionString))
                .Mappings(m => m.FluentMappings.AddFromAssemblyOf<OrderMap>())
                .BuildSessionFactory());

        services.AddScoped(sp =>
            sp.GetRequiredService<ISessionFactory>().OpenSession());

        services.AddScoped<NHibernateUnitOfWork>();
        services.AddScoped<NHibernateOrderRepository>();

        return services;
    }
}

11.3 Fluent mapping

using FluentNHibernate.Mapping;

public sealed class OrderMap : ClassMap<Order>
{
    public OrderMap()
    {
        Schema("dbo");
        Table("Orders");
        Id(x => x.OrderId).GeneratedBy.Assigned();

        Map(x => x.CustomerId).Not.Nullable();
        Map(x => x.Status).Length(32).Not.Nullable();
        Map(x => x.CreatedUtc).Not.Nullable();
        Map(x => x.PaidUtc).Nullable();
        Version(x => x.RowVersion).Generated.Always().UnsavedValue(null);

        HasMany(x => x.Lines)
            .KeyColumn("OrderId")
            .Cascade.AllDeleteOrphan()
            .Inverse();
    }
}

public sealed class OrderLineMap : ClassMap<OrderLine>
{
    public OrderLineMap()
    {
        Schema("dbo");
        Table("OrderLines");
        Id(x => x.OrderLineId).GeneratedBy.Assigned();
        Map(x => x.OrderId).Not.Nullable();
        Map(x => x.Sku).Length(80).Not.Nullable();
        Map(x => x.Quantity).Not.Nullable();
        Map(x => x.UnitPriceCents).Not.Nullable();
    }
}

11.4 Repository and unit of work

using NHibernate;
using NHibernate.Linq;

public sealed class NHibernateOrderRepository(ISession session)
{
    public Task<Order?> GetAsync(Guid orderId, CancellationToken ct) =>
        session.Query<Order>()
            .FetchMany(o => o.Lines)
            .SingleOrDefaultAsync(o => o.OrderId == orderId, ct);

    public Task SaveAsync(Order order, CancellationToken ct) =>
        session.SaveOrUpdateAsync(order, ct);
}

public sealed class NHibernateUnitOfWork(ISession session) : IAsyncDisposable
{
    private ITransaction? _transaction;

    public async Task BeginAsync(CancellationToken ct)
    {
        if (_transaction is not null) return;
        _transaction = session.BeginTransaction();
        await Task.CompletedTask;
    }

    public async Task CommitAsync(CancellationToken ct)
    {
        if (_transaction is null)
            throw new InvalidOperationException("No transaction has been started.");

        await session.FlushAsync(ct);
        await _transaction.CommitAsync(ct);
        await _transaction.DisposeAsync();
        _transaction = null;
    }

    public async ValueTask DisposeAsync()
    {
        if (_transaction is not null)
        {
            await _transaction.RollbackAsync();
            await _transaction.DisposeAsync();
        }
    }
}

11.5 Command handler

public sealed class PayOrderHandler(
    NHibernateOrderRepository repository,
    NHibernateUnitOfWork unitOfWork,
    TimeProvider clock)
{
    public async Task HandleAsync(Guid orderId, CancellationToken ct)
    {
        await unitOfWork.BeginAsync(ct);

        var order = await repository.GetAsync(orderId, ct)
            ?? throw new KeyNotFoundException($"Order {orderId} was not found.");

        order.MarkPaid(clock.GetUtcNow().UtcDateTime);

        await repository.SaveAsync(order, ct);
        await unitOfWork.CommitAsync(ct);
    }
}

This is where NHibernate shines: the domain object enforces rules, the session tracks changes, cascades save lines, and a version column can protect against concurrent writes. The code is object-oriented and transactional. The cost is that query behavior is less obvious than raw SQL, especially with lazy relationships and fetch plans.

11.6 Lazy loading, fetch plans, and N+1

Full ORMs can defer loading relationships until accessed. That is convenient inside a cohesive transaction but dangerous in JSON serialization, Razor views, logging, and debugging. A serializer walking an entity graph can trigger unexpected queries, load too much data, or fail after the session is disposed.

Production guidance:

  • Do not return NHibernate entities directly from public API endpoints.
  • Project to response DTOs inside an explicit session boundary.
  • Review generated SQL for high-volume endpoints.
  • Use fetch joins, batch fetching, projections, and query-specific DTOs to avoid N+1 patterns.
  • Keep sessions short and transaction scopes explicit.

11.7 NHibernate strengths

  • Rich domain modeling. Entities, associations, cascades, and aggregate persistence are first-class.
  • Unit of work. The session tracks identity and changes, reducing manual update code.
  • Mature feature surface. NHibernate has long-standing support for mappings, query languages, caching integrations, and extensibility.
  • Legacy compatibility. Many long-lived .NET systems already rely on NHibernate domain models.

11.8 NHibernate pitfalls

  • Hidden SQL. Generated queries must be observed and reviewed.
  • Session misuse. Long-lived sessions cause stale data, memory growth, and surprising flush behavior.
  • Mapping complexity. Mapping drift is a real maintenance cost.
  • Overuse for reads. Complex report queries are often clearer and faster as explicit SQL projections.
Use NHibernate when persistence should follow a rich object model and aggregate consistency is more important than direct SQL authorship at every call site.

12. Transactions, units of work, and consistency

Transactions are where the abstraction differences become operationally important.

12.1 Dapper transaction boundary

Dapper delegates transaction semantics to ADO.NET. You pass the transaction object into each call. The advantage is explicitness; the risk is forgetting to pass it to one command.

await using var connection = await factory.OpenConnectionAsync(ct);
await using var tx = await connection.BeginTransactionAsync(ct);

await connection.ExecuteAsync(insertOrder, order, transaction: tx);
await connection.ExecuteAsync(insertLines, lines, transaction: tx);
await connection.ExecuteAsync(insertOutbox, outboxMessage, transaction: tx);

await tx.CommitAsync(ct);

12.2 RepoDB transaction boundary

RepoDB similarly works well with explicit provider transactions. Generated operations, bulk operations, and raw operations must share the same connection and transaction when atomicity is required.

await using var connection = await factory.OpenConnectionAsync(ct);
await using var tx = await connection.BeginTransactionAsync(ct);

await connection.InsertAsync(orderRow, transaction: tx, cancellationToken: ct);
await connection.InsertAllAsync(lineRows, transaction: tx, cancellationToken: ct);
await connection.InsertAsync(outboxRow, transaction: tx, cancellationToken: ct);

await tx.CommitAsync(ct);

12.3 NHibernate transaction boundary

NHibernate coordinates changes through the session and flushes SQL statements before or during commit depending on flush mode and API usage. The session transaction should encompass the aggregate mutation.

using var tx = session.BeginTransaction();

var order = await session.GetAsync<Order>(orderId, ct);
order.MarkPaid(nowUtc);
await session.SaveAsync(new OutboxMessage(...), ct);

await tx.CommitAsync(ct);

12.4 Outbox pattern

When an ASP.NET Core service changes a database and publishes a message, do not publish directly inside the request transaction unless the infrastructure provides true atomicity across both resources. A safer common pattern is a transactional outbox: write the business change and an outbox row in the same database transaction, then let a background worker publish pending messages and mark them sent.

CREATE TABLE dbo.OutboxMessages (
    OutboxMessageId uniqueidentifier NOT NULL PRIMARY KEY,
    OccurredUtc     datetime2        NOT NULL,
    Type            nvarchar(300)    NOT NULL,
    PayloadJson     nvarchar(max)    NOT NULL,
    PublishedUtc    datetime2        NULL,
    Attempts        int              NOT NULL DEFAULT 0
);

Dapper makes the outbox insert explicit. RepoDB can insert the row with a generated operation. NHibernate can persist an outbox entity in the same session. The architectural rule is the same: the outbox write must share the same transaction as the state change it describes.

13. Performance engineering

ORM choice matters, but performance usually depends on the full path: indexes, query shape, result-set size, network latency, allocations, locks, isolation level, connection pooling, serialization, and retry behavior.

13.1 Connection pooling

Connection pooling is a foundational performance feature. Microsoft’s SQL Server pooling documentation explains that the pooler maintains physical connections per connection configuration and returns a pooled connection when available, instead of opening a new one.6 Therefore, the standard ASP.NET Core pattern is to create, open, use, and dispose logical connections quickly.

// Good: short-lived logical connection; provider pooling handles physical reuse.
await using var connection = await factory.OpenConnectionAsync(ct);
var row = await connection.QuerySingleAsync<OrderSummaryDto>(...);

// Bad: singleton open connection shared by all requests.
// This creates concurrency, lifetime, and failure-recovery problems.

13.2 Buffering versus streaming

Dapper supports buffered and non-buffered queries according to its README.2 Buffered queries materialize the result set immediately; streaming can reduce memory pressure but keeps the connection occupied while rows are consumed. Streaming is appropriate for exports and pipelines, not for casual controller responses where developers might accidentally hold connections during slow clients or downstream processing.

13.3 Projection discipline

Project only the data needed. This is natural with Dapper because SQL is explicit. With RepoDB, avoid loading whole rows for list screens if a projection is enough. With NHibernate, avoid returning entities for read-heavy list endpoints; use LINQ/HQL projections or a separate Dapper read model.

13.4 N+1 query prevention

N+1 happens when one query loads parent rows and then one query per parent loads children. Dapper avoids hidden N+1 but can still suffer from loops that issue repeated queries. RepoDB can suffer from repository methods called in loops. NHibernate can create N+1 through lazy navigation access. The prevention method differs, but the review question is identical: how many SQL statements does this endpoint execute for 1, 10, 100, and 1,000 parent rows?

13.5 Bulk ingestion

For large imports, compare four approaches: row-by-row commands, batched multi-row statements, provider-native bulk copy, and ORM bulk methods. RepoDB’s design often makes it a candidate for the fourth category, while Dapper is commonly paired with explicit bulk APIs or table-valued parameters, and NHibernate is generally not the best first choice for high-volume row ingestion unless carefully batched and flushed.

13.6 Measuring what matters

public sealed class QueryTimingMiddleware(RequestDelegate next, ILogger<QueryTimingMiddleware> logger)
{
    public async Task InvokeAsync(HttpContext context)
    {
        var started = Stopwatch.GetTimestamp();
        await next(context);
        var elapsed = Stopwatch.GetElapsedTime(started);

        if (elapsed > TimeSpan.FromMilliseconds(500))
        {
            logger.LogWarning("Slow request {Method} {Path} took {ElapsedMs} ms",
                context.Request.Method,
                context.Request.Path,
                elapsed.TotalMilliseconds);
        }
    }
}

Use database-side telemetry too: query store, execution plans, wait statistics, lock graphs, deadlock reports, and connection-pool counters. Application timers without database evidence can misdiagnose CPU, network, lock, and plan-cache problems.

14. Security and correctness

14.1 Parameterization

All three libraries can be used safely or unsafely. Dapper examples naturally use anonymous parameter objects; RepoDB generated operations parameterize values; NHibernate parameterizes generated SQL. The danger appears when developers concatenate user input into raw SQL, dynamic sort clauses, filter expressions, or table names.

// Dangerous: never concatenate untrusted input into SQL.
var sql = $"SELECT * FROM dbo.Orders WHERE Status = '{status}'";

// Safer: bind values as parameters.
var rows = await connection.QueryAsync<OrderRow>(
    "SELECT * FROM dbo.Orders WHERE Status = @Status",
    new { Status = status });

14.2 Dynamic sorting and filtering

Parameters protect values, not SQL identifiers. If a client can choose a sort column, map it through an allow-list.

private static string ToOrderBy(string? sort)
{
    return sort switch
    {
        "created" => "o.CreatedUtc DESC",
        "status" => "o.Status ASC, o.CreatedUtc DESC",
        "total" => "TotalCents DESC",
        _ => "o.CreatedUtc DESC"
    };
}

14.3 Over-posting and entity exposure

Do not bind API request bodies directly into persistence entities. That advice applies to Dapper row types, RepoDB mapped classes, and NHibernate entities. Use request DTOs and command objects. Public contracts should not expose persistence-only fields such as row versions, internal status transitions, tenant IDs, soft-delete flags, or audit columns unless intentionally designed.

14.4 Multi-tenancy

For tenant-aware systems, persistence abstractions must enforce tenant filters consistently. Dapper and RepoDB usually need explicit TenantId predicates or repository-level query builders. NHibernate can use filters or interceptors, but those mechanisms still require tests proving tenant isolation. The core security requirement is not “use an ORM feature”; it is “make cross-tenant reads and writes impossible by construction.”

14.5 Optimistic concurrency

With Dapper and RepoDB, optimistic concurrency is usually expressed in the WHERE clause. With NHibernate, version mappings can make concurrency part of the entity mapping. Either way, the application must decide what to do when zero rows are affected or a stale-object exception occurs.

UPDATE dbo.Orders
SET Status = @Status
WHERE OrderId = @OrderId
  AND RowVersion = @ExpectedRowVersion;

15. Testing strategies

Repository tests that mock an ORM often prove little. The high-risk behavior is the interaction between mapping, SQL, provider, transaction, and schema. Favor integration tests that run against a real database engine matching production as closely as practical.

15.1 Test pyramid for persistence

Test typePurposeExamples
Unit testsDomain rules independent of persistence.Order.MarkPaid rejects empty order; price calculation; command validation.
Repository integration testsMapping, SQL, transactions, provider behavior.Dapper query maps all columns; RepoDB bulk merge preserves uniqueness; NHibernate cascade saves lines.

15.2 Example integration test shape

public sealed class DapperOrderReadRepositoryTests : IAsyncLifetime
{
    private readonly TestDatabase _database = new();

    public async Task InitializeAsync()
    {
        await _database.StartAsync();
        await _database.ApplyMigrationsAsync();
        await _database.SeedOrdersAsync();
    }

15.3 SQL snapshot tests

For Dapper-heavy systems, snapshot important SQL files or embedded query strings. For NHibernate-heavy systems, capture generated SQL for critical use cases and assert query counts. For RepoDB, test generated behavior around partial updates, bulk operations, and key handling.

16. Migration and coexistence

Many ASP.NET Core systems do not choose one ORM forever. They evolve. The migration challenge is avoiding inconsistent persistence rules while two or more data-access styles coexist.

16.1 Dapper beside NHibernate

A common architecture uses NHibernate for transactional aggregate writes and Dapper for read models. The rule is that Dapper read models should not mutate NHibernate-owned aggregate tables unless the same invariants, audit rules, optimistic concurrency, and outbox rules are enforced. Reads are easy to split; writes require ownership.

16.2 RepoDB for administrative workflows

RepoDB can be introduced for back-office CRUD or bulk import while Dapper remains the query workhorse. Define which tables RepoDB can update and which require domain services. Otherwise, administrative convenience can bypass business invariants.

16.3 Strangler migration

  1. Inventory current persistence use cases: read projection, simple CRUD, complex aggregate write, bulk import, reporting, background synchronization.
  2. Classify each use case by risk and abstraction need.
  3. Introduce a new persistence implementation behind existing interfaces.
  4. Run dual reads in non-critical paths to compare results.
  5. Move writes only after transaction, concurrency, and audit semantics are reproduced.
  6. Remove unused mappings and queries aggressively after migration.

16.4 Database migrations

Dapper, RepoDB, and NHibernate are not a complete migration strategy by themselves. Teams commonly use tools such as Flyway, Liquibase, DbUp, RoundhousE, or EF Core migrations even when EF Core is not the runtime ORM. The important policy is that schema changes are versioned, reviewed, repeatable, and run in CI before application tests.

16.1 Final recommendations

  • Default to EF Core for mainstream ASP.NET Core CRUD, code-first migrations, relationship mapping, and LINQ-first teams.
  • Default to Dapper for SQL-heavy read APIs, reporting, operational dashboards, and stored-procedure integration.
  • Default to RepoDB for CRUD and bulk workflows where generated operations reduce repetitive code without committing to a full object graph ORM.
  • Default to NHibernate for domains with rich aggregate behavior, relationship cascades, and object-level transactional consistency.
  • Do not expose ORM entities from APIs. Use request and response DTOs.
  • Do not let ORM choice replace database design. Indexes, constraints, isolation levels, migrations, and query review remain mandatory.
  • Instrument generated and hand-written SQL. Every persistence stack needs slow-query visibility and query-count tests for critical paths.
  • Prefer explicit ownership. A table should have a clear write owner, even if several libraries read it.

The most robust ASP.NET Core persistence architectures are boring in the right places: short-lived connections, scoped units of work, explicit transactions, versioned schema migrations, observable SQL, parameterized values, tested mappings, and clear ownership. EF Core, Dapper, RepoDB, and NHibernate can all support that architecture. They simply optimize for different kinds of boring.
