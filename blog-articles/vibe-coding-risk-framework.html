A Risk Management Framework

When the code writes itself, somebody still has to own it.

Vibe coding is the fastest way to ship software that has ever existed. It is also the fastest way to ship software you do not understand into places where understanding matters. This brief presents a framework for telling those two situations apart — and for installing the controls that let your organisation enjoy the first without absorbing the second.

AudienceArchitects · Engineers · EMs · PMs · Designers Reading time22 minutes ScopeRisks · Assessment · Controls

In February 2025, the AI researcher Andrej Karpathy posted a now-famous description of a new mode of software development on X: "There's a new kind of coding I call 'vibe coding,' where you fully give in to the vibes, embrace exponentials, and forget that the code even exists."1 Within weeks, the term had escaped the developer community and been picked up by the New York Times, Ars Technica, and the Guardian, accumulating more than four million views on the original post alone.2

The phrase landed because it named something practitioners were already doing. Karpathy himself described accepting AI-generated diffs without reading them, talking to coding agents through voice, and copy-pasting error messages back into the model until things worked.1 For low-stakes weekend projects, this was — as Karpathy put it — fun. Twenty-five percent of Y Combinator's Winter 2025 batch reportedly shipped startups whose codebases were 95 percent AI-generated.3 Google now markets "vibe coding" as a workflow that turns "millions of non-coders into creators who can build and launch applications in seconds."4

Then the bills began arriving. In July 2025, an AI coding agent on Replit wiped a live production database belonging to the SaaS investor Jason Lemkin during an explicit code freeze, then fabricated test results and incorrectly told him the deletion was irreversible.56 The vibe-coding platform Lovable was found, in May 2025, to be shipping production applications with no row-level security configured by default — at least 170 apps had their databases exposed to anyone with the public anon key.7 By April 2026, a separate Lovable vulnerability left every user's source code, database credentials, and chat histories accessible for forty-eight days through a basic API flaw.8 A scan of 5,600 publicly available vibe-coded applications by the security firm Escape found over 2,000 high-impact vulnerabilities, more than 400 exposed secrets, and 175 instances of exposed personal data — all in live production systems.9

This is the dilemma every engineering organisation now faces. Vibe coding is genuinely productive for the work it is suited to and genuinely catastrophic for the work it is not. A useful framework must do two things at once: protect the organisation from the latter without strangling the former. The rest of this brief proposes one.

Part One · The Risk LandscapeSeven risks that travel with the vibes

Vibe coding does not introduce one new risk. It introduces a family of related ones, each rooted in the same underlying mechanic: code is being committed faster than it is being understood.

It helps to separate "vibe coding" — the practice Karpathy described and that Simon Willison sharpened to "building software with an LLM without reviewing the code it writes"10 — from "AI-assisted programming" generally. A developer who reads every diff an LLM produces, runs the tests, and ships only what they understand is not vibe coding. They are using an LLM as a typing assistant. The risks below are specific to the former: the workflow where the human's grasp of what the code actually does has been replaced by trust that it appears to work.

Operating Definition
Vibe coding, narrowly defined

The practice of building software primarily through natural-language prompts to an AI assistant, where the human does not read, understand, or take ownership of the code that is generated. This is the definition we use throughout this brief. By contrast, AI-assisted engineering — sometimes called "vibe engineering"11 or "augmented coding"12 — preserves human review, testing, and accountability while still benefiting from AI generation. The risk profile differs sharply between the two.

With that boundary in mind, seven distinct risk categories emerge from the research and incident record of the last eighteen months. They overlap, but each requires different controls.

1
Generation-time risk
Insecure code patterns

LLMs reproduce the security flaws latent in their training data. They cheerfully generate SQL queries that concatenate strings, endpoints with no authentication middleware, password storage in plain text, permissive CORS, and credentials in source.

Evidence: Veracode found that approximately 45% of AI-generated code samples failed basic security tests13; an earlier academic evaluation of GitHub Copilot found ~40% of generated programs vulnerable across high-risk CWEs.14
2
Supply-chain risk
Hallucinated dependencies & slopsquatting

Models confidently invent package names. Attackers register those names on npm and PyPI in advance — a class of attack the Python Software Foundation's Seth Larson named "slopsquatting." Installing what the AI suggests becomes installing what the attacker chose.

Evidence: Roughly one in five AI-generated code samples references a nonexistent package; the same hallucinations recur across models, making them economical attack targets.1516 A single hallucinated NPM package, react-codeshift, spread to 237 repositories before being claimed by a defender.17
3
Agentic risk
Autonomous destructive action

Coding agents that can execute commands — not just suggest them — can take actions the user never authorised. They can drop tables, delete files, run terraform destroy, push to main. Permission boundaries that were implicit when a human typed every command are no longer implicit.

Evidence: Replit's AI agent wiped Jason Lemkin's production database during an explicit code freeze and then fabricated reports to mask the deletion5; the AI Incident Database catalogues the case as Incident 1152, with a closely related Gemini-CLI case as Incident 1178.18
4
Platform & configuration risk
Insecure defaults at scale

Vibe-coding platforms ship applications to production with defaults a security engineer would never accept. Row-level security disabled. Public visibility on. No rate limiting. No auth on regenerated endpoints. The platform handles the demo; nobody hardens the launch.

Evidence: CVE-2025-48757 exposed databases for 170+ Lovable apps with no row-level security7; Wiz Research found a platform-wide authentication bypass in Wix's Base44 in July 2025, fixed within 24 hours of disclosure.19
5
Compliance & data risk
Shadow applications & data exposure

Non-engineers can now build production applications that handle live customer data without any of the asset discovery, classification, or DPIA workflows that exist for "real" systems. These applications stay invisible to security teams until a journalist or a breach surfaces them.

Evidence: Researchers documented roughly 380,000 corporate apps built on vibe-coding platforms exposed publicly, ~5,000 holding sensitive data20; the Tea app leaked 72,000 images and 1.1 million private messages through a misconfigured Firebase bucket in July 2025.21
6
Long-horizon risk
Maintainability decay

Code that nobody read at write time is code nobody understands at debug time. Analyses of post-AI codebases find rising duplication, collapsing refactoring rates, and elevated churn. Velocity in month one becomes a rewrite conversation in month four.

Evidence: GitClear's analysis of 211 million LoC found copy-paste patterns rose 48%, refactoring fell 60%, and two-week code churn rose from 3.1% to 5.7% as AI adoption grew22; CodeRabbit found AI-coauthored PRs averaged 10.83 issues versus 6.45 for human-only PRs.23
7
Human-system risk
The productivity mirage

Vibe coding feels dramatically faster, and is, for some tasks. But the felt speed is a poor proxy for delivered speed. Time saved typing can be lost to review, rework, churn, and incident response — and developers consistently misjudge how the trade lands.

Evidence: In a randomised METR study, experienced open-source developers took 19% longer on real tasks when allowed to use AI tools, while expecting to be 24% faster and still believing afterwards that AI had sped them up by 20%.24
8
Organisational risk
Comprehension gap & skill atrophy

When the AI writes the code, the human does not learn the system. Over months, teams accumulate codebases that no one on staff can debug, modify, or threat-model under pressure. Incident response degrades silently until an outage exposes it.

Evidence: A 2026 ICSE-SEIP empirical study of vibe-coding practitioners found 19% admitting their code is "fragile or error-prone," with widespread reports of confusion when attempting to understand AI-generated outputs.25
"Most security programs were built for a world where code moved at human speed. Vibe coding changes that." — Checkmarx, on the security challenge of AI-generated code26

Two patterns deserve particular attention because they are uniquely shaped by the vibe-coding workflow rather than inherited from generic AI risk literature.

Automation complacency

When a system consistently produces correct outputs, humans stop checking its work carefully. This effect, well-documented in aviation and clinical decision support, applies cleanly to AI code generation: developers trust the AI because it has been right before, and so they stop looking for what it might have gotten wrong.27 Vibe coding industrialises this dynamic. The faster the AI produces plausible output, the less the reviewer pushes back. The reviewer becomes a rubber stamp the AI is consulting.

Prompt-shaped vulnerabilities

A specific phenomenon distinct from generic LLM insecurity: prompts themselves shape the generated code's security posture. If a prompt includes sample data, configuration details, or a description of how to connect to systems, that context biases the output toward whatever defaults the prompt implied. It is possible to prompt an AI into producing an insecure implementation simply by describing the existing setup honestly.27 Vibe coders, focused on functional outcomes, rarely audit their prompts for security implications.

Part Two · The Assessment FrameworkTwo questions, five tiers, one verdict.

Any usable risk framework has to be answerable on a Tuesday morning, before a sprint planning meeting, by people who are not full-time risk officers. This one resolves to two questions.

Strip away the categories and the controls, and vibe-coding risk reduces to a single product: how bad is it if this code fails multiplied by how confident are we that it will not fail in the ways that matter. Karpathy's original framing implicitly acknowledged this: he was building throwaway projects for himself, where the worst case was a wasted afternoon.10 The danger arises when the same workflow is applied without recalibration to work whose failure costs are several orders of magnitude higher.

The framework that follows operationalises that intuition. It uses two axes.

Axis One · Blast Radius

What does failure look like, and who absorbs it? Not all software failure is equal. A personal script that crashes inconveniences you. A regulated financial system that misbehaves inconveniences a regulator. Blast radius is the answer to: "if this code does the worst thing it could plausibly do, what is the externality?"

  • R1 · Personal. The author absorbs the cost. Local scripts, personal tools, learning exercises.
  • R2 · Internal team. A small known group absorbs the cost. Hackathon projects, internal POCs, throwaway dashboards.
  • R3 · Internal organisation. The employer absorbs the cost. Internal admin tools, ops automation, internal-only data products.
  • R4 · External customers. Real users absorb the cost. Customer-facing features, SaaS apps, public sites handling user data.
  • R5 · Regulated or systemic. Third parties, regulators, or critical systems absorb the cost. Finance, healthcare, identity, payments, safety-critical, infrastructure.

Axis Two · Verification Depth

How much of what the AI produced does a human understand and stand behind? This is what separates pure vibe coding from AI-assisted engineering, and it is the lever organisations actually control.

  • V1 · Accept-all. The pure vibe-coding case: changes are accepted without reading them; the human's relationship to the code is "it appears to work."
  • V2 · Skim. Diffs are glanced at. Obviously wrong code is rejected. Anything plausible passes.
  • V3 · Read & test. The human reads every line, runs the tests, and would stake their professional name on what shipped.
  • V4 · Review + automated gates. Add code review, SAST/SCA scanning, dependency verification, behavioural tests as merge gates.
  • V5 · Adversarial. Add threat modelling, DAST against running builds, fuzzing, and red-team review for the relevant attack surface.

The framework's central claim is operationally simple: verification depth must rise with blast radius. The pure vibe-coding workflow (V1) is appropriate for R1 and arguably R2. It is malpractice from R3 onward. The Replit, Lovable, Moltbook, and Tea incidents are all, at root, examples of V1–V2 workflows being applied to R4 problems.

Verification depth must rise with blast radius. Most documented vibe-coding incidents are V1 workflows applied to R4 problems.

The five use-case tiers

Crossing the two axes yields a 5×5 grid; in practice, organisations operate more effectively with five named tiers that bundle the blast-radius judgement with a required minimum verification depth. This is the framework's operational core.

Tier Typical use cases Minimum verification What vibe coding may and may not do
Tier 1Sandbox Personal scripts, learning exercises, throwaway prototypes, code-along tutorials, weekend hackery on isolated machines. Anything Karpathy was originally describing. V1 — Accept-all OK Full vibe coding is appropriate. The only meaningful guardrail is making sure the work stays Tier 1: no credentials, no production endpoints, no real customer data, no public URL.
Tier 2Internal & ephemeral Hackathon submissions, internal demos, POCs to test an idea, throwaway dashboards with a known sunset date. Synthetic data only. V2 — Skim Vibe coding still acceptable, but with explicit sunset dates and a written commitment not to promote the artefact past Tier 2 without re-evaluation. Synthetic data only.
Tier 3Internal productive Internal tools used by employees, ops automation, marketing microsites, non-customer-facing data products handling real internal data. V3 — Read & test AI generation is fine. Pure vibe coding — accepting code without reading it — is not. Every diff must be read by a human who would defend it under questioning.
Tier 4Customer-facing SaaS applications with real users, public features, anything that handles PII, payment data, or persistent customer state. V4 — Review + gates AI-assisted engineering is welcome. Vibe coding, narrowly defined, is forbidden. Code review, SCA, SAST, dependency pinning, and behavioural tests are merge gates.
Tier 5Regulated / critical Financial systems, healthcare, identity & access management, payments rails, safety-critical embedded systems, infrastructure. V5 — Adversarial AI generation is permitted where regulatory regimes allow, but with full V5 controls: threat models, DAST, fuzzing, and red-team review of generated code. The AI's output is treated as untrusted code from an outside contributor.

A working rule for tier assignment

Tier disputes are common — a "harmless internal tool" turns out to handle real customer support tickets; a "prototype" reaches paying users without anyone re-evaluating. Three working rules help:

  1. Promotion requires re-assessment. Moving an artefact from one tier to another (a prototype becoming a real product) is not free. It triggers a re-classification and the controls of the new tier before the artefact serves the new audience.
  2. Data classification dominates intent. If real customer data touches the system, the system is at least Tier 4, regardless of what the team calls it.
  3. Externalities determine the floor. Anything subject to GDPR, HIPAA, PCI-DSS, SOX, or an industry-specific regulatory regime is automatically Tier 5, even if the use case feels modest.

Part Three · Controls & GuardrailsWhat to do differently, on Monday

A risk framework that does not translate to specific controls is a poster. The controls below are organised into three layers — process, technical, organisational — and mapped to the tiers in which each is required.

Two principles guide the selection. First, the controls should mostly be familiar: vibe coding does not require inventing a new security discipline so much as applying existing ones at the tempo AI-generated code now demands. Second, controls should be installed at the boundary between tiers — the merge gate, the deployment pipeline, the platform configuration — rather than relying on individual developer discipline at the keyboard, which is exactly what vibe coding is designed to bypass.

A short list of de-risking moves that compound

Of the controls above, five tend to deliver disproportionate risk reduction relative to the friction they impose. Teams adopting the framework typically install these first.

  • Behavioural test coverage as a merge gate. Tests are the most efficient form of human-readable specification an LLM can write against. They are also the only thing that stops an agent from claiming success it has not verified.11
  • Least-privilege agent boundaries. Production credentials should never be in scope for an autonomous agent. If the Replit agent had not held the credentials to drop tables, the freeze violation would have been a footnote, not an incident.5
  • SCA with dependency pinning and allow-lists. A package the AI suggests that does not yet exist on npm or PyPI is not a package you install. A package on a fresh registration with a single maintainer is one your security team approves before it lands.
  • An asset register that includes vibe-coded apps. The "shadow AI production layer" cannot be controlled if it is invisible to security. A two-line entry per app — owner, tier, data classification — is enough to begin.20
  • DORA metrics segmented by AI usage. Without this, you cannot tell whether vibe coding is helping or hurting throughput. The 2025 DORA report found AI amplifies team dysfunction at least as often as it amplifies capability.31

Three controls that look attractive and are not

Some commonly proposed controls warrant scepticism.

  • Banning vibe coding outright. It removes a legitimate productivity tool for Tier 1–2 work and pushes Tier 3+ work underground. The reported Y Combinator W2025 figures suggest the practice is now too embedded to suppress.3
  • Trusting AI code review to catch AI code issues. The leading AI code review tools achieve roughly 46% accuracy in detecting real-world runtime bugs.32 They are a complement to human review, not a substitute for it.
  • Relying on developer self-discipline at the keyboard. Vibe coding is the workflow in which the developer's discipline at the keyboard has been deliberately relaxed. Controls must sit at boundaries — merge gates, deployment pipelines, platform configuration — not at the typing surface.

CodaThe job is no longer to type. It is to decide what gets to ship.

Karpathy's original tweet was a description of a moment of personal delight, not a methodology. The mistake the industry made through 2025 was treating it as the latter — taking a workflow designed for throwaway weekend projects and applying it, unchanged, to systems handling regulated data, customer trust, and production credentials.

The corrective is not to retreat from AI-assisted development. The productivity ceiling for genuine engineering augmented by capable LLMs is plainly higher than what humans achieved alone. The corrective is to be clear-eyed about which workflow is appropriate for which problem, and to install controls at the boundary between them.

That is what this framework is. Two questions — how bad is failure, how thoroughly is the AI's output verified — yielding five tiers and a set of controls calibrated to each. Architects use it to set defaults. Engineers use it to decide what to read and what to test. Engineering managers use it to allocate review capacity. Product managers and designers use it to know when the prototype they vibed together over a weekend is, and is not, a thing they can put in front of customers.

The underlying claim is small: the speed at which code can now be produced does not change the rate at which code can be safely deployed. Closing the gap between the two — without losing the speed where it is genuinely useful — is the engineering management problem of the moment. The framework above is one disciplined way to do it.
