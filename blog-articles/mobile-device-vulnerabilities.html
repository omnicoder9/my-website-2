Mobile Device Vulnerabilities

A pocket-sized computer carries microphones, cameras, payment credentials, location history, and a permanent radio link to the outside world. This is a field guide to how that attack surface fails — at the device, the application, the radio, and the data layers — with the history, the mechanics, the code, and the citations behind each class of weakness.

Scope & threat model

The smartphone collapsed a dozen separate devices — phone, camera, wallet, map, key, and terminal — into one always-connected object, and in doing so it inherited every one of their threat models at once. Industry surveys now find that roughly four in five organizations consider mobile devices critical to their operations, yet many still protect them with little more than the operating system's defaults.3 The result is a uniquely rich target: a single compromise can yield a microphone, a location beacon, a set of synced credentials, and a tap-to-pay instrument.

This guide organizes mobile weaknesses into four layers, mirroring how an attacker actually moves: the device (the trust boundary of the OS itself), the application (the code running inside it), the connectivity stack (the radios that reach the outside world), and the data at rest on the hardware. Two reference frameworks anchor the discussion throughout — the OWASP Mobile Top 10, refreshed in 2024 for the first time since 2016,1 and NIST Special Publication 800-124 Revision 2, the U.S. government's enterprise mobility guidance.2 Where a vulnerability has a public exploit, a CVE, or a named campaign, it is cited; where a defense exists, it is shown in code.

⚑ On intent

Every attack below is documented publicly and discussed here to help defenders reason about exposure. The runnable code in this article is deliberately limited to detection and hardening — root/jailbreak checks, secure storage, permission hygiene. Offensive mechanics are described conceptually, not weaponized.

Part I — The device layerDevice security issues

Modern mobile operating systems are built on a chain of trust: a hardware root vouches for the bootloader, which vouches for the kernel, which enforces a per-application sandbox. The device-layer vulnerabilities in this section all share a single theme — they break, bypass, or fail to maintain that chain. When the chain holds, an app is a prisoner of its sandbox; when it breaks, the sandbox becomes a suggestion.

I.1 Rooting

Rooting is the act of obtaining superuser (root) privileges on an Android device, removing the restrictions that normally confine apps and the user to a limited account. Mechanically, modern rooting does not edit the read-only system partition; instead, tools such as Magisk patch the boot image to inject a privilege-escalation layer "systemlessly," which preserves over-the-air update compatibility and was designed to evade Google's SafetyNet and Play Integrity attestation checks.52 Once rooted, a device can run a su binary that grants any requesting process kernel-level power.

The security cost is structural. Research presented in 2025 found that popular rooting frameworks — Magisk, KernelSU, APatch, SKRoot — hook deep into the Android kernel, bypass standard security controls, and expose privileged interfaces to user-space apps; their authentication mechanisms are frequently fragile, especially in early development, leaving openings for an ordinary app to escalate to root.4 That is not hypothetical: CVE-2024-48336 ("MagiskEoP") let a locally installed app with no special privileges silently execute arbitrary code as root inside the Magisk app and escalate, by shipping a crafted package that Magisk loaded without verification — no user interaction required.5 Worse, users who root then disable security so banking and enterprise apps will still run, and who switch to newer, "less-detectable" root tools to dodge those apps' checks, often end up exposing themselves to exactly the kind of compromise the checks were meant to prevent.4

◆ Defense

Do client-side root checks for telemetry and friction, but make trust decisions server-side using hardware-backed attestation (Play Integrity on Android, DeviceCheck/App Attest on iOS). Never store a secret that matters in a way that assumes the sandbox is intact on a rooted device — assume it is not.

I.2 Jailbreaking

Jailbreaking is the iOS analogue of rooting: a privilege escalation that removes Apple's software restrictions so the user can sideload unsigned code, install tweaks, and access the filesystem. Crucially, the act bypasses the application sandbox, which inherently reduces security and makes the device more vulnerable to malware from outside the curated App Store.6 In the United States, jailbreaking phones has been legal under a DMCA exemption since 2010,6 but legality is orthogonal to safety.

The most consequential jailbreak in recent memory is checkra1n, built on checkm8 — a BootROM exploit released in 2019 by the researcher axi0mX. Because the BootROM is read-only memory burned into the silicon at manufacture, checkm8 is unpatchable by software update on the affected A5-through-A11 devices, and so it works against every iOS version those chips can run.78 The exploit is "tethered" in the sense that a reboot un-jailbreaks the device — but as Lookout's researchers noted at release, an attacker who jailbreaks a phone they briefly control can sideload a malicious app to gain a persistent foothold that survives the reboot.9 That is the real risk model for most people: not self-jailbreaking, but an adversary with momentary physical access ("evil maid") using an unpatchable hardware flaw against a high-value target.

I.3 Outdated OS versions

The third device-layer failure is mundane and devastating: devices that never receive the patch. Every month Google and Apple publish security bulletins that fix dozens of vulnerabilities, many rated critical for remote code execution or privilege escalation. The June 2026 Android bulletin, for instance, patched a critical Framework flaw (CVE-2025-48595) allowing remote privilege escalation with no user interaction, already under limited targeted exploitation.12 A device that does not install that update remains exploitable indefinitely.

Android's structural problem is fragmentation. Multiple OS versions coexist in the market alongside thousands of vendor-customized builds (Samsung, etc.), and updates roll out slowly and unevenly, leaving a wide window for adversaries to exploit known flaws.10 That same fragmentation creates subtler bugs: the 2014 "Pileup" research at IEEE S&P showed that the OS-update process itself can be abused — a low-privilege app installed on an old version can silently inherit dangerous permissions and capabilities when the device is upgraded, because the update logic must reconcile old and new permission models without breaking existing apps.10 The fix is operational, not clever: keep the OS and apps current, and retire hardware once it falls out of the vendor's support window, because an end-of-life device receives no patches at all.211

▸ Why this matters: the long tail

When Armis disclosed the BlueBorne Bluetooth flaws (covered in §III.1), it estimated that around 40% of affected devices — over two billion — would likely never be patched because they had reached end-of-life or would not receive vendor updates.25 Outdated OS versions do not merely delay safety; for much of the installed base, they permanently deny it.

Part II — The application layerMobile application risks

If the device layer is about the OS keeping its promises, the application layer is about the code running on top of it — code the user installs, often without reading what it asks for. Here the threat is not a broken sandbox but a sandbox that was handed the keys voluntarily.

II.1 Insecure app permissions

Both Android and iOS gate access to sensitive resources — location, microphone, camera, contacts, SMS — behind a permission model. The model is only as strong as the choices made on either side of it. On the developer side, the long-standing problem is over-privilege: empirical studies have found that roughly one-third of Android apps request more permissions than they actually use, widening the attack surface for no functional benefit and giving any malware that compromises the app a larger set of capabilities to abuse.13

The original Android model granted all requested permissions at install time, an all-or-nothing dialog that users routinely accepted to get the app. Android 6.0 (API 23) overhauled this so that dangerous permissions are requested at runtime, in context, letting users make informed decisions.13 Yet the gap persists: every permission an app holds is classified by Android as normal, dangerous, or signature, and research on remote-access trojan behavior found that the exact permission sets used to stealthily activate a microphone or camera are the same ones requested by benign apps with millions of installs — so the permission alone cannot distinguish friend from foe.14

II.2 Malicious apps

Some apps are not over-privileged by accident — they are weaponized by design. Mobile malware spans a spectrum from petty fleeceware to nation-state spyware.

At the high-volume end sits Joker, a malware family active since 2017 that hides inside otherwise functional apps — wallpapers, photo editors, scanners — and commits billing fraud by silently subscribing victims to premium services while intercepting the SMS confirmation codes, a scheme researchers call fleeceware.16 Joker has proven extraordinarily persistent on official stores: Zscaler's ThreatLabz found over 200 malicious apps on Google Play between June 2023 and April 2024 totaling nearly eight million installs, with Joker accounting for the largest single share at roughly 38%.15 Kaspersky has estimated that around 70% of Android users have encountered subscription trojans of this kind at some point.18

More dangerous are banking trojans such as Anatsa (also "TeaBot"), which masquerade as innocuous utilities — PDF readers, QR scanners, phone cleaners — to slip past store review, then deploy overlay attacks to steal banking credentials and abuse accessibility permissions to seize control of the device. One 2024 wave saw more than 90 malicious apps distribute Anatsa via Google Play, downloaded over 5.5 million times.17

At the apex is commercial spyware. NSO Group's Pegasus is the canonical example, repeatedly delivered through zero-click exploits that require no victim interaction at all. The FORCEDENTRY chain (CVE-2021-30860), discovered by Citizen Lab on the phone of a Saudi activist, exploited Apple's image-rendering library through a malicious iMessage attachment and bypassed the "BlastDoor" sandbox Apple had built specifically to stop such attacks.19 Google's Project Zero, after dissecting it, called it one of the most technically sophisticated exploits they had ever seen.20 The pattern recurred: in 2023 Citizen Lab caught the BLASTPASS chain compromising fully up-to-date iPhones through malicious PassKit images, again with zero clicks.21

⚠ Zero-click reality

Conventional advice — "don't tap suspicious links" — is irrelevant against a zero-click chain. The victim does nothing; a message arrives and the phone is compromised before any notification appears. The only durable defenses are rapid patching, attack-surface reduction (e.g. iOS Lockdown Mode, which disables risky message-attachment processing), and accepting that for high-risk targets, the most sensitive conversations do not belong on a general-purpose handset.

II.3 Untrusted app stores

The official stores are imperfect, but they apply review, signing, and rapid takedown. Untrusted third-party stores and sideloaded APKs apply none of this consistently, and they are where malware that is removed from Google Play tends to resurface. When Joker apps are reported and pulled from the Play Store, copies frequently remain available on third-party stores;16 a single Joker variant on Huawei's AppGallery infected more than 500,000 users.22 Trojanized "mod" versions of popular apps — a doctored WhatsApp, say — are a recurring delivery vehicle on alternative stores, bundling a payload into software the user believes is legitimate.18

The structural risk is twofold: sideloading requires the user to disable a platform safety control ("install unknown apps"), and the resulting binary has no trustworthy provenance. As regulators force the platforms to open to alternative distribution, this attack surface grows rather than shrinks, making on-device protections — Google Play Protect, app sandboxing, runtime permissions — more load-bearing than ever.

◆ Defense

Keep "install unknown apps" disabled by default; install only from the platform store or a vetted enterprise catalog; verify the developer, reviews, and requested permissions before installing; and in managed fleets, use Mobile Threat Defense to flag sideloaded or repackaged binaries.2

Part III — The radio layerConnectivity threats

A handset is a bundle of radios — Bluetooth, NFC, Wi-Fi, and cellular — each speaking a protocol designed under different assumptions, in different decades, with different security maturity. Connectivity attacks are attractive because they often need proximity rather than installed code, and the most dangerous ones exploit flaws in the standards themselves, which means a single design weakness translates into billions of vulnerable devices.26

III.1 Bluetooth attacks

Bluetooth's combination of ubiquity, always-on discoverability, and a sprawling protocol stack has made it a perennial target. The landmark event was BlueBorne (2017), a set of eight vulnerabilities disclosed by Armis affecting the Bluetooth implementations in Android, iOS, Windows, and Linux. Its danger lay in the preconditions: the attack required no pairing, no discoverable mode, and no user interaction — an attacker in radio range who knew the target's OS could achieve remote code execution or a man-in-the-middle position.24 Armis estimated BlueBorne could affect essentially all of the roughly 8.2 billion Bluetooth devices then in existence, with about 5.3 billion at direct risk.24 The flaws were tracked across platforms — including CVE-2017-0781 and CVE-2017-0782 (Android RCE) and CVE-2017-14315 (Apple LEAP RCE) — and patched in vendors' September 2017 updates.2523

Where BlueBorne was an implementation bug, the next wave struck the specification. The KNOB attack (Key Negotiation of Bluetooth, CVE-2019-9506), from Antonioli, Tippenhauer and Rasmussen at USENIX Security 2019, exploited the fact that the standard permits an encryption key with as little as one byte of entropy. A man-in-the-middle silently rewrites the entropy proposal during negotiation down to that minimum; the resulting key is trivially brute-forced, and because the flaw is in the protocol, it affected virtually all devices across Bluetooth versions 1.0 to 5.1.2726

KNOB was followed by BIAS (Bluetooth Impersonation AttackS, CVE-2020-10135) from EPFL in 2020, which abuses weaknesses in the authentication procedure to impersonate a previously paired device without possessing the link key. Chained together, BIAS and KNOB let an attacker impersonate a device, complete authentication, negotiate a low-entropy session key, and brute-force it — a full break of the paired channel.28 The 2020 BLURtooth work then targeted Cross-Transport Key Derivation, the mechanism that shares keys between Bluetooth Classic and Bluetooth Low Energy, allowing key overwrites across transports.29

◆ Defense

Turn Bluetooth off when unused (it shrinks the proximity attack surface to zero); keep firmware patched, since most of these flaws are fixed in updates; prefer devices that enforce a minimum encryption key length; and treat Bluetooth as an untrusted channel for anything sensitive.

III.2 NFC attacks

Near-Field Communication is often assumed safe because of its centimeter range — and that intuition is exactly what attackers exploit. The headline NFC threat is the relay attack: rather than break NFC's cryptography, the attacker simply extends its range over the internet. Academic work from Royal Holloway (Francis et al.) demonstrated a practical peer-to-peer relay using ordinary NFC phones and publicly available APIs, defeating the short-range assumption without touching secure memory or code signing.44

That research has now been industrialized into fraud. In 2020, students at TU Darmstadt released NFCGate, an open-source tool to capture, analyze, and relay NFC traffic; criminals weaponized its principles into malware such as NGate, which relays NFC data from a victim's payment card — via the victim's own phone — to an attacker waiting at an ATM.42 The current incarnation, dubbed Ghost Tap, links a stolen card to a mobile wallet on a mule's device (using malware-harvested card data and intercepted one-time passwords) and relays the tap-to-pay signal to point-of-sale terminals anywhere in the world, so the legitimate cardholder and card are nowhere near the transaction.41 Because each transaction is cryptographically valid, it slips past conventional fraud defenses; Group-IB tracked over US$355,000 in illegitimate transactions from a single point-of-sale vendor in one such operation.43

▸ The detection signal: latency

The physics that enables relay attacks also betrays them. Relaying a signal across the internet introduces measurable round-trip latency that a co-located tap never would. The proposed defense is for ATM and POS terminals to enforce strict distance-bounding or time-based checks, rejecting transactions whose timing implies the card is not physically present.41

III.3 Wi-Fi attacks

Wi-Fi attacks split into two families: breaking the encryption that protects a session, and tricking a device onto a network the attacker controls.

The defining encryption break is KRACK (Key Reinstallation Attack), disclosed by Mathy Vanhoef and Frank Piessens at ACM CCS 2017. It exploits the WPA2 four-way handshake by forcing a client to reinstall an already-in-use key, resetting the associated nonce and packet counters and enabling an attacker in range to decrypt, replay, and in some cases inject traffic.30 Critically, KRACK does not recover the Wi-Fi password — so changing it does not help — and it was tracked as ten distinct CVEs (CVE-2017-13077 through -13088).31 The attack was especially devastating against Android 6.0 and Linux, whose wpa_supplicant could be coerced into installing an all-zero encryption key, making decryption trivial.31 Vanhoef returned in 2021 with FragAttacks (fragmentation and aggregation attacks), a set of twelve flaws — several of them design defects in the Wi-Fi standard dating back to 1997 — that allow plaintext frame injection and data exfiltration on essentially every Wi-Fi device.32

The deception family is led by the evil twin: a rogue access point that clones a legitimate network's name (SSID) and, where the password is public, its credentials. Devices that have joined the real network before will often reconnect automatically — especially if the fake broadcasts a stronger signal — placing the attacker in a position to intercept traffic, harvest credentials, or inject content.33 The technique needs no protocol break at all; it exploits the device's eagerness to reconnect to a familiar name.

◆ Defense

Patch clients and access points (both endpoints are affected by KRACK and FragAttacks); prefer WPA3 where available; never transmit anything sensitive over an open or unverified network without a VPN; and treat HTTPS as a second line of defense, not a guarantee — connection downgrades exist. On managed devices, disable auto-join for open networks to blunt evil-twin reconnection.

III.4 Cellular interception

The cellular network is the one radio a phone can never turn off and still function — and parts of its signaling were designed in the 1970s and 1980s, before adversarial access was conceivable. Two attack families dominate.

Fake base stations (IMSI catchers / "Stingrays")

An IMSI catcher is a device that masquerades as a legitimate cell tower, exploiting the fact that phones connect to the strongest available signal and that, in the 2G/GSM era, base stations were never required to authenticate themselves to phones.37 Once a device connects, the catcher can force it to transmit its IMSI (the SIM's unique identifier) for tracking, and pinpoint its location.34 The pivotal technique is the 2G downgrade attack: even where 4G/5G is dominant, most phones retain 2G for backward compatibility, and an attacker can jam or coerce a device down onto GSM, where there is no mutual authentication and the A5/1 cipher was cracked in 2009 (and A5/0 provides no encryption at all), enabling full interception of calls and texts.36 3G and later added mutual authentication that closes the basic fake-tower attack, but sophisticated attacks can still downgrade those connections to legacy services that lack it.35 Once a niche government tool, this capability is now within reach of anyone with a software-defined radio and open-source base-station software such as OpenBTS;37 in the U.S. alone, the ACLU has identified at least 75 agencies across 27 states that own cell-site simulators.36

Signaling network attacks (SS7)

The second family attacks the network's backbone rather than the air link. Signaling System 7 (SS7) is the protocol suite that lets carriers worldwide route calls, deliver SMS, and manage roaming. It was built for a closed club of trusted operators and has essentially no authentication on its core services — so anyone with signaling access can, in effect, do as they please.38 In December 2014, German researchers Tobias Engel and Karsten Nohl demonstrated at the Chaos Communication Congress that SS7 weaknesses allow an attacker who knows only a target's phone number to intercept calls and texts and track location globally.38 The point was made unforgettably in 2016 when, for a CBS 60 Minutes segment, Nohl tracked and listened to the phone of U.S. Congressman Ted Lieu using only his number.39 Mechanically, an attacker uses commands such as Send Routing Information (SRI) and Provide Subscriber Information (PSI) to query a target's home register for their IMSI and current cell, and even modern 4G/5G networks using the Diameter protocol remain exposed where they interoperate with legacy SS7-based 2G/3G networks.40

⚠ Why SMS 2FA is the weak link

Both attack families converge on the same lesson: SMS is not a confidential channel. An IMSI catcher on the air link or an SS7 attack on the backbone can intercept the one-time passwords sent by text — which is precisely why SMS-based two-factor authentication fails in an estimated 90% of phone thefts and is considered the weakest of the common second factors.49 Prefer authenticator apps or hardware security keys.

◆ Defense

Disable 2G where the platform allows it (Android 12+ exposes a "Allow 2G" toggle that removes the legacy-GSM fallback IMSI catchers depend on);36 use end-to-end encrypted messaging and calling so the transport network never sees plaintext; move 2FA off SMS; and for high-risk individuals, assume location and metadata can leak at the network layer regardless of the handset.

Part IV — The data layerData protection issues

The final layer is the data itself, sitting in flash storage. The threats here are less about exotic exploits and more about what happens when the physical device — and everything on it — leaves the owner's control, or when the software stores secrets carelessly.

IV.1 Lost or stolen devices

The oldest mobile threat is also the most common: the phone simply goes missing. Estimates have long put smartphone losses on the order of 70 million per year, with only about 7% recovered.48 The hardware is rarely the real loss. A 2026 analysis found that one in three theft victims lose sensitive personal data, in large part because devices are often handed over unlocked and remote wipes are rare — only about 25% of users remotely wipe a stolen device.49 Stolen phones increasingly feed organized crime rather than local resale, used to bypass security features or reprogrammed and resold, and tied into large fraud operations.3

The defenses are mature but underused: a strong passcode plus biometrics, full-device encryption (covered next), and remote lock/wipe via Find My iPhone or Android's Find My Device. The catch is that these are most effective when configured before loss and when the device is locked at the moment of theft — an unlocked phone snatched mid-use defeats much of the protection stack.49

IV.2 Lack of encryption

Encryption is what turns a stolen phone from a data breach into an expensive paperweight. Modern iOS and Android both encrypt user data at rest by default, but the details — and the limits — matter.

On iOS, the feature is Data Protection, anchored in the Secure Enclave, a dedicated security coprocessor that generates and holds cryptographic keys in isolation; neither iOS, apps, nor even Apple can extract them.50 Each file gets a unique per-file key from a hardware AES engine: on A14/M1-class and later devices the cipher is AES-256 in XTS mode, with the per-file key passed through a NIST SP 800-108 key-derivation function.45 The system assigns each file a protection class (A through D), and this is the security-critical knob. Class A keys are evicted from memory when the device locks, so that data is unreadable until the user re-authenticates; but Class C — the default for most third-party app data — keeps its key available after the first unlock and does not evict it on lock.46 Android's equivalent, File-Based Encryption (FBE), similarly wraps file keys with hardware-bound keys (using AES-256-GCM) and supports cryptographic separation per user.47

⚑ The "After First Unlock" weak point

The practical limit of mobile encryption is operational state. A phone that is powered off (or that has not been unlocked since boot — "Before First Unlock") holds its most sensitive keys outside memory and resists forensic extraction. But once unlocked even once, an "After First Unlock" device keeps Class C / default-class keys resident in memory, which is exactly the state mobile forensic extraction tools target.46 Strong encryption is sacrificed for convenience the instant the device is unlocked — so for at-rest protection that actually holds, power the device fully off.

IV.3 Improper data storage

Platform encryption protects data at rest from outside the OS; it does nothing about an app that stores secrets carelessly inside its own sandbox. This is Insecure Data Storage — M9 in the OWASP Mobile Top 10 — and it is one of the most common real-world findings.51 The classic mistakes are storing authentication tokens, passwords, or PII in plaintext using Android SharedPreferences or iOS UserDefaults, both of which write unencrypted files into the app container. On a rooted or jailbroken device — or via a backup, or a forensic extraction in the "After First Unlock" state — those files are readable.

◆ Defense

Store credentials and keys in the platform keystore (Android Keystore / iOS Keychain), never in plaintext preferences; choose the most restrictive accessibility class the use case allows (e.g. …ThisDeviceOnly to block backup exfiltration); minimize what is persisted at all; and exclude sensitive files from automatic cloud backups. Validate against the OWASP MASVS storage requirements.51

SynthesisPulling the layers together

No single control secures a handset, because the failures span four independent layers. But the defenses rhyme, and a short, layered checklist closes the majority of the exposure described above.

LayerPrimary threatsHighest-leverage defenses
Device Rooting, jailbreaking, unpatched OS, end-of-life hardware Patch promptly; retire unsupported devices; avoid rooting/jailbreaking; use hardware attestation for trust decisions
Application Over-privilege, fleeceware, banking trojans, zero-click spyware Least-privilege permissions; official stores only; review apps before install; Lockdown Mode / MTD for high-risk users
Connectivity BlueBorne/KNOB/BIAS, NFC relay, KRACK/evil twin, IMSI catchers, SS7 Disable unused radios; patch firmware; VPN on untrusted Wi-Fi; disable 2G; move 2FA off SMS; E2E-encrypted comms
Data Loss/theft, weak at-rest protection, insecure local storage Strong passcode + biometrics; remote lock/wipe; power off to protect keys; keystore-backed secret storage

The throughline is a posture, not a product: assume the device can be lost, the radio can be hostile, the store can be wrong, and the OS can be a version behind — then make each layer fail safely rather than catastrophically. For enterprise programs, NIST SP 800-124 Rev. 2 formalizes this across the device lifecycle, pairing centralized management (EMM) with Mobile Threat Defense and clear policies for deployment, use, and disposal.2 For developers, the OWASP Mobile Top 10 and the accompanying MASVS provide the verification baseline.1 For everyone else, the unglamorous trio — update, lock, and encrypt — still defeats the overwhelming majority of attacks that ever reach a real phone.
