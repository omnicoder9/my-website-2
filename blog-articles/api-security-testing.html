API Security
Testing

Modern backend services live and die at the seam between authentication — proving who you are — and authorization — proving what you may touch. Almost every catastrophic API breach is a failure of the second, dressed up as a success of the first. This manual is about finding that seam before someone else does.

APIs are no longer a layer of the application — they are the application. The browser, the mobile app, the partner integration, and increasingly the autonomous agent all speak to the same JSON endpoints, and those endpoints expose business logic and sensitive data directly to anyone who can form an HTTP request. That exposure is the point of an API, and it is also the problem. Security testing for APIs is therefore less about exotic memory-corruption and more about a disciplined interrogation of trust: which identity is asserted, which object is requested, and whether the server ever bothers to check that the two belong together.

The numbers describe an industry that has not caught up to its own architecture. In one widely cited survey, the overwhelming majority of organisations reported experiencing at least one API security incident, and broken authorization alone accounts for something on the order of forty percent of observed API attacks.1 The financial weight behind those incidents is not trivial: industry reporting puts the average cost of a breach in the multiple-millions of dollars, and API-driven failures sit squarely in that distribution.2 The vulnerability that tops the charts is also the most boring to exploit — frequently nothing more than changing a number in a URL.3

Consider the pattern in the wild. Optus, one of Australia's largest carriers, leaked the records of roughly ten million customers — including passport and licence numbers — because an unauthenticated endpoint exposed customer data keyed by easily guessable identifiers; an access-control coding error had been fixed on one domain but missed on a forgotten, still-public one.42 Peloton's API returned full user profiles to anyone who asked, authenticated or not, because the backend never verified that the caller owned the requested record.5 T-Mobile has disclosed multiple API-related breaches, including one affecting around thirty-seven million customers that resolved in a 350-million-dollar settlement, and a separate incident touching roughly seventy-six million records.32 None of these required a zero-day. They required an attacker to notice that the lock on the front door says nothing about the locks on the individual apartments.

The login screen answers "are you a user?" An API endpoint must independently answer "are you this user, for this object, in this moment?" — and most of them forget to ask.

This manual walks the full surface of that question. It covers the two dominant API styles — REST and GraphQL — and the testing techniques specific to each. It works through the authentication and authorization failures that dominate the breach record: broken authentication, BOLA and IDOR, broken function-level authorization, mass assignment, and the abuse of business flows. It dissects the token and delegation machinery that underpins modern auth — JSON Web Tokens and OAuth 2.0 — where subtle implementation flaws turn cryptography into theatre. It maps everything back to the OWASP API Security Top 10 (2023), the canonical taxonomy for this domain,6 and closes with the labs, certifications, standards, regulations, and channels you need to build and keep the skill. Every load-bearing claim is referenced; the goal is a document you can act on and audit.

Tester's framing

Throughout, attacks are expressed as the HTTP transactions they actually are. The dark strip at the head of each section shows the request primitive — method, path, and the status code the server wrongly returns. If you can read that strip, you can reproduce the test. 01The modern attack surface

What makes an API different to test

A traditional web application renders state on the server and hands the browser a finished page; the attack surface is mediated by that rendering. An API does the opposite. It hands raw state — objects, identifiers, relationships — directly to a client that the server does not control, and trusts that client to ask only for what it should. Every object identifier in a response is a coordinate an attacker can perturb. Every field the client can send is a field the client can tamper with. The surface is wider, flatter, and far more machine-enumerable than a rendered UI, which is precisely why automated abuse scales so well against it.

Three architectural styles dominate, and they fail differently.

REST

Representational State Transfer, defined in Roy Fielding's 2000 dissertation, is an architectural style rather than a protocol: resources are named by URIs, manipulated through a uniform set of HTTP methods, and represented in a negotiable format — almost always JSON in practice.7 The style's virtue, a predictable mapping of nouns to URLs and verbs to methods, is also its testing handle. GET /api/orders/41 implies GET /api/orders/42; GET implies the existence of PUT, PATCH, and DELETE on the same resource. REST's resource-centric design means object identifiers are everywhere, which is exactly why object-level authorization is REST's defining failure mode.

GraphQL

GraphQL, developed at Facebook and now stewarded by the GraphQL Foundation, inverts the model: a single endpoint accepts a query language in which the client specifies precisely the shape of data it wants, traversing relationships in one round trip.8 This solves over-fetching and under-fetching elegantly and creates an entirely new threat surface in the process. A single endpoint defeats per-route controls; client-specified query shape defeats naive rate limiting; built-in introspection hands attackers a complete schema map; and the freedom to nest and alias arbitrarily turns one request into a denial-of-service or a brute-force amplifier. GraphQL does not enforce authorization by default — that responsibility falls entirely on the resolver layer, and the gap between route-level and field-level enforcement is where most GraphQL bugs live.

gRPC and the rest

Beyond these, gRPC (HTTP/2 plus Protocol Buffers) is increasingly common for internal service-to-service traffic, and SOAP persists in enterprise and financial systems. Their wire formats differ, but the security questions are identical: is the caller authenticated, is the caller authorized for this specific operation and object, and is input validated before it reaches an interpreter. The techniques in this manual translate across all of them — the transport changes, the trust questions do not.

Reconnaissance first

Before any of this, you have to find the API. Modern estates are riddled with shadow and zombie endpoints — undocumented, deprecated, or test interfaces left exposed — and roughly a fifth of APIs are estimated to be unmanaged. The most damaging breaches frequently involve an endpoint nobody remembered was live. Discovery (passive recon, traffic analysis, schema and documentation harvesting, endpoint brute-forcing) is the foundation everything else stands on, and it maps directly to OWASP's "Improper Inventory Management" risk discussed later.

02Testing methodology

A repeatable workflow, not a bag of tricks

Effective API testing is a methodology, not a checklist of payloads. OWASP's Web Security Testing Guide codifies a dedicated API testing methodology, and practitioner courses converge on the same broad sequence.9 The phases below structure the rest of this manual; each later section is a deep dive into one or more of them.

  1. Reconnaissance & discovery. Enumerate hosts, subdomains, and endpoints. Harvest OpenAPI/Swagger specs, GraphQL schemas (via introspection), mobile-app traffic, and JavaScript bundles. Build the inventory the defender often lacks.
  2. Endpoint & response analysis. Map every parameter, identifier, and field. Note where the response returns more than the UI shows (excessive data exposure) and where the client can send more than the UI offers (mass assignment).
  3. Authentication testing. Probe login, registration, password reset, MFA, token issuance, and API-key handling for weaknesses that let you become a user.
  4. Authorization testing. The core of API work. For every object and every function, ask whether a different, lower-privileged, or unrelated identity can reach it — horizontally (another user's data) and vertically (an admin function).
  5. Injection & input handling. Test where user input reaches an interpreter: SQL/NoSQL, OS commands, SSRF via server-side fetches, and GraphQL-specific injection.
  6. Rate, resource & business-logic abuse. Test the limits — or absence of limits — on requests, resource consumption, and sensitive flows that automation can weaponise.
  7. Reporting. Translate findings into reproducible HTTP transactions, business impact, and concrete remediation. A finding nobody can reproduce is a finding nobody will fix.

Two principles run through all seven phases. First, test as multiple identities. Authorization bugs are invisible from a single account; you need at least two users of the same role and ideally a privileged account, so you can ask the only question that matters — can user A reach user B's resource. Second, think in transactions, not pages. An API has no flow you must follow; you can call any endpoint in any order with any payload, and the server's assumptions about sequence and state are frequently where the logic breaks.

Defender's mirror

Every offensive phase has a defensive counterpart that belongs in the SDLC, not the pen test. Discovery ↔ a live, governed API inventory. Authorization testing ↔ a centralised, deny-by-default authorization layer invoked on every object access. Rate abuse ↔ per-user and per-object limits, not just per-IP. The most reliable way to pass an API pen test is to design as though one is always running.

03REST security testing

Interrogating resources, verbs, and the data you didn't ask for

PATCH /api/v2/users/me  {"role":"admin"} → 200 OK Mass assignment

REST testing begins with the uniform interface itself. Because methods are standardised, the first questions are mechanical: which verbs does each resource accept, and does the server enforce the right ones? An endpoint that reads with GET may silently accept PUT or DELETE from a user who was never meant to write. Method-override headers compound this — frameworks that honour X-HTTP-Method-Override, X-Original-URL, or X-Rewrite-URL let an attacker smuggle a forbidden verb or path past a front-end proxy that only filters the literal request line. The OWASP REST Security Cheat Sheet is the canonical baseline here: restrict methods explicitly, reject anything not on the allow-list, and never trust override headers from untrusted clients.10

Status codes are intelligence

The response code is the cheapest oracle in API testing. A 200 where you expected 403 is an authorization finding. A 401 that becomes a 200 when you add a spoofed header is a control bypass. A 429 that never appears under thousands of requests is a missing rate limit. The differential between codes — across identities, across small input changes — is the signal. Read it deliberately: 2xx means the operation succeeded, 401 means unauthenticated, 403 means authenticated-but-forbidden, and 404 is sometimes a 403 in disguise, used to hide the existence of a resource you can in fact reach by another route.

Excessive data exposure

REST endpoints frequently serialise an entire backing object and rely on the client UI to display only the relevant fields. The UI shows six fields; the JSON carries twenty-eight, including the ones the front-end quietly drops — internal flags, password hashes, tokens, neighbouring users' data. In the 2023 OWASP taxonomy this is folded, together with mass assignment, into Broken Object Property Level Authorization, whose root cause is the absence of authorization checks at the level of individual object properties rather than whole objects.11 The test is simple: read the raw response, not the rendered page, and look for any property the caller has no business seeing.

HTTPexcessive data exposure — read the wire, not the UI
GET /api/v2/users/me HTTP/1.1
Authorization: Bearer <valid-token>

// 200 OK — the app only renders name + avatar, but the body ships:
{
  "id": 1003,
  "name": "Dana",
  "email": "dana@example.com",
  "password_hash": "$2b$12$Q9...",   // ← should never leave the server
  "is_admin": false,            // ← tells the attacker the field exists
  "password_reset_token": "a1f3..."  // ← account-takeover primitive
}

Mass assignment

The mirror image of excessive exposure is mass assignment: the server binds the entire incoming JSON object to a model, so a client can set properties the form never offered. Send {"role":"admin"} or {"verified":true} or {"balance":999999} alongside the legitimate fields and watch whether they stick. The reconnaissance from excessive-exposure testing feeds this directly — every internal property you saw in a response is a candidate to write back. Defence is allow-listing the bindable fields explicitly and validating authorization per property, exactly as the BOPLA category prescribes.11

HTTPmass assignment — write a field the UI never exposed
PATCH /api/v2/users/me HTTP/1.1
Content-Type: application/json
Authorization: Bearer <low-priv-token>

{
  "displayName": "dana",        // the field the form offered
  "role": "admin",             // the field it didn't
  "emailVerified": true
}
// 200 OK + role now "admin"  →  vertical privilege escalation

CORS, content types, and parser tricks

A permissive CORS policy — reflecting arbitrary origins, or allowing credentials with Access-Control-Allow-Origin: * patterns — can let a malicious page read authenticated API responses from the victim's browser; test the Origin reflection behaviour directly. Content-type handling is another seam: APIs that accept multiple encodings (JSON, form, XML) may route to different parsers with different validation, and switching the body format is a documented way to slip past input filters and rate limits alike. XML parsers additionally invite XXE if external entities are not disabled. The discipline is the same everywhere — test what the server accepts, not what the documented client sends.

Real incident — Optus

The Optus breach was a textbook REST authorization failure: an internet-facing endpoint exposed customer records keyed by sequential, easily enumerated identifiers, with no server-side check that the caller owned the record. A fix applied to one domain was never propagated to a second, forgotten domain that stayed publicly reachable for years before it was exploited — roughly ten million customers' passport, licence, and identity data, extracted with nothing more sophisticated than incrementing an integer.42

04GraphQL security testing

One endpoint, one schema, and a dozen new ways to fail

POST /graphql  { __schema { types { name } } } → 200 OK Introspection on

GraphQL concentrates an entire API behind a single endpoint and a typed schema, which changes the testing playbook substantially. OWASP maintains a dedicated GraphQL Cheat Sheet and a GraphQL chapter in the Web Security Testing Guide; together they define the surface.129 The recurring theme is that GraphQL's conveniences — introspection, flexible query shape, batching — are each a security liability when defaults are left in place.

Introspection: the schema is the map

By default, most GraphQL servers answer introspection queries that return the complete schema — every type, field, mutation, deprecated field, and sometimes private fields never meant to be public. For an attacker this is reconnaissance handed over for free: the entire API, documented, in one request. OWASP and the major vendors are explicit that introspection (and developer IDEs such as GraphiQL or Playground) should be disabled or tightly restricted in production; Apollo's own guidance is to turn it off and to strip verbose errors and stack traces, and the GraphOS Router disables introspection by default.1213 When introspection is closed, also disable field suggestions — the "did you mean…" hints that leak field names one guess at a time.

GraphQLintrospection query — full schema disclosure
query IntrospectionMap {
  __schema {
    types { name fields { name } }
    mutationType { fields { name args { name } } }  // every write operation, named
  }
}
// If this returns the schema in production, you have a misconfiguration finding.

Authorization is on you — at the field level

GraphQL enforces no authorization of its own; the resolver for each field must do it. The common, dangerous pattern is a check at the route or top-level query, with no check on nested resolvers — so a caller who passes the outer guard can read sensitive child fields directly. The WSTG GraphQL chapter demonstrates exactly this: unauthenticated users reaching an auth query and extracting tokens, because permissions were assumed rather than enforced per field.9 In OWASP API terms, a User type that exposes email, role, and some internal score, with route-level auth but no per-field check on the internal field, is Broken Object Property Level Authorization realised in GraphQL.11 Test every field of every type as every identity; the object that's safe at the top can be unsafe two levels down.

Batching and aliases: brute force and DoS in one request

Because a single GraphQL request can contain many operations — through query batching or field aliasing — controls applied per HTTP request collapse. Rate limiting that counts requests is meaningless when one request carries five hundred aliased login mutations; the attacker brute-forces credentials or enumerates objects while the limiter sees a single call.12 This batching attack is, functionally, credential stuffing amplified by the protocol.

GraphQLalias batching — many attempts, one HTTP request, one rate-limit decrement
mutation Spray {
  a1: login(user:"victim", pass:"Winter2025!") { token }
  a2: login(user:"victim", pass:"Spring2025!") { token }
  a3: login(user:"victim", pass:"Summer2025!") { token }
  /* ... a4 ... a500 ... */
}
// One request. Five hundred guesses. Per-request rate limiting never fires.

Query depth and complexity: resource exhaustion

GraphQL relationships can be nested arbitrarily, and each level multiplies resolver work. A query that walks users → friends → posts → comments → author → friends fifteen levels deep can trigger an exponential number of backend calls and exhaust the connection pool — a denial of service that looks, on the wire, like one small query. The defences are well established: enforce a maximum query depth, score and cap query cost, demand pagination, set server-side timeouts, and limit the number of operations per batch. Depth limiting is supported by libraries such as graphql-depth-limit for JavaScript and MaxQueryDepthInstrumentation for graphql-java, and server-side batching/caching (Facebook's DataLoader) reduces duplicate resolver calls.12

Injection and SSRF still apply

GraphQL is not immune to the classics. User-supplied arguments flow into database queries, HTTP calls, and OS commands inside resolvers, so SQL/NoSQL injection, command injection, SSRF, and XSS in rendered responses all remain in scope — strict input validation against an allow-list of valid values, parameterised queries, and safe encoding are the same defences as anywhere else.12 Two GraphQL-specific extras worth probing: directive overloading and alias overloading, where a flood of repeated directives or aliases stresses the engine before resolution even begins.

Defence in depth for GraphQL

No single control suffices, because each only covers one vector: rate limiting alone ignores batching and nesting. Layer them — disable introspection and field hints in production, enforce authorization in every resolver, cap depth and complexity, limit operations per batch and aliases per query, paginate, and validate every argument. Plugins such as GraphQL Armor bundle several of these. The philosophy is defence in depth: when one layer is misconfigured, the next still catches the attack.12

05Authentication weaknesses

Becoming a user you are not

POST /api/login  × 10,000 → 200 · 200 · 200 … No lockout

Authentication is the mechanism by which the API identifies the caller, and it is the second-ranked risk on the OWASP list precisely because it is so often implemented incorrectly — letting attackers compromise tokens or exploit flaws to assume other identities, temporarily or permanently. The 2023 edition deliberately widened the category beyond user login to include machine and microservice authentication, and added newly emphasised failures such as permitting changes to sensitive information without password confirmation and failing to validate a JWT's expiration.14 Compromise the system's ability to know who is calling, and every other control is moot.

Credential attacks and missing throttling

The bread-and-butter tests are credential stuffing (replaying breach-corpus username/password pairs), brute force, and credential spraying (one common password against many accounts). What makes them succeed is the absence of compensating controls: no account lockout, no progressive delay, no MFA, weak password policy, and — critically — no effective rate limiting on the authentication endpoint. Nearly half of account-takeover attacks target authentication APIs with exactly these techniques, which is why rate limiting belongs on login endpoints specifically and not just on the marketing site. The test is direct: automate a large number of attempts and watch for the 429 that should appear and frequently does not.

Password reset and account recovery

Recovery flows are a rich seam because they bypass the password entirely. Probe for predictable or short reset tokens, tokens that don't expire or aren't single-use, and reset links whose domain is derived from a tamperable Host or X-Forwarded-Host header — host-header poisoning that sends the victim's reset link to an attacker-controlled domain. Excessive data exposure feeds this directly: if a profile endpoint leaks a password_reset_token (as in the REST example earlier), recovery becomes a single request.

MFA bypass

Multi-factor steps are frequently bolted on rather than enforced server-side. Test whether the second factor can be skipped by calling the post-MFA endpoint directly, whether a "mfa_passed": false field in the response can be flipped, and whether the OTP itself can be brute-forced because the verification endpoint has no rate limit — a six-digit code with no throttle is a million guesses, which is nothing to a script. MFA, correctly enforced, is one of the highest-leverage controls available; the operative word is enforced.

Tokens, sessions, and API keys

Examine the full lifecycle of whatever bearer credential the API issues. Long-lived tokens with no revocation path mean a single leak is permanent. Tokens placed in URLs leak through logs, referrers, and browser history. Sessions that don't rotate on privilege change invite fixation. And API keys deserve special scrutiny: a key is an identifier, not an authentication of a human, and APIs that rely on keys alone inherit every weakness of a static secret — keys hard-coded in mobile apps and JavaScript bundles, over-permissioned keys that grant far more than the integration needs, and keys that are never rotated. The guidance is unambiguous: do not authenticate with API keys alone, scope and expire tokens, and require step-up authentication for sensitive operations.

Authentication ≠ authorization

This section is about defeating authentication — becoming some valid identity. The next two are about defeating authorization — doing things that identity shouldn't. They are different failures with different fixes, and conflating them is the root cause of most API breaches: teams harden the login, declare victory, and leave every object and function trusting whoever got past the door.

06BOLA / IDOR — the #1 risk

Changing the number in the URL

GET /api/orders/1042  (you are user 1041) → 200 OK BOLA

Broken Object Level Authorization is the single most prevalent and damaging flaw in modern APIs, ranked first by OWASP and unmoved since the list's inception, because it is both trivial to exploit and structurally easy to introduce.153 The vulnerability formerly known broadly as Insecure Direct Object Reference (IDOR) is, in the API context, named BOLA: an endpoint accepts an object identifier from the user, retrieves the object, and returns it without verifying that the caller is entitled to that specific object. IDOR is the general class; BOLA is its API-specific framing, and they describe the same defect.

The mental model is an apartment building. Authentication is the lobby door — you prove you live in the building. Authorization is the lock on your individual unit. A BOLA-vulnerable API checks identity at the lobby and then lets you open any apartment by changing the number on the key. The caller is authenticated; the object is simply not protected.15

The exploit primitive

It is as plain as it looks. You are authenticated as user 1041; you request /api/orders/1042; the server returns user 1042's order. Automated tooling identifies these flaws in minutes, and the impact ranges from mass data scraping to manipulation of other users' transactions. The bug appears in horizontal form (peer data — another user at your privilege level) and vertical form (escalation to higher-privilege data), and it surfaces wherever object identifiers travel: path segments, query parameters, request bodies, custom headers, and inside GraphQL arguments and batch operations.

HTTPBOLA test — same token, someone else's object
// Authenticated as user 1041. Baseline:
GET /api/orders/1041   → 200 OK  (your order)

// Perturb the identifier:
GET /api/orders/1042   → 200 OK  (NOT your order)  ✕ BOLA

// Also test the write side — reading is not the only impact:
DELETE /api/orders/1042 → 204  ✕ you just deleted a stranger's order

// And the indirect places identifiers hide:
POST /api/export   {"account_id": 1042}   // body
GET  /api/report   X-Account-Id: 1042          // header

UUIDs help — but they are not authorization

Replacing sequential integers with UUIDs raises the cost of enumeration: an attacker can no longer simply increment. But a UUID is still a reference, and if it leaks — through a shared link, a referrer, a log, an earlier excessive-exposure response, or another endpoint that lists IDs — the missing ownership check still grants access. Randomised identifiers are defence in depth against discovery; they are not a substitute for verifying, on every object access, that this caller owns this object. Treating an unguessable ID as sufficient proof of access rights is a recurring root cause, especially in high-trust environments like telecom and finance.

Why it keeps happening

Object-level authorization is diffuse: it must be enforced in every function that touches a data source, across complex hierarchies of users, groups, and roles, and many frameworks offer poor primitives for it. Under delivery pressure, a developer adds an endpoint and wires up the query but forgets the ownership predicate; client-side state management that trusts client-provided IDs makes it worse. The fix is architectural — a centralised, deny-by-default authorization layer invoked on every object access, checking ownership or an explicit grant against the authenticated principal, rather than per-endpoint checks that are inevitably forgotten somewhere.15

Real incidents — BOLA in the wild

BOLA is the connective tissue of the breach record. The U.S. Postal Service's "Informed Visibility" API let any authenticated user query any other user's account details via wildcard parameters, exposing roughly sixty million users — and sat unpatched for over a year. Peloton exposed full user profiles by swapping user IDs; Parler was scraped of around seventy terabytes because endpoints served data by post ID with no ownership check; researchers demonstrated account takeover against Ferrari's systems; and CVE-2024-1313 was a BOLA flaw in Grafana, used by tens of millions. Optus, Coinbase, and T-Mobile round out a pattern that is less a series of accidents than a systemic blind spot.325

07Broken function-level authorization

Calling functions meant for someone above you

POST /api/admin/users  (as a standard user) → 200 OK BFLA

Where BOLA is about objects — whose data can you read — Broken Function Level Authorization is about functions: which operations can you invoke. It sits at #5 on the OWASP list and arises from complex access-control policies with unclear separation between administrative and regular functionality, letting attackers reach other users' resources or admin capabilities such as creating users, changing roles, or deleting records.16 The distinction matters for testing: a BFLA test targets the action, not a specific record.

BFLA is frequently more guessable than developers assume because administrative functions follow conventions. If /api/users/me exists, try /api/admin/users, /api/users, and /api/internal/.... If a normal user can GET a resource, test whether they can PUT, PATCH, or DELETE it. Replay requests captured from a privileged account using a low-privileged token and see what still succeeds. The signal, once more, is the differential: an endpoint that returns 200 to an account that should receive 403.

HTTPBFLA test — privileged action with an unprivileged token
// As an ordinary user, probe administrative functions by convention:
GET    /api/admin/users          → 200 OK   ✕ you can list all users
POST   /api/admin/users          → 201      ✕ you can create users
PATCH  /api/users/1042/role      {"role":"admin"}  → 200  ✕ role change

// Method-based BFLA: read is allowed, write should not be:
GET    /api/invoices/5500        → 200 (intended)
DELETE /api/invoices/5500        → 204 ✕ no function-level check on DELETE
Defence — deny by default, everywhere

Both authorization failures share one cure: a default-deny model in which every endpoint and every function explicitly declares the roles and conditions required to invoke it, enforced centrally rather than per-handler. Administrative functionality should live behind a distinct, consistently enforced boundary; roles and permissions should be checked server-side on every request; and the test for "can a lower-privileged identity do this" should be automated into the pipeline, because humans forget exactly the endpoint an attacker will find.16

08Rate limiting & resource abuse

When the limit isn't, and when there is no limit

POST /api/login  + X-Forwarded-For: <rotate> → 200 · 200 · 200 … Limit bypassed

Rate limiting is the control that turns a credential-stuffing run, a BOLA enumeration, or an expensive query from a viable attack into an expensive nuisance. Its absence — or its trivial bypass — is therefore a force multiplier for every other flaw. OWASP renamed the 2019 "Lack of Resources & Rate Limiting" category to Unrestricted Resource Consumption in 2023, broadening the lens: APIs consume bandwidth, CPU, memory, storage, and metered third-party services (SMS, email, biometrics), and unbounded consumption yields both denial of service and direct financial damage — a "denial of wallet" as cloud and per-request costs spiral.17

Bypassing IP-based limits

Most naive rate limiting keys on client IP, and most client IP determination trusts a header an attacker controls. The canonical bypass is spoofing the forwarded-for chain: applications and frameworks that read X-Forwarded-For, X-Real-IP, X-Client-IP, X-Originating-IP, or similar headers — sometimes even a doubled X-Forwarded-For — will attribute each spoofed value to a different "client," resetting the counter on every request.18 This is not theoretical: a published security advisory in Mastodon showed that its rate limiting, layered on a middleware that trusts the closest X-Forwarded-For and exempts localhost, could be bypassed entirely by an attacker spoofing 127.0.0.1 — defeating brute-force protection along with it.19

HTTPrate-limit bypass — a sampler of techniques
// 1. Spoof the client IP the limiter keys on (rotate the value):
POST /api/login
X-Forwarded-For: 127.0.0.1     // then .2, .3, ... or any random IP

// 2. Vary the path the limiter matches on:
POST /api/login/     // trailing slash
POST /api/Login      // case change, if matching is case-sensitive

// 3. Add a junk parameter so the request key differs:
POST /api/login?x=1

// 4. Change the body format or method (form ↔ JSON, POST ↔ PUT):
// 5. Append null bytes / whitespace to a parameter value.
// 6. Send concurrently — race the counter before it increments.

Beyond header tricks, attackers vary the request key (path casing, trailing slashes, extra parameters), switch body format or HTTP method, append null bytes or whitespace to values, and — most fundamentally — rotate source IPs at scale. Cloud providers offer effectively unlimited addresses, and residential-proxy services sell millions of real IPs, making pure IP-based limiting close to useless against a determined adversary.18 Race conditions deserve special mention: firing many requests concurrently can slip past a limiter (or a one-time-use check) before its counter updates, a class of bug that also breaks coupons, withdrawals, and "once per account" logic.

GraphQL batching, revisited

Recall that in GraphQL a single request can carry hundreds of aliased operations, so a per-request limiter sees one call while the server does hundreds of units of work. Rate limiting for GraphQL must therefore count operations and cost, not requests — the per-object and per-operation accounting the REST world rarely needs.

Unrestricted access to sensitive business flows

The 2023 list added a subtler sibling: Unrestricted Access to Sensitive Business Flows. Here nothing is technically "broken" — the API works as designed — but a flow that assumes human pace (buying limited-stock items, creating accounts, posting comments, reserving inventory) is exposed to automation with no compensating control, enabling scalping, spam, inventory hoarding, and fraud.20 Testing means identifying flows where excessive automated use harms the business and confirming whether device fingerprinting, behavioural analysis, CAPTCHA, or sequencing controls exist to slow them.

Defence — limit by identity, count by cost

Key limits to the authenticated principal (user/token), not just the IP; if you must read a forwarding header, configure the framework to trust only known proxies and take the correct hop. Layer per-user, per-endpoint, and per-object limits; apply cost-based accounting so a thousand-record export costs more budget than a ten-record read; add CAPTCHA or step-up on sensitive flows; and return 429 with Retry-After rather than failing open. Rate limiting is a control surface, not a single setting.17

09JWT attacks

When the signature is theatre

GET /admin  [ alg: "none", sig: "" ] → 200 OK Signature skipped

A JSON Web Token, defined in RFC 7519, is a compact, URL-safe credential carrying a set of claims; it is three Base64url segments — header, payload, signature — joined by dots, and its security rests entirely on the integrity of that signature.21 JWTs are attractive because they are stateless: the server can validate a token without a session lookup. That same statelessness means a forged or mis-validated token is accepted with no second opinion, which is why JWT handling is one of the most productive areas in API testing.22

The attacker's goal is to alter the payload — change role to admin, change sub to another user — and have the server accept it. Everything below is a way to make the signature stop mattering.

The alg: none attack

The JWT specification defines an "unsecured" mode where alg is none and the token carries no signature. Because the header is attacker-controlled, the test is to set alg to none, strip the signature (keeping the trailing dot), tamper the payload freely, and check whether the server accepts it — some libraries trust the header and validate nothing.23 If a token claiming "role":"user" can be rewritten to "role":"admin" with no signature and still works, the application is fully compromised.23

Algorithm confusion (RS256 → HS256)

The most elegant JWT attack exploits libraries that expose a single, algorithm-agnostic verify() that trusts the token's alg. The server intends asymmetric RS256: it signs with a private key and verifies with the public key, which it publishes freely. An attacker switches alg to the symmetric HS256 and signs the tampered token using the public key as the HMAC secret. The generic verifier, seeing HS256, dutifully computes an HMAC with the only key it has — the public key — and the signature matches. The attacker forges valid tokens without ever knowing the private key.24 Where the public key isn't published, it can sometimes be recovered from a pair of legitimate tokens (PortSwigger ships a tool that derives candidate values), so secrecy of the public key is not a defence.24

Conceptalgorithm confusion — public key becomes the HMAC secret
// Server expects RS256: verify(token, PUBLIC_KEY)
// Attacker sends HS256 and signs with the public key as the secret:

header  = { "alg": "HS256" }            // was RS256
payload = { "sub": "1041", "role": "admin" }  // tampered
sig     = HMAC_SHA256( header.payload, PUBLIC_KEY )

// Generic verify() runs HMAC with PUBLIC_KEY → matches → forged token accepted.

Weak secrets and key-sourcing tricks

HMAC-signed tokens are only as strong as their key. A human-memorable or low-entropy secret falls to offline brute force or dictionary attack once an attacker holds any valid token — tooling like hashcat and jwt_tool makes this routine. Header parameters that direct key lookup are another rich target: kid (key ID) is attacker-influenced and has led to SQL injection, path traversal, and command injection when used unsanitised, while jku and x5u (URLs pointing to a key set) invite SSRF and key substitution if the server fetches them without strict allow-listing.25

Missing claim validation

Even a correctly signed token must be checked. Common failures include not validating expiration (exp) — so old tokens live forever, a failure OWASP explicitly calls out — and not validating the audience (aud), letting a token minted for service A be replayed against service B. The audience check is among the most frequently omitted in practice. RFC 8725, the JWT Best Current Practices, is the authoritative remediation list.2514

Defence — RFC 8725 in one breath

Pin accepted algorithms server-side and never trust alg from the token; map each key to exactly one algorithm so RS256 cannot be downgraded to HS256 or none. Use strong, high-entropy keys. Treat kid, jku, and x5u as hostile input — validate or allow-list, never follow blindly. Always verify exp, iss, and aud, and use explicit token typing (typ) to prevent cross-protocol confusion.25

10OAuth 2.0 & OIDC abuse

Delegation is hard, and the seams leak tokens

GET /callback?code=…&redirect_uri=https://evil.tld → 302 Code exfiltrated

OAuth 2.0, specified in RFC 6749, is an authorization-delegation framework: it lets a third-party application obtain limited access to a resource on a user's behalf without handling the user's password.26 OpenID Connect (OIDC) layers authentication on top. The framework is widely deployed and notoriously difficult to implement correctly — most OAuth vulnerabilities are implementation flaws rather than protocol flaws, and they concentrate at two points: the redirect_uri, where the authorization code or token lands, and the state parameter, OAuth's CSRF token. Nearly every classic OAuth bug is one of these being absent, weak, or loosely validated.27

redirect_uri validation

After the user authorises, the provider redirects back to the redirect_uri carrying a code or token. If the authorization server validates that URI loosely — pattern-matching instead of exact matching, allowing subdomains, open redirects, or path traversal — an attacker can craft a flow that delivers the victim's code or token to an attacker-controlled endpoint. In the authorization-code flow this lets the attacker steal a code before it's used and redeem it at the legitimate callback, taking over the account without ever knowing the client secret.27 The test: fuzz redirect_uri with subdomains, path additions, traversal, and open-redirect chains, and confirm the server demands a complete, exact match.

The state parameter and CSRF

The state value binds the authorization response to the user's session; RFC 6749 §10.12 requires it for CSRF protection. Omit it and an attacker can force a victim's browser to complete a flow that links the attacker's identity to the victim's session — "forced OAuth profile linking," where the attacker subsequently logs in with their own social account and lands in the victim's account — or a login-CSRF that silently signs the victim into the attacker's account.27 Check for state in the initial authorization redirect, and that it is unguessable, single-session, and actually verified on return.

Implicit grant and PKCE

The legacy implicit grant returns the access token directly in the URL fragment (you'll see #access_token= at the callback); tokens in fragments leak through history, referrers, and any upstream open redirect, and the mode is now deprecated.28 Its replacement for public clients is the authorization-code flow with Proof Key for Code Exchange (PKCE, RFC 7636), in which the client commits to a secret code_verifier via a hashed code_challenge, so an intercepted code cannot be redeemed without the verifier.29 A common finding is a server that treats PKCE as optional — strip code_challenge/code_challenge_method from the request and test whether the token exchange still succeeds; if it does, public clients remain exposed to code interception.27

Hidden attack vectors

PortSwigger's research catalogued a class of subtler server-side OAuth/OIDC flaws beyond the classic redirect tricks: dynamic client registration that fetches attacker-supplied URLs (such as logo_uri or jwks_uri / sector_identifier_uri) and thereby becomes an SSRF-by-design primitive; redirect_uri session poisoning, where a hidden cross-domain request swaps in an untrusted client to redirect a token to an attacker; and WebFinger user enumeration via the OIDC discovery surface.30 These require understanding the provider's extension endpoints, and they are high-impact precisely because they live in machinery most testers ignore.

Standard — RFC 9700 (Jan 2025)

The current authority is RFC 9700, the OAuth 2.0 Security Best Current Practice, published January 2025, which updates and extends the older threat model (RFCs 6749/6750/6819) with practical mitigations. Its headline guidance: use the authorization-code flow with PKCE for all clients, enforce exact redirect_uri matching, prevent token replay, limit token privileges, and avoid both the implicit grant and the resource-owner-password-credentials grant. It also formalises newer threats — mix-up attacks, audience injection, and credential-leakage vectors — making it required reading for OAuth testing.28

11OWASP API Security Top 10 (2023)

The canonical taxonomy, mapped

Everything above maps to one reference framework: the OWASP API Security Top 10, whose second edition was published in 2023 and is the industry-standard awareness document for API risk.6 The list below is the authoritative ten, in OWASP's order, with a note on how each changed from the 2019 edition. Tags mark what is NEW, MODIFIED / RENAMED, or UNCHANGED in rank.

  1. Broken Object Level Authorization (BOLA) #1 UNCHANGED Endpoints expose object identifiers, creating a wide surface of object-level access-control issues. Object-level checks belong in every function that reads a data source using a user-supplied ID. The most prevalent and damaging API flaw.15
  2. Broken Authentication MODIFIED Authentication implemented incorrectly lets attackers compromise tokens or assume other identities. Broadened in 2023 beyond user login to machine/microservice auth, plus new emphases like sensitive changes without password confirmation and unvalidated JWT expiration.14
  3. Broken Object Property Level Authorization (BOPLA) NEW Merges 2019's Excessive Data Exposure and Mass Assignment. The root cause is missing or improper authorization at the level of individual object properties, leading to over-exposure or unauthorized manipulation of fields.11
  4. Unrestricted Resource Consumption RENAMED Formerly "Lack of Resources & Rate Limiting." Requests consume bandwidth, CPU, memory, storage, and metered services; unbounded consumption causes denial of service or runaway operational cost.17
  5. Broken Function Level Authorization (BFLA) #5 UNCHANGED Complex policies with unclear separation of admin and regular functions let attackers invoke functions they shouldn't — reaching other users' resources or administrative capabilities.16
  6. Unrestricted Access to Sensitive Business Flows NEW A business flow (buying, posting, reserving) exposed without compensating for harm from excessive automated use. Not necessarily an implementation bug — a design-level failure to consider abuse.20
  7. Server-Side Request Forgery (SSRF) NEW An API fetches a remote resource from a user-supplied URI without validation, letting an attacker coerce requests to unexpected destinations — even behind a firewall or VPN.31
  8. Security Misconfiguration MOVED UP APIs and their supporting systems carry complex configuration; missed hardening or insecure defaults (verbose errors, open CORS, debug endpoints, missing TLS) open the door to many attacks.32
  9. Improper Inventory Management RENAMED Formerly "Improper Assets Management." APIs expose more endpoints than web apps; without current documentation and an inventory of hosts and versions, deprecated and debug endpoints stay exposed — shadow and zombie APIs.33
  10. Unsafe Consumption of APIs NEW Developers over-trust data from third-party APIs and relax validation. Attackers compromise the integrated third party rather than the target directly, then ride the trusted channel inward.34
What changed in 2023

Two 2019 entries were retired — Injection and Insufficient Logging & Monitoring — on the grounds that, while real, they aren't API-specific and behave the same as in general web apps; the list deliberately focuses on risks unique to APIs. Three categories are genuinely new (Sensitive Business Flows, SSRF, Unsafe Consumption), one is a merger (BOPLA), and several were renamed to sharpen their root cause. BOLA, Broken Authentication, and BFLA — the authorization-and-identity core — held their positions, which tells you where the real-world losses keep coming from.6

12Defensive best practices

The build-side checklist

Testing finds the gaps; design closes them. The controls below consolidate the remediations referenced throughout this manual into a build-side checklist. The throughline is the same one the breach record keeps teaching: authenticate carefully, then authorize everything, on every request, by default-deny.

Authorization — the non-negotiables

  • Enforce object-level authorization on every access to a data source by user-supplied ID — verify the authenticated principal owns or is explicitly granted the object, not merely that they are logged in.15
  • Centralise authorization in a deny-by-default layer invoked uniformly, rather than per-endpoint checks that are forgotten somewhere.16
  • Enforce function-level authorization with a clear, consistent boundary around administrative operations; check on every method, not just GET.16
  • For GraphQL, enforce authorization in every resolver — field level, not just route level.12
  • Authorize at the property level: allow-list bindable fields (block mass assignment) and filter response fields (block excessive exposure).11
  • Use unpredictable identifiers (UUIDs) as defence in depth against enumeration — never as a substitute for the ownership check.

Authentication & tokens

  • Rate-limit authentication endpoints; add lockout/progressive delay; require MFA for sensitive operations and enforce it server-side.14
  • Issue short-lived, narrowly scoped tokens with a working revocation path; never rely on API keys alone for authentication.14
  • For JWTs: pin accepted algorithms server-side; map each key to one algorithm (no RS256→HS256/none downgrade); validate exp, iss, and aud; treat kid/jku/x5u as hostile; use explicit typing.25

OAuth 2.0 / OIDC

  • Use authorization-code flow with PKCE for all clients; enforce exact redirect_uri matching; require and verify state; avoid implicit and resource-owner-password grants.28
  • Treat dynamic-registration and discovery URLs (jwks_uri, logo_uri, sector_identifier_uri) as SSRF risks — allow-list and validate before fetching.30

Resource limits, input, and configuration

  • Apply per-user and per-object rate limits (not just per-IP); use cost-based accounting; add controls (CAPTCHA, fingerprinting) to sensitive business flows; trust forwarding headers only from known proxies.1720
  • Validate all input against an allow-list; use parameterised queries; disable XML external entities; for GraphQL, cap query depth and complexity and limit operations per batch.12
  • Harden configuration: disable verbose errors, stack traces, and introspection in production; lock down CORS; enforce TLS; remove debug endpoints.32
  • Maintain a live API inventory; version and sunset endpoints deliberately; scan continuously for shadow and zombie APIs.33
  • Validate data received from third-party APIs as rigorously as user input — do not trust the integration channel.34
Tools won't save a missing check

A WAF, an API gateway, and a SAST scanner are valuable, but none of them catches the object-ownership check your team is about to forget on the next endpoint. Schema validation will not see a missing per-object authorization predicate, and static analysis finds only a fraction of authorization findings. The durable control is design discipline plus authorization tests wired into CI — make "can a different identity reach this?" a question the pipeline asks automatically.

The NIST SP 800-204 series is particularly relevant to modern backends: it identifies the core features a microservices environment must provide — authentication and access management, secure service discovery, mutual TLS, monitoring, circuit breaking, load balancing, and throttling — and shows how API gateways and service meshes deliver them, with the explicit recommendation that API authentication not rely on API keys alone.3637 It dovetails with SP 800-207's zero-trust model, where every request is authenticated and authorized regardless of network position.38

Regulations that raise the stakes

Several legal regimes make API authorization failures expensive, not merely embarrassing:

  • GDPR (EU General Data Protection Regulation) governs personal data; its most severe penalty tier reaches up to 4% of annual global turnover or €20 million, whichever is higher — and a BOLA flaw exposing EU residents' data is squarely in scope.
  • PCI DSS (Payment Card Industry Data Security Standard, currently the v4.x generation) governs any API touching cardholder data, mandating strong access control, encryption, and audit logging.
  • HIPAA governs protected health information in the U.S.; health-data APIs that leak records through authorization gaps incur per-violation penalties and mandatory breach reporting.

These regimes share a logic that aligns neatly with the OWASP list: access to sensitive data must be controlled, logged, and demonstrably limited to authorised parties — which is exactly what object- and function-level authorization testing verifies.

14Tooling

The working toolkit

No tool finds an authorization bug for you — that requires testing as multiple identities and reasoning about ownership — but the right tooling makes recon, manipulation, and automation tractable. The categories below cover a practical API testing stack; curated, regularly updated lists are maintained by the major training programmes.39

  • Intercepting proxies. Burp Suite is the de-facto platform — its Repeater, Intruder, and extensions (JWT Editor for token tampering, InQL for GraphQL, Autorize for authorization testing, Param Miner) cover most of the workflow; OWASP ZAP is the strong open-source alternative, and mitmproxy excels at scripted interception.40
  • API clients. Postman and Insomnia for crafting requests, importing OpenAPI/GraphQL collections, and managing multiple identity profiles side by side.
  • Discovery & recon. Kiterunner and ffuf/gobuster for endpoint brute-forcing, Arjun for hidden parameter discovery, Amass and Shodan for external asset mapping, and Nmap for host and service enumeration.39
  • JWT. jwt_tool for tampering and cracking, jwt.io for inspection, hashcat for brute-forcing weak HMAC secrets, and the sig2n/rsa_sign2n utilities for deriving public keys in algorithm-confusion attacks.24
  • GraphQL. graphql-cop for automated checks, InQL and Clairvoyance for schema extraction (even with introspection disabled), GraphQL Voyager for visualisation, and GraphQL Armor as a defensive plugin.12
  • Automated scanning. Nuclei templates and OWASP-API-aware scanners (e.g. EthicalCheck) for broad first-pass coverage — useful for misconfiguration and known patterns, never a substitute for manual authorization testing.39

∞In closing

Test the seam

Strip away the protocols and the tooling and API security reduces to a single, stubborn question that most systems answer wrongly: given a valid identity, is this specific request — for this object, this function, this flow, at this moment — actually permitted? REST scatters object identifiers across every URL; GraphQL hides authorization gaps two resolvers deep; JWT and OAuth turn the question into cryptography that fails open when the alg header is trusted or the redirect URI isn't pinned. The vulnerabilities differ in mechanism and converge in cause. The OWASP API Security Top 10 keeps BOLA, broken authentication, and BFLA at the top year after year because the industry keeps building the front door beautifully and leaving the apartment doors unlocked.

The work, then, is disciplined and unglamorous: enumerate the surface, test as more than one identity, read the status-code differentials, and treat every identifier, field, token claim, and redirect as hostile until proven otherwise. Do that consistently — in the pen test and, more importantly, in the design — and the catastrophic breach that starts with someone changing a number in a URL simply never gets its first 200 OK.
