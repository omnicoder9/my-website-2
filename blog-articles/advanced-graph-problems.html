Orientation

Most graph problems are not hard because the algorithm is hard. They are hard because the graph is hidden. A grid of elevations, a list of plane tickets, a table of signal delays — none of these announces itself as a graph, and the entire difficulty is learning to see the vertices and edges that were there all along.

The five problems collected here sit just past the point where breadth-first and depth-first search stop being enough. Each is a small, self-contained puzzle, and each is a doorway into a pattern you will reuse for the rest of your competitive career: single-source shortest paths, minimum spanning trees, hop-constrained paths, bottleneck paths, and Eulerian tours. Once the pattern is visible, the implementation is almost mechanical. The skill worth building is the recognition that happens before the first line of code.

For each problem we state it plainly, walk a small example by hand, develop the intuition for why a particular algorithm is the right tool, give the complexity, and then show a complete, submission-ready solution in all six languages. Every solution is followed by the variations you are likely to meet and the pitfalls that quietly cost people the problem. The implementations follow the usual competitive convention: a single function or Solution class with standard headers and imports assumed, exactly as you would paste into a judge.

Notice that three of the five are shortest-path problems wearing different costumes. What changes between them is the definition of the cost of a path — sum of weights, sum under a hop limit, or the single largest weight on the path — and that one change is enough to demand a different algorithm. Learning to spot which cost function a problem is really asking about is most of the battle.

Pattern · Single-source shortest path

Network Delay Time

LeetCode 743 · Dijkstra

A signal starts at node k in a directed, weighted network of n nodes. Each edge [u, v, w] means a signal takes w time to travel from u to v. Return the time it takes for every node to receive the signal, or -1 if some node never does.

Worked example

times = [[2,1,1],[2,3,1],[3,4,1]], n = 4, k = 2  →  2

From node 2 the signal reaches node 1 at time 1, node 3 at time 1, and node 4 through node 3 at time 2. The network is fully informed only when the last node hears the signal, so the answer is the maximum of the shortest arrival times: 2.

Why this is a shortest-path problem

The time a single node takes to receive the signal is exactly the length of the shortest path from k to it — the signal will always arrive by the fastest available route. So the per-node answer is a single-source shortest-path computation, and because all edge weights are non-negative, Dijkstra is the natural choice.

The twist is the aggregation. Signals propagate in parallel down every edge at once, so the whole network is “done” at the moment the slowest-to-arrive node finally hears it. That makes the final answer the maximum over all shortest-path distances, not the sum. If any node is unreachable, its distance stays infinite and the answer is -1. This max-of-shortest-paths framing is the entire problem; everything else is a textbook heap-based Dijkstra.

Complexity

With a binary heap, O(E log V) time and O(V + E) space for the adjacency list and distance array.

Variations you will meet

  • Negative edge weights. Dijkstra breaks the moment an edge can be negative. Switch to Bellman–Ford (or SPFA) and the max-of-distances framing carries over unchanged.
  • Undirected network. Insert each edge in both directions when building the adjacency list; nothing else changes.
  • “Time for the k-th node” or “how many are reachable by time t.” Compute the same distance array, then sort or count instead of taking the maximum.

Pitfalls

  • Forgetting the unreachable case — an untouched infinite distance must produce -1, not a giant number folded into the maximum.
  • Index confusion. Nodes are 1-indexed here; size your distance array as n + 1 and ignore slot 0, or translate carefully.
  • Reprocessing settled nodes. Guard every pop with a staleness check (skip if the popped distance exceeds the recorded one) so the heap variant stays near-linear instead of degrading.

Pattern · Minimum spanning tree

Min Cost to Connect All Points

LeetCode 1584 · Prim (dense)

Given points on a plane, the cost to connect two points is their Manhattan distance. Return the minimum total cost to connect all points so that there is exactly one path between any two.

Worked example

points = [[0,0],[2,2],[3,10],[5,2],[7,0]]  →  20

The phrase “exactly one path between any two points” is the definition of a tree, and we want the cheapest tree that spans every vertex. That is a minimum spanning tree, and its total edge weight here is 20.

Spanning tree, not shortest path

It is tempting to reach for a shortest-path algorithm because the costs are distances, but that solves the wrong problem. We do not care about the distance between specific pairs; we care about choosing a set of edges that keeps everything in one connected piece for the least total weight. That is precisely the minimum spanning tree.

The decisive observation is that the graph is complete: every pair of points is a candidate edge, so there are V² edges. For a dense graph like this, the dense version of Prim’s algorithm — repeatedly absorbing the nearest outside vertex using a flat array scan instead of a heap — runs in O(V²) and avoids all heap overhead. Since you must at least look at every pair once to know the distances, O(V²) is optimal here. Kruskal with a sorted edge list or heap-based Prim also work, but they pay O(V² log V) to sort or push every edge, which is strictly worse on a complete graph.

Complexity

Dense Prim is O(V²) time and O(V) space — no explicit edge list, no priority queue.

Variations you will meet

  • A different distance metric. Euclidean or Chebyshev distance changes only the weight formula; watch for 64-bit overflow when squaring coordinates for Euclidean comparisons.
  • Forced or forbidden connections. Some edges are pre-built (free) or banned. Pre-union the forced pairs in a Union-Find and run Kruskal, skipping banned edges.
  • Maximum spanning tree. Flip the comparison to grab the most expensive edges — useful when “cost” is really a reliability or bandwidth you want to maximize.
  • Second-best MST or critical edges. Build the MST, then ask which single edge, added or removed, changes the answer; a staple of harder contest rounds.

Pitfalls

  • Materializing the full V² edge list for Kruskal when dense Prim needs none — wasteful in both time and memory at large n.
  • Integer overflow on some metrics. Manhattan distance is safe in 32 bits for typical limits, but accumulate the total in a 64-bit type to be certain.
  • Off-by-one in the “nearest outside vertex” scan — initialize the in-tree marker for the starting vertex correctly so it is never re-selected.

Pattern · Hop-constrained shortest path

Cheapest Flights Within K Stops

LeetCode 787 · Layered Bellman–Ford

There are n cities connected by flights [u, v, w] of price w. Find the cheapest price from src to dst using at most k stops, or -1 if there is no such route.

Worked example

flights = [[0,1,100],[1,2,100],[2,0,100],[1,3,600],[2,3,200]]
src = 0, dst = 3, k = 1  →  700

The route 0→1→2→3 costs only 400 but uses two stops, one too many. Within the one-stop budget the cheapest legal route is 0→1→3 at 700. The budget, not the price, is what makes this hard.

Why plain Dijkstra fails

Dijkstra settles each node once, the first time it is popped with its minimum cost. That is exactly wrong here. The cheapest way to reach a city might use too many stops, while a slightly pricier route with fewer stops could be the only one that still extends to the destination within budget. Once Dijkstra locks in the cheap-but-too-deep arrival, it discards the route you actually needed. The feasible state is not cost alone; it is the pair (city, stops used), and any correct method must respect both coordinates.

The key idea

Run Bellman–Ford for exactly k + 1 rounds. After i rounds, the distance array holds the cheapest price to each city using at most i edges. The one detail that makes or breaks the solution: each round must relax from a frozen snapshot of the previous round’s distances. If you relax in place, a single round can chain two or more edges together and silently blow past the hop budget.

Seen this way, Bellman–Ford’s defining property — that after i rounds it has considered all paths of at most i edges — is not a limitation to work around but the exact mechanism the problem is asking for. The number of rounds is the hop budget. An equivalent route is a BFS or Dijkstra over the explicit (node, stops) state space; the layered Bellman–Ford is just the most compact expression of the same idea.

Complexity

O(K·E) time — k + 1 rounds, each scanning every flight once — and O(V) space for two distance arrays.

Variations you will meet

  • Exactly k stops rather than at most k. Read the answer only from the layer for exactly that many edges instead of the running minimum.
  • Maximize value under a budget. Flip the relaxation to keep the best reward while a separate cost stays under a cap — the same layered sweep with a different objective.
  • Extra state such as fuel or fees. Widen the state to (node, stops, fuel); the layered structure still applies, the table just grows another dimension.
  • Negative prices. Bellman–Ford handles negative weights natively, which is another reason it is the right base algorithm here rather than Dijkstra.

Pitfalls

  • Relaxing in place. The classic, near-invisible bug: without a per-round snapshot you let one round use multiple edges and overshoot the stop limit.
  • Confusing k stops with k edges. A route with k stops uses k + 1 flights, so loop k + 1 times.
  • Overflow when adding to an “infinity” sentinel. Guard relaxations with an explicit reachability check before adding a weight.

Pattern · Bottleneck / minimax path

Swim in Rising Water

LeetCode 778 · Max-relax Dijkstra

On an n × n grid, cell (r, c) has elevation grid[r][c]. At time t the water level is t, and you may move between two adjacent cells only once both their elevations are at most t. Starting at the top-left, return the least time to reach the bottom-right.

Worked example

grid = [[0,2],[1,3]]  →  3

The destination cell itself sits at elevation 3, so you cannot arrive before time 3 no matter the route. At t = 3 every cell is submerged enough to traverse, so the answer is 3.

The cost of a path is its highest point

The time to follow a particular path is governed by its single highest cell — you must wait for the water to rise above the tallest obstacle on the route, and once it has, every other step on that route is already passable. So the cost of a path is the maximum elevation along it, and you want the path whose maximum is as small as possible. This is the classic minimax, or bottleneck, path problem, and it shows up far beyond grids: widest-bandwidth routing, reliability paths, and more.

The cleanest solution is Dijkstra with one change. Instead of relaxing a neighbour with distance + edge weight, relax it with max(distance so far, neighbour’s elevation). The priority queue then always expands the cell reachable with the smallest bottleneck, and the first time the destination is popped, that bottleneck is the answer. Two other approaches are worth knowing because they generalize differently:

  • Binary search the answer. Fix a time t and ask, with a plain BFS or DFS, whether the destination is reachable using only cells of elevation at most t. Binary search over t gives O(V log U).
  • Union-Find by rising water. Add cells in increasing elevation, union each with its already-added neighbours, and watch for the moment the start and end fall into the same component. The elevation at that moment is the answer — a direct cousin of Kruskal, because the bottleneck path always lies along the minimum spanning tree.

Complexity

Max-relax Dijkstra visits each of the V = n² cells once with heap operations, giving O(V log V) time and O(V) space.

Variations you will meet

  • Maximin / widest path. Maximize the smallest capacity on a route instead of minimizing the largest cost — flip the comparisons and the same machinery solves bandwidth-style problems.
  • Eight-directional movement. Add the four diagonals to the move set; the algorithm is unchanged.
  • Multiple sources or sinks. Seed the heap (or the Union-Find target test) with several start cells at once.

Pitfalls

  • Summing instead of taking the maximum. The whole problem hinges on the cost being a max, not a sum; a standard Dijkstra relaxation answers a different question.
  • Skipping the visited check and re-expanding cells, which inflates the running time.
  • Forgetting that the starting cell’s own elevation counts toward the bottleneck — seed the search with grid[0][0], not zero.

Pattern · Eulerian path

Reconstruct Itinerary

LeetCode 332 · Hierholzer’s algorithm

Given a list of airline tickets [from, to], reconstruct the itinerary that uses every ticket exactly once, beginning at JFK. If several valid itineraries exist, return the one that is smallest in lexical order. A valid itinerary is guaranteed to exist.

Worked example

tickets = [["MUC","LHR"],["JFK","MUC"],["SFO","SJC"],["LHR","SFO"]]
→  ["JFK","MUC","LHR","SFO","SJC"]

Each ticket is a directed edge, and “use every ticket exactly once” is the definition of an Eulerian path — a walk that traverses every edge of the graph exactly once, here starting from JFK.

Why naive greedy backtracking blows up

The obvious approach — always fly to the lexically smallest next airport — can strand you. Take the smallest edge and you may reach a dead end with tickets still unused, forcing a backtrack; on adversarial inputs that backtracking degrades badly. Hierholzer’s algorithm sidesteps it entirely by building the path in reverse via a post-order walk.

Keep each airport’s destinations sorted. Walk greedily to the smallest available destination, consuming edges as you go. When you hit a vertex with no remaining outgoing edges, that vertex is a dead end: append it to the route and step back. Because you append dead-ends in post-order and reverse the whole route at the end, the result is simultaneously a valid Eulerian path and the lexicographically smallest one. The post-order is the trick — it is the classic circuit-splicing argument made iterative, so a vertex is only finalized once every edge leaving it has been used.

Existence, for the curious

A directed graph has an Eulerian path when it is connected on its edge set and every vertex has equal in- and out-degree, with at most one vertex having one extra outgoing edge (the start) and at most one vertex one extra incoming edge (the end). This problem guarantees such a path exists, so you can reconstruct without first checking — but the same Hierholzer routine doubles as the check when existence is in doubt.

Complexity

O(E log E) dominated by keeping each airport’s destinations sorted (via a heap, multiset, or one upfront sort); the Hierholzer walk itself is linear in the number of tickets.

Variations you will meet

  • Eulerian circuit. When the route must return to its start, the same algorithm applies with balanced in- and out-degrees at every vertex.
  • Existence check only. Verify the degree conditions and edge-connectivity instead of (or before) building the path.
  • Largest lexical itinerary. Sort destinations in descending order so the stack yields the largest next airport.
  • Parallel tickets. Duplicate routes are just parallel edges; a multiset or heap of destinations handles the multigraph without special cases.

Pitfalls

  • Forgetting to reverse the collected route — the post-order produces the path backwards by construction.
  • Recursion depth. With many tickets a recursive Hierholzer can overflow the stack; prefer the explicit-stack iterative form shown here.
  • Letting the destination order drift so pop no longer yields the smallest option, which quietly breaks the lexical guarantee.
  • Treating it as an ordinary path search to a fixed target rather than an edge-covering walk — a different problem with a different answer.

Synthesis

Which algorithm, when

The hard part of a graph problem is matching it to a tool. These are the decision points that separate the five patterns — and a few neighbours worth contrasting against.

non-negative weights

Dijkstra. Standard single-source shortest paths. Aggregate with a max for “when is everyone reached.”

negative weights / cycle check

Bellman–Ford. Handles negatives and detects negative cycles where Dijkstra cannot.

hop limit or extra state

Layered Bellman–Ford or a Dijkstra over (node, stops). The constraint becomes another state dimension.

connect everything, cheapest

MST — dense Prim on complete graphs, Kruskal with Union-Find on sparse ones.

minimize the max / maximize the min

Max-relax Dijkstra, binary search on the answer, or Union-Find by sorted weight.

use every edge once

Hierholzer for an Eulerian path or circuit. Sort neighbours for the lexical variant.

use every vertex once

Hamiltonian path — NP-hard; bitmask DP only for small n. The deceptively similar sibling of Eulerian.

Field signals

Pattern recognition cues

Problems rarely name their pattern, but their wording leaks it. Train yourself on these signals and most of the recognition becomes automatic.

If the statement says…Reach for…
“time for all to receive,” “latest arrival,” “until everyone is reached”Shortest paths, then take the maximum
“connect all,” “minimum cost network,” “one path between any two”Minimum spanning tree
“at most k stops,” “within k moves,” “limited budget of steps”Hop-constrained / state-space shortest path
“minimize the maximum,” “rising water,” “bottleneck,” “widest path”Minimax path — max-relax Dijkstra
“use all tickets,” “traverse every road once,” “every edge exactly once”Eulerian path — Hierholzer
