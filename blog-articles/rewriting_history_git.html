Rewriting History Without Losing Your Team (or Your Mind)

“I just force-pushed a cleaned-up branch and three colleagues lost their work.” A technical guide to safely reshaping shared Git history, recovering from mistakes, and building team safety rails that make the dangerous path harder to take.

Audience: intermediate to senior engineers Format: practical technical article Includes command examples and team policy templates

Git makes history editable, which is both its superpower and its trapdoor. Local history rewriting is routine: amend a typo, reorder commits, squash the “try again” noise, or rebase on a new base branch. Shared history rewriting is different. Once another person, CI job, release branch, or dependent pull request has based work on your commits, a rewrite stops being an individual cleanup and becomes a distributed-systems migration.

This article is about doing that migration deliberately. We will treat git push --force as a production operation, use autosquash to turn chaotic local work into a reviewable sequence, keep stacked pull requests alive with git rebase --update-refs, recover lost commits with reflogs, bundles, and object inspection, and then encode the lessons into repository rules and team agreements.

1. Mental model: history is a graph, not a diary

A Git commit is not just a patch. It is an object that points to a tree, metadata, and one or more parent commits. A branch name is simply a movable reference to one commit. That reference gives humans a stable name, but the actual project history is the directed graph behind it. When you rebase, squash, amend, or filter history, Git usually does not “edit” the old commits. It creates new commits with new object IDs and moves a branch name to point at the new graph.

That distinction explains why a force push can look harmless to the person who performed it and catastrophic to everyone else. Locally, your branch says “feature/auth points to commit F5.” Remotely, your teammates may still have work based on the old F1 → F2 → ... → F40 chain. If you replace that chain with A → B → C → D → E and push by force, you have moved the public signpost. The old commits may still exist in someone’s clone, reflog, CI cache, or hosting provider for a while, but they are no longer reachable from the branch name your team agreed to use.

The rule of thumb Rewriting commits that only you have is cleanup. Rewriting commits that others may have based on is coordination. Rewriting protected release or default branches is an incident unless your team explicitly designed for it.

The rest of this article assumes you sometimes need to rewrite shared feature branches because real teams do: stacked pull requests need their bases moved, merge windows require a clean series, release notes depend on usable commit messages, and reviewers need a story instead of a transcript. The goal is not “never rewrite.” The goal is to rewrite with evidence, communication, tooling, and recovery paths.

2. The shared branch nightmare: why git push --force is a weapon

Consider a branch called feature/invoice-ledger. You created it from main, pushed it, and two colleagues branched off it for related work:

main
  │
  M0
  │
  A1---A2---A3---A4---A5  feature/invoice-ledger
                       │
                       B1---B2  feature/tax-rules
                       │
                       C1---C2  feature/export-ledger

Your feature branch has 40 commits because real development is iterative: tests failed, naming changed, an API was split, a reviewer suggested a smaller abstraction, and two commits are titled “WIP”. Before the merge window, you rewrite feature/invoice-ledger into five logical commits:

main
  │
  M0
  │
  L1---L2---L3---L4---L5  feature/invoice-ledger

Now the remote branch no longer contains A1..A40. The dependent branches feature/tax-rules and feature/export-ledger still contain commits based on old history. Their authors fetch and discover that normal pulls cannot fast-forward, PR diffs are strange, and merge conflict context no longer matches what they saw yesterday.

The Git documentation is blunt about the danger: normal pushes refuse to update a branch when the remote branch is not an ancestor of the commit being pushed, while --force disables that check and can cause the remote repository to lose commits.1 That does not mean the object bytes instantly vanish everywhere, but it does mean the shared reference has stopped protecting them.

The failure modes

Failure mode What happened Typical symptom
Remote-only teammate commit overwritten Someone pushed T1 after your last fetch. Your forced update moved the branch to a history that does not include T1. The teammate says “my commit disappeared from the branch.”
Dependent branch stranded A stacked branch points to commits that have been replaced. The PR shows unrelated changes or a huge diff.
CI and review state invalidated The commit IDs that CI tested and reviewers approved no longer exist on the public branch. Required checks rerun; review approvals may become stale.
Local clone confusion Teammates have old remote-tracking refs, stale branches, or worktrees checked out. git pull complains, rebases conflict, or local branches appear “ahead and behind.”
Do not normalize raw force pushes If a team uses git push --force casually, the team has converted a version-control system into a social trust exercise. Use safer options, but also assume command-line safety cannot replace team coordination.

3. Safer force pushes: --force-with-lease and --force-if-includes

The safer default for rewritten branches is:

git push --force-with-lease --force-if-includes origin HEAD:feature/invoice-ledger

This command still performs a forced update. It is not “safe” in the sense that no one can be disrupted. It is safer because it asks Git to verify assumptions before moving the remote reference.

--force-with-lease: “only overwrite what I think I saw”

A lease says, in effect: “Update the remote branch only if it still points to the commit I expect.” In its most casual form, git push --force-with-lease compares the remote branch against your local remote-tracking reference. If somebody pushed new work that you have not fetched, your local expectation is stale and the push should fail.

That protects against a common race, but the Git documentation highlights a subtle weakness: using --force-with-lease without an explicit expected value can interact badly with background fetches, because a cron job, IDE, or editor integration might update your remote-tracking refs without you actually reviewing the incoming commits.1 In other words, your computer may “know” about a teammate’s commit even if you do not.

For higher assurance, name the expected old remote value explicitly. One practical pattern is to tag the base you reviewed:

# Update your view intentionally.
git fetch origin feature/invoice-ledger

# Mark the remote state you have reviewed and are willing to replace.
git tag rewrite-base origin/feature/invoice-ledger

# Rewrite locally.
git rebase -i --autosquash origin/main

# Force-push only if the remote branch still equals rewrite-base.
git push --force-with-lease=refs/heads/feature/invoice-ledger:rewrite-base \
  origin HEAD:refs/heads/feature/invoice-ledger

# Remove the temporary local marker.
git tag -d rewrite-base

This version is more verbose, but the verbosity is part of the safety. You are encoding the exact public commit you agreed to replace.

--force-if-includes: “only if I incorporated what I fetched”

--force-if-includes adds another guard. Git describes it as allowing the forced update only if the tip of the remote-tracking ref has been integrated locally; it checks reachability from reflog entries and rejects the forced update if remote updates were not incorporated.1 It is most useful with --force-with-lease in the common case where the lease is based on remote-tracking refs rather than an explicit object ID.

Use both for day-to-day rewritten feature branches:

git push --force-with-lease --force-if-includes origin HEAD:my-feature

Use an explicit lease for sensitive shared branches:

expected=$(git rev-parse origin/my-feature)

# ... inspect, communicate, rebase ...

git push --force-with-lease=refs/heads/my-feature:$expected \
  origin HEAD:refs/heads/my-feature

Why these options do not eliminate blast radius

Leases prevent overwriting work you did not account for. They do not update dependent branches, preserve PR review context, explain the rewrite to your teammates, or recover branches checked out in someone else’s worktree. That is why a good rewrite workflow includes four steps:

  1. Announce: say which branch will be rewritten and when.
  2. Snapshot: create a backup ref or tag before rewriting.
  3. Rewrite: rebase, autosquash, and test locally.
  4. Publish: force-push with lease, then tell dependent branch owners how to rebase.
# Snapshot the old public tip in a remote-visible namespace.
git fetch origin feature/invoice-ledger
git branch backup/feature-invoice-ledger-before-rewrite origin/feature/invoice-ledger
git push origin backup/feature-invoice-ledger-before-rewrite

# Rewrite and publish with guards.
git rebase -i --autosquash origin/main
git test # replace with your project command
git push --force-with-lease --force-if-includes origin HEAD:feature/invoice-ledger

The backup branch changes the conversation. Instead of “I hope your old work is somewhere,” you can say “the old tip is preserved at backup/feature-invoice-ledger-before-rewrite until Friday.”

4. Autosquash: the fixup funnel

The cleanest history is often built while the messy history is still being written. Rather than waiting until the end of a feature to manually rearrange dozens of commits, use fixup commits as a funnel. Each time you find a small correction to an earlier commit, create a commit that explicitly targets it. Later, git rebase -i --autosquash moves those fixups next to their targets and marks them for squashing.

Git’s rebase documentation says --autosquash automatically handles commits whose messages start with squash!, fixup!, or amend!; it moves them after the commit they modify and changes the rebase todo actions accordingly.2 Git’s commit documentation recommends creating those markers with git commit --fixup, --fixup=amend:, --fixup=reword:, or --squash rather than hand-writing fragile subjects.3

The basic loop

Start with a logical commit:

git add src/ledger/model.ts test/ledger/model.test.ts
git commit -m "feat(ledger): add immutable invoice ledger model"

Later, you find a bug in that commit. Do not create “WIP fix bug.” Target the original commit:

# Find the target.
git log --oneline --decorate -n 10

# Stage the correction.
git add src/ledger/model.ts test/ledger/model.test.ts

# Create a fixup commit for the original logical commit.
git commit --fixup=HEAD~3

Your log now contains a machine-readable breadcrumb:

9c8d7e6 fixup! feat(ledger): add immutable invoice ledger model
4b3a2c1 test(ledger): cover export edge cases
2a1b0c9 feat(export): stream CSV writer
...

When you are ready to clean the series:

git fetch origin
git rebase -i --autosquash origin/main

The interactive todo list opens with the fixup automatically placed after the target:

pick 1a2b3c4 feat(ledger): add immutable invoice ledger model
fixup 9c8d7e6 fixup! feat(ledger): add immutable invoice ledger model
pick 2a1b0c9 feat(export): stream CSV writer
pick 4b3a2c1 test(ledger): cover export edge cases

Fixing content vs. fixing messages

Use the right flavor of fixup for the job:

Command Use when Result during autosquash
git commit --fixup=<commit> You changed code or tests belonging to an earlier commit. The content is folded into the target; the target message stays.
git commit --fixup=amend:<commit> You changed content and want to refine the target commit message. The content is folded in and the target message is replaced by the amended message.
git commit --fixup=reword:<commit> You only want to change the earlier commit message. The target message changes without adding staged content.
git commit --squash=<commit> You want to combine content but keep a chance to compose a combined message. Interactive rebase stops for message editing.

Make autosquash the default

For teams that frequently clean commit series, turn it on for interactive rebases:

git config --global rebase.autoSquash true

The Git configuration option rebase.autoSquash enables autosquash by default for interactive mode, while --no-autosquash can override it for a single rebase.2

Combine with executable checkpoints

During cleanup, you want the final sequence to be reviewable and bisectable. Add tests after each logical commit:

git rebase -i --autosquash --exec "npm test" origin/main

Git’s rebase documentation notes that --exec appends shell commands after lines that create commits; when combined with autosquash, intermediate commits inside a squash/fixup series do not receive separate exec lines, and checks run at the end of each resulting series.2 That behavior matches the intent: test the commits that will survive.

5. Conventional commits and reviewable history

Commit cleanup is easier when the desired shape is explicit. Conventional Commits gives a small grammar for commit messages:

<type>[optional scope]: <description>

[optional body]

[optional footer(s)]

The specification describes itself as a lightweight convention for explicit commit history and automation, with common types such as fix for bug patches and feat for features.4 You do not need to be dogmatic to benefit. For history rewriting, the convention gives you a target taxonomy: features, fixes, tests, refactors, build changes, docs, and breaking changes should not be indistinguishable blobs.

From WIP transcript to review sequence

A messy feature branch often looks like this:

2ddfca1 wip
0119cd2 try failing test
89aa912 fix lint
78dc154 add model
e321ccd rename thing
45ba999 review changes
11ee027 exports maybe
93d1175 test more
aabc012 fix test again

A cleaned series should read like a short design document:

feat(ledger): add immutable invoice ledger model
feat(export): stream ledger entries as CSV
fix(export): preserve decimal precision in totals
test(ledger): cover voided invoice edge cases
docs(ledger): document export invariants

Those five commits are not just prettier. They improve several engineering workflows:

  • Review: reviewers can inspect one idea at a time.
  • Bisect: if a regression appears, each commit is more likely to build and test.
  • Release notes: commit messages can feed changelog automation.
  • Rollback: reverting one logical change is less likely to revert unrelated work.
  • Ownership: code owners can reason about the scope of each change.

A practical commit-message policy

For branches that may be rewritten before merge, use this lightweight policy:

# Allowed subjects
feat(scope): imperative description
fix(scope): imperative description
refactor(scope): imperative description
test(scope): imperative description
docs(scope): imperative description
build(scope): imperative description
ci(scope): imperative description
chore(scope): imperative description

# Optional footer for breaking changes
BREAKING CHANGE: describe migration impact and required action

Examples:

feat(auth): require device challenge for risky login

Persist a pending challenge record before sending the notification so
retry logic can recover from transient provider failures.

Refs: SEC-1842
fix(payments): avoid duplicate settlement on retry

The gateway can return a timeout after accepting the request. Check the
idempotency key before creating a second settlement record.

During autosquash, --fixup=reword: is particularly useful when the code is correct but the story is not:

git commit --fixup=reword:abc1234

That creates an amend-style commit that will replace the target commit’s message during autosquash without changing its content.3

6. Rebasing stacked pull requests with --update-refs

Stacked pull requests split a large change into reviewable layers. Instead of one massive branch, you have a chain:

main
  │
  ├── pr/01-ledger-model
        │
        ├── pr/02-ledger-export
              │
              ├── pr/03-tax-rules

Stacking is powerful, but rebasing the bottom of the stack can be painful. If pr/01-ledger-model is rewritten, the branches above it still point into the old graph unless you move them too.

Git’s rebase --update-refs option automatically force-updates branches that point to commits being rebased, except branches checked out in a worktree.2 That exception matters: Git will not silently move a branch another worktree is actively using.

The problem without --update-refs

Suppose your stack looks like this:

M0
 └─ A1─A2        pr/01-ledger-model
       └─ B1─B2  pr/02-ledger-export
             └─ C1─C2  pr/03-tax-rules

You are on pr/03-tax-rules, and you need to rebase the whole stack onto updated origin/main. A plain rebase may rewrite the commits reachable from your current branch, but your local branch refs for pr/01 and pr/02 may still point at the old commits. You then have to manually reset each branch to the corresponding rewritten commit.

The survivable workflow

Enable the behavior globally if stacked PRs are a normal team workflow:

git config --global rebase.updateRefs true

Or use it explicitly:

git fetch origin
git switch pr/03-tax-rules
git rebase -i --autosquash --update-refs origin/main

After the rebase, inspect the rewritten stack:

git log --oneline --graph --decorate --boundary origin/main..HEAD
git branch --contains pr/01-ledger-model
git branch --contains pr/02-ledger-export

Then publish each branch with leases:

git push --force-with-lease --force-if-includes origin pr/01-ledger-model
git push --force-with-lease --force-if-includes origin pr/02-ledger-export
git push --force-with-lease --force-if-includes origin pr/03-tax-rules

When not to use it blindly

--update-refs moves local branch pointers automatically. That is exactly what you want when the stack is yours and all dependent branches are local. It is not a substitute for coordinating with teammates who own branches in the stack. If another developer has pr/02-ledger-export checked out in a worktree, Git will not update that branch automatically; if they own the corresponding remote branch, you should not publish over it without their agreement.

Stack rule The person rewriting the bottom of a shared stack owns the communication burden. Before pushing, list every branch that depends on the rewritten commits and tell owners whether you updated it, left it alone, or created a rescue branch.

7. When the damage is done: recovery with reflog, bundle, fetch, and fsck

Even careful teams will eventually have an incident. The right response is calm, mechanical, and evidence-preserving. Stop pushing. Stop running cleanup commands. Identify where the missing commit still exists. Move it to a named reference before garbage collection can make recovery harder.

First response: preserve what remains

On the machine that may still have the lost work:

# Do not run git gc.
# Do not prune.
# Capture current state for forensics.
git status
git log --oneline --decorate --graph --all -n 50
git reflog --date=iso --all > /tmp/reflog-snapshot.txt

The reflog records where local refs such as HEAD and branch names used to point. It is local to a repository clone, which is why the affected teammate’s laptop is often the best source of truth after an accidental rewrite.

Recover a commit from a teammate’s reflog

On the teammate’s machine, search for the missing branch state:

git reflog --date=iso
git reflog show feature/invoice-ledger --date=iso
git log --oneline --decorate --graph --all --reflog --grep="settlement"

When you find the lost commit, protect it with a branch:

git branch rescue/alex-lost-settlement 4f31abc
git log --oneline --decorate rescue/alex-lost-settlement -n 5

Now there is a stable name. You can push that branch directly if policy allows:

git push origin rescue/alex-lost-settlement

If the teammate cannot push to the shared repository, or you want a minimal artifact to transfer, create a Git bundle.

Transfer the recovered commit with git bundle

A bundle is a portable pack of Git objects and references. From the teammate’s machine:

# Include every object needed from main to the rescue branch.
git bundle create alex-lost-settlement.bundle \
  origin/main..rescue/alex-lost-settlement \
  rescue/alex-lost-settlement

Send alex-lost-settlement.bundle through your approved secure channel. On the maintainer’s machine:

git bundle verify alex-lost-settlement.bundle
git fetch alex-lost-settlement.bundle \
  rescue/alex-lost-settlement:refs/heads/rescue/alex-lost-settlement

git log --oneline --decorate --graph rescue/alex-lost-settlement

From there, integrate with cherry-pick, rebase, or merge depending on the incident:

# Example: apply just the recovered commit to the cleaned branch.
git switch feature/invoice-ledger
git cherry-pick 4f31abc
git push --force-with-lease --force-if-includes origin feature/invoice-ledger

Find dangling commits and blobs with git fsck --lost-found

If reflogs do not reveal the work, inspect unreachable objects. Git’s fsck --lost-found option writes dangling commits into .git/lost-found/commit/ and dangling blobs into .git/lost-found/other/; for blobs, it writes the blob contents rather than only the object name.5

git fsck --lost-found --no-reflogs

# Inspect dangling commits.
for c in .git/lost-found/commit/*; do
  echo "==== $c ===="
  git show --stat --oneline "$(basename "$c")" | head -40
done

# Inspect dangling blobs.
grep -R "function calculateSettlement" .git/lost-found/other || true

If you find a commit:

git branch rescue/dangling-commit 91e2abc
git show --stat rescue/dangling-commit

If you find only a blob, recover the file content manually and recommit it:

mkdir -p /tmp/recovered
cp .git/lost-found/other/ab/cdef... /tmp/recovered/settlement.ts
diff -u src/payments/settlement.ts /tmp/recovered/settlement.ts

Recovery decision tree

Question Command Next action
Does any teammate have the old branch? git reflog show branch Create rescue/* branch and push or bundle it.
Does CI or a fork still have the commit SHA? git fetch <remote> <sha-or-ref> Move the SHA to a rescue branch immediately.
Are the objects present but unreachable? git fsck --lost-found --no-reflogs Inspect dangling commits/blobs and branch useful commits.
Was the work never committed? Check editor backups, stash, IDE local history. Recover manually; Git cannot recover objects it never stored.
Incident rule Once work appears lost, do not run git gc, git prune, or aggressive cleanup. First create named rescue refs or bundles.

8. Repository safety rails: make safe behavior the default

Command discipline is not enough. Teams need repository-level constraints and social contracts that make the risky path visible.

Protected branches

GitHub protected branches can require pull-request reviews before merging, require status checks, restrict who can push, and block force pushes by default.6 GitHub’s documentation also warns that force pushing can remove commits that collaborators based work on, create merge conflicts, corrupt pull requests, delete branches, or point a branch to commits that were not approved in a pull request.6

A sensible baseline for main, release branches, and long-lived integration branches:

  • Require pull requests before merging.
  • Require at least one approving review; require code-owner review for critical paths.
  • Require status checks such as unit tests, static analysis, and build verification.
  • Block force pushes except for a very small break-glass group.
  • Block branch deletion for release and default branches.
  • Dismiss stale approvals or require approval of the most recent push for high-risk repositories.

Feature-branch policy

Feature branches are different. Blocking all force pushes can make stacked PRs and clean history painful. Instead, apply a lighter policy:

# Team convention for shared feature branches

1. Never use raw `git push --force`.
2. Use `git push --force-with-lease --force-if-includes`.
3. Announce rewrites of shared branches in the PR or team channel.
4. Before a major rewrite, push a backup branch:
   backup/<branch-name>-before-<date>
5. Keep backup branches for at least five working days.
6. If work is lost, everyone shares relevant reflog output before running GC.
7. The author rewriting the lowest branch in a stack must help dependent
   branch owners rebase or publish updated stack branches.

Local Git aliases

Aliases can lower friction:

git config --global alias.pushf \
  "push --force-with-lease --force-if-includes"

git config --global alias.lg \
  "log --oneline --graph --decorate --all"

git config --global alias.rescue-log \
  "reflog --date=iso --all"

Now the safe path is shorter:

git pushf origin HEAD:feature/invoice-ledger

Pre-push guard

For local guardrails, a pre-push hook can reject raw force pushes to protected branch patterns. Hooks are not a complete security boundary because users can bypass or forget them, but they are useful friction.

#!/usr/bin/env bash
# .git/hooks/pre-push
# Reject pushes to main/release branches from local clones unless explicitly allowed.

set -euo pipefail

remote_name="$1"
remote_url="$2"

while read -r local_ref local_sha remote_ref remote_sha; do
  case "$remote_ref" in
    refs/heads/main|refs/heads/release/*)
      echo "Blocked push to protected branch: $remote_ref" >&2
      echo "Use a pull request, or follow the break-glass procedure." >&2
      exit 1
      ;;
  esac
done

exit 0

For server-side enforcement, use your hosting platform’s protected-branch or ruleset features rather than relying on local hooks.

Garbage-collection agreement

Most recovery stories rely on the fact that lost commits are not instantly gone. Reflogs and unreachable objects create a recovery window. But that window is not a policy unless the team defines it. For repositories with high collaboration risk, set an explicit agreement:

  • During incidents, do not run manual garbage collection or pruning.
  • Before cleanup, affected contributors export reflog snapshots.
  • Before deleting backup branches, maintainers announce the deletion.
  • Critical rescue refs are converted into issues or postmortem links so future maintainers know why they exist.

9. Real-world example: cleaning a 40-commit branch into five logical commits

Scenario: the team is minutes from a merge window. feature/invoice-ledger has 40 commits. Two dependent branches exist: feature/tax-rules and feature/export-ledger. You need to turn the base branch into a five-commit story without derailing the dependent branches.

Initial topology

origin/main
  │
  M0
  │
  A1...A40  feature/invoice-ledger
       │
       B1...B8   feature/export-ledger
             │
             C1...C6  feature/tax-rules

Target history

origin/main
  │
  M0
  │
  L1 feat(ledger): add invoice ledger aggregate
  L2 feat(ledger): persist ledger entries transactionally
  L3 feat(export): stream ledger CSV with backpressure
  L4 fix(ledger): preserve decimal precision in totals
  L5 test(ledger): cover voids, refunds, and export edge cases
       │
       B1'...B8' feature/export-ledger
             │
             C1'...C6' feature/tax-rules

The actual command sequence

Step 1: Freeze and announce

# In the PR comment or team channel:
# "Rewriting feature/invoice-ledger now. Please do not push to
# feature/invoice-ledger, feature/export-ledger, or feature/tax-rules
# until I post the new tips. Backup branch will be created first."

Step 2: Fetch and snapshot

git fetch origin --prune

git branch backup/invoice-ledger-before-2026-05-24 \
  origin/feature/invoice-ledger

git push origin backup/invoice-ledger-before-2026-05-24

Step 3: Create fixup commits as needed

git switch feature/invoice-ledger

# Suppose a decimal precision fix belongs in an earlier commit:
git add src/ledger/money.ts test/ledger/money.test.ts
git commit --fixup=:/ledger aggregate

# Suppose the first commit message needs a clearer scope:
git commit --fixup=reword:HEAD~20

Step 4: Rebase the whole stack and update local refs

git switch feature/tax-rules

git rebase -i --autosquash --update-refs origin/main

The interactive todo is edited into five logical base commits, followed by the export and tax-rule commits. Because --update-refs is active, local branch pointers for the dependent branches are moved when their commits are rewritten, except for branches checked out in other worktrees.

Step 5: Verify the graph and run tests

git log --oneline --graph --decorate --all \
  --branches='feature/*' --branches='backup/*' -n 80

npm ci
npm test
npm run lint
npm run typecheck

Step 6: Publish from bottom to top

git push --force-with-lease --force-if-includes \
  origin feature/invoice-ledger

git push --force-with-lease --force-if-includes \
  origin feature/export-ledger

git push --force-with-lease --force-if-includes \
  origin feature/tax-rules

Step 7: Post the recovery instructions

# Team message:
# Rewrite complete.
# Backup: origin/backup/invoice-ledger-before-2026-05-24
#
# If you had local work on the old branch:
#   git fetch origin
#   git switch your-branch
#   git rebase --onto origin/feature/invoice-ledger \
#     origin/backup/invoice-ledger-before-2026-05-24 \
#     your-branch
#
# If you see missing commits, do not run git gc. Post your reflog.

Why this worked

The rewrite worked because it treated Git history as shared infrastructure. The backup branch preserved the old public tip. Autosquash transformed fixups into logical commits. --update-refs kept local stacked branches aligned. Leased force pushes avoided overwriting unaccounted remote updates. The final message gave dependent developers a concrete rebase command instead of a vague “pull again.”

Rebasing a teammate’s local dependent branch

If a teammate had a local branch alex/reporting based on the old feature/invoice-ledger, they can transplant only their work onto the new base:

git fetch origin

git switch alex/reporting

git rebase --onto origin/feature/invoice-ledger \
  origin/backup/invoice-ledger-before-2026-05-24 \
  alex/reporting

Read the command as: “take commits reachable from alex/reporting after the old backup base, and replay them onto the new feature branch.”

10. Operational playbooks

Playbook A: Clean a private branch

git fetch origin
git rebase -i --autosquash origin/main
npm test
git push --force-with-lease --force-if-includes origin HEAD:my-branch

Use when no one else is depending on the branch. Still avoid raw --force; good habits should not depend on branch criticality.

Playbook B: Clean a shared feature branch

git fetch origin
git branch backup/my-branch-before-rewrite origin/my-branch
git push origin backup/my-branch-before-rewrite

git switch my-branch
git rebase -i --autosquash origin/main
npm test

git push --force-with-lease --force-if-includes origin my-branch

Post the backup branch and the new tip in the PR. Ask dependent branch owners to confirm they have rebased successfully.

Playbook C: Rebase a stack you own

git fetch origin
git switch top-of-stack
git rebase -i --autosquash --update-refs origin/main

git log --oneline --graph --decorate origin/main..HEAD
npm test

git push --force-with-lease --force-if-includes origin stack/01-base
git push --force-with-lease --force-if-includes origin stack/02-api
git push --force-with-lease --force-if-includes origin stack/03-ui

Playbook D: Recover an overwritten teammate commit

# On teammate machine:
git reflog --date=iso --all | grep -i "keyword"
git branch rescue/teammate-lost-work <sha>
git bundle create teammate-lost-work.bundle \
  origin/main..rescue/teammate-lost-work \
  rescue/teammate-lost-work

# On maintainer machine:
git bundle verify teammate-lost-work.bundle
git fetch teammate-lost-work.bundle \
  rescue/teammate-lost-work:refs/heads/rescue/teammate-lost-work

Playbook E: Find unreferenced objects

git fsck --lost-found --no-reflogs

find .git/lost-found/commit -type f -maxdepth 1 -print |
while read -r obj; do
  sha=$(basename "$obj")
  echo "==== $sha ===="
  git show --stat --oneline "$sha" | head -60
done

Team checklist before rewriting shared history

  • Have I fetched deliberately and inspected the remote branch tip?
  • Do I know who owns dependent branches or stacked PRs?
  • Have I pushed a backup ref or tag?
  • Have I run tests after the final rewrite, not just before it?
  • Am I using --force-with-lease --force-if-includes rather than raw --force?
  • Have I posted recovery instructions?
  • Do I know how long the backup ref will remain?

Conclusion: rewrite like an operator, not a sculptor

Clean history is valuable. It makes code review sharper, bisecting easier, release notes clearer, and architecture decisions more legible. But clean history is not worth surprising your team. The professional version of history rewriting is operational: you preserve old refs, use leases, update dependent branches, publish instructions, and keep recovery tools close.

The next time you are tempted to type git push --force, pause. Ask what you believe about the remote branch, who else might have based work on it, how you will recover if that belief is wrong, and whether your command encodes those assumptions. If it does not, you are not cleaning history. You are gambling with it.
