Static Application Security Testing is the discipline of finding security defects in a program by reading it rather than running it. A SAST tool ingests source code, bytecode, or a compiled binary; builds a mathematical model of what the program can do; and reasons over that model to prove that certain bad things — an unsanitised database query, a hard-coded key, a buffer that can overflow — are reachable. It never launches the application, never sends it a request, and needs no test harness. In the vocabulary of software assurance it is a white-box technique: it sees the whole implementation, not just the outward behaviour.

That single property — analysis without execution — is the source of both SAST's greatest strength and its deepest limitation. It lets a scanner examine every branch of a program, including the error paths and privileged code that a running test would rarely hit, and it lets that examination happen the instant a developer types, long before anything is deployed. It also means the tool must reason about behaviour it can never observe, and a fifty-year-old theorem guarantees that such reasoning can never be exactly right. Everything interesting about how SAST works — the abstract syntax trees, the control-flow graphs, the lattices and the taint propagation — is machinery for making a fundamentally undecidable problem tractable enough to be useful.

This article traces that machinery from the inside out: the theory that makes it possible and bounds what it can do, the internal representations a scanner builds, the algorithms that run over them, the difference between matching patterns and understanding semantics, and the messy engineering realities — false positives, pipeline placement, tool selection — that decide whether any of it earns a place in a real development team.

01Definition & scope

Reading code to find the flaws before an attacker does

The formal definitions converge. The OWASP Foundation describes source-code analysis tools — its preferred umbrella for SAST — as tools that analyse source code or compiled versions of code to help find security flaws, and notes they can be embedded directly in a developer's editor.1 Practitioner literature adds the operational detail: SAST is a highly automated method that inspects source, bytecode, or binary during the early stages of the software development lifecycle, identifying injection flaws, cross-site scripting, insecure data handling, and the other weaknesses catalogued in industry standards such as the OWASP Top 10 and the MITRE Common Weakness Enumeration.2,3 The essential move is always the same: the code is treated as a static artifact to be parsed and reasoned over, not as a process to be exercised.

It helps to fix SAST's place in the wider family of application-security testing, because the categories are routinely confused. Four techniques dominate:

  • SAST (static) reads the code at rest. White-box, no execution, earliest possible feedback, and — crucially — it points at the exact line and the exact data-flow path that produced a finding, which is why it is prized for pinpointing the root cause of a defect rather than just its symptom.
  • DAST (dynamic) attacks a running application from the outside, black-box, sending crafted inputs and observing responses. It sees real runtime behaviour and configuration but cannot say which line is at fault, and it can only test paths it manages to reach.
  • IAST (interactive) instruments a running application from the inside, combining code visibility with live execution; it is essentially a hybrid that trades SAST's zero-setup for DAST's runtime fidelity.
  • SCA (software composition analysis) ignores your own code and instead inventories third-party dependencies, matching them against databases of known vulnerable versions. It answers a different question — not “did we write a bug?” but “did we import one?”

These are complements, not competitors. A mature program runs all four, and modern platforms increasingly bundle them. But SAST is the one that lives closest to the keyboard, and it is the only one that can find a vulnerability in code that has never been built or run even once.

The defining constraint of SAST is not that it looks at source code. It is that it must understand behaviour without ever observing it.

Because it operates on the implementation itself, SAST is also the technique most tightly coupled to a specific programming language. A DAST scanner speaks HTTP and is largely language-agnostic; a SAST engine must have a parser, a type model, and a library of security rules for every language and framework it supports. This is why tool coverage is described in languages — a scanner that is excellent for Java may be useless for Rust — and why the internal representations described in the next sections are built afresh for each language a tool understands.

02Origins

From lint traps to taint graphs

The lineage of SAST runs through two parallel streams that only converged in the 2000s: a stream of practical bug-finding tools, and a stream of program-analysis theory. Understanding SAST means understanding both, because every modern scanner is a theoretical idea wearing an engineer's clothes.

The first tool: Lint, 1978

The practical stream begins at Bell Labs. In 1978, while wrestling with portability bugs that surfaced as Unix was ported to a 32-bit machine, Stephen C. Johnson wrote a program to catch the mistakes that C compilers of the day sailed straight past — type mismatches across function boundaries, non-portable constructs, unused variables.4 He called it lint, after the fluff a clothes-dryer trap captures while leaving whole fabric intact, and it shipped to the world in 1979 with Version 7 Unix.5 Lint established the template that every SAST tool still follows: a separate pass over source code that flags suspicious constructs a compiler will happily accept. The whole modern category of “linters” carries its name.

The theory: data-flow analysis and abstract interpretation

The theoretical stream began five years earlier and reached deeper. In 1973 Gary Kildall unified the ad-hoc optimisations compilers were doing into a single framework, showing that a large class of program analyses could be expressed as the propagation of facts over a lattice until a fixpoint is reached.6 Frances Allen had already given compilers the control-flow graph as their organising structure.7 John Kam and Jeffrey Ullman generalised Kildall's conditions to the broader class of monotone frameworks that practical analyses actually satisfy.8 Then, in 1977, Patrick and Radhia Cousot delivered the field's crowning abstraction: abstract interpretation, a theory of sound approximation in which a program's real semantics are systematically replaced by a simpler abstract semantics over a lattice, so that finite computation can safely over-approximate infinite behaviour.9 This is the mathematics underneath essentially every serious static analyzer built since. The standard treatment of the whole apparatus — lattices, transfer functions, fixpoints, Galois connections — is Nielson, Nielson and Hankin's Principles of Program Analysis.10

Meta-compilation and the rise of commercial bug-finding

The streams merged around the turn of the millennium at Stanford. Dawson Engler's group argued that developers, not tool vendors, knew the rules their systems had to obey, and built a mechanism — meta-level compilation — that let programmers write small, system-specific checkers as compiler extensions.11 A companion insight, “bugs as deviant behaviour,” showed that a tool could infer likely rules statistically: if code calls lock before touching a structure ninety-nine times and forgets once, the once is probably a bug.12 They followed with a dedicated analysis language and engine,13 and in 2002 spun the work out as Coverity. Their retrospective on running that analyzer against tens of millions of lines of real-world code — the frank, funny A Few Billion Lines of Code Later — remains the best account of what actually happens when static analysis meets industrial software.14

The same years produced the commercial SAST market. Fortify Software was founded in 2003 in San Mateo and built the source-level, interprocedural taint engine that many enterprises still run; it passed to Hewlett-Packard in 2010, then Micro Focus, then OpenText.15 Checkmarx (2006) and Veracode (2006) rounded out the enterprise field; Veracode, founded by former L0pht and @stake researchers Chris Wysopal and Christien Rioux, took the unusual path of analysing compiled binaries rather than source.16 On the academic-tool side, Bill Pugh's group at the University of Maryland released FindBugs, whose “finding bugs is easy” thesis — that cheap bytecode pattern-matching catches a surprising number of real defects — put static analysis in the hands of ordinary Java developers.17,18 Brian Chess and Gary McGraw's Static analysis for security and McGraw's building security in movement reframed all of this explicitly around security rather than mere correctness.19,20

Code as data, and the developer-first turn

Two later ideas define the current era. The first is treating code as a database. Oxford research on querying source with Datalog — the CodeQuest system — became Semmle and its query language QL; GitHub acquired Semmle in 2019, made the engine's queries free for open source, and rebranded it CodeQL, which now compiles a program into a relational representation of its syntax, control flow, and data flow that analysts query for vulnerable patterns.21,22,23 The second is the developer-first, lightweight turn embodied by Semgrep: descended from a Facebook tool called sgrep (part of the pfff library, itself inspired by the Coccinelle program-transformation engine), it was reborn at the startup r2c around 2020 as a scanner whose rules look like the code they match, deliberately trading some depth for speed and approachability.24,25 Between the queryable database and the grep-like rule, these two tools bracket the design space that the rest of this article dissects.

03The bedrock

Why perfect static analysis is impossible — and useful anyway

Before any AST or algorithm, one result governs the entire field. In 1953 H. G. Rice proved that every non-trivial semantic property of a program is undecidable.26 A semantic property is one about behaviour — “does this program ever dereference a null pointer?”, “can attacker input reach this query?” — as opposed to a syntactic one like “does the source contain the word eval?”. Non-trivial means it holds for some programs and not others. The presence of a security vulnerability is the canonical example of exactly such a property. Rice's theorem, a generalisation of Turing's undecidability of the halting problem,27 therefore delivers an uncomfortable verdict: there is no algorithm that, for every program, always terminates and correctly answers whether it contains a given class of vulnerability.

If that were the end of the story, SAST could not exist. The escape hatch is to abandon the demand for a correct yes/no answer on every input and accept a one-sided error instead. An analysis can be built to always answer “safe” only when it is genuinely certain, and to answer “possibly vulnerable” whenever it cannot rule the danger out. Such an analysis is sound for the property: it never misses a real instance, at the price of sometimes flagging things that are actually fine. The formal engine for constructing such analyses in a principled way is precisely the Cousots' abstract interpretation — deliberately over-approximating the set of states a program can reach so that finite computation stays on the safe side of the truth.9

Soundness, completeness, and the two kinds of lie

Two failure modes bound every tool, and the vocabulary matters:

  • A false positive is a finding the tool reports that is not a real defect. Too many, and developers stop reading the output. The academic term for a tool that produces none is complete.
  • A false negative is a real defect the tool fails to report. Too many, and the tool provides false confidence. A tool that produces none is sound.

Rice's theorem says no tool can be both sound and complete on a Turing-complete language. Worse, recent work sharpens the geometry of the trade-off: any decidable over-approximation of a non-trivial property necessarily admits an infinite set of false positives. Perfect recall and perfect precision are not merely hard to hit simultaneously; the second is mathematically forbidden once you insist on the first.

“Soundiness”: the honest compromise every real tool makes

In practice, even tools that aspire to soundness quietly give it up in specific places. Reflection, dynamic code loading, eval, native calls, and heavy dependency injection defeat static reasoning; a genuinely sound treatment of them would flag so much that the tool would be useless. So real analyzers are sound for a well-behaved core of the language and deliberately unsound for the hard features — a stance a group of leading researchers formalised as soundiness in a 2015 manifesto, arguing that the virtue is not pretending to total soundness but being explicit about exactly which language features you do and do not model.28 This is not a defect of the discipline; it is the discipline. Every SAST result you read is an assertion within a modelled fragment of the language, and knowing the boundary of that fragment is what separates competent use of a tool from cargo-cult trust in it.

The consequence for the rest of this article is that all the sophistication to come — path sensitivity, interprocedural taint, points-to analysis — is not an attempt to reach the impossible top-right corner. It is an attempt to move as far toward it as the compute budget allows, shrinking both kinds of lie without ever eliminating either.

04Under the hood

The representations a scanner builds before it reasons

A SAST engine cannot reason about a text file. Its first job is to transform raw source into a series of progressively richer representations, each one exposing structure that the previous one hid. The pipeline below is essentially universal; where tools differ is how far along it they travel and how much precision they preserve at each step.

Lexing and parsing → the abstract syntax tree

Analysis begins where a compiler begins. A lexer turns the character stream into tokens; a parser assembles those tokens, according to the language grammar, into an abstract syntax tree (AST). The AST is “abstract” because it throws away everything inessential — parentheses, semicolons, whitespace, the literal spelling of operators — and keeps only the hierarchical structure of the program's meaning. An assignment becomes an Assign node with a target and a value; a string concatenation becomes a BinOp with two children. Pattern-matching tools operate largely at this level, which is why a scanner like Semgrep can match code by its structure rather than its exact text: db.execute(x) and db . execute( x ) parse to the identical AST.24

The AST already carries more than raw syntax. A good extractor annotates each node with the results of name resolution (which declaration does this identifier refer to?) and type information (what is its static type?). CodeQL's extractors, for instance, record exactly this syntactic and semantic data before importing it into the queryable database.23 That metadata is what lets an analysis distinguish a method call on a database connection from an identically named method on some harmless object — a distinction on which the accuracy of every security rule depends.

Control-flow graph → the order of things

An AST captures structure but not order of execution: it says a loop body exists, not that it may run zero times, nor which statements can follow which. To reason about what happens when the program runs, the engine lifts the AST into a control-flow graph (CFG), the representation Frances Allen introduced in 1970.7 Its nodes are basic blocks — maximal straight-line runs of instructions with no branches in or out except at their ends — and its edges are the possible transfers of control between them. Every conditional becomes a fork; every loop, a back-edge. The CFG is where an analysis can finally ask questions like “on which paths does this variable reach that call?”.

A key precision axis lives here. A path-sensitive analysis reasons about branch conditions and can conclude that the vulnerable code in one branch is unreachable when a guard holds; a cheaper path-insensitive analysis merges information at every join point and may conflate the safe and unsafe branches. Path sensitivity buys accuracy at real computational cost, and how much of it a tool affords is one of the main things that separates a fast linter from a deep enterprise engine.

SSA form → making data dependencies explicit

To track how values flow, many engines convert the CFG into static single assignment (SSA) form, the intermediate representation in which every variable is assigned exactly once. Where control-flow paths merge and two definitions of a variable meet, SSA inserts a special φ (phi) node to reconcile them. The efficient algorithm for placing those phi-functions, using the notion of dominance frontiers, was given by Cytron and colleagues in 1991 and is now standard in production compilers and analyzers alike.29 SSA's payoff for SAST is directly relevant: because each value has a single, uniquely named definition, the chain from where a value is defined to where it is used becomes trivial to follow — and following such chains is exactly what taint tracking does.

Going interprocedural: call graphs and points-to analysis

Everything so far concerns a single function. Real vulnerabilities cross function and file boundaries: untrusted input arrives in one method, is stored, and is used unsafely in another — a pattern the industry calls second-order injection. To follow data across calls, an engine needs a call graph (which functions can invoke which) and, in languages with pointers or references, a points-to (alias) analysis that determines which variables may refer to the same memory or object. The foundational algorithms here are Lars Andersen's inclusion-based analysis30 and Bjarne Steensgaard's near-linear-time unification-based approximation,31 which trade precision against scalability in opposite ways. Points-to analysis is notoriously the hardest and most expensive part of the pipeline, and the ability to do cross-file, interprocedural data-flow is the single feature that most cleanly divides deep commercial engines from lightweight open-source scanners: only a handful of tools trace tainted data reliably across file boundaries at scale.

05The core algorithm

Data-flow analysis: propagating facts to a fixpoint

With a control-flow graph in hand, the engine can run the workhorse of static analysis: data-flow analysis. The goal is to compute, at every program point, a sound over-approximation of some property — which definitions can reach here, which variables might be null, which values might be tainted. Kildall's 1973 insight was that an enormous variety of such analyses share one abstract structure, and that structure is worth stating precisely because taint analysis is simply one instance of it.6

The three ingredients

Every data-flow analysis is defined by three things:

  1. A lattice L of abstract facts — a partially ordered set with a meet operation (⊓) that combines information arriving from different paths, plus a top (⊤) and bottom (⊥). The lattice must have finite height for the analysis to terminate.
  2. A transfer function f[b] for each basic block, describing how that block transforms the incoming facts into outgoing facts. For soundness these functions must be monotone: more input information never produces less output information.8
  3. A direction and a meet: forward analyses (like reaching definitions and taint) push facts along control-flow edges and combine at merge points; backward analyses (like liveness) push the other way.

The lattice is the algebra of what the analysis knows. Taint tracking uses the simplest interesting one: a two-point lattice where clean sits below tainted, and the meet of anything with tainted is tainted — so if data arriving on any path is tainted, the merge is tainted. A sign analysis uses a slightly richer lattice; a constant-propagation analysis, richer still. Choosing the lattice is choosing the question.

Solving by iteration: the worklist algorithm

Given these ingredients, the solution is computed not by cleverness but by brute, guaranteed-to-converge iteration. Initialise every point to the lattice bottom, then repeatedly recompute each block's facts from its neighbours until nothing changes. Because the lattice has finite height and the transfer functions are monotone, facts only ever move upward, so the process must reach a stable least fixpoint — the tightest sound answer the abstraction admits. The classic implementation keeps a worklist of blocks whose inputs have changed:

monotone_dataflow.pypseudocode
# Generic monotone data-flow analysis by fixpoint iteration.
# L    : lattice of facts, with meet (⊓) and partial order (⊑)
# f[b] : monotone transfer function for basic block b
# cfg  : nodes = basic blocks, edges = possible control transfers

def analyze(cfg, L, f):
    # 1. every block starts at ⊥ ("nothing known yet")
    IN  = {b: L.bottom for b in cfg.nodes}
    OUT = {b: L.bottom for b in cfg.nodes}
    work = deque(cfg.nodes)          # all blocks start dirty

    while work:
        b = work.popleft()
        # 2. combine facts from every predecessor with the meet
        IN[b] = L.meet(OUT[p] for p in cfg.preds(b))
        # 3. push them through this block's transfer function
        new_out = f[b](IN[b])
        # 4. if the output moved, successors must be recomputed
        if new_out != OUT[b]:
            OUT[b] = new_out
            work.extend(cfg.succs(b))

    return IN, OUT   # least fixpoint — termination guaranteed
The engine of nearly all SAST: iterate the monotone transfer functions over the CFG until the lattice facts stop changing. Finite lattice height + monotonicity ⇒ convergence (Kildall 1973; Kam & Ullman 1977).

This single algorithm, instantiated with different lattices and transfer functions, computes reaching definitions for a compiler, nullability for a bug-finder, and — with a taint lattice and transfer functions that model sources, sinks, and sanitisers — the security analysis at the heart of SAST. The interprocedural version, which threads facts through function calls precisely rather than merging them crudely, was given its definitive formulation by Reps, Horwitz, and Sagiv, who reduced precise interprocedural data-flow to a graph-reachability problem in their 1995 IFDS framework — the theoretical basis for how modern engines follow taint across call boundaries.32

06The security overlay

Taint analysis: sources, sinks, and the paths between them

Data-flow analysis is generic; taint analysis is the specialisation that turns it into a security tool. The idea is to overlay security semantics on the analyzer's data-flow graph by declaring three kinds of program location:

  • Sources introduce untrusted or sensitive data — an HTTP request parameter, a form field, a file read, a network socket, an environment variable. Data from a source is marked tainted.
  • Sinks are security-sensitive operations that must not receive tainted data — a SQL query executor, a shell command, an HTML response writer, a file path, a deserializer.
  • Sanitisers (also called downgraders or validators) neutralise taint — a parameterised-query binding, an output encoder, an allow-list check. Passing tainted data through a sanitiser clears its taint bit.

The rule is then elegantly simple, and reduces security analysis to graph reachability: a vulnerability exists wherever there is a data-flow path from a source to a sink that does not pass through an appropriate sanitiser.32 Each such report is a “witness” — a concrete sequence of statements from the point untrusted data enters to the point it is used dangerously. These source/sink/sanitiser specifications are typically organised around the CWE catalogue, each weakness type carrying its own characteristic triples.3

A concrete vulnerability

Consider a textbook SQL-injection flaw (CWE-89). The endpoint reads an identifier from the query string and splices it directly into a SQL statement:

app/views.py · vulnerablepython
from flask import request

@app.route("/user")
def get_user():
    uid   = request.args.get("id")                  # ◄ SOURCE: attacker-controlled
    query = "SELECT * FROM users WHERE id = " + uid  # taint propagates
    cur   = db.execute(query)                       # ◄ SINK: raw SQL, no sanitiser
    return cur.fetchall()
A source-to-sink path with no sanitiser on it. Passing id=1 OR 1=1-- returns every row; worse payloads read or write arbitrary data.

The engine finds this not by matching the string execute — that would flag every safe query too — but by proving that the value handed to db.execute derives, through the concatenation, from request.args.get. Change the last two lines to bind the input as a parameter, and the taint is broken at the sink; the tool reports nothing because no tainted path remains:

app/views.py · fixedpython
    uid = request.args.get("id")                       # still a source…
    cur = db.execute("SELECT * FROM users WHERE id = ?", [uid])  # ✓ parameterised: taint cleared
The sanitiser is structural: the driver binds uid as data, never as SQL text, so it can never alter the query's meaning.

Why this is hard to do well

Getting taint analysis right at scale is where the deepest engineering lives. The path may cross many functions and files; the source data may be stored in a field and retrieved later; a sanitiser may be effective for one sink type (HTML encoding) but useless for another (SQL). The analysis must be interprocedural, ideally field- and object-sensitive, and must model the framework: a rule that does not know Flask's request object, or Spring's request mappings, will miss the source entirely. The FlowDroid analyzer for Android is a landmark demonstration of how much apparatus this demands — context-, flow-, field-, and object-sensitive taint tracking that is also aware of the application lifecycle.33 Commercial engines from Fortify, Checkmarx, Coverity, and Veracode, and the taint engines in CodeQL, SonarQube, and Semgrep's paid tier, all invest heavily in exactly this precision, because a taint tracker that only works within a single function catches only the most naive bugs.34

07Two philosophies

Pattern matching versus semantic understanding

Not all SAST tools reason at the same depth, and the industry has settled into a spectrum with two poles. At one end are pattern-based tools that match structural templates against the AST. At the other are semantic engines that build a full model of the program — control flow, data flow, type hierarchy, call graph — and reason over it. The distinction is not academic: it determines which bugs a tool can even express, let alone find.

The pattern end: rules that look like code

The lightweight approach, exemplified by Semgrep, is essentially a structural search-and-match over the syntax tree, enriched with limited data-flow. Its great virtue is approachability: a rule looks like the code it targets, with metavariables standing in for the parts you do not want to pin down, so a developer can write a useful check without a background in program analysis.24,25 Semgrep's taint mode lets a rule declare sources, sinks, and sanitisers directly:

rules/python-sqli.yamlsemgrep
rules:
  - id: python-sqli-string-concat
    languages: [python]
    severity: ERROR
    mode: taint                        # track flow, not just shape
    message: Untrusted input reaches a SQL execute() call.
    pattern-sources:
      - pattern: flask.request.$ANYTHING
    pattern-sinks:
      - pattern: $CURSOR.execute(...)
    pattern-sanitizers:
      - pattern: $CURSOR.execute("...", ...)   # parameterised form is safe
A Semgrep taint rule. $CURSOR and $ANYTHING are metavariables. The rule reads almost like the vulnerable code itself — the design goal of the pattern-first school.

The cost of this approachability is depth. Community-edition pattern tools analyse largely within a single function or file, and their maintainers say so plainly — cross-function, cross-file taint is reserved for the heavier engines.24 For a large class of real bugs that is enough, and running fast enough to sit in a pre-commit hook is often worth more than catching the last exotic flow.

The semantic end: code as a queryable database

The deeper approach treats the program as data. CodeQL compiles a codebase into a relational database of facts — every AST node, every control-flow edge, every data-flow relationship, the whole type hierarchy — and then executes queries against it in QL, a declarative language whose syntax resembles SQL but whose semantics come from Datalog.23,21 Because the entire program is a structure you can join over, a query can traverse arbitrary call chains and express a vulnerability as a logical predicate:

Sqli.ql · simplifiedcodeql / QL
/** Untrusted request data flowing into a SQL execute() call. */
import python
import semmle.python.dataflow.new.TaintTracking

module SqliConfig implements DataFlow::ConfigSig {
  predicate isSource(DataFlow::Node n) {
    n.asExpr() instanceof RequestArgsAccess          // framework-modelled source
  }
  predicate isSink(DataFlow::Node n) {
    exists(Call c |
      c.getFunc().(Attribute).getName() = "execute" and
      n.asExpr() = c.getArg(0))
  }
}
module SqliFlow = TaintTracking::Global<SqliConfig>;

from DataFlow::Node source, DataFlow::Node sink
where SqliFlow::flow(source, sink)
select sink, "Untrusted data from $@ reaches a SQL sink.", source, "here"
The same vulnerability class expressed as a global taint-tracking query. The engine handles the interprocedural search; the analyst only declares what counts as a source and a sink. This is the “code as data” paradigm that variant analysis with CodeQL has used to uncover hundreds of CVEs.22

Between these poles sit the deep commercial engines — Fortify, Checkmarx, Coverity — which perform whole-program interprocedural taint with proprietary abstractions, and the “bugs as deviant behaviour” style of statistical inference that Coverity pioneered, in which the tool learns a codebase's own implicit rules and flags the deviations.12 The right mental model is a dial from cheap-and-shallow to expensive-and-deep, and a serious program often runs tools from more than one point on it.

08What it can and cannot do

Strengths, weaknesses, and the false-positive tax

Where SAST is uniquely strong

  • Earliest possible feedback. Because it needs no running system, SAST can run in the editor as code is typed and on every commit — the cheapest moment to fix anything. This is the concrete meaning of “shift left.”
  • Total path coverage. A running test only exercises the paths it happens to hit; a static analyzer examines every branch, including rare error paths and privileged code that dynamic testing routinely misses.
  • Root-cause precision. A finding points at the exact line and, for taint results, the exact source-to-sink path. Developers get the cause, not just an externally observed symptom, which makes fixes fast and verifiable.
  • Whole-codebase reach, including legacy. It scans code that is hard to run, hard to reach, or simply old — large legacy systems where manual review is infeasible and no test harness exists.
  • Consistency and compliance. The same rules apply uniformly across millions of lines, and coverage maps cleanly onto standards — the OWASP Top 10, CWE, PCI DSS, and similar — which is why SAST is a fixture of regulated software assurance.2,3

Where it is structurally blind

  • No runtime or configuration context. SAST cannot see environment variables, deployment configuration, real authentication state, or how components actually interact at run time. Business-logic flaws and authorisation errors that depend on runtime state largely elude it — this is DAST's and IAST's territory.
  • Framework and dynamism blindness. Reflection, eval, dynamic dispatch, metaprogramming, and unmodelled frameworks break the analysis; a rule that does not know a framework's request object cannot see the source. This is the practical face of the soundiness compromise.28
  • It finds your code's bugs, not your dependencies'. A vulnerable third-party library is invisible to SAST; that is SCA's job.
  • Cost and scale. Deep interprocedural and points-to analysis is computationally expensive; full scans of large codebases can take from minutes to hours, which is why incremental, diff-aware scanning matters so much in practice.

The hardest problem in SAST is not detection. It is earning enough developer trust that the findings get read at all.

The false-positive tax

The single greatest obstacle to SAST's value is noise. Because sound analysis over-approximates, it reports flows that are technically possible but practically infeasible, and untuned scanners commonly generate false positives at rates high enough to bury the real findings. NIST's Static Analysis Tool Exposition evaluations found that only a minority of tool warnings were security-relevant, and independent measurements of large codebases routinely report false-positive rates well into the tens of percent.35 The human cost is documented: the classic study of why developers abandon static analysis tools traced it directly to false positives, poor result presentation, and workflow friction — not to any doubt that the underlying bugs were real.36 When a team investigates twenty findings, discovers fifteen are noise, and stops looking, the next genuine SQL-injection sits unread in the backlog.

The most important lesson from operators running SAST at the largest scale is that fix rate, not find rate, is the real metric. Google's account of building static analysis for its monorepo stresses integrating results into the developer workflow, showing them at code-review time, and ruthlessly suppressing checks that developers do not act on.37 Facebook's Infer team reached the same conclusion from a different direction: reporting issues on the specific code a developer is changing, at review time, dramatically raised the rate at which findings were actually fixed compared with dumping a batch report on an existing codebase.38 Both reframed the problem from “how many bugs can we find?” to “how many do we get fixed without eroding trust?” — the question that actually determines whether a SAST deployment succeeds. Techniques to rank and triage findings statistically, such as Z-ranking, exist precisely because raw output is unusable without prioritisation.39 To let teams compare tools on this axis objectively, the OWASP Benchmark seeds a deliberately insecure application with thousands of known true and false vulnerabilities and plots each tool on a true-positive-versus-false-positive chart whose diagonal marks the line of random guessing.40

09In the workflow

Where SAST fits in the DevOps pipeline

SAST's zero-execution property makes it the earliest automated security gate, and modern practice threads it through the whole development lifecycle rather than bolting it on at the end. When static analysis is embedded in the continuous-integration and continuous-delivery pipeline, the result is what the industry calls DevSecOps: security testing as an automatic, continuous part of building software rather than a separate audit.1 The guiding principle, articulated in the “building security in” movement, is that finding a defect during development costs a fraction of finding it in production.20

In a well-run pipeline, SAST appears at several points, each with a different depth/latency trade-off:

  • In the IDE, a lightweight analyzer or plugin flags issues as the developer types, before the code is even saved — the cheapest feedback of all, and the reason many vendors ship editor integrations.
  • On commit, a fast pre-commit hook runs quick rules so that obvious problems never enter version control. Speed matters more than depth here; a scan that takes minutes will be disabled.
  • On the pull request, the CI pipeline runs a deeper scan and — critically — a diff-aware one, reporting only issues introduced by the change rather than the entire backlog. This is the operating point where the Google and Facebook experience says findings actually get fixed, because they are attached to the code under review.37,38 Here a quality gate can block the merge if new high-severity issues appear.
  • On a schedule, a full, slow, whole-program scan runs nightly or weekly to catch deep interprocedural flows and to track the security posture of the entire codebase over time.

Speaking a common language: SARIF

Threading many tools through one pipeline created a Babel problem: every scanner emitted findings in its own format, making aggregation painful. The answer is the Static Analysis Results Interchange Format (SARIF), a JSON-based standard for static-analysis output ratified by OASIS as version 2.1.0 in 2020.41 SARIF lets a low-cost tool plug into a rich ecosystem of viewers and dashboards, and lets an organisation merge the output of many scanners into a single view — the mechanism behind, for example, code-scanning dashboards that ingest results from CodeQL, Semgrep, SpotBugs, Brakeman, and more. A minimal result carries the rule, a severity level, a message, and a precise location:

10The landscape

The tools: enterprise, open-source, language-specific

The market splits along two axes: commercial versus open-source, and general-purpose versus language-specific. No single tool is best for everyone, and OWASP explicitly declines to endorse any of them, maintaining only a neutral catalogue.1 The pragmatic pattern in serious programs is a layered one — a fast, developer-facing scanner in the pull-request loop, a deep engine on a schedule, and language-specific linters filling gaps — unified through SARIF.

Enterprise engines (Fortify, Checkmarx, Veracode, Coverity) buy depth, support, broad language coverage, and compliance reporting at significant cost and configuration weight; they remain the standard for large, regulated organisations that need cross-file taint and audit trails. Language-specific scanners (SpotBugs+Find Security Bugs, Bandit, Brakeman, gosec, ESLint) trade breadth for intimacy with one ecosystem's idioms and frameworks, and are cheap enough to run on every commit — several are the built-in analyzers behind larger platforms' SAST features.

11Doing it well

Best practices: from scanner to security outcome

Buying a SAST tool is easy; getting security value from it is not. The failure mode is universal — a scanner is switched on, produces a mountain of findings, overwhelms and alienates developers, and is quietly ignored. The practices that avoid it are consistent across the operators who have run static analysis at the largest scale.37,38

  1. Integrate into the workflow; do not bolt on at the end. Findings that appear in the editor and at code-review time, on the code being changed, get fixed; batch reports delivered to a separate dashboard do not. Meet developers where they already work.
  2. Scan diffs, and baseline the backlog. Report only issues a change introduces, and set aside pre-existing findings as a managed baseline. A team that must fix three hundred legacy warnings before shipping will simply turn the tool off; a team that must fix only what it just added will keep it on.
  3. Measure fix rate, not find rate, and tune ruthlessly. A check whose findings developers consistently dismiss is worse than no check — it trains people to ignore output. Suppress it. Use severity, confidence, and statistical ranking to surface the few findings that matter first.39 Track how many findings are actually remediated, and treat that number as the real score.37
  4. Write custom rules for your own invariants. The generic rule set does not know your frameworks, your internal sanitisers, or your organisation's forbidden APIs. Modern tools — Semgrep's YAML, CodeQL's QL, Checkmarx's CxQL — exist precisely so teams can encode security knowledge that a vendor never could. This is where a program moves from catching textbook bugs to enforcing its own security architecture.24,23
  5. Give developers actionable output. Every finding should carry the data-flow path, a clear explanation of why it is dangerous, and concrete remediation guidance. The research on tool abandonment points squarely at poor result presentation as a primary cause of failure, independent of detection quality.36
  6. Know — and document — the soundiness boundary. Understand which language features and frameworks your tool does not model, so its silence is never mistaken for a guarantee. A clean SAST run means “no findings within the modelled fragment,” not “no vulnerabilities.”28
  7. Treat SAST as one layer of defence in depth. It cannot see runtime behaviour, configuration, or dependency vulnerabilities. Pair it with DAST or IAST for runtime coverage and SCA for third-party risk. Cross-validating SAST findings against a running-application test is one of the most reliable ways to separate real exploitable flaws from theoretical ones.
12Coda

The shape of the problem does not change

The frontier of SAST today is the marriage of program analysis with machine learning. Large language models are being used as triage assistants — reading a tool's raw output and filtering the false positives that traditional ranking misses — and as generators of taint specifications and even of analysis queries. Early results are genuinely promising: LLM-based post-processing can discard a substantial fraction of false positives while preserving true findings, and hybrid pipelines that combine a classical analyzer's soundness with a model's contextual judgement point toward tools that are both thorough and quiet.

But it is worth being precise about what such tools can and cannot escape. An analyzer — classical or neural — that claimed to detect every vulnerability in every program with no false alarms would be claiming to decide an undecidable problem, and Rice's theorem forbids it as firmly for a transformer trained on billions of lines of code as for a hand-written data-flow pass.26 What machine learning changes is not the boundary but the nature of the approximation: where classical analysis uses formal abstractions with predictable failure modes, learned models use statistical patterns with different, sometimes complementary, blind spots. The most effective systems will almost certainly be hybrids that play the two against each other — the sound over-approximation of abstract interpretation to guarantee coverage, the learned model to prune the noise.

Which returns us to where the field began. Kildall's lattices, the Cousots' abstract interpretation, Reps and Sagiv's reachability, Engler's deviant behaviour, Semmle's queryable database, Semgrep's approachable rules — each was an answer to the same permanent question: how to say something rigorous and useful about a program's behaviour without ever running it, in a world where saying anything perfectly is provably impossible. SAST is the accumulated, practical, imperfect answer. Its value was never that it is right about everything. It is that, read at the right moment and tuned with care, it is right about enough — early enough, and cheaply enough — to matter.14
