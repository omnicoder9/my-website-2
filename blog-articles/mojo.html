Mojo: the language that wants to make your GPU go brrr

A Python-shaped systems language, an MLIR-shaped compiler, and a bet that the two-language problem can be retired through patient first-principles design.

Origins

Mojo is a programming language designed by Chris Lattner and developed by Modular Inc., a startup he co-founded in 2022 with Tim Davis, a former leader of Google's machine-learning APIs, compilers, and runtime infrastructure.3 Lattner's résumé is the kind that lends a new language gravitational pull on day one: he created the LLVM compiler infrastructure as a graduate project at the University of Illinois, then went on to co-found the MLIR project and architect Apple's Swift programming language.22 Mojo is, in a sense, the third act in a long meditation on what a compiler ought to look like.

The language was not the original product. Modular set out to build an AI inference engine and a portable code generator for heterogeneous hardware. According to Lattner's own account, "We weren't originally intending to build a language at Modular. We started building a very fancy code generator that had no frontend."8 The frontend grew out of necessity. Programming the modern AI stack — CPUs with matrix extensions, GPUs with warp primitives, TPUs and other accelerators — demanded a way to express common kernel-development patterns in MLIR, and the team eventually generalized those patterns into a language suitable for high-performance kernel development across all of them.21

The first publicly testable version of Mojo became available in May 2023 as a hosted JupyterHub called the Mojo Playground.1 Modular announced the language alongside its Modular Inference Engine, and the initial demos circulated widely thanks to an attention-grabbing benchmark: a Mandelbrot rendering kernel was reported as 35,000 times faster than equivalent Python after hardware acceleration was applied.3 Jeremy Howard of fast.ai wrote at the time that "Mojo isn't finished — but what's there is already mind-blowing, and it has been created by a very small team in a very short time."3

Local downloads followed in September 2023, beginning with Linux and adding macOS shortly afterward; by then more than 120,000 developers had signed up for the Playground and over 19,000 were active in the Discord and GitHub communities.23 The standard library was open-sourced under the Apache 2.0 license with LLVM exceptions in March 2024, an important milestone for a language that had begun life as proprietary software in incubation inside its parent company.15

Versions and trajectory

Mojo's release history is unusual in two respects. Modular adopted a calendar-style versioning scheme (year-dot-quarter, more or less) that produces version numbers like 24.2, 25.7, and 26.3 rather than the customary 0.x sequence. And the language remained explicitly pre-1.0 for nearly three years before entering beta — a deliberate pacing choice for a project whose authors did not want to commit to stability prematurely.

The recent simplifications are worth pausing on. Earlier versions of Mojo distinguished between def, which preserved Python-style dynamic semantics, and fn, which imposed stricter, performance-oriented typing rules. As of 26.2 that distinction has been dissolved into a single declaration form — a small change but a telling one, suggesting the language has grown confident enough in its own type system to stop hedging.10

Technical merits

Mojo's technical pitch begins with a substrate choice that almost no other production language has made: it is built from the ground up on MLIR — the Multi-Level Intermediate Representation framework that Lattner co-founded at Google.7 Where LLVM gave Swift, Clang, and Rust a single intermediate representation to lower to, MLIR offers a stack of dialects, each modelling a different layer of abstraction, from generic linear algebra down to specific accelerator instruction sets. According to Modular's vision document, "while you can use MLIR to create a flexible and powerful compiler for any programming language, Mojo is the world's first language to be built from the ground up with MLIR design principles."7

So, if Swift was "syntax sugar for LLVM," what's "syntax sugar for MLIR"? The answer is: Mojo.
— Jeremy Howard, fast.ai, on Mojo's launch

The practical consequence is that the same Mojo source can target heterogeneous hardware — CPUs, GPUs from different vendors, and AI accelerators — without the platform-specific rewrites that have historically forced developers to choose between CUDA, ROCm, Metal, or hand-rolled SIMD intrinsics.4 Mojo's compiler framework, internally code-named KGEN (for "kernel generator"), allows explicitly parametric code to be represented before instantiation, yielding faster compile times, clearer error messages, and the ability to compile the same source to multiple target devices.7

The language's other technical features round out a familiar but carefully composed list of systems-language capabilities:

Hybrid typing

Mojo supports static, dynamic, inferred, nominal, duck, strong, and affine typing within a single coherent system.1 Python developers can begin in a permissive mode and gradually opt into stricter guarantees. As Lattner has described it, "CPU and GPU high performance numeric programmers never want anything dynamic. They want full control over the machine […]. But we also allow you to start removing types, so you can get fully dynamic behavior if you'd like."8

Compile-time metaprogramming

The same Mojo language is used for compile-time programs as for runtime ones, with full support for imperative comptime logic — control flow, recursion, the lot — implemented as a distinct compilation stage between semantic analysis and lowering.18 The parameter system lets the compiler generate specialised versions of types or functions based on parameter values, much like C++ templates but with friendlier syntax. Generic structs declare their parameters between square brackets, mirroring Python's type hint conventions.18

Ownership-based memory management

Mojo borrows the ownership concept that Rust popularised, providing memory safety without garbage collection.25 Recent versions have added linear types (explicitly destroyed types) that the compiler statically guarantees cannot be leaked.10

SIMD-first numeric model

SIMD is treated as a first-class type rather than a wrapper over compiler intrinsics, encouraging programs to express vector operations directly. Combined with parallelism primitives that target multi-core CPUs and GPUs, this yields the headline numbers Mojo has been advertised on.25

Traits and zero-cost abstractions

Mojo's trait system, layered with conditional conformances via where clauses and default trait methods for mixin-style composition, has been a focal point of the language's roadmap to 1.0.9

A small Mojo snippet conveys the flavour better than a paragraph can. The following — drawn loosely from Modular's own kernel examples — illustrates compile-time specialisation for the target hardware:13

Mojo · GPU warp synchronization 
# Compile-time warp synchronization per hardware
@always_inline("nodebug")
fn syncwarp(mask: Int = -1):
    """Synchronizes threads within a warp using a barrier."""
    @parameter
    if is_nvidia_gpu():
        __mlir_op.`nvvm.bar.warp.sync`(mask)
    elif is_amd_gpu():
        # lowered to AMDGPU dialect intrinsics
        ...

The @parameter if branches are evaluated at compile time, meaning the GPU-specific intrinsic is selected during code generation — there is no runtime check. The MLIR operation is invoked directly from Mojo source using backtick syntax, since Mojo is designed to act as syntactic sugar for MLIR dialects without modifying the compiler itself.7

Performance results published by independent observers tend to land between two registers. ComplyAdvantage's engineering team reported that switching language-detection inference from Python to a Mojo-based path on Linux improved CPU inference speed "approximately 3 to 5 times" without other changes.19 A 2025 academic paper presented at SC25 ported four widely used HPC scientific kernels — a seven-point stencil, BabelStream, miniBUDE, and Hartree-Fock — to Mojo and compared them against hand-tuned CUDA and HIP baselines on NVIDIA and AMD GPUs, finding that Mojo's portable kernels held their own against vendor-specific implementations while requiring only a single source tree.17

Design philosophy

Mojo's designers have been unusually explicit about the values guiding their choices, and the values are not what one might expect from a language released by a Silicon Valley AI startup. The Mojo FAQ states the principle plainly: "We are guided more by pragmatism than novelty."8 The team will cheerfully acknowledge debts to Python, Cython, Rust, Swift, Zig, CUDA, C++, and MLIR rather than claim any of the ideas as original.1

The pragmatic frame translates into a specific aesthetic. Mojo deliberately reuses Python's syntax — its indentation rules, its keywords, its dunder methods — because the language's designers want Python developers to feel at home rather than re-learn. As Lattner put it on Developer Voices: "Python has one type, which is a reference to a Python object. In Mojo we said let's give this thing a name, which became PythonObject. Now you can create a Mojo type Int with conversions between PythonObject […]. Now it can be on the stack instead of boxed and heap allocated, it can be the size of the word on your machine or whatever properties you need."8 The dynamic universe is the default; you opt into the static one only where it pays.

We are guided more by pragmatism than novelty, but Mojo's use of MLIR allows it to scale to new exotic hardware types and domains in a way that other languages haven't demonstrated.
— Modular, Mojo FAQ

A second principle is what Modular calls a "library-first design." Many things that other languages bake into the compiler — including, in Mojo's case, the Int type itself — live in the standard library, written in Mojo, and can in principle be replaced or extended by users.7 The team has stated this is to keep the surface area of the compiler small and to ensure that the same metaprogramming abstractions available to library authors are available to everyone else.

The artistic and intangible factors are where Mojo permits itself the occasional flourish. The canonical file extension is .🔥 — the fire emoji, U+1F525 — with .mojo available as an ASCII alternative for the unconvinced.1 Modular's branding leans heavily on the same flame motif. It is a small, slightly knowing piece of theatre from a project that elsewhere takes itself extremely seriously, and it telegraphs an underlying sensibility: Mojo's designers think programming languages are also cultural objects, not just engineering deliverables.

The third philosophical commitment, articulated repeatedly across Modular's blog and roadmap, is the eventual dissolution of the two-language problem that has shaped AI engineering for a decade — prototyping in Python, productionising in C++ or CUDA.12 Mojo's stated long-term goal is to be a strict superset of Python, allowing the same code to evolve from research notebook to GPU kernel without crossing language boundaries.1 Whether that goal is finally achievable remains an open empirical question, but it is the organising ambition that explains nearly every other technical decision the language has made.

Popularity in context

By the conventional measures, Mojo remains a niche language whose mindshare exceeds its installed base. In the TIOBE Programming Community Index, which aggregates search-engine hits across roughly twenty-five sources, Mojo and Zig were described in late 2025 as "knocking on the top 50 door" at positions #51 and #56 respectively.20 The top of the chart remains dominated by Python (the very language Mojo proposes to extend), C, Java, C++, and C#; in May 2026, Python still led the index with roughly 20% rating share.24

The picture changes if one looks at community signals rather than aggregate search volume. By March 2026 the Mojo open-source ecosystem reportedly contained over 750,000 lines of code with an active community of 50,000 members.12 The Modular GitHub repository, Discord, and forum together represent a small but unusually active developer population — the kind that, for languages such as Rust and Go in their early years, eventually translated into mainstream adoption.

It is also worth noting where Mojo has chosen not to compete. The language is unapologetically targeted at AI infrastructure, kernel development, and the niche where Python already dominates research workflows. It has made no serious attempt to displace JavaScript, TypeScript, or the languages of mobile development. Whether it will ever break into top-twenty rankings depends largely on whether its current concentration in AI workloads broadens — and on whether the broader AI industry adopts Modular's stack as the dominant alternative to NVIDIA's CUDA monoculture.

Projects and products

The single most consequential product written in Mojo is the MAX framework itself — the Modular Accelerated eXecution platform whose core, according to Modular, "is written entirely in Mojo."9 MAX is an integrated suite for AI inference and serving, providing OpenAI-compatible endpoints, a Python API with PyTorch-like ergonomics, and a graph API exposed at the Mojo level for developers who need to construct custom inference pipelines.15 MAX is the answer to "what is Mojo actually used for in production?" — it is used to build the inference layer that hundreds of generative models can run on, with no vendor-specific code paths.14

The MAX 26.2 release in March 2026 added FLUX.2 diffusion image generation with reported speedups of over 4× against state-of-the-art baselines; 26.3 in May 2026 added video generation via Wan 2.2.10 Recent releases have also expanded hardware support to NVIDIA's B300, Jetson Thor, and DGX Spark systems alongside AMD's RDNA consumer GPUs.10

Outside Modular itself, the most cited production-grade Mojo deployment is by Qwerky AI, which has used Mojo to compile custom GPU kernels for the Mamba architecture and reported 50% faster GPU performance with deployments that work across both NVIDIA and AMD hardware — a portability story that would be operationally difficult to replicate in pure CUDA.12

In February 2026 Modular announced that BentoML — the open-source model-serving project widely used by ML platform teams — was joining the company, with a commitment to continued investment in BentoML's open-source roadmap.14 This represents one of the more significant corporate consolidations in the Mojo ecosystem to date, bringing an established open-source model-serving community into Modular's orbit.

The community has also produced a number of widely-shared open-source ports and reference implementations: llama2.mojo ports Llama 2 inference to a single Mojo file, and llm.mojo is a Mojo port of Andrej Karpathy's llm.c.14

Libraries and frameworks

The Mojo library ecosystem in 2026 remains small but is growing along recognisable lines: numerical computing libraries first, then web and systems plumbing, then domain-specific frameworks. Two community-curated lists — mojicians/awesome-mojo and coderonion/awesome-mojo-max-mlir — function as informal directories.14

NuMojo
N-dimensional array type and numerical algorithms, modelled after NumPy and SciPy. Supports native NDArray, Matrix, and ComplexNDArray types; targets vectorization, parallelization, and GPU acceleration.16
Mojmelo
A pure-Mojo implementation of machine-learning algorithms from scratch: linear and logistic regression, KNN, KMeans, SVM (primal and dual), Naive Bayes, decision trees, random forests, GBDT, PCA, LDA, AdaBoost.14
Basalt
A machine-learning framework built from scratch in pure Mojo — a deeper-stack alternative to Mojmelo, aiming at training as well as inference for neural-network workloads.14
Endia
A dynamic array library for scientific computing, comparable to PyTorch, NumPy, and JAX. Provides automatic differentiation, complex numbers, and JIT compilation via MAX.14
Lightbug
A simple and fast HTTP and web framework for Mojo — currently the most-cited piece of network plumbing in the ecosystem.14
SciJo
A high-performance numerical computation library inspired by NumPy, SciPy, and Scikit-HEP — particularly aimed at scientific computing including high-energy physics workflows.14
EmberJson
An ergonomic JSON library in pure Mojo, intended as a direct replacement for Python's json module.14
Decimo
Arbitrary-precision integer and decimal arithmetic, analogous to Python's int and decimal.Decimal.13
ArgMojo
A command-line argument parser inspired by Python's argparse, Rust's clap, and Go's flag.14
EasyDeL
An open-source library to accelerate model training and serving with cooperating Python and Mojo paths.14

Of the frameworks, MAX is the heavyweight: an end-to-end stack that combines model compilation, serving infrastructure, and a Python-facing API with optional Mojo escape hatches for kernel-level performance tuning.15 The MAX framework's recent direction is illustrative — version 26.1 graduated the Python API out of experimental status, adding PyTorch-like eager mode and a model.compile() path for production workloads, while stabilising the MAX LLM Book documentation.10 For developers who want to stay in Python but capture Mojo's underlying speedup, this is the intended pathway.

Integrations

Mojo's most consequential integration is, unsurprisingly, with Python itself. The language's Python interop layer allows existing Python libraries to be imported and called directly from Mojo, with bi-directional interoperability that supports gradual migration of performance-critical components.4 Mojo functions can be invoked from Python with minimal overhead, which has proved useful for cases where a PyTorch or JAX training loop needs to call into a custom data-preprocessing kernel.29

The MLIR integration runs even deeper. Because Mojo treats itself as syntactic sugar over MLIR dialect operations, it integrates not just with the LLVM-MLIR ecosystem in the abstract but with specific projects that consume the same intermediate representation: Torch-MLIR, which lowers PyTorch models to MLIR; ONNX-MLIR, which does the same for ONNX; IREE, a retargetable MLIR-based ML compiler from Google; and TPU-MLIR, used to compile networks for Sophgo TPUs.13 The shared IR means kernels written in Mojo can in principle be composed with code generated by other MLIR-aware tools.

On the hardware side, recent Mojo releases have steadily widened the supported target list. The 25.7 release added bfloat16 support on NVIDIA Grace Hopper and Grace Blackwell, expanded Apple silicon GPU coverage, and tightened the type system for GPU pointers.10 The 26.1 release expanded Apple silicon GPU support further, and 26.2 added NVIDIA B300, Jetson Thor, DGX Spark, and AMD RDNA consumer GPUs.10 The SC25 academic study on Mojo for HPC kernels confirmed that the same Mojo source could run portably across NVIDIA and AMD GPUs while remaining performance-competitive with vendor-specific CUDA and HIP baselines.17

Development environment integrations are now reasonably mature. Mojo ships with a Visual Studio Code extension, a Jupyter kernel for notebook workflows, and an LLDB-based debugger that can step seamlessly through hybrid Mojo, C, and C++ code in a single session.23 Installation is handled either through pip in existing Python environments or via the pixi package manager for greenfield projects.26 An AI-coding ecosystem has been built up around the language as well, with Modular shipping updated AI agent "skills" with each release so that LLM-based coding assistants can produce idiomatic Mojo and assist with GPU-kernel development.27

What it adds up to

Mojo in 2026 sits at an unusual point in its life cycle. The Mojo 1.0 beta announcement in May 2026 — combined with Modular's commitment to open-source the compiler alongside the final 1.0 release later in the year — signals that the language is approaching the stability that production users have been waiting for since 2023.11 The standard library is already open under Apache 2.0; the compiler is expected to follow.9

What Mojo has not yet achieved is mass adoption. Its TIOBE position remains outside the top 50, its corporate user base is concentrated in AI infrastructure shops, and its community, while energetic, is still measured in the tens of thousands rather than the millions. The language's eventual success will depend less on its technical merits — which are not in serious dispute — than on whether the broader AI industry finds the two-language problem painful enough to bet on a new substrate, and whether Modular's stewardship of the open-source ecosystem remains credible as the company commercialises the surrounding platform.

For now, Mojo is the rare new language whose existence is justified less by what it has done than by what it makes possible. It is built on what its designers explicitly acknowledge are not novel ideas — Python's syntax, Rust's ownership, MLIR's IR stack, CUDA-style kernel programming — and assembled with the discipline of a team that has built compilers before and knows where the rough edges typically end up. Whether that discipline produces a generational shift in AI programming, or merely a well-engineered tool for the people who care most about GPU kernels, is the central open question of the next two years.

Where to start, in one paragraph

If you want to try Mojo: install via pip install modular or the pixi package manager, work through the Mojo Manual at docs.modular.com/mojo, and explore the curated mojicians/awesome-mojo list for community libraries. If you want to follow the language's evolution closely, the Mojo changelog and the Modular GitHub releases page are the canonical sources. If you want to understand the design philosophy, read the Mojo Vision and FAQ documents — and watch Chris Lattner's Developer Voices interview, which is the most candid public account of how the language came to exist.
