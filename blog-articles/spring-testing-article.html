The Spring Framework has, since its inception in 2002, been one of the most widely adopted platforms in enterprise Java development. Alongside its core dependency injection and MVC capabilities, Spring has built a remarkably comprehensive testing infrastructure that spans unit isolation, slice-based integration testing, full context loading, and contract verification. This guide examines each layer of that infrastructure in depth, combining annotated source-code examples with an analysis of the architectural decisions that motivate each tool.

Modern Spring Boot applications are typically tested across multiple layers using a combination of JUnit 5, Mockito, and Spring's own annotations. The framework provides auto-configuration support that dramatically reduces the boilerplate required to stand up a test context, while slice annotations such as @WebMvcTest and @DataJpaTest allow developers to load only the components relevant to the layer under test — keeping test suites fast and diagnostically precise.[1]

01

History & Context

Spring's testing support predates even the first public release of JUnit 4. Rod Johnson's original Expert One-on-One J2EE Design and Development (2002) advocated strongly for dependency injection precisely because it makes components independently testable without requiring a running application server — a radical proposition at a time when EJB components could only be tested inside a container.[2]

The spring-test module was formalised as a first-class project component with Spring 2.0 (2006), introducing the AbstractTransactionalSpringContextTests base class and the @ContextConfiguration annotation. Spring 3.1 (2011) added the @ActiveProfiles annotation for environment-specific test configuration, and Spring 4.0 (2013) completed the transition to a purely annotation-driven model with @SpringRunner (formerly SpringJUnit4ClassRunner).

The major paradigm shift arrived with Spring Boot 1.4 (2016), which introduced slice testing annotations (@WebMvcTest, @DataJpaTest), the @SpringBootTest meta-annotation, and the @MockBean annotation. These additions codified opinionated testing patterns that the community had previously assembled by hand. Spring Boot 2.x refined the model further with JUnit 5 support via SpringExtension, and the Spring Boot 3.x series (2022–present) aligned the entire testing stack with Jakarta EE 9+ namespaces and introduced MockMvcTester as an AssertJ-native complement to the Hamcrest-based MockMvc.[3]

"The ability to test components outside of the application server container was one of Spring's founding design principles — and it remains the core motivation behind every subsequent testing annotation the framework provides."

@SpringBootTest

@SpringBootTest is the all-encompassing integration test annotation. It locates the class annotated with @SpringBootApplication and uses it as the configuration source to bootstrap a full ApplicationContext, loading all beans found through component scanning, auto-configuration, and explicit configuration classes.[4]

WebEnvironment Modes

The annotation's webEnvironment attribute controls how — or whether — a servlet container is started. By default, @SpringBootTest does not start any server; it instead sets up a mock servlet environment that can be probed via MockMvc.[5]

Mode Server Recommended Client
MOCK (default) No — mocked servlet environment MockMvc, MockMvcTester
RANDOM_PORT Yes — random available port WebTestClient, TestRestTemplate
DEFINED_PORT Yes — server.port property WebTestClient, TestRestTemplate
NONE No — no web infrastructure at all Direct service call

Integration Test Example

The following example demonstrates a full-context integration test. Note the use of @AutoConfigureMockMvc to inject a configured MockMvc when running with the default MOCK environment, and @TestPropertySource to override datasource properties for the test execution.[6]

EmployeeRestControllerIntegrationTest.java Java 
@ExtendWith(SpringExtension.class)
@SpringBootTest(
    webEnvironment = SpringBootTest.WebEnvironment.MOCK,
    classes = Application.class
)
@AutoConfigureMockMvc
@TestPropertySource(
    locations = "classpath:application-integrationtest.properties"
)
public class EmployeeRestControllerIntegrationTest {

    @Autowired
    private MockMvc mvc;

    @Autowired
    private EmployeeRepository repository;

    @Test
    public void givenEmployees_whenGetEmployees_thenStatus200() throws Exception {
        createTestEmployee("Alice");

        mvc.perform(MockMvcRequestBuilders
                .get("/api/employees")
                .contentType(MediaType.APPLICATION_JSON))
           .andExpect(status().isOk())
           .andExpect(content()
                .contentTypeCompatibleWith(MediaType.APPLICATION_JSON))
           .andExpect(jsonPath("$[0].name", is("Alice")));
    }
}
Note — Context Caching

Spring caches the ApplicationContext between test classes that share identical configuration metadata. This means that, in a large test suite, the expensive context bootstrap happens only once per unique configuration fingerprint, not once per test class. Annotations that modify context state — such as @DirtiesContext — force a fresh context to be created and should be used sparingly.

MockMvc & @WebMvcTest

MockMvc is Spring's primary tool for testing MVC controllers without starting a real HTTP server. It simulates the full Spring MVC request-processing pipeline — dispatcher servlet, handler mappings, handler adapters, message converters, view resolvers, and exception handlers — against a mocked servlet environment, providing HTTP semantics (status codes, headers, content negotiation) without network overhead.[7]

There are two principal ways to obtain a MockMvc instance in a Spring Boot test. The first, and most targeted, is via @WebMvcTest. The second is to combine @SpringBootTest with @AutoConfigureMockMvc. They serve different purposes and should not be mixed, as they create conflicting servlet environment configurations.[8]

@WebMvcTest — Slice Testing the Web Layer

@WebMvcTest initialises a sliced application context containing only MVC-relevant beans: @Controller, @ControllerAdvice, @JsonComponent, Converter, Filter, WebMvcConfigurer, and HandlerMethodArgumentResolver components. Service-layer beans are absent — they must be supplied as mocks via @MockBean. This constraint keeps the test fast and focused.[9]

@WebMvcTest — Use When
  • Testing a single controller class in isolation
  • Verifying JSON serialisation / deserialisation
  • Checking HTTP status codes and response headers
  • Validating @ControllerAdvice exception mapping
  • Testing Spring Security endpoint rules
@SpringBootTest + MockMvc — Use When
  • Testing the integration of multiple layers
  • Verifying behaviour with a real service/repository chain
  • End-to-end validation without a real HTTP port
  • Testing application-wide filter or interceptor chains

Controller Test with @WebMvcTest

SortingControllerUnitTest.java Java 
@WebMvcTest(SortingController.class)
class SortingControllerUnitTest {

    @Autowired
    private MockMvc mockMvc;

    // Service layer must be mocked — it is not part of the slice context
    @MockBean
    private SortingService sortingService;

    @Autowired
    private ObjectMapper objectMapper;

    @Test
    void whenSortMethodIsCalled_thenReturnSortedArray() throws Exception {
        List<Integer> input  = Arrays.asList(5, 3, 8, 1, 9, 2);
        List<Integer> sorted = Arrays.asList(1, 2, 3, 5, 8, 9);

        when(sortingService.sortArray(input)).thenReturn(sorted);

        mockMvc.perform(post("/")
                .contentType(MediaType.APPLICATION_JSON)
                .content(objectMapper.writeValueAsString(input)))
               .andExpect(status().isOk())
               .andExpect(content()
                    .json(objectMapper.writeValueAsString(sorted)));
    }
}

MockMvc Result Matchers

MockMvc's fluent API exposes a rich set of result-matcher factories. The most commonly used are imported statically from MockMvcResultMatchers:[10]

Matcher Purpose
status().isOk() Assert HTTP 200 response
content().contentType(…) Assert response Content-Type header
jsonPath("$.field", is(…)) Assert a JSON field value using JSONPath syntax
view().name("viewName") Assert the logical view name returned by a controller
model().attribute("key", …) Assert a model attribute value (Thymeleaf / MVC views)
header().string("X-Custom", "val") Assert a specific response header value
redirectedUrl("/path") Assert redirect location on 3xx responses
Spring Boot 3.x — MockMvcTester

Spring Boot 3.4 introduced MockMvcTester as an AssertJ-native wrapper around MockMvc. When AssertJ is on the classpath, @WebMvcTest auto-configures it alongside the standard MockMvc instance, offering a more idiomatic assertion style consistent with the rest of JUnit 5 / AssertJ based tests.

@MockBean & Mockito

@MockBean is a Spring Boot annotation that creates a Mockito mock object and registers it in the Spring ApplicationContext. If a bean of the same type already exists in the context, it is replaced by the mock. Mocked beans are automatically reset after each test method, preventing state leakage between test cases.[11]

This distinguishes @MockBean from plain Mockito's @Mock. With @Mock, the developer must manually inject mock objects using @InjectMocks or MockitoAnnotations.initMocks(…), and the object never enters the Spring context. @MockBean handles injection automatically by replacing the real bean in the context — making it the correct choice for any test that loads a Spring context.[12]

Service-Layer Test with @MockBean

UserServiceIntegrationTest.java Java 
@SpringBootTest
public class UserServiceIntegrationTest {

    // The real UserService bean is loaded from the context
    @Autowired
    private UserService userService;

    // The real UserRepository is replaced by this Mockito mock
    @MockBean
    private UserRepository userRepository;

    @Test
    public void testGetUserById() {
        // Arrange
        User mockedUser = new User("John", "Doe", 25);
        when(userRepository.findById(1L))
                .thenReturn(Optional.of(mockedUser));

        // Act
        User result = userService.getUserById(1L);

        // Assert
        assertNotNull(result);
        assertEquals("John", result.getFirstName());
        assertEquals("Doe",  result.getLastName());

        // Verify interaction
        verify(userRepository).findById(1L);
    }
}

@MockBean Deprecation in Spring Boot 3.4

Spring Boot 3.4 deprecated @MockBean and its companion @SpyBean, replacing them with @MockitoBean and @MockitoSpyBean respectively. The new annotations are semantically equivalent but align the naming with the Mockito ecosystem and remove the ambiguity of the "Bean" suffix in a Spring context.[13] Existing code using @MockBean continues to function but should be migrated over time.

Migration Example (Spring Boot 3.4+) Java 
// Before (deprecated as of 3.4)
@MockBean
private UserRepository userRepository;

// After (Spring Boot 3.4+)
@MockitoBean
private UserRepository userRepository;

"@MockBean not only creates a mock object — it registers that mock directly into the ApplicationContext, replacing any existing bean of the same type. This gives tests the flexibility to combine real dependency injection with selectively mocked fields."

JPA & Repository Tests

Testing the data access layer in isolation is supported by the @DataJpaTest annotation, which configures a sliced application context containing only JPA-relevant components: @Entity classes, Spring Data JPA repositories, and the Hibernate infrastructure. It sets spring.jpa.show-sql=true by default, makes all test data transactional and rolled back at the end of each test, and — unless instructed otherwise — replaces the application's DataSource with an embedded in-memory database.[14]

In-Memory Database Testing with H2

When the H2 database driver is present on the classpath, @DataJpaTest auto-configures it as the test datasource. The entire test lifecycle — schema creation, data seeding, test execution, and rollback — occurs in a JVM-local database that is discarded when the JVM exits, yielding a clean, reproducible test environment.[15]

pom.xml — Test Dependencies XML 
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-data-jpa</artifactId>
</dependency>
<dependency>
    <groupId>com.h2database</groupId>
    <artifactId>h2</artifactId>
    <scope>runtime</scope>
</dependency>
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-test</artifactId>
    <scope>test</scope>
</dependency>

Repository Test — @DataJpaTest with TestEntityManager

Spring provides a TestEntityManager within @DataJpaTest contexts that should be used to persist test fixtures rather than calling the repository's own save method — keeping arrangement logic separate from the system under test.[16]

UserRepositoryTest.java Java 
@DataJpaTest
@DirtiesContext
public class UserRepositoryTest {

    @Autowired
    private UserRepository userRepository;

    // Use TestEntityManager to persist fixtures — not the repo under test
    @Autowired
    private TestEntityManager testEntityManager;

    @Test
    void testGetByFirstName() {
        // Arrange: persist a user directly via JPA EntityManager
        User user = new User("John", "Wick", UUID.randomUUID());
        testEntityManager.persist(user);
        testEntityManager.flush();

        // Act
        List<User> results = userRepository.findByFirstName("John");

        // Assert
        assertFalse(results.isEmpty());
        assertEquals("Wick", results.get(0).getLastName());
    }
}

Limitations of H2 for Repository Tests

While H2 is convenient, it does not support all features of production databases such as PostgreSQL or MySQL. Vendor-specific SQL constructs (e.g., MySQL's STRAIGHT_JOIN, PostgreSQL's JSONB operators) cannot be tested reliably against H2. This divergence between test and production databases can produce a false sense of correctness and is a frequently-cited drawback of the in-memory approach.[17]

Spring Security Integration

When Spring Security is on the classpath, @WebMvcTest automatically scans WebSecurityConfigurer beans, meaning that secured endpoints require authenticated requests in tests. Spring Security provides a dedicated testing module (spring-security-test) that integrates seamlessly with MockMvc.[18]

AdminControllerTest.java — Security-Aware Test Java 
@Import(SecurityConfig.class)
@WebMvcTest(AdminController.class)
class AdminControllerTest {

    @Autowired
    private MockMvc mockMvc;

    @Test
    void anonymousUserShouldBeRedirectedToLogin() throws Exception {
        // Without authentication, Spring Security returns 302 → login
        mockMvc.perform(get("/admin"))
               .andExpect(status().is3xxRedirection());
    }

    @Test
    @WithMockUser(roles = "ADMIN")
    void adminUserShouldSeeAdminPage() throws Exception {
        mockMvc.perform(get("/admin"))
               .andExpect(status().isOk());
    }

    @Test
    void userWithJwtTokenShouldBeAuthenticated() throws Exception {
        mockMvc.perform(get("/admin")
                .with(SecurityMockMvcRequestPostProcessors
                        .jwt().authorities(new SimpleGrantedAuthority("ROLE_ADMIN"))))
               .andExpect(status().isOk());
    }
}

@WithMockUser associates an authenticated user principal with an entire test method or class. For per-request control, the SecurityMockMvcRequestPostProcessors class offers utility methods for all common authentication mechanisms including HTTP Basic, form-login, OAuth2, and JWT bearer tokens.[19]

07

Testcontainers Integration

Testcontainers is an open-source Java library that provides throwaway Docker container instances for integration tests. When combined with Spring Boot's @DataJpaTest or @SpringBootTest, it resolves the production-database-vs-H2 gap by running the actual production database engine in a container during the test lifecycle.[20]

ProductRepositoryTest.java — Testcontainers + @DataJpaTest Java 
@DataJpaTest
@AutoConfigureTestDatabase(replace = AutoConfigureTestDatabase.Replace.NONE)
@Testcontainers
public class ProductRepositoryTest {

    // Static container — shared across all test methods in this class
    @Container
    @ServiceConnection
    static PostgreSQLContainer<?> postgres =
        new PostgreSQLContainer<>("postgres:16-alpine");

    @Autowired
    private ProductRepository productRepository;

    @Test
    @Sql("classpath:/sql/seed-products.sql")
    void shouldReturnAllProducts() {
        List<Product> products = productRepository.findAll();
        assertEquals(2, products.size());
    }
}

The @ServiceConnection annotation, introduced in Spring Boot 3.1, instructs Spring Boot to auto-configure the datasource connection properties from the running container's exposed port, eliminating the need to manually set spring.datasource.url in a test property source.[21]

When to Use Testcontainers

Use Testcontainers when your repositories use vendor-specific SQL constructs, stored procedures, or features such as PostgreSQL's JSONB, ARRAY types, or full-text search. The overhead of spinning up a Docker container is acceptable for integration tests that run as part of a CI pipeline; for fast local feedback cycles, H2 is still a reasonable trade-off.

A Layered Testing Strategy

Effective Spring application testing follows a layered pyramid structure: many fast unit tests at the base, a moderate number of slice-level integration tests in the middle, and fewer full-context integration tests at the top. This distribution optimises the trade-off between feedback speed, diagnostic precision, and confidence in production behaviour.

Layer Annotation / Tool Context Size Speed
Unit test (no Spring) @ExtendWith(MockitoExtension.class) None Very fast
Web layer slice @WebMvcTest MVC beans only Fast
JPA layer slice @DataJpaTest JPA beans + DB Fast–Medium
Full context (mock env) @SpringBootTest (MOCK) Full application Medium
Full context (real server) @SpringBootTest (RANDOM_PORT) Full application + container Slow

Key Principles

1 — Prefer @WebMvcTest over @SpringBootTest for Controller Tests

Loading the full application context for a controller test is wasteful and slows the feedback cycle. @WebMvcTest starts in milliseconds and provides identical HTTP semantics for the web layer, including request mapping, serialisation, validation, and exception handling.

2 — Use @MockBean Sparingly

Each unique combination of mock configurations forces Spring to create a new application context, negating context caching benefits. Group tests that share a mock configuration into the same test class where possible. Consider @TestConfiguration inner classes to define shared fakes rather than using @MockBean per field across many test classes.

3 — Replace H2 with Real Databases for Critical Queries

Any repository that uses native queries, database-specific functions, or complex query DSLs should be tested against the same database type used in production, via Testcontainers. The cost of a container startup is justified by the elimination of false positives from H2 dialect compatibility issues.[22]

4 — Isolate Test Property Sources

Always place test-specific configuration in src/test/resources rather than conditionally branching within the main configuration. The @TestPropertySource annotation and Spring Profiles (via @ActiveProfiles("test")) provide clean separation without polluting production configuration.[23]

REST Documentation with Spring REST Docs

Spring REST Docs integrates with MockMvc to generate accurate, test-driven API documentation. Annotating your test class with @AutoConfigureRestDocs customises the MockMvc bean to emit AsciiDoc snippets on every request, which can then be assembled into a full API reference using the Asciidoctor build plugin — tying documentation accuracy to test passage.
