& Firmware Layer

From bare-metal register writes to containerised edge Linux — the software that lives directly on the device and makes cyber-physical systems real.

Domain — Embedded Systems / CPS Architecture Coverage — Bare-Metal · RTOS · Embedded Linux Citation Style — Vancouver

Every cyber-physical system — from a cardiac pacemaker to an autonomous forklift — is ultimately anchored to a piece of silicon. Between that silicon and the high-level software that reasons about the world sits a largely invisible stratum: the embedded firmware layer. This is the code that runs directly on the device, without the cushion of a general-purpose operating system, or with only the thinnest abstraction the application demands. It is the layer that determines whether a robotic arm stops in microseconds or milliseconds, whether a sensor reading is trustworthy, and whether a remote software update succeeds without bricking a device buried inside an industrial wall.

The embedded firmware layer is not monolithic. Three broad paradigms have evolved to serve radically different system requirements: bare-metal firmware, which executes directly against hardware registers with no operating-system scaffolding; Real-Time Operating Systems (RTOS), which impose scheduling and timing guarantees while remaining lightweight enough for microcontrollers; and embedded Linux, which brings the full power of the Linux kernel to more capable edge devices. Understanding when and why engineers reach for each approach is foundational to understanding modern cyber-physical systems architecture.

01 — HistoryThe Origins of Device-Level Software

The concept of software embedded permanently within a device predates the modern microprocessor. In the 1960s, read-only core rope memory in the Apollo Guidance Computer held fixed programs that could not be altered in the field — an early ancestor of what we now call firmware.1 The RTOS concept emerged through the same decade: the late 1960s and early 1970s saw industrial control, defence systems, and telecommunications demanding deterministic, predictable responses that general-purpose time-sharing systems could not guarantee.2

Through the 1970s and early 1980s, embedded programming meant writing directly in assembly language for specific microprocessors — the 8051, the Motorola 6800, the Zilog Z80. There was no abstraction; registers were manipulated by address. The term "bare-metal" was not yet coined, but the paradigm was firmly established. As 32-bit microprocessors arrived and C compilers matured, a generation of engineers gradually shifted from assembly to C, which allowed some degree of portability while preserving direct hardware access.2

Commercial RTOSes emerged to fill the gap between pure bare-metal programming and full operating systems. Wind River Systems, founded by Jerry Fiddler and Dave Wilner in 1983, released VxWorks 1.0 in 1987 — one of the first commercial RTOSes to achieve widespread adoption across aerospace, defence, and industrial automation.3 By the mid-2000s, the open-source movement began reshaping embedded software just as it had desktop computing, with FreeRTOS, OpenWRT, and eventually the Yocto Project democratising embedded development at every layer of the stack.

02 — Bare-Metal FirmwareProgramming Against the Machine

Bare-metal firmware is code that executes directly on a processor with no operating system layer between it and the hardware.4 The program interacts with peripherals by reading and writing memory-mapped registers defined in the microcontroller's datasheet; timing is managed via hardware timers and interrupt service routines (ISRs); and the developer is responsible for every aspect of system initialisation that a kernel would otherwise handle automatically.

Definition — Bare-Metal vs Firmware

The terms are often conflated. Firmware is the broader category — any software tightly coupled to hardware and stored in non-volatile memory, whether or not an OS is present. Bare-metal refers specifically to firmware that runs without any operating system. All bare-metal programs are firmware, but not all firmware is bare-metal.5

Structural Components

Bootloader

The bootloader is the first code to execute on power-up. It initialises the clock tree, configures memory regions, sets up stack pointers, and transfers control to the main application image. In secure systems, the bootloader performs cryptographic verification of the firmware image before launching it, a process known as secure boot.6 U-Boot is the dominant open-source bootloader for Linux-capable devices, while simpler MCU projects often use vendor-provided startup files or custom equivalents.

Hardware Abstraction Layer (HAL)

Even in bare-metal firmware, engineers frequently introduce a Hardware Abstraction Layer (HAL) — a thin set of functions that wraps register accesses behind a portable API.6 STMicroelectronics ships HAL libraries for its STM32 family; Nordic Semiconductor provides nRF-series HALs; ARM's CMSIS (Cortex Microcontroller Software Interface Standard) defines a vendor-neutral register-access layer for Cortex-M devices. The HAL does not eliminate direct hardware interaction; it organises and standardises it.

Interrupt Service Routines (ISRs)

In an OS-less system, ISRs are the primary mechanism for asynchronous event handling. When a hardware interrupt fires — a timer overflow, a UART character received, a GPIO edge — the processor saves its current context, jumps to the ISR address stored in the interrupt vector table, executes the handler, and returns. Effective ISR design is critical: handlers must execute in nanoseconds to microseconds to maintain hard real-time guarantees, deferring heavier processing to the main loop or a deferred interrupt handler.7

/* Bare-metal ISR — STM32 UART receive, direct register access */
void USART2_IRQHandler(void) {
    if (USART2->ISR & USART_ISR_RXNE) {        /* RX not empty flag */
        uint8_t byte = (uint8_t)(USART2->RDR); /* read clears flag    */
        ring_buf_put(&rx_buf, byte);             /* non-blocking enqueue */
    }
}

int main(void) {
    clock_init();   /* configure PLL, AHB/APB prescalers */
    gpio_init();
    usart_init(115200);
    __enable_irq();
    while (1) {
        process_rx_buffer(); /* main loop — deferred processing   */
        enter_low_power_mode();
    }
}

When Bare-Metal Is the Right Choice

Bare-metal programming excels in three scenarios. First, extremely resource-constrained devices — microcontrollers with 4–16 KB of flash and 1–4 KB of RAM where even a minimal RTOS kernel would exceed the available memory budget. Second, hard real-time systems where every clock cycle of scheduler overhead is unacceptable and timing must be deterministic to the instruction level. Third, ultra-low-power applications where the device spends most of its time in deep sleep and the minimal overhead of no-OS code matters for battery life.8

The trade-off is complexity. Without an OS, the developer must manually implement concurrency, communication between interrupt and main contexts, and any protocol stacks required. Bare-metal knowledge has become rarer as higher-level tools proliferate, making experienced bare-metal engineers increasingly valuable in mission-critical domains.1

Bare-metal implementation demands deeper knowledge of hardware components than most internal teams possess today — and when system failures occur, developers face challenges without the error logs that higher-level programs provide.

— Witekio, Bare Metal Firmware Development [1]

03 — Real-Time Operating SystemsScheduling, Guarantees, and Determinism

A Real-Time Operating System (RTOS) is a specialised operating system designed for applications where timing is as critical as functionality.2 Unlike general-purpose OSes that optimise for average throughput, an RTOS prioritises worst-case execution time and deterministic latency — the assurance that a high-priority task will execute within a bounded, predictable time after its triggering event.

The core abstraction is the task (or thread): a discrete unit of execution with its own stack, priority, and state. The RTOS scheduler decides which task runs at every context switch, typically using a preemptive priority-based algorithm: the highest-priority task that is ready to run preempts any lower-priority task. Additional scheduling modes — cooperative, round-robin, and hybrid — allow the developer to tune the concurrency model for the application.

FreeRTOS — The Open-Source Standard

FreeRTOS was created by Richard Barry in 2003 as an open-source RTOS for microcontrollers. Barry founded the project through his company Real Time Engineers Ltd., developing and promoting it for more than a decade.9 The system quickly gained traction because of its portability — written predominantly in C, it could be ported to new hardware by implementing a small set of platform-specific functions — its lightweight architecture, and its permissive licensing.10

A pivotal moment came in 2017 when Amazon Web Services acquired stewardship of FreeRTOS, announcing Amazon FreeRTOS at AWS re:Invent and integrating connectivity libraries, over-the-air update capabilities, and cloud-facing security enhancements.11 Barry joined AWS to continue development. By that time, FreeRTOS was being downloaded once every three minutes — a testament to its penetration across IoT, automotive, medical, and industrial embedded markets.9

FreeRTOS provides the core RTOS primitives: preemptive and cooperative task scheduling, queues for inter-task communication, binary and counting semaphores, mutexes with priority inheritance (to mitigate priority inversion), software timers, and an optional tickless idle mode for low-power applications.12

/* FreeRTOS task creation — sensor acquisition + processing pipeline */
void vSensorTask(void *pvParameters) {
    SensorReading_t reading;
    for (;;) {
        reading.value = adc_read_channel(ADC_TEMP);
        reading.timestamp = xTaskGetTickCount();
        xQueueSend(xSensorQueue, &reading, portMAX_DELAY);
        vTaskDelay(pdMS_TO_TICKS(10)); /* 100 Hz sampling, yields CPU */
    }
}

void app_main(void) {
    xSensorQueue = xQueueCreate(32, sizeof(SensorReading_t));
    xTaskCreate(vSensorTask,   "Sensor",  512, NULL, tskIDLE_PRIORITY+2, NULL);
    xTaskCreate(vProcessTask,  "Process", 1024, NULL, tskIDLE_PRIORITY+1, NULL);
    vTaskStartScheduler(); /* transfers control to FreeRTOS kernel */
}

Zephyr — The Linux Foundation's RTOS

Zephyr's lineage traces back to Virtuoso, an RTOS for digital signal processors developed by Belgian software company Eonic Systems in the 1990s. Wind River Systems acquired Eonic in 2001 and later evolved Virtuoso into "Rocket" RTOS — a lightweight, free-to-use (though not open-source) system targeted at IoT.13 In February 2016, Wind River contributed the Rocket kernel to the Linux Foundation, which relaunched it as the Zephyr Project with founding members Intel, Wind River, Synopsys, and NXP Semiconductors.14

Zephyr was explicitly designed to scale from simple bare-metal-like applications to complex multi-threaded systems, fitting into as little as 4 KB of RAM in its most minimal configuration.13 Its build system, based on CMake with Kconfig and devicetree configuration (inherited from the Linux kernel), allows the same RTOS to be used across a wide range of hardware with a consistent development experience. As of early 2025, Zephyr had the largest number of contributors and commits of any open-source RTOS, reflecting its strong community growth and wide adoption across silicon vendors.15

VxWorks — The Mission-Critical Workhorse

VxWorks occupies the high end of the RTOS spectrum. Introduced by Wind River Systems in 1987 as an enhancement to the VRTX kernel sold by Ready Systems, it was first commercially released when Wind River developed its own kernel in anticipation of losing its reseller contract.3 VxWorks became the de facto RTOS for safety-critical and defence-grade embedded systems: it powered NASA's Mars Pathfinder mission in 1997, demonstrating reliability across more than 1.5 million executed instructions without failure; the Spirit and Opportunity Mars Exploration Rovers in 2004; Curiosity in 2012; InSight in 2018; and Perseverance in 2021.3

Unlike FreeRTOS and Zephyr, VxWorks is a proprietary commercial product — certified to DO-178C Design Assurance Level A (DAL A) for avionics, IEC 61508 for industrial safety, and ISO 26262 for automotive. Its kernel can be as small as 20 KB but scales to multicore SMP and AMP configurations with a Type 1 hypervisor that allows VxWorks, Linux, and Android to run simultaneously on the same processor.16 In 2022, Wind River was acquired by Aptiv for $4.3 billion, underscoring the commercial value of its RTOS portfolio.3

Over 750 safety-critical programs have leveraged VxWorks, certified to DO-178C Design Assurance Level A — the highest assurance level for airborne software.

— Grokipedia, VxWorks [16]

Core RTOS Concepts

Preemptive Priority Scheduling

The RTOS scheduler assigns each task a numeric priority. When a higher-priority task becomes ready — because a timer fires, a semaphore is given, or data arrives on a queue — the scheduler immediately preempts the running lower-priority task, saves its context, and runs the higher-priority task. This property is what makes hard real-time deadlines achievable.

Priority Inversion and Inheritance

A classical RTOS hazard: a low-priority task holds a mutex that a high-priority task is waiting for; an intermediate-priority task runs instead, starving the high-priority task of the mutex indefinitely. The solution — priority inheritance — temporarily elevates the mutex-holding low-priority task to the ceiling priority of the waiting high-priority task until the mutex is released. Mars Pathfinder's 1997 system reset was caused by exactly this bug; NASA's ground team diagnosed and patched it remotely while the rover was on Mars.2

Tickless Mode and Power Management

Battery-powered CPS nodes spend the vast majority of their time idle. Both FreeRTOS and Zephyr support tickless idle: when no tasks are ready to run, the system clock is halted and a low-power timer is armed for the next scheduled wake event. This can reduce active current by orders of magnitude in sensor-node applications.12

04 — Embedded LinuxThe Full Stack on the Edge

For edge devices with sufficient computational resources — application processors, multicore SoCs, systems with 256 MB or more of RAM — the Linux kernel becomes viable, and with it come decades of driver support, networking stacks, package ecosystems, and developer familiarity. Embedded Linux occupies the upper tier of the firmware taxonomy: it is a full OS, albeit one carefully stripped and configured for resource-constrained hardware.

The distinction from desktop Linux is significant. Embedded Linux images are typically built from source, cross-compiled for a target architecture, configured to include only the drivers and services the device requires, and stored in flash memory. Boot times matter; image size matters; power consumption matters. The standard tools of embedded Linux development — build systems, cross-compilers, board support packages — form a specialised ecosystem that has evolved independently of enterprise Linux distributions.

Yocto Project — Build Your Own Distribution

The Yocto Project is not a Linux distribution — it is a framework for building one. Announced by the Linux Foundation in 2010 and officially launched in March 2011 in collaboration with 22 organisations including OpenEmbedded, its goal is to provide tools and processes that enable the creation of custom Linux distributions for embedded and IoT devices, independent of the host hardware architecture.17

The intellectual heritage of Yocto runs deeper: OpenEmbedded, the build framework that Yocto shares maintenance with, was established in 2003 by developers from the OpenZaurus project — itself an effort to build Linux for the Sharp Zaurus personal digital assistant.18 The OpenEmbedded build system uses BitBake, a task scheduler and metadata engine that parses recipe files (specifying how each software package is fetched, configured, compiled, and installed) to produce bootable firmware images.18

Yocto's architecture is layered: a Poky reference distribution provides the core build system and metadata; hardware-specific Board Support Package (BSP) layers add drivers and kernel configurations for the target device; distribution layers add application recipes and policies. Developers compose their image by selecting and stacking layers — enabling maximum reuse across product lines while maintaining hardware-specific customisation.19 In 2025, Yocto had more than 30 member companies including AWS, ARM, Intel, Qualcomm, Microsoft, and Wind River, generating an estimated $30 million USD in annual software value.19

Ubuntu Core — Immutable, Snap-Based IoT Linux

Ubuntu Core takes a fundamentally different approach from Yocto. Rather than building a bespoke distribution from source, Ubuntu Core delivers a complete Ubuntu operating system through Canonical's Snap package mechanism — including the kernel, base OS, and every application, all as self-contained snaps with bundled dependencies.20

The architectural implications are significant. Ubuntu Core's root filesystem is read-only (squashfs-based), making it highly resistant to corruption from power failures or interrupted writes. Updates are transactional: if an update fails, the system automatically rolls back to the previous revision.21 Snaps are confined by a security policy using AppArmor and seccomp, limiting what each application can access on the system. For fleet management scenarios — thousands of identical IoT devices receiving coordinated over-the-air updates — this model offers compelling operational advantages over traditional embedded Linux distributions.

Ubuntu Core's minimum requirements (600 MHz CPU, 128 MB RAM, 4 GB storage) position it above bare-metal and RTOS territory but below the heavy end of embedded Linux.22 It has been deployed in Erle-Copter drones, Dell Edge Gateways, and various industrial gateway appliances.

OpenWRT — Networking-Focused Embedded Linux

OpenWRT (Open Wireless Router) traces its origin to 2004, when Linksys released the firmware source for its WRT54G series of wireless routers under the GNU General Public License — an obligation imposed by the embedded Linux kernel the firmware contained. Developers recognised that the hardware was capable of far more than the stock firmware allowed, and began building a community Linux distribution specifically for the device.23

OpenWRT replaces the concept of a static, monolithic firmware image with a writable root filesystem and a package management system (opkg), currently offering approximately 8,000 optional packages.23 Its primary domain is network-centric embedded devices: residential gateways, access points, industrial routers, and IoT hubs. The LuCI web interface provides human-readable device management, while the ash shell gives developers full SSH access for scripting and diagnostics. OpenWRT has influenced a generation of embedded networking products; TP-Link, Xiaomi, ZyXEL, and D-Link firmwares have all derived from or built upon OpenWRT's codebase.

Tool Spotlight — Buildroot

A simpler alternative to Yocto, Buildroot generates minimal root filesystems for embedded targets with less configurability but significantly lower build complexity. It is favoured for resource-constrained devices and teams new to embedded Linux where Yocto's comprehensive feature set would add unnecessary overhead. Buildroot produces smaller, faster builds at the cost of the layered architecture and extensibility that Yocto provides.

05 — Selection FrameworkMatching Firmware Approach to System Requirements

Choosing between bare-metal, RTOS, and embedded Linux is not primarily a matter of preference — it is an engineering decision driven by measurable system constraints. The following framework captures the dominant decision factors.

In practice, sophisticated CPS architectures often combine layers. A multi-core SoC might run bare-metal firmware on a dedicated real-time core for low-latency motor control while Linux executes on the application core for HMI, cloud connectivity, and OTA management. Nordic Semiconductor's nRF5340, Texas Instruments' AM64x, and NXP's i.MX RT Fusion processors are designed around exactly this model.

06 — Innovations & Emerging TrendsThe Frontier of Embedded Software

Rust for Embedded Systems

The Rust programming language has emerged as a credible alternative to C and C++ for embedded firmware. Its ownership model eliminates entire classes of memory safety bugs — buffer overflows, use-after-free, data races — without a garbage collector or runtime overhead. The embedded-hal crate provides a vendor-neutral hardware abstraction API, and the embassy async runtime enables cooperative multitasking on microcontrollers using Rust's async/await syntax without an RTOS. Zephyr added formal Rust language support in 2024, marking a significant endorsement of Rust's role in the RTOS space.

Secure Boot and Firmware Trust Chains

As cyber-physical systems proliferate into critical infrastructure, the firmware layer has become a primary attack surface. Secure boot — the chain of cryptographic verification from bootloader to kernel to application — is now expected by industrial and automotive standards.6 MCUboot is an open-source secure bootloader maintained under the Zephyr Project umbrella that provides image signing, rollback protection, and encrypted firmware updates for RTOS-class devices. For embedded Linux, the integration of TPM chips and UEFI Secure Boot into application-processor platforms provides analogous guarantees.

Over-the-Air (OTA) Update Infrastructure

Devices deployed in physical infrastructure cannot be reliably updated by physical access. OTA update frameworks have therefore become a first-class concern at every layer of the embedded stack. For RTOS devices, AWS IoT OTA, the MCUmgr SMP protocol, and Nordic's FOTA stack handle firmware delivery and integrity verification. For embedded Linux, tools such as RAUC, SWUpdate, and Mender implement A/B partition update schemes that guarantee the device can always boot a known-good image.7

Containerisation at the Edge

A striking development of the late 2010s and 2020s has been the adaptation of container runtimes — Docker, containerd, and OCI-compatible alternatives — to embedded Linux devices. VxWorks added support for Open Container Initiative (OCI) standard containers in its modern releases, allowing the same DevSecOps toolchains used in cloud software to apply to embedded software deployment.16 Ubuntu Core's snap architecture is a form of this same principle: declarative, isolated, versioned software units that can be deployed and rolled back atomically.

AI/ML at the Firmware Layer

Machine learning inference is descending from cloud servers to microcontrollers. TensorFlow Lite for Microcontrollers, Edge Impulse's SDK, and ARM's Ethos NPU ecosystem enable sub-milliwatt neural network inference on bare-metal and RTOS targets. VxWorks explicitly lists TensorFlow Lite as a supported AI/ML framework.16 This trend collapses what was previously a cloud-only capability into the embedded layer, enabling CPS nodes to perform gesture recognition, anomaly detection, and predictive maintenance locally without network connectivity.

Zephyr had the largest number of contributors and commits compared to any other open-source RTOS as of early 2025 — a testament to the industry's bet on open, vendor-neutral embedded infrastructure.

— Developex, Zephyr RTOS Overview [15]

09 — ConclusionThe Software That Makes CPS Real

The embedded firmware layer is, in a meaningful sense, the interface between the digital world and the physical one. No matter how sophisticated the cloud infrastructure, the machine-learning models, or the network protocols above it, a cyber-physical system's behaviour ultimately reduces to code running on a specific processor, interacting with specific hardware, under real-time constraints that the physical world imposes without negotiation.

The three-tier taxonomy — bare-metal, RTOS, embedded Linux — reflects a genuine engineering spectrum. Bare-metal maximises control and minimises overhead, at the cost of developer burden and abstraction. RTOS provides scheduling, timing guarantees, and inter-task communication while remaining viable on constrained MCUs, forming the backbone of the world's automotive ECUs, industrial PLCs, and safety-critical instruments. Embedded Linux unlocks the full power of the open-source software ecosystem for edge devices with sufficient resources, enabling capabilities — rich networking, containerised applications, ML inference — that would be impractical on smaller targets.

These tiers are not mutually exclusive, and the most sophisticated CPS architectures deliberately combine them. The discipline of embedded firmware development — knowing the silicon, understanding the physics of timing, navigating the trade-offs between safety, security, and capability — remains one of the most technically demanding areas of software engineering, and one of the most consequential for the systems that will govern critical infrastructure in the decades ahead.
