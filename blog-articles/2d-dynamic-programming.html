A one-dimensional dynamic program walks along a line: the answer at position i depends on a handful of earlier positions, and a single array suffices to remember the past. The moment a problem forces you to track two coordinates at once — your progress through one string and your progress through another, your row and your column in a grid, the left and right boundaries of a subarray — the line buckles into a plane. The array becomes a matrix, and the entire craft of solving the problem reduces to one question: given the cells already computed, how do I fill this one?

This is the territory of two-dimensional dynamic programming, and it is where a large share of interview and contest problems live. The reason is structural. A great many natural questions are really about aligning, pairing, or partitioning two things, and those operations have a state space that is the product of two ranges. When you tabulate that product, you get a matrix; when you reason about how to fill each cell from its neighbours, you get an algorithm whose running time is, almost always, the area of the matrix multiplied by the work per cell.1

The good news is that nearly all of these problems are variations on a small number of templates. Learn to recognize the four or five archetypes — grid traversal, sequence alignment, string matching, interval merging, and the knapsack table — and most "hard" 2D DP problems collapse into a question of which neighbours a cell reads and what arithmetic combines them. This guide walks each archetype with a canonical problem, full runnable code in six languages, the recurrence drawn out cell by cell, and the variations you will actually meet.

00 — Foundations

The anatomy of a 2D table

Before any specific problem, fix the five components that define any tabulated dynamic program. Getting these explicit on paper, in this order, is the difference between a solution that writes itself and an afternoon of off-by-one debugging.2 The method itself — solving a problem by combining solutions to overlapping subproblems, stored once and reused — is Bellman's, and the discipline of naming the state before writing the loop is the most durable lesson in it.4

  1. State. What does dp[i][j] mean? Name it in a full sentence. "The length of the longest common subsequence of the first i characters of A and the first j characters of B." If you cannot finish that sentence, you do not yet have a DP.
  2. Transition. How is dp[i][j] built from already-known cells? This is the recurrence. In 2D it almost always reads from some subset of dp[i-1][j] (the cell above), dp[i][j-1] (the cell to the left), and dp[i-1][j-1] (the diagonal).
  3. Base cases. The edges of the matrix — usually row 0 and column 0 — that the recurrence cannot produce and must be seeded by hand. Empty-string and empty-prefix cases live here.
  4. Iteration order. Every cell a transition reads must already be filled. For up/left/diagonal dependencies, a simple row-major sweep works. Interval DP reverses this — it fills by increasing interval length instead.
  5. Answer. Which cell (or aggregate of cells) holds the result. Often dp[m][n], the far corner; sometimes a max over the last row, or dp[0][n-1] for interval problems.
Recognition signal

If a brute-force recursion has two parameters that both shrink toward a base case — solve(i, j) calling solve(i-1, j), solve(i, j-1), and so on — and the same (i, j) pair recurs along different paths, you are looking at a 2D DP. The recursion tree has overlapping subproblems indexed by a pair; the matrix is just that recursion memoized.

The padding convention is worth a word. For two-sequence problems it is almost always cleaner to size the table (m+1) × (n+1) and let row/column 0 represent the empty prefix. This buys you a uniform recurrence with no special-casing of the first character, at the cost of remembering that dp[i][j] refers to A[i-1] and B[j-1] — a one-index shift between the table and the strings. Almost every alignment-style recurrence below uses this convention.

Family I — Grid DP

The grid that counts itself

The most visual 2D DP is the one where the matrix is the problem. You are standing on a grid, you may move only in certain directions, and you want to count paths, minimize a cost, or maximize a region. Here dp[i][j] is simply "the answer for cell (i, j)", and the transition is dictated by which moves are legal. Because you may typically only move down or right, cell (i, j) depends on the cell above and the cell to the left — never on anything below or to the right — which is exactly why a top-to-bottom, left-to-right sweep fills the table in valid order.

Unique Paths

A robot sits in the top-left corner of an m × n grid and wants to reach the bottom-right corner, moving only down or right. How many distinct paths exist?6 The number of ways to reach a cell is the number of ways to reach the cell directly above it plus the number to reach the cell directly to its left, because those are the only two cells from which you can step into it.

dp[i][j] = dp[i-1][j] + dp[i][j-1]   // base: dp[i][0] = dp[0][j] = 1 (only one straight-line path)

The first row and first column are all 1: there is exactly one way to walk straight along an edge. Every interior cell sums its two predecessors. The answer waits in the far corner.

Closed form

Unique Paths has a combinatorial shortcut: every route makes exactly m-1 down-moves and n-1 right-moves in some order, so the answer is the binomial coefficient C(m+n-2, m-1). The DP matters anyway, because the instant you add obstacles or weights — the variations below — the closed form evaporates and the table is all you have.

Three variations you will actually meet

Unique Paths II — obstacles. Some cells are blocked. A blocked cell contributes zero paths and seeds nothing downstream; everything else is identical. Note the early exit when the start itself is blocked, and the per-cell guard that zeroes obstacles before summing neighbours.6

Minimum Path Sum. Each cell carries a cost; minimize the total along a down/right path. The structure is identical to Unique Paths, but instead of summing the two predecessors you take the minimum of them and add the current cell's cost. This swap — count becomes optimize — is the single most common 2D DP transformation.6

minPathSum(grid) — variation
func minPathSum(grid [][]int) int {
    m, n := len(grid), len(grid[0])
    dp := make([][]int, m)
    for i := range dp {
        dp[i] = make([]int, n)
    }
    dp[0][0] = grid[0][0]
    for j := 1; j < n; j++ {
        dp[0][j] = dp[0][j-1] + grid[0][j]
    }
    for i := 1; i < m; i++ {
        dp[i][0] = dp[i-1][0] + grid[i][0]
    }
    for i := 1; i < m; i++ {
        for j := 1; j < n; j++ {
            dp[i][j] = grid[i][j] + min(dp[i-1][j], dp[i][j-1])
        }
    }
    return dp[m-1][n-1]
}

Maximal Square. Given a binary matrix, find the largest all-ones square. Here the diagonal genuinely earns its keep: a cell can anchor the bottom-right of a square of side k only if the cells above, left, and diagonally-above-left can each anchor squares of side k-1. So dp[i][j] = 1 + min(up, left, diag) whenever the cell is a one. The three-way minimum is the recurrence's whole personality.6

maximalSquare(matrix) — variation
func maximalSquare(matrix [][]int) int {
    m, n := len(matrix), len(matrix[0])
    dp := make([][]int, m+1)
    for i := range dp {
        dp[i] = make([]int, n+1)
    }
    best := 0
    for i := 1; i <= m; i++ {
        for j := 1; j <= n; j++ {
            if matrix[i-1][j-1] == 1 {
                dp[i][j] = 1 + min(dp[i-1][j], min(dp[i][j-1], dp[i-1][j-1]))
                if dp[i][j] > best {
                    best = dp[i][j]
                }
            }
        }
    }
    return best * best
}
int maximalSquare(int[][] matrix) {
    int m = matrix.length, n = matrix[0].length, best = 0;

    for (int i = 1; i <= m; i++)
        for (int j = 1; j <= n; j++)
            if (matrix[i - 1][j - 1] == 1) {
                dp[i][j] = 1 + Math.min(dp[i - 1][j],
                              Math.min(dp[i][j - 1], dp[i - 1][j - 1]));

Pattern · grid DP

State is the cell itself. Read up, left, and (sometimes) diagonal. Choose your combinator by the question: sum for counting, min/max for optimizing, 1 + min(three) for "largest uniform region". Seed the first row and column, sweep row-major, read the corner.

Family II — Two-Sequence DP

Aligning two sequences

Now the two indices stop being spatial coordinates and become positions in two different strings. dp[i][j] answers a question about the first i characters of A against the first j characters of B. This is the richest and most examined family in all of 2D DP, because alignment underlies diff tools, spell-checkers, DNA sequence comparison, and a long tail of contest problems.1 The defining move is comparing A[i-1] with B[j-1] and branching on whether they match.

Longest Common Subsequence

A subsequence keeps relative order but may drop characters. The longest common subsequence (LCS) of two strings is the longest sequence appearing in both. When the current characters match, they can both extend the best alignment of the two shorter prefixes, so we add one to the diagonal. When they differ, we must drop one of them, taking the better of "ignore A's last character" or "ignore B's last".1

if A[i-1] == B[j-1]: dp[i][j] = dp[i-1][j-1] + 1
else: dp[i][j] = max(dp[i-1][j], dp[i][j-1])

func longestCommonSubsequence(a, b string) int {
    m, n := len(a), len(b)
    dp := make([][]int, m+1)
    for i := range dp {
        dp[i] = make([]int, n+1)
    }
    for i := 1; i <= m; i++ {
        for j := 1; j <= n; j++ {
            if a[i-1] == b[j-1] {
                dp[i][j] = dp[i-1][j-1] + 1
            } else {
                dp[i][j] = max(dp[i-1][j], dp[i][j-1])
            }
        }
    }
    return dp[m][n]
}

    for (let i = 1; i <= m; i++)
        for (let j = 1; j <= n; j++)
            dp[i][j] = a[i - 1] === b[j - 1]
                ? dp[i - 1][j - 1] + 1
                : Math.max(dp[i - 1][j], dp[i][j - 1]);
    return dp[m][n];
}
Reading the table. Every cell dp[i][j] is the LCS length of the first i characters of A and the first j of B. Row 0 and column 0 are all zeros — the LCS of anything with the empty string is empty — which is exactly why the padded convention pays off: the match branch reads dp[i-1][j-1] without a bounds check. The answer waits in the bottom-right corner, and the path that produced it can be recovered by walking backwards, a trick we return to in the section on reconstruction.

Edit Distance

Edit distance — the Levenshtein distance — is the minimum number of single-character insertions, deletions, and substitutions that turn one string into another. It is the optimization twin of LCS: where LCS counts what two strings share, edit distance counts what separates them. The Wagner–Fischer algorithm that tabulates it is one of the oldest and most cited dynamic programs in computer science.3

The recurrence turns on a single comparison. If A[i-1] equals B[j-1], the last characters cost nothing and we inherit dp[i-1][j-1] unchanged. Otherwise we pay one operation on top of the cheapest of three predecessors: dp[i-1][j] is a deletion from A, dp[i][j-1] is an insertion into A, and dp[i-1][j-1] is a substitution. The base cases are no longer zeros: dp[i][0] = i because emptying a string of length i takes i deletions, and dp[0][j] = j symmetrically.

Recurrence. dp[i][j] = dp[i-1][j-1] if A[i-1] == B[j-1], else 1 + min(dp[i-1][j], dp[i][j-1], dp[i-1][j-1]). Seed the borders with dp[i][0]=i and dp[0][j]=j — the single most common source of wrong answers here is leaving them at zero.

Variations on the alignment table

Once the two-sequence skeleton is in hand, a whole catalogue of problems is just a change to the match and mismatch branches. The table shape, the iteration order, and the padded borders never move.

Distinct Subsequences. Count how many distinct subsequences of A equal the whole of B. Here dp[i][j] is the number of ways the first i characters of A can spell the first j of B. Family IIIString matching with a pattern

String-matching DP looks like two-sequence DP — two indices, a padded matrix, the same loop shape — but one of the two strings is a pattern whose characters carry special powers. A single pattern character can match zero things, one thing, or many things, and that elasticity is what forces a table rather than a linear scan. The two canonical problems are regular-expression matching and wildcard matching; they differ only in how generously the special characters behave.

Regular expression matching

Match a string s against a pattern p in which . matches any single character and * matches zero or more of the character immediately before it. The whole string must match the whole pattern. The presence of * is what makes this two-dimensional: a star can erase its preceding element or repeat it, and you cannot know which choice is right without looking ahead, so you tabulate both.7

Let dp[i][j] be true when the first i characters of s match the first j of p. When p[j-1] is an ordinary character or ., the cell is true exactly when the previous cell dp[i-1][j-1] is true and the characters line up. When p[j-1] is *, two doors open. The zero-match door drops the star and its preceding character, reading dp[i][j-2]. The many-match door applies only when the preceding pattern character matches the current text character, and it consumes one character of s while keeping the star, reading dp[i-1][j]. The cell is true if either door is.

The empty-string seam. dp[0][0] is true, and the first row needs care: a pattern like a*b*c* can match the empty string, so for each * at position j you must propagate dp[0][j] = dp[0][j-2]. Forgetting this row is the classic regex-DP bug — the table is correct everywhere except the one place the recursion bottoms out.

Family IVInterval DP

Interval DP is the family that breaks the comfortable up-and-left habit. Here dp[i][j] describes a contiguous interval of a single sequence — from index i to index j — and a cell depends on shorter intervals nested inside it. Because a longer interval is built from shorter ones, you cannot fill row by row; you fill by increasing interval length, or equivalently iterate i downward while j runs upward, so that every sub-interval a cell needs is already done.8

Iteration order is the whole game. A grid DP that loops i and j both ascending will quietly read garbage in interval DP. The fix is to let the outer loop be interval length (1, 2, 3, …) or to run the row index from high to low. Get this wrong and the table looks plausible but the answer is nonsense.

Longest palindromic subsequence

A clean on-ramp to interval thinking: find the longest subsequence of a string that reads the same forwards and backwards. Let dp[i][j] be the answer for the substring from i to j. If the endpoints match, they wrap a palindrome and contribute two to whatever lives inside: 2 + dp[i+1][j-1]. If they differ, drop one end or the other and take the better: max(dp[i+1][j], dp[i][j-1]). Single characters are palindromes of length one — the diagonal of the table — and we build outward from there.8

Burst balloons

Burst Balloons is the problem that makes people respect interval DP. You have a row of balloons, each painted with a number; bursting balloon k earns nums[left] · nums[k] · nums[right] coins where left and right are the still-unburst neighbours at that moment. Maximize the total. The trap is that the neighbours change as balloons pop, so the naive "which balloon first" decision couples everything to everything.8

The unlock is to reason about the last balloon to burst in an interval rather than the first. Pad the array with virtual 1s at both ends. Define dp[left][right] as the most coins obtainable by bursting every balloon strictly between left and right. If balloon k is the last one to go in that open interval, its neighbours at that instant are exactly left and right — they are fixed, because everything else between them is already gone. So the gain is nums[left]·nums[k]·nums[right] + dp[left][k] + dp[k][right], and we maximize over every split point k. That reframing — fixing the last action so the boundaries stop moving — is the signature trick of the whole family, shared with matrix-chain multiplication and optimal binary search trees.1

O(n³), and that's expected. Two indices define the interval and a third scans the split point, so interval DP is cubic in the sequence length rather than quadratic. For contest constraints this means interval DP is viable up to a few hundred elements, not a few thousand — a sizing instinct worth internalizing before you start coding.

func maxCoins(nums []int) int {
    n := len(nums)
    vals := make([]int, n+2)
    vals[0], vals[n+1] = 1, 1
    for i, v := range nums {
        vals[i+1] = v
    }
    sz := n + 2
    dp := make([][]int, sz)
    for i := range dp {
        dp[i] = make([]int, sz)
    }
    for length := 2; length < sz; length++ {
        for left := 0; left+length < sz; left++ {
            right := left + length
            for k := left + 1; k < right; k++ {
                gain := vals[left]*vals[k]*vals[right] + dp[left][k] + dp[k][right]
                if gain > dp[left][right] {
                    dp[left][right] = gain
                }
            }
        }
    }
    return dp[0][sz-1]
}

Family VKnapsack tables

The 0/1 knapsack is the archetype of a 2D DP whose two axes are different kinds of thing: one index ranges over items, the other over a resource budget. Given items with weights and values and a capacity W, choose a subset that maximizes value without exceeding the weight budget — each item taken at most once. Let dp[i][w] be the best value using the first i items within capacity w. For each item you either skip it, inheriting dp[i-1][w], or take it when it fits, gaining dp[i-1][w - weight[i]] + value[i].2

The "first i items" axis is what makes the choice clean: because every row only reads the row above it, the take/skip decision never double-counts an item. This is also the hook for the classic space optimization — iterate the weight axis backwards in a single row — which we cover next alongside the general technique.

func knapsack(weights, values []int, capacity int) int {
    n := len(weights)
    dp := make([][]int, n+1)
    for i := range dp {
        dp[i] = make([]int, capacity+1)
    }
    for i := 1; i <= n; i++ {
        for c := 0; c <= capacity; c++ {
            dp[i][c] = dp[i-1][c]
            if weights[i-1] <= c {
                take := dp[i-1][c-weights[i-1]] + values[i-1]
                if take > dp[i][c] {
                    dp[i][c] = take
                }
            }
        }
    }
    return dp[n][capacity]
}

Why the row-above structure matters. Counting problems (coin-change ways), subset-sum feasibility, and the bounded knapsack are all the same table with the body verb changed — sum of ways, logical OR, or max value. The item axis stays; only the cell's combine rule moves. This is the knapsack analogue of the alignment-family insight from earlier.

Space optimization and reconstruction

Two refinements turn a working table into a contest-grade one: shrinking its memory, and recovering the actual solution rather than just its score.

Rolling rows. Whenever a cell reads only the current row and the one directly above — which covers grid DP, every two-sequence problem, and pattern matching — the full matrix is wasteful. You only ever need two rows, and with a little care, one. The LCS below keeps a single array, overwriting it left to right while stashing the soon-to-be-clobbered diagonal value in a temporary. This drops space from O(mn) to O(min(m,n)) without touching the time bound.5

longestCommonSubsequence(a, b) — O(n) space, two rows
func longestCommonSubsequence(a, b string) int {
    m, n := len(a), len(b)
    prev := make([]int, n+1)
    curr := make([]int, n+1)
    for i := 1; i <= m; i++ {
        for j := 1; j <= n; j++ {
            if a[i-1] == b[j-1] {
                curr[j] = prev[j-1] + 1
            } else {
                curr[j] = max(prev[j], curr[j-1])
            }
        }
        prev, curr = curr, prev
    }
    return prev[n]
}

The cost of rolling. A one-row table gives you the optimal value but throws away the information needed to reconstruct the optimal choice. If the problem asks for the actual subsequence, the alignment, or the chosen items — not just their count or score — keep the full matrix, or use Hirschberg's divide-and-conquer to get the sequence back in linear space at the cost of a constant factor in time.5

Reconstruction. The value lives in one corner; the decisions live in the path that reaches it. Start at the answer cell and walk backwards, at each step asking which predecessor the recurrence chose. For LCS: if the characters matched, the move came from the diagonal and that character belongs to the answer — emit it and step to (i−1, j−1); otherwise step toward the larger of up or left. Reverse what you collected. The same backtracking recovers the edit script for Wagner–Fischer (which neighbour you came from names the operation) and the balloon order for interval DP (the stored split point k).3
