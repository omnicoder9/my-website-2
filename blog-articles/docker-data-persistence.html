The Persistence Problem

An Editorial on State, Storage, and the Ephemeral Machine
Cloud-Native Engineering · Storage Architecture

Containers were designed to forget. Your data needs to remember.

Docker made it trivial to package an application. But the same isolation that gave containers their elegance also made them amnesic by default — a constraint that shaped a decade of patterns, plugins, and protocols for keeping state alive outside the box.

§ 01 — Prologue

A Brief History of Forgetting

On 15 March 2013, Solomon Hykes walked onto the lightning-talk stage at PyCon in Santa Clara and demonstrated, in five minutes, an unreleased tool that the dotCloud team had built to solve their own platform-as-a-service problem. He noted that, for developers, "shipping code to the server is hard."[12] The audience — expecting perhaps thirty people in a side room, but instead facing a main-stage crowd of several hundred — reacted with a level of excitement that surprised the speaker.[13] Someone posted the unfinished project page to Hacker News and accused the team of vaporware. Two weeks later, dotCloud open-sourced the engine.[13]

That engine was Docker. It was written in Go, used the Linux kernel's existing namespace and cgroup primitives, and originally relied on LXC and AuFS to do its work; one year later, with the release of version 0.9, Docker replaced LXC with its own component, libcontainer.[1] The company renamed itself from dotCloud, Inc. to Docker, Inc. in 2013, and the project rapidly became the substrate on which an entire generation of cloud-native tooling was built.[1]

But Docker shipped with a property that was, depending on one's perspective, either a feature or a bug: containers were stateless. A container's filesystem was an ephemeral, copy-on-write overlay on top of a stack of immutable image layers, and anything written inside it vanished the moment the container was removed. For the new generation of microservices, that was a virtue. For databases, queues, content stores, machine-learning checkpoints, and every other workload that needed to remember anything, it was a problem.

This article is about that problem and the layered, sometimes contradictory, set of mechanisms the Docker ecosystem evolved to solve it — from the writable container layer to bind mounts, from named volumes to volume driver plugins, and ultimately to the cross-orchestrator Container Storage Interface that now mediates between containerized workloads and the storage systems of every major cloud.

§ 02 — Foundations

The Ephemeral Container

To understand why persistence is a separate problem, it helps to look at what a container's filesystem actually is. A Docker image is not a monolithic blob. It is a stack of read-only layers, each produced by an instruction in a Dockerfile and stored independently on disk. When the engine starts a container from that image, it places a thin writable layer on top of the stack and presents the union of all those layers as a single filesystem to the process inside.[4]

The mechanism that makes this work is called a union filesystem. On modern Linux hosts, the default storage driver is overlay2, which uses the kernel's OverlayFS implementation — merged into the mainline kernel at version 3.18 and requiring at minimum kernel 4.0 for the overlay2 driver.[5] OverlayFS combines a stack of lower (read-only) directories with an upper (read-write) directory and exposes a merged view to userspace. A container reading a file traverses the layers top-down until the file is found; writes always go to the upper layer.[5]

When a container modifies a file that exists only in a lower image layer, the storage driver performs a copy-on-write (CoW): the file is copied up to the writable layer before being changed.[6] This means image layers stay immutable and shareable across many containers, and the writable layer stays as small as possible. But it also means two important things for persistence.

Figure 1 · Container Filesystem Stack

First, writing through the union filesystem is slower than writing to the underlying host disk directly: every CoW operation has to copy the full file (not just the changed bytes) before applying modifications, which can introduce real latency for large files.[5] Volumes bypass this overhead entirely; they mount directly on the backing filesystem.[7]

Second, and more importantly, the writable layer is bound to the container's lifecycle. The Docker documentation puts the consequence plainly: data written to the container's writable layer "is destroyed along with the container" and cannot easily be moved elsewhere.[2] An nginx container that logs to /var/log/nginx inside the writable layer, a Postgres container that writes to /var/lib/postgresql/data, an application that caches uploads to /uploads — all of them lose everything the moment they are removed.

A container's writable layer is destroyed along with the container. To make data survive its container, you have to place it somewhere else. — Docker Engine Storage Documentation[2]

Docker's answer to this problem is a small, deliberate vocabulary of options for placing data outside the writable layer. There are three: volumes, bind mounts, and tmpfs mounts. The first two persist data to the host disk; the third keeps it in memory. The rest of this article is, in essence, an exploration of how each works, when to reach for which, and how the ecosystem has extended the volume mechanism to reach storage systems that live far beyond the host.

§ 03 — The Canonical Path

Volumes: The Canonical Path

Volumes are the Docker-managed solution to persistence. When you create a volume, Docker provisions a directory under its own storage area — by default /var/lib/docker/volumes/ on Linux hosts — and takes responsibility for its lifecycle.[3] The official documentation is unambiguous about the preference: "Volumes are the preferred mechanism for persisting data generated by and used by Docker containers."[3]

The reason for that preference is decoupling. Volumes are not dependent on the host's directory layout or operating system. They can be created, listed, inspected, backed up, and destroyed through the Docker CLI without touching the host filesystem directly. They can be attached to multiple containers at once. And on Docker Desktop for Mac and Windows, where the underlying filesystem is not native Linux, volumes perform substantially better than bind mounts.[3]

Creating and using a named volume

The basic interaction surface is small. Volumes can be created explicitly with the docker volume create command, or implicitly the first time a container references one.

Shell · Creating and attaching a named volume
# Explicitly create a named volume
docker volume create app-data

# List, inspect, and remove
docker volume ls
docker volume inspect app-data
docker volume rm app-data

# Attach to a container using the modern --mount syntax
docker run -d \
  --name postgres \
  --mount source=app-data,target=/var/lib/postgresql/data \
  -e POSTGRES_PASSWORD=secret \
  postgres:16

# Equivalent using the older shorthand -v
docker run -d \
  --name postgres \
  -v app-data:/var/lib/postgresql/data \
  -e POSTGRES_PASSWORD=secret \
  postgres:16

The two flags — --mount and -v — do the same job, but they differ in temperament. --mount is more verbose and explicit, parsing its options as comma-separated key/value pairs; -v is a colon-separated shorthand. The shorthand has a notable quirk: if you reference a volume that does not yet exist with -v, Docker will silently create it. With --mount, the same reference produces an error. For that reason, --mount is the recommended syntax in production scripts, where surprises are not a virtue.[2]

Anonymous and named volumes

There are two kinds of volumes. A named volume is one with a human-readable identifier (like app-data above) and a stable lifetime independent of any specific container. An anonymous volume is one Docker creates on the fly when an image's Dockerfile declares a VOLUME instruction or when a container is started with -v /some/path (no source name); these are given a long hexadecimal ID and are easily forgotten on disk.

Anonymous volumes accumulate quickly in development. The docker volume prune command removes any volume not currently in use by at least one container; running it periodically is a sensible hygiene practice on developer machines.

Sharing data and pre-populating from an image

Because volumes are independent objects, the same volume can be attached to multiple containers at once. This is useful for sidecar patterns (a logging agent reading from the same directory the main application writes to) and for sharing static assets among horizontally scaled workers. It is, however, the application's responsibility to coordinate concurrent writes — the filesystem will not do it for you.

Docker also has a small but useful behavior for first-time mounts: if a volume is empty and you mount it into a non-empty directory inside a container, the contents of that directory in the image are copied into the volume before it is mounted.[3] This is how, for instance, the official Postgres image pre-populates a fresh volume with its initial data directory the first time it starts.

Note · Backups
Because volumes live under /var/lib/docker/volumes/ on the host, you can back them up by running a short-lived container that mounts the volume and writes a tar archive to the host: docker run --rm -v app-data:/data -v $(pwd):/backup alpine tar czf /backup/app-data.tgz -C /data .. Restoring follows the symmetric pattern. This idiom — "the throwaway container with two mounts" — is one of the standard Docker plays.
§ 04 — The Host Window

Bind Mounts: The Host Window

If volumes are Docker's preferred abstraction, bind mounts are its oldest one. Bind mounts have been available in Docker since its earliest days for persisting data,[14] and they do exactly what their name suggests: they map a specific file or directory on the host machine into a path inside the container. Unlike volumes, where Docker manages everything under /var/lib/docker/volumes/, a bind mount uses an arbitrary absolute path on the host that the user chooses.[15]

This has a few important consequences. The first is that the data is not isolated by Docker. Both processes inside the container and processes outside it — any user on the host with the right permissions, any other application, any cron job — can read and modify the same files simultaneously.[4] That property is precisely what makes bind mounts indispensable in some contexts and dangerous in others.

The development use case

The classic legitimate use is local development. When you bind-mount your project's source directory into a container, edits in your editor on the host are reflected immediately inside the container. There is no rebuild step, no image push, no container restart for code changes — the file the running process is reading is, in a real sense, the file your editor is writing.

Shell · Bind-mounting a project directory
# Bind-mount the current working directory into /app inside the container
docker run -it --rm \
  --mount type=bind,source="$(pwd)",target=/app \
  -w /app \
  node:20 \
  npm run dev

# Same thing with the -v shorthand
docker run -it --rm \
  -v "$(pwd)":/app \
  -w /app \
  node:20 \
  npm run dev

# Read-only bind mount for configuration that the container shouldn't modify
docker run -d \
  --mount type=bind,source=/etc/myapp/config.yml,target=/etc/myapp/config.yml,readonly \
  myapp:latest

The semantic differences between --mount and -v become more pronounced with bind mounts than with volumes. With -v, if the host path does not exist, Docker creates it (as a directory) automatically; with --mount, the same situation produces an error.[2] The --mount behavior — fail loudly when something is missing — is what you want in production. The -v behavior is convenient interactively but has been the source of countless puzzled debugging sessions in which a typo in a path silently produced an empty directory at an unexpected location on the host.

Why bind mounts are "limited"

Compared to volumes, bind mounts have a clear set of limitations. They depend on the host's directory structure and operating system, so a docker run command that works on a developer's laptop may fail on a CI runner where the path does not exist. They cannot be managed through the Docker CLI — there is no docker bind command analogous to docker volume. They cannot be driven by storage plugins to back onto cloud services. And they expose the host's filesystem to whatever processes are running inside the container, which is a security concern whenever the container does not need that surface.[3]

The conventional wisdom — expressed across the LogRocket guide, the Docker docs, and the broader community — is that bind mounts should be reserved for cases where you specifically need shared access between host and container, and volumes should be used for everything else.[14] Use bind mounts for source code in development, for read-only configuration files, for sharing Unix sockets like /var/run/docker.sock. Use volumes for databases, queues, application state, and anything that needs to survive a host migration.

§ 05 — Memory as Disk

tmpfs: When You Want Nothing to Persist

The third option exists for a different goal entirely. A tmpfs mount creates a filesystem that lives only in the host's RAM and is destroyed when the container stops.[8] It is, in a sense, the opposite of persistence — an explicit guarantee that data will not be written to disk.

This is useful in two scenarios. The first is sensitive data: API tokens, decrypted secrets, session keys, anything that should not leave a trace on the host filesystem after the container exits. The second is performance: certain workloads benefit dramatically from in-memory I/O. A test database that does not need durability, a build tool with heavy intermediate file churn, an Nginx cache directory — all are good candidates.[10]

Shell · tmpfs mount for sensitive temp files
# Mount a 100MB tmpfs at /app/secrets with restrictive permissions
docker run -d \
  --name secure-app \
  --mount type=tmpfs,destination=/app/secrets,tmpfs-size=104857600,tmpfs-mode=0700 \
  my-secure-app

# Shorthand version (no per-mount configuration options)
docker run -d \
  --name secure-app \
  --tmpfs /app/secrets:size=100m,mode=0700 \
  my-secure-app

There are real constraints. tmpfs mounts work only on Linux hosts.[8] They cannot be shared between containers — each container gets its own private tmpfs.[8] They consume RAM, so unbounded growth can starve the host. The tmpfs-size option puts a ceiling on this, and on production hosts it should almost always be set explicitly. Permissions set on a tmpfs may reset after a container restart, which is worth knowing if your application relies on specific file modes inside the mount.[8]

The --tmpfs shorthand and the --mount type=tmpfs form differ in flexibility, much as their volume and bind-mount counterparts do: the shorthand cannot express the full set of options. For anything beyond a size limit and a mode, --mount is required.[8]

§ 06 — The Wider Ecosystem

Integrations and the CSI Era

Everything described so far happens on a single Docker host. But the question of where containers' data should live becomes much more interesting when you have a cluster — when containers can move between hosts, and the volume that was on Host A yesterday needs to be reachable from Host B today. The Docker ecosystem evolved two distinct extension points to address this, one specific to Docker and one that has since become an industry standard.

Docker volume plugins

The first extension point is the Docker volume plugin system, sometimes called a volume driver. When you create a volume, you can specify a driver other than the default local:

Shell · Driver-backed volumes
# NFS-backed volume using the built-in local driver with type=nfs
docker volume create \
  --driver local \
  --opt type=nfs \
  --opt o=addr=192.168.1.50,nfsvers=4,rw \
  --opt device=:/exports/shared \
  shared-data

# CIFS / SMB-backed volume (e.g., for Windows file shares)
docker volume create \
  --driver local \
  --opt type=cifs \
  --opt device=//fileserver/share \
  --opt o=username=svc,password=secret,vers=3.0 \
  smb-data

# Install and use an S3 volume plugin
docker plugin install rexray/s3fs --grant-all-permissions
docker volume create -d rexray/s3fs my-bucket

The Docker documentation describes the idea concisely: volume drivers "let you abstract the underlying storage system from the application logic," so that a service using an NFS-backed volume today can be reconfigured to use cloud object storage tomorrow without any change to the service itself.[3] Plugins exist for AWS EBS and EFS, Azure File storage, Google Cloud Persistent Disks, GlusterFS, Ceph, Portworx, IPFS, and many others.[7] Some of these are managed Docker plugins installable via docker plugin install; others run as separate services that implement the Volume Plugin API over a Unix socket.

The architecture is straightforward. A plugin exposes the Volume Plugin API to the Docker daemon — a small set of HTTP-over-Unix-socket calls for Create, Remove, Mount, Unmount, Path, and so on. When a container references a volume managed by the plugin, the daemon calls into the plugin to provision the backing storage and to perform the mount; from the container's perspective, the result is an ordinary filesystem mount.

The Container Storage Interface

Docker's volume plugin API was not the only one of its kind. Kubernetes had its own in-tree volume plugins; Mesos had FlexVolume; Cloud Foundry had its own model. Storage vendors faced a depressing matrix: write and maintain a separate driver for each orchestrator, or simply not support the ones you couldn't afford.

The Container Storage Interface (CSI) was proposed in 2017 as a joint effort between contributors from Docker, Kubernetes, Mesos, and Cloud Foundry to fix this fragmentation with a single, cluster-level standard.[9] The first specification, CSI v0.1, was published in December 2017.[16] Kubernetes 1.9 shipped an alpha implementation in early 2018,[10-csi] and CSI reached general availability in Kubernetes 1.13 in January 2019.[17]

The promise was modest but consequential: a storage plugin written once to the CSI specification should run unmodified across any conforming orchestrator. The mechanism is a gRPC API exposed over a Unix domain socket, structured around three services — Identity, Controller, and Node — that together cover the full lifecycle of a volume from provisioning through attachment, mounting, and teardown.[17]

Figure 2 · CSI Architecture (Generic)

The practical effect is that CSI has become, for orchestrated container workloads, what Docker volume plugins were for single-host Docker: the integration surface between containerized applications and the storage systems they need. Every major cloud provider ships CSI drivers for their block-storage and file-storage offerings; every distributed storage system worth using does too. Kubernetes deprecated its in-tree volume plugins in favor of CSI drivers years ago, completing a migration that the broader ecosystem now treats as a given.[18]

For pure Docker deployments — single-host or Docker Swarm — the older volume plugin system is still the integration point, and it remains adequate for most non-clustered workloads. For anything orchestrated by Kubernetes, CSI is the answer, and the volume itself is a higher-level Kubernetes object (a PersistentVolume backed by a PersistentVolumeClaim) that points to a CSI driver in the cluster.

Backup and snapshot tooling

Sitting on top of CSI is a small but rich layer of data-protection tooling. Velero, originally an open-source Kubernetes backup tool, integrates with CSI's volume-snapshot extension to perform consistent backups of stateful workloads.[19] Rook, the Ceph operator for Kubernetes, exposes Ceph's snapshotting and replication capabilities through CSI. Portworx, Pure Storage's PX-Backup, and a number of vendor-specific tools play in the same space. Backup is, in the modern stack, increasingly a property of the storage layer rather than the application, and CSI is what makes that decoupling possible.

§ 07 — In Practice

Patterns in Production

A few concrete patterns recur across the production Docker stacks one is likely to encounter in the wild. None of them are surprising once stated; all of them are worth stating, because they encode hard-won experience about how the abstractions described above interact with real workloads.

The database pattern

Stateful services — Postgres, MySQL, MongoDB, Redis with persistence, and so on — should always use named volumes, never bind mounts.[14] On Linux, this gives you the best write performance the storage stack can offer; on Docker Desktop, it avoids the substantial penalty of bind mounts crossing the VM boundary. Volumes are also easier to back up via the throwaway-container idiom shown earlier, and they survive container recreation cleanly — an important property when the next deploy will destroy and recreate the container.

docker-compose.yml · A typical stateful service
services:
  postgres:
    image: postgres:16
    restart: unless-stopped
    environment:
      POSTGRES_PASSWORD_FILE: /run/secrets/db_pass
    volumes:
      - pgdata:/var/lib/postgresql/data    # named volume for state
      - ./initdb:/docker-entrypoint-initdb.d:ro   # bind mount, read-only, for init scripts
    tmpfs:
      - /tmp:size=100M                     # ephemeral scratch
    secrets:
      - db_pass

volumes:
  pgdata:
    # default 'local' driver; could swap for NFS, EBS, Ceph, ...

secrets:
  db_pass:
    file: ./secrets/db_pass.txt

The example above shows all three mount types working together: a named volume for persistent state, a read-only bind mount for one-time initialization scripts, and a tmpfs for scratch space. This is a common shape.

The development pattern

For local development, bind-mount your source code into the container so that editor edits are reflected immediately, and keep dependencies (a node_modules volume, a .cargo cache volume) in named volumes so they survive container rebuilds and don't conflict with anything the host has installed. The slightly counterintuitive trick — mounting a named volume over a directory that exists in the bind-mounted source — is how you keep the container's view of node_modules separate from whatever your host happens to have:

docker-compose.yml · Local development with hot reload
services:
  app:
    build: .
    command: npm run dev
    ports: ["3000:3000"]
    volumes:
      - .:/app                  # bind: source code, hot-reload
      - node_modules:/app/node_modules   # named volume: container's own deps
volumes:
  node_modules: {}

The secrets pattern

Sensitive material that must not be written to disk should go through Docker secrets (which mount as in-memory files at /run/secrets/<name> in Swarm) or, in standalone Docker, through a tmpfs mount with a tight size limit and mode 0700. Never bake credentials into images; never bind-mount your host's ~/.aws directory into a production container.

The "do not use volumes for this" pattern

Volumes are not the right tool for everything. They are a poor fit for distributed databases that have their own replication and clustering protocols, where a shared filesystem can defeat the consistency guarantees the database itself is providing. They are a poor fit for object storage workloads — if you want to put files in S3, use the S3 API from the application, do not bolt on an S3-backed volume plugin that pretends S3 is a POSIX filesystem (it is not, and the impedance mismatch shows up in latency, atomicity, and cost). They are a poor fit for ephemeral caches that should reset on container restart — that is what tmpfs is for.

A general rule of thumb

The LogRocket guide puts it as succinctly as anyone: if you are ever in doubt about what mechanism to reach for, use volumes.[14] Reach for bind mounts when you specifically need shared access between host and container. Reach for tmpfs when you specifically need data not to persist. Everything else is a volume.

§ Coda

Coda: The Shape of the Problem

There is something slightly philosophical about the persistence problem in containers. The whole appeal of the container model — the property that made Docker spread the way it did — was that containers were small, fast, immutable, and disposable. You could build them, ship them, and throw them away without consequence. The instant you needed to remember anything, that elegance had to be punctured. State had to live somewhere that was, by design, outside the model.

The mechanisms described in this article are the puncture points: volumes, bind mounts, tmpfs, and the wider ecosystem of plugins and CSI drivers that extend them. They are, taken together, the answer to a question that the container model itself refuses to answer. They are also, fittingly, the part of the stack where the most serious operational decisions get made — because state is where workloads accumulate value, and the choice of how to persist it determines what kinds of failures are recoverable and what kinds are not.

The good news, in 2026, is that the abstractions have matured. The volume API and the CSI specification have both stabilized; the major storage systems have first-class integrations; the tooling for backups, snapshots, and migrations is no longer a research problem. The bad news, such as it is, is that nothing about persistence is as cheap or as automatic as the container model made the rest of the application lifecycle look. The data is still the hardest part.

It was always going to be.
