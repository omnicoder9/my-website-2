Bit manipulation is the practice of treating an integer not as a number but as what it physically is: a fixed-width row of binary digits. Once you adopt that view, a surprising number of problems collapse. A linear scan with a hash set becomes a single XOR. A nested loop over subsets becomes a counter from 0 to 2ⁿ. The cost is conceptual, not computational — you must learn to read what the operators do to the bits, not what they do to the value.

This guide is organised as a problem set. We begin with the operators and the dozen identities worth memorising, then work through eleven problems that recur in interviews and contests. Each comes with the key insight, a worked binary trace where it helps, complexity, common variations, and a reference implementation in six languages — switch between them with the bar at the top of the page, or the tabs on any code block.1,2

Section 00 — Foundations

The Primitives

Every technique in this guide is assembled from six operators. They act bit-by-bit and in parallel: a single 64-bit AND processes all 64 columns in one machine instruction. That parallelism is the whole reason bit tricks are fast.

OperatorSymbolRuleReading
AND&1 only if both bits are 1Masking — keep / test bits
OR|1 if either bit is 1Setting — turn bits on
XOR^1 if the bits differToggling — flip / cancel bits
NOT~Flip every bitComplement — one's complement
Left shift<<Move bits up, fill 0Multiply by 2ᵏ
Right shift>>Move bits downDivide by 2ᵏ (sign-aware)

Three properties of XOR do most of the heavy lifting and deserve to be internalised. XOR is its own inverse, it has an identity, and it does not care about order:

# the three XOR identities x ^ x = 0 // self-inverse — a value cancels itself x ^ 0 = x // identity — 0 leaves a value untouched a ^ b = b ^ a // commutative & associative — order is free

Put together, these say that XOR-ing a long stream of values makes every pair of equal values vanish, regardless of how they were interleaved. That single fact powers problems 01, 07, 08 and 09.

The Anatomy of an Integer

A 32-bit signed integer is a row of 32 cells. Bit 0 (rightmost) is the units place; bit 31 (leftmost, outlined in teal) is the sign bit in two's-complement representation. Click any cell to toggle it and watch how the operators below reshape the pattern.

Two buttons there are worth dwelling on. n & (n-1) always erases exactly one bit — the lowest set bit — because subtracting 1 flips that bit to 0 and turns every 0 below it into a 1, so the AND keeps only the higher bits. n & -n does the opposite: it isolates that same bit and discards everything else. These two appear constantly.3

The Essential Idioms

Below is the reference card. Each idiom is a small composition of the six operators that has a name and a purpose. Memorise the expression and, more importantly, why it works — contests reward recognising these on sight.

x & (1 << k)
Test bit k: non-zero iff that bit is set.
x | (1 << k)
Set bit k to 1, leaving the rest alone.
x & ~(1 << k)
Clear bit k to 0.
x ^ (1 << k)
Toggle bit k.
x & (x - 1)
Clear the lowest set bit. Loop until 0 to count bits.
x & -x
Isolate the lowest set bit. Core of the Fenwick tree.
x & (x - 1) == 0
True when x is 0 or a power of two.
x | (x - 1)
Set all bits below the lowest set bit.
x ^ (x >> 1)
Convert binary to Gray code.
~x + 1
Two's-complement negation — equals -x.
(x >> k) & 1
Read the value of bit k as 0 or 1.
x << 1 | x >> 1
Neighbour mask — common in grid & DP problems.
Why & (x-1) clears the lowest bit

Take x = 101102. Then x-1 = 101012 — subtraction borrowed through the trailing zeros, flipping the lowest 1 off and the zeros below it on. AND-ing the two keeps only the bits they agree on, which are exactly the bits above the one that flipped: 101002. One set bit gone, every time.

01
XOR Cancellation

Single Number

Problem

Given a non-empty array in which every element appears exactly twice except for one, find the element that appears only once. Required: linear time, constant extra space.

The constant-space requirement rules out a hash set. But recall the XOR identities: x ^ x = 0 and x ^ 0 = x. If we XOR every element of the array together, each duplicated value pairs off and cancels to zero, leaving only the lonely element XOR-ed against zero — that is, the lonely element itself. Order is irrelevant because XOR is commutative.

# array = [4, 1, 2, 1, 2] — fold left with XOR 0 ^ 4 = 4 4 ^ 1 = 5 5 ^ 2 = 7 7 ^ 1 = 6 6 ^ 2 = 4 // the pairs cancelled; 4 survives
Time O(n) Space O(1) One pass

Variations

  • Missing number. If the array should contain 0…n with one value missing, XOR the values together and XOR in every index plus n — see problem 07.
  • Find the duplicate. The same fold finds a single duplicate among otherwise-unique values.
  • Appears three times. XOR no longer cancels triples; this needs a different counting scheme — see problem 08.
02
Population Count

Counting Set Bits

Problem

Given a 32-bit integer, return how many of its bits are set to 1 — its Hamming weight or population count.

The naïve method tests the last bit with n & 1 and shifts right, looping 32 times. It works, but it always does 32 iterations regardless of the input. Brian Kernighan's algorithm is better: n & (n - 1) clears the lowest set bit, so looping until n reaches zero runs exactly once per set bit. A number with three 1s costs three iterations, not thirty-two.3,4

The fixed-width loop pays for every column; Kernighan's loop pays only for the columns that are lit.
Time O(set bits) Space O(1) ≤ 32 iterations

Variations & the production answer

  • Hardware intrinsics. In a real codebase you would call the built-in: bits.OnesCount32 (Go), Integer.bitCount (Java), std::popcount (C++20), u32::count_ones (Rust), BitOperations.PopCount (C#). These compile to a single POPCNT instruction.5
  • Hamming distance. The number of differing bits between a and b is the population count of a ^ b.
  • Parallel SWAR count. For constant-time counting without intrinsics, the bit-twiddling "divide and conquer" sum collapses pairs, then nibbles, then bytes.
03
Bitwise Dynamic Programming

Counting Bits, 0 to n

Problem

Given an integer n, return an array ans of length n + 1 where ans[i] is the number of set bits in i.

You could call problem 02 on every value, giving O(n log n). But the answers are not independent. Dropping the lowest bit of i — that is, computing i >> 1 — produces a smaller number whose answer we have already calculated. The bit we shifted off is simply i & 1. That gives the recurrence:

dp[i] = dp[i >> 1] + (i & 1) # i = 13 = 1101 # i >> 1 = 6 = 110 -> dp[6] already known = 2 # i & 1 = 1 # dp[13] = 2 + 1 = 3

Each entry is one array lookup and two cheap operations, so the whole table is built in a single linear pass. An equivalent recurrence, dp[i] = dp[i & (i-1)] + 1, uses Kernighan's idea instead of a shift.

Time O(n) Space O(n) DP on bit structure

Why this matters

This problem is the gateway to bitmask dynamic programming: states are integers, and transitions are bit operations on those integers. The Travelling Salesman Problem, assignment problems, and subset-sum DP all lean on precomputing popcounts of masks exactly this way.1,2

04
Bit-by-Bit Construction

Reverse Bits

Problem

Given a 32-bit unsigned integer, return the integer obtained by reversing the order of all 32 bits.

Build the result one bit at a time. Each iteration peels the last bit off the input with n & 1, then makes room in the result by shifting it left and OR-ing the peeled bit into the freshly vacated slot. After 32 rounds the first input bit has been pushed all the way to the top of the result.

# input = ...0011 (low bits shown) result = 0 result = (result << 1) | (n & 1) // take bit, n >>= 1 # round 1: result = 1 # round 2: result = 11 # round 3: result = 110 ... and so on for 32 rounds
Language trap — the right shift

The input must be shifted with a logical (zero-filling) shift. Java's >> is arithmetic and copies the sign bit; you must use >>>. JavaScript/TypeScript has the same split — use >>> and finish with >>> 0 to read the result as unsigned. Go, Rust, C++ and C# avoid the issue by using genuinely unsigned types.

Time O(32) Space O(1) Constant work

Variations

  • Reverse in O(log w). Swap adjacent bits, then pairs, then nibbles, then bytes, then halves — five masked swaps reverse a 32-bit word.
  • Repeated calls. If reverseBits is called many times, precompute a 256-entry byte-reversal table and assemble the word from four lookups.
05
Arithmetic From Logic

Sum Without + or −

Problem

Compute the sum of two integers a and b without using the + or - operators.

Reconstruct what an adder circuit does. Addition in any column produces two things: a sum bit and a carry into the next column. XOR is addition that ignores the carry — 1 ^ 1 = 0, exactly the sum bit with the carry dropped. AND detects precisely the columns where a carry is generated — 1 & 1 = 1 — and that carry belongs one column to the left, so shift it up by one.

# loop: fold the carry back in until there is none sum = a ^ b // addition without carries carry = (a & b) << 1 // the carries, moved left one place a, b = sum, carry // repeat until carry (b) is 0

Each iteration moves the carry at least one bit higher, so within the width of the integer the carry runs out of room and the loop terminates — at which point a holds the answer. Subtraction is the same routine with b negated, and negation is ~b + 1.

Language trap — signed overflow

Left-shifting a value into the sign bit is undefined behaviour for signed integers in C++ and panics in debug builds of Rust. The implementations below compute the carry in an unsigned type and cast back. Go, Java and C# define wrap-around for their signed integers, so the plain expression is safe there.

Time O(32) Space O(1) Full adder

Structure Detection

Power of Two

Problem

Determine whether a positive integer is an exact power of two.

A power of two has exactly one set bit. Clearing the lowest set bit of such a number leaves zero — so n & (n - 1) == 0 is true precisely when n has zero or one set bits. Guard with n > 0 to exclude zero and negative inputs, and the test is a single branch with two operations.

Time O(1)Space O(1)

Related one-liners

  • Power of four. A power of two whose single bit sits in an even position: n > 0 && (n & (n-1)) == 0 && (n & 0x55555555) != 0.
  • Round up to a power of two. Smear the highest bit downward with shifts and ORs, then add one.
07
XOR Cancellation

Missing Number

Problem

An array contains n distinct numbers drawn from the range 0…n. Exactly one value from that range is absent — find it.

XOR every index 0…n together with every array value. Every number that is present appears once as an index and once as a value, so it cancels. The one number that is missing appears only as an index, and survives. This avoids both the summation approach (which can overflow) and any extra space.

Time O(n)Space O(1)Overflow-safe
08
Per-Bit State Machine

Single Number II

Problem

Every element of an array appears exactly three times except one, which appears once. Find it in linear time and constant space.

XOR cancels pairs, not triples, so the problem 01 trick fails. Instead, track each bit column modulo 3. Across the whole array, every bit contributed by a tripled value is counted three times and so vanishes mod 3; the lonely value's bits remain. The elegant form uses two accumulators, ones and twos, that together encode a count of 0, 1 or 2 for every column simultaneously and reset to 0 on the third occurrence.3,6

The two-counter state machine

Think of (twos, ones) as a two-bit counter per column, cycling 00 → 01 → 10 → 00. The two update lines advance every column's counter by one wherever the incoming bit is 1. After processing all elements, ones holds the bits seen a number of times not divisible by three — the answer.

Time O(n)Space O(1)Generalises to any k

Variation

A more transparent (if slightly slower) solution loops over all 32 bit positions, counts how many array values have that bit set, and keeps the bit if the count is not a multiple of three. It generalises immediately to "appears k times".

09
Partition by a Bit

Single Number III

Problem

Exactly two elements appear once; every other element appears twice. Return the two singletons, in any order.

XOR-ing the whole array now leaves a ^ b, the XOR of the two unknowns — useful but not yet an answer. The breakthrough: because a and b differ, a ^ b has at least one set bit, and in that bit position the two numbers disagree. Isolate any such bit with x & -x and use it to split the array into two groups. Each group contains exactly one singleton plus pairs, so XOR-ing each group reduces it to problem 01.

# step 1: xorAll = a ^ b (pairs cancel) # step 2: lowBit = xorAll & -xorAll -> a bit where a, b differ # step 3: bucket each value by (value & lowBit), XOR each bucket # bucket-with-bit -> a bucket-without -> b
Time O(n)Space O(1)Two passes

Integer as a Set

Subset Enumeration

Problem

Generate every subset (the power set) of a set of n distinct elements.

An n-bit integer is a subset: bit i set means element i is included. There are exactly 2ⁿ such integers, so a single counter from 0 to (1 << n) - 1 walks through every subset, with no recursion and no backtracking state. For each mask, test bit i with mask & (1 << i) to decide membership.

Time O(n · 2ⁿ)Space O(2ⁿ)

The companion idiom — submask enumeration

A frequent follow-up is iterating only the submasks of a given mask. The loop for (s = mask; s > 0; s = (s - 1) & mask) visits every subset of mask in descending order, and across all masks it costs only O(3ⁿ) in total — the backbone of subset-sum and SOS (sum-over-subsets) dynamic programming.1

Common Prefix

Bitwise AND of a Range

Problem

Given two integers left ≤ right, return the bitwise AND of every integer in the inclusive range [left, right].

AND-ing the whole range one value at a time is too slow when the range is wide. Observe that a bit survives the AND only if it is 1 in every number of the range — and the only bits with that property are the bits of the common binary prefix of left and right. Any bit where they differ flips to 0 somewhere inside the range. So shift both numbers right until they are equal, counting the shifts, then shift the common prefix back into place.

Time O(log range)Space O(1)

Pitfalls & Language Gotchas

Bit code is unforgiving — a wrong shift type produces a plausible but incorrect answer rather than a crash. These are the traps that cost contest points.

Signed versus logical right shift

Arithmetic right shift copies the sign bit; logical right shift fills with zero. Java and JavaScript/TypeScript expose both — >> and >>> — and any algorithm that treats a value as unsigned (reversing bits, hashing, masking) must use >>>. Go, Rust, C++ and C# choose the shift behaviour from the operand's type, so the fix there is to use an unsigned type.

Shift amount out of range

Shifting a 32-bit value by 32 or more is undefined in C and C++, and in Java/JavaScript the shift count is silently taken modulo 32 — so 1 << 32 is 1, not 0. When you need a full-width mask use the language's max-value constant or a 64-bit type.

The 1 << n overflow

In bitmask DP with n ≥ 31, the literal 1 is a 32-bit int and 1 << 31 overflows into the sign bit. Write the literal as a 64-bit value — 1L in Java/C#, 1LL in C++, 1u64 in Rust — before shifting.

Operator precedence

In the C family &, | and ^ bind looser than the comparison operators. x & 1 == 0 parses as x & (1 == 0). Always parenthesise: (x & 1) == 0.

JavaScript's 32-bit ceiling

Every bitwise operator in JavaScript and TypeScript first coerces its operands to signed 32-bit. Numbers above 2³¹ - 1 lose their high bits, and a result you expect to be unsigned must be normalised with >>> 0. For wider work, use BigInt.

Note on negation

The idiom x & -x relies on two's-complement representation, where -x == ~x + 1. Negating the most-negative integer overflows; Rust's debug builds panic on it, which is why the implementations here reach for wrapping_neg when the input could be unbounded.
