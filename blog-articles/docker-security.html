Docker Security  ·  a technical field manual

Everything Is a Process Pretending to Be a Machine

A container is not a small computer. It is an ordinary Linux process wearing a costume stitched from namespaces, cgroups, and capability masks. Docker security is the study of how convincingly that costume holds — the kernel primitives that draw the walls, the escapes that walk straight through them, and the layered discipline that turns a container into a boundary worth trusting.

01The shape of the problem

A container is a process with a good costume

The single most important fact about Docker security is also the one most easily forgotten: a container is not a lightweight virtual machine. There is no second kernel, no hypervisor drawing a hardware boundary, no guest operating system. A running container is an ordinary Linux process — sharing the exact same kernel as the host and every other container on the box — that the kernel has been persuaded to lie to about what it can see and touch.

Docker does not virtualize hardware. It composes a handful of kernel features so that a process believes it owns the machine. Namespaces govern what the process can see — its own process tree, its own network interfaces, its own view of the filesystem. Control groups (cgroups) govern what it can consume — how much CPU, memory, and I/O it is allowed. Capabilities, seccomp, and a Linux Security Module such as AppArmor or SELinux govern what it can ask the kernel to do — which of the roughly three hundred system calls are permitted, and with what privilege.1 Assembled together, these make a process feel like a server.

The security consequence follows directly. None of these mechanisms is a wall by default; each is a dial, and Docker's historical defaults left many of them turned toward convenience rather than confinement. The isolation you get is emergent from configuration, not an intrinsic property of “being a container.” This is why two containers running the identical image can present wildly different attack surfaces depending on the flags they were launched with.

The core asymmetry

Because every container shares the host kernel, a kernel vulnerability exploited from inside a container lands on the host. The 2016 Dirty COW privilege-escalation bug (CVE-2016-5195), executed inside an otherwise well-isolated container, still yielded root on the underlying machine.23 No amount of namespace configuration closes a hole in the code that all namespaces run on top of. Defense has to assume the boundary is porous.

The U.S. National Institute of Standards and Technology captured the shape of this problem in Special Publication 800-190, the Application Container Security Guide, still the canonical government reference on the subject. It organizes container risk into five tiers — image risks, registry risks, orchestrator risks, container (runtime) risks, and host-OS risks — precisely because a container's security is never a single property but a chain that runs from the Dockerfile a developer writes to the kernel the workload finally executes on.4 Break any link and the rest of the chain is decorative.

This field manual walks that chain. It begins with where the confinement primitives came from and what each actually enforces, turns to the concrete ways attackers defeat them — with real, catalogued vulnerabilities — and then builds back up: how to harden the image, how to harden the runtime, when ordinary namespace isolation is not enough, and which tools make the whole thing auditable. The scope is deliberately the single host and the Docker Engine; orchestration platforms such as Kubernetes are a strict superset of this surface, with their own control plane to defend, but they inherit every primitive discussed here.

02A short history of confinement

Isolation was invented for tidiness, not for enemies

Container security is haunted by its own origin story. Nearly every mechanism Docker relies on was designed to keep well-behaved workloads from stepping on each other — multi-tenancy, resource fairness, clean packaging — and not to withstand a hostile process actively trying to break out. Understanding that inheritance explains why “secure by default” was never the starting point.

The lineage begins in 1979 with chroot, the Unix system call that changes a process's apparent root directory. It was the first primitive that let a program run with a restricted view of the world — but only of the filesystem. A chrooted process still shared the same process table, the same network stack, the same user database, and the same everything else as the rest of the host. The walls went up in exactly one direction, and escaping them was trivial for a privileged process. It was confinement as convenience, not as security.

The kernel learns to compartmentalize

Real isolation arrived incrementally, as the Linux kernel grew a family of namespaces. Each namespace wraps a single global system resource and hands each participating process its own private instance of it. Mount namespaces came first, in 2002 (kernel 2.4.19) — which is why their clone flag is the generic CLONE_NEWNS, named before anyone imagined there would be others.5 UTS and IPC namespaces followed in 2006, PID namespaces in 2008, and network namespaces in 2009. The watershed was the user namespace, developed largely by Eric Biederman and considered complete in kernel 3.8 in 2013: by letting an unprivileged host user map to root inside a namespace, it made genuinely rootless containers possible for the first time.5 Cgroup namespaces arrived in 2016 and time namespaces in 2020, bringing the count to eight.6

In parallel, Google engineers Paul Menage and Rohit Seth began work in 2006 on what they first called “process containers” — a way to limit and account for the CPU, memory, and I/O consumed by a group of processes. To avoid the already-overloaded word container, the feature was renamed control groups in late 2007 and merged into the mainline kernel in version 2.6.24, released in January 2008.7 A major redesign produced cgroup v2, with its single unified hierarchy, in kernel 4.5 in 2016; v2 is now the default on modern distributions and is what container runtimes increasingly target.8

Definition · the two axes of a container

Namespaces answer “what can this process perceive?” — they partition visibility. Cgroups answer “how much can this process use?” — they partition consumption. The two are orthogonal: you can have one without the other. A container is simply a process placed inside a coordinated set of both, plus filters on what it may ask the kernel to do.

From primitives to a product

These features were powerful but forbidding. Wiring namespaces and cgroups together by hand demanded deep systems knowledge, and one missing piece — the setns() system call that lets a process join an existing namespace — did not land until Linux 3.0 in 2011, which is why early tooling could not offer anything resembling docker exec.9 LXC (Linux Containers), first released in 2008, packaged the primitives into something usable, and it was LXC that Docker originally sat on top of.

Docker itself was demonstrated publicly by Solomon Hykes at PyCon in Santa Clara in March 2013, as an internal project of the platform-as-a-service company dotCloud, and released as open source that same month.2 Its genius was not a new isolation mechanism but a new developer experience: a standard image format built on copy-on-write layers, a simple build tool, and a distribution model captured in the slogan “Build, Ship, Run.” One year later, with version 0.9 in March 2014, Docker replaced its dependency on LXC with libcontainer, a native Go component that programmed namespaces and cgroups directly — the ancestor of today's runc.9

The last piece of scaffolding was standardization. In 2015 the Open Container Initiative was founded to define vendor-neutral specifications for image format and runtime behavior; runc became the reference low-level runtime implementing the OCI runtime spec.10 The modern stack is layered accordingly: an orchestrator (Kubernetes, Swarm) drives a high-level runtime (the Docker Engine via containerd), which drives a low-level runtime (runc or crun), which finally speaks to the kernel primitives. Its tenth anniversary was marked, fittingly, by a retrospective in the Communications of the ACM.11

03The primitives that draw the walls

Six mechanisms, each with a blind spot

To reason about Docker security you have to know what each underlying control actually enforces — and, just as importantly, what it does not. Docker composes six kernel mechanisms. Three govern perception and resources (namespaces, cgroups, and the special case of user namespaces); three govern what a process may ask the kernel to do (capabilities, seccomp, and a Linux Security Module). Every one of them has a gap that a later section will show attackers exploiting.

Namespaces — partitioning what a process can see

A namespace gives a process a private instance of some global kernel resource. Modern Linux has eight kinds: mount, PID, network, UTS (hostname), IPC, user, cgroup, and time.5 When Docker starts a container it places the process in fresh instances of most of them, so the process sees its own root filesystem, its own process tree in which it is PID 1, its own network interfaces, and its own hostname. You can watch the machinery directly with ordinary tooling:

shellnamespaces
# list the namespaces on the host and how many processes share each
$ lsns
#        NS TYPE   NPROCS   PID USER  COMMAND
# 4026531835 cgroup    142     1 root  /sbin/init
# 4026531836 pid       142     1 root  /sbin/init
# 4026532191 mnt         1  3821 nginx nginx: master   ← a container's own mount ns

# create an isolated PID + user namespace by hand, mapping yourself to root
$ unshare --user --map-root-user --fork --pid --mount-proc bash
# inside: you appear to be root (uid 0) and the shell is PID 1,
# but on the host you are still your unprivileged user.
# id -u        # -> 0
# ps -e        # -> only processes inside this namespace are visible

The security-critical member of the family is the user namespace. Without it, a process running as root inside a container is running as real uid 0 as far as the kernel is concerned — it differs from host root only by its capability set and the other filters in place. With a user namespace, container uid 0 maps to some high, unprivileged uid on the host, so even a full container breakout lands as a nobody rather than as root.12 This is the single most valuable isolation upgrade Docker offers, and, tellingly, it is off by default.

Blind spot · not everything is namespaced

Several kernel resources have no namespace and therefore leak across the boundary: the kernel keyring, kernel modules, much of /sys and parts of /proc, and — before time namespaces — the system clock. This is why the container runtime and profiles must actively block syscalls like keyctl and init_module: namespaces alone do not contain them.

Control groups — partitioning what a process can consume

Cgroups are the availability half of isolation. By capping a container's memory, CPU shares, process count, and I/O bandwidth, they prevent one workload from starving the host or its neighbors — the classic defense against a runaway fork bomb or a memory-exhaustion denial of service.8 On a host without limits, a single container can consume all available memory and trigger the kernel out-of-memory killer against unrelated processes, taking down the whole node. The mechanism is exposed as a pseudo-filesystem under /sys/fs/cgroup, and Docker sets the knobs for you from --memory, --cpus, --pids-limit, and related flags.

The important caveat: cgroups protect availability only. They say nothing about confidentiality or integrity. A container held to a strict memory limit can still read secrets it should not or make a syscall that escapes the box — those are the province of the next three mechanisms.

Capabilities — slicing up the powers of root

Historically, a Unix process was either root (able to do anything) or not. Linux capabilities break the monolith of root into roughly forty distinct privileges — CAP_NET_BIND_SERVICE to bind low ports, CAP_SYS_ADMIN for a sprawling grab-bag of administrative operations, CAP_SYS_MODULE to load kernel modules, and so on — so that a process running “as root” can hold only the slice it truly needs.13 Docker uses this aggressively: by default it drops most capabilities and retains a curated set of fourteen, including CHOWN, DAC_OVERRIDE, SETUID, SETGID, NET_BIND_SERVICE, NET_RAW, and KILL.14

Fourteen is still generous for most applications. The hardening pattern — and the one from any serious baseline — is to drop everything and add back only what the workload provably needs:

shellcapabilities
# the least-privilege pattern: drop ALL, add back the minimum
$ docker run --cap-drop=ALL --cap-add=NET_BIND_SERVICE myapp
# a plain web service usually needs zero added capabilities:
$ docker run --cap-drop=ALL myapp

# NEVER do this in production -- it grants ALL capabilities,
# disables seccomp, and disables AppArmor in one flag:
$ docker run --privileged myapp   # effectively: no container boundary at all
The one capability to fear

CAP_SYS_ADMIN is so broad it is often called “the new root.” It permits mounting filesystems, manipulating namespaces, and a long list of operations that individually enable container escape. Treat any request to add it — or to run --privileged, which grants it and everything else — as a red flag demanding justification. The same goes for SYS_MODULE, SYS_PTRACE, DAC_READ_SEARCH, and SYS_RAWIO.

Seccomp — a firewall for system calls

Even with a minimal capability set, a container can invoke hundreds of system calls, and any one of them might harbor a kernel bug. Seccomp (secure computing mode) lets a process install a Berkeley Packet Filter that the kernel consults on every syscall, deciding whether to allow it, kill the process, or return an error.15 Docker ships a default seccomp profile — enabled since Docker 1.10 — that works as an allowlist: it permits the large majority of the roughly three hundred-plus syscalls a normal workload needs and denies several dozen dangerous ones, among them kexec_load (loading a new kernel), init_module (loading kernel modules), keyctl (the un-namespaced keyring), ptrace in most cases, and the mount and reboot families.16 A custom profile can tighten this further to exactly the syscalls a given binary uses:

seccomp-profile.jsonjson
{
  "defaultAction": "SCMP_ACT_ERRNO",          // deny everything by default
  "architectures": ["SCMP_ARCH_X86_64"],
  "syscalls": [
    {
      "names": [
        "accept4", "bind", "listen", "read", "write",
        "open", "close", "epoll_wait", "futex", "mmap"
      ],
      "action": "SCMP_ACT_ALLOW"            // only these are permitted
    }
  ]
}
# apply it:  docker run --security-opt seccomp=seccomp-profile.json myapp

Building such a profile means observing the workload — typically running it briefly with strace or an audit-mode profile to enumerate the syscalls it actually makes, then allowlisting precisely those. The payoff is a dramatically smaller kernel attack surface. The pitfall to remember: --privileged disables seccomp entirely, in every version of Docker, even if you pass an explicit profile.16

AppArmor and SELinux — mandatory access control

The final layer is a Linux Security Module enforcing mandatory access control on top of the ordinary discretionary file permissions. Docker applies a default AppArmor profile (docker-default) on Debian- and Ubuntu-family hosts, restricting what files and interfaces a container may touch regardless of Unix ownership.17 On Red Hat-family systems the equivalent is SELinux, which labels container processes (for example container_t) and confines them to their own labeled resources. This layer is not theoretical: SELinux in enforcing mode was one of the few configurations that blunted the CVE-2019-5736 runc escape before hosts were patched, because the malicious container was denied permission to overwrite the runtime binary.18

Notice the pattern across all six: each is genuinely useful and each has a documented gap. That is not a flaw in Docker so much as the nature of shared-kernel isolation. Security comes from stacking the layers so that defeating the boundary requires defeating several independent mechanisms at once — the defense-in-depth argument this manual returns to at the end. First, though, it helps to see the walls fail.

04How the walls fail

The four ways a container stops containing

A container escape is any technique by which code confined to a container gains code execution or data access on the host, or on another container. In practice, escapes arrive through four doors: a bug in the shared kernel, a bug in the runtime, a misconfiguration by the operator, or a compromise of the image supply chain. The first two are rare and severe; the last two are common and, empirically, the source of most real-world container incidents. A defender has to close all four.

Door one — the shared kernel

Because there is only one kernel, a kernel privilege-escalation bug reachable from inside a container is a host compromise. Dirty COW (CVE-2016-5195), a race condition in the kernel's copy-on-write handling, is the standing example: exploited from within a container, it granted root on the host despite otherwise sound namespace isolation.32 The defensive implication is stark and non-negotiable: patch the host kernel promptly, and, for genuinely untrusted workloads, do not rely on the shared kernel as the sole boundary at all — a theme §7 develops.

Door two — the runtime

The runtime that builds the container — runc and, for image builds, BuildKit — is itself attackable, and two disclosures define the genre.

CVE-2019-5736, disclosed in February 2019 by the researchers of Dragon Sector and coordinated by runc maintainer Aleksa Sarai, allowed a malicious container to overwrite the host's runc binary and thereby execute code as root on the host.1920 The trick abused /proc/self/exe, the kernel “magic link” that points at a process's own executable. By replacing a container binary with a reference to /proc/self/exe, an attacker with root inside the container could get a file descriptor to the host's runtime and, in a busy loop from a second process, reopen it for writing the instant runc exited — sidestepping the kernel's ETXTBSY protection — then overwrite it.21 The next invocation of any Docker command ran the attacker's code. It affected Docker before 18.09.2, and the mitigations that blunted it — user-namespace remapping, a read-only runtime binary, and SELinux in enforcing mode — are precisely the layered controls this manual recommends.18

Five years later, in January 2024, Snyk researcher Rory McNamara disclosed Leaky Vessels — four vulnerabilities in core container plumbing, headlined by CVE-2024-21626 in runc (all versions up to 1.1.11).22 Here runc inadvertently leaked an open file descriptor pointing into the host filesystem (a handle to the host's /sys/fs/cgroup). By setting a container's working directory — via the Dockerfile WORKDIR instruction or the runtime process.cwd — to a path like /proc/self/fd/7, an attacker could cause the container process to start inside a host directory and then walk out to the full host root filesystem with the privileges of the engine.23 Crucially, it could be triggered not only by running a malicious image but by building one from a malicious base — a FROM line was enough.24 The companion BuildKit flaws (CVE-2024-23651, -23652, -23653) attacked the build cache and mount handling. The fix was runc 1.1.12 and BuildKit 0.12.5; the lesson is that even your build pipeline is an attack surface.25

Door three — misconfiguration

The most common container compromises require no zero-day at all. They exploit choices the operator made. Three anti-patterns dominate.

Mounting the Docker socket. The Unix socket /var/run/docker.sock is the Docker API, owned by root. Any process that can talk to it can instruct the daemon to launch a new container — privileged, with the host root filesystem bind-mounted in. Handing a container that socket is therefore equivalent to handing it unrestricted root on the host.2 It is the single most exploited escape vector, and mounting it read-only does not help — it only raises the effort slightly.

Anti-pattern · never do this

Mounting the daemon socket into a container is game over:

docker run -v /var/run/docker.sock:/var/run/docker.sock  ← grants root on host
docker -H tcp://0.0.0.0:2375 ...  ← unauthenticated remote root
docker run --privileged -v /:/host alpine  ← host fs mounted, no boundary

The same reasoning condemns exposing the daemon over an unauthenticated TCP port and --privileged, which disables seccomp and AppArmor and grants every capability in a single flag.

Dangerous bind mounts. Passing host paths such as /, /etc, /proc, or a block device into a container hands the workload direct access to host state. If a mount is truly needed, scope it narrowly and mount it read-only with the :ro suffix.

Door four — the image supply chain

The container you run is only as trustworthy as the image it came from, and images are where the largest volume of real vulnerabilities lives — not in exotic escapes but in ordinary, outdated packages. NIST SP 800-190 places image risks — known-vulnerable components, embedded secrets, and configuration defects — at the top of its taxonomy for exactly this reason.4 Three failure modes recur:

  • Vulnerable dependencies. A base image or application layer carrying unpatched CVEs ships those CVEs into production. This is why image scanning in CI is table stakes.
  • Untrusted or typosquatted base images. Public registries host malicious images with names a keystroke away from popular ones; a careless FROM pulls attacker code into your build.
  • Embedded secrets. Because image layers are immutable and independently inspectable, a secret written in one layer and “deleted” in a later one still sits in the image history for anyone to extract.
shelllayer forensics
# a secret removed in a later layer is still recoverable from history
$ docker history --no-trunc myapp:latest
# RUN echo 'AWS_SECRET=AKIA...' > /tmp/creds   ← still present in that layer
# RUN rm /tmp/creds                            ← deletion does NOT erase history

$ docker save myapp:latest | tar -tf - # each layer is a separate, inspectable tarball

Taken together, the four doors argue for a defense that is deliberately redundant: keep the kernel and runtime patched to shut doors one and two, refuse the dangerous configurations that open door three, and treat every image as guilty until scanned to narrow door four. The remaining sections build that defense from the image outward.

05Hardening the image

The smallest thing you can ship is the safest

Defense begins before anything runs, in the Dockerfile. Almost every property that makes a container hard to attack — a minimal attack surface, no baked-in secrets, a non-root process, known-good provenance — is decided at build time. The governing principle is subtraction: every package, shell, and interpreter you leave out of an image is one an attacker cannot turn against you.

Choose a minimal base

Most images inherit far more than they use. A full ubuntu or debian base drags in a package manager, a shell, and dozens of libraries a single Go binary will never call — each a potential CVE and a ready tool for post-exploitation. The remedy is to start as small as the workload allows. Google's distroless images ship an application's language runtime and nothing else — no shell, no package manager, no busybox — which both shrinks the image and removes the interactive footholds attackers rely on.26

BaseApprox. sizeShell / pkg mgrBest forCost

distroless~2–20 MBNoneCompiled apps + runtimes (Java, Go, Python, Node)No shell to debug

The two extremes deserve emphasis. scratch, the empty image, is ideal for a statically linked binary and leaves literally nothing for an attacker to use. Distroless is the pragmatic default for compiled applications: small, shell-free, and available in a nonroot variant. Alpine remains popular where a shell is genuinely needed for debugging or entrypoint scripting, at the price of occasional musl-libc incompatibilities with glibc-built software.

Build in one stage, ship from another

Multi-stage builds let you keep your compilers, headers, and build tooling out of the final image entirely. You compile in a heavy builder stage, then copy only the resulting artifact into a minimal runtime stage — so the shipped image contains the binary and nothing that produced it.27

Dockerfile · multi-stage (Go → distroless)dockerfile
# ---- build stage: full toolchain, discarded after ----
FROM golang:1.23 AS build
WORKDIR /src
COPY go.mod go.sum ./
RUN go mod download
COPY . .
# static binary: no libc dependency, runnable on scratch/distroless
RUN CGO_ENABLED=0 go build -trimpath -ldflags="-s -w" -o /app ./cmd/server

# ---- runtime stage: distroless, non-root, no shell ----
FROM gcr.io/distroless/static-debian12:nonroot
COPY --from=build /app /app
USER nonroot:nonroot
EXPOSE 8080
ENTRYPOINT ["/app"]        # exec form: the app is PID 1, no shell wrapper

Never run as root inside the container

By default a container process runs as root, and if it is compromised, that root — absent a user namespace — is uncomfortably close to host root.2 A USER instruction that drops to a dedicated, unprivileged account is one of the highest-value single lines in a Dockerfile. Do the privileged work (installing packages) first, then switch:

Dockerfile · non-root on alpinedockerfile
FROM node:22-alpine
WORKDIR /app
COPY package*.json ./
RUN npm ci --omit=dev && npm cache clean --force
COPY . .
# create an unprivileged user and hand it ownership
RUN addgroup -S app && adduser -S app -G app && chown -R app:app /app
USER app                    # everything below runs unprivileged
EXPOSE 3000
CMD ["node", "server.js"]   # exec form, no shell as PID 1

Pin provenance, and keep secrets out

Reproducibility is a security property. Pinning a base image by digest (FROM debian:12-slim@sha256:…) guarantees you build on the exact bytes you audited, immune to a mutated tag; pinning package and dependency versions does the same downstream.28 Secrets, meanwhile, must never enter an image layer via ENV or COPY — they persist in history. Use build-time secret mounts, which are never written to a layer:

Dockerfile · secret mount (never persisted)dockerfile
# syntax=docker/dockerfile:1
FROM node:22-alpine
# the .npmrc token is mounted only for this RUN, not baked into any layer
RUN --mount=type=secret,id=npmrc,target=/root/.npmrc npm ci --omit=dev
# build with:  docker build --secret id=npmrc,src=$HOME/.npmrc .

Scan every image, and sign what you ship

Finally, treat scanning as a build gate, not an afterthought. Tools such as Trivy, Grype, and Docker's built-in Scout inspect an image's installed packages against vulnerability databases and can fail a pipeline on high-severity findings, keeping known-bad images out of production.293031

shell · scan as a CI gateshell
# fail the build if any HIGH or CRITICAL vulnerability is present
$ trivy image --severity HIGH,CRITICAL --exit-code 1 myapp:latest

# Docker Scout: a fast overview and full CVE listing
$ docker scout quickview myapp:latest
$ docker scout cves --only-severity critical,high myapp:latest

Scanning tells you an image is free of known flaws; signing tells you it is the image you built. Enabling Docker Content Trust (export DOCKER_CONTENT_TRUST=1) — or signing with a tool like cosign — ensures the daemon pulls and runs only images bearing a trusted signature, closing the door on registry tampering and typosquatted substitutes.32 A hardened image is minimal, pinned, non-root, secret-free, scanned, and signed. What remains is to run it safely.

06Hardening the runtime

Turn every dial toward confinement

A hardened image can still be run carelessly. The runtime flags are where the dials of §3 get set, and the discipline is uniform: grant the minimum, forbid the dangerous, and assume any single control might fail. None of these settings is expensive; their value is in the stack.

Drop privilege at every level

Shed capabilities. Start from nothing and add only what the workload needs — the pattern from §3, now as an operational default rather than an exception. A typical web service needs none; one binding a low port needs exactly one.14

Run unprivileged, and remap the user. Pair the Dockerfile's USER with a runtime --user as belt-and-suspenders, and turn on user-namespace remapping so that in-container root maps to an unprivileged host uid — the mitigation that would have neutralized CVE-2019-5736 outright.12

Forbid new privileges. The no-new-privileges option stops a process from gaining privilege through setuid or setgid binaries, breaking a whole class of escalation chains — even if a setuid sudo sits in the image, the exec returns an error.2

Make the filesystem and resources hostile to abuse

Read-only root filesystem. Running with --read-only and mounting small writable tmpfs areas only where the app truly writes makes most persistence techniques — dropped payloads, web shells, tampered binaries — simply fail, because there is nowhere on disk to write them.

Bound the resources. Memory, CPU, process-count, and file-descriptor limits are the defense against denial of service: a container that cannot allocate unbounded memory cannot invoke the host out-of-memory killer against its neighbors, and a capped --pids-limit defuses fork bombs.4

Keep the profiles on. Retain Docker's default seccomp and AppArmor profiles — and tighten them with a workload-specific seccomp allowlist where you can. The one thing never to do is disable them, whether explicitly with seccomp=unconfined or implicitly with --privileged.16

shell · a hardened docker runshell
docker run -d --name myapp \
  --read-only --tmpfs /tmp --tmpfs /run \        # nowhere to drop a payload
  --cap-drop=ALL --cap-add=NET_BIND_SERVICE \      # least privilege
  --security-opt=no-new-privileges:true \          # block setuid escalation
  --security-opt seccomp=/etc/docker/seccomp.json \  # custom syscall allowlist
  --user 10001:10001 \                             # unprivileged uid
  --memory=256m --cpus=0.5 --pids-limit=128 \       # DoS bounds
  --network=app-net \                            # no default bridge free-for-all
  myapp:latest

The same posture expressed declaratively in Compose — the form most teams actually check into a repository:

compose.yaml · hardened serviceyaml
services:
  api:
    image: myapp:latest
    read_only: true
    tmpfs: [/tmp, /run]
    user: "10001:10001"
    cap_drop: [ALL]
    cap_add: [NET_BIND_SERVICE]
    security_opt:
      - no-new-privileges:true
      - seccomp:/etc/docker/seccomp.json
    mem_limit: 256m
    pids_limit: 128
    cpus: 0.5
    read_only: true
    networks: [app-net]
    restart: on-failure:5
networks:
  app-net: {}

Run the daemon itself without root

Even a perfectly configured container sits under a daemon that has historically run as root — so a compromise of dockerd or the runtime is a compromise of the host. Rootless mode removes that last privileged component by running the daemon and all its containers as an unprivileged user, so that a breakout lands as that user rather than as root.33 For environments where security is paramount it is a strongly recommended configuration; teams that want a daemonless model with the same property often reach for Podman instead.34

shell · rootless Dockershell
$ dockerd-rootless-setuptool.sh install
$ export DOCKER_HOST=unix:///run/user/$(id -u)/docker.sock
# the daemon and every container now run as your user —
# a container breakout is no longer a path to host root.

Set against the escapes of §4, the effect of stacking these controls is qualitative, not incremental. A single misconfiguration — a mounted socket, a stray --privileged — leaves the boundary one kernel bug from collapse. With capabilities dropped, a read-only filesystem, a tight seccomp profile, a mandatory-access-control profile, user-namespace remapping, and a rootless daemon all in force, an attacker must defeat several independent mechanisms in sequence. That is the whole argument.

07When namespaces aren’t enough

Giving the workload its own kernel

Every control so far shares one weakness: they all run on the same kernel as the workload. For most software — your own services, running your own code — that is an acceptable trust boundary, and the layered hardening of §6 is the right answer. But for genuinely untrusted code, the shared kernel is a liability no amount of namespace tuning fixes. NIST SP 800-190 puts the alternative plainly: where the risk warrants it, treat the kernel as belonging to a single tenant and add another layer of separation entirely.4

The question is never how fast the runtime is. It is how much you trust the code — and whether one kernel bug should be allowed to end the conversation.

Three workloads reliably cross that line: multi-tenant platforms running different customers' code on shared nodes; continuous-integration and function-as-a-service systems executing arbitrary user scripts; and, increasingly, systems that run LLM-generated or autonomous-agent code whose behavior no one has reviewed. For these, the industry reaches for sandboxed runtimes that restore a virtual-machine-grade boundary without surrendering the density and speed that made containers attractive. Two designs dominate.

gVisor — an application kernel in user space

Open-sourced by Google in 2018, gVisor interposes a user-space “application kernel” between the container and the host.35 Its component runsc — a drop-in replacement for runc — intercepts the container's system calls and services most of them itself, in a sandbox written in Go for memory and type safety, rather than passing them to the host kernel. The container therefore talks to gVisor's Sentry, not to the host directly, and the host's exposed attack surface shrinks to the small set of syscalls the Sentry itself needs. Crucially, gVisor drops its own privileges before any untrusted code runs.36 The cost is compatibility — a handful of syscalls are unimplemented — and overhead on syscall- and network-intensive workloads, which is why it fits untrusted services better than high-frequency, kernel-bound ones.

Kata Containers and Firecracker — a real VM, made lightweight

Kata Containers takes the other route: each container (or pod) runs inside its own lightweight virtual machine with a dedicated, minimal guest kernel, launched by a hypervisor.37 A kernel exploit that succeeds inside the container escapes only the guest kernel and then confronts the hypervisor boundary — hardware-enforced isolation — rather than landing on the host. Kata remains OCI-compatible, so it slots into existing Docker and Kubernetes tooling. The hypervisor underneath is often Firecracker, AWS's purpose-built micro-VM monitor: it presents a deliberately minimal device model, boots in a fraction of a second, and is the isolation technology beneath AWS Lambda and Fargate, demonstrating that VM-grade boundaries and serverless-grade density are not mutually exclusive.38

RuntimeIsolation mechanismBoundary an escape hitsOverheadUse when
runcNamespaces + cgroups + filters on the shared kernelHost kernel (shared)Near zeroFirst-party, trusted workloads
gVisor (runsc)User-space application kernel intercepting syscallsSentry, then a small host-syscall surfaceLow–moderate; higher on syscall/network-heavy appsUntrusted code, multi-tenant services

The decision rule is about trust, not benchmarks. Reach for a sandboxed runtime precisely when a single kernel vulnerability in your container stack would be unacceptable — when you are running code you did not write and cannot vouch for. For everything else, a well-hardened runc container is both sufficient and cheaper. Knowing which regime you are in is itself a security decision, and one worth making explicitly rather than by default.

08The toolchain

Make the discipline enforceable and observable

Hardening you cannot verify is hope, not security. A mature container practice wraps the controls of the previous sections in tooling that enforces them before deployment and watches for their failure at runtime. The ecosystem falls into three phases — scan, audit, and detect — anchored by a few authoritative standards.

Scan early

Vulnerability scanners inspect an image's contents against known-CVE databases and belong in continuous integration, gating every build. Trivy and Anchore's Grype are the widely adopted open-source options; Grype pairs with Syft to generate a software bill of materials for supply-chain tracking, while Docker Scout is built into the Docker CLI.293130 The goal is not a clean report for its own sake but a hard stop: no image with critical vulnerabilities reaches production.

Audit against a benchmark

Where scanners judge image contents, benchmarks judge configuration. The CIS Docker Benchmark is the consensus reference for securely configuring the daemon, host, images, and containers, and Docker's own docker-bench-security script automates checking a host against it.3940 Above the benchmark sit the two documents this manual has leaned on throughout: NIST SP 800-190 as the risk framework, and the OWASP Docker Security Cheat Sheet as the working practitioner's checklist.42

Detect at runtime

No pre-deployment control catches a zero-day exploited in production, so the final phase watches running containers for the behaviors that signal an escape in progress — an unexpected shell spawned inside a container, a write to a sensitive path, an outbound connection to an unknown host. Falco, a CNCF project, taps kernel events to flag exactly these anomalies; Tetragon and Cilium apply eBPF to the same end.41 Runtime detection is what turns the layered defense from a static posture into something that alerts when a layer is being tested.

The through-line is “security as code”: every control expressed as a scan rule, a benchmark check, a policy, or a detection so that it is applied consistently, automatically, and at scale — never depending on an engineer remembering to add a flag. Tools do not replace the hardening; they make it repeatable and auditable, which is what separates a secure fleet from a lucky one.

09A hardened baseline

Everything, in one place

The controls of §5 through §8 are most useful assembled into a single reference a team can copy and adapt. Below is a complete hardened Compose stack — a minimal, pinned, non-root image from §5 run under the full runtime posture of §6, on an internal network, with a health check and bounded logging. It is deliberately opinionated: the safe defaults are present, and relaxing any one of them should be a conscious, documented choice.

compose.yaml · the reference stackyaml
services:
  api:
    build:
      context: .
      dockerfile: Dockerfile          # multi-stage, distroless, non-root (§5)
    image: myapp:1.4.2@sha256:<digest>  # pinned by digest
    read_only: true                    # no writable rootfs
    tmpfs: [/tmp, /run]               # scratch space only
    user: "10001:10001"                # unprivileged
    cap_drop: [ALL]                    # start from nothing
    cap_add: [NET_BIND_SERVICE]        # add back only what's needed
    security_opt:
      - no-new-privileges:true        # block setuid escalation
      - seccomp:/etc/docker/seccomp.json
      - apparmor:docker-myapp
    mem_limit: 256m
    pids_limit: 128
    cpus: 0.5                         # DoS bounds (§6)
    healthcheck:
      test: ["CMD", "/app", "-healthcheck"]
      interval: 30s
      timeout: 3s
      retries: 3
    logging:
      driver: json-file
      options: { max-size: "10m", max-file: "3" }
    networks: [backend]           # never the default bridge
    restart: on-failure:5

networks:
  backend:
    internal: true                   # no egress to the host network

Every line above maps back to a specific failure mode from §4. Read the mapping the other way — from door to defense — and the baseline becomes a checklist:

Escape door (§4)Controls that close it
Shared kernel — a kernel bug reachable from the containerPatch the host promptly; drop capabilities; keep seccomp on; for untrusted code, a sandboxed runtime (gVisor / Kata)
Runtime bug — a flaw in runc or the daemonPatch runc/BuildKit; user-namespace remapping; rootless daemon; read-only filesystem
Misconfiguration — over-granted privilegeNever mount docker.sock; never --privileged; --cap-drop=ALL; no-new-privileges; non-root user; internal networks
Supply chain — a poisoned image or dependencyMinimal pinned base; CI vulnerability scanning; signed images; no secrets in layers

Making this enforceable is the last step. In an orchestrated environment, the same policy is expressed as admission control — a Kubernetes admission controller or the Pod Security Standards rejecting any workload that requests privilege, runs as root, or mounts a host path — so that the baseline is applied by the platform rather than trusted to each author. The configuration above is the unit; policy-as-code is how it scales to a fleet without eroding.

10The discipline

A boundary you build, not one you’re given

The question that opens this manual — is a container secure? — has no answer, because it is the wrong question. A container is not secure or insecure any more than a process is; it is configured, and its security is exactly the sum of the dials you have turned. The isolation is real but partial, drawn by namespaces, cgroups, capabilities, seccomp, and mandatory access control, every one of which defaults toward convenience and must be deliberately tightened toward safety.

What makes the practice tractable is that each layer is cheap. Dropping capabilities costs one flag. A read-only filesystem costs one flag. A non-root user costs one line. A minimal base image often makes the build faster. None of these is a heavy investment, and none, alone, is sufficient — a determined attacker will find the gap in any single control. Their power is cumulative. The container becomes a boundary worth trusting only when a breakout must defeat a dropped capability set and a read-only filesystem and a seccomp profile and a mandatory-access-control policy and a user namespace and a rootless daemon, in sequence, on a patched kernel. That is the whole discipline: stack cheap, independent layers until the combination is the wall that no one of them could be.

Three habits sustain it. Patch relentlessly — the shared kernel and the runtime are your true perimeter, and the CVEs of §4 were all fixed before most fleets were breached by them. Default to least privilege — grant nothing, add back only what a workload proves it needs, and encode that as policy so it holds at scale. And know your trust level — run first-party code in a well-hardened container, and give genuinely untrusted code a real kernel of its own. Do these, and the abstraction that opened this piece stops being a warning and becomes a working description of something safe.

A container is a process pretending to be a machine. Security is the art of holding it to the pretense — of ensuring that when the mask slips, as one day it will, there is another mask beneath it, and another beneath that, all the way down to a kernel you have kept sound. Build the layers. The boundary is yours to make.
