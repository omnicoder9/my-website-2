§ 01 — Overview

What is curl?

HTTP/HTTPS FTP/FTPS SFTP/SCP WebSocket SMTP LDAP Gopher

curl (stylised in lowercase) is a command-line tool and library for transferring data with URLs. First released by Daniel Stenberg in 1997 as a one-line HTTP fetcher, it has grown into the most widely deployed URL-transfer utility ever written, supporting over 28 network protocols and shipping in nearly every Linux distribution, macOS, and Windows 10/11 by default. [1]

The name stands for Client URL. curl operates without user interaction—it is designed to work perfectly within shell scripts, CI pipelines, and automation tasks. Its non-interactive nature, combined with a rich option set and predictable exit codes, makes it the reference tool for HTTP debugging and API testing. [2]

Devices with libcurl

curl is built on libcurl, a free, thread-safe, IPv6-compatible, feature-rich client-side URL transfer library. libcurl is embedded in thousands of products—PlayStation consoles, Tesla vehicles, iOS, Android, Spotify, and practically every server-side language's HTTP client library. [3]

§ 02 — Program Internals

How curl Works

Understanding curl's internal architecture demystifies its behaviour under failure conditions and helps you write faster, more reliable automation. curl the command-line tool is a thin wrapper around libcurl: it parses CLI flags, configures a libcurl handle, executes the transfer, and reports results. All the real work happens inside the library. [4]

libcurl Architecture

libcurl is structured around two interfaces: the easy interface for simple synchronous transfers and the multi interface for concurrent, non-blocking transfers. The CLI tool uses the easy interface for single requests, falling back to the multi interface when parallel downloads (--parallel) are requested. [4]

└─────────────────────┬──────────────────────────────────┘

│ ┌──────────────┐ ┌──────────────┐ ┌───────────────┐ │
│ │ URL parser │ │ DNS resolver │ │ Protocol mux │ │
│ └──────────────┘ └──────────────┘ └───────────────┘ │
│ ┌──────────────┐ ┌──────────────┐ ┌───────────────┐ │
│ │ HTTP/1.1 │ │ HTTP/2 │ │ HTTP/3 QUIC │ │
│ └──────────────┘ └──────────────┘ └───────────────┘ │
│ ┌──────────────────────────────────────────────────┐ │
│ │ TLS backends: OpenSSL · GnuTLS · wolfSSL · Schannel│ │

└────────────────────────────────────────────────────────┘

libcurl multiplexes I/O using the platform's most efficient event-notification mechanism—epoll on Linux, kqueue on macOS/BSD, and I/O Completion Ports on Windows—allowing thousands of concurrent transfers without threading overhead. [4]

Request Lifecycle

Every curl invocation walks through a deterministic sequence of phases. Understanding this sequence tells you exactly where a failure occurred and which flag addresses it:

1. URL Parse → scheme, host, port, path, query extracted
2. DNS Resolution → getaddrinfo() or c-ares async resolver
3. TCP Connect → three-way handshake (controlled by --connect-timeout)
4. TLS Handshake → cert verify, cipher negotiation (--cacert, --insecure)
5. Protocol Setup → HTTP/1.1 upgrade or HTTP/2 ALPN negotiation
6. Send Request → request line + headers + optional body
7. Receive Response → status line, headers, body (chunked or Content-Length)
8. Redirect Follow → 3xx Location: re-enters at step 1 (--location)
9. Cleanup → connection recycled into pool for reuse

Connection reuse is on by default when making multiple requests in a single curl invocation (e.g., with --next). This avoids repeated TCP and TLS setup, mirroring what browsers and HTTP clients do. [4]

Exit Codes

curl returns a non-zero exit code on failure, making error handling straightforward in scripts:

CodeMeaning
0Success
1Unsupported protocol
2Failed to initialise
3URL malformed
6Could not resolve host (DNS failure)
7Could not connect to host (TCP refused)

bash — using exit codes in scripts
# Fail fast on curl error in a script
curl --fail --silent https://api.example.com/health || {
    echo "Health check failed, exit code: $?"
    exit 1
}

# --fail returns exit code 22 on HTTP 4xx/5xx responses
# Without --fail, curl exits 0 even on HTTP 404
§ 03 — GET Requests

Basic GET Requests

A bare URL argument issues an HTTP GET request—the simplest possible curl invocation. The response body is written to stdout by default. This makes curl trivially composable with other Unix tools. [2]

OptionDescription
-sSilent mode — suppress progress meter and error messages
-SShow errors even when -s is set (used together: -sS)
-LFollow HTTP 3xx redirects automatically
--fail / -fReturn exit code 22 on HTTP 4xx/5xx (default: always exit 0)
-GForce GET when combined with -d (appends data as query string)
--max-redirs NLimit redirects to N (default 30)
bash — GET examples
# Simplest possible request — body to stdout
curl https://api.github.com/repos/curl/curl

# Silent: suppress progress meter, show errors only on failure
curl -sS https://httpbin.org/get

# Follow redirects (HTTP → HTTPS, www → apex, etc.)
curl -L http://github.com

# Fail with non-zero exit on 4xx/5xx — essential in scripts
curl -sfL https://api.example.com/status

# Append query parameters via -G + -d (URL-encodes the data)
curl -G https://api.example.com/search \
  -d "q=curl+tutorial" \
  -d "page=1" \
  -d "limit=20"
# Equivalent to: /search?q=curl+tutorial&page=1&limit=20

# Request a specific URL with a fragment (hash is client-only, stripped)
curl -sS "https://api.example.com/users?sort=created&order=desc"

# Specify HTTP version explicitly
curl --http1.1 https://api.example.com/users
curl --http2   https://api.example.com/users
curl --http3   https://api.example.com/users   # requires QUIC build
ℹ

Always quote URLs containing & or ? characters to prevent shell interpretation. Use single quotes for URLs containing single quotes, or escape with \.

Testing with httpbin.org

httpbin.org is a widely used HTTP testing service that echoes back request details as JSON—ideal for inspecting exactly what curl sends:

bash

§ 04 — Headers

Working with Headers

HTTP headers carry metadata about the request and response: content type, authentication tokens, caching directives, encoding, and more. curl provides distinct flags for retrieving server headers without a body, including them with the body, and injecting custom headers into requests. [5]

OptionDescription
-I / --headSend a HEAD request — retrieve response headers only, no body
-i / --includeInclude response headers in the output, followed by body
-H / --headerPass a custom request header (repeatable for multiple headers)
-D FILEWrite response headers to FILE (instead of stdout)
-e URLSet the Referer header
-ASet the User-Agent header (shorthand for -H "User-Agent: ...")

Inspecting Response Headers

bash — -I (HEAD request)
# HEAD request: headers only, no body transferred
curl -I https://www.github.com

HTTP/2 301
location: https://github.com/
content-type: text/html
server: GitHub.com
x-github-request-id: A1B2:3C4D:5E6F:7890

# Useful for: checking redirects, content-type, cache headers,
# last-modified, and server software without downloading body

# Check if a resource supports byte-range requests (download resumption)
curl -I https://releases.ubuntu.com/24.04/ubuntu-24.04.iso | grep -i accept-ranges
accept-ranges: bytes
bash — -i (headers + body)
# -i includes status line and headers before the body
curl -i https://httpbin.org/get

HTTP/2 200
date: Mon, 01 Jan 2024 00:00:00 GMT
content-type: application/json
access-control-allow-origin: *

{
  "headers": { ... },
  "url": "https://httpbin.org/get"
}

# Write headers to a separate file, body to stdout
curl -D headers.txt https://api.example.com/users

Sending Custom Request Headers

bash — -H
# Set Accept header to request JSON
curl -sS -H "Accept: application/json" https://api.example.com/users

# Multiple headers — repeat -H for each
curl -sS \
  -H "Accept: application/json" \
  -H "Accept-Language: en-US,en;q=0.9" \
  -H "X-Request-ID: $(uuidgen)" \
  https://api.example.com/users

# Override the Host header (virtual hosting)
curl -H "Host: staging.example.com" https://10.0.1.50/api/status

# Remove a default header curl adds automatically
# (use "HeaderName:" with empty value to delete it)
curl -H "User-Agent:" https://httpbin.org/headers    # strips User-Agent
curl -H "Accept:"     https://httpbin.org/headers    # strips Accept

# Conditional GET — only download if resource changed
curl -H "If-None-Match: \"etag-value-here\"" https://api.example.com/data
curl -H "If-Modified-Since: Tue, 01 Jan 2024 00:00:00 GMT" https://api.example.com/data
⚠

When you override the Content-Type header with -H while also using -F (multipart), curl loses the auto-generated boundary parameter. Only override Content-Type on -F uploads if you know the exact boundary string.

§ 05 — Downloading Files

Downloading Files

By default, curl writes the response body to stdout. For file downloads, you redirect output to disk using -o (explicit filename) or -O (use the remote filename). Resumption, parallel downloads, and integrity checking round out curl's download toolkit. [2]

OptionDescription
-o FILEWrite output to FILE instead of stdout
-OWrite output to a file named after the remote URL's filename
-C -Resume a previous file transfer (auto-detect offset with -)
-LFollow redirects — essential when download links use short URLs
--create-dirsCreate local directory hierarchy for -o path if it doesn't exist
--parallel / -ZFetch multiple URLs in parallel (curl 7.66+)
--parallel-max NMax simultaneous parallel transfers (default: 50)
-#Progress bar instead of progress meter
bash — -o and -O
# -o: specify exact output filename
curl -L -o ubuntu.iso https://releases.ubuntu.com/24.04/ubuntu-24.04-desktop-amd64.iso

# -O: use the filename from the URL path
curl -L -O https://releases.ubuntu.com/24.04/ubuntu-24.04-desktop-amd64.iso
# → saves as: ubuntu-24.04-desktop-amd64.iso

# Show a visual progress bar
curl -L -O -# https://example.com/largefile.tar.gz

# Suppress all output (silent download)
curl -sSL -o /dev/null https://api.example.com/warmup
bash — Resume downloads with -C
# Resume a partial download — curl asks the server for byte offset
curl -C - -L -O https://example.com/largefile.tar.gz

# Resume from a specific byte offset
curl -C 1048576 -o file.tar.gz https://example.com/largefile.tar.gz

# Resilient download loop (retry on failure)
while ! curl -C - -L -O --retry 5 --retry-delay 3 https://example.com/file.iso; do
    echo "Retrying..."; sleep 5
done
bash — Parallel downloads
# Download multiple files in parallel (curl 7.66+)
curl --parallel --parallel-max 4 -L \
  -O https://example.com/file1.tar.gz \
  -O https://example.com/file2.tar.gz \
  -O https://example.com/file3.tar.gz

# Download into subdirectory, creating dirs as needed
curl -L -o downloads/2024/report.pdf \
  --create-dirs \
  https://example.com/reports/2024/report.pdf
bash — Integrity verification
# Download and verify SHA-256 checksum
curl -L -O https://example.com/archive.tar.gz
curl -L -O https://example.com/archive.tar.gz.sha256
sha256sum --check archive.tar.gz.sha256

# Time-limited download — abort if total time exceeds 60s
curl --max-time 60 -L -O https://example.com/file.zip
§ 06 — Sending Data

POST, PUT & PATCH Requests

When you pass data with -d, curl automatically switches to POST and sets Content-Type: application/x-www-form-urlencoded. Use -X to explicitly override the HTTP method for PUT, PATCH, and DELETE operations, or for cases where you need POST without a body. [5]

OptionDescription
-X METHODSet the HTTP method explicitly (GET, POST, PUT, PATCH, DELETE, …)
-d DATASend DATA as the request body; implies POST
--data-urlencode STRURL-encode the data before sending (handles special characters)
-d @FILERead request body from FILE (@- reads from stdin)
--data-raw DATAPOST data without processing @ as filename
--data-binary @FILEPost binary file contents without newline conversion
bash — POST (form-encoded)
# POST with form-encoded data (default Content-Type)
curl -d "username=alice&password=s3cret" https://example.com/login

# Multiple -d flags — curl joins them with &
curl -d "name=Alice" -d "email=alice@example.com" -d "role=admin" \
  https://api.example.com/users

# URL-encode special characters automatically
curl --data-urlencode "message=Hello World! & Goodbye?" \
  https://api.example.com/messages

# POST body from a file
curl -d @payload.txt -H "Content-Type: application/xml" \
  https://api.example.com/import

# POST body from stdin (pipe)
echo '{"event":"deploy"}' | curl -sS -d @- \
  -H "Content-Type: application/json" \
  https://hooks.example.com/webhook
bash — PUT, PATCH, DELETE
# PUT — replace a resource entirely
curl -X PUT https://api.example.com/users/42 \
  -H "Content-Type: application/json" \
  -d '{"name":"Alice","email":"alice@example.com","role":"admin"}'

# PATCH — partial update (RFC 5789)
curl -X PATCH https://api.example.com/users/42 \
  -H "Content-Type: application/json" \
  -d '{"email":"newalice@example.com"}'

# DELETE — no body required
curl -X DELETE https://api.example.com/users/42

# DELETE with confirmation body
curl -X DELETE https://api.example.com/buckets/my-bucket \
  -H "Content-Type: application/json" \
  -d '{"confirm":true}'

# OPTIONS — check allowed methods for a resource
curl -X OPTIONS -i https://api.example.com/users
HTTP/2 204
Allow: GET, HEAD, POST, OPTIONS
⚠

-X sets the method string only — it does not otherwise change curl's behaviour. Using -X GET with -d will still send a body, which is technically valid HTTP but widely unsupported. To send query-string data on a GET, use -G + -d instead.

§ 07 — JSON APIs

Working with JSON APIs

JSON is the lingua franca of modern REST APIs. curl handles JSON by letting you set the right headers — there is no built-in JSON parsing, but that is by design: curl is a transport tool. Pair it with jq for parsing (see §12). [5]

Setting Content-Type: application/json is the caller's responsibility. curl will never infer this from the body shape — it is just bytes.

bash — JSON POST
# Standard JSON POST — two required headers
curl -sS -X POST https://api.example.com/users \
  -H "Content-Type: application/json" \
  -H "Accept: application/json" \
  -d '{"name":"Alice","email":"alice@example.com","role":"admin"}'

# Multi-line JSON body — use $'...' quoting for readability
curl -sS -X POST https://api.example.com/articles \
  -H "Content-Type: application/json" \
  -d $'{
    "title": "curl Reference",
    "tags": ["networking", "cli", "http"],
    "published": true,
    "metadata": {"author": "Alice", "version": 2}
  }'

# POST from a pre-built JSON file
curl -sS -X POST https://api.example.com/batch \
  -H "Content-Type: application/json" \
  --data-binary @batch-payload.json
bash — Building JSON dynamically with shell variables
# Safe JSON construction with jq --arg (handles escaping) NAME="O'Brien & Associates" EMAIL="obrien@example.com" curl -sS -X POST https://api.example.com/contacts \ -H "Content-Type: application/json" \ -d "$(jq -n --arg n "$NAME" --arg e "$EMAIL" \ '{name: $n, email: $e}')" # Unsafe: embedding vars directly (breaks on quotes/special chars) # curl -d "{\"name\":\"$NAME\"}" ... ← DO NOT DO THIS
bash — JSON PATCH and PUT
# JSON Merge Patch (RFC 7396) — partial update
curl -sS -X PATCH https://api.example.com/users/42 \
  -H "Content-Type: application/merge-patch+json" \
  -d '{"email":"updated@example.com"}'

# JSON Patch (RFC 6902) — operation-based patch
curl -sS -X PATCH https://api.example.com/users/42 \
  -H "Content-Type: application/json-patch+json" \
  -d '[
    {"op":"replace","path":"/email","value":"new@example.com"},
    {"op":"add","path":"/tags/-","value":"premium"}
  ]'
ℹ

curl 7.82.0+ adds --json DATA, a shorthand that sets both Content-Type: application/json and Accept: application/json while sending DATA as the body — combining three common flags into one.

bash — curl --json (7.82+)
# --json is shorthand for -H Content-Type:application/json
#                        + -H Accept:application/json
#                        + -d DATA
#                        + implicit POST
curl --json '{"name":"Alice"}' https://api.example.com/users

# Read JSON from file
curl --json @payload.json https://api.example.com/users
§ 08 — Authentication

Authentication

curl supports every mainstream HTTP authentication scheme: HTTP Basic, Digest, NTLM, Negotiate (Kerberos), AWS SigV4, and the ubiquitous OAuth 2.0 Bearer token pattern. The right mechanism depends on the API you are targeting. [6] [7]

OptionDescription
-u USER:PASSHTTP Basic authentication (sends Base64-encoded credentials)
-u USERPrompt for password interactively (does not appear in shell history)
--digestUse HTTP Digest authentication (challenge–response, MD5)
--ntlmUse NTLM authentication (Windows/Active Directory environments)
--negotiateUse Negotiate / SPNEGO (Kerberos) authentication
--oauth2-bearer TOKENSet the Bearer token for OAuth 2.0
--aws-sigv4 "aws:amz:REGION:SERVICE"Sign request with AWS Signature Version 4
--anyauthLet curl pick the most secure method the server advertises
--netrcRead credentials from ~/.netrc

HTTP Basic Authentication

bash — Basic auth
# Inline credentials — visible in ps and shell history (avoid in production) curl -u alice:supersecret https://api.example.com/admin # Prompt for password (safer — not stored in history) curl -u alice https://api.example.com/admin # Environment variable pattern (safest for scripts) curl -u "${API_USER}:${API_PASS}" https://api.example.com/admin # .netrc file: avoids credentials on command line entirely # ~/.netrc: # machine api.example.com # login alice # password supersecret curl --netrc https://api.example.com/admin # Basic auth is just Base64(user:pass) in the Authorization header: # Authorization: Basic YWxpY2U6c3VwZXJzZWNyZXQ= # Never use over plain HTTP — always HTTPS

Bearer Token (OAuth 2.0)

Bearer tokens are the dominant authentication mechanism for modern REST APIs. The token is presented in the Authorization header with the Bearer scheme, as specified in RFC 6750. [7]

bash — Bearer tokens
# Standard Authorization: Bearer header
TOKEN="eyJhbGciOiJSUzI1NiIsInR5cCI6IkpXVCJ9..."

curl -sS -H "Authorization: Bearer $TOKEN" \
  https://api.example.com/profile

# Equivalent shorthand (curl 7.88+)
curl -sS --oauth2-bearer "$TOKEN" \
  https://api.example.com/profile

# Full OAuth 2.0 flow: obtain token first, then use it
TOKEN=$(curl -sS -X POST https://auth.example.com/token \
  -H "Content-Type: application/x-www-form-urlencoded" \
  -d "grant_type=client_credentials" \
  -d "client_id=${CLIENT_ID}" \
  -d "client_secret=${CLIENT_SECRET}" \
  | jq -r '.access_token')

curl -sS -H "Authorization: Bearer $TOKEN" \
  https://api.example.com/data
bash — API key patterns
# API key in header (GitHub-style)
curl -sS -H "Authorization: token ghp_xxxxxxxxxxxx" \
  https://api.github.com/user/repos

# API key in header (custom scheme)
curl -sS -H "X-API-Key: ${STRIPE_API_KEY}" \
  https://api.stripe.com/v1/customers

# API key as query parameter (avoid — keys appear in server logs)
curl -sS "https://api.example.com/data?api_key=${KEY}"

# AWS Signature v4 (for AWS API calls without SDK)
curl --aws-sigv4 "aws:amz:us-east-1:execute-api" \
  -u "${AWS_ACCESS_KEY_ID}:${AWS_SECRET_ACCESS_KEY}" \
  https://api-id.execute-api.us-east-1.amazonaws.com/prod/resource
⚠

Never hardcode credentials in scripts committed to version control. Use environment variables ($TOKEN), a secrets manager, or ~/.netrc. Consider using a .curlrc file with oauth2-bearer = TOKEN for persistent configuration that stays outside source control.

§ 09 — File Uploads

File Uploads

curl supports two principal upload mechanisms: -F for multipart/form-data (HTML form uploads, the norm for file APIs), and -T for raw PUT-based uploads used in FTP, WebDAV, and HTTP streaming upload scenarios. [8]

OptionDescription
-F NAME=@FILEUpload FILE as a multipart form field named NAME
-F NAME=VALUEAdd a text form field (no @)
-F NAME=@FILE;type=MIMESpecify MIME type for the upload
-F NAME=@FILE;filename=NAMEOverride the filename sent to the server
-T FILEUpload FILE using HTTP PUT or FTP STOR
--upload-file FILELong form of -T

Multipart Form Upload (-F)

Multipart uploads encode the file and metadata as a MIME multipart document, matching what a browser sends when submitting <input type="file">. curl sets Content-Type: multipart/form-data; boundary=… automatically. [8]

bash — -F multipart uploads
# Single file upload
curl -sS -F "file=@/path/to/photo.jpg" \
  https://api.example.com/upload

# File + additional text fields
curl -sS \
  -F "avatar=@profile.jpg;type=image/jpeg" \
  -F "username=alice" \
  -F "caption=My profile photo" \
  https://api.example.com/profile/avatar

# Override the filename sent to the server
curl -sS -F "file=@/tmp/tmp_12345.jpg;filename=invoice.jpg" \
  https://api.example.com/documents

# Multiple files in one request
curl -sS \
  -F "files[]=@image1.jpg" \
  -F "files[]=@image2.jpg" \
  -F "files[]=@image3.jpg" \
  https://api.example.com/gallery/batch

# Read file content from stdin
cat report.pdf | curl -sS -F "file=@-;filename=report.pdf;type=application/pdf" \
  https://api.example.com/reports

Raw PUT Upload (-T)

bash — -T PUT upload
# HTTP PUT — sends file as raw body (no multipart wrapper)
curl -sS -T localfile.bin \
  -H "Content-Type: application/octet-stream" \
  https://storage.example.com/bucket/object-key

# FTP upload
curl -T report.csv ftp://ftp.example.com/uploads/report.csv \
  -u "${FTP_USER}:${FTP_PASS}"

# SFTP upload (requires SSH key or password)
curl -T backup.tar.gz sftp://user@server.example.com/home/user/backups/ \
  --key ~/.ssh/id_ed25519

# S3 pre-signed URL upload (no AWS credentials needed in curl)
curl -sS -T photo.jpg "${PRESIGNED_PUT_URL}" \
  -H "Content-Type: image/jpeg"
ℹ

For large uploads, add --expect100-continue to send an Expect: 100-continue header. The server acknowledges the headers before the body is sent — avoiding wasting bandwidth uploading to a URL that returns 401 or 413.

§ 10 — Cookies

Cookies

Cookies are key–value pairs stored by the client and sent back to the server on subsequent requests to the same domain. They underpin session management across the web. curl has a full cookie engine that reads, writes, and stores cookies in the Netscape cookie file format. [9]

OptionDescription
-b COOKIESend a cookie as a string ("name=value") or read from a file
-c FILEWrite cookies received from server to FILE (cookie jar)
-b FILERead cookies from FILE and send them (combined with -c FILE for persistence)
--cookie-jar FILELong form of -c
-j / --junk-session-cookiesDiscard session cookies (non-persistent) when loading from jar
bash — Sending cookies
# Send a single cookie string
curl -sS -b "session=abc123" https://example.com/dashboard

# Send multiple cookies (semicolon-separated)
curl -sS -b "session=abc123; theme=dark; locale=en-US" \
  https://example.com/dashboard

# Load cookies from a Netscape format cookie jar file
curl -sS -b cookies.txt https://example.com/profile
bash — Full session with cookie persistence
# Step 1: Login — server sets Set-Cookie, we save to jar
curl -sS -X POST https://example.com/login \
  -H "Content-Type: application/json" \
  -d '{"username":"alice","password":"secret"}' \
  -c cookies.txt   # save cookies from response

# Step 2: Authenticated request — send saved cookies back
curl -sS https://example.com/api/profile \
  -b cookies.txt   # read and send cookies

# Step 3: Cookie-maintaining session — both -b and -c together
# (reads existing cookies AND saves new ones from each response)
curl -sS https://example.com/api/data \
  -b cookies.txt -c cookies.txt

# Inspect the cookie jar (Netscape format)
cat cookies.txt
# Netscape HTTP Cookie File
# https://curl.se/docs/http-cookies.html
example.com  FALSE  /  TRUE  1735689600  session  abc123xyz
example.com  FALSE  /  FALSE  0          theme    dark

# Full browser-like session with User-Agent and Referer curl -sS -L \ -A "Mozilla/5.0 (X11; Linux x86_64) AppleWebKit/537.36" \ -e "https://example.com/" \ -b cookies.txt -c cookies.txt \ https://example.com/members/dashboard # Discard session cookies when reloading from jar (-j) # Simulates browser restart (session cookies should not persist) curl -sS -j -b cookies.txt -c cookies.txt \ https://example.com/profile
ℹ

The cookie jar file is only written when curl exits. If you use the same jar for both -b and -c in a long-running script, consider using a fresh jar name per session and rotating it to avoid concurrency issues.

§ 11 — Troubleshooting

Troubleshooting

When a curl request misbehaves, three tools diagnose almost any problem: -v shows the full TLS and HTTP conversation; -w extracts precise timing metrics; and the timeout flags give you control over when curl gives up and why. [2]

Verbose Output (-v)

The -v flag prints every step of the request lifecycle to stderr, prefixed with directional markers: > for sent data, < for received data, and * for informational messages (TLS, redirects, connection pool).

# Send verbose output to a log file for later analysis
curl -v https://api.example.com 2>curl-debug.log
bash — --trace and --trace-ascii
# --trace: full hex+ASCII dump of all data transferred
curl --trace dump.txt https://api.example.com/users

# --trace-ascii: ASCII-only trace (more readable for text protocols)
curl --trace-ascii trace.txt https://api.example.com/users

# --trace-time: add timestamps to trace output
curl --trace-ascii - --trace-time https://api.example.com/users 2>&1 | head -40

Write-Out Metrics (-w)

The -w flag formats a custom output string after the transfer using named variables. It is the fastest way to collect HTTP status codes, timing breakdowns, and byte counts from curl in scripts. [2]

bash — -w format variables
# Print HTTP status code only (suppress body with -o /dev/null)
curl -o /dev/null -sS -w "%{http_code}\n" https://api.example.com/health
200

# Full timing breakdown
curl -o /dev/null -sS -w "\
  DNS lookup:    %{time_namelookup}s\n\
  TCP connect:   %{time_connect}s\n\
  TLS handshake: %{time_appconnect}s\n\
  TTFB:          %{time_starttransfer}s\n\
  Total:         %{time_total}s\n\
  Downloaded:    %{size_download} bytes\n" \
  https://api.example.com/users

  DNS lookup:    0.012340s
  TCP connect:   0.034567s
  TLS handshake: 0.087654s
  TTFB:          0.123456s
  Total:         0.145678s
  Downloaded:    2048 bytes

# Check HTTP code in a script
HTTP_CODE=$(curl -o /dev/null -sS -w "%{http_code}" https://api.example.com/status)
if [ "$HTTP_CODE" != "200" ]; then
  echo "Unexpected status: $HTTP_CODE" >&2
  exit 1
fi
Key -w variables
──────────────────────────────────────────────────────────
%{http_code} HTTP response status code (200, 404, 500…)
%{time_namelookup} Seconds for DNS resolution
%{time_connect} Seconds until TCP connection established
%{time_appconnect} Seconds until TLS/SSL handshake complete
%{time_pretransfer} Seconds until ready to send (post-headers)
%{time_starttransfer}Seconds until first byte received (TTFB)
%{time_total} Total seconds for the entire transfer
%{size_download} Bytes downloaded
%{size_upload} Bytes uploaded
%{speed_download} Average download speed (bytes/second)
%{remote_ip} IP address of the server
%{remote_port} Port number of the server
%{url_effective} Final URL after redirects
%{ssl_verify_result} SSL peer certificate verification result (0 = OK)
%{num_redirects} Number of redirects followed

Timeouts

Two distinct timeout flags address different failure modes in the request lifecycle:

OptionDescription
--connect-timeout SECSMax seconds to wait for TCP connection (and TLS). Fires exit code 28.
--max-time SECSMax seconds for the entire transfer including body. Fires exit code 28.
--speed-limit BYTESAbort if transfer speed drops below BYTES/s for more than --speed-time seconds
--speed-time SECSDuration below --speed-limit before aborting (default: 30s)
--retry NRetry failed transfers up to N times
--retry-delay SECSWait SECS between retries
--retry-max-time SECSTotal time limit across all retries
--retry-connrefusedTreat connection refused as a retryable error
bash — Timeout and retry patterns
# Tight timeout for health checks
curl --connect-timeout 2 --max-time 5 \
  -o /dev/null -sS -w "%{http_code}" \
  https://api.example.com/health

# Retry with exponential backoff (via --retry-delay)
curl --retry 5 --retry-delay 2 --retry-max-time 60 \
  --retry-connrefused \
  --connect-timeout 5 --max-time 30 \
  -sSf https://api.example.com/data

# Abort a stalled download (under 1KB/s for 10s)
curl --speed-limit 1000 --speed-time 10 \
  -L -O https://example.com/largefile.iso

# Wait for a service to come online (startup probe pattern)
until curl --connect-timeout 1 --max-time 2 \
       -sSf http://localhost:8080/health; do
  echo "Waiting for service..."; sleep 2
done
ℹ

--connect-timeout covers only the TCP + TLS phases. Once the connection is established, only --max-time limits the transfer. For APIs with slow cold starts, set --connect-timeout low and --max-time higher.

§ 12 — Advanced API Testing

curl + jq

jq is a lightweight, portable JSON processor that operates as a stream transformer — it reads a JSON value, applies a filter expression, and writes the result. Piped together, curl -s … | jq '…' is the standard UNIX idiom for API exploration and scripting. [10]

curl fetches. jq shapes. Together they form a composable JSON API client from the command line.

jq Filter Essentials

bash — Basic jq filters
# . (identity) — pretty-print the entire JSON response
curl -s https://api.github.com/repos/curl/curl | jq '.'

# .key — extract a single field
curl -s https://api.github.com/repos/curl/curl | jq '.stargazers_count'
35621

# .key.nested — nested field access
curl -s https://api.github.com/repos/curl/curl | jq '.owner.login'
"curl"

# -r (--raw-output) — strip surrounding quotes from strings
curl -s https://api.github.com/repos/curl/curl | jq -r '.full_name'
curl/curl

# .[] — iterate over array elements
curl -s https://api.github.com/repos/curl/curl/tags | jq -r '.[] | .name'
curl-8_7_1
curl-8_7_0
curl-8_6_0
bash — Selecting, mapping, and filtering
# map() — transform each element of an array
curl -s https://api.example.com/users \
  | jq 'map({id, name, email})'

# select() — filter elements matching a condition
curl -s https://api.example.com/users \
  | jq '[.[] | select(.active == true)]'

# Chained: filter active admins, extract emails
curl -s https://api.example.com/users \
  | jq -r '[.[] | select(.active and .role == "admin")] | .[].email'

# length — count elements
curl -s https://api.example.com/users | jq 'length'
247

# sort_by + reverse — sort array by field, descending
curl -s https://api.example.com/posts \
  | jq 'sort_by(.created_at) | reverse | .[0:5]'   # top 5 newest

# group_by — aggregate users by role
curl -s https://api.example.com/users \
  | jq 'group_by(.role) | map({role: .[0].role, count: length})'
bash — Constructing output objects
# Build a new object from response fields
curl -s https://api.github.com/repos/curl/curl \
  | jq '{
      name: .full_name,
      stars: .stargazers_count,
      language: .language,
      updated: .updated_at
    }'

# @csv — output as CSV
curl -s https://api.example.com/products \
  | jq -r '.[] | [.id, .name, .price] | @csv'
1,"Widget Pro",29.99
2,"Widget Lite",9.99

# @tsv — tab-separated (pipe into column -t for alignment)
curl -s https://api.example.com/users \
  | jq -r '.[] | [.id, .name, .email, .role] | @tsv' \
  | column -t
bash — Scripting patterns with curl + jq
# Paginate an API and collect all results
PAGE=1
ALL_USERS="[]"
while true; do
  RESP=$(curl -s "https://api.example.com/users?page=${PAGE}&limit=100" \
    -H "Authorization: Bearer $TOKEN")
  ITEMS=$(echo "$RESP" | jq '.data')
  COUNT=$(echo "$ITEMS" | jq 'length')
  [ "$COUNT" -eq 0 ] && break
  ALL_USERS=$(echo "$ALL_USERS $ITEMS" | jq -s 'add')
  PAGE=$((PAGE + 1))
done
echo "$ALL_USERS" | jq 'length'

# POST using jq-constructed JSON and capture response fields
RESPONSE=$(curl -sS -X POST https://api.example.com/tasks \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer $TOKEN" \
  -d "$(jq -n --arg t "Deploy v2.0" --arg p "high" \
        '{title:$t, priority:$p, status:"pending"}')")

TASK_ID=$(echo "$RESPONSE" | jq -r '.id')
echo "Created task: $TASK_ID"

# Health-check all microservices, report unhealthy ones
SERVICES=("auth:8001" "users:8002" "orders:8003" "payments:8004")
for SVC in "${SERVICES[@]}"; do
  NAME=${SVC%%:*}
  PORT=${SVC##*:}
  STATUS=$(curl -s --connect-timeout 1 --max-time 3 \
    -o /dev/null -w "%{http_code}" \
    http://localhost:${PORT}/health)
  [ "$STATUS" != "200" ] && echo "UNHEALTHY: $NAME (HTTP $STATUS)"
done
bash — API response comparison and diffing
# Diff two API responses (e.g. staging vs production)
diff \
  <(curl -s https://staging.api.example.com/config  | jq -S '.') \
  <(curl -s https://production.api.example.com/config | jq -S '.')

# Monitor an endpoint for changes (poll every 30s)
PREV=""
while true; do
  CURR=$(curl -s https://api.example.com/status | jq -Sc '.data')
  if [ "$CURR" != "$PREV" ]; then
    echo "[$(date)] Status changed: $CURR"
    PREV="$CURR"
  fi
  sleep 30
done
