§ 01

The Principle of Least Privilege

Origins and Formal Roots

The principle of least privilege (PoLP) — the idea that every program, process, and user should operate with the minimum set of permissions necessary to perform its designated function and no more — is one of the most consequential concepts in the history of computer security. Its rigorous articulation came in 1975 when Jerome H. Saltzer and Michael D. Schroeder, working on the Multics system at MIT, published their landmark survey "The Protection of Information in Computer Systems."[1] That paper enumerated eight design principles for protection systems that remain as applicable today as they were half a century ago.

The full set — economy of mechanism, fail-safe defaults, complete mediation, open design, separation of privilege, least privilege, least common mechanism, and psychological acceptability — constitutes a design philosophy, not a checklist.[1] Saltzer and Schroeder observed that privilege separation is best achieved by granting authority in units no larger than the task requires, and revoked as soon as the task completes. The Multics experience had made the costs of ignoring this obvious: complex interaction patterns between subsystems with excessive authority led to subtle, exploitable failures.

Every program and every privileged user of the system should operate using the least amount of privilege necessary to complete the job.

— Jerome H. Saltzer & Michael D. Schroeder, 1975[1]

The companion intellectual framework came two years earlier, from David Bell and Leonard LaPadula, who constructed a mathematical model of mandatory access control for the US Department of Defense.[2] The Bell-LaPadula model assigned security levels to both subjects (processes, users) and objects (files, memory, devices) and defined two core security properties: the simple security property (no read up — a subject at security level L cannot read an object at level L' > L) and the *-property (no write down — a subject at level L cannot write to an object at level L' < L). These properties together prevent Trojan horse attacks: a high-clearance process cannot leak data to a low-clearance object on an attacker's behalf.

The Bell-LaPadula model captured information flow constraints; Saltzer and Schroeder captured operational privilege constraints. Together, they established the two axes along which modern privilege separation mechanisms operate.

Threat Model Implications

The threat model implied by PoLP is both simple and perpetually relevant: any piece of software may contain defects. Buffer overflows, integer underflows, logic errors, race conditions, injection vulnerabilities, and use-after-free bugs are not theoretical concerns — they are empirical regularities of software development at scale. If a flawed component operates with elevated privileges, an attacker who exploits that flaw inherits those privileges. The blast radius of a single exploited vulnerability is exactly the privilege footprint of its host process.

~400 Linux syscalls exposed to every process by default
8,700 Lines of C in the formally verified seL4 kernel
50+ Years since Saltzer & Schroeder's eight principles
40 POSIX capability bits dividing root's monolithic power

This motivates the defense-in-depth posture that underlies modern security engineering. Rather than relying on a single barrier to privilege escalation, mature systems layer independent mechanisms: privilege dropping after initialization, file capability bits, mandatory access control profiles, syscall filters, capability descriptors, and namespace isolation. Each layer degrades independently; their combination requires an attacker to defeat multiple orthogonal controls.

The mechanisms surveyed in this article represent the principal implementations of the PoLP in practice, tracing from the raw Unix permission model to formally verified microkernels.

§ 02

Unix Privilege Separation: setuid, sudo, and Daemon Design

Unix, designed by Ken Thompson and Dennis Ritchie at Bell Labs beginning in 1969,[3] introduced a deceptively simple privilege model. Every process carries a real user ID (UID) and real group ID (GID); filesystem objects carry permission bits controlling read, write, and execute access for owner, group, and world. A single privileged account — root, UID 0 — can bypass virtually all permission checks. This superuser model was pragmatic for a research system; it became a security liability as Unix proliferated into production environments across industries and eventually into the global internet.

The setuid Mechanism

The setuid bit, set on an executable via chmod u+s, instructs the kernel to execute that binary with the file owner's effective UID rather than the invoking user's UID. When /usr/bin/passwd, owned by root, is executed by an ordinary user, the resulting process runs with effective UID 0, enabling it to write to /etc/shadow — a file inaccessible to unprivileged users. This solves a genuine delegation problem: users must be able to change their passwords without receiving direct write access to the credential store.

The Linux kernel maintains three distinct UID values per process:[4]

  • Real UID (RUID): The identity of the user who invoked the process
  • Effective UID (EUID): The identity used for permission checks at each system call
  • Saved set-user-ID (SUID): A saved copy of the EUID prior to privilege reduction, enabling restoration

The setuid(2) system call exhibits context-sensitive behavior. An unprivileged process (EUID ≠ 0) calling setuid(x) succeeds only if x equals the RUID or the saved SUID; if successful, all three IDs are set to x — the privilege drop is permanent. A privileged process (EUID 0) calling setuid(x) sets all three IDs unconditionally, irrevocably dropping root.[4] The seteuid(2) call modifies only the EUID, permitting temporary privilege reduction with subsequent restoration via seteuid(saved_suid).

This asymmetry enables the canonical privilege-drop pattern: a setuid-root program acquires root at exec time, performs whatever privileged operation is required (binding a low port, accessing a protected file), then calls setuid(getuid()) to permanently relinquish root for the remainder of its execution. Any subsequent exploit of the process yields only the invoking user's privileges.

Vulnerabilities of setuid Programs

The setuid mechanism has accumulated a troubled history of exploited vulnerabilities.[5] Three vulnerability classes are endemic to this design:

Time-of-check-to-time-of-use (TOCTOU) races arise when a setuid program checks a condition (does this file exist? is it owned by the calling user?) and then acts on it. An attacker who controls the filesystem can substitute a different object — commonly via a symlink — between the check and the use. The classic /tmp race: a program creates /tmp/workfile, the attacker atomically replaces it with a symlink to /etc/passwd, and the subsequent write (privileged, as EUID 0) corrupts the target. Kernel mitigations /proc/sys/fs/protected_symlinks and protected_hardlinks, introduced in Linux 3.6, prevent following symlinks in sticky directories when the symlink owner differs from the follower's UID.

Environment variable attacks: setuid programs inherit the invoking process's environment. The LD_PRELOAD attack — injecting a malicious shared library to override standard functions — was the archetypal example, now largely mitigated by glibc suppressing LD_PRELOAD, LD_LIBRARY_PATH, and related variables for setuid executables. PATH manipulation, IFS injection in shell scripts, TERMINFO exploitation, and GCONV_PATH attacks represent the same pattern applied to different environment variables.

Shell execution from setuid programs: Any setuid program that invokes a shell — whether explicitly via system(3), popen(3), or implicitly by exec-ing a shell script — is vulnerable to argument injection, command injection, and environment attacks mediated through the shell. The -p flag to bash prevents effective UID from being reset to RUID, but the interaction between setuid and shell interpreters remains a perennial source of vulnerabilities.

sudo and the sudoers Model

The sudo utility, originally written by Bob Coggeshall and Cliff Spencer at the State University of New York at Buffalo around 1980 and substantially redesigned and maintained by Todd Miller,[6] provides a flexible delegation mechanism superior to raw setuid-root binaries. Rather than granting a fixed binary omnipotent root access, sudo allows administrators to specify precisely what commands a given user may run, as which target user, on which hosts, subject to password authentication, and with comprehensive logging.

The /etc/sudoers file (and the /etc/sudoers.d/ drop-in directory) encodes privilege grants as policy rules:

# Grant alice permission to restart nginx as root on any host
alice ALL=(root) /usr/bin/systemctl restart nginx

# Allow ops group to run all commands without re-authentication
%ops ALL=(ALL:ALL) NOPASSWD: ALL

# Bob may run specific read-only diagnostics on webserver01
bob webserver01=(root) /sbin/ifconfig, /usr/bin/netstat, \
    /bin/ss

# Prevent privilege escalation through editors
Cmnd_Alias EDITORS = /usr/bin/vi, /usr/bin/vim, /usr/bin/nano
alice ALL=(root) !EDITORS
sudoers

The rule format is <user_spec> <host_spec>=(<runas_user>:<runas_group>) <command_list>. Sudoers wildcards (*) and argument-free allowances are common misconfigurations. A rule permitting /usr/bin/find * effectively grants arbitrary file read, since find accepts -exec sh \;. The GTFOBins project documents hundreds of such privilege escalation vectors through legitimate binaries.[6]

⚠ Security Note

sudo -s and sudo -i grant shell access, effectively equivalent to sudo su. If any user in sudoers has these permissions, they have unrestricted root access. The !authenticate and NOPASSWD directives, applied carelessly, eliminate the authentication barrier that sudo's audit trail depends on.

Modern sudo hardening practices include using sudoedit (rather than sudo $EDITOR) for editing files — sudoedit copies the target to a temporary file, invokes the editor as the unprivileged user, then copies the result back, preventing editor escape sequences from granting a shell. Enabling the requiretty flag prevents invocation from non-interactive contexts (cron jobs, remote code execution). Logging to both syslog and a separate secure log, optionally in a centralized SIEM, provides the audit trail that justifies sudo's administrative overhead.

Daemon Privilege Separation

The most principled approach to Unix privilege management is the privilege-separated daemon architecture, exemplified by OpenSSH's privsep design, developed by Niels Provos and Markus Friedl around 2002 and formally described at USENIX Security 2003.[7]

The pre-privsep OpenSSH architecture ran a single process with effective UID 0, parsing arbitrary network input, performing host key cryptographic operations, and executing user authentication logic — all in the same privilege domain. A single exploitable bug anywhere in this code, in a program that was not merely reachable from the network but actively accepting connections from the entire internet, would yield immediate root compromise.

Provos and Friedl's redesign decomposed the daemon into a privilege hierarchy:[7]

  • A monitor process running as root, responsible only for operations that genuinely require privilege — accessing host private keys, validating passwords against PAM or the shadow database, and invoking login(1) — implementing each operation as a narrow, audited handler
  • A pre-authentication slave running as the unprivileged sshd user, chrooted to /var/empty (a directory with no useful files), responsible for all network I/O and most protocol parsing
  • A post-authentication slave, which after a successful login gets setuid()'d to the target user and handles the session

Communication between monitor and slave occurs over a Unix domain socket pair, using a simple, stateful request-response protocol. The monitor implements a strict state machine that rejects out-of-sequence requests — a compromised slave cannot, for example, request a session setup before completing authentication.[7] Even total compromise of the slave process yields no path to root: the slave can only request operations the monitor is willing to perform, in the states the monitor is prepared to accept them.

Wietse Venema's Postfix mail transfer agent embodies the same philosophy, applied to a more complex service.[8] Postfix consists of many specialized, short-lived daemons — smtpd (network reception), cleanup (message normalization), qmgr (queue management), local (local delivery), smtp (outbound delivery) — each running as the restricted postfix user, communicating via a carefully designed IPC substrate enforced by a master daemon. No single component has access to another's data or privilege. Venema designed Postfix explicitly as a security architecture first and an MTA second, in deliberate contrast to Sendmail, which by 2000 had accumulated decades of setuid vulnerabilities that made it one of the most exploited services on the early internet.

§ 03

Capability-Based Security Models

The capability model represents a fundamentally different approach to access control from the identity-based discretionary access control (DAC) of standard Unix or even the label-based mandatory access control of Bell-LaPadula. In a capability system, access rights are unforgeable, communicable tokens: possession of a capability to an object grants the specified access, without consulting any central authority, access control list, or ambient identity. The security check is the capability check; if you hold the token, you may act.

The formal theory was first articulated by Jack Dennis and Earl Van Horn in 1966,[9] in the context of systems design at MIT. A capability is comparable to a physical key: you may hand it to another agent, who then possesses the same access rights — except that unlike physical keys, capabilities can be attenuated (a read-write capability can be downgraded to read-only before transfer) and revoked (by invalidating the underlying object reference).

The Confused Deputy Problem

The key motivation for capability systems over ACL-based systems is illustrated by the confused deputy problem, described by Norm Hardy in 1988.[10] Consider a Fortran compiler service that is granted privileged access to a billing resource, allowing it to charge users for CPU time consumed. When a user submits a compilation job with input parameters crafted to name the billing file as the output path, the compiler — acting in its "deputy" capacity on behalf of the user, with its elevated billing authority — overwrites the billing database. The compiler cannot distinguish between "this output path was specified by the system's authority" and "this output path was provided by a potentially malicious user," because both are encoded the same way: as a pathname string in the ambient authority context.

In a capability system, the compiler would receive distinct capabilities: one for the billing resource (from the system), and one for the user-specified output location (from the user). These capabilities carry their authority in themselves and cannot be confused. The compiler's billing capability cannot be exercised on a user-supplied path, because the user's capability to their output file and the system's capability to the billing resource are separate, non-interchangeable tokens.

The confused deputy problem arises because ambient authority — the identity-based privilege that permeates everything a process does — cannot be partitioned. Capabilities solve this by making authority explicit and per-object.

POSIX Capabilities in Linux

Linux implements a partial capability system through POSIX capabilities,[11] which decompose root's monolithic privilege into approximately 40 distinct capability bits. Each process maintains five sets:

  • Effective (CapEff): Currently active capabilities, checked by the kernel at each privileged operation
  • Permitted (CapPrm): The superset of capabilities the process is allowed to have effective
  • Inheritable (CapInh): Capabilities that can survive across execve(2) if the file also has them inheritable
  • Bounding (CapBnd): A ceiling — no capability can enter Permitted unless it is in the bounding set (available since Linux 2.6.25)
  • Ambient (CapAmb): Capabilities that are automatically inherited by exec'd programs even without file capability bits (available since Linux 4.3)

File capabilities, stored as extended attributes (security.capability) on executable binaries, allow specific capabilities to be granted upon exec without the full setuid-root mechanism.[11] A web server requiring only privileged port binding need not run as root:

# Grant capability to bind privileged ports (effective + permitted)
setcap 'cap_net_bind_service=ep' /usr/local/bin/caddy

# Inspect capabilities on a binary
getcap /usr/local/bin/caddy
# /usr/local/bin/caddy: cap_net_bind_service=ep

# Programmatic capability manipulation with libcap
#include <sys/capability.h>

cap_t caps = cap_get_proc();
cap_value_t cap_list[] = { CAP_NET_BIND_SERVICE };

/* Move cap to permitted only, not effective */
cap_set_flag(caps, CAP_EFFECTIVE, 1, cap_list, CAP_CLEAR);
cap_set_proc(caps);
cap_free(caps);
C

The =ep suffix grants the capability in both Effective and Permitted sets at exec time. An =p grant places it in Permitted only — the process must explicitly raise it to Effective with cap_set_proc(3), useful for programs that want to minimize their window of elevated privilege.

Object Capability Systems

POSIX capabilities are not a complete capability model in the Dennis-Van Horn sense. They represent per-process privilege amplification flags, not per-object unforgeable tokens. CAP_SYS_ADMIN is particularly problematic: it encompasses mounting filesystems, configuring audit, creating namespaces, performing device I/O control, and dozens of other operations, making it effectively equivalent to root for most practical purposes. The granularity of the POSIX capability set is insufficient for fine-grained confinement.

True capability-based systems have been realized in research and production:

  • EROS (Extremely Reliable Operating System),[12] designed by Jonathan Shapiro, implemented a persistent capability system emphasizing confinement and verifiability. Its successor, Coyotos, informed the design of seL4.
  • seL4[13] implements capabilities as the sole mechanism for all kernel object access, with machine-checked proofs of the capability system's correctness — covered in depth in §07.
  • Capsicum[28] realizes capability principles within the FreeBSD kernel and POSIX API — covered in §06.
  • The E language[14] and its descendants (Pony, Wyvern, Crochet) embed capability safety into the type system, making it a compile-time property: if a module does not receive a capability as a parameter, it cannot exercise that authority, regardless of what other authority the runtime holds.
§ 04

Mandatory Access Control: SELinux and AppArmor

Discretionary access control — the Unix permission model where file owners control who may access their resources — is insufficient against compromised processes. A setuid binary running as root, once exploited, can access any resource root can access, regardless of DAC settings on those resources. Mandatory access control (MAC) addresses this by enforcing a global security policy that neither individual users nor processes can override, regardless of their DAC permissions.

SELinux: Flask Architecture and Type Enforcement

Security-Enhanced Linux (SELinux) originated at the National Security Agency as the Flask security architecture,[15] first implemented in the Mach microkernel and the Fluke research operating system at the University of Utah. Peter Loscocco and Stephen Smalley led the effort to port Flask to Linux, releasing the first public patch in December 2000.[16] Following the upstream integration of the Linux Security Modules (LSM) framework — a hook-based system allowing multiple security modules to intercept and mediate kernel operations — in Linux 2.6 (late 2003), SELinux became the reference MAC implementation for the Linux kernel.

Flask introduces a policy engine that intercepts every security-relevant operation (file open, socket connect, IPC send, signal delivery, capability use) via LSM hooks and consults a policy database to determine whether the operation is permitted. The policy is evaluated against security contexts — structured labels attached to every process and every kernel object.

The SELinux security context is a four-part label visible on any object:

# Show SELinux context of files
$ ls -Z /var/www/html/index.html
system_u:object_r:httpd_sys_content_t:s0  /var/www/html/index.html

# Show SELinux context of processes
$ ps -eZ | grep httpd
system_u:system_r:httpd_t:s0  1234 ?  00:00:01 httpd

# Format: user:role:type:sensitivity[:categories]
#         ↑       ↑      ↑    ↑
#         RBAC    RBAC   TE   MLS
bash

SELinux's three core policy mechanisms work in concert:[16]

Type Enforcement (TE): Every process runs in a domain (a type applied to processes); every object has a type. The policy consists of allow rules specifying which domains may perform which operations on which object types. Absence of an allow rule means denial — SELinux is default-deny.

Role-Based Access Control (RBAC): Domains are associated with roles, and users are associated with sets of permitted roles. Roles provide organizational structure: the sysadm_r role grants system administration domains; user_r grants only ordinary user domains.

Multi-Level Security (MLS): Bell-LaPadula sensitivity labels for classified environments. MLS enables enforcement of strict information flow control — ensuring that a process handling data labeled Secret cannot write to objects labeled Unclassified.

# Type Enforcement policy fragment
# httpd_t domain may read files of type httpd_sys_content_t
allow httpd_t httpd_sys_content_t:file { read getattr open };

# httpd_t may connect to TCP sockets on http_port_t
allow httpd_t http_port_t:tcp_socket { name_connect };

# httpd_t may NOT write to shadow_t (no allow rule = deny)
# httpd_t may NOT bind ports other than http_port_t, https_port_t

# Macro-driven policy in the reference policy framework
apache_content_template(httpd)

# Audit2allow: convert denial logs to candidate rules
$ ausearch -m avc -ts today | audit2allow -M mymodule
$ semodule -i mymodule.pp
selinux te

Red Hat Enterprise Linux and Fedora ship with the "targeted" policy, which enforces SELinux only on a curated set of high-risk daemons (httpd, named, sshd, dhcpd, postgresql, and others) while placing all other processes in the permissive unconfined_t domain. The "strict" and "MLS" policies confine all subjects, including the login shell and desktop environment, at the cost of substantially more complex policy development and maintenance.

ℹ Operational Note

SELinux boolean flags (getsebool -a, setsebool) allow runtime policy adjustments without recompiling policy modules. Booleans represent common configuration choices — httpd_can_network_connect, httpd_enable_cgi — and are the preferred mechanism for adapting the reference policy to site-specific requirements.

AppArmor: Path-Based MAC

AppArmor was developed by Immunix (founded by Crispin Cowan and colleagues), first described as "SubDomain" in 2000,[17] and subsequently acquired by Novell and then Canonical. It takes a deliberately different design philosophy from SELinux: rather than labeling every filesystem object with a security type, AppArmor defines per-application profiles that specify which filesystem paths, network operations, and capability flags a program may use.

The path-based model trades expressiveness for operational simplicity. Writing an AppArmor profile for a new application requires understanding what files it accesses, which can often be determined empirically using AppArmor's learning mode (complain mode), which logs violations without enforcing them:

# /etc/apparmor.d/usr.sbin.nginx
#include <tunables/global>

/usr/sbin/nginx {
  #include <abstractions/base>
  #include <abstractions/nameservice>

  # Capabilities required by nginx
  capability net_bind_service,
  capability setuid,
  capability setgid,
  capability dac_override,

  # Configuration read access (single level)
  /etc/nginx/             r,
  /etc/nginx/**           r,

  # Content (recursive read)
  /var/www/html/**        r,

  # Logging (write)
  /var/log/nginx/         rw,
  /var/log/nginx/**       rw,

  # PID file
  /run/nginx.pid          rw,

  # Network: TCP on inet/inet6
  network inet  tcp,
  network inet6 tcp,

  # Explicit denials (enforced over allow rules)
  deny /etc/shadow        r,
  deny /proc/sysrq-trigger rw,
  deny /sys/**            rw,
}
apparmor

AppArmor path matching uses glob patterns: /* matches one level, /** matches recursively. Profiles are compiled from text to binary format and loaded into the kernel via the AppArmor LSM, where they are enforced at every VFS operation, socket call, and capability check.

The path-based model has known limitations. Hard links allow reaching a file at a path other than its canonical location; bind mounts allow making a file appear at an entirely different path. SELinux's inode-label model is immune to these — the label follows the inode regardless of which path is used to access it. AppArmor's path model can be circumvented by an attacker who can arrange a bind mount or hard link, though these operations themselves typically require privilege.

TOMOYO, Smack, and Landlock

The Linux LSM framework accommodates multiple concurrent MAC systems. Beyond SELinux and AppArmor, several other frameworks are noteworthy:

TOMOYO Linux,[18] developed by NTT and mainlined in Linux 2.6.30, implements a path-based MAC system with a distinctive learning mode that automatically generates policy by observing program behavior during a training phase. TOMOYO treats policy construction as a systems administration task rather than a security engineering task — a pragmatic choice for deployment in environments without dedicated security staff.

Smack (Simplified Mandatory Access Control Kernel),[19] mainlined in Linux 2.6.25 and used in the Tizen embedded operating system, uses short string labels (smack labels) on files and processes. The rule set is intentionally simple: a subject label can read from objects with the same label or with an explicit access mode entry in the Smack rules table. Smack targets resource-constrained systems where SELinux's policy complexity would be prohibitive.

Landlock,[20] merged in Linux 5.13 (2021), is architecturally distinct from all the above: it is an unprivileged, self-restriction mechanism. An application can impose a Landlock ruleset on itself — restricting its own filesystem access — without requiring any system-wide policy deployment or administrator involvement. This makes it analogous to OpenBSD's pledge(2) and directly addresses the use case of sandboxing applications from within the application itself:

#include <linux/landlock.h>
#include <sys/syscall.h>

/* Define which access types this ruleset handles */
struct landlock_ruleset_attr rs_attr = {
    .handled_access_fs =
        LANDLOCK_ACCESS_FS_EXECUTE  |
        LANDLOCK_ACCESS_FS_READ_FILE |
        LANDLOCK_ACCESS_FS_WRITE_FILE |
        LANDLOCK_ACCESS_FS_MAKE_REG,
};
int rs_fd = landlock_create_ruleset(&rs_attr, sizeof(rs_attr), 0);

/* Allow read-only access to /var/www */
struct landlock_path_beneath_attr path_attr = {
    .allowed_access = LANDLOCK_ACCESS_FS_READ_FILE,
    .parent_fd = open("/var/www", O_PATH | O_DIRECTORY),
};
landlock_add_rule(rs_fd, LANDLOCK_RULE_PATH_BENEATH, &path_attr, 0);

/* Enforce: this is irreversible */
prctl(PR_SET_NO_NEW_PRIVS, 1, 0, 0, 0);
landlock_restrict_self(rs_fd, 0);
C

Once landlock_restrict_self() is called, the restriction applies to the calling process and all its children, and cannot be lifted. Landlock version 3 (Linux 5.19) extended coverage to network operations (TCP connect and bind), and version 4 (Linux 6.7) added control over ioctl operations.

§ 05

seccomp and Berkeley Packet Filter (BPF) Syscall Filtering

The Linux kernel exposes approximately 400 system calls, each representing both functionality and attack surface. A web server's legitimate operational requirements — socket I/O, file reads, memory allocation, process management — encompass perhaps 50 of those calls. The remaining 350-odd, including ptrace(2), pivot_root(2), init_module(2), kexec_load(2), and perf_event_open(2), provide an attacker with powerful kernel exploitation primitives if they can reach the post-exploitation phase. Seccomp (secure computing) addresses this by allowing processes to restrict their own syscall surface.

seccomp Strict Mode

Original seccomp, contributed by Andrea Arcangeli and merged in Linux 2.6.12 (2005),[21] implements a maximally restrictive mode: upon calling prctl(PR_SET_SECCOMP, SECCOMP_MODE_STRICT), a process is limited to exactly four system calls: read(2), write(2), exit(2), and sigreturn(2). Any other syscall causes an immediate SIGKILL. The rationale was to allow compute services to accept untrusted code and execute it with near-zero kernel attack surface — essentially a sandboxed computation model where all I/O setup occurs before the sandboxed computation begins. The restriction was too severe for most real applications, and strict mode saw minimal deployment outside of specialized compute sandboxes.

BPF Origins and the eBPF Evolution

Seccomp-BPF, merged in Linux 3.5 (2012),[22] transformed seccomp into a practical tool by allowing user-defined filters expressed in Berkeley Packet Filter (BPF) bytecode. BPF was developed by Steven McCanne and Van Jacobson at Lawrence Berkeley National Laboratory and presented at the USENIX Winter Conference in 1993.[23] Its original purpose was efficient packet filtering for tools like tcpdump(1): rather than copying every network packet to user space for the application to evaluate filter expressions, BPF programs execute in the kernel, discarding unmatched packets before they incur the copy overhead.

The BPF virtual machine is a simple register machine with an accumulator, index register, a small scratch memory store, and an implicit pointer to the packet (or, in seccomp-BPF's case, to the seccomp_data structure containing the syscall number and arguments). BPF programs are validated before installation to guarantee termination (no loops, bounded instruction count) and memory safety (all loads are bounds-checked). These properties — in-kernel execution, formal verifiability, event-driven triggering — made BPF's model attractive far beyond packet filtering.

Alexei Starovoitov and Daniel Borkmann extended BPF substantially in Linux 3.18 (2014), creating extended BPF (eBPF).[24] eBPF is a general-purpose JIT-compiled in-kernel virtual machine with 10 64-bit registers, a read-only frame pointer, a 512-byte stack, persistent map data structures shared between kernel and user space, and hook points spanning virtually every kernel subsystem: kprobes and uprobes for dynamic tracing, tracepoints for static instrumentation, XDP for network packet processing at the NIC driver level, tc (traffic control) for socket-level filtering, cgroup hooks for per-cgroup enforcement, LSM hooks for security policy, and socket operations for network policy.

The eBPF verifier performs abstract interpretation of all possible execution paths in the program before it is loaded, proving that the program cannot dereference invalid pointers, read uninitialized memory, or access kernel memory outside of defined regions.[24] This in-kernel verification is a form of lightweight formal verification — not a full proof in the Isabelle/HOL sense, but sufficient to prevent memory safety violations and ensure program termination.

/* seccomp-BPF filter using libseccomp */
#include <seccomp.h>

int install_sandbox_filter(void) {
    scmp_filter_ctx ctx;

    /* Default action: kill entire process group */
    ctx = seccomp_init(SCMP_ACT_KILL_PROCESS);
    if (!ctx) return -1;

    /* Allowlist: basic I/O and memory management */
    seccomp_rule_add(ctx, SCMP_ACT_ALLOW, SCMP_SYS(read), 0);
    seccomp_rule_add(ctx, SCMP_ACT_ALLOW, SCMP_SYS(write), 0);
    seccomp_rule_add(ctx, SCMP_ACT_ALLOW, SCMP_SYS(close), 0);
    seccomp_rule_add(ctx, SCMP_ACT_ALLOW, SCMP_SYS(exit_group), 0);
    seccomp_rule_add(ctx, SCMP_ACT_ALLOW, SCMP_SYS(brk), 0);
    seccomp_rule_add(ctx, SCMP_ACT_ALLOW, SCMP_SYS(mmap), 0);
    seccomp_rule_add(ctx, SCMP_ACT_ALLOW, SCMP_SYS(munmap), 0);
    seccomp_rule_add(ctx, SCMP_ACT_ALLOW, SCMP_SYS(fstat), 0);

    /* openat: allow read-only flags only */
    seccomp_rule_add(ctx, SCMP_ACT_ALLOW, SCMP_SYS(openat), 1,
        SCMP_A2(SCMP_CMP_MASKED_EQ, O_WRONLY | O_RDWR, 0));

    /* Futex for threading primitives */
    seccomp_rule_add(ctx, SCMP_ACT_ALLOW, SCMP_SYS(futex), 0);

    /* Commit the filter — this applies to all threads */
    int rc = seccomp_load(ctx);
    seccomp_release(ctx);
    return rc;
}
C

Seccomp Filter Actions

Seccomp-BPF filters inspect the seccomp_data structure — containing the syscall number, architecture identifier, instruction pointer, and up to six argument values — and return one of eight possible actions:[22]

  • SECCOMP_RET_ALLOW: Permit the syscall unconditionally
  • SECCOMP_RET_KILL_PROCESS: Terminate the entire thread group with SIGSYS (Linux 4.14+)
  • SECCOMP_RET_KILL_THREAD: Terminate only the calling thread
  • SECCOMP_RET_TRAP: Deliver SIGSYS, allowing a signal handler to inspect the blocked call
  • SECCOMP_RET_ERRNO(e): Return the specified errno to the caller without entering the kernel
  • SECCOMP_RET_TRACE: Notify a ptrace supervisor and await its decision
  • SECCOMP_RET_LOG: Allow but write to the seccomp audit log
  • SECCOMP_RET_USER_NOTIF: Block and notify a user-space supervisor via a file descriptor, which may then approve, reject, or spoof the call (Linux 5.0)[25]

SECCOMP_RET_USER_NOTIF is particularly significant: it enables a container manager or supervisor process to intercept and handle syscalls on behalf of a sandboxed process, implementing policies more complex than BPF bytecode allows. The syscall arguments are read from /proc/PID/mem by the supervisor, which can then respond with SECCOMP_USER_NOTIF_FLAG_CONTINUE (allow), an error code (deny), or a spoofed return value (intercept).

Argument Filtering Caveats

Filtering on syscall arguments presents subtle correctness issues. The filter sees raw register values at syscall entry — it cannot follow pointers into user space (doing so would introduce TOCTOU races between filter evaluation and kernel handling). For openat(2), the filter can check flags and the dirfd value, but cannot inspect the pathname string. A filter wishing to allow openat to /proc/self/status but block all other paths must use SECCOMP_RET_USER_NOTIF with a supervisor that reads the path from the process's address space.

Architectural multiplicity is another correctness concern. On x86_64, 32-bit syscalls issued via int 0x80 use different syscall numbers and a different kernel path than 64-bit syscalls issued via syscall. A filter checking only 64-bit syscall numbers could be bypassed by issuing the equivalent 32-bit syscall. Correct filters must check BPF_ARCH against AUDIT_ARCH_X86_64 as a first step and reject any non-matching architecture unconditionally.

Deployment: Chrome, Docker, and systemd

Google Chrome was the first widely deployed application to use seccomp-BPF as a core sandbox layer.[26] The Chrome renderer process — responsible for parsing HTML, executing JavaScript, and rendering graphics — operates under an aggressive seccomp filter that blocks the vast majority of syscalls. A fully exploited renderer process cannot call execve(2), fork(2), clone(2), socket(2), or connect(2). Even code-execution capability within the renderer is largely contained: the attacker controls only the renderer's address space and the small set of allowed syscalls, severely limiting pivot capability. Chrome's sandbox layers seccomp-BPF with a separate broker process (analogous to Capsicum's casper daemon) for operations requiring cross-sandbox resource access.

Capsicum: Fine-Grained Capabilities in FreeBSD

Capsicum, developed by Robert Watson, Jonathan Anderson, Ben Laurie, and Kris Kennaway at the University of Cambridge and Google, and first presented at USENIX Security 2010,[28] represents one of the most successful practical realizations of object capability principles in a mainstream POSIX operating system. Its core insight is that the Unix file descriptor is already an unforgeable kernel-managed reference to a kernel object — it requires only the addition of per-descriptor rights restrictions to become a genuine capability.

Capability Mode and Descriptor Rights

Capsicum introduces a process state called capability mode, entered by calling cap_enter(2). Once in capability mode, the process loses access to all global namespaces: it cannot open files by pathname, resolve hostnames, manipulate arbitrary processes, or perform any operation that requires naming a resource by its global identifier. All resource access must occur through file descriptors obtained before entering capability mode or explicitly received from a trusted component over a socket.

Capability mode is a one-way transition — there is no cap_exit(). This unidirectionality is a security property: it means the confinement cannot be escaped by calling a single function. The process's future access to resources is entirely determined by the set of file descriptors it holds at the moment of cap_enter(), and whatever rights those descriptors carry.

File descriptor rights are refined using cap_rights_limit(2), which applies a bitmask of permitted operations to a descriptor. Rights can only be reduced, never expanded — monotonic restriction ensures that a compromised subcomponent cannot escalate its own descriptor rights:

#include <sys/capsicum.h>

/* Step 1: Acquire resources before entering capability mode */
int config_fd = open("/etc/app/config.toml", O_RDONLY);
int sock_fd   = socket(AF_INET, SOCK_STREAM, 0);
int log_fd    = open("/var/log/app.log", O_WRONLY | O_APPEND);

/* Step 2: Restrict each descriptor to minimum required rights */
cap_rights_t r_read, r_write, r_net;

cap_rights_init(&r_read, CAP_READ, CAP_SEEK, CAP_FSTAT);
cap_rights_limit(config_fd, &r_read);

cap_rights_init(&r_write, CAP_WRITE, CAP_FSTAT);
cap_rights_limit(log_fd, &r_write);

cap_rights_init(&r_net, CAP_CONNECT, CAP_RECV, CAP_SEND,
                         CAP_SETSOCKOPT, CAP_GETSOCKOPT);
cap_rights_limit(sock_fd, &r_net);

/* Step 3: Enter capability mode — no return */
cap_enter();

/* From here: read config succeeds, open("/etc/passwd") fails ECAPMODE */
read(config_fd, buf, sizeof(buf));   /* OK */
open("/etc/passwd", O_RDONLY);      /* ECAPMODE */
connect(sock_fd, &addr, addrlen);   /* OK — sock_fd has CAP_CONNECT */
write(sock_fd, data, len);          /* ENOTCAPABLE — only CAP_SEND allowed */
C

The full rights taxonomy in Capsicum covers not just the obvious I/O operations but also fine-grained control over socket operations, ioctl commands (via cap_ioctls_limit(2)), and fcntl commands (via cap_fcntls_limit(2)). A descriptor for a socket can be restricted to allow CAP_RECV and CAP_SEND but deny CAP_BIND, CAP_LISTEN, and CAP_CONNECT — appropriate for a socket that has already been connected before capability mode was entered.

Sandboxing Real Programs

Watson et al. demonstrated that Capsicum could be integrated into real-world programs with modest modifications.[28] Key examples from the FreeBSD base system:

  • tcpdump(8): Opens its capture interface via pcap_create() and output save file before calling cap_enter(). The entire packet processing and decoding loop then runs in capability mode.
  • dhclient(8): Enters capability mode after binding its UDP socket on port 68, restricting the socket descriptor to network I/O only.
  • ping(8): Restricted to a capability for the raw ICMP socket and a descriptor for /etc/hosts name resolution.
  • kdump(1): Sandboxes the per-process output stage in a child process under capability mode.

The key observation was that most programs naturally acquire all their resources at startup — during argument parsing, configuration file reading, and socket setup — and then enter a processing loop that operates on those resources without needing new ones. Capsicum's capability mode formalizes this pattern, making the pre-/post-initialization boundary an explicit security boundary.

The casper(8) Daemon

Some operations inherently require ambient authority: resolving hostnames requires querying DNS or reading /etc/hosts; looking up users requires reading /etc/passwd; generating random numbers requires accessing /dev/urandom. Capsicum's solution is casper(8)[28] — a trusted supervisor daemon that provides ambient services through capability-safe channels.

A Capsicum-ized process opens a Casper channel before entering capability mode. The channel is a file descriptor representing a connection to the Casper daemon. After cap_enter(), the process can request services through this channel:

#include <libcasper.h>
#include <casper/cap_dns.h>
#include <casper/cap_pwd.h>

/* Open Casper channels before entering capability mode */
cap_channel_t *capcas  = cap_init();
cap_channel_t *capnet  = cap_service_open(capcas, "system.dns");
cap_channel_t *cappwd  = cap_service_open(capcas, "system.pwd");

/* Restrict DNS channel to specific domains only */
const char *domains[] = { "example.com", "api.example.com", NULL };
cap_dns_type_limit(capnet, "NAME");   /* forward lookups only */

/* Enter capability mode */
cap_enter();

/* Resolve a hostname via Casper — works despite capability mode */
struct addrinfo *res;
cap_getaddrinfo(capnet, "api.example.com", "https", NULL, &res);

/* Look up a user by name via Casper */
struct passwd *pw = cap_getpwnam(cappwd, "nobody");
C

The Casper daemon performs each operation in its own isolated process and can apply additional filtering on top of capability mode's restrictions. A DNS Casper service configured to resolve only example.com subdomains will reject lookups for arbitrary hostnames, even if the sandboxed process requests them. This allows construction of multi-level capability hierarchies where each capability is scoped as narrowly as the protocol permits.

Comparison: pledge(2) and unveil(2) on OpenBSD

OpenBSD's pledge(2) system call,[29] introduced in OpenBSD 5.9 (2015) and designed by Theo de Raadt, takes a different approach to the same problem. Rather than capability descriptors, pledge defines named promise categories — strings representing groups of related syscalls:

/* Restrict this process to stdio, filesystem reads, and network */
pledge("stdio rpath inet dns", NULL);

/* Common promise groups: */
/* stdio  — read, write, basic I/O multiplexing */
/* rpath  — open(O_RDONLY), stat, readdir */
/* wpath  — open(O_WRONLY) */
/* cpath  — creat, mkdir, rename, unlink */
/* inet   — socket(AF_INET), connect, recv, send */
/* dns    — getaddrinfo, getnameinfo (libc wrappers) */
/* proc   — fork, exec (NOT available after first pledge) */
/* exec   — execve */
/* id     — setuid, setgid, getgroups */
/* tty    — terminal operations */
C

After a pledge() call, a violation causes SIGABRT delivery (or SIGKILL in more recent OpenBSD versions) and a syslog entry identifying the violating process and the denied operation. The pledge set can be progressively narrowed but never expanded. A common pattern is to start with a broad pledge (including proc exec during initialization), then re-pledge to a more restrictive set after setup is complete.

unveil(2),[30] introduced in OpenBSD 6.4 (2018), complements pledge by restricting filesystem visibility. A process that calls unveil("/var/www", "r") followed by unveil(NULL, NULL) can only access /var/www and its descendants with read permission — all other filesystem paths are invisible, as if they do not exist (returning ENOENT rather than EPERM):

/* Reveal only the paths this process needs */
unveil("/var/www/html", "r");       /* read content */
unveil("/var/log/app", "rw");       /* read/write logs */
unveil("/etc/app/config.toml", "r");/* read config */
unveil(NULL, NULL);                  /* lock - no more unveil calls */
C

Pledge and unveil are intentionally coarser than Capsicum's per-descriptor model. The promise categories are broad, and the filesystem restrictions are path-based (sharing AppArmor's susceptibility to hardlink/bind-mount circumvention). The design philosophy prioritizes adoption simplicity over expressiveness — and the results validate this tradeoff. By OpenBSD 7.4, virtually the entire base system (including the web server, email stack, DNS resolver, and most utilities) operates under both pledge and unveil, demonstrating that a coarser model can achieve meaningful security improvements at acceptable engineering cost for real-world software.

Microkernel Architectures and Minimal Trusted Computing Base

The trusted computing base (TCB) of a system is the set of hardware and software components whose correct, non-malicious behavior is required for the system's security properties to hold. It is distinct from the set of components the system trusts — the TCB is the set that must be trusted because the security policy cannot be enforced without it. An exploitable vulnerability anywhere in the TCB can undermine the system's entire security posture.

In a monolithic kernel — Linux, Windows, macOS XNU — the TCB encompasses the entire kernel: millions of lines of device drivers, filesystem implementations, network stacks, and subsystems, all executing in ring 0 with unrestricted access to physical memory, I/O ports, and all CPU registers. A single kernel memory corruption vulnerability anywhere in this code base can yield arbitrary kernel code execution. A 2023 study of Linux kernel CVEs showed that device drivers and filesystem code account for over 60% of kernel vulnerabilities, yet both run in ring 0 with full kernel privileges.

A microkernel is wrong if anything is included in it which can be prevented from being there.

— Jochen Liedtke, "On μ-kernel Construction," SOSP 1995[31]

The microkernel architecture attempts to minimize the TCB by moving as much kernel functionality as possible into unprivileged user-space servers. The kernel itself retains only those mechanisms that genuinely require hardware privilege: address space management, inter-process communication, thread scheduling, and hardware interrupt dispatch.

Mach: The First Widely Adopted Microkernel

The Mach microkernel, developed at Carnegie Mellon University beginning in 1985 by Richard Rashid, Avadis Tevanian, and colleagues,[32] was the first microkernel to achieve broad adoption in production systems. Mach's key abstractions are:

  • Task: A protected address space together with a port namespace, roughly corresponding to a Unix process
  • Thread: An execution context within a task
  • Port: A protected, kernel-managed message queue, serving as the fundamental IPC primitive and as a capability-like reference
  • Message: A typed data structure containing data and port rights, sent atomically between tasks

Mach's port model has genuine capability properties: a send right to a port allows transmitting messages to it; a receive right allows reading messages from it; a send-once right allows a single message transmission. Rights are transferred via messages, with the kernel mediating transfers and maintaining type safety. Tasks that do not hold a send right to a port cannot contact the service behind it.

Mach became the basis for macOS (via NeXTSTEP), GNU Hurd, and several research operating systems. However, Mach's practical IPC performance was substantially worse than monolithic kernel system call overhead.[33] A Unix system call implemented in a user-space BSD server layered atop Mach required at minimum two context switches and two IPC operations — enter kernel, IPC to BSD server, execute, IPC back, return. This overhead motivated either collapsing servers back into the kernel (creating a compromise architecture, as macOS does) or a fundamental rethinking of microkernel design.

L4: Minimality as Proof

Jochen Liedtke at GMD (the German national research center for information technology) published a highly influential analysis in 1993[33] arguing that Mach's poor IPC performance was not a fundamental property of microkernels but a consequence of specific implementation choices — Mach's IPC path was too long, traversed too many layers, and failed to exploit processor architecture efficiently.

Liedtke's original L4 microkernel demonstrated IPC performance one to two orders of magnitude faster than Mach, by implementing the IPC path entirely in hand-optimized assembly, fitting critical data structures in CPU registers, and minimizing the number of kernel boundary crossings per IPC operation. His 1995 SOSP paper[31] articulated the minimality principle quoted above and demonstrated that a microkernel providing only address spaces, threads, and IPC could support the full POSIX API via user-space servers with acceptable performance overhead.

The L4 family diversified into multiple research and production implementations: L4Ka::Pistachio, NOVA, Fiasco.OC (used in automotive systems), OKL4 (used in cellular basebands), and ultimately seL4. Each iteration refined the tradeoffs among IPC performance, formal tractability, and real-world applicability.

seL4: Formal Verification of a General-Purpose Kernel

seL4, developed at NICTA (now CSIRO's Data61) by Gerwin Klein, Kevin Elphinstone, Gernot Heiser, and a team of formal methods researchers and systems programmers,[13] represents the most significant advance in microkernel security: a fully functional, performance-competitive OS kernel with machine-checked proofs of correctness and security.

The seL4 verification establishes a chain of refinement between three levels of specification:

  1. Abstract specification: A mathematical specification of seL4's intended behavior, written in Isabelle/HOL, defining what the kernel should do in response to each operation
  2. Executable specification: A Haskell implementation directly derived from the abstract specification, executable for testing
  3. C implementation: The approximately 8,700-line C implementation actually deployed, proved to be a refinement of the executable specification

The refinement proof means that any property provable about the abstract specification holds for the C implementation. Two key security properties have been proved:[13]

  • Integrity: A subject cannot modify a kernel object unless it holds a write capability to that object — the capability system correctly enforces authority
  • Confidentiality (proved subsequently): Information can flow from high-confidentiality domains to low-confidentiality domains only through explicitly authorized channels — the system enforces noninterference

The proof effort was enormous — the initial verification consumed approximately 11 person-years and produced around 200,000 lines of Isabelle/HOL proof script. The companion proof of absence of unsafe C behaviors (no buffer overflows, no null pointer dereferences, no use-after-free, no undefined behavior) in the C implementation adds further assurance.

seL4 uses a capability-based access control model throughout. Every kernel object — thread control blocks (TCBs), address spaces (CNodes and VSpaces), endpoints (IPC channels), notification objects, and untyped memory — is accessed exclusively through capabilities held in CNodes. The Untyped capability represents physical memory; retyping an Untyped capability creates other kernel objects, making memory allocation itself a capability-controlled operation. No operation on any kernel object is possible without holding an appropriate capability.

seL4 has seen deployment in DARPA's HACMS (High-Assurance Cyber Military Systems) program, in Boeing autonomous helicopter platforms, in automotive systems (particularly trusted execution environments), and as the basis for the secure RISC-V platform developed by the CHERI project.

MINIX 3 and the Intel Management Engine Discovery

Andrew Tanenbaum's MINIX 3,[34] designed as a highly reliable, self-repairing operating system for embedded and reliability-critical contexts, gained unexpected notoriety in 2017 when security researchers Maxim Goryachy and Mark Ermolov revealed that Intel's Management Engine — a separate processor embedded in virtually all Intel chipsets since 2008 — runs a modified MINIX 3 kernel.[35]

The Intel ME operates at a privilege level below the main operating system kernel, in a domain often called "ring -3." The ME has access to system RAM (via DMA), to the network interface (including a dedicated network stack operating independently of the host OS), to cryptographic hardware, and to the EC (embedded controller). It cannot be disabled by the host operating system — attempts to do so typically result in system shutdown or reset.

This revelation illustrated a fundamental point about TCB analysis in modern systems: the trusted computing base of an x86 platform extends well below the operating system to include firmware, microcode, out-of-band management processors, and the baseboard management controller. The total TCB of a modern server runs to tens of millions of lines of firmware code, most of it unaudited, unverified, and running at privilege levels the host OS cannot observe. Microkernel principles applied at the OS level are necessary but not sufficient — a fully verified OS atop an unverified firmware TCB remains vulnerable to firmware-level attacks.

Fuchsia and Zircon: Capability Principles at Scale

Google's Fuchsia operating system[36] uses the Zircon kernel — a microkernel-influenced design that represents perhaps the most ambitious contemporary attempt to realize capability principles in a production general-purpose OS. In Zircon, every kernel object is accessed through a handle — an unforgeable reference analogous to a Mach port right or a Capsicum capability descriptor. Processes, threads, virtual memory objects, channels, sockets, and event objects are all created and accessed through handles.

Handles can be transferred between processes via Zircon channels (bidirectional pipe-like IPC primitives), enabling capability delegation. A parent process creates a child with specific handles — the child can only access what it has been explicitly given. Fuchsia's component framework layers atop this: each software component declares its capability requirements in a component manifest, and the component manager mediates routing of capabilities between components based on a static topology. A component that does not declare a requirement for (say) the writable filesystem service cannot receive a capability to it, regardless of runtime attempts to obtain one.

User Namespaces and Privilege Escalation Mitigation

Linux namespaces partition global kernel resources, creating isolated instances that a group of processes perceives as their own private resource view.[37] They are the kernel mechanism that makes container technologies like Docker, Podman, and Kubernetes possible. Each namespace type isolates a different category of global state, and multiple namespace types can be combined to create environments that are substantially isolated from the host system.

User namespaces,[38] merged in Linux 3.8 (2013) after a multi-year development effort by Eric Biederman, are the most security-significant and architecturally complex namespace type. A user namespace defines a mapping between UIDs and GIDs inside the namespace and UIDs/GIDs in the parent namespace:

bash

A process with effective UID 0 within a user namespace has capabilities within that namespace's scope, but only with respect to resources owned by that namespace. It can create files that appear UID 0 within the namespace (but map to UID 1000 on the host), set up new namespaces nested within its user namespace, and perform operations allowed by its in-namespace capabilities. It cannot affect resources in the parent namespace or owned by other users.

This design enables a key property: an ordinary unprivileged user can create a user namespace in which they appear as root and have full capabilities, using only those capabilities to set up an isolated environment. No setuid binary, no sudo, no privileged helper is required.

Rootless Containers

Rootless containers leverage user namespaces to run container workloads without any privileged component in the host environment.[39] The runtime (Podman, rootless Docker, or an OCI runtime like crun) executes entirely as an unprivileged user, creating user, mount, PID, UTS, and IPC namespaces using clone(2) or unshare(2) with no elevated privilege:

# Rootless container with Podman — no root, no setuid
$ id
uid=1000(user) gid=1000(user) groups=1000(user)

$ podman run --rm -it alpine:latest sh

# Inside the container
/ # whoami
root
/ # id
uid=0(root) gid=0(root)

# From the host — the container "root" is actually UID 100000
$ ps -eo user,pid,cmd | grep sh
100000   45678  sh
bash

Several technical challenges arise in rootless operation that do not exist for root-privileged container runtimes:

  • Networking: Creating veth pairs and configuring iptables requires CAP_NET_ADMIN in the network namespace, which requires creating the network namespace inside a user namespace. User-space networking via slirp4netns provides TCP/IP connectivity without kernel-level network configuration.
  • Overlay filesystems: overlay2 storage requires kernel support for unprivileged overlay mounts (added in Linux 5.11 with CONFIG_OVERLAY_FS_REDIRECT_ALWAYS_FOLLOW=y). Prior to this, fuse-overlayfs — a FUSE implementation of overlayfs semantics — provides an alternative at some performance cost.
  • Port binding: Binding to ports below 1024 inside a rootless container requires CAP_NET_BIND_SERVICE within the network namespace or the net.ipv4.ip_unprivileged_port_start sysctl to be set appropriately.

Privilege Escalation Vectors and CVE History

Unprivileged user namespace creation introduced significant new kernel attack surface.[40] Within a user namespace, an unprivileged process gains capabilities it did not have before, allowing it to exercise code paths — network namespace configuration, filesystem mounting, netfilter rule manipulation — that were previously accessible only to root. Kernel bugs in those newly reachable paths become locally exploitable privilege escalation vulnerabilities.

The pattern appears repeatedly in CVE history:

Distribution Responses and Kernel Mitigations

Linux distributions have responded to the user namespace attack surface with various restriction mechanisms:

# Debian/Ubuntu (historical): disable unprivileged user namespace creation
sysctl kernel.unprivileged_userns_clone=0

# Fedora (default): full user namespace access for unprivileged users
# RHEL 9.x: unprivileged user namespaces enabled (changed from RHEL 8)

# Ubuntu 24.04: AppArmor-mediated user namespace restriction
# Programs must have AppArmor profile permitting userns creation
# OR be in the allowlist at /proc/sys/kernel/apparmor_restrict_unprivileged_userns

# Max user namespaces per user (0 = disabled)
sysctl user.max_user_namespaces=0
bash

Ubuntu 24.04 LTS introduced a new AppArmor-based restriction mechanism: by default, unprivileged user namespace creation is blocked unless the calling process's AppArmor profile explicitly permits it (via the userns abstraction). This approach allows rootless container tools (Podman, Docker Desktop, Buildah) to be allowlisted while preventing arbitrary programs from creating user namespaces.

Linux 6.7 introduced infrastructure for LSM-mediated namespace creation control, allowing security modules to apply per-process restrictions. The KRSI (Kernel Runtime Security Instrumentation) framework, leveraging eBPF LSM hooks, allows user-space security daemons to implement custom namespace creation policies in eBPF programs without kernel modifications.

Defense-in-Depth for Container Security

Effective container security requires all isolation mechanisms in concert. The following stack represents current best practice for a hardened OCI container:

Kubernetes security contexts encode this layering declaratively:

apiVersion: v1
kind: Pod
spec:
  securityContext:
    runAsNonRoot: true
    runAsUser: 1000
    runAsGroup: 3000
    fsGroup: 2000
    seccompProfile:
      type: RuntimeDefault   # containerd/crio built-in seccomp
    sysctls: []
  containers:
  - name: app
    image: registry.example.com/app:v1.4.2
    securityContext:
      allowPrivilegeEscalation: false
      readOnlyRootFilesystem: true
      capabilities:
        drop: [ALL]
        add: [NET_BIND_SERVICE]  # only if binding < 1024
yaml

The allowPrivilegeEscalation: false flag sets the no_new_privs bit on the container process, preventing it from gaining additional privileges via setuid binaries or file capabilities within the container. Combined with a read-only root filesystem and dropped capabilities, this configuration substantially degrades an attacker's ability to persist, pivot, or escalate even after gaining code execution within the container.

Namespace persistence enables more complex security architectures: namespaces can be pinned by bind-mounting /proc/PID/ns/TYPE to a filesystem path, then re-entered by processes that obtain the path. This mechanism underlies ip netns exec, nsenter(1), and the rootless container runtime's mechanism for attaching to a running container's namespace without re-creating it.

§ 09

Conclusion: The Layered Trust Hierarchy

The mechanisms surveyed across this article represent successive refinements of a single architectural principle: trust should be earned, scoped, and minimized — not assumed, ambient, and universal. From the raw Unix setuid bit's narrow delegation of specific elevated operations, through Capsicum's unforgeable per-descriptor capability tokens, through seL4's machine-checked proofs of noninterference, each mechanism is an attempt to enforce that software components do only what they are intended to do, even when they are compromised.

The historical arc is illuminating. The setuid mechanism (1970s) addressed a genuine problem — delegating specific privileged operations to user programs — with a mechanism that created new problems of its own: race conditions, environment variable attacks, the implicit privileged context persisting across the entirety of execution. Each subsequent mechanism can be read as a response to the residual attack surface of its predecessors.

sudo and privilege-separated daemons (1980s-2000s) refined the Unix model by separating the monitor (high-privilege, small, audited) from the worker (low-privilege, large, potentially vulnerable). POSIX capabilities (1990s-2000s) fragmented root's monolithic authority into distinct permission bits, enabling partial privilege without full root. MAC frameworks (2000s) introduced policy-enforced behavioral constraints that survive even root compromise. Seccomp-BPF (2012) extended these constraints to the kernel API surface, denying attackers the syscall primitives needed to escalate. Capsicum (2010) and OpenBSD pledge (2015) brought object capability discipline into mainstream POSIX, making the capability model a viable engineering choice rather than a research curiosity. Formally verified microkernels (seL4, 2009+) extended assurance from "we believe this is correct" to "we have proved this is correct." User namespaces (2013) enabled privilege isolation without the complexity of privilege — at the cost of new kernel attack surface that continues to produce exploitable CVEs.

No single mechanism is sufficient. The appropriate deployment for a security-sensitive workload combines all available layers:

  • Privilege separation and prompt privilege dropping in daemon design
  • File capabilities in preference to setuid-root binaries
  • seccomp-BPF filters appropriate to each component's syscall requirements
  • MAC profiles (SELinux or AppArmor) for all confined services
  • Capsicum or pledge for applications where the pattern fits
  • User and mount namespaces for container workload isolation
  • Capability dropping (drop ALL, add only needed) in container security contexts
  • Microkernel-based trusted execution for the highest-assurance components

The trend toward formal verification, visible in seL4's refinement proofs and the eBPF verifier's abstract interpretation, suggests that the next decade will move from the current engineering-focused approach — we believe this implementation is correct — toward mathematical guarantee — we have proved this implementation is correct. Landlock's ongoing formalization, the CHERI project's hardware-enforced capability architecture, and the extension of seL4 verification to new architectures and configurations all point in this direction.

Saltzer and Schroeder's eight principles, articulated in 1975 for MIT's Multics time-sharing system, remain the canonical design framework. The machinery implementing those principles has grown enormously more sophisticated — from permission bits to verified microkernels — but the underlying insight has not changed: minimize the surface of trust, separate the privileges that must exist, and fail safely when violations occur.
