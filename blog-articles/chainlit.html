ChainlitTurning a Python function into a production chat interface, one decorator at a time

Chainlit is the open-source framework that took the frontend out of the critical path for LLM application development. This guide covers why it exists, how its decorator model actually works, what it plugs into, how it holds up as production enterprise infrastructure, and where it sits against the rest of the conversational-AI tooling landscape.

★ 12.2k GitHub stars[1] Apache 2.0 license[1] 11.7k+ dependent projects[1] current release v2.11.1[1] community-maintained since May 2025[1]
Read this first

Chainlit is an open-source Python framework, first released in 2023 by Paris-based Chainlit SAS, that renders a polished, ChatGPT-style web interface directly from a handful of Python decorators — no HTML, CSS, or JavaScript required.[1] It exists to remove the frontend build-out (WebSocket plumbing, message-streaming UI, session state) as a blocker between a working LLM pipeline and a usable product.

As of 1 May 2025 the original team stepped back from active development and the project moved to community maintenance under a formal Maintainer Agreement, with Chainlit SAS providing no warranty on future updates.[1] The framework nonetheless continues to ship regular releases — v2.11.1 at the time of writing — and remains one of the most-used purpose-built UI layers for LLM applications, with two disclosed high-severity CVEs in its history that are instructive for anyone planning a production deployment.

01

Why Chainlit exists

Every team that has tried to ship a large language model to real users runs into the same wall at roughly the same point. The model works. The prompt is tuned. The retrieval pipeline returns the right documents. And then someone asks the obvious question: how do people actually talk to this thing? Before frameworks like Chainlit existed, the honest answer was to hand-build a chat surface — a FastAPI or Flask backend, a WebSocket layer for token streaming, a React or Vue frontend for message bubbles, typing indicators, and file uploads, and enough session-management code to keep one user's conversation from bleeding into another's. One write-up on the framework's origins put it plainly: Chainlit was built to address a critical gap in the AI application development process: the need for a straightforward way to transform machine learning models into interactive, user-friendly applications without extensive web development experience.[2]

That gap was not really about difficulty in the abstract sense — competent teams can and did build bespoke chat frontends. It was about where engineering time went. A tutorial aimed at newcomers frames the same problem from the other side: building a user-friendly chat interface that can effectively interact with these models often requires significant frontend development skills, skills that the data scientists and ML engineers writing the actual model logic frequently do not have and should not need to acquire just to demo their work.[3] Chainlit's answer is to collapse the frontend into a set of Python decorators that a backend engineer can read and modify without touching HTML, CSS, or JavaScript at all.

What Chainlit actually is

Chainlit is an open-source Python package, licensed under Apache 2.0, for building conversational interfaces on top of arbitrary Python logic.[1] The official framing is direct: it is a way to build production-ready Conversational AI starting from a couple of lines of Python, with authentication, data persistence, multi-step reasoning visualization, and multi-platform deployment as first-class, documented capabilities rather than bolt-on afterthoughts.[4] Under the hood it pairs a FastAPI and python-socketio backend with a prebuilt React frontend that ships inside the Python package itself, so pip install chainlit is genuinely sufficient — there is no separate frontend build step for the common case.[5]

The project was incorporated in April 2023 as Chainlit SAS in Paris, founded by Willy Douhard and Dan Constantini.[6] Its early growth tracked the first wave of retrieval-augmented-generation prototypes: the framework's own cookbook from that period leans heavily on OpenAI, Anthropic, LangChain, LlamaIndex, ChromaDB, and Pinecone, which is to say it was built by and for people wiring together exactly the RAG stacks that were becoming the default LLM application pattern in 2023.[1] By the time of writing, the company's public profile lists it as a small, Paris-headquartered software house with early backing from Kima Ventures and Snaw Ventures.[7]

The mechanical reason this works is worth stating precisely, because it explains almost everything else in this guide. A Chainlit application is a normal Python module. You import chainlit, decorate a handful of functions, and run chainlit run app.py. The server that starts is a real ASGI process; the frontend it serves is a compiled single-page app that talks to that process over a WebSocket for streaming and over REST for everything else.[5] One breakdown of the framework's internals groups what that server quietly handles on the application's behalf into a short list: routing model calls through a simple provider interface, running the WebSocket layer that carries real-time communication between frontend and backend, handling user sessions, conversation history, and application state, and rendering rich content including markdown, code with syntax highlighting, and data visualizations — none of which the application author writes themselves.[2] Because the boundary between your logic and the UI is a set of decorator-registered callbacks rather than an API contract you design yourself, there is very little surface area for the two halves to drift out of sync as the application grows.

The classes of problem it is built to solve

In practice, teams reach for Chainlit for a fairly small number of recurring situations, and it is worth naming them directly rather than leaving the value proposition abstract:

  • Internal knowledge assistants. The ChatGPT for our data request — a chat surface in front of proprietary documents, a database, or an internal API — is the single most common use case, precisely because it used to require an ML engineer, a backend engineer, and a frontend engineer, and now does not.[5]
  • Agent and RAG observability. Chainlit's Step concept renders every intermediate tool call, retrieval, or sub-agent invocation as an expandable trace in the same thread as the final answer, which turns a black-box pipeline into something a reviewer can actually audit turn by turn.
  • Embedded copilots. The same application logic can be shipped as a standalone web app, mounted inside an existing product as a chat bubble via the Copilot widget, or exposed as a bot on Slack, Discord, or Microsoft Teams, without rewriting the underlying hooks.
  • Fast, credible prototypes. Because the interface is generated rather than hand-built, a working, presentable demo of a LangChain or LlamaIndex pipeline is minutes away rather than a sprint away — useful for stakeholder reviews, hackathons, and early customer validation before anyone commits frontend engineering budget.[3]

It helps to see the trade concretely, layer by layer, rather than take the time savings on faith. One deep-dive lays the old and new stacks side by side:

LayerTraditional stackChainlit equivalent
UIReact + Tailwind, hand-builtAuto-generated React chat UI

None of this makes Chainlit a general-purpose web framework, and it is not trying to be one. It is opinionated about exactly one interaction shape — a threaded conversation, possibly with rich attachments and tool traces — and it trades away the flexibility of a blank-canvas frontend for the speed of never having to build that shape from scratch. Section 5 returns to what that trade-off costs against the alternatives; the rest of this section and the next look at how the model actually works in code.

02

The decorator model, in practice

Chainlit's entire public surface is a set of lifecycle hooks — plain async functions registered with a decorator — plus a small vocabulary of objects those hooks send back to the client. Learning the framework is mostly a matter of learning which hook fires when, and which object to construct in response. The full taxonomy the documentation exposes covers session lifecycle hooks, multimodal hooks, and platform hooks, alongside message, step, element, action, chat-profile, chat-settings, and input-widget types.[8]

Getting to the point of having something running takes three commands and no configuration: install the package, run its bundled demo to confirm the toolchain works, then point the same command at a real module with hot reload turned on so edits show up without a manual restart.[1]

terminalbash
1pip install chainlit
2chainlit hello
3
4# scaffold done -- now run your own module with hot reload
5chainlit run app.py -w

The simplest possible application only needs two of these hooks. on_chat_start fires once when a session opens and is the natural place to greet the user or initialize per-session state; on_message fires on every user turn and is where the actual work happens:

app.pypython
1import chainlit as cl
2
3@cl.on_chat_start
4async def on_chat_start():
5    await cl.Message(content="Welcome. Ask me anything about your data.").send()
6
7@cl.on_message
8async def main(message: cl.Message):
9    thinking = cl.Message(content="Thinking...")
10    await thinking.send()
11
12    answer = await call_my_llm(message.content)
13
14    thinking.content = answer
15    await thinking.update()

Two details in that snippet matter more than they look. First, cl.Message is a real object with a lifecycle: calling .send() renders it, and mutating .content followed by .update() edits the same bubble in place rather than appending a new one — which is how a Thinking… placeholder gets swapped for a final answer without a flicker. Second, per-session state travels through cl.user_session, a dict-like store scoped to the current WebSocket connection, so two users hitting the same process never see each other's conversation.

Steps: making the reasoning visible

Most real applications are not a single call to a model — they retrieve documents, call tools, invoke sub-agents, or run several chained prompts before producing a final answer. Chainlit's Step primitive exists specifically to make that intermediate work visible without cluttering the final response. Wrapping a function in @cl.step(type="tool") turns every call of that function into a collapsible trace, timed and nested correctly, that sits above the assistant's reply in the same thread:

app.pypython
1import chainlit as cl
2
3@cl.step(type="tool")
4async def search_filings(query: str):
5    # any expensive or side-effectful call: DB, API, subprocess
6    results = await run_search(query)
7    return results
8
9@cl.on_message
10async def main(message: cl.Message):
11    # the Step renders as a collapsible trace above the final answer
12    findings = await search_filings(message.content)
13    await cl.Message(content=summarize(findings)).send()

This is a small piece of API surface with an outsized effect on trust. A reviewer looking at an agent's output can expand the search_filings step, see the exact query and the exact results the model reasoned over, and judge the final answer accordingly — the same pattern shows up repeatedly in write-ups of Chainlit-based internal tools as the reason business stakeholders were willing to sign off on an LLM-backed workflow they could not otherwise inspect.

Streaming

Because the backend is already an async process talking to the frontend over a WebSocket, token-by-token streaming is a matter of calling stream_token in a loop rather than building a Server-Sent-Events endpoint by hand:

app.pypython
1import chainlit as cl
2from openai import AsyncOpenAI
3
4client = AsyncOpenAI()
5
6@cl.on_message
7async def main(message: cl.Message):
8    reply = cl.Message(content="")
9    await reply.send()
10
11    stream = await client.chat.completions.create(
12        model="gpt-4-turbo",
13        messages=[{"role": "user", "content": message.content}],
14        stream=True,
15    )
16    async for chunk in stream:
17        token = chunk.choices[0].delta.content or ""
18        await reply.stream_token(token)
19
20    await reply.update()

Elements, actions, and structured input

Beyond text, a message can carry elements — images, PDFs, audio, video, plain files, Plotly and Matplotlib figures, and both pandas and polars dataframes render natively, alongside a dedicated TaskList element for showing multi-step progress. Actions attach clickable buttons to a message that route back into a Python callback, which is the usual mechanism for confirmation prompts, follow-up suggestions, or human-in-the-loop approval steps:

app.pypython
1import chainlit as cl
2
3@cl.on_message
4async def main(message: cl.Message):
5    actions = [
6        cl.Action(name="approve", payload={"id": "req-42"}, label="Approve"),
7        cl.Action(name="reject", payload={"id": "req-42"}, label="Reject"),
8    ]
9    await cl.Message(
10        content="This refund exceeds the auto-approval threshold. Proceed?",
11        actions=actions,
12    ).send()
13
14@cl.action_callback("approve")
15async def on_approve(action: cl.Action):
16    await process_refund(action.payload["id"])
17    await cl.Message(content="Refund approved and queued.").send()

That pattern — a message with attached actions, and a decorated callback keyed to the action's name — is the whole mechanism behind human-in-the-loop gates: an agent proposes a refund, a database write, or an email send, and a human clicks approve or reject before anything irreversible happens, with the click routing straight back into ordinary Python rather than a separate approval microservice. For anything that needs structured input rather than a binary decision, the input-widget set covers select boxes, sliders, switches, tagged multi-selects, radio groups, date pickers, and plain text fields, exposed through ChatSettings so a user can tune an application's behavior (model choice, temperature, retrieval depth) from a settings panel rather than a slash command.[8]

Multimodal and tool sessions

The same hook pattern extends to voice, with on_audio_start, on_audio_chunk, and on_audio_end covering streamed microphone input, and to external tool servers through the Model Context Protocol, covered in full in the next section. Both are additive: an application that only implements on_message continues to work exactly as before if audio or MCP hooks are never registered.

One deployment, several personas

A detail that surprises teams coming from single-purpose chatbot builders is that a single Chainlit deployment does not have to mean a single assistant. set_chat_profiles returns a list of named ChatProfile objects — each with its own name, description, icon, and optional starter prompts — and the picker built from that list is the first thing a user sees on the login screen. The same on_message hook then branches on cl.user_session.get("chat_profile") to route to a different model, system prompt, or tool set per profile, which is how one deployment ends up serving, say, a Contracts assistant and a Financial filings assistant behind the same URL and the same authentication layer rather than two separate applications:

app.pypython
1import chainlit as cl
2
3@cl.set_chat_profiles
4async def chat_profiles():
5    return [
6        cl.ChatProfile(
7            name="Contracts",
8            markdown_description="Answers questions over the signed contract corpus.",
9            starters=[
10                cl.Starter(label="Renewal terms", message="What are our Q3 renewal terms?"),
11            ],
12        ),
13        cl.ChatProfile(
14            name="Financial filings",
15            markdown_description="Answers questions over 10-K and 10-Q filings.",
16        ),
17    ]
18
19@cl.on_message
20async def main(message: cl.Message):
21    profile = cl.user_session.get("chat_profile")
22    retriever = retrievers[profile]
23    await cl.Message(content=await answer(retriever, message.content)).send()

Profiles are the coarse-grained switch; ChatSettings (built from the input-widget set) is the fine-grained one, letting a user tune retrieval depth, temperature, or model choice from a panel without either a slash command or a redeploy.

03

What it plugs into

Chainlit deliberately does not ship its own orchestration layer, model client, or vector store — it assumes you already have, or will choose, those pieces yourself, and it stays out of the way. The app.py calls out through plain Python arrow in Figure 1 is the important part: nothing about the framework requires a particular SDK. That said, the documentation maintains first-class, worked integrations for the orchestration and model layers most teams reach for — LangChain and LangGraph, LlamaIndex, Semantic Kernel, Autogen, and Embedchain on the orchestration side, and OpenAI, Mistral AI, vLLM, LM Studio, Hugging Face endpoints, and the LiteLLM proxy on the model-serving side.[8] The framework's own cookbook has, since its earliest releases, demonstrated the same pattern against OpenAI, Anthropic, ChromaDB, and Pinecone, which is a fair proxy for the RAG stack it was originally built to sit in front of.[1]

The orchestration integrations are not just import compatibility — the documented pattern for LangChain and LlamaIndex specifically is a callback handler that Chainlit provides and the orchestration library calls into as it runs, so every chain step, retriever call, and agent action is reported back to the UI as a Step without the application author writing any bridging code themselves.[8] That is a meaningful design choice: it means adopting Chainlit does not require restructuring an existing LangChain or LlamaIndex pipeline, only attaching a callback handler to whatever chain or query engine already exists.

Tools, over the Model Context Protocol

Since MCP support landed, Chainlit apps can attach to external tool servers at runtime rather than only at build time: a user clicks a plug icon in the composer, supplies a connection command or URL, and any tools that server exposes become available to the model for the rest of that session.[9] The framework supports all three MCP transports — a local stdio subprocess, remote Server-Sent Events, and remote streamable HTTP — and surfaces the connection lifecycle through two more hooks:

app.pypython
1import chainlit as cl
2from mcp import ClientSession
3
4@cl.on_mcp_connect
5async def on_mcp_connect(connection, session: ClientSession):
6    result = await session.list_tools()
7    tools = [{"name": t.name, "description": t.description} for t in result.tools]
8    mcp_tools = cl.user_session.get("mcp_tools", {})
9    mcp_tools[connection.name] = tools
10    cl.user_session.set("mcp_tools", mcp_tools)
11
12@cl.on_mcp_disconnect
13async def on_mcp_disconnect(name: str, session: ClientSession):
14    # release any per-connection state here
15    pass

The security-relevant detail sits in configuration, not code. As of version 2.7.0, MCP support is opt-in: the mere presence of an on_mcp_connect handler no longer silently enables the feature, and stdio connections must name an explicit allow-list of executables before a user can launch one from the browser.[10] Left at its defaults, an application with MCP enabled but no allow-list would let any connected user spawn arbitrary local processes on the server — the allow-list exists precisely to close that off:

config.tomltoml
1[features.mcp]
2enabled = true
3
4[features.mcp.stdio]
5enabled = true
6# only these executables may be launched as local MCP servers
7allowed_executables = ["npx", "uvx"]
8
9[features.mcp.sse]
10enabled = true
11
12[features.mcp.streamable-http]
13enabled = true

A recent comparison of MCP-capable chat front ends is a useful way to place Chainlit among its nearest neighbors here: against ready-to-run applications like LibreChat, AnythingLLM, and Open WebUI, the same review concluded that Chainlit offers the most developer control for building custom, code-first experiences, at the cost of requiring Python to do anything at all, where the other three are usable out of the box.[11] Section 5 develops that trade-off in more depth.

Persistence: data layers and storage providers

By default a Chainlit app remembers nothing once a process restarts. Production deployments attach a data layer to persist threads, steps, elements, and feedback, most commonly through SQLAlchemyDataLayer, which was built against Postgres but works with any database SQLAlchemy supports given a connection string, plus a storage client for large binary elements on S3, Azure Blob, or Google Cloud Storage.[12]

app.pypython
1import chainlit as cl
2from chainlit.data.sql_alchemy import SQLAlchemyDataLayer
3
4@cl.data_layer
5def get_data_layer():
6    return SQLAlchemyDataLayer(
7        conninfo="postgresql+asyncpg://user:pass@host/db"
8    )

The data-layer story also carries the clearest signal of how the project's relationship with its original commercial sponsor has changed. Chainlit's own hosted observability platform, Literal AI, has been sunset as a product the framework depends on: the changelog now states plainly that LiteralAI is being sunset and will be removed in one of the next releases, directing users to the official data layer instead, and telemetry that used to be on by default is opt-in going forward.[13] Anyone standardizing a data layer today should treat the SQLAlchemy or DynamoDB-backed options, not a Literal AI integration, as the durable choice.

Identity: authentication providers

Chainlit applications are public by default, which is worth repeating because it is easy to miss: turning on authentication is an explicit step, not the out-of-the-box behavior.[14] Once a CHAINLIT_AUTH_SECRET is set, an application can authenticate users through three mechanisms, and more than one can be active at once. OAuth covers the largest surface, with prebuilt providers for Google, GitHub, Azure AD (multi- or single-tenant), Okta, Auth0, and AWS Cognito, plus a generic OpenID Connect provider for anything else, each translating a provider-specific token exchange into the same internal User object.[15] Password authentication runs through a developer-supplied callback against whatever user store already exists, and header-based authentication delegates the actual check to an upstream reverse proxy — Cloudflare Access or an AWS Application Load Balancer are the examples the documentation gives — and simply trusts a signed header it receives.[16] Successful logins of any kind are issued as an HS256 JWT stored in an HTTP-only cookie, chunked across multiple cookies automatically if the token exceeds roughly 3,000 characters, since browsers cap individual cookies well below that.[16]

app.pypython
1import chainlit as cl
2
3@cl.password_auth_callback
4def auth_callback(username: str, password: str):
5    if verify_credentials(username, password):
6        return cl.User(identifier=username, metadata={"role": "analyst"})
7    return None
8
9@cl.header_auth_callback
10def header_auth_callback(headers) -> cl.User | None:
11    # trust an upstream proxy (Cloudflare Access, AWS ALB) that has
12    # already authenticated the request
13    token = headers.get("x-user-jwt")
14    return verify_upstream_token(token)

Channels: shipping the same app more than one way

The last integration surface is distribution. A single app.py can be run as a standalone web app, mounted onto an existing FastAPI service as an ASGI sub-application via mount_chainlit[17], embedded into another site as a floating chat bubble through the Copilot widget, or deployed as a bot on Slack, Discord, or Microsoft Teams, generally without touching the underlying hooks. Slack in particular ships two distinct integration modes: a websocket-based Socket Mode that needs no public-facing endpoint at all, and a webhook-based HTTP Mode for teams that would rather terminate the connection at their own edge; both dispatch into the same on_message hook an application already has.[10]

app.pypython
1from fastapi import FastAPI
2from chainlit.utils import mount_chainlit
3
4app = FastAPI()
5
6@app.get("/health")
7def health_check():
8    return {"status": "ok"}
9
10# mounts the Chainlit app as an ASGI sub-application on /chainlit
11mount_chainlit(app=app, target="my_cl_app.py", path="/chainlit")
Field note

The Copilot widget is the integration path most likely to surprise a team the first time they use it. Because the widget must run inside a third-party page's origin, the usual password_auth_callback flow does not carry over cleanly, and one practitioner walkthrough is candid that passing an authenticated user's identity from the host page into the embedded Chainlit session is the hardest part of the whole setup, describing it as the point where you tend to lose control over the user experience.[18] Plan for header-based or token-based authentication at the embed boundary rather than assuming the standard login screen will simply appear inside the widget.

04

Running it in production

Getting a Chainlit app onto a colleague's screen and getting it onto a production endpoint that survives autoscaling, load balancing, and an audit are different problems, and the gap between them is mostly ordinary web-operations discipline rather than anything specific to LLMs. The official deployment guidance is narrow and worth following exactly: always pass -h in production so the CLI does not try to open a local browser window server-side, bind to 0.0.0.0 rather than the development default of 127.0.0.1, and set --root-path if the app is served from a subpath.[19]

The one piece of infrastructure guidance that is easy to skip and expensive to skip is about WebSockets. Because Chainlit's UI depends on a persistent socket connection, any autoscaled deployment must enable sticky sessions (session affinity) at the load balancer, and even that is not always sufficient — the documentation notes that load balancers can still struggle to consistently route a client back to the same container, and recommends pinning the transport to websocket only in config.toml when that happens.[19] Teams that skip this step tend to discover it the same way: intermittent could not reach the server errors that only appear under real concurrent load, not in a single-user smoke test.

Containers and cloud targets

The framework's own cookbook ships a minimal, working pattern for AWS ECS — a slim Python base image, the app copied in, dependencies installed, and a container command of chainlit run app.py -h --host 0.0.0.0 --port 8080 — and the same Dockerfile shape ports cleanly to Cloud Run, Kubernetes, or any other container platform with minimal edits:[20]

Dockerfiledocker
1FROM python:3.10-slim
2# Apple Silicon build hosts: FROM --platform=linux/amd64 python:3.10-slim
3
4ENV PYTHONUNBUFFERED=1
5WORKDIR /app
6
7COPY requirements.txt /app/
8RUN pip install -r requirements.txt
9COPY . /app/
10
11EXPOSE 8080
12CMD ["chainlit", "run", "app.py", "-h", "--host", "0.0.0.0", "--port", "8080"]

For teams that want a managed target rather than owning the container platform themselves, Ploomber Cloud is a documented option that layers Auth0-based authentication and usage analytics on top of a standard Chainlit container, requiring only the app's existing CHAINLIT_AUTH_SECRET and a handful of provider secrets.[21]

The most instructive production reference, though, is a full enterprise architecture rather than a minimal container: AWS's own foundational-llm-chat sample deploys a Chainlit application through the CDK with Amazon Cognito for identity, Amazon Bedrock as the model backend, ECS across multiple availability zones for compute, and CloudFront in front of it for global distribution, with MCP connections deliberately kept in per-session memory rather than any shared server-side store so that one user's connected tools are never visible to another.[22] It is a useful checklist precisely because every piece of it is answering a question the minimal Docker Compose tutorial glosses over: who is allowed in, where does the model actually run, and how does session isolation hold up once MCP tool connections are part of the picture.

That last question is worth answering explicitly rather than assuming. In the AWS reference implementation, each user's connected MCP servers live only in that user's own in-memory session, with an explicit design note that connections are NOT stored in ECS containers, DynamoDB, or S3, specifically so that User A cannot access User B's MCP tools — complete separation, and each session carries its own multi-day timeout independent of every other user's.[22] A team building its own data layer on top of Chainlit's defaults should treat that same guarantee — one user's tool connections, credentials, and thread history are never reachable from another user's session — as a hard requirement to re-verify against its own storage design, not an assumption inherited for free.

Governance: what "community-maintained" means for a production dependency

Since 1 May 2025, Chainlit has run under a different governance model than the one most of its adopters signed up for. The original team stepped back from active development; a group of maintainers now handles code review, releases, and security under a formal Maintainer Agreement, and the project's own README is explicit that Chainlit SAS provides no warranties on future updates.[1] The commercial product built by the same founders, Literal AI, has wound down in stages since: its self-hosted enterprise Docker image was discontinued on 31 October 2025, and its hosted cloud service — dashboards, online evaluation, A/B testing, and the prompt playground — is no longer available, leaving the open-source data layer as the only surviving piece.[23] The founders themselves have since moved on to a new venture, described as an effort to transform any API into an MCP or A2A agent, which is a fair indication of where they now see the more interesting problem.[24]

None of this has stopped releases: v2.11.1 shipped in late April 2026, continuing a steady cadence of point releases through the community-maintenance period.[1] But it is a materially different risk profile than a venture-backed company with a support SLA, and it belongs in the same architecture-review conversation as the choice of model provider or cloud region: patches depend on volunteer maintainer bandwidth, and there is no vendor to escalate to if a critical bug lands on a Friday.

Security track record

That governance shift matters more once the framework's actual vulnerability history is on the table. Three disclosed, high-severity issues are relevant to anyone running Chainlit in front of real users or real cloud credentials.

CVE-2025-68492 — authorization bypass

Versions before 2.8.5 failed to verify that a user requesting a conversation thread actually owned it: an authenticated attacker who could guess or enumerate another user's thread identifier could view that thread's contents or transfer ownership of it, a textbook insecure-direct-object-reference flaw (CWE-639).[25]

CVE-2026-22218 — arbitrary file read (CVSS 7.7)

Versions before 2.9.4 let an authenticated client attach a custom element with a user-controlled file path; the server would copy the referenced file into the attacker's own session, generating an element identifier that could then be handed straight back to the standard file-retrieval endpoint to download the contents — disclosing configuration files, database credentials, or source code to anyone who could reach the app and log in.[26]

CVE-2026-22219 — server-side request forgery (CVSS 7.0)

The same custom-element mechanism, exploited through its URL field instead of its path field, forced the server to fetch an attacker-chosen URL. On an AWS EC2 instance still running the original Instance Metadata Service (IMDSv1), that SSRF was enough to reach the metadata endpoint, pull the instance's IAM role credentials, and move laterally inside the AWS account — instances that had already been upgraded to IMDSv2 were not vulnerable to that specific escalation.[27]

Security firm Zafran, which disclosed the file-read and SSRF pair, found both in the course of surveying publicly reachable Chainlit servers and reported turning up live deployments tied to enterprises and academic institutions — not just personal projects — while Chainlit itself was seeing roughly 700,000 downloads a month, or on the order of five million a year, from PyPI at the time.[28] Both issues were fixed in version 2.9.4, released 24 December 2025, ahead of the public advisory in January 2026.[29] Coverage of the disclosure made a point that generalizes well beyond this one framework: the flaws were not exotic prompt-injection or model-alignment failures, but familiar server-side problems of the kind web applications have dealt with for two decades, arriving with unusually high stakes because the vulnerable process typically sits next to cloud credentials, internal APIs, and a data layer full of conversation history.[28]

A short enterprise checklist

  • Pin an exact Chainlit version in your lockfile and track releases deliberately; do not float on latest given the maintenance model described above.
  • Turn on authentication explicitly — the framework will not do it for you — and prefer delegating to an existing identity provider (OAuth or a header-auth proxy) over a hand-rolled password store.
  • If you deploy on AWS EC2 rather than a managed container service, confirm IMDSv2 is enforced regardless of Chainlit's patch level; it closes off an entire class of SSRF-to-credential-theft chains, not just this one.
  • Treat any custom Element-handling code path as untrusted input, since both 2026 CVEs trace back to the same feature; review anything your own code adds on top of it with the same scrutiny.
  • Budget for the data layer and object storage from the start — retrofitting persistence onto an app that has been running stateless is a bigger migration than it looks.

Where it sits in a crowded field

Chainlit shares shelf space with three genuinely different kinds of tool, and conflating them is the most common way to end up disappointed with whichever one gets picked. There are general-purpose Python app frameworks that happen to be usable for chat; there are ready-to-run chat applications that need no Python at all; and there are enterprise conversational-AI platforms built around visual flow design rather than code. Chainlit's actual competition is only the first category — but it is worth placing against all three.

Streamlit, Gradio, and the rest of the Python app-framework family

Streamlit and Gradio predate Chainlit and solve an adjacent but different problem: both let a Python script become a shareable web app, and both have since added chat-shaped components on top of a fundamentally different execution model. A detailed side-by-side of the three summarizes the practical difference well: Chainlit has features that Streamlit and Gradio have to bolt on awkwardly: native streaming, message threading, step-by-step agent reasoning display, user feedback collection, authentication, persistent conversation history, because Streamlit and Gradio were designed first for data apps and model demos respectively, and picked up chat support afterward, whereas Chainlit started from the chat thread as the unit of design.[30] The same comparison is candid about the cost of that focus: Chainlit's component library is smaller than Streamlit's, it has no native charts, dataframes-as-widgets, or maps, and a non-chat UI — a dashboard, a multi-page admin tool — is an awkward fit for it in a way it is not for Streamlit.[30]

A broader survey of UI frameworks for AI agents, scored across setup speed, streaming quality, multimodal support, customization, and production readiness, put Chainlit and the Vercel AI SDK ahead of Streamlit and Gradio specifically on the production-readiness axis, crediting Chainlit's built-in authentication and observability, while ranking Streamlit highest for teams whose agent needs to sit alongside data visualizations and Gradio highest for throwaway demos and teaching.[31] Put simply: choose Streamlit if the product is a data app that happens to talk back; choose Gradio if the job is a five-minute shareable demo of a model, keeping in mind that its state management stays noticeably more primitive than either Streamlit's or Chainlit's once an interaction becomes multi-step rather than a single inference call[30]; choose Chainlit if the product is the conversation itself. Panel, Reflex, and Mesop round out this cluster as smaller, more general-purpose alternatives to Streamlit rather than direct Chainlit competitors.

The Vercel AI SDK: the same idea, a different stack

The closest philosophical peer to Chainlit is not a Python tool at all. The Vercel AI SDK targets the same conversational-AI-specific niche — streaming-first, built for chat and agent UIs rather than general apps — but from React and Next.js, using shadcn/ui-based components (branded as AI Elements) for teams that want pixel-level control over the interface and are willing to own a JavaScript frontend to get it.[31] The honest framing is a stack decision more than a quality judgment: teams already committed to Python and wanting a UI without owning frontend code reach for Chainlit; teams already committed to React and wanting first-class design-system control reach for the Vercel AI SDK.

Ready-to-run chat applications: Open WebUI, LibreChat, AnythingLLM

A second category needs no code at all for the common case. Open WebUI, LibreChat, and AnythingLLM are complete, self-hostable ChatGPT-style applications: install, point them at a model or provider, and there is a working multi-user chat product with no app.py to write. A comparison of how all four handle MCP tool connections draws the line precisely: LibreChat is positioned as a leading enterprise-focused open-source ChatGPT alternative… among the first platforms to fully implement Model Context Protocol server support, AnythingLLM leans toward guided, document-centric RAG setups, and Open WebUI trades native MCP support for adaptability through a compatibility shim — while Chainlit alone requires Python programming to create custom chat applications, the deliberate trade for going from ready to use to fully programmable.[11] If the requirement is genuinely stand up a ChatGPT-like tool for the company, one of these three will get there faster than writing a single line of Chainlit. If the requirement is build a specific product with custom logic embedded in the conversation, that programmability is the entire point.

Enterprise conversational-AI platforms: Botpress, Rasa, and the visual-builder tier

The third category is the one Chainlit is least likely to be compared against in practice, but it is where most conversational-AI budget in large enterprises actually goes: visual-builder platforms such as Botpress, Rasa, Microsoft's Bot Framework and Copilot Studio, Google's Dialogflow CX, Voiceflow, and Kore.ai. These tools trade code for either a drag-and-drop flow designer (Botpress, Copilot Studio, Dialogflow) or a structured dialogue-management layer with its own DSL (Rasa's stories and, more recently, its CALM framework), and they bundle concerns — channel deployment, analytics, human handoff, compliance certifications — that a Chainlit application would otherwise have to assemble from separate pieces.[32] Rasa in particular is the closest thing to a code-first peer in this tier: it is self-hosted from the start, Apache-licensed at its core, and aimed at teams with strict data-residency requirements, but it is built around dialogue management and intent recognition as the primary abstraction rather than around rendering a chat UI, which makes it a poor substitute for Chainlit's actual job even though the two are sometimes mentioned in the same breath.[33] Botpress, for its part, no longer offers a self-hostable open-source distribution at all — the legacy v12 line is sunset and Botpress Cloud is the only supported path for new development, which is precisely the dependency that a self-hosted framework like Chainlit is chosen to avoid.[32]

Field note — exit cost

The governance risk described in Section 4 is real, but it is bounded by the license underneath it. Because Chainlit is Apache 2.0 and the React frontend ships prebuilt inside the same repository, a team that concludes the maintainer group has gone quiet is not locked in the way it would be with a closed-source vendor: forking the project, or simply freezing on a known-good version while budgeting the occasional security patch in-house, is a legitimate and unglamorous contingency plan. That is a materially different position than being stuck on an abandoned SaaS platform, and it is worth weighing against the governance concern rather than letting the concern stand alone.

None of this makes Chainlit strictly better or worse than its neighbors — it makes it narrower. It gives up the general-purpose flexibility of Streamlit, the zero-code convenience of Open WebUI, and the compliance tooling of an enterprise platform like Rasa or Botpress, in exchange for being the fastest path from a Python function to a production-shaped conversation, with the reasoning behind that conversation visible the whole way through. For the specific job of turning custom LLM logic into a UI without a frontend team, that remains a genuinely useful trade, and it is the reason the framework has kept a substantial installed base through a founder transition, a governance change, and two disclosed CVEs that would have sunk a less differentiated tool.[1]

The state of things, July 2026

Chainlit is, at the same time, a maturing piece of infrastructure and an orphaned one. The releases keep coming, the integration surface keeps growing — MCP support, an official data layer, an increasingly capable Copilot widget — and the installed base is large enough that a single framework's file-handling bug became a multi-outlet security story rather than a footnote.[1] At the same time, it is no longer backed by a company with a commercial incentive to keep it healthy; it is backed by whoever shows up to review pull requests under a Maintainer Agreement. Both things are true, and a production decision should weigh both rather than either one alone.

What that tension actually looks like day to day is unglamorous: a maintainer group merging dependency bumps and security fixes competently, a founder team that has moved on to the next problem, and a user base large enough that its mistakes make the trade press. None of that is unique to Chainlit — it is close to the median outcome for infrastructure that starts as a venture-backed product and ends as community-run open source — but it is worth naming plainly rather than folding into the usual actively maintained line on a landing page. A team adopting it today is adopting a tool, not a vendor relationship, and the operational habits in Section 4 (pinned versions, explicit authentication, IMDSv2, treating custom Element handling as untrusted input) are the price of that trade rather than optional hardening.

For the narrow job it was built for — turning a working piece of Python LLM logic into a chat interface a real user can sit down in front of, with the reasoning behind each answer visible and auditable — it remains, as of this writing, one of the fastest paths available. Whether that job is the one your project actually has is the question Section 5 is meant to help answer.
