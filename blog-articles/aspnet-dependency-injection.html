ASP.NET Core Dependency Injection: Lifetimes, Scrutor, and Beyond

Published: 28 May 2026

1. Introduction

Dependency injection (DI) is a first-class citizen in ASP.NET Core, embedded into the framework from its inception. The built‑in container – Microsoft.Extensions.DependencyInjection – provides a lightweight, performant, and opinionated way to manage service lifetimes and composition[1]. While this default container covers the majority of scenarios, it intentionally omits features such as automatic assembly scanning, interception, and decoration. The open-source library Scrutor fills these gaps, extending the built‑in container without replacing it[2].

This article explores the three core service lifetimes (transient, scoped, singleton) in depth, details the history and capabilities of Scrutor, and demonstrates practical examples integrating both the standard container and Scrutor’s extensions. All facts are referenced using the Vancouver citation style.

2. Dependency Injection in ASP.NET Core: A Brief History

Before ASP.NET Core, DI in .NET web applications relied on third‑party containers such as Unity, Autofac, or Ninject. With the release of ASP.NET 5 (later renamed ASP.NET Core 1.0) in June 2016, Microsoft introduced a built‑in DI abstraction in the Microsoft.Extensions.DependencyInjection namespace[1]. The design was intentionally minimal:

  • An IServiceCollection to register services.
  • An IServiceProvider as the composition root.
  • Three clear lifetimes: Transient, Scoped, Singleton.
  • Support for open generics and factory-based registration.

Crucially, the DI container remains replaceable; developers can swap in Autofac, Lamar, or others. However, many projects prefer to stay on the default stack and augment it with scanning and decoration – the exact purpose that Scrutor was built to serve[2].

Scrutor was created by Kristian Hellang and first released on NuGet in August 2016. The name derives from the Latin scrūtārī, meaning “to search” or “to examine thoroughly” – a nod to its assembly scanning capabilities[3]. As of version 5.0 (2024), it supports .NET 6 through .NET 9 and introduces interception helpers that work with the built‑in container.

3. Built‑in Service Lifetimes

Understanding lifetimes is essential for correct resource management and thread safety. ASP.NET Core defines three[1,4]:

LifetimeCreatedDisposedTypical Use
Transient Every time they are requested from the container. When the service scope ends (the consumer does not dispose them). Lightweight, stateless services (e.g., IEmailSender, IValidator).
Scoped Once per scope (in ASP.NET Core, one scope per HTTP request). At the end of the scope (request). Per‑request data, unit‑of‑work (e.g., DbContext, IUserContext).
Singleton Once per application lifetime (root container). When the application shuts down. Stateless, thread‑safe, expensive‑to‑create services (e.g., cache, configuration).

3.1 Registration Examples

// Standard Microsoft.Extensions.DependencyInjection
builder.Services.AddTransient<IEmailService, SmtpEmailService>();
builder.Services.AddScoped<IOrderRepository, SqlOrderRepository>();
builder.Services.AddSingleton<ICacheProvider, RedisCacheProvider>();

3.2 Lifetime Traps and Best Practices

  • Captive dependencies: Never inject a scoped service into a singleton; the scoped instance will be kept alive beyond its intended scope, leading to disposed‑object exceptions or stale data[5].
  • Transient inside singleton: Allowed, but each transient instance becomes a permanent child of the singleton (no re‑creation per request).
  • Validation: In development mode, ASP.NET Core validates scopes to catch captive dependencies early. Enable with Host.CreateDefaultBuilder default settings or ValidateScopes = true.

4. The Role of Scrutor: Extending Microsoft.Extensions.DependencyInjection

4.1 What is Scrutor?

Scrutor is not a replacement DI container. It sits on top of Microsoft.Extensions.DependencyInjection and provides extension methods for IServiceCollection to automate registration and apply decoration patterns[2,3]. Key features include:

  • Assembly scanning for convention‑based registration.
  • Service decoration (wrapper/decorator pattern).
  • Open‑generic decoration.
  • Lifetime control per discovered type.
  • Interception support (via Scrutor.Intercept).

Because Scrutor operates directly on IServiceCollection, it integrates seamlessly with any application that uses the default container, including ASP.NET Core, Worker Services, and Blazor[2].

4.2 Scanning for Convention‑Based Registration

Manual registration of dozens of services becomes tedious. Scrutor’s Scan() method allows you to select assemblies, filter types, and specify lifetime and interface mapping fluently. 

// Assembly scanning: register all classes that implement IMyService
services.Scan(scan => scan
    .FromAssemblyOf<IMyService>()               // Scan the assembly containing IMyService
    .AddClasses(classes => classes.AssignableTo<IMyService>()) // Filter
    .AsImplementedInterfaces()                  // Register as their implemented interfaces
    .WithScopedLifetime());                     // Set lifetime

// Multiple rules in one scan
services.Scan(scan => scan
    .FromAssemblies(Assembly.GetExecutingAssembly())
        .AddClasses(c => c.Where(t => t.Name.EndsWith("Service")))
            .AsSelf()
            .WithTransientLifetime()
        .AddClasses(c => c.AssignableTo<IValidator>())
            .AsImplementedInterfaces()
            .WithSingletonLifetime());

Scrutor can also use attributes like [ServiceDescriptor] (from Microsoft.Extensions.DependencyInjection, not built-in but often user-defined) or its own [Service] marker attribute when combined with .UsingAttributes().

4.3 Decoration and Interception

The decorator pattern is a powerful way to add cross‑cutting concerns (logging, caching, validation) without modifying original services. Scrutor provides Decorate() and Intercept() methods. 

// Decorate all instances of IRepository<T> with CachedRepository<T>
services.Decorate(typeof(IRepository<>), typeof(CachedRepository<>));

// Keyed decoration (since .NET 8 with keyed services)
services.Decorate<IHandler, LoggingHandler>(serviceKey: "default");

// Interception using a proxy (requires Scrutor.Intercept and Castle.Core)
services.Scan(scan => scan
    .FromAssemblyOf<IMyService>()
    .AddClasses()
    .UsingRegistrationStrategy(RegistrationStrategy.Skip)
    .AsImplementedInterfaces()
    .WithScopedLifetime());
services.Intercept<IMyService, LoggingInterceptor>();

The Decorate method rewrites the ServiceDescriptor so that the original implementation is injected into the decorator’s constructor, while the decorator itself is registered as the service[6]. Interception works similarly but uses a proxy generator (Castle DynamicProxy) under the hood.

4.4 Advanced Features

  • Conditional decoration: TryDecorate() only applies if the service exists.
  • Service lifetime overriding: Combine scanning with .WithLifetime() to enforce a specific lifetime for all matched types.
  • Open‑generic support: Both scanning and decoration handle generic types naturally.
  • Registration strategies: RegistrationStrategy.Skip, .Replace, .Append, etc., control what happens when a service is already registered.

5. Practical Examples

5.1 Basic Registration via Scanning

Consider a typical clean architecture project with services, repositories, and validators. Register them all in a few lines:

using Scrutor;

var builder = WebApplication.CreateBuilder(args);
builder.Services.Scan(scan => scan
    .FromApplicationDependencies()   // all assemblies referenced by app
        .AddClasses(c => c.AssignableTo<IUseCase>())
            .AsSelf()
            .WithScopedLifetime()
        .AddClasses(c => c.AssignableTo<IRepository>())
            .AsImplementedInterfaces()
            .WithScopedLifetime()
        .AddClasses(c => c.InNamespaces("MyApp.Validators"))
            .AsImplementedInterfaces()
            .WithTransientLifetime());

5.2 Implementing the Decorator Pattern

Wrap every IOrderService with a logging decorator that records execution time.

// Original service
public class OrderService : IOrderService { /* ... */ }

// Decorator
public class LoggingOrderService : IOrderService
{
    private readonly IOrderService _inner;
    private readonly ILogger<LoggingOrderService> _logger;
    public LoggingOrderService(IOrderService inner, ILogger<LoggingOrderService> logger)
    {
        _inner = inner;
        _logger = logger;
    }
    public async Task ProcessAsync(Order order)
    {
        var sw = Stopwatch.StartNew();
        await _inner.ProcessAsync(order);
        _logger.LogInformation("Order processed in {Elapsed} ms", sw.ElapsedMilliseconds);
    }
}

// Registration: register original, then decorate
builder.Services.AddScoped<IOrderService, OrderService>();
builder.Services.Decorate<IOrderService, LoggingOrderService>();

5.3 Full Web API Integration

A typical ASP.NET Core Web API using both built‑in DI and Scrutor. The following example scans the assembly, decorates the repository, and configures DbContext with a scoped lifetime. 

using Microsoft.EntityFrameworkCore;
using Scrutor;

var builder = WebApplication.CreateBuilder(args);
builder.Services.AddDbContext<AppDbContext>(options =>
    options.UseSqlServer(builder.Configuration.GetConnectionString("Default")),
    ServiceLifetime.Scoped);

// Scrutor scan for services and repositories
builder.Services.Scan(scan => scan
    .FromAssemblies(typeof(Program).Assembly)
        .AddClasses(c => c.AssignableTo<IRepository>())
            .AsImplementedInterfaces()
            .WithScopedLifetime()
        .AddClasses(c => c.AssignableTo<IService>())
            .AsSelf()
            .WithTransientLifetime());

// Add a caching decorator for all IRepository<T>
builder.Services.Decorate(typeof(IRepository<>), typeof(CachedRepository<>));

// Controllers are already registered by AddControllers()
builder.Services.AddControllers();

var app = builder.Build();
app.MapControllers();
app.Run();

6. Integration with Other Tools and Frameworks

Scrutor does not replace the container; it works alongside any library that uses IServiceCollection. This makes it compatible with:

  • Entity Framework Core: AddDbContext and repository scanning.
  • MediatR / FluentValidation: Often combined to auto‑register handlers and validators.
  • Refit / HttpClientFactory: Register typed clients and decorate them for resilience.
  • ASP.NET Core Minimal APIs and Blazor: Same DI composition root.

Moreover, Scrutor can be used in any .NET host (console, worker, MAUI) because it only needs Microsoft.Extensions.DependencyInjection[2]. It does not interfere with third‑party containers if you decide to swap later; you simply stop using Scrutor extensions.

7. Source Code Insights

Scrutor’s implementation is a masterclass in fluent API design over IServiceCollection. The core types reside in the Scrutor namespace:

  • ServiceCollectionExtensions: Entry points (Scan, Decorate, Intercept).
  • ITypeSelector / IImplementationTypeSelector: Fluent interfaces for filtering classes.
  • DecorationStrategy / RegistrationStrategy: Control how descriptors are modified.

When you call Scan(), it builds an internal list of ServiceDescriptor objects and then calls IServiceCollection.Add() for each – exactly what you would do manually, but automated. Decoration re‑wires the dependency graph by replacing the original descriptor with a factory that constructs the decorator, passing the original as a dependency[6].

The library is distributed under the MIT license, and the source code is available on GitHub[3]. Development remains active, with community contributions adding support for newer .NET features like keyed services and interception.

8. Conclusion

The built‑in DI container in ASP.NET Core provides a solid foundation with its three lifetimes – transient, scoped, and singleton. Scrutor elegantly extends this foundation with assembly scanning, decoration, and interception, all while preserving the container’s simplicity and replaceability. By understanding these tools and their proper usage, developers can keep their composition roots clean, reduce boilerplate, and apply cross‑cutting concerns without heavy‑weight alternatives. The combination of Microsoft’s DI abstractions and Scrutor’s fluent extensions has become a de facto standard in many modern .NET applications.
