1. The animation mental model

Angular animation work now has two overlapping models. The first is the modern model: write animation behavior in CSS and let Angular attach or remove animation classes at the moment an element enters or leaves the DOM. Angular exposes this model through animate.enter and animate.leave, which are special compiler-supported animation APIs rather than ordinary directives [1]. The second model is the older @angular/animations package, a TypeScript DSL built around trigger(), state(), transition(), style(), animate(), query(), stagger(), group(), sequence(), animation(), and useAnimation() [2,3,4].

The distinction matters because the Angular team documents @angular/animations as deprecated as of Angular v20.2 and recommends native CSS with animate.enter and animate.leave for new code [5]. However, many production Angular applications still contain the legacy DSL, and the DSL remains the clearest way to explain Angular-specific concepts such as animation triggers, wildcard states, void, :enter, :leave, query-driven child animation, and reusable parameterized animation factories [2,3,4].

Practical recommendation: for a new Angular application, prefer CSS animations, animate.enter, animate.leave, and Router View Transitions. For an existing application already using @angular/animations, understand the DSL well enough to maintain it safely and migrate incrementally.

2. Modern Angular animations vs. the legacy DSL

Concern Modern Angular approach Legacy @angular/animations approach
Element enters DOM animate.enter="class-name" applies CSS classes while the animation is active [1]. transition(':enter', ...), alias for void => * [2].
Element leaves DOM animate.leave="class-name"; Angular removes the element after the longest animation or transition completes [1]. transition(':leave', ...), alias for * => void [2].
Stateful component transition Toggle CSS classes with template bindings or signals [5]. trigger(), state(), and transition() [2].
List sequencing CSS delays, custom properties, or JavaScript animation libraries. query() plus stagger() [3].
Route transition Angular Router integration with the browser View Transitions API [6]. Route-state-driven trigger attached around <router-outlet>.
Reusable animation Shared CSS keyframes, classes, CSS custom properties, and design tokens [5]. animation() and useAnimation() with runtime parameters [4].

The rest of this article teaches both models. The examples use standalone components and modern Angular template control flow where possible, but the DSL examples are still shown because triggers, transitions, complex sequences, and reusable animations are established terms from the legacy animation package [2,3,4].

3. Transitions and triggers

In the legacy DSL, a trigger is a named animation definition that is attached to an element in a component template. A state describes the styles associated with a named value, and a transition describes what animation should run when the bound value changes from one state to another [2].

3.1 Basic open/closed trigger

The following example uses a classic disclosure panel. The TypeScript metadata declares an animation trigger named panel. The template binds that trigger using [@panel]="isOpen() ? 'open' : 'closed'". 

// panel.component.ts
import { Component, signal } from '@angular/core';
import {
  animate,
  state,
  style,
  transition,
  trigger
} from '@angular/animations';

@Component({
  selector: 'app-panel',
  standalone: true,
  template: `
    <button type="button" (click)="toggle()">
      {{ isOpen() ? 'Close' : 'Open' }}
    </button>

    <section class="panel" [@panel]="isOpen() ? 'open' : 'closed'">
      <h2>Build details</h2>
      <p>This content expands and collapses with an Angular trigger.</p>
    </section>
  `,
  styles: [`
    .panel {
      overflow: hidden;
      border: 1px solid #d1d5db;
      border-radius: 12px;
      padding: 1rem;
      margin-top: 1rem;
    }
  `],
  animations: [
    trigger('panel', [
      state('open', style({
        height: '*',
        opacity: 1,
        transform: 'translateY(0)'
      })),
      state('closed', style({
        height: '0px',
        opacity: 0,
        transform: 'translateY(-8px)',
        paddingTop: '0',
        paddingBottom: '0'
      })),
      transition('closed => open', [
        animate('180ms cubic-bezier(0.2, 0, 0, 1)')
      ]),
      transition('open => closed', [
        animate('140ms cubic-bezier(0.4, 0, 1, 1)')
      ])
    ])
  ]
})
export class PanelComponent {
  readonly isOpen = signal(false);

  toggle(): void {
    this.isOpen.update(value => !value);
  }
}

The wildcard value * has two meanings in this API. In transition expressions it can match any state, while in a style value it can tell Angular to compute and animate to or from the current style value [2]. The void state represents an element that is not yet in the view or has left the view; Angular also provides :enter and :leave aliases for void => * and * => void respectively [2].

3.2 Reducing transition explosion with wildcards

If a component has several states, declaring every pairwise transition becomes noisy. Angular's wildcard transition syntax lets one rule match multiple changes [2]. 

trigger('saveState', [
  state('idle', style({ opacity: 1, transform: 'scale(1)' })),
  state('saving', style({ opacity: 0.7, transform: 'scale(0.98)' })),
  state('saved', style({ opacity: 1, transform: 'scale(1)' })),
  state('error', style({ opacity: 1, transform: 'translateX(0)' })),

  // Any state entering "saving" uses a short compression animation.
  transition('* => saving', [
    animate('120ms ease-out')
  ]),

  // Any state entering "error" shakes.
  transition('* => error', [
    animate('70ms', style({ transform: 'translateX(-4px)' })),
    animate('70ms', style({ transform: 'translateX(4px)' })),
    animate('70ms', style({ transform: 'translateX(0)' }))
  ]),

  // Return from any terminal state to idle.
  transition('* => idle', [
    animate('160ms ease-out')
  ])
])
Pattern: use explicit state-to-state transitions when direction matters, such as closed => open versus open => closed. Use wildcard transitions when the destination state is more important than the origin state.

3.3 Animation callbacks

Angular trigger bindings can emit animation lifecycle callbacks such as start and done events. These are useful for instrumentation, focus management, and deferred work that should not happen until motion completes [2]. 

<section
  [@panel]="isOpen() ? 'open' : 'closed'"
  (@panel.start)="onAnimationStart($event)"
  (@panel.done)="onAnimationDone($event)">
  ...
</section>
import { AnimationEvent } from '@angular/animations';

onAnimationDone(event: AnimationEvent): void {
  if (event.toState === 'open') {
    // Example: move focus into a newly opened panel.
    this.firstFocusableElement()?.focus();
  }
}

4. Enter and leave animation

Modern Angular's recommended entry point is CSS. With animate.enter, Angular applies the named class while the element enters the DOM and removes the class after the animation is complete. With animate.leave, Angular applies the leave class and removes the element only after the longest transition or keyframe animation completes [1].

4.1 Modern CSS-first enter/leave example

// toast.component.ts
import { Component, signal } from '@angular/core';

@Component({
  selector: 'app-toast-demo',
  standalone: true,
  template: `
    <button type="button" (click)="show.set(true)">Show toast</button>

    @if (show()) {
      <aside
        class="toast"
        animate.enter="toast-enter"
        animate.leave="toast-leave">
        <strong>Saved</strong>
        <span>Your changes were persisted.</span>
        <button type="button" (click)="show.set(false)" aria-label="Dismiss">×</button>
      </aside>
    }
  `,
  styleUrl: './toast.component.css'
})
export class ToastDemoComponent {
  readonly show = signal(false);
}
/* toast.component.css */
.toast {
  display: flex;
  gap: 0.75rem;
  align-items: center;
  max-width: 32rem;
  padding: 0.9rem 1rem;
  border: 1px solid #cbd5e1;
  border-radius: 0.9rem;
  background: white;
  color: #0f172a;
  box-shadow: 0 18px 48px rgba(15, 23, 42, 0.18);
}

.toast-enter {
  animation: toast-in 180ms cubic-bezier(0.2, 0, 0, 1);
}

.toast-leave {
  animation: toast-out 140ms cubic-bezier(0.4, 0, 1, 1);
}

@keyframes toast-in {
  from {
    opacity: 0;
    transform: translateY(12px) scale(0.98);
  }

  to {
    opacity: 1;
    transform: translateY(0) scale(1);
  }
}

@keyframes toast-out {
  from {
    opacity: 1;
    transform: translateY(0) scale(1);
  }

  to {
    opacity: 0;
    transform: translateY(8px) scale(0.98);
  }
}

@media (prefers-reduced-motion: reduce) {
  .toast-enter,
  .toast-leave {
    animation-duration: 1ms;
  }
}

This style keeps animation ownership in CSS. Angular controls the lifecycle: class insertion, class removal, and delayed DOM removal for leave animations [1]. The CSS owns timing, easing, transform values, and reduced-motion behavior.

4.2 Legacy equivalent with :enter and :leave

trigger('toastMotion', [
  transition(':enter', [
    style({ opacity: 0, transform: 'translateY(12px) scale(0.98)' }),
    animate(
      '180ms cubic-bezier(0.2, 0, 0, 1)',
      style({ opacity: 1, transform: 'translateY(0) scale(1)' })
    )
  ]),
  transition(':leave', [
    animate(
      '140ms cubic-bezier(0.4, 0, 1, 1)',
      style({ opacity: 0, transform: 'translateY(8px) scale(0.98)' })
    )
  ])
])

In legacy code, :enter and :leave are expressive because they map directly to insertion and removal. The migration issue is that this behavior is now usually simpler and more portable when written as CSS keyframes and classes [1,5].

5. Complex sequences

Complex Angular animations coordinate more than one element or more than one animation step. The legacy package documents four central sequencing functions: query() finds inner elements, stagger() applies a cascading delay, group() runs animation steps in parallel, and sequence() runs steps one after another [3].

5.1 Staggered list entry with query() and stagger()

// task-list.component.ts
import { Component, signal } from '@angular/core';
import {
  animate,
  query,
  stagger,
  style,
  transition,
  trigger
} from '@angular/animations';

interface Task {
  id: number;
  title: string;
  owner: string;
}

@Component({
  selector: 'app-task-list',
  standalone: true,
  template: `
    <button type="button" (click)="load()">Load tasks</button>

    <ul class="tasks" @listIntro>
      @for (task of tasks(); track task.id) {
        <li class="task-card">
          <strong>{{ task.title }}</strong>
          <span>{{ task.owner }}</span>
        </li>
      }
    </ul>
  `,
  animations: [
    trigger('listIntro', [
      transition(':enter', [
        query('.task-card', [
          style({
            opacity: 0,
            transform: 'translateY(16px)'
          }),
          stagger(45, [
            animate(
              '220ms cubic-bezier(0.2, 0, 0, 1)',
              style({
                opacity: 1,
                transform: 'translateY(0)'
              })
            )
          ])
        ], { optional: true })
      ])
    ])
  ]
})
export class TaskListComponent {
  readonly tasks = signal<Task[]>([]);

  load(): void {
    this.tasks.set([
      { id: 1, title: 'Index new documents', owner: 'Maya' },
      { id: 2, title: 'Review audit trail', owner: 'Noah' },
      { id: 3, title: 'Deploy search worker', owner: 'Ava' }
    ]);
  }
}

The { optional: true } option avoids throwing when the query finds no matching elements. This is a practical guard for dynamic lists whose content may be empty at the moment an animation runs.

5.2 Parallel animation with group()

Parallel animation is useful when different properties need different timings. The Angular documentation describes group() as a way to run multiple animation steps in parallel [3]. 

trigger('cardEnterLeave', [
  transition(':enter', [
    style({
      opacity: 0,
      transform: 'translateY(20px)',
      filter: 'blur(6px)'
    }),
    group([
      animate(
        '180ms cubic-bezier(0.2, 0, 0, 1)',
        style({ opacity: 1 })
      ),
      animate(
        '260ms cubic-bezier(0.2, 0, 0, 1)',
        style({ transform: 'translateY(0)' })
      ),
      animate(
        '320ms ease-out',
        style({ filter: 'blur(0)' })
      )
    ])
  ]),
  transition(':leave', [
    group([
      animate('120ms ease-in', style({ opacity: 0 })),
      animate('160ms ease-in', style({ transform: 'translateY(12px)' }))
    ])
  ])
])

5.3 Sequential animation with sequence()

Sequential animation is useful when one step must finish before the next begins. Angular's sequence() function runs animation steps one after another [3]. 

trigger('saveButton', [
  transition('idle => saving', [
    sequence([
      animate('80ms ease-out', style({ transform: 'scale(0.96)' })),
      animate('120ms ease-out', style({ transform: 'scale(1)' })),
      animate('200ms ease-out', style({ opacity: 0.72 }))
    ])
  ]),
  transition('saving => saved', [
    sequence([
      animate('120ms ease-out', style({ opacity: 1 })),
      animate('180ms ease-out', style({ backgroundColor: '#16a34a' }))
    ])
  ])
])

5.4 Parent and child animations

Complex Angular components often contain nested triggers. The legacy documentation notes that query() can be followed by animateChild() to run child animations that might otherwise be blocked by the parent animation [3]. 

import {
  animate,
  animateChild,
  query,
  style,
  transition,
  trigger
} from '@angular/animations';

export const pageWithChildren = trigger('pageWithChildren', [
  transition(':enter', [
    style({ opacity: 0 }),
    animate('120ms ease-out', style({ opacity: 1 })),
    query('@*', animateChild(), { optional: true })
  ]),
  transition(':leave', [
    query('@*', animateChild(), { optional: true }),
    animate('100ms ease-in', style({ opacity: 0 }))
  ])
]);

5.5 Reordering lists and tracking identity

When list items are reordered, Angular needs stable identity to keep track of which DOM element corresponds to which data item. The Angular documentation recommends using tracking for lists that may reorder during runtime so animation can apply to the correct elements [3]. 

@for (task of tasks(); track task.id) {
  <li class="task-card" @cardEnterLeave>
    {{ task.title }}
  </li>
}
Avoid animating layout-heavy properties by default. Height, width, margin, top, and left can trigger layout work. Prefer opacity and transform for high-frequency UI motion, and reserve layout animation for cases where the visual meaning requires it.

6. Reusable animations

Reusable animation has two forms in Angular applications. In modern Angular code, reuse typically means shared CSS classes, keyframes, custom properties, and design tokens. In legacy DSL code, reuse means exporting animation definitions from a shared TypeScript file with animation() and consuming them with useAnimation(); Angular supports runtime replacement of animation parameters such as height, opacity, color, and timing [4].

6.1 Modern reusable CSS animation

/* motion.css */
:root {
  --motion-fast: 120ms;
  --motion-standard: 200ms;
  --motion-emphasized: cubic-bezier(0.2, 0, 0, 1);
}

.motion-fade-in-up {
  animation: fade-in-up var(--motion-standard) var(--motion-emphasized);
}

.motion-fade-out-down {
  animation: fade-out-down var(--motion-fast) ease-in;
}

@keyframes fade-in-up {
  from {
    opacity: 0;
    transform: translateY(var(--motion-distance, 12px));
  }

  to {
    opacity: 1;
    transform: translateY(0);
  }
}

@keyframes fade-out-down {
  from {
    opacity: 1;
    transform: translateY(0);
  }

  to {
    opacity: 0;
    transform: translateY(var(--motion-distance, 8px));
  }
}
<aside
  class="toast"
  style="--motion-distance: 16px"
  animate.enter="motion-fade-in-up"
  animate.leave="motion-fade-out-down">
  Saved successfully.
</aside>

This approach makes animation tokens shareable across Angular, plain HTML, Web Components, and non-Angular contexts. It also aligns with Angular's migration guidance, which shows reusable native CSS animations as the replacement for animation factories in the deprecated package [5].

6.2 Legacy reusable animation factory

// shared-animations.ts
import {
  animate,
  animation,
  style
} from '@angular/animations';

export const fadeScale = animation([
  style({
    opacity: '{{ fromOpacity }}',
    transform: 'scale({{ fromScale }})'
  }),
  animate(
    '{{ duration }} {{ easing }}',
    style({
      opacity: '{{ toOpacity }}',
      transform: 'scale({{ toScale }})'
    })
  )
], {
  params: {
    fromOpacity: 0,
    toOpacity: 1,
    fromScale: 0.98,
    toScale: 1,
    duration: '180ms',
    easing: 'cubic-bezier(0.2, 0, 0, 1)'
  }
});
// menu.component.ts
import { Component, signal } from '@angular/core';
import {
  transition,
  trigger,
  useAnimation
} from '@angular/animations';

import { fadeScale } from './shared-animations';

@Component({
  selector: 'app-menu',
  standalone: true,
  template: `
    <button type="button" (click)="open.update(v => !v)">Menu</button>

    @if (open()) {
      <nav class="menu" @menuMotion>
        <a href="/profile">Profile</a>
        <a href="/settings">Settings</a>
      </nav>
    }
  `,
  animations: [
    trigger('menuMotion', [
      transition(':enter', [
        useAnimation(fadeScale)
      ]),
      transition(':leave', [
        useAnimation(fadeScale, {
          params: {
            fromOpacity: 1,
            toOpacity: 0,
            fromScale: 1,
            toScale: 0.98,
            duration: '120ms',
            easing: 'ease-in'
          }
        })
      ])
    ])
  ]
})
export class MenuComponent {
  readonly open = signal(false);
}
Pattern: store reusable animation primitives near design-system code. Use names based on user experience intent, such as fadeScale, listStagger, or routeSlide, rather than component-specific names such as settingsPageAnimation.

7. Route transitions

Route transitions are animations that occur when Angular Router changes the visible view. Angular Router now documents built-in integration with the browser's View Transitions API, currently described by Angular as developer preview and dependent on browser support [6]. The browser API works by capturing the old view, executing a DOM update callback, capturing the new view, and animating between those two states [6,7].

7.1 Modern route transitions with Angular Router View Transitions

In current Angular applications, the preferred route-transition direction is usually Router View Transitions. A typical setup enables view transitions in the router providers and defines CSS for the browser's transition pseudo-elements. 

// app.config.ts
import { ApplicationConfig } from '@angular/core';
import {
  provideRouter,
  withViewTransitions
} from '@angular/router';

import { routes } from './app.routes';

export const appConfig: ApplicationConfig = {
  providers: [
    provideRouter(
      routes,
      withViewTransitions()
    )
  ]
};
/* styles.css */

/* The names of these pseudo-elements come from the browser View Transitions API. */
::view-transition-old(root) {
  animation: route-out 140ms ease-in both;
}

::view-transition-new(root) {
  animation: route-in 180ms cubic-bezier(0.2, 0, 0, 1) both;
}

@keyframes route-out {
  from {
    opacity: 1;
    transform: translateY(0);
  }

  to {
    opacity: 0;
    transform: translateY(8px);
  }
}

@keyframes route-in {
  from {
    opacity: 0;
    transform: translateY(10px);
  }

  to {
    opacity: 1;
    transform: translateY(0);
  }
}

@media (prefers-reduced-motion: reduce) {
  ::view-transition-old(root),
  ::view-transition-new(root) {
    animation-duration: 1ms;
  }
}

View Transitions are attractive for route changes because the browser owns the capture and interpolation between old and new visual states. Angular Router participates in the navigation lifecycle by preparing navigation, starting the view transition when routes are ready, activating and deactivating routes inside the transition callback, and finalizing the transition after rendering completes [6].

7.2 Legacy route transition trigger

Existing codebases may still animate route transitions using the legacy animation DSL. The usual structure is: wrap the router-outlet, read route data from the activated route, bind it to a trigger, and animate :enter and :leave views together. 

// app.routes.ts
import { Routes } from '@angular/router';

export const routes: Routes = [
  {
    path: '',
    loadComponent: () => import('./home.page').then(m => m.HomePage),
    data: { animation: 'HomePage' }
  },
  {
    path: 'settings',
    loadComponent: () => import('./settings.page').then(m => m.SettingsPage),
    data: { animation: 'SettingsPage' }
  }
];
// app.component.ts
import { Component } from '@angular/core';
import { RouterOutlet } from '@angular/router';
import {
  animate,
  group,
  query,
  style,
  transition,
  trigger
} from '@angular/animations';

@Component({
  selector: 'app-root',
  standalone: true,
  imports: [RouterOutlet],
  template: `
    <main class="route-shell" [@routeMotion]="routeKey(outlet)">
      <router-outlet #outlet="outlet"></router-outlet>
    </main>
  `,
  styles: [`
    .route-shell {
      position: relative;
      display: block;
      min-height: 100vh;
      overflow: hidden;
    }
  `],
  animations: [
    trigger('routeMotion', [
      transition('* <=> *', [
        query(':enter, :leave', [
          style({
            position: 'absolute',
            inset: 0,
            width: '100%'
          })
        ], { optional: true }),

        query(':enter', [
          style({
            opacity: 0,
            transform: 'translateX(24px)'
          })
        ], { optional: true }),

        group([
          query(':leave', [
            animate(
              '140ms ease-in',
              style({
                opacity: 0,
                transform: 'translateX(-16px)'
              })
            )
          ], { optional: true }),

          query(':enter', [
            animate(
              '180ms cubic-bezier(0.2, 0, 0, 1)',
              style({
                opacity: 1,
                transform: 'translateX(0)'
              })
            )
          ], { optional: true })
        ])
      ])
    ])
  ]
})
export class AppComponent {
  routeKey(outlet: RouterOutlet): string | undefined {
    return outlet?.activatedRouteData?.['animation'];
  }
}

The important layout detail is absolute positioning for entering and leaving route components during the transition. Without it, both views can affect layout while they coexist temporarily. This is one reason Router View Transitions are a cleaner fit for new applications: the browser-level API is designed specifically to animate between visual states of the page [6,7].

7.3 Directional route transitions

Some products need navigation direction: forward navigation slides one way, backward navigation slides the other. Use route metadata to assign each route an order, then bind a computed direction to the transition trigger. 

// routes
export const routes: Routes = [
  { path: '', component: HomePage, data: { order: 0 } },
  { path: 'projects', component: ProjectsPage, data: { order: 1 } },
  { path: 'projects/:id', component: ProjectDetailPage, data: { order: 2 } }
];
// Conceptual direction calculation
previousOrder = currentOrder;
currentOrder = Number(outlet.activatedRouteData['order'] ?? 0);

direction = currentOrder > previousOrder ? 'forward' : 'backward';

Directional route animation can improve spatial comprehension in wizard-like flows, but it can be confusing for non-linear navigation. For dashboards, settings pages, and deep links, fade or shared-element transitions are usually less misleading than slide metaphors.

8. Performance, accessibility, and testing

8.1 Performance rules

Prefer transform and opacity

Transform and opacity animations usually avoid expensive layout recalculation and are the default choice for frequent UI transitions.

Keep duration purposeful

Most microinteractions feel responsive when they complete in roughly 100–250ms. Longer motion should communicate meaningful spatial change, not decorate latency.

Stagger sparingly

Staggering improves scanability for short lists, but it can slow experienced users when applied to long or frequently changing lists.

Avoid accidental layout animation

Animating height from 0 to * is convenient in the legacy DSL, but CSS transform-based reveal patterns often perform better.

8.2 Reduced motion

CSS supports the prefers-reduced-motion media feature, which allows sites to reduce or replace non-essential animation for users who have requested less motion at the operating-system or browser level [8]. Angular applications should honor this preference in global motion utilities and component-local animation CSS. 

@media (prefers-reduced-motion: reduce) {
  *,
  *::before,
  *::after {
    scroll-behavior: auto !important;
  }

  .motion-fade-in-up,
  .motion-fade-out-down,
  .toast-enter,
  .toast-leave {
    animation-duration: 1ms !important;
    transition-duration: 1ms !important;
  }
}

8.3 Testing animation-heavy components

Animation tests should verify state, DOM lifecycle, and accessibility behavior rather than pixel-perfect frames. For unit tests, prefer deterministic assertions:

  • the element is inserted when the state becomes visible;
  • the leave action eventually removes the element;
  • focus moves only after expansion completes when focus management depends on motion;
  • the component remains usable when reduced motion disables meaningful duration;
  • route transitions do not break direct navigation or browser back/forward behavior.

In end-to-end tests, consider disabling or shortening animation globally unless the animation itself is the feature under test. This reduces flakiness caused by timing, CPU load, and browser scheduling.

9. Migration strategy

Angular's migration guidance says that @angular/animations was deprecated as of v20.2 and that new animate.enter and animate.leave features can replace animations based on the package with plain CSS or JavaScript animation libraries [5]. A pragmatic migration plan is:

  1. Inventory triggers. Find all component animations: [] metadata and categorize by type: enter/leave, state transition, list sequence, route transition, or reusable factory.
  2. Migrate simple enter/leave first. Replace :enter and :leave triggers with animate.enter, animate.leave, and CSS keyframes.
  3. Move state animation to CSS classes. Replace state() declarations with class bindings such as [class.open]="isOpen()" and [class.closed]="!isOpen()".
  4. Preserve complex sequences until their UX is stable. Rewriting query(), stagger(), and nested animateChild() behavior may require design-system-level CSS utilities or a third-party animation library.
  5. Move route transitions to Router View Transitions where browser support is acceptable. Keep legacy fallbacks for browsers or flows where View Transitions are unavailable.

9.1 Before and after: state transition

// Before: legacy trigger
<section [@panel]="isOpen() ? 'open' : 'closed'">...</section>
/* After: class-driven CSS */
<section class="panel" [class.open]="isOpen()" [class.closed]="!isOpen()">
  ...
</section>

10. Conclusion

Animation in Angular is best understood as a lifecycle coordination problem. Angular knows when components, template blocks, and routes enter or leave the rendered tree; CSS and browser APIs are increasingly the preferred engines for visual motion. The legacy @angular/animations DSL is still valuable for maintaining existing applications and for modeling sophisticated transitions with triggers, queries, staggered lists, parallel groups, sequential steps, reusable factories, and route outlet transitions. For new development, the center of gravity has moved toward CSS keyframes, CSS transitions, animate.enter, animate.leave, and Angular Router's View Transitions integration.

The durable architectural rule is simple: keep motion semantic, reusable, accessible, and cheap. Use Angular to identify lifecycle moments; use CSS variables and keyframes to express design-system motion; use View Transitions for whole-route visual continuity; and reserve complex JavaScript-driven sequencing for cases where the interaction truly needs it.
