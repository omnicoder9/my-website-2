Nmap for Active Directory

The domain controller is a noisy machine. It announces itself in nine open ports, answers anonymous LDAP, leaks usernames through Kerberos error codes, and negotiates SMB in ways that betray a decade of compatibility debt. This is a field study of how a thirty-year-old port scanner reads the Windows domain at the network layer — and where it stops.

Active Directory is, at its root, a set of network services bolted onto a replicated database. A domain controller does not advertise itself as "the keys to the kingdom"; it advertises itself as a Kerberos key distribution center on TCP 88, an LDAP responder on 389 and 636, a Global Catalog on 3268, and a file and RPC endpoint on 135, 139, and 445. Every one of those listeners speaks a documented protocol, and every protocol leaks structure to an unauthenticated stranger on the wire. Nmap — the Network Mapper — is the instrument that turns those leaks into a map. It is rarely the tool that finishes an Active Directory engagement, but it is almost always the tool that starts one.

This dossier treats Nmap as a network-layer reconnaissance platform aimed specifically at Windows domains. We move from the scanner's history, through the port fingerprint that distinguishes a domain controller from any other Windows host, and into the five capability areas where the Nmap Scripting Engine does real work against AD: domain controller discovery, LDAP enumeration, Kerberos enumeration, SMB security assessment, and trust discovery. Throughout, the assumption is an expert reader who already knows what a TGT and a sAMAccountName are, and who wants precision about what the scanner can and cannot extract.

01 — Lineage

Origins of the scanner

Nmap entered the world not as a product but as a magazine article. In September 1997, Gordon Lyon — writing under the handle Fyodor — published The Art of Port Scanning in issue 51 of Phrack, bundling roughly two thousand lines of Linux-only C source alongside the prose.1 The release consolidated a scattering of scanning techniques — half-open SYN scans, FIN and Xmas probes, decoy evasion — into one coherent utility, and Lyon has coordinated its development ever since.2 Over the following decade the codebase migrated from C to C++, gained TCP/IP stack fingerprinting for operating-system detection, acquired service and version probes, and was ported well beyond Linux to Windows, macOS, and the BSDs.2

The feature that matters most for Active Directory work arrived later. On 10 December 2006, the Nmap Scripting Engine (NSE) shipped as part of Nmap 4.21ALPHA1, carrying an initial set of twenty-three scripts for tasks as humble as detecting SSHv1 or querying recursive DNS; a year later, Nmap 4.50 marked the project's tenth anniversary.3 NSE transformed the scanner from a port enumerator into an extensible probing framework. Its scripts are written in embedded Lua — the engine pairs a lightweight Lua interpreter with a C++ network framework and a library layer (nselib) that exposes protocol implementations for SMB, LDAP, Kerberos, MSRPC, and dozens more.4 Today the project sits in its 7.9x series, with 7.98 released in August 2025 and 7.99 subsequently published for download, still distributed under the same open model.5

A protocol implementation written once, in Lua, against the wire — that is what separates Active Directory enumeration from a generic port sweep.

On the significance of the NSE library layer

For domain reconnaissance this matters because the interesting questions about Active Directory are not "is port 389 open?" but "what naming contexts does this directory hold, which SASL mechanisms will it accept, and does this username exist?" Answering those requires speaking LDAP and Kerberos, not merely fingerprinting a banner — and NSE is where that protocol fluency lives.

02 — The fingerprint

The domain's network signature

Active Directory's network behaviour is unusually legible. The directory service itself runs inside the Lsass.exe process on each domain controller, and Microsoft documents that domain controllers, member computers, and application servers require connectivity to that service over a set of hard-coded ports, supplemented by an ephemeral range — 49152–65535 on modern Windows, or the legacy 1025–5000 on releases predating Windows Server 2008 and Vista — for the dynamic RPC endpoints that the endpoint mapper hands out.6 The fixed ports are the ones that give a domain controller away.

No single port proves a domain controller — plenty of hosts expose 445, and a bare LDAP server might listen on 389. What identifies a DC is the combination. A host answering simultaneously on Kerberos (88), LDAP (389), and the Global Catalog (3268), with kpasswd on 464 and LDAPS on 636 alongside, is a domain controller with near-certainty; the Global Catalog ports in particular are not present on a domain member. Run against a real DC, Nmap's service detection resolves these listeners to the labels it carries in its services database — and the resulting report reads almost like a checklist of the AD role:7

03 — Discovery

Domain controller discovery

Finding domain controllers is the first move of any AD assessment, and it has two complementary paths: ask the network's own locator service, or sweep for the port fingerprint. The elegant path is DNS. Active Directory publishes its locators as SRV records under a well-known namespace, and Nmap's dns-srv-enum script walks exactly the set that matters — Global Catalog, the Kerberos KDC, the Kerberos password-change service, and the LDAP servers among them.9 Where you know (or can guess) the domain's DNS name, this is faster and quieter than a port sweep because the directory volunteers the answer:

srv-locator — enumerate AD SRV records via DNS
$ nmap --script dns-srv-enum --script-args "dns-srv-enum.domain=corp.example.com"

| dns-srv-enum:
|   Active Directory Global Catalog
|     service          prio  weight  host
|     3268/tcp         0     100     dc01.corp.example.com
|     3268/tcp         0     100     dc02.corp.example.com
|   Kerberos KDC Service
|     88/tcp           0     100     dc01.corp.example.com
|   LDAP Servers
|_    389/tcp          0     100     dc01.corp.example.com

The query path mirrors what every domain-joined machine does at boot: it resolves names such as _ldap._tcp.dc._msdcs.<domain>, _kerberos._tcp.<domain>, and _gc._tcp.<domain> to locate writable DCs and Global Catalog servers. The SRV approach hands you hostnames and priorities directly, which you then resolve and confirm.

The blunt-instrument path is the port fingerprint. When you have an address range but no domain name, you scan for the diagnostic combination and let the open-port pattern surface the controllers. A pragmatic sweep skips host discovery (DCs frequently filter ICMP), targets only the AD-relevant ports, and reports just the live listeners:

dc-sweep — locate DCs across a subnet by port pattern
$ nmap -Pn -n --open -T4 -p 88,389,3268 10.10.0.0/22 -oG dc-sweep.gnmap

# Hosts answering on all three of 88 + 389 + 3268 are domain controllers.
# Grep the greppable output for the pattern, then re-scan the hits with -sV.
$ grep "88/open" dc-sweep.gnmap | grep "3268/open"
Practitioner note

Treat 88 + 389 + 3268 together as the decision rule. Port 3268 is the discriminator: a member server or workstation will show 445 and often 135/139, but never the Global Catalog. Confirming a candidate with smb-os-discovery (next section) then yields the DC's FQDN, the NetBIOS and DNS domain names, and the forest name in a single probe.

04 — Directory

LDAP & the Global Catalog

LDAP is the directory's query interface, standardised in RFC 4511, and an Active Directory domain controller exposes it on 389 (and the TLS variant on 636), with the forest-wide Global Catalog on 3268/3269.10 Every LDAP server publishes an unauthenticated entry called the rootDSE — the DSA-specific entry — whose operational attributes describe the server's own capabilities and the partitions it holds; RFC 4512 defines attributes such as namingContexts, supportedControl, supportedSASLMechanisms, and supportedLDAPVersion.11 The rootDSE is meant to be readable before binding, and that is precisely what Nmap's ldap-rootdse retrieves.

The script's value against AD is that a Windows directory returns far more in its rootDSE than a minimal OpenLDAP instance — naming contexts, the schema and configuration partitions, the DNS application partitions, the default and root-domain naming contexts, the dsServiceName pointing at the NTDS settings object, and a long list of supported controls and SASL mechanisms — and historically Windows servers served all of it without authentication.12 In a single anonymous request you obtain the distinguished-name skeleton of the entire forest:

rootdse — anonymous map of the forest partitions
$ nmap -p 389 --script ldap-rootdse 10.10.30.10

389/tcp open  ldap
| ldap-rootdse:
|   namingContexts: DC=corp,DC=example,DC=com
|   namingContexts: CN=Configuration,DC=corp,DC=example,DC=com
|   namingContexts: CN=Schema,CN=Configuration,DC=corp,DC=example,DC=com
|   namingContexts: DC=DomainDnsZones,DC=corp,DC=example,DC=com
|   namingContexts: DC=ForestDnsZones,DC=corp,DC=example,DC=com
|   defaultNamingContext: DC=corp,DC=example,DC=com
|   rootDomainNamingContext: DC=corp,DC=example,DC=com
|   dnsHostName: dc01.corp.example.com
|   dsServiceName: CN=NTDS Settings,CN=DC01,CN=Servers,CN=Default-First-Site-Name,...
|   supportedSASLMechanisms: GSSAPI
|   supportedSASLMechanisms: GSS-SPNEGO
|_  supportedLDAPVersion: 3

From the rootDSE you graduate to content with ldap-search, which performs an actual LDAP search and returns matching entries. If no credentials are supplied, the script consults the Nmap registry — using an account recovered by ldap-brute if one was found — and otherwise falls back to an anonymous bind; by default it derives its search base from the defaultNamingContext exposed in the rootDSE, iterating over the available naming contexts if none is offered.13 Against AD it can enumerate user, computer, and group objects with their sAMAccountName values, and the underlying LDAP library understands AD-specific encodings — Tom Sellers added decoding of the binary objectSID attribute in 2017, and the library converts the directory's 100-nanosecond-since-1601 timestamps into human-readable form.14

ldap-search — authenticated enumeration of principals
$ nmap -p 389 --script ldap-search --script-args 'ldap.username="CORP\\svc_read",
    ldap.password=R3ad0nly!,ldap.qfilter=users,ldap.attrib=sAMAccountName' 10.10.30.10

| ldap-search:
|   Context: DC=corp,DC=example,DC=com
|     dn: CN=Administrator,CN=Users,DC=corp,DC=example,DC=com
|         sAMAccountName: Administrator
|     dn: CN=krbtgt,CN=Users,DC=corp,DC=example,DC=com
|         sAMAccountName: krbtgt
|     dn: CN=SQL Service,CN=Users,DC=corp,DC=example,DC=com
|_        sAMAccountName: svc_sql

The rootDSE is the directory volunteering its own table of contents — partitions, controls, and supported authentication, served before you have proven who you are.

On anonymous LDAP exposure

One operational subtlety: the Global Catalog on 3268 holds a partial, read-only replica of every domain in the forest, so pointing ldap-search there rather than at 389 widens enumeration from one domain to the whole forest in a single query — at the cost of returning only the attributes flagged for GC replication. For an attacker mapping a multi-domain forest, that trade is usually worth taking.

05 — Authentication

Kerberos enumeration

Kerberos, defined for version 5 in RFC 4120, is Active Directory's default authentication protocol, and its design hands an enumerator a precise oracle.15 When a client requests a ticket-granting ticket with an Authentication Service request (AS-REQ), the key distribution center on port 88 responds differently depending on whether the named principal exists. If the account is unknown, the KDC returns KRB5KDC_ERR_C_PRINCIPAL_UNKNOWN; if the account exists, the server either returns the error KRB5KDC_ERR_PREAUTH_REQUIRED — demanding pre-authentication — or, for an account configured without pre-authentication, returns a TGT outright in an AS-REP.16 That divergence is a username-validity check that requires no password and, crucially, no successful logon.

Nmap's krb5-enum-users automates exactly this. It brute-forces a list of candidate usernames against the Kerberos service, classifies each by the returned error code, and reports the valid ones; it works against Active Directory and needs only a valid Kerberos realm to operate.17 The Hacker Recipes documents the mechanism precisely as Nmap implements it — an AS-REQ per candidate, the PRINCIPAL_UNKNOWN error for invalid names, and either a TGT or the PREAUTH_REQUIRED error for valid ones.16 Because the script never submits a password, it does not increment the badPwdCount attribute and so does not trip account lockout — a property that makes Kerberos username enumeration markedly safer for the operator than SMB or LDAP password spraying.

krb5-enum-users — validate accounts via KDC error codes
$ nmap -p 88 --script krb5-enum-users --script-args krb5-enum-users.realm='CORP.EXAMPLE.COM',
    userdb=/opt/wordlists/ad-users.txt 10.10.30.10

88/tcp open  kerberos-sec
| krb5-enum-users:
|   Discovered Kerberos principals
|       administrator@CORP.EXAMPLE.COM
|       svc_sql@CORP.EXAMPLE.COM
|       jdoe@CORP.EXAMPLE.COM
|_      backup@CORP.EXAMPLE.COM  (no pre-auth — AS-REP returned)

An account that answers with an AS-REP rather than a pre-auth demand is more than a confirmed username: it is a candidate for AS-REP roasting. Where the "Do not require Kerberos pre-authentication" flag is set on a user object, anyone can request an AS-REP on that user's behalf and receive a blob encrypted under a key derived from the account's password hash, which can then be cracked offline.18 Pre-authentication has been enabled by default since Kerberos v5, so a roastable account is a misconfiguration — and Nmap's error-code classifier flags exactly these accounts as a by-product of enumeration.

Where Nmap hands off

Nmap confirms which accounts exist and which lack pre-authentication; it does not retrieve and crack the roastable hashes, perform password spraying through Kerberos, or carry out Kerberoasting against service principal names. Those workflows belong to purpose-built tooling — Kerbrute for high-speed enumeration and spraying without lockout, and Impacket's GetNPUsers/GetUserSPNs or Rubeus for harvesting AS-REP and TGS material.19 The discipline is to use the scanner for the cheap, low-risk validation step and switch tools once you need to act on credentials.

06 — SMB

SMB security assessment

SMB is the richest single surface Nmap reaches in a Windows domain, because the protocol carries both file access and a great deal of remote administration via named-pipe MSRPC. The first probe is almost always smb-os-discovery, which determines the operating system, computer name, domain, workgroup, and current time over SMB on port 445 or 139 — context that calibrates everything that follows, including the FQDN, the DNS and NetBIOS domain names, and the forest name.20

Message signing and the relay problem smb-security-mode · smb2-security-mode

The most consequential SMB misconfiguration Nmap surfaces is the absence of required message signing. SMB signing comes in two flags — enabled and required — and the SMBv2/3 negotiate response encodes them in a security-mode field, where bit 0x01 marks signing enabled and 0x02 marks it required.21 The smb2-security-mode script sends an SMB2_COM_NEGOTIATE for each supported dialect and parses that field, reporting one of four states; its older sibling smb-security-mode reports the SMBv1 view and famously labels the historical default as "message signing disabled (dangerous, but default)."22 The reason it is dangerous is direct: when a server does not require signing, an attacker positioned in the middle can negotiate signing off and relay authentication, which is the foundation of SMB-relay and NTLM-relay attacks.22

signing-check — audit SMB message signing posture
$ nmap -p 445 --script smb2-security-mode,smb-security-mode 10.10.30.40

445/tcp open  microsoft-ds
| smb2-security-mode:
|   3:1:1:
|_    Message signing enabled but not required     # <- relay-exposed
| smb-security-mode:
|   account_used: guest
|   authentication_level: user
|   challenge_response: supported
|_  message_signing: disabled (dangerous, but default)
Reading the result correctly

Domain controllers require SMB signing by default, so the relay exposure lives on the broader estate of member servers and workstations, not the DCs themselves — and recent Windows releases have begun enabling signing requirements more aggressively, narrowing the historical gap. The state to flag is enabled but not required (or disabled): those hosts are viable relay targets. Note also that when SMBv3 is the only dialect and encryption is in force, the signing field can read as a false positive, so corroborate before reporting.

Enumeration over MSRPC smb-enum-*

The smb-enum-* family rides named-pipe MSRPC calls — chiefly the SAMR and SRVSVC interfaces — to pull domain and host structure. Their reach depends heavily on access level, and on modern Windows that usually means authenticated access:

The SMB enumeration scripts and their RPC underpinnings
ScriptWhat it returnsMechanism & access
smb-enum-domainsDomains and password / lockout policy, the domain SID, creation timeSAMR bind → EnumDomains → QueryDomainInfo2; generally needs credentials (anonymous only worked against Windows 2000).23
smb-enum-usersUser accounts via SID/RID resolutionDetermines the server SID, then cycles RIDs; lineage traces to enum.exe / sid2user / user2sid.24

The practical lesson is that the anonymous "null session" enumeration that once made these scripts devastating against Windows 2000 and early 2003 is largely closed on current domain controllers, which restrict anonymous SAMR and share enumeration. Supplied with even a low-privilege domain credential, however, the same scripts become a fast inventory of users, groups, shares, and live sessions.

Vulnerability detection smb-vuln-ms17-010

Nmap also carries SMB vulnerability checks, the most enduring of which is smb-vuln-ms17-010 for the EternalBlue remote-code-execution flaw in SMBv1, tracked as CVE-2017-0143 and weaponised at scale by the WannaCry and NotPetya outbreaks.28 The script — authored by Paulino Calderon — connects to the IPC$ tree, issues a transaction on FID 0, and infers patch status from whether the server returns STATUS_INSUFF_SERVER_RESOURCES, cross-checking against error codes known to be returned by patched systems; it has been validated across Windows XP through Server 2016.29 Years after the patch shipped, it remains a reliable canary for unmanaged SMBv1 hosts inside a domain.

07 — Trust

Trust discovery

Trust relationships define the lateral geography of a forest — which domains will honour authentication from which others, in which direction, and across what boundary. This is the capability where Nmap is weakest, and it is worth being candid about why. Trusts are represented in the directory as trustedDomain objects beneath the CN=System container of each domain's naming context, carrying attributes such as trustPartner, trustDirection, and trustType. Nmap can reach them only indirectly, by aiming an authenticated ldap-search at those objects:

trust-objects — read trustedDomain entries via LDAP
$ nmap -p 389 --script ldap-search --script-args 'ldap.username="CORP\\svc_read",
    ldap.password=R3ad0nly!,ldap.qfilter=custom,ldap.searchattrib="objectClass",
    ldap.searchvalue="trustedDomain",ldap.attrib={trustPartner,trustDirection,trustType}' 10.10.30.10

| ldap-search:
|   Context: DC=corp,DC=example,DC=com
|     dn: CN=dev.example.com,CN=System,DC=corp,DC=example,DC=com
|         trustPartner: dev.example.com
|         trustDirection: 3   (bidirectional)
|_        trustType: 2        (uplevel / AD)

There are two further indirect signals. First, the rootDSE and especially the Global Catalog reveal multiple naming contexts where a forest spans several domains, which hints at the trust topology even without reading trust objects directly. Second, the RPC endpoint mapper on port 135 is the transport for the LSARPC interface that legitimate tools query for trust enumeration — but Nmap maps and fingerprints those RPC endpoints rather than driving the trust-specific LSA calls itself; doing the latter requires speaking interfaces such as LSARPC, SamRPC, and the directory replication (DRS) interface that the endpoint mapper resolves to dynamic ports.30 NetBIOS-based trusts to legacy or Samba-backed domains add a further wrinkle, since they may still rely on the older NetBIOS ports that Microsoft documents as required for trust scenarios.8

Where Nmap hands off

For authoritative trust mapping, the field moves to dedicated tooling: nltest /domain_trusts from a domain context, Impacket's RPC clients against LSARPC, and BloodHound's collectors, which model trusts as edges in the attack graph alongside group membership and session data. Nmap's role here is to confirm the controllers, prove that LDAP and RPC are reachable, and — with a credential — read the trust objects as raw directory entries. It establishes the perimeter; the graph tools draw the map.

08 — Method

A coherent workflow

The capabilities above are most effective sequenced rather than fired at random. A disciplined AD reconnaissance pass with Nmap moves from broad and cheap to narrow and protocol-specific: locate the controllers, establish identity context, then run the targeted NSE scripts that each open service warrants. The pattern below collapses that into a working pipeline.

ad-recon — a staged Nmap pass against a domain
# 1 — Locate domain controllers by the diagnostic port pattern
$ nmap -Pn -n --open -p 88,389,3268 10.10.0.0/16 -oA 01-dc-sweep

# 2 — Identify role & gather domain/forest names from each candidate
$ nmap -Pn -sV -p 88,135,139,389,445,464,636,3268,3269 \
    --script smb-os-discovery,ldap-rootdse 10.10.30.10 -oA 02-dc-detail

# 3 — Validate accounts through Kerberos (no lockout risk)
$ nmap -p 88 --script krb5-enum-users \
    --script-args krb5-enum-users.realm='CORP.EXAMPLE.COM',userdb=users.txt 10.10.30.10

# 4 — Assess SMB posture across the member estate
$ nmap -Pn -p 445 --open \
    --script smb2-security-mode,smb-vuln-ms17-010 10.10.0.0/22 -oA 04-smb-posture

# 5 — With a credential: enumerate directory + shares + sessions
$ nmap -p 389,445 --script ldap-search,smb-enum-shares,smb-enum-sessions \
    --script-args 'ldap.username=...,ldap.password=...,smbusername=...,smbpassword=...' 10.10.30.10
Operational security

None of this is quiet. Kerberos AS-REQ floods generate Event ID 4768 on the KDC, anonymous LDAP and SAMR queries are logged, and SMBv1 probing lights up modern EDR. Authorised assessments should scope timing deliberately — the -T templates and --max-rate exist for a reason — and operators should assume that every script in this dossier is observable by a competent blue team. The legality bright line is simple: scan only systems you own or are explicitly contracted to test.

09 — Defence

The defender's mirror

Every technique here implies a control. Reading the dossier from the blue side yields a concise hardening checklist, much of it grounded in Microsoft's own port and firewall guidance. Segment the network so that the AD service ports are reachable only from the systems that legitimately need them, following the documented port requirements rather than exposing controllers broadly, and restrict the dynamic RPC range as Microsoft describes for the LSA RPC services.630 Require SMB signing across the estate — not merely on the controllers, which enforce it by default — to close the relay surface that smb2-security-mode reports, and retire SMBv1 entirely to eliminate the EternalBlue class.8

At the directory layer, tighten anonymous LDAP and SAMR so that the rootDSE remains the only thing an unauthenticated client can read and bulk principal enumeration requires a bind. At the authentication layer, audit every user object for the "Do not require Kerberos pre-authentication" flag and remove it wherever it is not justified, since those are precisely the accounts Nmap's Kerberos enumeration flags as AS-REP-roastable; pair that with a password policy strong enough that a recovered AS-REP hash is computationally useless.18 Finally, instrument detection: a single source generating a burst of Event ID 4768 entries for non-existent principals is the network signature of exactly the username enumeration described above, and decoy accounts that should never authenticate make excellent tripwires.19

10 — Boundaries

Limits & the wider toolchain

Nmap's honest place in an Active Directory engagement is the network-layer opener. It excels at discovery — proving which hosts are controllers, which services answer, and what their unauthenticated surfaces reveal — and at a specific set of protocol probes that fall out naturally from speaking LDAP, Kerberos, and SMB. It validates usernames, reads the forest's partition skeleton, audits signing posture, and detects a handful of high-impact vulnerabilities. What it does not do is model the domain as a system of relationships or act on the credentials it helps you find.

The moment the work shifts from "what is here?" to "how do these objects relate, and how do I traverse them?", the scanner gives way. Authenticated directory collection and attack-path analysis move to BloodHound and its collectors; credential operations — relaying, roasting, spraying, ticket manipulation — move to Impacket, NetExec, Rubeus, and Kerbrute. Used well, Nmap is the instrument that tells those tools where to point. It reads the domain's network signature with thirty years of refinement behind it, and then knows when to stop.

It establishes the perimeter; the graph tools draw the map. Knowing where the scanner ends is as much a skill as knowing how to drive it.
