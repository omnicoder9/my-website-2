Kubernetes has become the de-facto runtime for containerised workloads. Yet deploying new versions of an application correctly — with zero downtime, safe rollback paths, and minimal blast radius — remains one of the hardest operational challenges teams face.

Deployment patterns encode hard-won engineering wisdom about how to ship software reliably at scale. This reference covers the six most important patterns in production Kubernetes environments: Blue-Green deployments, CI/CD automation, GitOps practices, Helm chart management, Canary deployments, and Rolling Updates with rollbacks. Each section presents the core concept, historical context, worked examples with annotated YAML, tooling recommendations, and caveats.

01
Deployment Strategy

Blue-Green Deployments

Blue-Green deployment is a strategy that reduces downtime and risk by running two nearly identical production environments simultaneously.3 The currently live environment is labelled blue, while the new version is deployed to a cloned environment labelled green.4 While 100% of user traffic continues flowing to the blue environment, operators perform validation and automated testing against the green environment. Once all quality gates pass, a single routing change shifts all traffic to green, making it the new production environment.5

"Blue-Green deployment is especially valuable in Kubernetes environments, cloud-based CI/CD pipelines, and microservices architectures where high availability, risk mitigation, and deployment confidence are paramount."6

It is worth noting that native Kubernetes does not automatically implement this pattern out of the box.7 Teams must build it using Kubernetes primitives — primarily Deployment and Service resources — or adopt purpose-built operators such as Argo Rollouts. The pattern can be implemented consistently across cloud providers including Amazon EKS, Azure AKS, and Google GKE.8

How It Works

1
Deploy the Blue Version

The original application version is deployed as the blue Deployment, with a label such as version: v1. A Kubernetes Service is configured with a selector matching this label, routing all user traffic to the blue pods.9

2
Deploy the Green Version in Parallel

A separate green Deployment is created for the new application version, labelled version: v2. Both deployments run concurrently, but the Service continues routing traffic exclusively to blue.9

3
Validate Green in Production

Automated test suites, smoke tests, and synthetic monitoring are run against the green environment. This provides thorough validation in a production-like environment without impacting live users.7

4
Switch Traffic

After confirming green is stable, the Service's selector is updated from version: v1 to version: v2. This atomic change redirects all incoming traffic to the green pods.9

5
Retain or Remove Blue

The previous blue Deployment can be kept on standby to enable an instant rollback, or deleted to reclaim resources after the green environment is confirmed stable.7

YAML Example

YAML — Blue Deployment 
apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp-blue
  labels:
    app: myapp
    version: v1
spec:
  replicas: 3
  selector:
    matchLabels:
      app: myapp
      version: v1
  template:
    metadata:
      labels:
        app: myapp
        version: v1
    spec:
      containers:
        - name: myapp
          image: myapp:1.0.0
          ports:
            - containerPort: 8080
YAML — Service (traffic switch) 
apiVersion: v1
kind: Service
metadata:
  name: myapp
spec:
  selector:
    app: myapp
    version: v2  # changed from v1 → v2 to cut over
  ports:
    - port: 80
      targetPort: 8080

Automation Pattern

CI/CD Integration

Continuous Integration and Continuous Delivery (CI/CD) pipelines automate the entire software lifecycle — from the moment a developer pushes a commit, through building and testing, to deploying changes into a Kubernetes cluster.11 In a CI/CD model, no human manually applies manifests; instead, every step is triggered by version control events and governed by automated quality gates.

"Deployment strategies are crucial for minimising downtime, ensuring application stability, and providing a seamless experience for users during updates. Kubernetes offers built-in mechanisms to facilitate controlled rollouts, enabling safe and efficient delivery of new features or bug fixes to production environments."11

Example: GitHub Actions Kubernetes Deploy

YAML — GitHub Actions Workflow 
name: Build and Deploy to Kubernetes
on:
  push:
    branches: [main]

jobs:
  build-and-deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Build Docker image
        run: |
          docker build -t myapp:${{ github.sha }} .
          docker push registry.example.com/myapp:${{ github.sha }}

      - name: Set up kubectl
        uses: azure/setup-kubectl@v3

      - name: Deploy to Kubernetes
        run: |
          helm upgrade --install myapp ./charts/myapp \
            --set image.tag=${{ github.sha }} \
            --namespace production \
            --wait --timeout 5m

Operations Pattern

GitOps

GitOps is a set of practices for managing infrastructure and application configurations using Git repositories as the single source of truth for declarative state.12 Rather than pushing changes to a cluster via pipelines, a GitOps operator running inside the cluster continuously compares the desired state declared in Git with the actual live state, and automatically reconciles any drift.13

According to the Cloud Native Computing Foundation (CNCF), GitOps adoption has reached mainstream status, with 100% of survey respondents stating they plan to embrace GitOps methodology within the next two years.1

"GitOps gives you version-controlled deployments, easier rollbacks via Git history, auditable infrastructure changes, faster onboarding for new team members, and better separation of concerns between developers and operations."12

Core GitOps Principles

1
Declarative Infrastructure

All cluster state is expressed declaratively in Git — Kubernetes manifests, Helm values files, Kustomize overlays. There are no imperative shell commands in the delivery path.

2
Git as the Source of Truth

Every change to production begins as a pull request. Git history provides a full audit trail of who changed what and when, without separate change management tooling.

3
Automated Reconciliation

An in-cluster operator continuously observes both Git state and cluster state, and automatically corrects any divergence — providing self-healing infrastructure.14

4
Rollback via Git Revert

Rolling back a bad deployment is as simple as reverting the offending commit. The operator detects the change and re-converges the cluster to the previous known-good state.

Leading GitOps Tools: Argo CD vs Flux CD

Two tools dominate the GitOps landscape for Kubernetes. Both are CNCF graduated projects that monitor Git repositories and reconcile cluster state.15

Aspect Argo CD Flux CD
Architecture API server + repo server + controller + Redis15 Modular controllers, CLI-driven16
UI Rich Web UI CLI-centric
Multi-cluster Excellent — centralised dashboards17 Good — native OCI/S3 source support16
Best for Enterprise governance, visibility16 Automation-first, resource-efficient16

Package Management

Helm Charts

Helm is the package manager for Kubernetes.18 As microservices multiply, each with their own Deployments, Services, ConfigMaps, and Ingresses, manually managing those manifests becomes error-prone and time-consuming — a problem the community has dubbed "YAML sprawl".19 Helm addresses this by bundling related Kubernetes resources into reusable, versioned units called charts, making deployments reproducible, configurable, and version-controlled.19

"Think of Helm like apt, yum, or Homebrew for Kubernetes. Helm renders your templates and communicates with the Kubernetes API from wherever you want to run it — a laptop, CI/CD pipeline, or any other environment."20

Helm reached a major milestone in late 2025 with the release of version 4.0.0 — the first major upgrade in six years and Helm's 10th anniversary under CNCF stewardship.21 Helm 4 introduces native support for server-side apply, a rebuilt plugin system with WebAssembly support, modern Go logging interfaces, and improvements in chart signing and testing automation.21 As of April 2026, the stable release is v4.1.4.22

Core Concepts

C
Chart

The package — a directory with a defined structure containing templates, default values, and metadata. Charts can be published to repositories like Artifact Hub.

R
Release

An installed instance of a chart. You can install the same chart multiple times under different release names; each is an independent, versioned instance.23

V
Values

A values.yaml file provides defaults that consumers can override at install time. When helm install runs, Helm injects values into templates and renders standard Kubernetes YAML.23

Chart Directory Structure

Shell — Chart Layout 
myapp/
├── Chart.yaml          # chart metadata (name, version, appVersion)
├── values.yaml         # default configuration values
├── templates/
│   ├── deployment.yaml # Go-templated Kubernetes manifests
│   ├── service.yaml
│   ├── ingress.yaml
│   └── _helpers.tpl    # reusable template partials
└── charts/             # sub-chart dependencies

Common Helm Commands

Shell — Helm Lifecycle 
# Install a release
helm install my-nginx bitnami/nginx \
  --namespace production --create-namespace

# Upgrade with new values
helm upgrade my-nginx bitnami/nginx \
  --set replicaCount=3 --namespace production

# View release history
helm history my-nginx --namespace production

# Roll back to a previous revision
helm rollback my-nginx 2 --namespace production

# Render templates locally (dry-run)
helm template my-nginx ./myapp -f values-prod.yaml

# Validate chart for errors
helm lint ./myapp

Example: Parameterised Deployment Template

YAML — templates/deployment.yaml 
apiVersion: apps/v1
kind: Deployment
metadata:
  name: {{ .Release.Name }}-{{ .Chart.Name }}
  labels:
    app.kubernetes.io/managed-by: Helm
    app.kubernetes.io/version: {{ .Chart.AppVersion | quote }}
spec:
  replicas: {{ .Values.replicaCount }}
  template:
    spec:
      containers:
        - name: {{ .Chart.Name }}
          image: "{{ .Values.image.repository }}:{{ .Values.image.tag }}"
          resources:
            {{- toYaml .Values.resources | nindent 12 }}

Progressive Delivery

Canary Deployments

The canary deployment technique gradually rolls out a new version of an application by initially directing a small percentage of users or traffic to it, while the majority continues using the stable version.24 The name comes from the "canary in a coal mine" — miners used canaries to detect toxic gases; in software terms, the canary release detects production problems by exposing only a fraction of traffic to potential issues.25

"No matter how thorough your testing, production traffic has a way of exposing bugs that staging never could. Canary deployment is a battle-tested strategy that lets you ship changes incrementally, catching problems before they affect all your users."25

Traffic-Splitting Approaches

Method Mechanism Granularity Best For
Replica Ratio Kubernetes round-robin load balancing across pods Coarse (≈ replica %) Simple canaries without a service mesh25
Ingress Weights NGINX / Traefik annotations on Ingress resources Medium Lightweight, edge-level splitting26
Service Mesh Istio / Linkerd VirtualService weights Precise (exact %) Microservice architectures, header-based routing27
Progressive Operator Flagger / Argo Rollouts automate shifting + analysis Fully automated Large-scale production with metric-driven promotion26

Example: Replica-Based Canary

YAML — Stable + Canary Deployments 
# 9 stable replicas → ~90% traffic
apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp-stable
spec:
  replicas: 9
  selector:
    matchLabels: { app: myapp }
  template:
    metadata:
      labels: { app: myapp, version: stable }
    spec:
      containers:
        - { name: myapp, image: myapp:v1 }
---
# 1 canary replica → ~10% traffic
apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp-canary
spec:
  replicas: 1
  selector:
    matchLabels: { app: myapp }
  template:
    metadata:
      labels: { app: myapp, version: canary }
    spec:
      containers:
        - { name: myapp, image: myapp:v2 }
---
# Single Service selects both via shared 'app' label
apiVersion: v1
kind: Service
metadata:
  name: myapp
spec:
  selector: { app: myapp } # matches ALL pods

Scaling the Canary

Shell — Progressive Traffic Increase 
# Phase 1: 10% canary (1 / 10 total)
kubectl scale deployment myapp-canary --replicas=1
kubectl scale deployment myapp-stable --replicas=9

# Phase 2: 50% canary (5 / 10 total) — after metrics look healthy
kubectl scale deployment myapp-canary --replicas=5
kubectl scale deployment myapp-stable --replicas=5

# Phase 3: 100% — stable removed, canary becomes production
kubectl scale deployment myapp-stable --replicas=0

# Emergency rollback: scale canary to zero instantly
kubectl scale deployment myapp-canary --replicas=0

Real-World Example: E-commerce Checkout

Consider an e-commerce platform rolling out a redesigned checkout system. Instead of releasing the new flow to all customers simultaneously, the team directs 5% of traffic to the canary version. If metrics show improved conversion rates with no increase in error rates, traffic is gradually increased until 100% of customers use the new system. If problems surface, traffic is immediately rolled back to the old version, minimising disruption while still gaining valuable production insights.26

Monitoring Requirements

A successful canary rollout requires continuous monitoring of both stable and canary versions. Standard metrics to track include HTTP error rates, p99 latency, CPU and memory usage, and business-level KPIs. Prometheus and Grafana are commonly deployed for real-time dashboarding and alert-driven rollback triggers.28

Default Strategy

Rolling Updates & Rollbacks

Rolling updates are the default deployment strategy in Kubernetes. They enable application updates without downtime by gradually replacing old pods with new ones, ensuring that a minimum level of service remains available throughout the process.29 Unlike Recreate deployments — which terminate all existing pods before starting new ones, causing unavoidable downtime — RollingUpdate coordinates old and new replica sets simultaneously.30

"The primary advantage of rolling updates is zero-downtime deployment, which is critical for production environments. Kubernetes retains a history of deployments, enabling easy rollbacks to a previous stable version if issues are detected."31

Key Parameters

maxSurge Max extra pods above desired count32
maxUnavailable Max pods offline simultaneously32
minReadySeconds Soak time before pod is considered available

Both maxSurge and maxUnavailable can be specified as absolute numbers or percentages, giving teams fine-grained control scaled to their deployment size.33

For example, setting maxSurge: 2 and maxUnavailable: 1 on a 5-replica deployment allows Kubernetes to create up to 7 pods during the update while never dropping below 4 ready pods.34 If you prioritise rollout safety, setting maxUnavailable: 0 provides a strong guarantee that the desired replica count is maintained throughout.35

Rolling Update YAML

YAML — RollingUpdate Strategy 
apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp
spec:
  replicas: 5
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 2       # at most 7 pods total during update
      maxUnavailable: 1  # at least 4 pods always ready
  minReadySeconds: 30  # soak 30s before considering pod healthy
  selector:
    matchLabels:
      app: myapp
  template:
    metadata:
      labels:
        app: myapp
    spec:
      containers:
        - name: myapp
          image: myapp:2.0.0
          readinessProbe:         # essential for safe rolling updates
            httpGet:
              path: /healthz
              port: 8080
            initialDelaySeconds: 10
            periodSeconds: 5

Rollback Commands

When a bad deployment is detected, Kubernetes can immediately roll back to the previous revision. The Deployment controller initiates a new rollout using the previous configuration, processed as another rolling update.35

Shell — Rollback Operations 
# View rollout status in real time
kubectl rollout status deployment/myapp

# View revision history
kubectl rollout history deployment/myapp

# Roll back to the previous revision
kubectl rollout undo deployment/myapp

# Roll back to a specific revision number
kubectl rollout undo deployment/myapp --to-revision=3

# Pause a rollout mid-flight (e.g., to investigate metrics)
kubectl rollout pause deployment/myapp

# Resume a paused rollout
kubectl rollout resume deployment/myapp

Pitfalls to Avoid

PitfallImpactMitigation
Using :latest image tag Rollbacks reference a mutable tag — unpredictable33 Pin image tags to immutable digests or semantic versions
Missing readiness probes Traffic sent to pods that are not yet ready Define readinessProbe on every container
No graceful shutdown Long-lived connections (gRPC, WebSocket) severed35 Implement preStop hooks and terminationGracePeriodSeconds
Generic maxSurge/maxUnavailable defaults May overload cluster or reduce availability too far35 Tune based on workload capacity headroom and startup time

Summary

Strategy Comparison

Strategy Downtime Rollback Resource Cost Complexity Best Use Case
Rolling Update None Automatic via kubectl Low Low Default; most stateless services
Blue-Green None Instant selector flip High (2×) Medium Zero-risk production cut-overs
Canary None Scale canary to 0 Low–Medium High Risk-sensitive feature releases
Recreate Yes Re-deploy old version Very Low Very Low Dev/non-critical; stateful workloads

Decision Guide

Strategy Selection Decision Tree
Can you tolerate any downtime? ├── No → Is instantaneous rollback required? │ ├── Yes → Can you afford 2× resources? │ │ ├── Yes → Blue-Green Deployment │ │ └── No → Rolling Update │ └── No → Do you need gradual traffic exposure? │ ├── Yes → Canary Deployment │ └── No → Rolling Update └── Yes → Is this dev/staging or a stateful workload? └── Yes → Recreate Strategy
