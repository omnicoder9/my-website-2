§ 01

The Problem, Precisely

"There are only two hard things in Computer Science: cache invalidation and naming things."
— Phil Karlton, Netscape principal engineer, circa 1996–1997[1]

Phil Karlton's aphorism is so widely repeated that its origins have become apocryphal — Tim Bray, who heard it directly, is among the few primary sources.[1] The witticism lands because it is simultaneously a joke and a precise technical claim. Naming is a problem of abstraction and human cognition. Cache invalidation is a problem of distributed agreement about the truth of a mutable world in the presence of bounded observation and partial failure. It just looks simpler.

The core tension is this: caching exploits the temporal and spatial locality of data access to trade memory capacity for speed. You keep a copy of something close to the consumer so it needn't be fetched from the authoritative source on every request. The moment the authoritative source changes, you have a consistency problem. The cached copy is now a lie. The question is: how do you know? When do you care? What do you do about it?

These deceptively simple questions force a confrontation with the most uncomfortable tradeoffs in distributed systems: consistency vs. availability, freshness vs. throughput, and simplicity vs. correctness. As Brewer's CAP theorem formalizes, a distributed system cannot simultaneously guarantee consistency, availability, and partition tolerance.[2] Every caching architecture encodes an implicit choice among these dimensions, usually informally, often wrongly.

⚠ The fundamental difficulty

Cache invalidation is hard not because the algorithms are complex, but because it requires solving distributed consensus under real-world constraints: network partitions, clock skew, concurrent writes, partial failures, and indefinite latency. Any solution involves compromises that most systems teams do not make explicitly.

What makes invalidation distinct from related problems — consistency, replication, synchronization — is that it is asymmetric. Writes are rare; reads are common. A cache exists precisely because reads dominate. The write path is the exceptional case that must somehow reach every cached copy, including those whose existence the writer may not know about. This is fundamentally an observer problem: the thing being observed must somehow notify observers it does not know exist.

Before examining invalidation strategies, it helps to understand the layers at which caching occurs, because the invalidation mechanisms appropriate to each layer differ dramatically.

The consistency model weakens as we move down the stack, not by accident but by necessity: the farther a cache sits from the authoritative source, the higher the network cost of guaranteeing freshness, and the higher the traffic justifying the cache in the first place. The browser cache protecting a single user's session and the CDN edge node serving millions of concurrent users are solving the same abstract problem but with dramatically different operational constraints.

§ 03

Historical Arc

Cache invalidation is not a problem invented by the web. It is a fundamental tension that emerged the moment computer architects noticed that main memory was slower than computation — a gap that has only widened over six decades.

1965
Wilkes proposes cache memories

Maurice Wilkes introduced the concept of a "slave memory" — a small fast store acting as a transparent intermediary between the processor and main store — in a landmark paper at the IFIP Congress.[3] Wilkes recognized that the slave must be kept coherent with main store, a problem he considered "the principal difficulty."

1968
IBM System/360 Model 85 — first commercial CPU cache

The IBM 360/85 shipped with a 16 KB cache, managing coherence in hardware. With only a single CPU sharing memory, the invalidation problem was manageable: writes directly updated the cache and main memory (write-through).[4]

1983
Goodman formalizes cache coherence

James Goodman's landmark paper on snoopy cache-based multiprocessors laid the formal groundwork for bus-snooping coherence protocols, demonstrating how caches on a shared bus could observe each other's traffic to maintain consistency.[5]

1984
MESI protocol — the Illinois Protocol

Papamarcos and Patel at the University of Illinois published the MESI protocol, introducing four cache-line states (Modified, Exclusive, Shared, Invalid) and a snooping mechanism that remains the foundation of virtually every modern multicore processor.[6]

1995
HTTP caching — RFC 1945 / 2068

HTTP/1.0 introduced the Expires header; HTTP/1.1 (RFC 2068, then RFC 2616) introduced the Cache-Control directive family, ETags, conditional GET, and the Vary header — establishing the web's TTL-plus-validation invalidation model.[7]

1997
Consistent hashing — distributing cache space

Karger et al. published consistent hashing at STOC, solving the problem of distributing cache entries across a cluster with minimal reshuffling when nodes are added or removed.[8] This made large-scale distributed caches viable.

2003
Memcached — distributed caching enters the mainstream

Brad Fitzpatrick built Memcached for LiveJournal over a weekend, solving a database bottleneck that had made the site unusable at scale.[9] Memcached made a deliberate design choice: no invalidation primitives beyond TTL and explicit delete. Simplicity over feature richness.

2006
Varnish Cache — VCL and BAN invalidation

Poul-Henning Kamp designed Varnish Cache with a programmable invalidation layer (VCL) and introduced the BAN mechanism: rather than purging individual objects, a BAN is a predicate stored in memory and evaluated lazily as objects are fetched — allowing O(1) invalidation of arbitrarily large cache populations.[10]

2009
Redis — rich invalidation semantics

Salvatore Sanfilippo released Redis with built-in pub/sub, keyspace notifications, Lua scripting, and expiry. Redis gave applications the primitives to build event-driven invalidation on top of a distributed cache, a significant architectural leap beyond Memcached's key-value delete model.[11]

2013
Facebook Memcache paper — invalidation at planetary scale

Nishtala et al. published the definitive account of caching at Facebook's scale — trillions of cache lookups per day — documenting novel solutions to invalidation during replication lag (mcrouter's "delete replication"), thundering herds (leases), and stale reads during failover (cold cluster warmup).[12]

2015–
Cache tags / surrogate keys at the CDN edge

Fastly introduced surrogate keys; Cloudflare introduced cache tags; Varnish formalized the concept in core. This allowed CDN operators to invalidate structured subsets of edge-cached content in milliseconds, collapsing the historically-enormous gap between "delete everything" and "delete one URL."[13]

§ 04

Invalidation Strategies

There is no universally optimal cache invalidation strategy. The choice depends on data mutability, read:write ratios, acceptable staleness windows, operational complexity budget, and the consistency requirements of the consuming application. What follows is a comprehensive taxonomy of the principal strategies with their mechanics, tradeoffs, and appropriate use cases.

4.1 TTL-Based Expiry (Time-to-Live)

The simplest and most widely deployed strategy: cache entries carry a maximum age, after which they are treated as expired and must be re-fetched. No write-time coordination is required. The cache is self-healing: stale data will eventually be replaced regardless of producer behavior. The staleness window is bounded by the TTL value.

Tradeoff

TTL conflates two distinct questions: "Is this entry still valid?" and "How long should I keep it?" Data with highly variable mutation rates (a product price vs. a user's name) cannot be efficiently served by a single TTL policy. Short TTLs reduce staleness but increase origin load. Long TTLs improve hit ratios but extend inconsistency windows after writes.

The correct TTL is often unknown at write time, and wrong in any case because data lifetimes are non-uniform. A TTL of 60 seconds on a configuration entry that changes once a year is wasteful; a TTL of 60 seconds on a real-time inventory count is stale within milliseconds of a sale. Systems that use a single TTL policy for all data are implicitly accepting this mismatch.

Python TTL Cache with per-key expiry (simplified) 
import time
import threading
from typing import Any, Optional
from dataclasses import dataclass, field

@dataclass
class CacheEntry:
    value:      Any
    expires_at: float    # absolute epoch seconds

class TTLCache:
    """Thread-safe TTL cache with per-key expiry and passive expiration."""

    def __init__(self, default_ttl: float = 60.0) -> None:
        self._store: dict[str, CacheEntry] = {}
        self._lock  = threading.RLock()
        self.default_ttl = default_ttl

    def set(self, key: str, value: Any, ttl: Optional[float] = None) -> None:
        ttl = ttl if ttl is not None else self.default_ttl
        with self._lock:
            self._store[key] = CacheEntry(
                value=value,
                expires_at=time.monotonic() + ttl
            )

    def get(self, key: str) -> Optional[Any]:
        with self._lock:
            entry = self._store.get(key)
            if entry is None:
                return None                              # cold miss
            if time.monotonic() > entry.expires_at:
                del self._store[key]                        # passive expiry
                return None                              # stale miss
            return entry.value

    def invalidate(self, key: str) -> bool:
        with self._lock:
            return self._store.pop(key, None) is not None

    def _expired(self, entry: CacheEntry) -> bool:
        return time.monotonic() > entry.expires_at

4.2 Write-Through

On every write, the cache and the authoritative store are updated atomically (or near-atomically) before the write is acknowledged to the caller. The cache is therefore always consistent with the store — at the cost of write latency, since the write path now includes both the cache layer and the store.

Write-through is most appropriate when reads are frequent and writes are rare, when read latency is critical, and when the cache and store are co-located or connected by a reliable, low-latency link. It does not handle external mutations — writes made directly to the underlying store bypassing the cache will still create stale entries.

4.3 Write-Back (Write-Behind)

Writes are acknowledged to the caller immediately after updating the cache; the write to the authoritative store is deferred and batched. This dramatically reduces write latency and can coalesce multiple writes to the same key into a single store operation. The tradeoff is durability: data in the cache-but-not-the-store is vulnerable to loss if the cache node fails before the write is flushed.

Write-back is the dominant policy in CPU L1/L2 caches, where the performance gain is essential and the hardware can coordinate coherence. At the application level it requires careful handling of failure modes (write-ahead logging, idempotent retries) to avoid data loss.

4.4 Cache-Aside (Lazy Population)

The application is explicitly responsible for cache population and invalidation. On a cache miss, the application fetches the data from the authoritative store, populates the cache, and returns the value. On a write, the application updates the store and then either deletes or updates the cache entry. The cache is never written directly by the producer; it is always populated lazily by the consumer on miss.

Go Cache-aside pattern with Redis and invalidation-on-write 
package cache

import (
    "context"
    "encoding/json"
    "errors"
    "fmt"
    "time"

    "github.com/redis/go-redis/v9"
)

type UserRepository struct {
    redis *redis.Client
    db    Database       // authoritative store
    ttl   time.Duration
}

const userKeyFmt = "user:%d"

// GetUser implements cache-aside: check cache, fall back to DB, populate cache.
func (r *UserRepository) GetUser(ctx context.Context, id int64) (*User, error) {
    key := fmt.Sprintf(userKeyFmt, id)

    // 1. Check cache.
    raw, err := r.redis.Get(ctx, key).Bytes()
    if err == nil {
        var u User
        if jsonErr := json.Unmarshal(raw, &u); jsonErr == nil {
            return &u, nil   // cache hit
        }
    } else if !errors.Is(err, redis.Nil) {
        return nil, fmt.Errorf("redis GET: %w", err)
    }

    // 2. Cache miss — fetch from authoritative store.
    u, err := r.db.GetUser(ctx, id)
    if err != nil {
        return nil, fmt.Errorf("db GetUser: %w", err)
    }

    // 3. Populate cache asynchronously (fire-and-forget with deadline).
    go func() {
        data, _ := json.Marshal(u)
        _ = r.redis.SetEx(ctx, key, data, r.ttl).Err()
    }()

    return u, nil
}

// UpdateUser writes through to DB first, then invalidates the cache entry.
func (r *UserRepository) UpdateUser(ctx context.Context, u *User) error {
    // CRITICAL: Update the DB *before* invalidating the cache.
    // Reverse order creates a race: another goroutine repopulates the cache
    // with stale data between the delete and the DB write completing.
    if err := r.db.UpdateUser(ctx, u); err != nil {
        return fmt.Errorf("db UpdateUser: %w", err)
    }
    key := fmt.Sprintf(userKeyFmt, u.ID)
    if err := r.redis.Del(ctx, key).Err(); err != nil {
        // Non-fatal: cache will expire eventually. Log and continue.
        logWarn("cache invalidation failed for key %s: %v", key, err)
    }
    return nil
}
The update-then-invalidate ordering requirement

In cache-aside, the ordering of "write to store" and "delete from cache" is not arbitrary. Deleting the cache entry before the store write completes opens a race: a concurrent reader triggers a miss, queries the store, finds the old value, and repopulates the cache — all before your write lands. The new write then invalidates a now-stale entry, but the cache repopulation with old data has already occurred. Write to the authoritative store first; invalidate the cache after.

4.5 Read-Through

Unlike cache-aside, where the application orchestrates cache misses, read-through places the miss-handling logic inside the cache layer itself. On a miss, the cache fetches from the store, populates itself, and returns the value — the application never talks to the store directly. This simplifies application code but couples the cache to the store interface.

4.6 Event-Driven (Push) Invalidation

Rather than relying on TTL expiry or write-time coordination, event-driven invalidation uses a message bus or pub/sub system to broadcast invalidation signals when the authoritative source changes. Cache consumers subscribe to these signals and either delete the relevant entries or schedule background refreshes.

Python Redis pub/sub cache invalidation — publisher and subscriber 
# ── Publisher: runs in the service that owns the data ──────────────────────
import redis, json

r = redis.Redis(host='localhost', port=6379, decode_responses=True)

def update_product(product_id: int, new_price: float) -> None:
    # 1. Persist to authoritative store (DB omitted for brevity).
    db.execute("UPDATE products SET price=? WHERE id=?", new_price, product_id)

    # 2. Publish invalidation event on a typed channel.
    payload = json.dumps({"entity": "product", "id": product_id, "op": "update"})
    vr.publish("cache_invalidation:product", payload)


# ── Subscriber: runs in every service instance with a product cache ────────
import threading

local_cache: dict[str, Any] = {}
pubsub = vr.pubsub()
pubsub.subscribe("cache_invalidation:product")

def _invalidation_worker() -> None:
    for message in pubsub.listen():
        if message["type"] != "message":
            continue
        event = json.loads(message["data"])
        cache_key = f"product:{event['id']}"
        local_cache.pop(cache_key, None)    # evict from process-local cache
        vr.delete(cache_key)                  # evict from shared Redis cache

# Run listener in daemon thread; or use an async event loop.
threading.Thread(target=_invalidation_worker, daemon=True).start()

Event-driven invalidation can achieve near-instantaneous consistency across all caches in a cluster, but it introduces coupling to the message bus, requires exactly-once or at-least-once delivery guarantees, and must handle the window between the write and the delivery of the invalidation event (during which any reader may serve stale data). It also does not handle the case where a subscriber is temporarily offline — on reconnect it must either re-subscribe from a durable log or fall back to TTL-bounded staleness.

4.7 Stale-While-Revalidate

Formalised in RFC 5861, stale-while-revalidate decouples serving from freshness verification.[14] When a cached entry is past its max-age but within the stale-while-revalidate window, the cache serves the stale copy immediately and triggers an asynchronous background revalidation with the origin. The user experiences zero additional latency; the cache self-heals within one request cycle.

HTTP stale-while-revalidate response header 
# Serve fresh for 60 s. After 60 s, serve stale while revalidating for up to 3600 s.
Cache-Control: max-age=60, stale-while-revalidate=3600

# Serve stale if the origin is unreachable for up to 86400 s (1 day).
Cache-Control: max-age=60, stale-if-error=86400

This pattern shifts the consistency model from "never serve stale" to "serve stale for a bounded window." It is particularly well-suited to content that changes infrequently but must appear fresh and where revalidation latency would be user-visible. It should not be used for data with hard consistency requirements: financial balances, inventory counts, access control rules.

§ 05

CPU Cache Coherence: The MESI Protocol

The most thoroughly engineered cache invalidation system in existence is not a web framework — it is the cache coherence hardware in every modern multicore processor. The MESI protocol (Modified, Exclusive, Shared, Invalid), introduced by Papamarcos and Patel in 1984,[6] defines a finite state machine for each cache line that governs when a cache may serve a local copy and when it must invalidate.

M
Modified
This cache has the only up-to-date copy. Main memory is stale. The cache must write back before another core can read.
E
Exclusive
This cache has the only copy; main memory is also current. The line can be written without broadcasting an invalidation — transition to M is silent.
S
Shared
Multiple caches hold clean copies. A write requires broadcasting an invalidation to all sharers before transitioning to M.
I
Invalid
The cache line does not contain valid data. A read triggers a cache miss and a fetch from another cache or main memory.

Coherence is maintained by a snooping mechanism: every cache controller monitors the shared bus (or in modern ring/mesh topologies, the interconnect) and reacts to transactions it observes. When core A writes to a cache line currently in state S on core B, the bus transaction broadcasts an invalidation; core B transitions its copy from S to I. Core A transitions from S to M. If core B subsequently reads the line, it sees a miss, signals a cache-to-cache transfer, and core A writes back.

State transitions for a MESI cache line (from the perspective of one core): ┌─────────────────────────────────────────────────────────────────────────┐ │ │ │ ┌───────┐ │ │ local read ───── │ │ ──── local write ────────────┐ │ │ (hit) │ E │ │ │ │ │ │ ▼ │ │ ┌──────────────►│Exclus.│◄─────────────────────────┌─────┐ │ │ │ └───────┘ │ │ │ │ │ │ bus read (other core) │ M │ │ │ │ │ → transition to Shared │Modif│ │ │ │ ▼ │ │ │ │ local ─┤ ┌───────┐ local write → └──┬──┘ │ │ read │ snoop ◄──│ │──── invalidate all sharers │ │ │ (miss) │ write │ S │ → transition to M │ │ │ │ │ │ │ │ │ │ │Shared │◄──── bus read (shared hit) ───┘ │ │ │ └───────┘ │ │ │ │ snoop write / bus invalidate │ │ │ ▼ │ │ └──────────────┌───────┐ │ │ │ │◄── any operation on invalid line │ │ │ I │ │ │ │Invalid│ │ │ └───────┘ │ └─────────────────────────────────────────────────────────────────────────┘

Figure 1 — Simplified MESI state transitions; actual protocols add HITM (hit-in-modified) states and snoop filter optimisations.

Modern processors extend MESI with additional states: MOESI (Modified, Owned, Exclusive, Shared, Invalid) adds an Owned state that allows a cache to supply data to other caches without writing back to memory, used in AMD processors. Intel's MESIF adds a Forward state designating one cache as the designated responder among sharers.[15]

TLB Shootdowns: OS-Level Cache Invalidation

The Translation Lookaside Buffer (TLB) is a cache of virtual-to-physical address mappings. When the OS modifies a page table entry — on munmap, fork/CoW, or permission change — it must invalidate the corresponding TLB entries on all CPUs that might have cached them. This is a TLB shootdown.

On x86, this is accomplished by sending an inter-processor interrupt (IPI) to each affected CPU, which executes INVLPG to flush the specific mapping or MOV CR3 to flush the entire TLB. TLB shootdowns are a significant source of latency in systems with many CPUs and high virtual memory churn — a reminder that hardware-level cache invalidation carries real costs even at nanosecond timescales.[16]

§ 06

Distributed Cache Invalidation

When caches are distributed across multiple nodes — as in any horizontally-scaled service — the invalidation problem acquires the full weight of distributed systems theory. A write on one node must reach every cache replica across the cluster, but the network is unreliable, nodes may fail mid-operation, and you cannot know in advance which nodes have cached a given key.

6.1 Delete Replication: Facebook's Approach

At Facebook's scale (trillions of cache lookups per day), the dominant approach for cache invalidation is delete replication via a dedicated invalidation daemon — McSqueal (later mcrouter).[12] When a MySQL replica applies a binlog event representing a write, McSqueal reads the binlog, derives the affected cache keys, and sends DELETE commands to the appropriate Memcached pools.

This architecture has important properties: it is asynchronous, it decouples the application write path from cache invalidation latency, and it provides a durable record (the binlog) of all mutations that can be replayed on reconnect. The tradeoff is that the cache remains stale for the replication lag window — typically tens of milliseconds, occasionally longer during failover.

6.2 Leases: Solving the Thundering Herd on Invalidation

The Facebook Memcache paper introduced leases to address two problems simultaneously: stale sets and thundering herds after invalidation.[12]

When a key is invalidated, the next request that misses receives a lease token — a unique identifier that grants exclusive write permission for that key for a short window (typically 10 seconds). Subsequent requests that also miss are asked to wait or retry. When the key-holder fetches the data and calls set with the lease token, the cache validates the token and accepts the write. If the token has been invalidated by a subsequent write, the set is rejected and the key-holder discards its data.

This is, in essence, an optimistic lock at the cache layer — preventing concurrent repopulation by racing readers and preventing stale sets from an old computation completing after a write has already invalidated and repopulated the key.

6.3 Cache Invalidation in Event-Sourced Systems

Event sourcing — where the authoritative state is a log of events rather than a snapshot — provides a natural invalidation substrate. Every write is an event published to a durable log (Kafka, Kinesis, Pulsar). Cache consumers can subscribe to the event stream and invalidate (or recompute) cache entries as events arrive. The log provides durability, ordering, and a replay mechanism for consumers that fall behind.

The fundamental advantage over ad-hoc pub/sub is that the event log is the authoritative record. A consumer that was offline for an hour can replay the missed events and bring its cache to full consistency with the source — without polling, without TTL gaps, and without relying on a separate invalidation channel that might have different reliability characteristics than the write path.

Python Kafka-driven cache invalidation consumer 
from confluent_kafka import Consumer
import json, redis

cache = redis.Redis(host="localhost", decode_responses=True)

consumer = Consumer({
    "bootstrap.servers": "kafka:9092",
    "group.id":          "cache-invalidator",
    "auto.offset.reset": "earliest",   # replay from beginning on restart
    "enable.auto.commit": False,
})
consumer.subscribe(["domain.events"])

def cache_key_for_event(event: dict) -> str | None:
    """Derive the cache key affected by an event. Return None if not cacheable."""
    entity, op, eid = event.get("entity"), event.get("op"), event.get("id")
    if op not in {"updated", "deleted"}:
        return None
    return f"{entity}:{eid}"

try:
    while True:
        msg = consumer.poll(timeout=1.0)
        if msg is None or msg.error():
            continue
        event = json.loads(msg.value())
        key   = cache_key_for_event(event)
        if key:
            cache.delete(key)
            print(f"Invalidated: {key} (offset {msg.offset()})")
        consumer.commit(asynchronous=False)  # commit after successful invalidation
finally:
    consumer.close()
§ 07

HTTP and CDN Caching

HTTP caching is governed by RFC 7234,[17] which defines a layered model of cache directives, validation mechanisms, and freshness computation. Unlike most distributed cache systems, HTTP caching operates through the request/response layer — the cache is a man-in-the-middle between client and server, and invalidation must work through the same HTTP interface.

7.1 Freshness and Validation

RFC 7234 defines freshness as a function of the response's age relative to its max-age or Expires header. A fresh response may be served without contacting the origin. A stale response must be validated with a conditional request before serving.

The two validation mechanisms are: ETags (entity tags — a hash or opaque version identifier for the response body, sent in If-None-Match conditional requests) and Last-Modified dates (sent in If-Modified-Since requests). If the origin responds with 304 Not Modified, the cache updates its headers and serves the existing body. If the origin responds with 200, the cache replaces its stored copy.

The critical insight is that HTTP caching has no push invalidation model in the base protocol. The origin cannot reach out and tell caches to invalidate a stored response — it can only respond to validation requests. This is why CDN cache invalidation APIs exist as out-of-band mechanisms: they are compensating for a fundamental gap in the HTTP model.

7.2 The Vary Header: Caching's Silent Complicator

The Vary header instructs caches that a response's cacheability depends on specific request headers. A response with Vary: Accept-Language must be cached separately for each language variant requested. A response with Vary: Cookie is essentially uncacheable at a shared cache because every user has unique cookies.

Vary does not affect invalidation directly, but it dramatically complicates it: to invalidate a resource that varies, you must invalidate every variant. CDNs typically handle this by storing all variants under a common cache key namespace and purging the entire namespace on invalidation — which is one of the motivating use cases for surrogate keys.

7.3 CDN Invalidation: APIs and Propagation

CDN invalidation is a fundamentally different problem from application-level cache invalidation. A CDN operates hundreds of points of presence (PoPs) globally, each caching potentially millions of objects. When you publish a new version of a CSS file at /static/styles.css, you need every PoP to stop serving the old version — across multiple continents, within seconds, without affecting availability.

Major CDN providers offer URL-based purge APIs, but these have significant limitations. Akamai's "Fast Purge" API promises propagation within 5 seconds globally.[18] Cloudflare's Cache Purge API accepts up to 30 URLs per request. Purging by prefix or pattern is available from some providers. Purging an entire zone ("cache everything for this domain") is the nuclear option — it works, but it guarantees an origin flood as the CDN cache warms up from scratch.

§ 08

Surrogate Keys and Cache Tags

Surrogate keys (Fastly) and cache tags (Cloudflare, Varnish) are among the most significant innovations in HTTP cache invalidation of the past decade.[13] They solve a fundamental mismatch: URL-based purging is a one-to-one operation (one URL per purge), but the relationship between content and cached objects is many-to-many.

Consider a product page on an e-commerce site. It caches images, prices, availability data, related recommendations, and category breadcrumbs — all potentially from different data sources with different mutation rates. A single change to the product record should invalidate all cache objects that include product data for that SKU. Without surrogate keys, you need to know every URL that includes that product and purge each individually — which is impossible at scale.

How Surrogate Keys Work

The origin server annotates each response with a Surrogate-Key (Fastly) or Cache-Tag (Cloudflare) header containing a space-separated list of opaque identifiers representing the content's logical dependencies:

HTTP Surrogate key / cache tag response headers 
# Fastly surrogate keys
HTTP/1.1 200 OK
Surrogate-Key: product-42371 category-electronics brand-samsung price-tier-2
Surrogate-Control: max-age=3600

# Cloudflare cache tags
HTTP/1.1 200 OK
Cache-Tag: product-42371,category-electronics,brand-samsung
Cache-Control: public, max-age=3600

# Varnish (xkey module)
HTTP/1.1 200 OK
xkey: product-42371 category-electronics brand-samsung

The CDN stores the mapping from each tag to every cached object annotated with it. When a product record changes, the origin calls the CDN purge API with the tag product-42371. The CDN invalidates every cached object across every PoP that carries that tag — potentially thousands of objects — in a single API call, typically within seconds.

Python Fastly surrogate key purge on data update
import httpx, os FASTLY_SERVICE_ID = os.environ["FASTLY_SERVICE_ID"] FASTLY_API_TOKEN = os.environ["FASTLY_API_TOKEN"] async def purge_surrogate_key(key: str) -> None: """ Invalidate all CDN objects tagged with `key` across all PoPs. Uses Fastly's soft-purge (marks stale, allows stale-while-revalidate). """ url = f"https://api.fastly.com/service/{FASTLY_SERVICE_ID}/purge/{key}" async with httpx.AsyncClient() as client: resp = await client.post( url, headers={ "Fastly-Key": FASTLY_API_TOKEN, "Fastly-Soft-Purge": "1", # soft-purge: serve stale until fresh } ) resp.raise_for_status() async def update_product(product_id: int, data: dict) -> None: await db.update_product(product_id, data) # write to DB await purge_surrogate_key(f"product-{product_id}") # invalidate all CDN copies # All pages, API responses, images tagged product-{product_id} are now stale.

Varnish's implementation — the xkey VMOD — uses a BAN-based approach internally. A BAN is a stored predicate evaluated lazily against cached objects as they are fetched, avoiding the need to maintain an explicit tag-to-object index in memory. The tradeoff is that stale objects are not immediately purged from memory — they are only invalidated when next requested, which means the memory footprint is not immediately reduced.[10]

§ 09

Failure Modes: When Cache Invalidation Goes Wrong

Cache invalidation failures are a category of bug that is disproportionately severe: they tend to be silent (the system appears functional), persistent (they self-heal only after TTL expiry or a deployment), and potentially catastrophic (serving incorrect data to users at scale). Here are the principal failure modes.

9.1 The Cache Stampede (Thundering Herd)

When a popular cache entry expires or is invalidated, the next N requests all see a miss simultaneously and all proceed to query the origin. If N is large and the origin query is expensive, the origin is overwhelmed — potentially cascading into a full outage. The cache, designed to protect the origin, has become the mechanism of its destruction.

The canonical solution is probabilistic early expiration, proposed by Vattani et al.[19] Rather than serving from cache until the exact TTL boundary and then causing a synchronized miss, each request uses a randomized function that increases the probability of early revalidation as the TTL approaches zero. The first request to decide to revalidate (probabilistically) does so in the background; subsequent requests continue to serve from cache.

Python Probabilistic early expiration (XFetch algorithm) 
import time, math, random
from dataclasses import dataclass
from typing import Callable, Any

@dataclass
class XFetchEntry:
    value:      Any
    expires_at: float    # absolute monotonic time
    delta:      float    # time (seconds) taken to recompute value

def xfetch_get(
    key:      str,
    store:    dict[str, XFetchEntry],
    fetch_fn: Callable[[], Any],
    ttl:      float = 60.0,
    beta:     float = 1.0,          # aggressiveness: 1.0 is optimal in most cases
) -> Any:
    """
    XFetch (Optimal Probabilistic Cache Stampede Prevention).
    Serves cached value with probability that increases as TTL approaches zero.
    Early revalidation happens on the serving request, not a background job.
    """
    now   = time.monotonic()
    entry = store.get(key)

    if entry is not None:
        # XFetch formula: early-revalidate if random() < beta * delta * -log(uniform(0,1))
        # Equivalently: revalidate if expires_at - now < beta * delta * -log(rand)
        time_to_expiry = entry.expires_at - now
        should_revalidate = time_to_expiry - beta * entry.delta * math.log(random.random()) < 0

        if not should_revalidate:
            return entry.value   # serve cached value

    # Recompute (on cache miss OR probabilistic early revalidation).
    t_start = time.monotonic()
    value   = fetch_fn()
    delta   = time.monotonic() - t_start

    store[key] = XFetchEntry(
        value=value,
        expires_at=time.monotonic() + ttl,
        delta=delta,   # recorded so future requests know recompute cost
    )
    return value

9.2 Stale Sets: Writing Stale Data into a Fresh Cache

A stale set occurs when a slow reader fetches data from the origin, and by the time it attempts to populate the cache, a write has already invalidated the key and a subsequent reader has already repopulated it with fresh data. The slow reader's SET now overwrites fresh data with stale data — the exact opposite of what the cache should provide.

This is the problem that Facebook's lease mechanism was specifically designed to solve. Without leases, stale sets are possible whenever the application uses a read-then-set pattern without version checking. The fix is to use conditional SET commands or generation-number versioning: the read captures a version number from the cache, and the SET is only accepted if the current version number has not changed.

9.3 Cache Avalanche

Cache avalanche occurs when a large number of cache entries expire simultaneously — typically because they were all populated in the same batch operation with the same TTL. On expiry, all corresponding origin queries fire at once.

The standard mitigations are: jitter (add a random offset to TTLs so entries expire at different times), cache warming (pre-populate before expiry), and graduated background refresh (refresh the cache before entries expire, rather than on demand).

Python TTL jitter to prevent cache avalanche 
import random

def jittered_ttl(base_ttl: float, jitter_factor: float = 0.2) -> float:
    """
    Returns a TTL between base_ttl * (1 - jitter_factor) and base_ttl.
    A jitter_factor of 0.2 spreads expiry across a 20% window.
    """
    lower = base_ttl * (1 - jitter_factor)
    return random.uniform(lower, base_ttl)

# Instead of: redis.setex(key, value, 3600)
# Use:         redis.setex(key, value, jittered_ttl(3600))
#              → expiry between 2880 s and 3600 s, distributed uniformly

9.4 Inconsistency During Partial Failure

In any distributed system, invalidation is not atomic. A cache-aside write succeeds in updating the DB but fails to delete the cache key (network error, Redis node restart). The cache now holds stale data indefinitely — or until TTL expiry. This is an unavoidable consequence of the two-phase nature of cache-aside writes.

Mitigations include: short TTLs as a safety net, write-through with transactional semantics (some Redis Cluster setups), CDC-based invalidation (which derives invalidations from the DB's own commit log and is therefore reliable), or accepting bounded staleness as a product decision.

9.5 The Negative Cache Problem

Negative caching — caching the absence of a resource — is important for performance (avoiding repeated DB queries for non-existent keys) but creates a specific invalidation problem: when a previously-absent entity is created, the negative cache entry must be invalidated. This requires the write path to know which keys were negatively cached, which may be impossible in a decentralized architecture.

DNS suffers from this acutely: a negatively cached NXDOMAIN response (RFC 2308) for a hostname will continue to be served for its negative TTL even after the hostname is created — affecting zone delegation propagation and new service launches.[20]

§ 10

Research Frontiers

10.1 Caffeine and W-TinyLFU: Rethinking Eviction as Invalidation

Eviction policy and invalidation policy are distinct concerns, but they interact: an optimal eviction policy keeps the most valuable entries in cache, reducing the need for re-population after invalidation. Caffeine, the modern Java caching library that replaced Guava Cache,[21] implements the Window TinyLFU (W-TinyLFU) admission policy — maintaining a frequency sketch (a compact probabilistic data structure) to estimate access frequency, and only admitting entries that are accessed more frequently than the entry they would evict.

W-TinyLFU achieves near-optimal cache hit ratios by effectively predicting which entries are likely to be needed again, dramatically reducing the number of re-population events that follow evictions — which, from a consistency standpoint, are indistinguishable from invalidation-driven misses.

10.2 CRDT-Based Cache Consistency

Conflict-free Replicated Data Types (CRDTs) offer a mathematical framework for managing replicated mutable state without coordination — operations are designed to be commutative and associative so that any ordering of concurrent operations converges to the same result.[22] Applied to caching, CRDT-based approaches can represent cache entries as grow-only sets, counters with merge semantics, or last-writer-wins registers with causal timestamps, allowing cache replicas to converge without explicit invalidation coordination.

Antidote DB is an example of a system combining CRDT semantics with strong eventual consistency guarantees, which can serve as both the authoritative store and the caching layer with predictable consistency properties.[23]

10.3 Change Data Capture for Cache Invalidation

Change Data Capture (CDC) derives a stream of database mutations from the database's own write-ahead log, providing a reliable, ordered, and comprehensive record of all writes. Systems like Debezium (for MySQL, PostgreSQL, MongoDB) can publish CDC events to Kafka, which are then consumed by cache invalidation services. This approach has several advantages over application-level invalidation: it catches writes from any application (including migrations and manual admin queries), it is transactionally consistent with the database write, and it provides a durable log for replay on consumer restart.

CDC as the authoritative invalidation substrate

The combination of CDC (for reliable write detection), Kafka (for durable event transport), and surrogate key purge APIs (for efficient CDN invalidation) represents the current state of the art for cache invalidation at scale: reliable, comprehensive, replay-capable, and decoupled from application write paths.

10.4 Lease-Based Caching in Distributed Databases

Conditional leases — short-duration locks granted to cache-populating readers, invalidated on writes, and with fine-grained scope — remain an active research and engineering area. Google's Spanner uses a variant of read-write leases for external consistency.[24] The core insight is that a lease is a lightweight form of distributed coordination that avoids the full overhead of two-phase commit while providing stronger freshness guarantees than TTL alone.

§ 11

Implementation Patterns and Engineering Guidance

11.1 Versioned Cache Keys

The most reliable cache invalidation for immutable content (CSS, JS bundles, static images) is to never invalidate: use content-addressable versioned URLs. A file's URL contains a hash of its content; when the content changes, the URL changes. Old URLs can be cached forever. New URLs are always cold-cached. No invalidation required.

Shell Content-hash URL generation (the only truly correct cache invalidation) 
# Generate content-hashed filenames for static assets
sha256sum styles.css | awk '{print $1}' | head -c 16
# → a3f9c12b44e8d701

# Deploy as /static/styles.a3f9c12b44e8d701.css
# Set Cache-Control: public, max-age=31536000, immutable
# The response is cacheable for 1 year — zero invalidation required.

11.2 The Two-Phase Cache Invalidation Pattern

For mutable resources where versioned URLs are impractical, the two-phase pattern provides strong consistency guarantees with minimal coordination overhead. On write: (1) mark the existing cache entry as "purging" — serve stale with a header indicating revalidation is in progress; (2) write to the authoritative store; (3) delete the cache entry. New readers will trigger a miss and repopulate with fresh data. In-flight readers continue to serve stale (phase 1) rather than hitting the origin during the write window.

11.3 Choosing a Consistency Model

Use case Acceptable staleness Recommended strategy
Financial balances, inventory counts, access control Zero No caching, or write-through with very short TTL (<1 s) for read-heavy paths
User profile data, preferences, configuration Seconds Cache-aside with event-driven invalidation; TTL as safety net
Product catalog, CMS content, search indexes Minutes CDN caching with surrogate key purge on write; stale-while-revalidate
Analytics dashboards, aggregate counts Minutes to hours TTL-based with generous expiry; scheduled background refresh
Static assets (CSS, JS, images) Infinite (versioned) Content-hash URLs; immutable Cache-Control; no invalidation
DNS records TTL field (operator choice) Low TTL before planned changes; awareness of negative caching (RFC 2308)

11.4 Observability: Making Cache State Visible

Cache invalidation bugs are notoriously difficult to debug precisely because caches make state invisible. The system appears correct until a specific sequence of reads and writes exposes the inconsistency — and by then the cache has often self-healed. Engineering teams should instrument their caches with the following metrics as a baseline:

Cache hit rate by key namespace (detect which namespaces have poor effectiveness or abnormal miss rates following writes), invalidation lag (time from DB write to cache invalidation completion, derived from CDC timestamps), stale read rate (if your cache supports conditional reads with ETags, track how often a revalidation returns 304 vs 200), and cache error rate (failed SET or DEL operations that may leave stale entries). Coupling these metrics to deployment events reveals whether new code changes are altering cache behavior in unexpected ways.
