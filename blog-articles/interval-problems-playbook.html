An interval problem is a set of segments on a line and a question about how they relate — do they collide, combine, cover, or count? The naive instinct is to compare every pair, which is O(n²) and almost never the intended solution. The decisive move is to impose a total order on the endpoints. Once the input is sorted by start or by finish, the segments line up so that each one need only be reconciled with the frontier of what came before — the rightmost end seen, the contents of a heap, the running balance of a sweep. The pairwise blow-up disappears and an O(n log n) sort plus an O(n) walk does the work. Nearly every problem below is a variation on that single idea; what changes is which endpoint you sort by and what state you carry across the scan.

Sorting is not a preprocessing step you do before the real algorithm. For intervals, the sort order is the algorithm.The recurring theme

01The overlap predicate

Everything starts with deciding whether two segments touch. For closed intervals [a₁, b₁] and [a₂, b₂], they intersect if and only if

a₁ ≤ b₂  ∧  a₂ ≤ b₁

Read it as “each one starts before the other ends.” The equivalent symmetric form, max(a₁, a₂) ≤ min(b₁, b₂), is more useful in practice because the overlapping region, when it exists, is exactly [max starts, min ends] — the same expression hands you both the test and the intersection. Negating the predicate gives the two disjoint cases: one interval lies entirely left of the other (b₁ < a₂) or entirely right (b₂ < a₁).

The middle row is where bugs live. When endpoints coincide (b₁ = a₂), whether the segments “overlap” depends entirely on your interval semantics, and that choice ripples through every comparison operator you write.

02Representation & conventions

Three decisions are worth making deliberately before writing a line of code, because each one silently fixes a < versus ≤ somewhere downstream.

Closed vs. half-open

A closed interval [s, e] includes both endpoints; meetings that share an instant conflict. A half-open interval [s, e) excludes its right end, so a meeting ending at 10 and one starting at 10 do not clash. Scheduling problems are almost always half-open: a room freed at 10:00 is available at 10:00. Concretely, half-open semantics use start < prevEnd to detect conflict (touching is fine), while closed semantics use start ≤ prevEnd. Pick one and apply it uniformly; mixing them is the single most common source of off-by-one failures on the meeting-room family.

Which endpoint to sort by

This is the choice that selects your algorithm:

  • Sort by start when you process intervals left to right and reconcile each against an evolving set of currently-active intervals — merging, room assignment, sweeps.
  • Sort by finish when you are choosing a maximal compatible subset. Committing to whichever interval frees up soonest leaves the most room for the rest; this is the exchange argument behind the entire greedy family.

Coordinate scale

When endpoints are small integers (say ≤ 10⁶), a difference array indexed directly by coordinate is unbeatable: range updates become two point writes and a prefix sum. When coordinates are large or sparse, coordinate-compress them to ranks first, or switch to an event-based sweep that never materialises the line. Ties in a sweep need a deliberate rule — for half-open intervals, process a −1 (an ending) before a +1 (a start) at equal coordinates so that abutting segments hand off cleanly.

Mental model

Carry the smallest possible frontier across the scan. Sometimes that is a single integer (the last end you kept). Sometimes it is a min-heap of active ends, or a running balance. The art is recognising which scalar or structure is sufficient.

03Pattern I — Sort & merge

Sort by start; walk once. Maintain a single open run and absorb each interval that begins at or before the run’s current end, extending the end to the larger of the two. The instant an interval begins strictly after the open run, the run is sealed and a new one opens. Because every interval is pushed and extended a constant number of times, the walk is linear and the O(n log n) sort dominates.

The proof that one pass suffices is the sort itself: after sorting by start, any interval that overlaps the open run must overlap it at the run’s right end — there is no way to “reach back” past an already-sealed run, because a later start cannot precede an earlier one.

Insert Interval — merge against a sorted list

When the existing intervals are already sorted and disjoint, inserting one new interval does not require a fresh sort. The list splits into three contiguous zones relative to the newcomer: intervals entirely to its left, the block it overlaps, and intervals entirely to its right. Emit the left block untouched, fold the overlapping block into the new interval by widening its bounds, emit the result, then emit the right block. One linear pass, O(n), no sort.

Edge case

The boundary between “left” and “overlapping” uses existing.end < new.start (strict) so that intervals merely touching the newcomer get merged under closed semantics. The boundary into “right” uses existing.start ≤ new.end. Get either comparison backwards and touching intervals are dropped or duplicated.

04Pattern II — Greedy by endpoint

A surprising number of interval problems reduce to one classic: from a set of intervals, keep the largest subset that is mutually non-overlapping. The optimal strategy is greedy — sort by finishing time and always take the interval that ends earliest among those still compatible. Each early finish preserves the maximum runway for everything after it. The argument is a standard exchange: any optimal solution can be rewritten, without losing size, so that its first interval is the earliest finisher, then induction handles the rest.

The complement of that maximisation is Non-overlapping Intervals: the minimum number of removals to eliminate all conflicts equals n minus the size of the largest compatible subset. So the same greedy, counting the intervals you are forced to drop (each one whose start falls before the end of the last interval you kept), solves it directly. Sort by end, keep a running prevEnd, and increment a counter on every conflict.

Same skeleton, three problems

Minimum arrows to burst balloons (LC 452) asks for the fewest points that stab every interval — that is the number of kept intervals in the greedy, since each new shot is fired at the end of the earliest-finishing uncovered balloon. Maximum compatible scheduling returns the kept count; non-overlapping returns the dropped count; interval stabbing returns the same kept count from the dual view. One sort-by-end loop, three answers.

05Pattern III — The sorted scan

The simplest scan answers a yes/no question. Meeting Rooms I asks whether a single person can attend every meeting — equivalently, whether the intervals are pairwise disjoint. Sort by start, and a conflict can only occur between neighbours: if any meeting begins before its predecessor ends, the answer is no. Sorting removes the need to test all pairs, because a non-adjacent collision would force an adjacent one too.

This is the degenerate case of the sweep that follows: the maximum number of simultaneously active intervals is at most one if and only if the schedule is conflict-free.

06Pattern IV — Sweep line & active sets

When the question is “how many overlap at once,” you need the peak occupancy, not just its existence. Meeting Rooms II — the minimum number of rooms — equals the maximum number of meetings live at any instant. Two equivalent techniques compute it.

Heap of active end times

Sort by start and keep a min-heap of the end times of meetings still in progress. For each new meeting, if the soonest-ending active meeting has already finished (its end is at or before the new start), pop it — that room is reused. Then push the new end. The heap’s size is the live count; its maximum over the run is the answer. Each interval is pushed once and popped at most once, so the heap operations total O(n log n).

Chronological sweep with ±1 events

Decompose each interval into two events: +1 at its start, −1 at its end. Sort all 2n events by coordinate, breaking ties so that endings precede starts under half-open semantics. Sweep left to right keeping a running balance; the maximum balance is the peak overlap. This is the same answer as the heap, computed with a sort and a single accumulator instead of a priority queue — usually the faster constant and the more flexible base for variants that need to know when the peak occurs.

The tie-break decides the semantics

At equal coordinates, ordering −1 before +1 treats intervals as half-open [s, e): a meeting ending at t frees the room for one starting at t, and they share no overlap. Flip the order — +1 before −1 — and touching intervals count as overlapping, the closed-interval convention. Choosing the wrong one is an off-by-one that only shows up on adjacency.

Difference arrays — range updates without a heap

When the coordinate space is small and bounded, the sweep can skip events entirely. A difference array records each interval’s effect as +v at its start index and −v just past its end; a single prefix sum then reconstructs the value at every point. Car Pooling is the canonical example: board passengers at a stop, drop them at another, and check that the running occupancy never exceeds capacity. Building the array and sweeping it is O(n + R) for range bound R — no sorting, no comparisons.

07Pattern V — Offline queries with a heap

Minimum Interval to Include Each Query is the synthesis problem: for each query point, report the length of the smallest interval that contains it. Answering each query independently would be quadratic. The trick is to answer them offline — sort the queries ascending and stop treating them as independent.

Sort intervals by start and queries by value. Sweep a pointer through the intervals: as the query value rises past an interval’s start, that interval becomes a candidate and is pushed into a min-heap keyed by length (carrying its end alongside). Before reading the answer for the current query, evict from the heap any interval whose end lies before the query — it can no longer cover this or any later (larger) query, which is exactly why ascending order makes the eviction permanent. Whatever remains on top of the heap is the shortest interval still covering the point. Each interval enters and leaves the heap once, so the whole thing is O((n + q) log (n + q)).

Why offline works

Two monotone sweeps cooperate. Because queries only grow, an interval admitted for one query stays admitted for all later ones until its end is passed — so additions are monotone. Because the eviction test compares each interval’s end against an increasing query value, a removed interval never returns — so removals are monotone too. Monotone in, monotone out: amortised linear heap traffic.

08Pattern VI — Two pointers on two lists

When two already-sorted, internally-disjoint interval lists must be combined, no heap or re-sort is needed — a single merge-style walk suffices. Interval List Intersections computes, for each overlapping pair, the region [max starts, min ends], emitting it when non-empty. The pointer that advances is the one whose current interval ends first, because it can have no further intersections to the right; its partner might still meet the next interval in the other list. Linear in the combined size.

The same two-pointer shape generalises to interval union, difference, and gap-finding over sorted lists — including Employee Free Time (LC 759), where you merge everyone’s sorted busy intervals and report the gaps between consecutive merged blocks.

The last row covers the My Calendar series (LC 729 / 731 / 732), where bookings arrive one at a time and each must respect an overlap limit (no double-booking, no triple-booking, or report the running maximum). It is the difference-array sweep made incremental on a balanced tree.

The recurring traps

  • The touching boundary. Decide closed vs. half-open once. Half-open uses strict < for conflict; closed uses ≤. Every meeting-room failure traces back here.
  • Sort-key confusion. Merging needs sort-by-start; the greedy family needs sort-by-finish. Sorting the greedy by start is the most common wrong answer that still passes small tests.
  • Sweep tie-breaks. At equal coordinates, end-before-start gives half-open behaviour. Decide it on purpose, not by accident of comparator.
  • Mutating shared rows. In languages with reference semantics, extending out.back() can alias the input; clone when the output must be independent.
  • Integer overflow. Interval sizes (end − start) and difference-array sums can exceed 32 bits on large bounds — widen accordingly.
  • Empty input. Greedy and scan loops that read element 0 before the loop must guard the empty case.
