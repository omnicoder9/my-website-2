01 Foundations

A History of Web Performance Thinking

Performance anxiety is older than the modern web. What has changed is our precision in diagnosing it, our vocabulary for discussing it, and our willingness to argue about it.

The modern discipline of frontend performance began to cohere in the mid-2000s, largely through the work of Steve Souders at Yahoo!. His 2007 book High Performance Web Sites codified 14 rules for faster pages — rules that, stripped of their era-specific dressing, remain relevant today: minimise HTTP requests, use a Content Delivery Network, add expiration headers, compress components, put stylesheets at the top, put scripts at the bottom, avoid CSS expressions, make JavaScript and CSS external, reduce DNS lookups, minify JavaScript, avoid redirects, remove duplicate scripts, configure ETags, make Ajax cacheable.[1] Souders also created YSlow, a browser extension that graded pages against these rules and introduced millions of developers to the concept of automated performance auditing.

The period from 2008 to 2013 brought structural change. Smartphones began to dominate traffic and revealed how badly the performance practices of the desktop era scaled to constrained devices and mobile networks. Ilya Grigorik joined Google and produced work — culminating in his 2013 book High Performance Browser Networking — that pushed browser engineers and web developers to think deeply about the TCP handshake, TLS overhead, and the latency characteristics baked into the physical network.[2] This era also produced the HTTP Archive, a project that crawled the web to produce longitudinal data on how page weight, request counts, and technology choices evolved across the broader web.[6]

In 2015, Google engineers Paul Lewis and Paul Irish introduced the RAIL model — Response, Animation, Idle, Load — a user-centric framework for thinking about performance budgets.[4] RAIL was important because it tied performance targets to human perception thresholds rather than arbitrary technical benchmarks: responses under 100ms feel instantaneous, animations at 60fps feel smooth, idle time could be used to front-load work, and load should complete in under a second on a 3G connection. The same year saw HTTP/2 standardised in RFC 7540, which multiplexed requests over a single connection and eliminated the head-of-line blocking that had driven the proliferation of hacks like domain sharding and CSS sprites.[18]

Google launched Lighthouse as an open-source automated auditing tool in 2016,[15] and it was subsequently integrated into Chrome DevTools, making sophisticated performance analysis available to any developer who could open a browser. PageSpeed Insights, which had existed since 2010 as a simpler tool, gained a second life by wrapping Lighthouse scores alongside field data drawn from the Chrome User Experience Report (CrUX), creating a useful tension between what a controlled lab test finds and what real users actually experience.[13]

The most significant recent development was the announcement of Core Web Vitals in May 2020 — a defined, opinionated set of metrics representing load performance, visual stability, and interactivity — coupled with the signal that they would become a Google Search ranking factor.[3] Whatever one thinks of the methodology, this announcement transformed frontend performance from a craft concern into a business imperative for the SEO-sensitive majority of the web. It also provoked a new wave of metric gaming that the industry is still reckoning with. In March 2024, Interaction to Next Paint (INP) replaced First Input Delay (FID) as the Core Web Vitals interactivity metric, addressing a well-documented shortcoming of FID's narrow measurement window.[11]

"The fastest page is the one with the fewest requests."

— paraphrased from Steve Souders, 2007, on the dominant bottleneck of the HTTP/1 era

The lesson of this history is that performance best practices are context-dependent and era-dependent. What Souders said about minimising HTTP requests was correct for HTTP/1.1; it became less universally true with HTTP/2 multiplexing. Advice about render-blocking scripts still holds but has been refined by async, defer, and module scripts. Understanding why a rule exists is the only reliable defence against applying it when the underlying context has changed.

02 Measurement

The Metrics That Matter

A metric without a threshold is decoration. A threshold without an understanding of what it measures is a target for gaming.

Performance metrics fall into two broad categories: lab metrics, measured in a controlled test environment with simulated network and CPU throttling, and field metrics (also called Real User Monitoring, or RUM), collected from actual users in production. Both are necessary and neither is sufficient alone. Lab data is reproducible and actionable for debugging; field data reflects the true distribution of user experiences across devices, locations, and network conditions.[14]

Core Web Vitals and Key Timing Metrics

TTFB Time to First Byte
≤ 800ms > 1800ms

Time from navigation start until the first byte of the response arrives. Reflects server performance, routing, and CDN effectiveness.

FCP First Contentful Paint
≤ 1.8s ≤ 3.0s > 3.0s

Time until the browser renders the first piece of DOM content (text, image, canvas, SVG). A leading indicator of perceived responsiveness.

LCP Largest Contentful Paint
≤ 2.5s ≤ 4.0s > 4.0s

Time until the largest content element in the viewport becomes visible. A Core Web Vital representing loading performance.[9]

CLS Cumulative Layout Shift
≤ 0.1 ≤ 0.25 > 0.25

Measures unexpected layout shifts weighted by impact fraction and distance fraction. A Core Web Vital representing visual stability.[10]

INP Interaction to Next Paint
≤ 200ms ≤ 500ms > 500ms

The 98th-percentile interaction latency across a page session. Replaced FID as the Core Web Vitals interactivity metric in March 2024.[11]

TBT Total Blocking Time
≤ 200ms > 600ms

Sum of the blocking portions (beyond 50ms) of long tasks between FCP and TTI. The primary lab proxy for INP; highly correlated with main-thread congestion.

SI Speed Index
≤ 3.4s > 5.8s

How quickly content is visually populated during load, derived from a video-frame analysis. Popularised by WebPageTest; less predictable as a single number than it appears.

TTI Time to Interactive
Deprecated

Removed from Lighthouse scoring in version 10. Found to be too variable and difficult to explain to users; TBT is now the preferred proxy for interactivity.

The Problem with Any Single Number

Every metric is a compression: it trades fidelity for communicability. FCP can be satisfied by a loading spinner — technically the first contentful paint — while the user sees no meaningful content. LCP is fooled by a large background image that is present before useful text. CLS can be gamed by reserving space for ads before they load, even if that space is wrong and shifts later. INP, being a 98th-percentile figure, can miss systematic latency that affects the majority of users if it appears only once per session.[26]

The thresholds themselves were determined by examining the relationship between metric values and user survey responses about perceived performance, combined with field data correlations.[26] This gives them a real empirical basis, but the 75th-percentile aggregation (the level at which a page is considered to pass a Core Web Vitals threshold) means a site can have excellent median performance and still fail if a significant tail of users experience poor conditions.

The RAIL Model

RAIL (Response, Animation, Idle, Load) provides a user-centric framing for performance targets: responses to user input should complete in under 100ms; animations should render a frame within 16ms (60fps); idle time should be used in chunks under 50ms to stay responsive; and load should deliver an interactive experience in under 5 seconds on a mid-range device and slow network.[4] Unlike the Core Web Vitals thresholds, RAIL is a design philosophy rather than a measurement system — it tells you what to aim for rather than how to score yourself.

03 Internals

The Critical Rendering Path

Before you can optimise rendering, you need to know what the browser is actually doing between receiving the first byte and painting pixels on screen.

The browser's rendering process, from network bytes to visible pixels, follows a deterministic sequence. Understanding where it can stall is the foundation of every meaningful optimisation.

The Sequence

1. HTML parsing. As bytes arrive, the browser's HTML parser constructs the Document Object Model (DOM). The parser is incremental — it can begin before the full document is downloaded — and is interrupted whenever it encounters a <script> tag without async or defer.[20]

2. CSS Object Model construction. CSS files, inline styles, and <style> blocks are parsed into the CSS Object Model (CSSOM). CSS is render-blocking: the browser will not render any content until it has fully parsed all CSS it has encountered, because a later rule could change the appearance of already-processed elements. This is the correct behaviour, but it means that every stylesheet added to the critical path has a cost.

3. Render tree construction. The DOM and CSSOM are merged into a render tree, which contains only visible nodes with their computed styles. Elements with display: none are excluded; pseudo-elements like ::before are included.

4. Layout (Reflow). The browser calculates the exact position and size of every element in the render tree. This is an expensive operation proportional to the size of the DOM and the complexity of the layout.

5. Paint. The browser fills in the pixels for each node — text, colours, images, borders, shadows. Different layers may be painted independently.

6. Compositing. If the page has multiple layers (created by transform, opacity, will-change, or position: fixed elements), the GPU compositor combines them. Composited animations (translating or fading a compositor-promoted element) bypass layout and paint entirely — this is why GPU-accelerated CSS animations are far cheaper than those that trigger layout.

Where the Pipeline Breaks

JavaScript running on the main thread can interrupt the rendering process at any point. A long task — any JavaScript execution that runs for more than 50ms without yielding — blocks the browser from processing user input, running paint, or handling other work. The main thread is single-threaded, and everything from DOM manipulation to event handlers to requestAnimationFrame callbacks competes for its time.[25]

Layout thrashing is a particularly damaging pattern: JavaScript that interleaves reads of layout-triggering properties (like offsetHeight, getBoundingClientRect()) with DOM writes forces the browser to perform a synchronous layout on each cycle, negating its ability to batch layout work.[20] Libraries like FastDOM exist to batch reads and writes; the better solution is architectural — avoid the pattern altogether.

// Layout thrashing — forces synchronous layout on every iteration
for (const el of elements) {
  el.style.width = el.offsetWidth + 'px'; // read → write → read → write
}

// Correct — batch reads, then writes
const widths = elements.map(el => el.offsetWidth);  // reads
elements.forEach((el, i) => { el.style.width = widths[i] + 'px'; }); // writes

Speculative Parsing and Preloading

Modern browsers implement a preload scanner that reads ahead in the HTML stream, independently of the main parser, to discover resources like images, scripts, and stylesheets. The preload scanner fires off requests for these resources before the main parser has reached them, reducing the cost of render-blocking resources. However, it only works with static HTML — resources injected by JavaScript or hidden in CSS are invisible to it, which is why <link rel="preload"> hints exist: to proactively tell the browser about resources the preload scanner cannot find.[35]

04 Best Practices

What To Do: The Optimization Catalogue

There is no silver bullet. Every optimisation addresses a specific bottleneck, and the first step is always measuring to know which bottleneck actually dominates.

Image Optimisation

Images consistently account for the largest share of page weight on the median web page, frequently exceeding the combined weight of JavaScript, CSS, and fonts.[6] The decisions made at the point of image encoding and delivery have larger cumulative impact than most JavaScript optimisations.

Format Selection

AVIF (AV1 Image File Format), standardised by the Alliance for Open Media in 2019, provides superior compression at equivalent visual quality versus JPEG and WebP, particularly for photographic content at lower bitrates.[16] WebP, developed by Google, delivers roughly 25–34% smaller files than JPEG at equivalent quality for most photographic content.[17] Use AVIF where browser support allows, WebP as a broad fallback, and JPEG/PNG as the final fallback. The <picture> element makes this straightforward:

<picture>
  <source srcset="photo.avif" type="image/avif">
  <source srcset="photo.webp" type="image/webp">
  <img src="photo.jpg" alt="..." width="800" height="500" loading="lazy">
</picture>

Always set explicit width and height attributes on images. This allows the browser to reserve the correct space before the image loads, preventing layout shift (CLS). Always compress images — uncompressed PNGs exported from design tools at full resolution are a common source of multi-megabyte assets that should be kilobytes.

Loading Strategy

Use loading="lazy" for all below-the-fold images. Use fetchpriority="high" on your LCP image to signal to the browser that it should be loaded as early as possible. Use srcset and sizes to deliver appropriately dimensioned images to each viewport width — serving a 2400px image to a 375px mobile screen wastes bandwidth proportional to the square of the scale factor.

Font Optimisation

Web fonts introduce two distinct performance problems: the network round-trip to fetch them, and the rendering behaviour while they load.[29] Flash of Invisible Text (FOIT) — where text is rendered as invisible while the font loads — and Flash of Unstyled Text (FOUT) — where system fonts show initially and are replaced on load — are both caused by font loading behaviour.

The font-display descriptor controls this behaviour. font-display: swap allows the browser to use a system font immediately and swap it when the web font loads, eliminating FOIT at the cost of a reflow. font-display: optional gives the browser a very short window to load the font, then permanently falls back to the system font for that page load — the most performance-friendly option but requires accepting that some users will see the system font.[31]

Use <link rel="preload" as="font" crossorigin> for fonts that are used in above-the-fold content. Subset fonts to include only the Unicode ranges your content actually uses — a full Latin typeface including all weights can be 500KB+, while a subsetted version for English text is often under 30KB. Google Fonts handles subsetting automatically via the text parameter for static subsets, or by serving per-script subsets via unicode-range.[29]

JavaScript: Load Less, Parse Less, Execute Less

JavaScript is uniquely expensive among web resources: unlike images, which are simply decoded, JavaScript must be downloaded, parsed, compiled, and executed. On a median mobile device, this processing overhead can make a 1MB JavaScript bundle take 4–5 seconds to become interactive.[7]

Code Splitting

Modern bundlers (webpack, Rollup, Vite, esbuild) support splitting the JavaScript bundle into smaller chunks loaded on demand. Route-based splitting — loading the code for the current page and deferring the rest — is the highest-leverage form. Dynamic import() syntax makes this straightforward:

// Load the heavy chart library only when the dashboard route is active
const ChartModule = await import('./charts');
ChartModule.render(data);

Tree Shaking

Tree shaking eliminates unreachable code from the final bundle. It depends on ES Module static imports, which can be statically analysed. Avoid wildcard imports (import * as _) from large libraries; prefer named imports from packages that declare "sideEffects": false in their package.json.

Deferral and Async Loading

Scripts with defer are downloaded in parallel with HTML parsing and executed in order after parsing completes. Scripts with async are downloaded in parallel and executed immediately when available, interrupting parsing. Use type="module" scripts where possible — they are deferred by default and always strict-mode. Reserve async for truly independent third-party scripts where order does not matter.

CSS: Critical Path and Dead Code

The critical CSS technique inlines the styles required to render above-the-fold content in a <style> tag in the document head, then loads the full stylesheet asynchronously. This eliminates the render-blocking round-trip for the stylesheet, allowing the browser to paint the initial viewport without waiting for an external file.

Tools like Critical and PurgeCSS automate the extraction of critical styles and the removal of unused rules respectively. The Coverage tab in Chrome DevTools shows what percentage of loaded CSS and JavaScript is actually executed — it is common to find 60–80% of a CSS framework's rules unused on any given page.

CSS Containment

The content-visibility: auto property (now broadly supported) instructs the browser to skip layout and paint for off-screen elements until they are about to enter the viewport.[34] On pages with long, complex content, this can dramatically reduce initial rendering time. Pair it with contain-intrinsic-size to avoid CLS from the browser reserving no space for the skipped content.

Network and Protocol

HTTP/2 multiplexes multiple requests over a single connection, eliminating head-of-line blocking and making per-connection overhead irrelevant for most sites.[18] HTTP/3, standardised in RFC 9114, addresses the remaining head-of-line blocking at the transport layer by using QUIC (UDP-based) instead of TCP, improving performance on lossy connections such as mobile networks.[19] Both are transparent to most application code — the primary requirement is server and CDN support.

Enable Brotli compression (generally 15–25% better than gzip for text assets) at the server level. Ensure Cache-Control headers are set correctly: long-lived caches with content-hashed filenames for static assets, short-lived or no-cache for HTML documents that must reflect the latest version.

Resource Hints

Resource hints let you tell the browser about work it should do before it has discovered the need itself:

<!-- Preconnect to a critical third-party origin -->
<link rel="preconnect" href="https://fonts.gstatic.com" crossorigin>

<!-- Preload the LCP image -->
<link rel="preload" as="image" href="hero.avif"
  fetchpriority="high" type="image/avif">

<!-- Prefetch the next likely page for instant navigation -->
<link rel="prefetch" href="/dashboard">

preconnect performs the DNS lookup, TCP handshake, and TLS negotiation early. preload fetches a resource at high priority before the browser has discovered it through normal parsing. prefetch fetches a resource at low priority for anticipated future navigations. Over-use of preload competes with the browser's own priority scheduler and can harm performance; instrument before deploying broadly.

The 14KB First Round Trip

TCP slow-start means the first 14KB of data (the initial congestion window) is transmitted before the connection establishes its throughput. Content that fits within 14KB of the initial response — the HTML document including inlined critical CSS and any inlined critical JavaScript — can be rendered before any subsequent round trips complete. This limit is approximate and depends on the network path, but the principle is sound: minimise what is required to render the first frame.[2]

05 Anti-Patterns

What Not To Do: Anti-Patterns and Their Costs

Most real-world performance problems are not caused by failing to implement an optimisation. They are caused by actively doing something harmful — often unintentionally, often inherited.

Third-Party Script Sprawl

Analytics, chat widgets, A/B testing tools, ad networks, social embeds, and CRO scripts are each, individually, understandable decisions. Collectively, they are frequently the largest performance liability on the page. Each third-party script: introduces an additional DNS lookup and connection; runs in the same main thread as the page's own code; often loads additional resources; and is outside the development team's control when the vendor updates it.[8]

The correct response is not to forbid third-party scripts but to inventory and audit them regularly, load them with async or defer where possible, and consider consolidating multiple tools into a single Tag Manager — though Tag Managers themselves introduce overhead and require careful governance.

The Third-Party Trap

A page can achieve a perfect Lighthouse score in a test environment and fail Core Web Vitals for real users because a third-party script loads asynchronously after the test window closes but before users attempt to interact with the page. Field data is the only honest measure of third-party impact.

Importing Entire Libraries for One Feature

The canonical example is loading the entire Moment.js library (roughly 67KB minified + gzipped including a locale) to format a single date. The Intl.DateTimeFormat API, built into every modern browser at zero bundle cost, handles the majority of date-formatting use cases. Similarly, loading the full lodash library (import _ from 'lodash') when only debounce is needed ships tens of kilobytes of dead code — the same function is available as a standalone package (lodash.debounce) or importable from the ES Modules build (import { debounce } from 'lodash-es') with tree shaking.[21]

The Bundlephobia tool (discussed in the tooling section) makes the bundle cost of npm packages visible before installation, which is the right time to evaluate them — not after they have propagated through the codebase.

Render-Blocking Resource Chains

CSS @import rules create sequential dependencies: the browser must download and parse the first stylesheet before it discovers the second. Three @import statements can triple the time the browser spends waiting for styles before it can render anything. Use build tools to inline @import chains into a single file, or use multiple <link> elements in the HTML head, which the browser can download in parallel.

Excessive DOM Size

Lighthouse warns when a page has more than approximately 1,400 DOM nodes. Large DOMs increase memory usage, slow style calculations (more nodes must be matched against CSS rules), and make layout, paint, and compositing more expensive. Modern JavaScript frameworks that render large lists without virtualisation are a common cause — libraries like react-window or TanStack Virtual render only the visible portion of a long list.

Unset Image Dimensions and Dynamic Content

Injecting content above existing content after the page has loaded — without reserving space in advance — causes layout shift. The most common causes of poor CLS scores are: images without explicit width and height attributes; web font swaps that change text dimensions; dynamically injected banners, cookie notices, or GDPR dialogs; and ad slots that expand from a placeholder size.[10] Reserve space with CSS aspect-ratio for images, min-height for dynamic content slots, and use font-display: optional to prevent font-swap shifts.

document.write()

Calling document.write() after the document has loaded causes the parser to halt, re-open the document stream, and insert the new content — a guaranteed full page re-parse. Chrome has blocked document.write() calls that insert parser-blocking scripts on slow connections since 2016. Its legitimate use cases have all been superseded by DOM APIs. Avoid entirely.

Not Compressing Static Assets

Serving uncompressed JavaScript, CSS, and HTML is a straightforward error with a large, immediate cost. Brotli and gzip are implemented in every major web server and CDN and require only a configuration flag. A 500KB uncompressed JavaScript bundle is typically around 150–180KB with gzip and 130–160KB with Brotli. This reduction is free from the application's perspective.

Memory Leaks

Memory leaks in long-lived single-page applications accumulate over a session, eventually causing the browser to slow down as the garbage collector spends more time on a large heap. Common causes: closures that retain large DOM subtrees; global event listeners never removed when components unmount; timers and intervals that are never cleared; and cached data structures that grow without bound. The Memory panel in Chrome DevTools and heap snapshot comparison are the primary diagnostic tools.

06 Tooling

Tooling: A Comparative Analysis

Tools reveal different aspects of the same reality. Using only one is like diagnosing a patient with only a thermometer.

Lighthouse & PageSpeed Insights Lab / Field

Lighthouse is an open-source automated auditing engine maintained by Google.[15] It runs a controlled test — simulating a mid-range device on a throttled connection — and produces 0–100 scores for Performance, Accessibility, Best Practices, and SEO, alongside specific diagnostics and opportunities. PageSpeed Insights (PSI) wraps Lighthouse lab results alongside field data from the Chrome User Experience Report (CrUX), making it the most accessible combination of lab and real-user data.[13]

Strengths
  • Integrated into Chrome DevTools — zero installation required
  • Actionable, specific recommendations with estimated savings
  • PSI combines lab and CrUX field data in one view
  • Scriptable via CLI and Node API for CI integration
  • Scores correlate reasonably with Core Web Vitals field thresholds
Limitations
  • Simulated throttling does not accurately represent all real device/network combinations
  • High score variability (±5–10 points) between runs on the same URL
  • Scores can be gamed without improving real user experience
  • Lab run does not capture third-party scripts loading after the test window
  • No support for authenticated pages or multi-step user flows without scripting
WebPageTest Lab / Advanced

Created by Patrick Meenan (originally at AOL, now at Google), WebPageTest is the most powerful and flexible public performance testing tool available.[12] It runs tests from real devices in global locations across real connection profiles, produces detailed waterfall charts, filmstrip views, and video comparisons. It supports scripted multi-step tests, custom metrics, and experimental features like the Opportunities panel that maps findings directly to Core Web Vitals impact.

Strengths
  • Real devices (not emulation) across global test agents
  • Detailed waterfall: every request, timing, and priority
  • Filmstrip and video capture for perceived performance analysis
  • Multi-step scripting for authenticated flows
  • Exportable data and API access for automation
  • Comparison views across multiple URLs or runs
Limitations
  • Test queues on the free tier can be slow during peak hours
  • Steeper learning curve than Lighthouse for newcomers
  • Results not directly comparable to Lighthouse scores
Chrome DevTools Browser / Diagnostic

Chrome DevTools is the most granular diagnostic environment available without purchasing a specialised tool. The Performance panel records a detailed trace of all browser activity — JavaScript execution, style calculations, layout, paint, and compositing — visualised as a flame chart. The Network panel shows every request with timing, size, priority, and caching headers. The Coverage panel identifies unused JavaScript and CSS. The Memory panel enables heap snapshot analysis and allocation tracking.

Strengths
  • Complete visibility into main thread activity and frame rendering
  • Identifies specific functions causing long tasks
  • Network panel shows actual caching, compression, and priorities
  • Coverage panel quantifies dead code
  • Can test local builds and authenticated sessions
Limitations
  • Requires manual interpretation — no automatic scoring
  • Profiling adds overhead; the observer affects the observed
  • Only available in Chrome (though Firefox and Safari have analogues)
  • Performance traces for complex SPAs can be difficult to read
Bundlephobia Bundle Analysis

Bundlephobia calculates the bundle cost of any npm package before installation.[21] Enter a package name and it shows the minified size, minified + gzipped size, whether the package is tree-shakeable, download time estimates on different connections, and historical size trends. It also shows the transitive dependencies that would be included. This is most valuable at the point of package selection — cheaper to discover a 400KB dependency before it is in your package.json than after it has propagated through the codebase.

Strengths
  • Instant cost estimate before committing to a dependency
  • Shows treeshakeability, transitive deps, historical trends
  • Package comparison view for evaluating alternatives
Limitations
  • Estimates the full package cost — not the subset your code actually uses
  • Does not account for polyfills, peer dependencies, or runtime environment
  • Gzip estimates; your server may use Brotli (slightly better)
Squoosh Image Compression

Squoosh, built by the Google Chrome Labs team, is a browser-based image compression tool that uses WebAssembly-compiled codecs to compress images entirely client-side.[22] It supports MozJPEG, WebP, AVIF, OxiPNG, and JPEG XL, provides a visual comparison slider between original and compressed versions, and shows the file size savings in real time. A CLI version (@squoosh/cli) enables integration into build pipelines.

Strengths
  • AVIF support with visual quality comparison — best for evaluating modern formats
  • No server upload — images processed locally in WASM
  • Excellent for one-off compression experiments and format decisions
  • CLI for build pipeline integration
Limitations
  • Manual tool — not suitable for automated bulk processing at scale
  • AVIF encoding is slow (WASM overhead) for large images
  • Does not handle responsive image generation (multiple sizes)

Lab vs Field: The Fundamental Tension

Lab tools (Lighthouse, WebPageTest) are reproducible, actionable, and testable in CI. Field tools (CrUX, RUM) reflect what users actually experience. The two routinely disagree. A page can score 95 on Lighthouse and fail Core Web Vitals for real users because of device diversity, network variance, and third-party scripts that load after the test window.[14] Use lab tools to find and fix problems; use field data to confirm whether the fix worked for users.

07 Controversies

Controversies, Debates, and Schools of Thought

Performance engineering is not a settled discipline. Reasonable engineers disagree — sometimes sharply — on fundamentals. These disagreements are worth understanding, not resolving prematurely.

The Business Case: Does Performance Actually Matter?

The performance-as-business-value argument is built on a small corpus of widely-cited studies. The most famous is the attribution, via Greg Linden's 2006 blog post about a presentation by Marissa Mayer, that Amazon found every 100ms of additional latency reduced sales by 1%.[28] A 2020 Deloitte study commissioned by Google found that a 0.1-second improvement in mobile page speed increased conversion rates by 8% for retail sites and reduced bounce rates by 8%.[5]

These numbers are frequently cited and should be taken seriously — but also critically. They describe specific populations, specific changes, specific contexts. The causal mechanism (slower page → more abandonment) is plausible and well-supported directionally, but the specific percentages do not straightforwardly transfer across industries, user bases, or the particular bottleneck being addressed. Treat these studies as directional evidence that performance has business consequences, not as quantitative predictions for your specific situation.

Debate 01 — The JavaScript Framework Cost
The Framework Pragmatist View
  • Code splitting and tree shaking mitigate bundle size
  • Good developer ergonomics leads to faster iteration and fewer bugs
  • SSR, SSG, and streaming modes recover much of the cost
  • React, Vue, Svelte scores on CWV are comparable to non-framework sites in practice
  • Developer velocity is itself a business resource
The Minimal-JS View
  • A 200KB+ React runtime has a fixed cost before any app code runs
  • Median mobile devices in global markets cannot process large JS bundles quickly
  • Many sites ship SPAs for content that does not need interactivity
  • The platform (HTML, CSS, Web Components) has caught up to much of what frameworks provided
  • Alex Russell: frameworks are "a unilateral budget cut" on users[24]

Tim Kadlec's analysis of CrUX data found that pages built with Angular had significantly higher JavaScript transfer sizes than those built with React, which in turn had higher sizes than pages with no detectable framework.[8] This is an observational finding that reflects how each framework tends to be used — not that React inherently produces worse output than handwritten code. The more important variable is often how disciplined a team is about bundle management, not which framework they chose.

Debate 02 — AMP: Accelerated Mobile Pages
The Case For AMP (circa 2015–2019)
  • Pages cached on Google's infrastructure loaded near-instantly from search results
  • Constrained AMP HTML prevented the worst performance anti-patterns
  • Publishers on slow infrastructure got fast distribution for free
  • Achieved measurable performance improvements for sites that could not otherwise
The Case Against AMP
  • URLs in Google's cache were google.com/amp/... — breaking publisher identity
  • Required learning a proprietary subset of HTML for a single platform's benefit
  • Critics argued it was Google using performance as a pretext for ecosystem control
  • The speed benefit derived mainly from pre-caching, not AMP HTML itself
  • With Core Web Vitals, Google removed the AMP requirement for Top Stories eligibility[23]

The AMP controversy was clarifying for the field in a useful way. It forced a distinction between two things that had been conflated: page performance as a property of the page itself, versus delivery speed as a property of the distribution infrastructure. The lesson was that pre-caching content on a CDN close to users produces fast delivery regardless of AMP. Google's subsequent move — making Core Web Vitals the ranking signal rather than AMP compliance — accepted this distinction.

Debate 03 — Rendering Architecture: SSR vs CSR vs SSG vs Islands
Server-Side Rendering (SSR) & Static Generation (SSG)
  • HTML is delivered ready-to-paint; no JavaScript required for first paint
  • Good TTFB + FCP + LCP by default
  • Content is indexable without JavaScript execution
  • SSG is the fastest possible delivery: pre-built HTML served from CDN edge
Client-Side Rendering (CSR) & SPA
  • Subsequent navigations are instant (no full page reload)
  • Rich interactions without server round-trips
  • Moves server costs to the client; scales cheaply
  • Modern SSR (React 18 streaming) recovers some of the gap with selective hydration

The Islands Architecture — popularised by Astro and advocated by frameworks like Eleventy with partial hydration — represents an emerging synthesis: server-render the document by default, then hydrate only the interactive components ("islands") that actually need client-side JavaScript.[33] This approach delivers the LCP and FCP characteristics of SSR with the interactivity of a framework-powered component, at a dramatically lower JavaScript payload. It is not appropriate for every application, particularly those with highly stateful global interactions, but it is increasingly the preferred default for content-heavy sites.

Debate 04 — Metric Gaming and Goodhart's Law
Metrics are a useful forcing function
  • Having a number creates accountability and enables regression detection
  • Core Web Vitals correlate with user satisfaction in Google's research
  • The SEO pressure gets performance taken seriously in boardrooms
  • Even imperfect metrics guide better decisions than no metrics
When a measure becomes a target, it ceases to be a good measure
  • Goodhart's Law[32]: teams optimise for the score, not the user experience
  • Delaying non-critical JavaScript past the measurement window inflates scores
  • Reserving ad slots avoids CLS without making the experience better
  • Lighthouse score is a proxy; treating the proxy as the goal corrupts it

Progressive Enhancement: Philosophy vs Pragmatism

The doctrine of Progressive Enhancement — building a baseline experience in HTML, layering CSS, then adding JavaScript — was codified in the mid-2000s and has been defended most articulately by Jeremy Keith.[30] The argument is not primarily about performance; it is about resilience: a site built with progressive enhancement works when JavaScript fails, when a user's network drops a script, when an accessibility tool doesn't handle a framework. Performance is a consequence of the approach rather than its motivation.

The counter-position is not that progressive enhancement is wrong but that it is impractical at scale for genuinely complex applications. A spreadsheet editor, a collaborative document tool, or a video-conferencing interface cannot be meaningfully progressive — they are fundamentally JavaScript applications. The debate is most useful when it forces the question: does this specific feature actually need to be a JavaScript component? — rather than as an absolute architectural stance.

"The web is resilient by design. We keep trying to undermine that by making everything depend on JavaScript working perfectly."

— Jeremy Keith, Resilient Web Design, 2016

Perceived Performance vs Actual Performance

A site can feel faster than it is, and it can feel slower than it is. Perceived performance — the user's subjective sense of speed — is determined partly by the actual timing and partly by what is shown during that time. Skeleton screens (placeholder layouts that show content structure before data arrives), optimistic UI updates (reflecting a user action immediately before server confirmation), and progressive image loading (displaying a low-quality placeholder while the full image loads) all improve perceived speed without changing a single metric.

This creates a tension: metrics optimise for when something happens; UX design also shapes how something feels while it is happening. Both matter, and neither fully substitutes for the other. An application that loads in 3 seconds with a well-designed skeleton feels different from one that loads in 3 seconds with a blank white screen, even though they produce identical Lighthouse scores.

The Bundling Debate Post-HTTP/2

One of Souders' original rules was to minimise HTTP requests, and for years the practice of bundling dozens of small JavaScript and CSS files into a single large file was justified by the per-connection overhead of HTTP/1.1. HTTP/2 multiplexing changed the underlying premise: multiple small files could now be requested over a single connection without that overhead.[18]

This has not entirely vindicated unbundled delivery, however. Parse time — the cost to the JavaScript engine of processing a file — scales with the number of bytes, not the number of requests. A hundred small modules and one large bundle with the same total byte count will still perform similarly in terms of parse and compilation time. Code-split bundles (a small number of route-level chunks) remain the pragmatic optimum: fewer than a full bundle, more than a single monolith. The development-mode pattern of Vite — serving unbundled ES Modules to the browser in development — works well in practice because modern browsers have fast module loaders, but production builds are still bundled for the same reasons.

08 Process

Performance Budgets and the Engineering Culture of Speed

A performance budget is a commitment. It is only as valuable as the process that enforces it.

A performance budget sets explicit limits on the metrics and resource sizes that a page may not exceed. Budget categories typically include: total JavaScript size, total image weight, LCP target, TBT target, and third-party request count. The budget is then enforced in CI: a pull request that would push the JavaScript bundle past 200KB is rejected before merging, just as a pull request that would push test coverage below a threshold would be.

Tools like Lighthouse CI (@lhci/cli), Calibre, and SpeedCurve's Budget feature integrate with GitHub, GitLab, and similar systems to make performance a first-class quality gate. The key organisational requirement is that the budget must be set by the team and defended by the team — a budget imposed from outside without developer buy-in tends to be worked around rather than maintained.

Setting a Useful Budget

The most defensible starting point for a budget is not an aspirational number but a competitive one: measure the performance of direct competitors' pages under the same conditions, then set your budget 20% faster than the best competitor on the metrics that most correlate with your conversion goals. This roots the budget in a business rationale rather than an engineering ideal, which makes it easier to defend when the budget blocks a feature launch.

The Culture Problem

Most persistent performance problems are not technical. They are organisational: a marketing team that can add third-party scripts without engineering review; a product manager who can specify a feature without a performance impact assessment; an engineering culture where "it works" means "it renders" rather than "it renders fast enough for the device and network of our median user." Technical tools can enforce constraints, but they cannot substitute for a team that considers performance a correctness property rather than an optimisation phase.

The Device and Network Empathy Gap

Development machines are fast. The median developer machine has a multi-core high-clock-speed CPU, 16–32GB RAM, an SSD, and a gigabit network connection. The median user of many global consumer applications has a lower-specification Android device, 2–4GB RAM, and an LTE connection that varies between 4G and 2G throughput depending on location. Chrome DevTools' device throttling and CPU throttling modes — combined with the practice of testing on real budget Android hardware — are the only way to build genuine empathy for this gap.[7]

09 Synthesis

Conclusion

Frontend performance is a discipline that rewards empiricism over dogma. The practices described in this article are not universal laws but probabilistic recommendations, validated in specific contexts. Rules about HTTP request minimisation had to be revisited when HTTP/2 changed the protocol's cost model. Advice about server-side rendering has been complicated by streaming SSR and islands architecture. The "avoid JavaScript frameworks" position and the "frameworks are fine" position have both been partially vindicated by the evidence, which means both are also partially wrong.

The field has made genuine progress. The Core Web Vitals initiative, for all its controversy, gave the web a shared vocabulary for performance that has measurably moved the industry's attention toward user-centric timing. The tooling ecosystem — Lighthouse, WebPageTest, Bundlephobia, Squoosh, real-device profiling in DevTools — is the best it has ever been. The understanding of what happens between a network request and a painted pixel is deeper and more widely distributed than it was when Souders wrote his 14 rules.

What has not changed: the fundamental constraint is human attention and patience, and the cost of wasting them. Users abandon slow pages; slow pages lose conversions; slow pages lose users to competitors who are faster.[5] The specific mechanism — whether it is a render-blocking stylesheet, an oversized image, a memory leak, or a third-party script running amok — is a diagnostic question. The reason to care about the answer is always the same: speed is a feature, and it belongs to the user.

Speed is not a technical property of a website. It is a quality-of-life property of the people who use it.
