§ 1 — Background & Motivation

From N×M Chaos to a Universal Connector

Before November 2024, connecting an AI model to an external data source was an artisanal affair. Every pairing — a chatbot to a database, a coding copilot to a code repository, an AI assistant to a calendar — demanded its own bespoke integration. For M AI applications and N data sources, the result was, as Anthropic engineers would later document, an "N×M integration problem": a combinatorial explosion of connectors, each fragile, non-reusable, and built against vendor-specific APIs.[1]

The pattern had antecedents. OpenAI's 2023 "function-calling" API allowed models to generate structured calls to external tools, and the ChatGPT plugin framework attempted standardised access to web services. Yet each approach remained siloed within its own product ecosystem, requiring developers to implement integrations separately for every AI platform they targeted.[2]

The inspiration for a better approach came from an adjacent domain. The Language Server Protocol (LSP), originally proposed by Microsoft in 2016, solved an analogous problem in software development: rather than every code editor implementing custom support for every programming language, LSP defined a single interface between editors and language-analysis servers. The same editor could then gain syntax highlighting, auto-completion, and diagnostics for dozens of languages by simply speaking LSP.[3]

Anthropic engineers David Soria Parra and Justin Spahr-Summers applied this insight to AI integration. The protocol they created re-uses the message-flow ideas of LSP and is transported over JSON-RPC 2.0. On 25 November 2024, Anthropic open-sourced the Model Context Protocol (MCP) with initial SDKs for Python and TypeScript.[2]

"MCP eliminates the integration tangle by defining a single protocol both on the client side and on the server side — a bit like USB-C for AI, connecting any model to any resource."

§ 2 — Protocol Design

What Model Context Protocol Actually Is

At its most precise, the Model Context Protocol is an open standard that defines a stateful, JSON-RPC 2.0-based communication contract between AI applications and external data sources or tools.[7] It specifies not just message formats but the full lifecycle of a connection: how capabilities are negotiated at initialisation, how tools and resources are discovered, how data flows in both directions, and how sessions are terminated cleanly.

Three defining properties characterise MCP at the protocol level:

  • Stateful sessions. Unlike REST, which is inherently stateless, MCP maintains a persistent session between client and server. This allows incremental notifications, server-initiated sampling requests, and the avoidance of repeated capability negotiation on every call.[7]
  • Transport agnosticism. The JSON-RPC 2.0 message format is invariant across all transport mechanisms — standard I/O, Server-Sent Events, or Streamable HTTP. The same application logic works locally and remotely.[8]
  • Capability negotiation. Clients and servers declare their supported features during an initialisation handshake. This enables backward-compatible evolution: a client can connect to older servers without errors by discovering what subset of the protocol they support.[9]

The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", and "SHOULD" in the specification are interpreted per RFC 2119 and RFC 8174, giving the standard formal normative weight.[1]

JSON-RPC 2.0 Message Structure

All MCP communication is carried in one of three JSON-RPC 2.0 message types:[9]

JSON-RPC MESSAGE TYPES// 1. Request — expects a response
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "tools/call",
  "params": { "name": "read_file", "arguments": { "path": "/etc/hosts" } }
}

// 2. Response — success or error
{
  "jsonrpc": "2.0",
  "id": 1,
  "result": { "content": [ { "type": "text", "text": "127.0.0.1 localhost" } ] }
}

// 3. Notification — one-way, no response expected
{
  "jsonrpc": "2.0",
  "method": "notifications/tools/list_changed"
}

A message with method and id is a Request; one with result or error and an id is a Response; one with method but no id is a Notification. This self-describing format makes MCP messages straightforward to parse and route.[10]

The Initialisation Handshake

Every MCP connection begins with a mandatory capability-negotiation sequence before any data exchange can occur:[9]

INIT SEQUENCE// Step 1: Client sends initialize with its capabilities
Client → Server: { "method": "initialize", "params": {
  "protocolVersion": "2025-11-25",
  "capabilities": { "roots": { "listChanged": true },
                     "sampling": {} },
  "clientInfo": { "name": "MyAgent", "version": "1.0.0" }
}}

// Step 2: Server responds with its capabilities
Server → Client: { "result": {
  "protocolVersion": "2025-11-25",
  "capabilities": { "tools": { "listChanged": true },
                     "resources": { "subscribe": true } },
  "serverInfo": { "name": "FileServer", "version": "2.1.0" }
}}

// Step 3: Client acknowledges — session is now active
Client → Server: { "method": "notifications/initialized" }
§ 3 — System Architecture

Hosts, Clients, and Servers

MCP defines a three-tier client-host-server architecture. Understanding the distinct role of each tier is prerequisite to building correctly with the protocol.[11]

The MCP Host

The host is the process that orchestrates the entire MCP ecosystem. It is the AI-powered application — Claude Desktop, a coding assistant like Cursor, a custom agent framework — that the end user directly interacts with. The host is responsible for: spawning and managing MCP client instances; enforcing security policies and connection permissions; coordinating LLM integration; and ensuring that multiple clients operate in isolation from one another.[11]

A single host can run many MCP clients simultaneously, each maintaining its own independent session with a different server. This multiplicity enables an agent to, in principle, draw context from a filesystem server, a database server, and a web-search server within a single response cycle.

The MCP Client

Each client is spawned by the host to manage one specific server connection. The client-server relationship is strictly 1:1: a client connects to exactly one server and that server is isolated from all others the host may be managing.[11] The client is responsible for the JSON-RPC lifecycle — sending requests, receiving responses, and forwarding notifications — and for translating the protocol's outputs into a form the host's LLM can use.

The MCP Server

The server is the program that exposes context, tools, and resources. Crucially, the designation "server" describes a functional role, not a physical location: an MCP server may run as a local subprocess on the same machine as the client (using stdio transport), or as a remote cloud service reachable over HTTPS.[12] The server responds to client requests, pushes notifications when its state changes, and — via the sampling primitive — can itself initiate LLM calls through the client.

§ 4 — Core Primitives

What Servers Can Expose

MCP servers communicate capability through three core primitives, plus an experimental fourth. These primitives define the vocabulary of what an AI application can discover, retrieve, and act upon.[12]

Tools

Executable functions the AI can invoke. Examples: database queries, API calls, file operations, shell commands. Treat tool calls as arbitrary code execution — consent and sandboxing are essential.[1]

Resources

Data sources that provide contextual information: file contents, database rows, API responses, live sensor data. Resources are read-oriented; they enrich the model's context without implying execution.[12]

Prompts

Reusable prompt templates or workflows that servers expose as first-class objects. Clients can enumerate and instantiate prompts, enabling server-managed, version-controlled instruction sets.[12]

ℹ

Experimental: Tasks. The 2025-11-25 specification introduces a Tasks primitive — durable execution wrappers for deferred result retrieval and status tracking. Tasks are designed for expensive computations, multi-step workflow automation, and batch processing where the result cannot be returned synchronously within a single request-response cycle.[12]

Sampling: Server-Initiated LLM Calls

One of MCP's most powerful — and security-sensitive — features is sampling: the ability for a server to request that the client perform an LLM inference call on its behalf. This enables agentic workflows where a server pauses mid-task to ask clarifying questions or generate intermediate content, all within the established session and without requiring the server to hold independent API credentials.[13]

It is also the feature that, as Palo Alto Networks Unit 42 documented in December 2025, creates the most significant attack surface: malicious servers can exploit sampling to inject adversarial instructions into LLM context.[14]

§ 5 — Transport Mechanisms

How Messages Actually Move

MCP is transport-agnostic: any channel that supports bidirectional JSON-RPC 2.0 message exchange can carry the protocol. In practice, three mechanisms dominate production deployments.[8]

Transport Use Case Auth Model Latency
stdio Local processes, CLI tools, IDE plugins OS user permissions (no network auth needed) Minimal — no network stack
Streamable HTTP Remote / cloud servers (current standard) Bearer tokens, API keys, OAuth 2.0 Network-dependent
HTTP + SSE Legacy remote deployments Bearer tokens, API keys Network-dependent

Standard Input/Output (stdio)

In stdio transport, the host application spawns the MCP server as a child process and pipes its stdin and stdout streams to the client. JSON-RPC messages are newline-delimited; diagnostic output is emitted on stderr and must never be mixed into stdout.[16]

stdio offers three important properties for local deployments: there is no network exposure, so no ports are opened; the server's lifecycle is tied to the client's — when the host closes, the subprocess terminates; and authentication is handled entirely by OS process permissions, which removes the need for token management.[15]

Streamable HTTP

The current recommended transport for remote servers. Clients send JSON-RPC requests via HTTP POST to a single endpoint; servers may respond immediately or stream responses using Server-Sent Events (SSE) over the same or a separate connection. MCP recommends OAuth 2.0 for obtaining authentication tokens, and TLS is assumed for all remote deployments.[12]

HTTP + Server-Sent Events (SSE) — Legacy

The original remote transport (protocol version 2024-11-05) uses two HTTP interaction patterns simultaneously: a persistent GET connection to an /sse endpoint for server-to-client events, and separate HTTP POST requests to a /messages endpoint for client-to-server data. This split creates statefulness complexity — the server must maintain a mapping from SSE connection IDs to incoming POST requests — and is now superseded by Streamable HTTP for new implementations.[17]

STDIO TRANSPORT — TYPESCRIPT SDKimport { Server } from '@modelcontextprotocol/sdk/server/index.js';
import { StdioServerTransport } from '@modelcontextprotocol/sdk/server/stdio.js';

const server = new Server({ name: 'my-local-server', version: '1.0.0' });

// Register a tool
server.setRequestHandler(CallToolRequestSchema, async (req) => {
  const { name, arguments: args } = req.params;
  if (name === 'read_file') {
    const content = await fs.readFile(args.path, 'utf-8');
    return { content: [{ type: 'text', text: content }] };
  }
});

const transport = new StdioServerTransport();
await server.connect(transport);
§ 6 — Deployment Modes

Creating and Deploying MCP Servers

An MCP server stores and shares conversation data, tools, and resources for AI agents using a standardised protocol for agent memory management. Deployment decisions hinge on latency requirements, scale, security posture, and who will access the server.[18]

Local

Local Desktop Deployment

The MCP server runs directly on the user's machine as a subprocess, communicating via stdio. Software, runtimes, and any required model files are installed locally.

  • Zero network attack surface — no open ports
  • OS user permissions handle all access control
  • Server lifecycle tied to the host application
  • Ideal for personal tools, IDE plugins, offline use
  • Cannot serve multiple users concurrently
Remote / Cloud

Remote Cloud Deployment

The server is containerised or deployed as a virtual machine on AWS, Azure, or GCP, exposing a public HTTPS endpoint and using Streamable HTTP transport.

  • Serves multiple AI clients simultaneously
  • Requires explicit auth (OAuth, API keys, mTLS)
  • CORS configuration needed for browser-based clients
  • Horizontal scaling via load balancers
  • Ideal for enterprise shared services

Building a Minimal MCP Server (Python)

Anthropic publishes SDKs in Python, TypeScript, C#, and Java. The following example uses the Python SDK and the FastMCP framework — a higher-level abstraction that Thoughtworks added to its November 2025 Technology Radar as a Trial entry for simplifying server development.[10]

PYTHON — FastMCPfrom fastmcp import FastMCP
import httpx

mcp = FastMCP("weather-server")

# Expose a tool — the LLM can call this
@mcp.tool()
async def get_weather(city: str) -> str:
    """Fetch current weather for a city."""
    async with httpx.AsyncClient() as client:
        r = await client.get(
            f"https://wttr.in/{city}?format=3"
        )
        return r.text

# Expose a resource — read-only context for the LLM
@mcp.resource("config://settings")
def get_settings() -> str:
    return '{"units": "metric", "lang": "en"}'

# Run locally via stdio (default)
if __name__ == "__main__":
    mcp.run()

Scaling Context with Code Execution

As the number of connected MCP servers grows, loading all tool definitions into the context window on every request becomes prohibitively expensive. Anthropic engineering has documented an approach to this: rather than declaring all tool schemas upfront, the agent is given code-execution capabilities and a filesystem of tool definitions it can read on demand.[13]

A search_tools tool allows the agent to locate relevant definitions before loading them. A detail_level parameter lets the agent choose between loading just tool names, names with descriptions, or full schemas with parameter definitions, preserving context budget for the actual task.[13]

§ 7 — Security Considerations

Attack Surfaces and Mitigations

The MCP specification explicitly acknowledges that tools "represent arbitrary code execution and must be treated with appropriate caution."[1] As adoption scaled through 2025, the attack surface became a significant research focus. In April 2025, security researchers released a comprehensive analysis concluding that multiple outstanding vulnerabilities exist in MCP ecosystems, including prompt injection, tool permissions enabling data exfiltration, and "lookalike tools" capable of silently replacing trusted ones.[2]

⚠

Protocol-level vs. implementation-level vulnerabilities. Research from 2026 distinguishes between vulnerabilities intrinsic to MCP's architectural choices — such as the trust model during capability negotiation — and implementation bugs in specific servers. Both categories require attention; the former demands protocol-level mitigations, while the latter requires code review and secure coding practices.[6]

Prompt Injection via Tool Descriptions

The most pervasive attack class involves embedding adversarial instructions inside tool metadata — descriptions, parameter annotations, or response content — that the LLM receives as part of its context. Because the LLM processes tool descriptions as natural language, a malicious server can coerce the agent into calling unintended tools or exfiltrating data.[14,19]

A documented real-world scenario from May 2025 involved GitHub repository issues containing injected instructions that caused AI assistants to forward private repository data to attacker-controlled endpoints.[20]

Tool Poisoning

Tool poisoning refers to the replacement or modification of legitimate tool definitions with adversarial variants. "Rug pull" attacks alter tool descriptions after the user has reviewed and trusted them; "lookalike" tools use names nearly identical to trusted ones. The MCPTox benchmark, published in 2025, demonstrated that such attacks pass into AI agent context at alarmingly high rates when clients lack rigorous metadata validation.[19]

A backward-compatible mitigation is for client implementations to: (a) display initial tool descriptions to users before first use; and (b) alert users whenever those descriptions change mid-session.[23]

Command Injection via stdio

OX Security's 2026 disclosure advisory identified a systemic command injection vulnerability in MCP stdio configurations affecting multiple servers and generating ten CVEs. When tool input is passed unsanitised to shell commands, prompt-injected payloads containing shell metacharacters (e.g., ; rm -rf /tmp) can achieve remote code execution on the host running the MCP server.[21] The mitigation is straightforward: use execFile instead of exec, pin the command, and pass arguments as array elements rather than interpolating them into a command string.

Least Privilege Violation

During capability negotiation, servers declare all capabilities they support upfront, including those the client has not requested. This violates the principle of least privilege: a client connecting for a narrow task receives a broader capability footprint than necessary, enlarging the blast radius of any compromise.[6]

Specification-Level Security Requirements

The MCP specification SHOULD-level requirements include: building robust consent and authorisation flows; providing clear documentation of security implications; implementing appropriate access controls; following security best practices in integrations; and considering privacy implications in feature design.[1] These remain normative guidance rather than enforcement mechanisms — the protocol cannot enforce them at the wire level.

Threat Vector Mitigation
Prompt injection Tool descriptions, resource content Input sanitisation; user confirmation for sensitive actions
Tool poisoning / rug pull Modified tool metadata Display and alert on description changes; trust pinning
Command injection (stdio) Shell-interpolated tool args Use execFile; treat all args as untrusted values
Sampling abuse Malicious server → LLM via sampling User consent before sampling; sandboxed LLM calls
Least privilege violation Broad capability negotiation Request-scoped capability filtering at the host level
OAuth proxy bypass mcp-remote credential exposure Enforce narrow token scopes; rotate credentials frequently
§ 8 — Ecosystem & Innovations

The MCP Ecosystem in 2026

Within eighteen months of its release, MCP has generated an ecosystem that Thoughtworks describes as "vibrant" in its November 2025 Technology Radar, spanning major industry players and independent open-source contributors simultaneously.[10]

Reference Servers and SDKs

Anthropic maintains open-source reference implementations for Google Drive, Slack, GitHub, Git, Postgres, and Puppeteer. SDKs are available in Python, TypeScript, C#, and Java.[2] The community has produced tens of thousands of servers spanning virtually every category of enterprise software, navigable via marketplace directories like MCP.so.[10]

Notable Tools and Frameworks

FastMCP is a Python framework that significantly reduces the boilerplate required to implement an MCP server, listed as Trial in Thoughtworks' Technology Radar. Context7, also in the Technology Radar, addresses a common LLM failure mode — generating code against outdated library versions — by providing MCP servers with current, version-specific documentation.[10]

Docker MCP Toolkit provides containerised server packaging with an integrated OAuth gateway, addressing authentication scope vulnerabilities that led to real-world data exfiltration incidents in 2025.[20]

Governance: The Agentic AI Foundation

In December 2025, Anthropic donated MCP to the Agentic AI Foundation (AAIF), a directed fund under the Linux Foundation co-founded with Block and OpenAI, with backing from AWS, Google, Microsoft, and Cloudflare. This governance structure signals that the protocol's infrastructure layer is intended to remain open, interoperable, and vendor-neutral — an explicit architectural decision to avoid Anthropic maintaining a competitive moat through a proprietary integration standard.[5]

MCP Apps: Interactive UI Extension

MCP Apps (formerly mcp-ui), formalised under the SEP-1865 specification in early 2026, extends the base protocol to standardise the delivery of interactive user interfaces — React dashboards, forms, data visualisations — from MCP servers to host applications. Where the base MCP specification is restricted to text and structured data, MCP Apps enables servers to push rendered UI components, bringing tool results out of the chat transcript and into interactive widgets.[2]

Integration with the Broader Agentic Stack

MCP now integrates with Microsoft Semantic Kernel, Azure OpenAI, and Cloudflare's serverless compute platform. The AAIF's broader agentic stack combines MCP for tool connectivity with AGENTS.md (OpenAI's markdown-based standard for agent project instructions, adopted by over 60,000 open-source repositories) and goose (Block's open-source agent framework), forming what amounts to a full open stack for agentic application development.[5]
