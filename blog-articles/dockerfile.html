Infrastructure as code · the build recipe

The Dockerfile
a manifest for machines

A Dockerfile is a plain-text recipe that a builder reads, line by line, to assemble a container image. It turns the messy, undocumented ritual of “setting up a server” into a versioned artifact you can diff, review, and rebuild identically anywhere.

FROMRUNCOPYCMD // build → ship → run

01

The manifest

A Dockerfile is a text document that contains every command you might otherwise type by hand to assemble an image. You point docker build at it, and the builder executes those instructions in succession to produce a container image automatically.1 There is nothing exotic in the file itself — it is closer to a shell script than a programming language — but the discipline it imposes is the whole point.

Before Dockerfiles, “setting up the server” was an oral tradition: a wiki page that drifted out of date, a senior engineer's memory, a shell history no one could reconstruct. A Dockerfile replaces that ritual with an artifact. It lives in version control next to the code it serves; it can be diffed, reviewed in a pull request, and rebuilt to produce the same image on a laptop, in CI, and on a production node. This is the core claim of infrastructure as code: the recipe is the environment, and the environment is only ever as trustworthy as the recipe that produced it.

By packaging an application together with its dependencies in a single standardized, open format, Docker sharply lowered the friction between the people who write software and the people who operate it — developers could concentrate on what runs inside the container while operators concentrated on what surrounds it, and the phrase that stuck was build, ship, run.2 The Dockerfile is the “build” half of that promise: the deterministic front end that every later stage depends on.

Here is a complete, unremarkable Dockerfile for a Python web service. Read it top to bottom; the builder does exactly that.

Dockerfile./Dockerfile
# syntax=docker/dockerfile:1
FROM python:3.12-slim          # a known, pinned starting point
WORKDIR /app                    # all later paths are relative to here

# copy the dependency manifest first, on its own, for cache reuse
COPY requirements.txt ./
RUN pip install --no-cache-dir -r requirements.txt

# then the source, which changes far more often
COPY . ./

EXPOSE 8000                       # documentation of the listening port
USER nobody                       # drop out of root before running
CMD ["gunicorn", "-b", "0.0.0.0:8000", "app:app"]

Every idea in this article is already latent in those ten lines: a pinned base image, a working directory, an ordering chosen for the build cache, the difference between build-time and run-time, the decision to run as a non-root user, and a default command expressed as a list rather than a string. The rest of the piece pulls each of those apart — but first, where the box itself came from.

02

Lineage of the box

Isolation, older than the cloud

Containers did not begin with Docker, and neither did the ideas in a Dockerfile. The oldest thread reaches back to 1979, when the chroot system call arrived during development of the Seventh Edition of Unix. It let a process be handed a new apparent root directory, narrowing its view of the filesystem; a few years later, in 1982, it was folded into BSD.3 chroot could move the floor under a process, but it did not stop that process from seeing the rest of the system's users, network, or processes.

Those walls were built later. In 2000 FreeBSD extended the concept into jails, partitioning a host into independent mini-systems each with its own filesystem, users, and network interface. Solaris answered in 2004 with Zones, virtualizing the operating system while sharing a single kernel, and in 2006 engineers at Google shipped “process containers” to cap and account for the resource use of a group of processes. That feature was renamed control groups — cgroups — a year later, and merged into the Linux kernel in release 2.6.24 in 2008.4

Cgroups answered the question of how much a process could consume. The companion mechanism, namespaces, answered what a process could see, arriving incrementally from the early 2000s — mount namespaces landed in Linux 2.4.19 — and later rewritten, so that a second generation, cgroups v2, appeared in Linux 4.5.5 With both primitives in the mainline kernel, a user-space manager could finally knit them together. LXC, released in the summer of 2008, was the first reasonably complete Linux container manager: a library and set of tools wrapping namespaces and cgroups without kernel patches. It would become Docker's original engine.6

The lightning talk

On 15 March 2013, Solomon Hykes stepped onto a lightning-talk stage at PyCon in Santa Clara and demonstrated Docker publicly for the first time, compressing the problem it solved into five words: shipping to the server is hard.7 The five-minute slot landed far harder than anyone expected. For many developers in the room it was a first encounter with cgroups and namespaces at all, and Docker's genuine contribution was less a new kernel feature than an act of democratization — using those primitives no longer demanded a systems-administration background.8

Docker had grown out of dotCloud, a platform-as-a-service company Hykes co-founded in the 2010 Y Combinator batch. The team had already been running many LXC containers in production, each backed by a layered filesystem image, and had written an internal tool to smooth over LXC's rough and unstable edges; that tool quietly became the container runtime the whole company pivoted toward.9 Weeks after the talk, Docker was released as open source under the Apache 2.0 license. At first it used LXC as its execution environment, but with version 0.9 in 2014 it swapped LXC for its own component, libcontainer, written in Go, and in 2017 Docker spun its open-source core out into the Moby project.10

Standardizing the box

Explosive adoption created a fresh risk: fragmentation. If every vendor shipped its own image format and runtime, “run anywhere” would quietly fracture. In June 2015, Docker and a coalition of industry players established the Open Container Initiative (OCI) under the Linux Foundation, and Docker donated its runtime, runC, as the reference implementation. The OCI today maintains three specifications — the runtime, image, and distribution specs — that let a compliant image be produced by one toolchain and executed by any compliant runtime.11 runC first appeared as version 0.0.1 in July 2015 and reached 1.0.0 in June 2021; the image-format work split into its own specification in March 2016.12

The image and runtime specifications both reached 1.0 in July 2017, a little over two years after the group formed. That was the moment a container image stopped being a Docker feature and became an industry standard.13 The practical consequence for this article is quiet but total: the artifact a Dockerfile produces is not a proprietary blob but an OCI image, and the file you write is one particular — and very popular — way of describing one.

03

The layered image

To understand why a Dockerfile is written the way it is, you have to understand the thing it builds. An image is not a disk snapshot. It is a stack of read-only layers, and almost every instruction in the file adds one.

A stack of diffs, unified at read time

The mechanism that stacks the layers is a union filesystem. Docker's default storage driver, overlay2, uses the Linux kernel's OverlayFS to present several directories as one. The image's layers become read-only lower directories; when you start a container the runtime adds a single writable upper directory on top, and the unified view the process actually sees is the merged directory. The overlay2 driver natively supports up to 128 stacked lower layers and requires a reasonably modern kernel.14

Because the lower layers are read-only and shared, editing a file that lives in one of them does not rewrite it in place. OverlayFS copies the file up into the writable layer on first write — copy-on-write — and modifies the copy, leaving the original untouched beneath it. This has a consequence that surprises newcomers and shapes half of Dockerfile best practice: deleting a large file in a later layer does not shrink the image. The bytes still sit in the lower layer, hidden behind a whiteout marker, and they still ship.14

Concretely, each layer is a directory holding a diff — the exact set of filesystem changes that layer introduces — identified by a content hash. Two layers with identical content are stored only once on disk and shared across every image that references them, which is why your fifth image built on the same base pulls almost instantly: the shared base layers are already present.15 The base image supplied by FROM is simply the first layer in the stack, and each subsequent filesystem-changing instruction lays another on top.15

Image, container, tag, digest

Four words get used loosely and are worth pinning down. An image is an immutable, content-addressed template — identified by the SHA-256 digest of its contents.11 A container is a running or stopped instance of that image, distinguished only by its thin writable layer and some runtime configuration; delete it and the underlying image is unharmed. A tag such as python:3.12-slim is a mutable, human-friendly pointer that today references one digest and tomorrow may reference another when the maintainers rebuild it. A digest such as sha256:9b2f… is that immutable content hash. The distinction matters because a Dockerfile that pins only a tag is not fully reproducible: the same file, built next month, can resolve to a different base.

The build context

When you run docker build ., the trailing dot is not decoration — it is the build context. The client packages that directory and hands it to the builder, and COPY and ADD can only reach files that live inside it. A bloated context (a stray node_modules, a .git directory, a multi-gigabyte dataset) is slow to transfer and easy to leak into an image by accident, so a .dockerignore file is used to exclude paths from the context before it is ever sent.1 Treat the context as an explicit input to the build, not as “whatever happens to be in the folder.”

04

The build engine

A Dockerfile is a description, not a program that runs itself. Something has to read it, decide what work is needed, do that work in isolation, and package the result. Since Docker Engine 23.0 that something is BuildKit, and knowing how it thinks explains most of what a Dockerfile author can and cannot do.

From instructions to a graph

On modern engines, docker build is executed by BuildKit by default; Docker Desktop adopted it earlier, and on older engines you opted in with the DOCKER_BUILDKIT=1 environment variable.16 At its heart is a binary intermediate representation called LLB (low-level build), and the design analogy is explicit and useful: LLB is to a Dockerfile what LLVM IR is to C. A component called a frontend parses a human-readable build definition — the Dockerfile frontend is named dockerfile.v0 — and lowers it into LLB, a content-addressable dependency graph of build operations.16

Representing a build as a graph rather than a linear script is what unlocks BuildKit's speed. It can detect and skip build stages whose output is never used, build independent stages in parallel, transfer only the changed files of your build context between successive builds, and drive an entirely rewritten cache that tracks checksums of the build graph and mounted content instead of guessing from image history.17 The legacy builder ran instructions one after another and compared images with heuristics; BuildKit reasons about dependencies directly.

Client, daemon, and the solver

The word frontend is borrowed from compiler theory, and the borrowing runs deep. When you type docker build, the CLI is only a client: it sends the request over gRPC to a daemon, and your Dockerfile together with its build context travels to a BuildKit instance that may be embedded in the engine or running on another machine entirely. BuildKit has been embeddable in the Docker Engine since v18.09 and became the default in v23; it was deliberately split out of the engine's codebase so it could evolve on its own release cadence, and alternative builders such as kaniko and buildah exist for environments where running the Docker daemon is unwelcome.18

Inside BuildKit, the frontend turns each instruction into nodes of a directed acyclic graph — source operations that fetch a base image or context, exec operations that run a command, and file operations that copy or move files — and a solver walks that graph, running each step in a throwaway container (by default through runc, or the containerd runtime) and caching the result by its content hash.19 Two instructions that do not depend on each other can therefore run at the same time, and any node whose inputs are unchanged is served straight from cache.

The syntax directive

Because frontends are distributed as ordinary images, the first line of a modern Dockerfile can pin which frontend interprets the rest of the file: # syntax=docker/dockerfile:1. This decouples the Dockerfile language from the engine version, so features like heredocs and cache mounts become available without upgrading Docker itself.16 It looks like a comment, and to a naive parser it is; to BuildKit it is a version contract.

05

The instruction set

Dockerfile instructions fall into two families. Some build the filesystem and therefore create layers — FROM, RUN, COPY, ADD. Others only write metadata into the image's configuration — ENV, EXPOSE, CMD, ENTRYPOINT, LABEL, and friends — and add no layer of their own. Keeping that distinction in mind is the single fastest way to reason about image size and cache behavior.

FROM — the base

FROM sets the base image and begins a build stage; on its own it is almost always the first instruction in the file, and its argument becomes the first layer of the image.1 The choice of base is the most consequential line in the whole Dockerfile: it dictates the operating-system libraries you inherit, the size floor, and the vulnerability surface you start from.

Pin it. A bare FROM python silently tracks whatever latest resolves to; a tag such as python:3.12-slim is better, and pinning the exact content with a digest is best for true reproducibility. The special image scratch is an explicitly empty base for statically linked binaries, and FROM base AS name gives a stage a name so a later stage can copy from it.1

DockerfileFROM variants
# pin to an immutable digest -- the same bytes, forever
FROM python:3.12-slim@sha256:9b2f...e4a1

# an empty base: nothing but what you add
FROM scratch

# a parameterized, named stage (see multi-stage builds)
ARG GO_VERSION=1.22
FROM golang:${GO_VERSION} AS build

WORKDIR — the cursor

WORKDIR sets the working directory for every RUN, CMD, ENTRYPOINT, COPY, and ADD that follows. If the directory does not exist it is created, relative paths compound across multiple WORKDIR lines, and it is the correct tool for changing directory — a RUN cd /app affects only that one instruction's shell and is forgotten immediately after.1 Always give it an absolute path.

DockerfileWORKDIR
WORKDIR /app          # now /app
COPY . .            # copies context into /app
RUN pwd              # prints /app, not the build root

COPY and ADD — getting files in

Both instructions copy files from the build context into the image, and both create a layer. COPY is the plain, predictable one: it copies files and directories, and nothing else. ADD has two extra powers — it can automatically extract a local tar archive into the destination, and it can fetch a file from a remote URL. Precisely because it does more and its behavior is less obvious, the standing guidance is to reach for COPY by default and use ADD only when you specifically want tar extraction or a remote fetch.1

CapabilityCOPYADD
Copy local files and directories from the build contextyesyes
Auto-extract a local tar archive into the imagenoyes

Recommended defaultpreferredonly for extraction / fetch

Both accept flags worth knowing: --from=build copies out of a named stage instead of the context, --chown=uid:gid sets ownership as it copies, and --chmod sets permissions — each avoiding an extra RUN chown layer.1

DockerfileCOPY
COPY --chown=app:app src/ /app/src/
COPY --from=build /out/server /usr/local/bin/server

RUN — doing work at build time

RUN executes a command in a new layer on top of the current image and commits the result. This is where packages are installed, source is compiled, and directories are created — anything that must happen while building. It is the instruction most often confused with CMD: RUN actually runs a command and commits it at build time, whereas CMD only records the command to run later, at container start.1

Because every RUN is its own layer and layers are copy-on-write, cleanup must happen inside the same RUN that created the mess. Splitting apt-get install and rm -rf /var/lib/apt/lists/* into two instructions leaves the package lists sitting in the earlier layer, shipping forever beneath a whiteout. Chain related steps with && and clean up in one shot:

DockerfileRUN · the apt pattern
RUN apt-get update && apt-get install -y --no-install-recommends \
      ca-certificates curl \
    && rm -rf /var/lib/apt/lists/*

Under BuildKit, RUN gains mounts that never enter the image: a cache mount persists a package or compiler cache across builds, and a secret mount exposes a credential to a single command without baking it into a layer.17

DockerfileRUN --mount
# reuse pip's download cache between builds; nothing is committed
RUN --mount=type=cache,target=/root/.cache/pip \
    pip install -r requirements.txt

# a build-time secret, mounted at a path, never stored in the image
RUN --mount=type=secret,id=npmrc,target=/root/.npmrc \
    npm ci

ENV and ARG — variables, two kinds

The two look similar and behave very differently. ARG defines a build-time variable, set on the command line with --build-arg; it exists only during the build and only from the line that declares it to the end of that stage. An ARG written before the first FROM is special-cased: it can be used in FROM lines but must be redeclared after FROM to be visible inside the stage.1 ENV defines an environment variable that is written into the image's configuration and is therefore present at run time, visible to your application and to every later instruction.1

Two consequences follow. First, build arguments are not a secure way to pass secrets — they can surface in the image's build history and logs, which is exactly why the secret mount above exists.1 Second, substitution is textual and scoped: ${VAR} expands, ${VAR:-default} supplies a fallback, a leading backslash escapes it, and a variable holds the same value throughout a single instruction — so in ENV a=hello then ENV a=bye b=$a, the value of b becomes hello, not bye.1

DockerfileARG vs ENV
ARG NODE_VERSION=20          # build-time only
FROM node:${NODE_VERSION}-slim

ARG NODE_VERSION                 # redeclare to use inside the stage
ENV APP_ENV=production       # persists into the running container
ENV PATH=/app/node_modules/.bin:$PATH

EXPOSE — documentation, not a firewall

EXPOSE declares which ports the containerized application listens on, and it is metadata only. It does not publish anything; it does not open a port on the host. To actually reach the service you still pass -p 8080:8080 to docker run (or a ports mapping in Compose).1 Think of it as a signed note to the next human: “this image intends to serve on 8080.”

DockerfileEXPOSE
EXPOSE 8080/tcp        # declaration; publishing happens at run time

CMD vs ENTRYPOINT — what runs, and who wins

This is the pair everyone trips over, and the confusion is really about two separate axes: form (exec vs shell) and role (default vs fixed). Get both straight and the behavior becomes mechanical.

Form: exec vs shell

RUN, CMD, and ENTRYPOINT each accept two forms. The exec form is a JSON array — ["executable", "arg1"], double quotes required — and it does not start a shell; the executable becomes PID 1 and receives signals directly, but there is no variable substitution. The shell form — executable arg1 — is wrapped in /bin/sh -c, which gives you shell features like $VAR expansion and pipes at the cost of an intervening shell process.1

Dockerfiletwo forms
CMD ["nginx", "-g", "daemon off;"]   # exec: nginx is PID 1
CMD nginx -g "daemon off;"             # shell: runs /bin/sh -c '...'

Role: default vs fixed

CMD supplies defaults for the container. There can be only one CMD — if you write several, only the last takes effect — and any arguments the user passes to docker run override it entirely.1 ENTRYPOINT declares the command that always runs; arguments from docker run are appended to it rather than replacing it. The idiomatic combination is an exec-form ENTRYPOINT naming the fixed executable and an exec-form CMD supplying default, overridable arguments — and when they are paired this way, both should be in exec form.1

Dockerfilethe idiomatic pair
ENTRYPOINT ["python", "app.py"]     # always: python app.py
CMD ["--port", "8000"]            # default, overridable arg
# docker run img              -> python app.py --port 8000
# docker run img --port 9000  -> python app.py --port 9000

The interaction, exhaustively

Every combination resolves deterministically. Read the table as “what becomes PID 1”; ep is the ENTRYPOINT executable and p1 the CMD argument.

ENTRYPOINT ↓  /  CMD →(no CMD)CMD ["p1"] · execCMD p1 · shell

Two rows of that table are traps. The red cell is the classic mistake: a shell-form CMD underneath an exec-form ENTRYPOINT is wrapped in /bin/sh -c and handed to the entrypoint as a single argument, so ENTRYPOINT ["python"] with CMD app.py tries to run python /bin/sh -c app.py and fails; both must be exec form.1 The whole bottom row is the other trap: a shell-form ENTRYPOINT runs as a child of /bin/sh -c, so it is not PID 1, it does not receive signals, and it silently ignores CMD and any run-time arguments.1

Dockerfilewrong vs right
# WRONG: shell-form CMD under exec-form ENTRYPOINT
ENTRYPOINT ["python"]
CMD app.py            # -> python /bin/sh -c app.py  (broken)

# RIGHT: both exec form
ENTRYPOINT ["python"]
CMD ["app.py"]         # -> python app.py

The entrypoint-script pattern

A common production shape is an entrypoint script that performs setup — waiting on a database, running migrations, fixing permissions — and then hands control to whatever CMD requested. The critical last line is exec "$@": exec replaces the shell with the target process so it becomes PID 1 and receives SIGTERM for a graceful shutdown, and "$@" forwards the container's arguments (the CMD).

Dockerfile + entrypoint.shhandoff
ENTRYPOINT ["/docker-entrypoint.sh"]
CMD ["node", "server.js"]

# --- docker-entrypoint.sh ---
#!/bin/sh
set -e
./wait-for-db.sh
exec "$@"          # become PID 1; run the CMD (node server.js)
PID 1 and signals

Prefer exec form for the process you actually want running. In shell form your app is a child of /bin/sh, which may not forward SIGTERM, so docker stop waits out its grace period and then kills the container hard — connections dropped, work lost.1 Exec form (or exec "$@") makes your process PID 1 so it can shut down cleanly.

06

The cache

The build cache is why the order of instructions is a design decision, not a matter of taste. Every layer is keyed by its inputs; when those inputs are unchanged, BuildKit reuses the cached layer instead of rebuilding it, and it keeps reusing until it hits the first instruction whose inputs have changed.17 That instruction — and every instruction below it — is rebuilt.

The practical rule follows directly: order instructions from least to most frequently changing. A base image and system packages change rarely; application source changes constantly. If a frequently changing instruction sits high in the file, it invalidates the expensive layers beneath it on every build.

The textbook mistake is copying the whole source tree in before installing dependencies. Any edit to any file changes the copied layer, so the slow npm install or pip install below it re-runs every single time. The fix is to copy the dependency manifest alone first, install against it, and only then copy the rest of the source — so that editing a source file invalidates just the final, cheap copy.1

Two sharp edges are worth internalizing. First, for RUN the cache key is the literal command string — BuildKit does not look at what the command actually does — so a standalone RUN apt-get update can be served from a stale cache while the world moves on; always chain update with the install that depends on it. Second, keep churn out of the build context entirely: a .dockerignore that excludes .git, logs, and local build output stops incidental file changes from invalidating a COPY ..1 For caches that should survive a cache miss — a package download directory, a compiler cache — the BuildKit --mount=type=cache from the previous section persists them across builds without committing them to a layer.17

07

Multi-stage builds

The heaviest tension in a Dockerfile is that building software and running it need different tools. Compiling needs a full toolchain — compilers, headers, package managers, hundreds of megabytes of it. Running needs almost none of that. Multi-stage builds resolve the tension inside a single file.

They arrived in Docker 17.05, on 4 May 2017, in the same release that let you use an ARG in a FROM line and added a --target flag to build one chosen stage.20 Before them, teams achieved the same result with the awkward “builder pattern”: one Dockerfile to compile, a second to package, and a shell script to shuttle the artifact between them. Multi-stage folds all of that into one file with more than one FROM.

The mechanism is simple and strict. Each FROM begins a new stage; you do the heavy work in an early stage, and in the final stage you COPY --from only the finished artifact across. Everything else in the build stage — the compiler, the intermediate objects, the source itself — is left behind and never ships. This is precisely the standard way to land on a minimal or distroless final image: build in a fat stage, then copy just the binary into a base that has no shell and no package manager.21

DockerfileGo → distroless
# syntax=docker/dockerfile:1

# ---- build stage: heavy, holds the full Go toolchain ----
FROM golang:1.22 AS build
WORKDIR /src
COPY go.mod go.sum ./
RUN go mod download
COPY . .
RUN CGO_ENABLED=0 go build -ldflags="-s -w" -o /out/server ./cmd/server

# ---- final stage: tiny, holds only the binary ----
FROM gcr.io/distroless/static-debian12:nonroot
COPY --from=build /out/server /server
EXPOSE 8080
ENTRYPOINT ["/server"]

The CGO_ENABLED=0 produces a statically linked binary that needs no system libc, which is what lets the final stage be something as spare as distroless/static or even scratch. The size difference between the two stages is routinely an order of magnitude or more, and the diagram makes the flow concrete: only the arrow crosses.

Stages are also useful beyond size. You can COPY --from=nginx:latest to lift a file straight out of another published image, and you can build to an intermediate stage for debugging with docker build --target build . — running the fat stage interactively when something in compilation goes wrong.20

08

Hardening & practice

Almost every Dockerfile best practice reduces to three moves: ship less, run with less privilege, and pin what you depend on. Each one improves size, security, and reproducibility at the same time.

Ship less: minimal base images

The base image sets both the size floor and the vulnerability floor of everything above it. A full distribution — or a language image's default tag — carries a shell, a package manager, and hundreds of utilities your application never runs, and every one of them is attack surface and something a scanner can flag. The most effective single change most Dockerfiles can make is to choose a smaller base.

Distroless images take this to its logical end: pioneered by Google's distroless project and assembled with the Bazel build tool, they contain only your application and its runtime dependencies — no shell, no package manager, nothing else. Size, in this framing, is mostly a proxy for the number of components in the image, and fewer components means less that can go wrong or be targeted. A completely empty base, scratch, is the floor: a statically linked binary can live there alone, and the resulting images are startlingly small — a Go “hello world” around two megabytes, an equivalent C program only a few hundred bytes.22

Distroless is not literally nothing, though. The widely used gcr.io/distroless/static image is roughly two megabytes and still ships CA certificates, timezone data, and an /etc/passwd entry, so most applications simply work without you assembling those pieces by hand.23 The choice between scratch and distroless is a real trade-off: scratch is smaller but you own every file in it, including certificates you must keep current; distroless is a hair larger, but its maintainers rebuild and patch it, and its “nonroot” variants run as an unprivileged user by default. Statically compiled Go and Rust binaries (built with CGO_ENABLED=0 or the musl target) fit scratch; anything that still needs libc wants a distroless base or a minimal distribution like Alpine.24

Run with less privilege: drop root

By default the process inside a container runs as root, and that root is not cosmetic — it maps to real capabilities on the host kernel. A USER instruction that switches to an unprivileged account before CMD sharply limits the blast radius of a compromise, which is exactly why distroless nonroot variants and hardened base images ship running as a non-root uid already.24 Removing the shell compounds the benefit: with no /bin/sh in the image, an attacker who achieves code execution cannot simply drop into a terminal to explore the filesystem, install tooling, or pivot.21 It does not make exploitation impossible, but it removes the easy path.

Pin what you depend on

Pin base images by tag and, where reproducibility matters, by digest, so the image you tested is byte-for-byte the image you ship and you decide when to take an upstream update.1 Pin package versions where it is practical, and never rely on latest. The counterweight to pinning is a cadence: a pinned image that is never rebuilt slowly rots as new vulnerabilities are discovered beneath it, so pair pinning with regular rebuilds and scans — the maintainers of good base images do exactly this on your behalf, which is part of what you are buying when you choose one.24

The short version

Pick the smallest base that runs your app. Build in one stage, ship from a minimal one. Copy the dependency manifest before the source. Clean up in the same RUN. Switch to a non-root USER. Pin the base, and rebuild on a schedule. Keep a .dockerignore. Prefer exec-form ENTRYPOINT so your process is PID 1.

09

The toolchain

You do not have to hold all of this in your head. A small toolchain will lint the file, x-ray the image, and scan it for known vulnerabilities — ideally in CI, on every change.

Lint the file: hadolint

hadolint parses a Dockerfile into an abstract syntax tree and checks it against a body of rules — each prefixed DL — covering security, efficiency, and style, and it embeds ShellCheck to lint the bash inside your RUN instructions as well. Its rules encode much of this article as machine checks: pin versions, clean up apt lists, use an absolute WORKDIR, and — pointedly — the last USER should not be root. It runs as a standalone binary, a container, an editor plugin, a pre-commit hook, or a CI step.25

X-ray the image: dive

dive is an interactive terminal UI that walks a built image layer by layer, showing precisely what each layer added and flagging wasted space, so you can see the stray files, forgotten caches, and accidental copies that inflate an image before you ship it.26 It turns the abstract layer model from earlier into something you can scroll through.

Scan for vulnerabilities: Trivy and Docker Scout

Trivy scans an image for known vulnerabilities in both OS packages and language dependencies and drops cleanly into a CI pipeline; Docker Scout offers similar CVE analysis integrated with the Docker tooling (docker scout cves).24,26 This is where minimalism pays off in the most measurable way: fewer packages means fewer findings to triage. The cost of neglect is measurable too — an analysis of Dockerfiles across more than eleven thousand open-source repositories found recurring inefficiencies, half-jokingly called “Docker smells,” that added on average about 4.6% to image size and, in some cases, more than ten percent.26

Finally, the builder itself is pluggable. Where running the Docker daemon is undesirable — unprivileged CI, Kubernetes-native pipelines — kaniko and buildah build OCI images without it,18 and tools such as DockerSlim take an existing image and strip it down by removing the parts your application never touches.26

10

Everything, a manifest

A container is, in the end, an ordinary process wearing a set of kernel features that let it pretend to be a machine. The Dockerfile is what makes the pretense reproducible — the written record that turns a one-off act of setup into an artifact anyone can rebuild.

Read back through the file with the layered image in mind and every rule falls out of one fact: each instruction that touches the filesystem settles as a layer, layers are read-only and copy-on-write, and only what you explicitly keep gets shipped. That is why you clean up inside the RUN that made the mess, why you copy the dependency manifest before the source, why you build in a fat stage and ship from a spare one, and why the smallest, least privileged base is almost always the right default.

None of it is magic, and that is the point. A good Dockerfile is small, pinned, unprivileged, ordered for the cache, and split into stages. It reads top to bottom, exactly like the sediment it deposits — a legible stack of decisions that rebuilds into the same strata on a laptop, in a pipeline, or on a production node. Build, ship, run begins with a file you can read.2
