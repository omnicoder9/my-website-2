01Why a stack

The stack is the most unassuming of the classical data structures: a list you may only touch at one end. Yet a startling fraction of competitive programming reduces, once you see it correctly, to putting something down now so you can pick it up at exactly the right moment later. That single idea—deferral—is what turns a last-in-first-out container into one of the sharpest tools in a contestant's kit.

Mechanically there is almost nothing to a stack. It supports three operations—push, pop, and peek (inspect the top)—and a textbook implementation performs each in constant time1. Knuth catalogues it among the fundamental linear-list structures, alongside the queue and the deque, distinguished only by where insertions and deletions are allowed2. The whole structure is defined by a discipline rather than a shape: the element you remove is always the one most recently added.

What earns the stack its place in every contest toolkit is not those mechanics but the shape of the problems it dissolves. Two mental models cover almost all of them.

Last-in-first-out processing. When the natural order of work is "most recent first," a stack is the data structure of the order itself: undo histories, depth-first search, recursion, and the parsing of anything nested. The most recently opened context is the first that must be closed.

Deferred resolution. This is the model that wins contests. You encounter an element whose fate cannot be decided yet—an open bracket with no partner, a bar whose rectangle has no right edge, a day with no warmer day in sight—so you record the obligation on the stack and move on. When the resolving event finally arrives, it settles the obligations on top of the stack, in reverse order, and they are removed. The stack holds precisely the set of "still-open" items, and it holds them in exactly the order they will be closed.

A stack is the right tool whenever you need to process elements in last-in-first-out order, or whenever you need to track something that will be resolved later.The working definition

The practical heuristic follows immediately. When you catch yourself thinking "I'll deal with this once I see what comes next," you are describing a stack. The rest of this article is a tour of the canonical problems—and, more importantly, the handful of patterns beneath them—with complete, idiomatic implementations you can switch between Go, Java, C++, Rust, C#, and TypeScript using the bar above.

02The stack in six languages

Every language ships a stack; only the spelling differs. The four operations—push, pop, peek, and emptiness—are what the abstract data type promises, independent of how the bytes are laid out3. In practice you rarely reach for a dedicated Stack class: a dynamic array (a slice, a vector, a Vec, a plain array) is a stack if you only append and remove at the end, and it is usually the fastest option because of cache locality.

A few language-specific notes worth internalizing before the problems begin:

  • In Java, prefer ArrayDeque to the legacy java.util.Stack. The old class extends Vector and synchronizes every method—needless overhead in a single-threaded solver, and it iterates bottom-to-top, which surprises people.
  • In C++, std::stack is an adapter over std::deque by default. When you need raw speed or index access, back it with a vector (std::stack<int, std::vector<int>>) or simply use a vector directly.
  • In Rust, Vec::pop returns Option<T>, so the empty case is part of the type—pattern-match it and the borrow checker keeps you honest.
  • Everywhere, peeking or popping an empty stack is the canonical bug: undefined behaviour in C++, a panic in Rust, a NoSuchElementException in Java. Guard the read.

03Balanced structure: Valid Parentheses

Bracket matching is the purest expression of deferred resolution, and it appears in some guise in a remarkable number of problems. Given a string of (), [], and {}, decide whether every bracket is closed by the correct type in the correct order.

The insight that makes the code almost trivial: when you read an opening bracket, you do not yet know where it ends, but you know exactly which closer it will demand. So push that expected closer. When a closing bracket arrives, it must equal whatever is on top—the most recently opened, still-unclosed context. If the top disagrees, or the stack is empty, the string is invalid. If the stack is empty at the end, every obligation was discharged.

valid_parentheses
C++
func isValid(s string) bool {
    stack := []byte{}
    for i := 0; i < len(s); i++ {
        switch s[i] {
        case '(':
            stack = append(stack, ')')   // push the closer we now owe
        case '[':
            stack = append(stack, ']')
        case '{':
            stack = append(stack, '}')
        default: // a closing bracket: it must settle the most recent debt
            if len(stack) == 0 || stack[len(stack)-1] != s[i] {
                return false
            }
            stack = stack[:len(stack)-1]
        }
    }
    return len(stack) == 0
}
boolean isValid(String s) {
    Deque<Character> stack = new ArrayDeque<>();

            case '[' -> stack.push(']');
            case '{' -> stack.push('}');
            default  -> {                  // closing bracket must match the top
                if (stack.isEmpty() || stack.pop() != c) return false;
            }
        }
    }
    return stack.isEmpty();
}

function isValid(s: string): boolean {
    const stack: string[] = [];
    for (const c of s) {
        if (c === "(") stack.push(")");        // push the closer we now owe
        else if (c === "[") stack.push("]");
        else if (c === "{") stack.push("}");
        else if (stack.pop() !== c) return false;  // undefined on empty -> false
    }
    return stack.length === 0;
}

Pushing the expected closer (rather than the opener) is a small stylistic win: the match check becomes a single equality, with no lookup table. It also sidesteps a notorious trap in Java—comparing two boxed Character objects with == tests reference identity, not value, so always unbox to a primitive (as the code above does by comparing against the primitive char c).

The pattern generalizes in several directions that recur in contests:

  • Minimum Add to Make Parentheses Valid. With a single bracket type, the stack collapses to an integer: track the count of open brackets and a separate count of insertions needed. Recognizing that a one-symbol stack is just a counter is a routine memory optimization.
  • Score of Parentheses and nesting-depth problems push partial results rather than symbols: on ( push a frame, on ) fold the top frame into its parent.
  • Longest Valid Parentheses. Keep a stack of indices with a base sentinel of -1; the length of a valid run is the current index minus the index now exposed at the top after a pop. (A two-pass left/right counting scan solves it in O(1) space.)
  • Generate Parentheses is the inverse problem—enumerating every valid string—and we return to it once the monotonic material is in hand.

04The augmented stack: Min Stack

Design a stack that also reports its minimum element in O(1) time, while keeping push, pop, and top at O(1). The naive approach—scan for the minimum on demand—is O(n) per query, which is exactly what we must avoid. The fix is the recurring theme of the augmented stack: spend a little extra memory to carry a running answer alongside each element.

The cleanest version stores, with every value, the minimum of the stack at the moment that value was pushed. Because a stack only ever grows and shrinks at the top, the minimum-so-far is monotone with respect to stack height: when you pop, the previous minimum is simply the one recorded on the element now exposed. No recomputation is ever required.

min_stack
C++
type MinStack struct {
    data []int
    mins []int // mins[i] = minimum among data[0..i]
}

func Constructor() MinStack { return MinStack{} }

func (s *MinStack) Push(val int) {
    s.data = append(s.data, val)
    if len(s.mins) == 0 || val <= s.mins[len(s.mins)-1] {
        s.mins = append(s.mins, val)
    } else {
        s.mins = append(s.mins, s.mins[len(s.mins)-1])
    }
}

func (s *MinStack) Pop()       { s.data = s.data[:len(s.data)-1]; s.mins = s.mins[:len(s.mins)-1] }
func (s *MinStack) Top() int   { return s.data[len(s.data)-1] }
func (s *MinStack) GetMin() int { return s.mins[len(s.mins)-1] }
class MinStack {
    // each frame carries {value, min-so-far}
    private final Deque<int[]> stack = new ArrayDeque<>();

    public void push(int val) {
        int min = stack.isEmpty() ? val : Math.min(val, stack.peek()[1]);
        stack.push(new int[]{val, min});
    }
    public void pop()   { stack.pop(); }
    public int top()    { return stack.peek()[0]; }
    public int getMin() { return stack.peek()[1]; }
}
class MinStack {
    stack<pair<int,int>> st; // {value, min-so-far}
public:
    void push(int val) {
        int mn = st.empty() ? val : min(val, st.top().second);
        st.push({val, mn});
    }
    void pop()   { st.pop(); }
    int top()    { return st.top().first; }
    int getMin() { return st.top().second; }
};
struct MinStack {
    stack: Vec<(i32, i32)>, // (value, min-so-far)
}

impl MinStack {
    fn new() -> Self { MinStack { stack: Vec::new() } }

    fn push(&mut self, val: i32) {
        let min = match self.stack.last() {
            Some(&(_, m)) => m.min(val),
            None => val,
        };
        self.stack.push((val, min));
    }
    fn pop(&mut self)    { self.stack.pop(); }
    fn top(&self) -> i32 { self.stack.last().unwrap().0 }
    fn get_min(&self) -> i32 { self.stack.last().unwrap().1 }
}
public class MinStack {
    // each frame carries (value, min-so-far)
    private readonly Stack<(int val, int min)> stack = new();

        int min = stack.Count == 0 ? val : Math.Min(val, stack.Peek().min);
        stack.Push((val, min));
    }
    public void Pop()    => stack.Pop();
    public int Top()     => stack.Peek().val;
    public int GetMin()  => stack.Peek().min;
}
class MinStack {
    // each frame is [value, min-so-far]
    private stack: [number, number][] = [];

    push(val: number): void {
        const min = this.stack.length === 0
            ? val
            : Math.min(val, this.stack[this.stack.length - 1][1]);
        this.stack.push([val, min]);
    }
    pop(): void   { this.stack.pop(); }
    top(): number { return this.stack[this.stack.length - 1][0]; }
    getMin(): number { return this.stack[this.stack.length - 1][1]; }
}

Two refinements are worth knowing. First, instead of duplicating the minimum on every frame, keep a second stack that records a new value only when it is less than or equal to the current minimum; this saves memory when minima change rarely (push to the min-stack on val ≤ min, and pop it only when the value leaving equals the current min). Second, an elegant single-stack trick stores each entry encoded as a difference from the current minimum, so one stack of integers suffices—at the cost of needing 64-bit arithmetic to avoid overflow when values straddle a wide range.

The same idea—a stack carrying an associative running aggregate—gives a Max Stack for free, and underlies the classic Implement Queue using Two Stacks: an "in" stack receives pushes, an "out" stack serves pops, and you transfer the whole in-stack into the out-stack only when the latter empties. Each element moves between stacks at most once, so the queue's operations are O(1) amortized—our first taste of the accounting that makes the next pattern fast.

05The monotonic stack

If you learn one stack technique for contests, learn this one. A monotonic stack is a stack you deliberately keep sorted—every value below the top is no larger than it (a non-decreasing stack), or no smaller (a non-increasing stack). The discipline is enforced on the way in: before pushing a new element, you pop everything that would violate the order. The revelation is that those pops are not overhead—they are the answers. The moment an element is popped is the moment its pending query gets resolved.

Consider the archetype, Next Greater Element: for each position, find the next position to its right holding a larger value. Maintain a stack of indices whose values are strictly decreasing. Each new value pops every smaller index off the top—and the new value is, by construction, the next-greater element for every index it pops. Whatever survives the pops is still waiting for a larger value to its right; the new index joins them.

Monotonic stack · next-greater template
# answer[i] = index of the next element greater than a[i] (or NONE)
stack ← empty            # will hold indices, a[] values decreasing
for i in 0 .. n-1:
    while stack not empty and a[i] > a[stack.top]:
        j ← stack.pop      # a[i] is j's answer — resolved now
        answer[j] ← i
    stack.push(i)         # i waits for ITS next greater element
# flip > to < for next-smaller; iterate n-1..0 for previous-* queries

Why is this efficient? The inner while loop looks dangerous, but each index is pushed exactly once and popped at most once over the entire sweep. The total number of pops is therefore bounded by n, so the whole algorithm is O(n) despite the nested loop—the textbook result of amortized analysis1, whose accounting framework was formalized by Tarjan4. Monotonic-stack sweeps are a staple of competitive programming references for exactly this reason56.

Daily Temperatures

This is Next Greater Element wearing a thin disguise: given daily temperatures, for each day report how many days you must wait for a warmer one (or 0 if none comes). Instead of storing the next-greater index, store the distance to it. The stack holds indices of days still awaiting a warmer day, temperatures decreasing from bottom to top; today pops—and answers—every colder day it dominates.

daily_temperatures
C++
func dailyTemperatures(temps []int) []int {
    n := len(temps)
    res := make([]int, n)
    stack := []int{} // indices whose answer is still pending; temps decreasing
    for i := 0; i < n; i++ {
        for len(stack) > 0 && temps[i] > temps[stack[len(stack)-1]] {
            j := stack[len(stack)-1]
            stack = stack[:len(stack)-1]
            res[j] = i - j // today resolves day j
        }
        stack = append(stack, i)
    }
    return res
}

The same skeleton, with the comparison flipped or the iteration reversed, solves an entire family:

  • Next Greater Element II handles a circular array by iterating over 2n positions with a modular index, so wrap-around neighbours are considered.
  • Online / Offline Stock Span asks for the run of consecutive prior days with price no greater than today's—a previous-greater query answered by the mirror-image stack.
  • Trapping Rain Water can be read as a monotonic-stack problem: a decreasing stack of bar indices, where each pop closes off a horizontal basin bounded by the incoming taller bar on the right and the new top on the left, contributing (min(left, right) − floor) × width units of water. (Two pointers solve it in O(1) space; the stack version generalizes more readily.)
  • Sum of Subarray Minimums uses the contribution technique: each element is the minimum of (distance to the previous strictly-smaller element) × (distance to the next smaller-or-equal element) subarrays. The asymmetry between strict and non-strict is what prevents double-counting equal values—a subtle but essential detail—and both boundaries come from monotonic stacks, with the running sum taken modulo the required prime.
  • Remove K Digits and Remove Duplicate Letters turn a greedy lexicographic choice into a monotonic stack: keep the result non-decreasing by popping a larger preceding character whenever doing so is still affordable.

06Largest Rectangle in a Histogram

This is the problem that elevates the monotonic stack from a trick to a technique, and it is a near-universal interview and contest staple. Given a histogram of unit-width bars, find the area of the largest axis-aligned rectangle that fits inside it. A rectangle is limited by its shortest bar, so the question for each bar is: how far left and how far right can a rectangle of this bar's height extend? It can extend until it meets a strictly shorter bar on either side—precisely the previous-smaller and next-smaller boundaries that monotonic stacks compute.

Maintain a stack of indices with increasing heights. When the incoming bar is shorter than the bar on top, the top bar can grow no further right, so its maximal rectangle is now fully determined: its right boundary is the current index i, its left boundary is the index newly exposed beneath it, and its width is i − newTop − 1 (or i when the stack empties, meaning the bar extends all the way to the left). Finalize that rectangle, then continue popping. A virtual zero-height bar appended at the end (the sentinel) guarantees every remaining bar is flushed and measured.

largest_rectangle
C++
func largestRectangleArea(heights []int) int {
    n := len(heights)
    stack := []int{} // indices with strictly increasing heights
    best := 0
    for i := 0; i <= n; i++ {
        h := 0 // virtual zero-height sentinel at i == n flushes the stack
        if i < n {
            h = heights[i]
        }
        for len(stack) > 0 && heights[stack[len(stack)-1]] >= h {
            top := stack[len(stack)-1]
            stack = stack[:len(stack)-1]
            height := heights[top]
            width := i
            if len(stack) > 0 {
                width = i - stack[len(stack)-1] - 1
            }
            if area := height * width; area > best {
                best = area
            }
        }
        stack = append(stack, i)
    }
    return best
}
long largestRectangleArea(int[] heights) {
    int n = heights.length;
    Deque<Integer> stack = new ArrayDeque<>(); // increasing heights
    long best = 0;
    for (int i = 0; i <= n; i++) {
        int h = (i == n) ? 0 : heights[i]; // sentinel flushes everything
        while (!stack.isEmpty() && heights[stack.peek()] >= h) {
            int height = heights[stack.pop()];
            int width = stack.isEmpty() ? i : i - stack.peek() - 1;
            best = Math.max(best, (long) height * width); // 64-bit: avoid overflow
        }
        stack.push(i);
    }
    return best;
}

fn largest_rectangle_area(heights: Vec<i32>) -> i64 {
    let n = heights.len();
    let mut stack: Vec<usize> = Vec::new(); // increasing heights
    let mut best: i64 = 0;
    for i in 0..=n {
        let h = if i == n { 0 } else { heights[i] }; // sentinel flush
        while let Some(&top) = stack.last() {
            if heights[top] >= h {
                stack.pop();
                let height = heights[top] as i64;
                let width = match stack.last() {
                    Some(&prev) => (i - prev - 1) as i64,
                    None => i as i64,
                };
                best = best.max(height * width); // i64 avoids overflow
            } else {
                break;
            }
        }
        stack.push(i);
    }
    best
}

            const height = heights[stack.pop()!];

    return best;

Tracing [2,1,5,6,2,3]: the bars of height 5 and 6 sit on the stack until the 2 at index 4 arrives. Popping the 6 yields a width-1 rectangle; popping the 5 yields a width-2 rectangle of area 10—the answer—because its left boundary is now the bar of height 1 at index 1. Two details decide correctness in practice. Use ≥ (pop on equal heights) together with the width formula above, so runs of equal bars are measured at full width. And compute the area in 64-bit arithmetic: with heights up to 104 and widths up to 105, the product overflows a 32-bit integer.

The boundary machinery here is reusable far beyond one problem:

  • Maximal Rectangle in a binary matrix reduces to this directly: for each row, treat the number of consecutive 1s ending at each column as a histogram, and run the routine on every row. The result is O(rows × cols).
  • The very same previous-/next-smaller boundaries drive Sum of Subarray Minimums, the "largest rectangle under a skyline" variants, and a host of contribution-counting problems—which is why mastering this single sweep pays compound interest.

07Backtracking and the call stack: Generate Parentheses

Where is the stack in generating all valid parenthesizations? It is the call stack. Producing every balanced string of n pairs is a constrained depth-first search, and the recursion's own stack holds the string being built. The constraints are tiny and total: you may append ( while fewer than n have been opened, and you may append ) only while there are open brackets left unmatched (closed < open). Maintain that invariant and every completed string is valid by construction—no separate validity check is ever needed.

A few things worth noticing. The number of results is the n-th Catalan number, Cn = (2n)! / ((n+1)!·n!), so any correct algorithm is Ω(Cn); the win is in never generating an invalid candidate, not in beating that bound. The C++, Java, C#, and Rust versions use the mutate-then-undo idiom—append a character, recurse, remove it—which is the in-place analogue of pushing and popping on an explicit stack. The Go and TypeScript versions instead pass an immutable string forward; in Go this matters, because reusing a slice via append across the two branches can alias the backing array and corrupt results, a footgun immutable strings sidestep.

More broadly, an explicit stack is the standard way to convert recursion to iteration—to avoid stack overflow on adversarial deep inputs, or to gain fine control over traversal order. Iterative tree walks (pre-, in-, and post-order), iterative DFS on graphs, and flood fills are all the call stack made manifest as a data structure you manage yourself.

08Expression evaluation

Stacks are the engine of expression evaluation, and the connection runs deep: matching brackets and parsing nested grammar are the same act of deferral, which is why a pushdown automaton—a finite machine plus a stack—is exactly what recognizes context-free languages. Three idioms cover most contest needs.

Reverse Polish (postfix) notation is the simplest. With operands already in postfix order, a single stack suffices: push numbers; on an operator, pop the two most recent operands, apply, and push the result. There is no precedence logic at all—the ordering did that work in advance. Mind only the operand order, since subtraction and division are not commutative.

Reverse Polish notation · one operand stack
stack ← empty
for token in expression:
    if token is a number:
        stack.push(token)
    else:                       # a binary operator
        b ← stack.pop            # mind the order: b is the right operand
        a ← stack.pop
        stack.push(apply(token, a, b))
return stack.pop             # the single remaining value

Infix to postfix is the shunting-yard algorithm, due to Dijkstra8: scan the infix tokens, emit operands immediately, and use an operator stack governed by precedence and associativity to decide when a pending operator must be output before the incoming one. Direct infix evaluation is Dijkstra's two-stack method—one stack for operands, one for operators—resolving an operator when a closing parenthesis or a higher-precedence boundary appears; Sedgewick and Wayne present it cleanly for fully parenthesized expressions7. The contest workhorses Basic Calculator I–III are applications: a running value plus the last operator handles a single precedence level, and a stack saves and restores that state across nested parentheses.

09Collapse and simulation

A final family uses the stack to model a process that resolves locally: each incoming token either extends the structure or cancels and condenses whatever sits on top—the "reduce" step of a stack machine.

  • Decode String (3[a2[c]] → accaccacc): on [ push the current partial string and its repeat count; on ] pop them and splice the repeated segment back. Nested multipliers compose without special cases.
  • Asteroid Collision: push right-moving asteroids; an incoming left-mover annihilates smaller right-movers off the top until it is destroyed, both vanish on a tie, or it survives to be pushed. The stack is the set of asteroids that have not yet collided.
  • Simplify Path: split a Unix path on /, push directory names, pop on .., and ignore . and empty segments—the stack is the canonical path.
  • Backspace String Compare: build each string on a stack where # pops the last character, then compare. (A right-to-left two-pointer scan does it in O(1) space.)

10Recognizing the pattern

Step back from the individual problems and the decision becomes almost mechanical. The signal in the problem statement tells you not only that a stack applies but which flavour.

The cost profile is reassuringly uniform. The individual operations are O(1); a single left-to-right (or right-to-left) sweep that pushes and pops each element once runs in O(n) time and O(n) space, dropping to O(1) extra space in the degenerate cases where the stack collapses to a counter.

Pitfalls that cost submissions

Empty-stack reads. Always guard peek/pop; the empty case is undefined behaviour in C++ and a thrown exception or panic elsewhere.

Off-by-one widths. In the histogram and span problems, the width after a pop is i − newTop − 1, not i − newTop—and it becomes i when the stack empties.

Integer overflow. height × width and subarray-sum accumulations overflow 32 bits; compute in 64-bit (or modulo the required prime).

Strict vs. non-strict comparisons. When counting contributions over equal values, one boundary must be strict and the other inclusive, or you double-count.

Language traps. Java: never compare boxed Integer/Character with ==, and prefer ArrayDeque over Stack. C++: pick the underlying container deliberately. Go: beware append aliasing when branching. Rust: usize subtraction underflows—order the operands so the result is non-negative.

The throughline of every problem here is the one idea we started with. A stack lets you defer a decision—cheaply, and in a way that automatically resurfaces the right pending item at the right time. Master the recognition (the table above) and the accounting (each element in and out at most once), and a whole stratum of contest problems stops being a collection of tricks and becomes a single, familiar move.
