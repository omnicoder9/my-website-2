Introduction

Android is the world's most widely deployed operating system, powering more than three billion active devices as of 2023.[1] Its success is built on a carefully designed application framework that imposes a structured, component-based model on every app — a model that is at once flexible enough to support millions of diverse applications and predictable enough for the runtime to manage resources automatically.

Unlike traditional desktop programs with a single entry point (main()), Android applications are composed of loosely coupled, independently activatable building blocks called components. The Android runtime can instantiate any component on demand, even reaching into another application's components when the user navigates across app boundaries. This architecture enables powerful features such as deep-linking, inter-app data sharing, and cross-app navigation — but it also demands that developers understand each component's purpose, lifecycle, and concurrency model precisely.[2]

"An Android app is not a monolith with a single entry point — it is an ecosystem of components that the operating system can start, stop, and reclaim independently."

This article provides a comprehensive technical reference for all five Android application components — Activities, Services, Broadcast Receivers, Content Providers, and Intents — covering their design rationale, lifecycle semantics, concurrency considerations, and best practices with annotated Kotlin source examples.

Architecture Overview

Android's application framework is layered above a Linux kernel. User-space system services — such as the Activity Manager Service (AMS), the Package Manager, and the Window Manager — mediate all component interactions via Binder IPC, a high-performance inter-process communication mechanism custom-built for Android.[3]

Every Android application runs in its own Linux process with a unique user ID. The Dalvik/ART virtual machine hosts each app process, providing memory isolation between apps. The system can kill any app process under memory pressure, relying on component lifecycle callbacks to allow apps to save state beforehand.[4]

Component Has UI? Activated By Primary Use Case
Activity Yes Intent Single UI screen; user interaction
Service No Intent / Bind Long-running background work
BroadcastReceiver No System broadcast / Intent Respond to system or app-level events
ContentProvider No ContentResolver Structured data sharing across apps
Intent — Application code Message/command to activate components

AndroidManifest.xml

Every app must include an AndroidManifest.xml file at the root of the project. The manifest declares all components the app exposes, as well as hardware and software features the app requires, permissions the app needs, and the minimum API level the app supports.[5]

ℹ

Components that are not declared in the manifest cannot be started by the system or other apps. The Package Manager reads the manifest at install time to register the app's capabilities with the system.

<!-- AndroidManifest.xml (abridged) -->
<manifest xmlns:android="http://schemas.android.com/apk/res/android"
    package="com.example.myapp">

    <uses-permission android:name="android.permission.INTERNET" />

    <application
        android:label="@string/app_name"
        android:theme="@style/AppTheme">

        <!-- Activity declaration -->
        <activity android:name=".MainActivity"
            android:exported="true">
            <intent-filter>
                <action android:name="android.intent.action.MAIN" />
                <category android:name="android.intent.category.LAUNCHER" />
            </intent-filter>
        </activity>

        <!-- Service declaration -->
        <service android:name=".MusicService"
            android:foregroundServiceType="mediaPlayback" />

        <!-- BroadcastReceiver -->
        <receiver android:name=".BootReceiver"
            android:exported="true">
            <intent-filter>
                <action android:name="android.intent.action.BOOT_COMPLETED" />
            </intent-filter>
        </receiver>

        <!-- ContentProvider -->
        <provider
            android:name=".DataProvider"
            android:authorities="com.example.myapp.provider"
            android:exported="false" />

    </application>
</manifest>

Activities

Component 01 — android.app.Activity

An Activity is the fundamental unit of user interaction on Android. It represents a single screen with a user interface, typically corresponding to one task the user wishes to perform — composing an email, browsing a photo gallery, or entering a destination in a maps app.[6]

Architecturally, an Activity hosts a window managed by the Window Manager service. The UI is described as a hierarchy of View and ViewGroup objects inflated from XML layout resources (or, increasingly, declared via Jetpack Compose). User input events — touch, keyboard, sensor — are dispatched through this view hierarchy.[7]

A minimal Activity subclass in Kotlin:

class MainActivity : AppCompatActivity() {

    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        setContentView(R.layout.activity_main)

        // Restore saved state if returning from system kill
        savedInstanceState?.let {
            val userInput = it.getString("KEY_USER_INPUT")
            // restore UI state...
        }
    }

    override fun onSaveInstanceState(outState: Bundle) {
        super.onSaveInstanceState(outState)
        outState.putString("KEY_USER_INPUT", binding.editText.text.toString())
    }
}
Kotlin
⚠

Activities should never perform long-running operations on the main thread. Doing so blocks the UI and risks an Application Not Responding (ANR) dialog if the block exceeds 5 seconds. Use coroutines, WorkManager, or a bound Service for background work.[8]

Activity Lifecycle

The Activity lifecycle is one of the most critical concepts for any Android developer to master. The system controls when Activities are created, started, resumed, paused, stopped, and destroyed — often in response to user navigation or memory pressure — calling corresponding callback methods that the developer overrides to manage state correctly.[9]

State Preservation

When the system destroys an Activity due to a configuration change (e.g. screen rotation), it recreates it immediately. The ViewModel component from Android Jetpack survives configuration changes and is the recommended place to hold UI-related data.[10] Transient UI state (text field content, scroll position) is saved via onSaveInstanceState() and restored in onCreate() or onRestoreInstanceState().

// Surviving configuration changes with ViewModel
class UserViewModel : ViewModel() {
    private val _users = MutableStateFlow<List<User>>(emptyList())
    val users: StateFlow<List<User>> = _users.asStateFlow()

    init { fetchUsers() }

    private fun fetchUsers() {
        viewModelScope.launch {
            _users.emit(repository.getUsers())
        }
    }
}

Task Back Stack

Android groups related Activities into tasks. A task is a collection of activities that users interact with when trying to accomplish a goal. Activities are arranged in a stack — the back stack — in the order each was opened. Pressing the system Back button pops the top Activity from the stack, returning the user to the one beneath it.[11]

Activity A — Launcher (bottom)
Activity B — List Screen
Activity C — Detail Screen (top / foreground)
← FOREGROUND

Launch Modes

The android:launchMode attribute in the manifest controls how new instances of an Activity relate to the back stack. The four modes are standard, singleTop, singleTask, and singleInstance.[12]

Launch Mode New Instance? Back Stack Behaviour
standard Always Multiple instances allowed; default behaviour
singleTop If not on top Reuses top instance; calls onNewIntent()
singleTask If not in task Clears activities above existing instance
singleInstance Once ever Isolated in its own task

Services

Component 02 — android.app.Service

A Service is an application component that can perform long-running operations in the background without providing a user interface. It can be started by another component and continues running even if the user switches to another application.[13]

Services run on the main thread of their hosting process by default. This is a common source of bugs: blocking the main thread in a Service causes the same ANR risk as in an Activity. Services should launch coroutines or hand work off to a thread pool.

Service Types

Type Started How? Lifecycle Example Use
Started startService() / startForegroundService() Runs until stopSelf() or stopService() File download
Bound bindService() Lives while at least one client bound Music playback API
Foreground startForeground() Persistent notification; system won't kill GPS tracker, music player
Background (limited) startService() OS restricts since API 26; prefer WorkManager Periodic sync

Since Android 8.0 (API 26), the system imposes significant restrictions on background services: apps running in the background may not create background services at all. The recommended alternative is WorkManager, which handles deferrable, guaranteed background work in a battery-efficient manner.[14]

// Foreground Service for music playback
class MusicService : Service() {

    private val binder = MusicBinder()

    inner class MusicBinder : Binder() {
        fun getService(): MusicService = this@MusicService
    }

    override fun onStartCommand(intent: Intent?, flags: Int, startId: Int): Int {
        val notification = buildNotification()
        startForeground(NOTIFICATION_ID, notification)
        startPlayback()
        return START_STICKY  // Restart if killed; preserve intent
    }

    override fun onBind(intent: Intent): IBinder = binder

    override fun onDestroy() {
        stopPlayback()
        super.onDestroy()
    }
}
Kotlin

Service Return Values

onStartCommand() returns an integer that controls restart behaviour after a process kill:

  • START_STICKY — Restart; deliver null intent (media playback)
  • START_NOT_STICKY — Do not restart (one-shot jobs)
  • START_REDELIVER_INTENT — Restart; re-deliver last intent (file download)

Broadcast Receivers

Component 03 — android.content.BroadcastReceiver

A Broadcast Receiver is a component that enables the system to deliver events to the app outside of a normal user flow, allowing the app to respond to system-wide broadcast announcements.[15] Examples include the device completing boot, battery level changing, network connectivity changing, or an SMS arriving.

Receivers can be registered in two ways: statically via the manifest (survives app restarts; note the restrictions since API 26) or dynamically via Context.registerReceiver() (tied to the registering component's lifecycle).[16]

⚠

The system dispatches broadcasts on the main thread. onReceive() must complete within 10 seconds or the system will raise an ANR. For any work taking longer, call goAsync() to obtain a PendingResult and complete work on a background thread, or schedule a JobService.[17]

// Statically registered receiver (manifest declares it)
class BootReceiver : BroadcastReceiver() {

    override fun onReceive(context: Context, intent: Intent) {
        if (intent.action == Intent.ACTION_BOOT_COMPLETED) {
            // Schedule recurring sync after device restarts
            WorkManager.getInstance(context)
                .enqueue(OneTimeWorkRequestBuilder<SyncWorker>().build())
        }
    }
}

// Dynamic registration (tied to Activity lifecycle)
class NetworkActivity : AppCompatActivity() {

    private val receiver = object : BroadcastReceiver() {
        override fun onReceive(ctx: Context, intent: Intent) {
            updateNetworkStatus()
        }
    }

    override fun onResume() {
        super.onResume()
        val filter = IntentFilter(ConnectivityManager.CONNECTIVITY_ACTION)
        registerReceiver(receiver, filter)
    }

    override fun onPause() {
        unregisterReceiver(receiver)
        super.onPause()
    }
}
Kotlin

Ordered vs. Normal Broadcasts

Normal broadcasts are delivered to all registered receivers simultaneously in undefined order. Ordered broadcasts are delivered to receivers one at a time in priority order; each receiver can propagate or abort the broadcast to the next.[18] Since API 26, many implicit broadcast intents are no longer delivered to statically registered receivers; dynamic registration or JobScheduler/WorkManager must be used instead.

Content Providers

Component 04 — android.content.ContentProvider

A Content Provider manages access to a structured set of data and provides mechanisms for defining data security.[19] It is the standard Android API for sharing data between processes. System providers expose contacts, calendar events, media files, and call logs through this mechanism. Apps access a provider through a ContentResolver, which communicates with the provider via Binder IPC.

Data is addressed with a content URI of the form:

content://<authority>/<path>/[<id>]

For example: content://com.android.contacts/contacts/42 refers to contact record 42 in the system contacts provider.

// Implementing a ContentProvider
class TaskProvider : ContentProvider() {

    private lateinit var db: TaskDatabase

    override fun onCreate(): Boolean {
        db = TaskDatabase.getInstance(context!!)
        return true
    }

    override fun query(
        uri: Uri, projection: Array<String>?,
        selection: String?, selectionArgs: Array<String>?,
        sortOrder: String?
    ): Cursor? {
        return when (URI_MATCHER.match(uri)) {
            TASKS   -> db.taskDao().getAllCursor()
            TASK_ID -> db.taskDao().getByIdCursor(ContentUris.parseId(uri))
            else    -> throw IllegalArgumentException("Unknown URI: $uri")
        }
    }

    override fun getType(uri: Uri): String =
        when (URI_MATCHER.match(uri)) {
            TASKS   -> "vnd.android.cursor.dir/vnd.example.tasks"
            TASK_ID -> "vnd.android.cursor.item/vnd.example.task"
            else    -> throw IllegalArgumentException("Unknown URI")
        }

    // insert(), update(), delete() omitted for brevity
    override fun insert(uri: Uri, values: ContentValues?): Uri? = null
    override fun update(uri: Uri, values: ContentValues?, sel: String?, args: Array<String>?) = 0
    override fun delete(uri: Uri, sel: String?, args: Array<String>?) = 0
}
Kotlin
✦

With android:exported="false", a ContentProvider is only accessible within the same app. Setting it to true with permission controls enables cross-app data sharing. The Photo Picker API (Android 13+) is now preferred over direct MediaStore access for photo selection, improving user privacy.[20]

Accessing a System Provider

// Query system contacts via ContentResolver
val cursor = contentResolver.query(
    ContactsContract.Contacts.CONTENT_URI,
    arrayOf(ContactsContract.Contacts.DISPLAY_NAME_PRIMARY),
    null, null,
    "${ContactsContract.Contacts.DISPLAY_NAME_PRIMARY} ASC"
)
cursor?.use {
    while (it.moveToNext()) {
        val name = it.getString(0)
        Log.d("Contacts", name)
    }
}

Intents

Component 05 — android.content.Intent

An Intent is a messaging object used to request an action from another app component. It is the primary mechanism for late runtime binding between components: an Intent can activate an Activity, start or bind to a Service, or deliver a broadcast to a BroadcastReceiver.[21]

Explicit Intents

An explicit Intent specifies the fully qualified class name of the target component. Explicit Intents are used for intra-app navigation, where the developer knows exactly which component to start.[22]

// Explicit Intent — navigate to DetailActivity
val intent = Intent(this, DetailActivity::class.java).apply {
    putExtra("ITEM_ID", item.id)
    putExtra("ITEM_TITLE", item.title)
}
startActivity(intent)

// Retrieve in DetailActivity.onCreate()
val itemId   = intent.getLongExtra("ITEM_ID", -1L)
val itemTitle = intent.getStringExtra("ITEM_TITLE")
Kotlin

Implicit Intents

An implicit Intent declares a general action to perform without naming the target component. The Android system finds a suitable component by comparing the Intent to declared Intent Filters across all installed apps.[23] If multiple components can handle the Intent, the system presents a chooser dialog.

// Implicit Intent — share text via any compatible app
val shareIntent = Intent().apply {
    action = Intent.ACTION_SEND
    putExtra(Intent.EXTRA_TEXT, "Check out this article!")
    type = "text/plain"
}
startActivity(Intent.createChooser(shareIntent, "Share via"))

// Implicit Intent — open a URL in the browser
val webpage = Uri.parse("https://developer.android.com")
val webIntent = Intent(Intent.ACTION_VIEW, webpage)
if (webIntent.resolveActivity(packageManager) != null) {
    startActivity(webIntent)
}
Kotlin
ℹ

Since Android 11 (API 30), package visibility restrictions mean you must declare the packages your app interacts with in the manifest via <queries> elements, or use QUERY_ALL_PACKAGES permission (with Play Store justification), before resolving implicit Intents targeting other apps.[24]

PendingIntent

A PendingIntent wraps an Intent and grants another application (or the system) the right to fire that Intent on your app's behalf — with your app's identity and permissions. PendingIntents are used with notifications, AlarmManager, and widgets.[25]

// PendingIntent for a notification action
val actionIntent = Intent(this, MusicService::class.java).apply {
    action = "ACTION_PAUSE"
}
val pendingIntent = PendingIntent.getService(
    this, 0, actionIntent,
    PendingIntent.FLAG_UPDATE_CURRENT or PendingIntent.FLAG_IMMUTABLE
)
Kotlin

Intent Filters

An Intent Filter is a declaration in the manifest (or set programmatically) that specifies what types of Intents a component can respond to. The filter describes the component's capability in terms of three elements: action, data (URI and MIME type), and category.[26]

The system evaluates a potential match using all three elements — an Intent must pass all three tests to be delivered to the component:

Filter Element Purpose Example
<action> What the component does android.intent.action.SEND
<category> Additional qualifications android.intent.category.DEFAULT
<data> URI scheme / MIME type android:mimeType="image/*" 
<!-- Activity that receives shared images -->
<activity android:name=".ImageShareActivity">
    <intent-filter>
        <action   android:name="android.intent.action.SEND" />
        <category android:name="android.intent.category.DEFAULT" />
        <data     android:mimeType="image/*" />
    </intent-filter>
    <!-- Also handle multiple images -->
    <intent-filter>
        <action   android:name="android.intent.action.SEND_MULTIPLE" />
        <category android:name="android.intent.category.DEFAULT" />
        <data     android:mimeType="image/*" />
    </intent-filter>
</activity>

Binder IPC

Under the hood, all inter-component and inter-process communication in Android is mediated by Binder, a custom IPC mechanism built into the Linux kernel as a character device (/dev/binder). Binder's design allows method calls across process boundaries to look nearly identical to local method calls from the developer's perspective, while the kernel handles memory mapping, thread management, and security enforcement.[27]

When an Activity calls startActivity(), it is calling into the Activity Manager Service via Binder. When a client binds to a Service, the Service returns an IBinder reference, which the client uses to issue remote procedure calls. For high-level usage, Android's AIDL (Android Interface Definition Language) automates the generation of Binder proxy and stub classes.[28]

// AIDL definition: IRemoteService.aidl
interface IRemoteService {
    /** Returns a process identifier to the caller. */
    int getPid();
    void basicTypes(int anInt, long aLong, boolean aBoolean,
                    float aFloat, double aDouble, String aString);
}

// Service implementing the AIDL stub (Kotlin)
class RemoteService : Service() {
    private val binder = object : IRemoteService.Stub() {
        override fun getPid() = Process.myPid()
        override fun basicTypes(anInt: Int, aLong: Long,
                                 aBool: Boolean, aFloat: Float,
                                 aDouble: Double, aString: String) { }
    }
    override fun onBind(intent: Intent): IBinder = binder
}
Kotlin / AIDL

The Application Class

The Application class is the base class for maintaining global application state. It is instantiated before any other class when the app's process is created. One or more activities, services, and receivers constitute an application instance, and they share the same Application object within the same process.[29]

Custom Application subclasses are the appropriate place to initialise app-wide singletons such as dependency injection frameworks (Hilt, Dagger), analytics SDKs, or logging infrastructure.

class MyApplication : Application() {

    override fun onCreate() {
        super.onCreate()
        // Initialise Hilt (DI), Timber (logging), etc.
        Timber.plant(Timber.DebugTree())
    }
}
Kotlin

Jetpack & Modern Android Development

Android Jetpack is a suite of libraries that complement and extend the core framework components. Its architecture components — ViewModel, LiveData, Room, WorkManager, and the Navigation Component — address the most common pain points developers encounter when working with the raw Activity/Service/Intent API.[30]

Jetpack Library Replaces / Wraps Problem Solved
ViewModel Activity/Fragment state Survives config changes; no memory leaks
Room SQLiteOpenHelper + ContentProvider Type-safe ORM; coroutine/Flow support
WorkManager JobScheduler / AlarmManager / Service Guaranteed deferred background work
Navigation Intent / FragmentManager Declarative nav graph; safe args; deep links
DataStore SharedPreferences Async, coroutine-safe preference storage

The contemporary architecture recommendation from Google is MVVM with a Repository layer, often combined with Jetpack Compose for UI and Hilt for dependency injection. This approach decouples business logic from the Android framework lifecycle, making components individually testable and the codebase dramatically more maintainable.[31]
