Proof of Identity

A field guide to API authentication — the mechanisms that decide who gets through the door, why the modern web settled on tokens, and how to choose among Basic, JWT, OAuth, OpenID Connect and SAML without getting burned.

Topic  API Authentication Format  Technical Reference Citations  Vancouver

Every request that arrives at an API is, in effect, a stranger knocking. Authentication is the process of answering a single question before any resource changes hands: is this caller who it claims to be? Get that answer wrong and everything downstream — rate limits, billing, authorization, audit logs — is built on sand.

It helps to separate two words that are routinely conflated. Authentication establishes identity; authorization decides what an established identity is permitted to do. The distinction is not pedantic. The Open Worldwide Application Security Project (OWASP) explicitly warns engineers that some of the best-known standards in this space are not authentication at all — OAuth is an authorization framework, and API keys identify a client application rather than authenticate a human user.1 Treating one as the other is a recurring source of breaches.

Over HTTP, authentication rides on a small, well-defined framework. RFC 7235 specifies the challenge–response pattern that nearly every scheme below reuses: a server rejects an unauthenticated request with a 401 Unauthorized status and a WWW-Authenticate header describing how to proceed, and the client retries with an Authorization header carrying its credentials.2 What changes from scheme to scheme is the content of that header and the trust machinery behind it.

“Don't reinvent the wheel in authentication, token generation, or password storage. Use the standards.”

That advice — paraphrased from OWASP's API Security guidance1 — is the spine of this article. The sections that follow walk through the major standardized mechanisms in roughly the order a working system tends to encounter them: the blunt simplicity of Basic Authentication, the stateless economy of tokens and JSON Web Tokens, the server-memory model of cookies and sessions, and finally the three federation heavyweights — OAuth 2.0, OpenID Connect and SAML — that let one party vouch for a user to another.

§ 01

A short history of letting strangers in

The vocabulary of API authentication accumulated over roughly two decades. Each standard below was a reaction to a concrete failure of the one before it — usually the discovery that users were handing their passwords to software that had no business holding them.

2005

OpenID 1.0 and SAML 2.0

Brad Fitzpatrick, creator of LiveJournal, publishes the first OpenID authentication protocol, aiming at decentralized, user-centric login.14 The same year, OASIS ratifies SAML 2.0 as a standard, superseding SAML 1.1.15

2006

OAuth begins as a side problem

While building Twitter's OpenID implementation, engineer Blaine Cook finds there is no open standard for delegating API access without sharing a password — the question that becomes OAuth.10

2007

OAuth Core 1.0

A small group of implementers, soon joined by Google and AOL, releases the OAuth Core 1.0 final draft in October.10

2012

OAuth 2.0 — RFC 6749

The IETF publishes the OAuth 2.0 Authorization Framework, which replaces and obsoletes the OAuth 1.0 protocol (RFC 5849).9

2014

OpenID Connect

OpenID Connect 1.0 is officially released — a fresh identity layer built on top of OAuth 2.0 rather than the older XML-based OpenID.14

2015

JWT and Basic Auth get formal specs

RFC 7519 standardizes the JSON Web Token,5 and RFC 7617 re-specifies the "Basic" HTTP scheme, obsoleting the long-serving RFC 2617.3

2023

OWASP API Security Top 10

"Broken Authentication" is listed as the second-most-critical API risk (API2:2023), reflecting how often the mechanisms below are implemented incorrectly.1

§ 02

Basic Authentication

RFC 7617 Stateless Credential-bearing HTTPS required

The "Basic" HTTP authentication scheme is the simplest mechanism still in common use. Defined by RFC 7617 — which obsoletes the older RFC 2617 — it transmits credentials as a user-id and password pair, joined by a colon and encoded with Base64.3 The encoded string is placed in the Authorization header, prefixed with the keyword Basic. 

# Client request after a 401 challenge
GET /api/v1/account HTTP/1.1
Host: example.com
Authorization: Basic YWxhZGRpbjpvcGVuc2VzYW1l

# The token decodes trivially to:  aladdin:opensesame

That last comment is the entire problem. Base64 is an encoding, not encryption — it is fully and trivially reversible by anyone who sees it. RFC 7617's own Security Considerations are blunt: Basic Authentication is not a secure method of user authentication and does not protect the transmitted data, which crosses the network in cleartext.3 Mozilla's documentation reaches the same conclusion and states plainly that the scheme offers very poor security and must be paired with HTTPS/TLS so the transport layer can encrypt the exchange.2

When it still earns its keep

Despite its weakness, Basic Auth is widely supported and easy to set up,2 which keeps it alive for internal service-to-service calls behind TLS, quick prototypes, and machine clients where a long random string is used in place of a human password. It should never guard a public, user-facing API on its own.

§ 03

Token-based authentication

Pattern Stateless Bearer credential

Token-based authentication breaks the habit of sending a password with every request. The user authenticates once; in exchange the server issues a unique, time-limited access token. From then on the client presents that token to reach protected resources, and the original credentials are never transmitted again while the token remains valid.4

The defining property is statelessness. With a self-contained token, no server-side session record is persisted; the credential is attached to each request — conventionally as Authorization: Bearer <token> — and validated on arrival.4 Because nothing about the caller lives in a particular server's memory, any node in a cluster can answer any request, which makes the pattern scale cleanly across horizontally distributed services.4 Tokens also travel well: the same token can authenticate browser apps, mobile clients and cross-domain calls without the cookie machinery described later.4

  1. The client submits credentials once to an authentication endpoint.
  2. On success, the server generates and returns an access token.
  3. The client stores the token and attaches it to every subsequent request.
  4. The server validates the token and serves the resource — no re-login required until the token expires.4

"Token" is a category, not a format. A token can be an opaque random string whose meaning is looked up in a store, or it can be a structured, signed document that carries its own meaning. The dominant example of the second kind — and the reason token-based auth became the default of the modern API — is the JSON Web Token.

§ 04

JSON Web Tokens (JWT)

RFC 7519 Self-contained Signed (JWS) URL-safe

A JSON Web Token, standardized as RFC 7519, is a compact and URL-safe way of representing a set of claims to be transferred between two parties.5 Those claims are encoded as a JSON object and carried inside a JSON Web Signature (JWS) structure — which lets them be digitally signed or integrity-protected with a message authentication code — or, less commonly, inside a JSON Web Encryption (JWE) structure.5 In practice "JWT" almost always means a signed JWS token: anyone can read it, but its contents cannot be altered without detection.

Anatomy of the token

Three parts, two dots

A signed JWT is three Base64URL-encoded segments joined by period characters.6 Each segment has a distinct job:

  • Header — a small JSON object naming the token type and the signing algorithm, for example HS256.6
  • Payload — the JSON Web Token Claims Set: the name/value pairs asserting information about the subject.5
  • Signature — the value that binds header and payload together and lets a recipient verify integrity. It is produced by signing the encoded header and payload with the algorithm named in the header.6
// 1 — HEADER
{ "alg": "HS256", "typ": "JWT" }

// 2 — PAYLOAD (claims)
{ "sub": "1234567890",
  "iss": "https://auth.example.com",
  "exp": 1716100000 }

// 3 — SIGNATURE
HMACSHA256( base64url(header) + "." + base64url(payload), secret )

// Final token: header.payload.signature
eyJhbGciOiJIUzI1NiJ9.eyJzdWIiOiIxMjM0..."

RFC 7519 registers a starter set of registered claim names, deliberately abbreviated to keep tokens compact. Among the most used are iss (issuer — who minted the token), sub (subject — who the token is about), aud (audience — who it is for), exp (expiration time, after which the token must be rejected), nbf (not-before) and iat (issued-at). None are individually mandatory; applications declare which they require.5

Signing

How a JWT becomes trustworthy

The signature is what separates a JWT from a sticky note. The JWT family supports several algorithm classes: symmetric HMAC variants (HS256/384/512), which use one shared secret for both signing and verification; and asymmetric RSA (RS*) and ECDSA (ES*) variants, which sign with a private key and verify with a public one — useful when many services must validate tokens without holding the power to mint them.7 Verification means recomputing the signature and checking it against the algorithm declared in the header; if it fails, the token has been tampered with or did not come from a trusted issuer.6

Why JWTs took over

Because a signed JWT carries identity and permissions inside itself, a service can make authentication and authorization decisions without a database lookup or a server-side session store.7 That statelessness reduces lookups and improves scalability, which is why JWTs are the natural fit for distributed, microservice-style architectures where traditional session storage would be impractical.7

The same self-containment cuts both ways. A signed-but-not-encrypted JWT is readable by anyone, so it must hold no secrets. And because validity lives in the token rather than the server, a stolen token is usable until it expires — RFC 7519 anticipates this with the exp claim, and well-built systems keep access-token lifetimes short while handling renewal separately. The specification also defines an alg value of none for unsecured tokens; accepting it in production is a classic, dangerous mistake, which is why verifiers must strictly enforce the expected algorithm.7

§ 05

Cookie-based (session) authentication

Stateful Server-side session Browser-native

Cookie-based authentication is the older, server-centric counterpart to tokens. HTTP is fundamentally stateless — each request arrives with no memory of the last — yet a logged-in experience requires the server to recognize a returning user. Sessions solve this by keeping the state on the server.8

  1. The user logs in with credentials.
  2. The server creates a session, stores the session data in memory or a database, and generates a unique session ID.
  3. That session ID is sent back to the browser and stored in a cookie.
  4. The browser automatically attaches the cookie to every later request; the server matches the session ID against its stored session data to recognize the user.8

The contrast with tokens is precise. Session-based authentication is stateful: it relies on server memory, so the server must track every active session — and in a multi-server deployment that session state has to be shared or replicated, which constrains scaling.4 A self-contained token carries all the necessary information itself, letting servers validate a request without storing anything, which is what makes the token model easier to scale horizontally.8 The redeeming strength of sessions is the mirror image: because the server owns the state, a session can be expired or invalidated instantly, whereas revoking a still-valid stateless token is genuinely harder.4

Hardening the cookie

A session cookie is only as safe as its flags. Setting HttpOnly keeps the cookie out of reach of client-side scripts; Secure forces transmission over HTTPS only; and the SameSite attribute restricts third-party use, mitigating Cross-Site Request Forgery (CSRF) attacks.8 Sessions remain an excellent default for single-domain web applications; their friction shows up with mobile clients, single-page apps, and cross-origin APIs.4

A useful clarification: cookies versus the Authorization header is a question of transport, while sessions versus tokens is a question of where state lives. The two axes are independent — a server can perfectly well deliver a JWT inside a cookie, or expect a stateful session ID in an Authorization header. Neither approach is universally better.4

§ 06

OAuth 2.0 — delegated authorization

RFC 6749 Authorization framework Delegated access

OAuth answers a question the schemes above do not: how can a third-party application be granted limited access to a user's data without ever seeing that user's password? Before OAuth, the only option was for the user to hand the password to the third party outright — an arrangement that gave the third party unlimited, unrevocable, impersonating power.

The OAuth 2.0 Authorization Framework, RFC 6749, was published by the IETF in October 2012 and replaced and obsoleted the earlier OAuth 1.0 protocol of RFC 5849.9 Its abstract states the goal directly: OAuth 2.0 enables a third-party application to obtain limited access to an HTTP service, either on behalf of a resource owner — by orchestrating an approval interaction between the owner and the service — or on the application's own behalf.9 Instead of a password, the third party receives an access token that carries a defined scope and an expiry, and that can be revoked independently.10

Origins

A valet key for the web

OAuth grew out of a practical itch. Around November 2006, while working on Twitter's OpenID implementation, Blaine Cook needed a way to delegate access to the Twitter API and found no open standard for it. With collaborators including Chris Messina, David Recordon and Larry Halff, an effort coalesced into a small implementers' group in early 2007 — attracting interest from Google — and produced the OAuth Core 1.0 final draft in October 2007.10 The community's own analogy endures: OAuth is the valet key of the web — a credential that lets a service do a specific, limited job without unlocking everything you own.10

The crucial caveat

OAuth is authorization, not authentication

This is the single most consequential point about OAuth, and the one most often gotten wrong. OAuth is an authorization framework: it lets you grant access to your resources without sharing your identity or its secret parts.10 An access token says this app may read your photos; it does not reliably say who you are. OWASP states the rule without qualification — OAuth is not authentication — and using it as a login mechanism on its own is sometimes called "pseudo-authentication".1 Closing that gap is precisely the job of the next standard.

§ 07

OpenID and OpenID Connect

OIDC 1.0 Authentication layer Single sign-on ID token (JWT)

OpenID is an open standard for decentralized authentication: it lets a person prove their identity to many websites using one set of credentials held by an identity provider, instead of creating a separate password at every site. The original protocol was developed in 2005 by Brad Fitzpatrick while he was building LiveJournal; it let cooperating sites — called relying parties — defer authentication to a single identity provider.14 Early OpenID grew quickly but had real design limits, and the version most systems use today is a different beast.

The modern standard

OpenID Connect: identity on top of OAuth

OpenID Connect (OIDC), officially released in 2014, is a thin identity layer built directly on the OAuth 2.0 framework.14 It exists precisely to fix the gap from the previous section. Where OAuth produces an access token that describes permissions, OIDC adds an ID token — a security token, issued by the authorization server, that lets a client verify the identity of the end user and obtain basic profile information about them.12 Crucially, that ID token is itself a JSON Web Token, signed with public-key cryptography, tying this section back to §04.12

The vocabulary maps onto OAuth's: the OpenID Provider (OP) authenticates the user, and the Relying Party (RP) is the application that trusts the OP's verdict. Each time a user signs in, they are redirected to their provider, authenticate there, and are redirected back — which is the mechanism behind "single sign-on" (SSO) and the familiar Sign in with… button.13 Providers publish their endpoints and public signing keys at a well-known discovery document, so client libraries can configure themselves automatically.12

The one-line summary

OAuth 2.0 governs what an application is allowed to do; OpenID Connect, layered on top, establishes who the user is. OIDC verifies identity and issues the ID token; OAuth handles delegated access to resources.13 They are complementary, and modern login systems almost always use them together.

§ 08

SAML — the enterprise federation standard

OASIS SAML 2.0 XML-based Identity federation Enterprise SSO

The Security Assertion Markup Language (SAML) addresses the same goal as OpenID Connect — single sign-on and cross-domain identity — but it predates the JSON era and is built on XML. SAML 2.0 was ratified as an OASIS Standard in March 2005, superseding the earlier SAML 1.1.15 Two decades later it remains entrenched in enterprise environments, corporate intranets and government systems.

SAML 2.0 is an XML-based protocol that uses security tokens containing assertions to pass information about a principal — usually an end user — between two parties: a SAML authority called the Identity Provider (IdP) and a SAML consumer called the Service Provider (SP).15 By passing signed assertions instead of credentials, SAML enables web-based, cross-domain SSO and removes the need to distribute multiple authentication tokens to the same user.15

What an assertion carries

Assertions, statements and signatures

A SAML assertion is a packet of trusted facts. It can convey several kinds of statement about its subject — that the subject was authenticated, the attributes describing them, and authorization decisions — and each assertion identifies the issuing identity provider so the recipient knows whom it is trusting.16 The framework communicates user authentication, entitlement and attribute information between trusted IdPs and SPs.16 Integrity rests on XML digital signatures: assertions are signed (and may be encrypted) so a service provider can be confident the contents were not forged or altered in transit.

A typical browser SSO flow begins when a user requests a protected resource at a service provider. If no valid security context exists, the SP issues a SAML authentication request and the user's browser is directed to the identity provider; the IdP authenticates the user and returns a signed response containing an assertion, which the SP validates before granting access.15 The pattern is conceptually identical to OIDC's redirect dance — the difference is the XML envelope and the heavier protocol surface, which is why newer consumer and mobile applications generally reach for OpenID Connect while SAML holds its ground in established enterprise estates.

§ 09

The mechanisms, side by side

No single mechanism is "best"; each encodes a different trade-off between simplicity, statefulness, and the trust relationships it can express. The table below summarizes the eight approaches discussed above.

Comparison — API authentication mechanisms
Mechanism Primary purpose State model Best-fit context
Basic Auth Identify a caller via username + password Stateless Internal calls behind TLS; prototypes — never alone in public
Token-based Re-usable credential after a single login Typically stateless APIs, mobile and SPA clients, distributed services
JWT Self-contained, signed claims transport Stateless Microservices; stateless sessions; the format inside OIDC
Cookie / session Maintain a logged-in browser session Stateful (server-side) Single-domain web apps needing instant revocation
OAuth 2.0 Delegated, scoped access for third parties Token-based Granting apps limited access without sharing passwords
OpenID Connect Verify user identity (authentication) Token-based (ID token) Consumer SSO; "Sign in with…"; modern login
SAML 2.0 Cross-domain identity federation Assertion-based Enterprise & government SSO; established estates
§ 10

Getting it wrong — and getting it right

Choosing a standard is the easy part; implementing it correctly is where systems fail. OWASP ranks Broken Authentication as the second-most-critical API risk in its 2023 API Security Top 10 (API2:2023), noting that authentication in APIs is genuinely complex and that engineers frequently misplace its boundaries.1 Exploited weaknesses translate directly into account takeover and unauthorized access.

OWASP's guidance condenses into a handful of durable rules — each of which pushes back against a temptation visible somewhere in the sections above:

  • Use the standards; don't improvise. Do not reinvent authentication, token generation, or password storage — use established, reviewed mechanisms.1
  • Know every entry point. Map all authentication flows — web, mobile, deep links, one-click sign-in — because an overlooked flow is an unguarded door.1
  • Treat password recovery like login. Credential recovery and "forgot password" endpoints need the same brute-force, rate limiting and lockout protection as the login endpoint itself.1
  • Resist automated attacks. Apply anti-brute-force controls — stricter than ordinary rate limits — plus account lockout or CAPTCHA and weak-password checks, to blunt credential stuffing and dictionary attacks.1
  • Don't confuse the credential types. API keys authenticate client applications, not human users, and should not be used for user authentication.1
  • Never trust cleartext transport. Because Basic Auth and bearer tokens are readable by anyone who intercepts them, every scheme here depends on TLS/HTTPS underneath it.2

Two cross-cutting themes recur. The first is transport: every mechanism above — Basic credentials, session cookies, JWTs, OAuth tokens, SAML assertions — assumes an encrypted channel and offers little protection without one.2 The second is revocation: the statelessness that makes tokens scale also makes them hard to cancel mid-life, which is the practical argument for short token lifetimes, careful storage, and — where instant invalidation matters — server-side sessions.4

§ 11

Choosing well

The eight mechanisms in this guide are not competitors so much as a layered toolkit. A single production system might use Basic Auth between two internal services, issue JWTs to its mobile clients, keep server-side sessions for its web dashboard, accept OAuth tokens from partner integrations, offer OpenID Connect for consumer login, and speak SAML to a corporate customer's identity provider — each chosen because it fits a specific trust relationship.

A workable decision path follows the questions this article has raised in order. Is the caller a machine or a person? — that separates API keys and Basic Auth from user-facing flows. Does state need to live on the server or in the credential? — that separates sessions from tokens. Is a third party involved, and do you need to know who the user is or only what an app may do? — that separates plain OAuth from OpenID Connect. And is the counterparty a modern app or an established enterprise? — that often decides OIDC versus SAML.

Whatever the answer, the constant holds: authentication is the foundation every other guarantee rests on. Reach for a reviewed standard, run it over TLS, implement it against published guidance rather than intuition, and treat the stranger at the door with the seriousness the question deserves.1
