Comprehensive Internationalization in Angular with @angular/localize

Internationalization (i18n) enables an Angular application to reach a global audience by adapting its user interface to different languages and regions. Angular provides a robust, compile‑time i18n solution built on the @angular/localize package. This article explores the key concepts—locale identifiers, translation file extraction, handling multiple locales, and building locale‑specific application bundles—with practical examples and clear explanations.

1. Introduction to Angular’s i18n Model

Angular’s internationalization approach is template‑based and performs locale‑specific compilation. Unlike runtime translation libraries that load message catalogs in the browser, Angular’s built‑in i18n compiles the translated text directly into the application bundle for each target locale. This results in faster rendering and smaller payloads because no translation resolution happens at runtime [1].

The process revolves around three pillars:

  • Marking static text in component templates with the i18n attribute.
  • Extracting those messages into a standard translation file (XLIFF).
  • Building and serving separate bundles for each required locale, each containing the translated text.

The @angular/localize package, available from Angular 9 onward, provides the core runtime for translated templates as well as the CLI tooling that drives message extraction and locale‑aware builds [1][2].

2. Setting Up @angular/localize

To enable i18n in a project, add the @angular/localize package. For new Angular CLI workspaces (v17+) this package is included by default. In existing projects, run: 

ng add @angular/localize

The package must also be imported into the application’s main polyfills if you plan to use runtime locale data for features such as date, number, and currency pipes. The CLI automatically adds import '@angular/localize/init'; to the polyfills.ts file when using the schematic [1].

3. Locales by ID

A locale identifier (locale ID) follows the Unicode CLDR convention, typically a combination of a language code and an optional region subtag [3]. Examples include en-US, fr, de-DE, and es-MX. Angular uses these identifiers to:

  • Load the appropriate locale data for pipes (DatePipe, CurrencyPipe, etc.).
  • Select the correct translation file during the build.
  • Set the LOCALE_ID token, which can be injected for further customization.

When configuring multiple locales in angular.json, each locale is represented by its ID, enabling the CLI to produce a distinct output for each one.

4. Marking Text for Translation

Any static text that should be translated is annotated with the i18n attribute in the component template. The attribute can optionally include a meaning and description to help translators. 

<h1 i18n="site header|An introduction heading@@siteHeader">
  Welcome to our store
</h1>

<p i18n>Your cart is empty.</p>

Angular also supports ICU MessageFormat for handling plurals and selections directly in templates. This allows a single message to adapt based on a numeric value or a string selector [5]. 

<span i18n>
  {minutes, plural,
    =0 {just now}
    =1 {one minute ago}
    other {{{minutes}} minutes ago}
  }
</span>

The ICU expressions are fully extracted and translated in the same XLIFF file, preserving the format placeholders and selectors.

5. Extracting Translation Files

The Angular CLI provides the extract-i18n command to generate a source translation file from all marked messages in the project. The default output format is XLIFF 1.2, but XLIFF 2.0 and XMB are also available [1]. 

ng extract-i18n --output-path src/locale

This creates a file named messages.xlf (or messages.xlf2 for XLIFF 2.0) in the specified output directory. The generated file contains a <trans-unit> for each unique message, with a unique ID derived from the source text and optional meaning [4].

A typical entry looks like: 

<trans-unit id="siteHeader" datatype="html">
  <source>Welcome to our store</source>
  <context-group purpose="location">
    <context context-type="sourcefile">src/app/app.component.html</context>
    <context context-type="linenumber">1</context>
  </context-group>
  <note priority="1" from="description">An introduction heading</note>
</trans-unit>

6. Working with Translation Files

After extraction, the file is duplicated and handed to translators. For a French version, one would create a copy and add the <target> element inside each <trans-unit>: 

<trans-unit id="siteHeader" datatype="html">
  <source>Welcome to our store</source>
  <target>Bienvenue dans notre boutique</target>
  ...
</trans-unit>

The translated file is then stored alongside the source, for example src/locale/messages.fr.xlf. Angular’s build system can reference this file when compiling the application for a specific locale.

It is important to keep the original message ID and structure intact. The @angular/localize runtime matches the ID from the template at compile time and replaces the source text with the translated target [1].

7. Building for Multiple Locales

To produce separate application bundles for multiple languages, the project’s angular.json must define each locale under the i18n section. The sourceLocale represents the original language of the templates, and the locales map lists every target locale together with its translation file path. 

{
  "projects": {
    "my-app": {
      "i18n": {
        "sourceLocale": "en-US",
        "locales": {
          "fr": {
            "translation": "src/locale/messages.fr.xlf",
            "baseHref": "/fr/"
          },
          "de": {
            "translation": "src/locale/messages.de.xlf",
            "baseHref": "/de/"
          }
        }
      },
      "architect": {
        "build": {
          "options": {
            "localize": true
          }
        }
      }
    }
  }
}

Setting "localize": true in the build options enables locale‑aware compilation. When the project is built (ng build), the CLI generates one output folder per defined locale, each containing the fully translated application. The baseHref property ensures the <base href> tag points to the correct sub‑directory [2].

If only a subset of locales is desired, localize can accept an array of locale IDs, e.g., "localize": ["fr"]. This allows selective building without altering the configuration repeatedly.

8. Serving and Testing

During development, the localize option can also be used with ng serve. However, because Angular serves only one locale at a time, you must specify the target locale. The simplest way is to create a dedicated build configuration per locale: 

"configurations": {
  "fr": {
    "localize": ["fr"]
  },
  "de": {
    "localize": ["de"]
  }
}

Then run: 

ng serve --configuration=fr

The development server will compile the application with French translations and serve it on the default port. Note that ng serve with localization does not automatically switch languages; it only compiles one locale per serving session [2].

9. Beyond Static Builds: Dynamic Locale Switching

Angular’s compile‑time i18n does not natively support switching languages at runtime without a full page reload, because the translations are baked into the template factory. For applications that need dynamic language changes (e.g., user selection from a dropdown), a common strategy is to deploy each locale under its own URL path and use server‑side redirection or a simple page reload. Alternatively, the loadTranslations() function from @angular/localize can be used in combination with Just‑In‑Time (JIT) compilation to load translation data during bootstrap, but this approach carries a performance cost and is not recommended for production without careful consideration [1].

10. Conclusion

The @angular/localize package provides a cohesive, compile‑time i18n pipeline that integrates deeply with Angular’s rendering engine. By marking templates, extracting XLIFF translation files, and configuring locale‑specific builds, developers can efficiently serve fully translated applications to diverse audiences. Following the locale‑by‑ID convention ensures compatibility with CLDR locale data, while the multi‑locale build support in angular.json makes it easy to generate all required bundles in a single command.

Adopting this official approach not only guarantees optimal performance but also leverages the standard XLIFF format, enabling seamless collaboration with professional translation tools and localization teams.
